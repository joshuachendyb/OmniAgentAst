## 一、分析背景

本分析基于当前（2026-06-13）实际代码架构，该架构与 2026-06-12 版旧文档（`001` 号）描述的 CRSS/Intent 多系统架构完全不同：

| 维度 | 旧文档(001号) | 当前实际 |
|------|-------------|---------|
| 意图检测 | CRSS 4层评分 + 多意图分类 | **无意图检测，单 UniversalAgent** |
| 工具隔离 | 5类 IntentType → 5类 AgentConfig | **无分类隔离，统一工具注册表** |
| 映射层数 | 5层映射 | **0层映射，直接_loaded_categories** |
| 工具发现 | CRSS 静态配置 | **tool_search 动态发现** |
| LLM协议 | 混合(text+FC) | **纯 FC-only** |

**分析范围**: 同 001 号文档分析范围，但基于实际存在的文件路径。

### 2.2 工具动态加载链路

```

  │
  ▼
tool_search(query) → 在全局注册表搜索匹配工具

  │  self._loaded_categories.add(cat)
  │  self._tool_manager.load_category(cat)

  ▼
下一轮_ call_llm():
  self._get_openai_tools() → tool_registry.to_openai_tools(categories=self._loaded_categories)
  → 只暴露已加载分类的工具给LLM
```

### 2.3 工具执行链路

```
handle_action(agent, parsed, chunk_buffer)
  │
  ├─ check_safety_and_confirm(agent, all_calls, step)
  │     ├─ for each call: safety_checker.check_before_execute(tool_name, params)
  │     │     ├─ 安全级别判定: _get_effective_safety_level()
  │     │     ├─ 已知风险检测: _check_known_risks()
  │     │     └─ 返回 {blocked, requires_confirmation, ...}
  │     ├─ blocked → ErrorStep + FAILED
  │     └─ requires_confirmation → IncidentStep(authorization_required) → await确认

  └─ build_observation(agent, ...)
        ├─ ActionToolStep × N
        ├─ observation 文本 (format_llm_observation)
        ├─ _append_observation → fc_context → message_builder
        └─ ObservationStep × 1
```

## 三、发现的问题

### ⚠️ 问题1（P0）：`_find_tool` 兜底全局注册表，绕过分类隔离

**位置**: `tool_retry_engine.py:116-122`

```python
def _find_tool(self, action: str) -> Optional[Callable]:
    tool = self._tools.get(action)              # ① 先查Agent已加载的工具
    if tool is None:
        from app.services.tools.registry import tool_registry
        tool = tool_registry.get_implementation(action)  # ② 兜底到全局注册表!
    return tool
```

**问题**: 第②行是全局 `ToolRegistry` 单例，包含**所有分类的全部工具**。

**风险场景**:
- LLM 通过 `tool_search` 发现了一个未加载分类的工具（比如 `execute_shell_command`）
- LLM 直接调用该工具名（即使它不在当前 `_loaded_categories` 中）
- `_find_tool` 第①步失败后，第②步从全局注册表找到并执行

**缓解因素**:
- ✅ `_get_openai_tools` 现在正确传 `categories=self._loaded_categories`（不是 `None`），所以 LLM **不可见**未加载分类的工具
- ❌ 但 `tool_search` 搜索结果包含所有工具名，LLM 可以从中"看到"未加载分类的工具名

**实际攻击路径**: `tool_search` → LLM知道某工具名 → LLM直接调用 → `_find_tool` 全局兜底 → 工具执行（通过安全检查后）

**风险等级**: **P0-紧急**

---

### ⚠️ 问题2（P1）：`return_direct` 完整功能是死代码

**现象**: `return_direct` 在3处定义/传递，但**既不传播到 ObservationStep，也不在 ReAct 循环中被检查**。

**死代码链路**:

```
工具层: _response.py → build_success/build_error 接受 return_direct 参数
  │  仅当值≠False时才写入result
  ▼
Agent层: tool_result_factory.py → create_tool_result 接受 return_direct
  │  委托 _response, 追加 return_direct
  ▼
格式化层: observation_formatter.py → build_execution_result_dict
  │  result["return_direct"] = execution_result.get("return_direct", False)
  ▼
action_handler.py: build_observation → 构建 ObservationStep
  ❌ 没有传 return_direct=first_result.get("return_direct", False) ← **缺失**
  → ObservationStep 永远是 return_direct=False
  ▼
react_cycle.py: _process_single_step
  ❌ 没有在 handler 返回后检查 observation 的 return_direct ← **缺失**
  → 循环无条件继续
```

**两处缺失**:
1. `action_handler.py:147-157` — `ObservationStep` 构造函数没有传入 `return_direct`
2. `react_cycle.py` — `_process_single_step` / `run_react_cycle` 没有检查 `return_direct`

**后果**: 即使某个工具设置了 `return_direct=True` 表示"请直接返回这个结果作为答案"，系统也会忽略，继续走 ReAct 循环。如果 `action_handler` 修复了缺失①，React 循环仍然不会停止。

**风险等级**: **P1-高**（功能缺失，影响依赖此特性的工具）

### ⚠️ 问题3（P1）：`tool_search` 无分类过滤，暴露所有工具

**位置**: `meta_tools.py:38-64`

```python
def tool_search(query: str) -> Dict[str, Any]:
    all_tools = tool_registry._tools    # 全局所有工具！
    for name, metadata in all_tools.items():
        # 在所有工具中搜索，无分类过滤
        ...
    top_results = scored[:10]
    return build_success(..., llm_data={"matches": [{"name": r["name"], "category": r["category"]} for r in top_results[:10]]})
```

**问题**: `tool_search` 不检查当前 Agent 的 `_loaded_categories`，返回的工具包含所有分类。虽然 `_auto_inject_from_search` 会加载匹配的分类，但 LLM 也可以在下一轮直接调用匹配结果的工具名。

**与问题1的组合风险**:
1. LLM 调用 `tool_search("文件操作")` → 返回 `write_text_file` (FILE分类，未加载)
2. LLM 下一轮直接调 `write_text_file` → `_find_tool` 全局兜底找到 → 执行

**风险等级**: **P1-高**（与问题1组合可达 P0 效果）

---

### ⚠️ 问题4（P2）：`lazy_loader` 冷启动缺陷 — 新增分类永不注册

**位置**: `lazy_loader.py:53-54`

```python
def ensure_tools_registered() -> None:
    global _tools_registered, _registered_categories
    if _tools_registered:      # ← 一旦设为 True，永久跳过
        return
    ...
    if not _failed:
        _tools_registered = True   # ← 全量成功后锁定
```

**问题**: 
- 当 `_tools_registered = True` 后，整个函数立即返回
- 后续在 `CATEGORY_MODULES` 中新增分类永远不会被注册
- `_registered_categories` 记录的只是旧分类列表

**触发条件**: 首次全量注册成功 → 新增分类到 `CATEGORY_MODULES` → 新分类不被注册

**影响范围**: 热更新/插件扩展场景

**风险等级**: **P2-中**

### ⚠️ 问题5（P2）：并行工具调用的安全检查无累积风险评估

**位置**: `action_handler.py:36-75`

```python
async def check_safety_and_confirm(agent, all_calls: List[Dict], step: int):

        safety_result = safety_checker.check_before_execute(call["tool_name"], call["tool_params"])
```

- 没有"批量操作"的概念

**风险等级**: **P2-中**

### ⚠️ 问题6（P2）：`llm_call_count` 每次 run_react_cycle 重置

**位置**: `initialize_run_state.py:22`

```python
def initialize_run_state(self, task, task_id, context=None):
    self.llm_call_count = 0   # ← 每次运行重置
```

**问题**: 
- `react_cycle.py:91` 用 `agent.llm_call_count < max_steps` 控制循环次数
- 每次 `run_react_cycle` 调用重置计数器，意味着 max_steps 是"每轮"限制而非"Agent 生命周期"限制
- 当前架构是单次 Agent（每个请求创建一次），所以目前无害
- 但如果 Agent 被复用（比如长连接场景），限制会被绕过

**风险等级**: **P2-中**

### ⚠️ 问题7（P3）：`tool_search` 评分机制过于简单

**位置**: `meta_tools.py:46-50`

```python
for word in query_words:
    if word in name_lower:      # 名字匹配 +10
        score += 10
    if word in desc_lower:      # 描述匹配 +5
        score += 5
if name_lower in query_lower or query_lower in name_lower:
    score += 8                  # 精确匹配 +8
```

**问题**:
- 纯关键词匹配，无语义理解
- 短关键词（如"文件"）可能匹配不相关工具
- 没有对分类做加权（同类工具应排序在一起）
- 没有结合 LLM 的 embedding 做语义搜索

**风险等级**: **P3-低**

### ⚠️ 问题8（P3）：`extract_status` 缺少 code 时默认 SUCCESS，可能隐藏错误

**位置**: `observation_formatter.py:84`

```python
def extract_status(result: dict) -> str:
    code = result.get("code", SUCCESS_CODE)   # ← 如果 code 缺失，默认成功
```

**问题**: 
- 如果工具返回的 result dict 意外没有 `code` 字段，会被当作成功
- 原本可能是错误信息放在了 `message` 中但没有设置 `code`
- 导致 LLM 看到的是成功态，但实际上执行异常

**风险等级**: **P3-低**

## 六、核心矛盾总结

```
设计意图: 动态加载 → 只有已加载分类的工具可被 LLM 使用
实际效果: 动态加载 → LLM 可通过 tool_search 发现任何工具并直接调用
```

与旧文档（001号）的核心矛盾相似，但表现形式不同：

| 旧架构 | 当前架构 |
|-------|---------|
| 5类 Intent → 5类 AgentConfig → 分类隔离 | 单 Agent → 动态加载 → 无硬隔离 |
| 问题：多Agent但隔离被 `_find_tool` 绕过 | 问题：单Agent但 `tool_search` + `_find_tool` 组合使"动态加载"设计形同虚设 |
| 本质：分类隔离的设计意图 vs 实现鸿沟 | 本质：动态加载的设计意图 vs `tool_search` 信息泄露 + `_find_tool` 未限制 |

**两个 P0/P1 问题的组合**（`_find_tool` 全域兜底 + `tool_search` 暴露所有工具）使得"动态加载"设计失去了安全边界。LLM 可以通过 `tool_search` 发现任何工具，然后直接通过 `_find_tool` 调用它，完全绕过加载流程。
