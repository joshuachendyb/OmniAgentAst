## 一、当前ReAct实现情况及问题分析

### 1.1 ReAct实现现状

当前后端 chat.py 中的流式响应实现了8种type，分别是：
- start、thought、action_tool、observation、final、chunk、error、incident

代码位置：`backend/app/api/v1/chat.py`

### 1.2 意图检测函数问题

**问题现象**：LongCat模型无法执行文件操作（查看桌面、查看E盘），只返回"不具备执行能力"等文本答案。

**问题根因**：`detect_file_operation_intent()` 函数有两个问题：

1. **完整词匹配**：必须精确匹配"查看文件"或"查看目录"，无法识别"查看桌面目录的文件有什么"等自然语言
2. **阈值0.6太高**：导致意图检测失败后，程序走普通LLM路径，LLM不知道有工具可用

**时间线**：
- 3月5日：阈值从0.2提高到0.6
- 3月11日：重写为完整词匹配
- 3月8日：用户最后一次正常使用
- 3月10日后：突然失效

**已修复**：2026-03-14将阈值从0.6降到0.2，并扩展关键词列表。

## 三、第5章和第11章关键内容总结 【2026-03-19 05:39:16 小沈追加】

### 3.1 8种type分类

| 场景 | 流程 | 使用的type |
|------|------|-----------|
| **普通对话** | start → chunk → final | start, chunk, final |
| **ReAct** | start → thought → action_tool → observation → ... → final | start, thought, action_tool, observation, final |
| **ReAct（意外）** | start → ... → incident → ... → final | + incident |

### 3.2 8种type说明

| type | 说明 | 普通对话 | ReAct中 | step |
|------|------|---------|---------|------|
| start | 任务开始 | ✅ | ✅ | ✅ |
| thought | LLM思考 | 🔴 | ✅ | ✅ |
| action_tool | 执行动作 | 🔴 | ✅ | ✅ |
| observation | 执行结果 | 🔴 | ✅ | ✅ |
| final | 最终回复 | ✅ | 🔴 | ✅ |
| chunk | 流式片段 | ✅ | 🔴 | 🔴 |
| error | 错误 | ✅ | ✅ | ✅ |
| incident | 执行状态 | ✅ | ✅ | ✅ |

### 3.3 关键设计点

1. **chunk与ReAct互斥**：普通对话有chunk，ReAct没有chunk
2. **ReAct退出点**：只在thought阶段，action_tool=finish时进入final
3. **observation设计**：根据文档，observation阶段应该再调用LLM决定下一步

### 3.4 字段设计原则（来自第5章）

| 原则 | 说明 |
|------|------|
| 原则1 | 每个type只表达当前阶段信息，不嵌套 |
| 原则2 | type名称不能出现在字段名 |
| 原则3 | 字段完整准确 |
| 原则4 | 字段名称清晰准确、互不重叠 |
| 原则5 | 字段来源必须准确说明 |

## 四、新8种type的字段汇总及阶段的UI布局设计说明

本文档定义了流式ReAct API的8种type，用于前端识别和处理不同类型的消息。

### 4.0  板块总体布局（通用）

**5种核心type对应5种板块模式**，大体结构相同，内部显示信息和样式略有差异。

#### 板块结构图

```
┌─────────────────────────────────────────────────┐
│ [步骤1][start] [正在思考...]              [状态]│  ← 第一行：步骤+type名称+占位+状态
│ ─────────────────────────────────────────────── │
│                                                 │
│ [内容区...]                                      │  ← 内容区（第一行下方）
│   - 根据不同type显示不同内容                      │
│   - 分行显示                                    │
│                                                 │
└─────────────────────────────────────────────────┘
 
┌─────────────────────────────────────────────────┐
│ [步骤2][thought] [正在思考...]             [状态]│
│ ...
```

#### 第一行格式

| 位置 | 内容 | 说明 |
|------|------|------|
| 左侧 | 步骤No. + type名称 | 固定显示 |
| 中间偏左 | "正在思考..."占位 | 仅start/thought/observation在LLM响应前显示 |
| 右侧 | 状态 | ✅/⚠️/🚫 等 |

#### 显示规则

1. **板块组成**：每个type板块由**第一行** + **内容区**组成
2. **瀑布式显示**：板块接着板块显示，依次向下排列
3. **无折叠**：各板块之间不需要折叠功能
4. **第一行"正在思考..."占位**：
   - 仅适用于 start、thought、observation（这3种调用LLM）
   - 位置：第一行的中间偏左（在步骤No.和type名称之后）
   - LLM响应前显示"正在思考..."占位
   - LLM响应后隐藏"正在思考..." 
   - action_tool、final、chunk、error、incident 不调用LLM，无此占位

#### 4.1.3 UI显示布局

**总体布局结构**：

```
┌─────────────────────────────────────────────────┐
│ [步骤1][start] [正在思考...]              [✅] │  ← 第一行：步骤+type名称+占位+状态
│ ─────────────────────────────────────────────── │
│ [内容区...]                                      │  ← 内容区（第一行下方）
└─────────────────────────────────────────────────┘
```

**显示说明**：
- **第一行**（从左到右）：
  - 左侧：步骤No. + type名称
  - 中间偏左："正在思考..."占位（仅LLM响应前显示）
  - 右侧：状态（✅/⚠️/🚫）
- **内容区**：第一行下方，显示LLM返回的content内容

**一、核心逻辑**

```

    ↓

**二、check() 方法的判断逻辑**（`shell_security.py:527-555`）

| 检查步骤 | 逻辑 | 返回值 |
|---------|------|--------|
| 空命令 | `if not command` | `is_safe=True, risk=""` |
| **第1步：检查危险命令黑名单** | 遍历 `DANGEROUS_COMMANDS`（包含 bash/cmd/powershell/chinese 四类，共约60+条） | `is_safe=False, risk="检测到危险命令: xxx"` |
| **第2步：检查危险命令模式** | 遍历 `DANGEROUS_PATTERNS`（正则表达式，约20+条，如 `rm -rf /`、`format C:`、`dd of=/dev/` 等） | `is_safe=False, risk="检测到危险命令模式: xxx"` |
| 均未命中 | - | `is_safe=True, risk=""` |

**危险命令黑名单示例**（`shell_security.py:140-183`）：
- **Bash/Linux**：`rm -rf /`、`sudo su`、`chmod 777 /`、`nc -e`、`bash -i`、`cat /etc/shadow`
- **CMD/Windows**：`del /f /s /q`、`rd /s /q`
- **PowerShell**：`Remove-Item -Recurse -Force`、`Stop-Computer -Force`
- **中文危险指令**：格式化、重启系统、关闭防火墙等

**危险命令模式示例**（`shell_security.py:434-453`）：
- `rm\s+-[rf]+\s+/\s*` → 递归删除根目录
- `format\s+[a-zA-Z]:?` → 格式化磁盘
- `dd\s+if=/dev/zero` → 磁盘覆写

**三、get_risk_level() 方法的判断逻辑**（`shell_security.py:570-595`）

| risk_level | 触发关键词 |
|-----------|-----------|
| `critical` | `rm -rf`、`format`、`mkfs`、`dd if` |
| `high` | `kill -9`、`sudo`、`su`、`chmod 777`、`nc -e`、`ncat`、`bash -i`、`reg add` |
| `medium` | 其他任何匹配到的危险命令/模式 |
| `safe` | `is_safe=True`（check未命中） |

**二、触发场景**

| 错误码 | 触发场景 | 前端操作 |
|--------|---------|---------|
| TIMEOUT | LLM调用超时 | 显示超时，可重试 |
| NETWORK_ERROR | 网络问题 | 显示网络错误，可重试 |
| LLM_ERROR | LLM返回异常 | 显示LLM错误，可重试 |
| TOOL_ERROR | 工具执行失败 | 显示工具错误，显示详情 |
| FILE_NOT_FOUND | 文件不存在 | 显示路径错误 |
| PERMISSION_DENIED | 权限不足 | 显示权限错误 |
| SECURITY_BLOCKED | 安全拦截 | 显示拦截原因，**不可重试** |
| INVALID_INPUT | 参数错误 | 显示输入错误 |
| INTERNAL_ERROR | 系统错误 | 显示系统错误，可重试 |

**三、前端处理逻辑**

```javascript
if (data.type === 'error') {
    // 显示错误提示
    showErrorDialog({
        code: data.code,
        message: data.message,
        step: data.step
    });

    // 判断是否可重试
    if (data.retryable) {
        if (data.retry_after) {
            // 倒计时自动重试
            startCountdown(data.retry_after);
        } else {
            // 显示重试按钮
            showRetryButton();
        }
    } else {
        // 不可重试，显示结束
        endSession();
    }
}
```

**四、与incident.retrying的区别**

| 特性 | error | incident.retrying |
|------|-------|-------------------|
| 来源 | 工具执行失败 | 工具执行失败自动重试 |
| 循环 | 不参与ReAct | 参与ReAct（重试中） |
| 用户感知 | 明确错误提示 | 静默重试 |
| UI | 显示错误对话框 | 显示重试进度 |

### 4.8 type=incident（Agent执行状态）

**说明**：Loop执行过程中的意外事件，需要Agent进行处理的，整合了中断、暂停、恢复、重试四个状态。

---

#### 4.8.1 字段汇总表

| 字段 | 类型 | 必要性 | 含义 | 来源 | 5.8 | 实现否 |
|------|------|--------|------|------|--------|--------|
| type | string | 固定值 | 固定为 "incident" | 固定值 | ✅ | ℹ️ |
| incident_value | string | 必要 | 具体状态值 | 后端生成 | ✅ | ℹ️ |
| message | string | 必要 | 事件消息 | 后端生成 | ✅ | ℹ️ |

**incident_value 取值说明**：

| 值 | 说明 | 触发者 | UI需要操作 |
|-----|------|--------|-----------|
| interrupted | 任务中断 | 用户主动/系统被动 | 显示中断提示 |
| paused | 任务暂停 | 用户主动 | 显示恢复/取消按钮 |
| resumed | 任务恢复 | 用户主动 | 显示继续执行提示 |
| retrying | 任务重试中 | 系统自动 | 显示重试提示 |

---

#### 4.8.2 字段用途说明

**一、用于逻辑判断的字段**

| 字段 | 用途 | 说明 |
|------|------|------|
| type | 判断消息类型 | 固定为 "incident" |
| incident_value | 判断当前状态 | 决定UI显示什么操作按钮 |

**二、用于显示的字段（直接用于output，内容不改变）**

| 显示位置 | 字段 | 显示内容 |
|---------|------|---------|
| 第一行左侧 | type | "incident" |
| 第一行中间 | incident_value | 状态值（interrupted/paused/resumed/retrying） |
| 第一行右侧 | 状态标签 | ⏸️暂停 / ▶️继续 / 🔄重试 / ⏹️中断 |
| 内容区 | message | 事件消息 |

#### 4.8.5 补充说明

**一、incident与其他type的关系**

| 特性 | incident | thought/action_tool/observation | final/error |
|------|---------|-------------------------------|-------------|
| 是否在ReAct循环中 | ❌ 不是 | ✅ 是 | ❌ 不是（是结束） |
| 是否参与循环 | ❌ 不参与 | ✅ 参与 | ❌ 不参与 |
| 是否有step | ❌ 无step | ✅ 有step | ✅ 有step |
| 是否调用LLM | ❌ 不调用 | ✅ 调用 | ✅ 可能调用 |

**二、触发场景**

| incident_value | 触发场景 | 前端操作 |
|-------------|---------|---------|
| paused | 用户点击暂停按钮 | 显示继续/取消按钮 |
| resumed | 用户点击继续按钮 | 关闭按钮，继续流式输出 |
| retrying | 工具执行失败自动重试 | 显示重试进度，可取消 |
| interrupted | 用户点击中断/系统强制中断 | 显示中断提示，流式结束 |

**三、前端处理逻辑**

```javascript
if (data.type === 'incident') {
    switch (data.incident_value) {
        case 'paused':
            // 显示暂停提示 + 继续/取消按钮
            showPauseDialog(data.message);
            break;
        case 'resumed':
            // 显示恢复提示 + 关闭按钮
            showResumedNotice(data.message);
            break;
        case 'retrying':
            // 显示重试提示 + 可选取消按钮
            showRetryingNotice(data.message);
            break;
        case 'interrupted':
            // 显示中断提示 + 结束会话
            showInterruptedNotice(data.message);
            endSession();
            break;
    }
}
```

**四、与用户确认的区别**

| 特性 | incident (paused) | 用户确认 (is_need_confirm) |
|------|-------------------|---------------------------|
| 触发原因 | 用户主动暂停 | LLM要求确认 |
| 显示内容 | "是否继续" | 确认内容 |
| 按钮 | 继续/取消 | 确认/拒绝/修改 |
| 流程影响 | 可恢复继续 | 可修改命令 |
