## 一、设计要求

### 1.1 start阶段的处理流程

```
chat_stream.py:633-640
  1. 构建 display_name（AI显示名称）
       ↓ chat_stream.py:644-645
  2. 执行 security_check → 安全检查结果（blocked/is_safe/risk_level/risk）
       ↓ chat_stream.py:647-653
  3. 初始化 step_counter = 0，定义 next_step()
       ↓ chat_stream.py:655-670
  4. 构建 start_data 字典（含"正在思考..."占位）
       ↓ chat_stream.py:673
  5. yield start_data（发送给前端）
       ↓ chat_stream.py:677-678
  6. 保存到数据库
       ↓ chat_stream.py:681
  7. if not is_safe → yield error，return
       ↓ 安全通过后
  8. 组织LLM输入（用户消息 + security_check）
       ↓
  9. 调用LLM（流式响应）
       ↓ LLM返回
 10. LLM返回：content + is_clear + is_need_confirm
       ↓ LLM流结束
 11. 判断状态图标，统一 yield 给前端（content + 状态图标）
       ↓
 12. 进入 thought 阶段
```

#### 3.2.1 组织LLM输入（步骤8）

```python
# 构建LLM输入prompt
# 用户消息
user_message = request.messages[-1].content if request.messages else ""

is_safe = security_check_result.get('is_safe', True)
risk_level = security_check_result.get('risk_level', '')
risk = security_check_result.get('risk', '')
blocked = security_check_result.get('blocked', False)

# LLM prompt
llm_prompt = f"""用户输入: {user_message}

安全检查结果:
- 是否安全: {is_safe}
- 风险等级: {risk_level}
- 风险描述: {risk}
- 是否拦截: {blocked}

请分析:
1. 用户意图是什么？
2. 输入是否清晰明确？
3. 如果不安全或输入不清晰，说明原因？
4. 是否需要用户确认？
"""
```

#### 3.2.2 调用LLM（步骤9-10）

```python
# 调用LLM流式接口
async for chunk in ai_service.stream(llm_prompt):
    # 累积content
    content += chunk
    # 实时yield给前端（可选）
    # yield {'content': chunk, 'type': 'start_chunk'}

# 从LLM返回中提取字段（需要设计返回格式）
# 假设LLM返回JSON格式：
# {
#   "is_clear": true/false,
#   "is_need_confirm": false,  # 🔴 已废弃：固定为False，不使用用户确认功能
#   "analysis": "分析内容...",
#   "clarification_question": "如果需要澄清..." 
# }

# 解析LLM返回
import json
try:
    result = json.loads(content)
    is_clear = result.get('is_clear', True)
    # 🔴 is_need_confirm 固定为 False，不使用用户确认功能
    is_need_confirm = False
    analysis = result.get('analysis', '')
    clarification = result.get('clarification_question', '')
    content = analysis if is_clear else clarification
except:
    # 如果解析失败，使用默认值
    is_clear = True
    is_need_confirm = False  # 🔴 固定为False
    content = content
```

## 五、注意事项

### 5.1 安全性

- LLM调用失败时，需要有fallback逻辑
- 如果LLM解析失败，需要使用默认值（is_clear=True, is_need_confirm=False）

### 5.2 性能

- LLM调用应该使用流式接口，避免长时间阻塞
- content可以实时yield给前端，改善用户体验

### 5.3 兼容性

- 第二次yield的数据格式需要与前端协商
- 需要确保前端能正确解析status_icon字段
