## 三、从 Windows 桌面助手角度分析

### 3.1 核心差距

当前 Agent 只能操作文件（读写删改查），不能与 Windows 系统真实交互。

**桌面助手应该能做的，当前做不到**：

| 用户需求 | 需要的能力 | 当前状态 |
|---------|----------|---------|
| "帮我打开记事本" | 启动应用 | ❌ 不支持 |
| "把这段话复制到剪贴板" | 剪贴板操作 | ❌ 不支持 |
| "截个屏看看" | 截图 | ❌ 不支持 |
| "帮我搜索一下天气" | 网络搜索 | ❌ 不支持 |
| "帮我打开这个网页" | 浏览器控制 | ❌ 不支持 |
| "提醒我下午3点开会" | 通知 | ❌ 不支持 |
| "查看 CPU 使用率" | 系统信息 | ❌ 不支持 |
| "关闭这个程序" | 进程管理 | ❌ 不支持 |

## 五、工具框架建议

### 5.1 现在最重要的事

不是急着加工具，而是**先把工具框架搭好**：

| 优先级 | 事项 | 原因 |
|--------|------|------|
| **P0** | 工具注册机制 | 支持动态添加新工具 |
| **P0** | 工具描述标准化 | LLM 需要准确的工具描述 |
| **P1** | 工具安全机制 | 新工具需要安全审查 |
| **P1** | 工具测试框架 | 每个工具需要独立测试 |
| **P2** | 新工具实现 | 按需求逐步添加 |

### 5.2 工具注册标准

框架搭好后，加一个新工具只需要：

```
1. 定义 Pydantic 模型（参数）
2. 写函数（实现）
3. 注册到工具列表
```

### 5.3 当前工具定义标准（已符合）

```python
class LaunchAppInput(BaseModel):
    """launch_app 工具的输入参数"""
    app_name: str = Field(
        description="应用名称（如 notepad、chrome、explorer）"
    )
    args: Optional[str] = Field(
        default=None,
        description="启动参数（可选）"
    )
```
