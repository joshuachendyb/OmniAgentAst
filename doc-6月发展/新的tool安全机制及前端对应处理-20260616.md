## 1 背景

5级安全枚举 `ToolSafetyLevel`（`read_only/safe/destructive/dangerous_sandbox/dangerous`）已废弃，替换为**二进制bool + check_fn** 的确认判定机制。本文档说明新机制的判定规则、配置方法、前后端数据流及前端显示处理。

## 2 后端双保险确认判定

### 2.1 判定流程图

```
┌──────────────────────────────────────────────┐
│              传入参数 params                   │
│         ┌───────────┴───────────┐             │
│         ▼                       ▼             │
│  action_confirmation     needs_confirmation    │
│  有该action的配置？      工具级的默认配置        │
│         │                       │              │
│   ┌─────┴──────┐         ┌─────┴──────┐        │
│   │  True/False │         │ True/False  │       │
│   └─────┬──────┘         └─────┬──────┘        │
│         │                       │              │
│         └──────────┬────────────┘              │
│                    ▼                            │
│         needs_confirm = 最终判定结果            │
│                    │                            │
│         ┌──────────┴──────────┐                 │
│         ▼                     ▼                 │
│     True: 需要确认        False: 直接执行       │
│     safety_level=          safety_level=        │
│     "destructive"          "safe"               │
└──────────────────────────────────────────────┘
```

### 2.2 双保险规则

| 层级 | 属性 | 说明 | 示例 |
|------|------|------|------|
| **工具级默认** | `needs_confirmation: bool` | 该工具所有调用是否需确认 | `register(needs_confirmation=True)` |
| **action级覆盖** | `action_confirmation: Dict[str, bool]` | 特定action的确认策略，优先于工具级 | `register(action_confirmation={"write": True})` |

**优先级**：

> action级（指定了该action）→ 工具级默认（未指定action或action无配置时fallback）

### 2.3 为什么需要两个级别（设计意图）

**核心原因**：同一个工具的不同调用，危险性不一样。用一个开关管所有，要么过度拦截、要么拦截不足。

**反面例子**：假如只有一个工具级 `needs_confirmation`

```
file_operation(needs_confirmation=True)
→ 读文件也要弹窗确认 → 用户烦死

file_operation(needs_confirmation=False)
→ delete文件不弹窗 → 不安全
```

**不管设 True 还是 False，总有一边不对。** 所以需要两个级别配合：

| 级别 | 粒度 | 含义 |
|------|------|------|
| **工具级默认** `needs_confirmation` | 粗（全工具） | 该工具大部分调用的**默认**确认策略 |
| **action级覆盖** `action_confirmation` | 细（单个action） | 只在 `params` 有 `action` 字段时才介入，覆盖指定action的默认策略 |

**协作关系**：

> 工具级定基调（大部分调用用这个），action级修特例（少数危险动作单独加强）。

实际效果：

```python
file_operation(
    needs_confirmation=False,                # 基调：大部分操作安全
    action_confirmation={"delete": True}     # 特例：只有delete危险
)

# 调用效果：
# {action: "read",   path: "a.txt"} → 不确认（工具级 False）
# {action: "list",   path: "/"}     → 不确认（工具级 False）
# {action: "delete", path: "a.txt"} → 弹确认（action级 True 覆盖）
# {path: "a.txt"}（无action）        → 不确认（工具级 False，action级不触发）
```

### 2.4 判定示例

#### 示例1：shell默认需确认，action=read不需

```
execute_shell_command:
  needs_confirmation=True
  action_confirmation={"write": True}

→ action=write 时 needs_confirm=True  （action级指定）
→ action=read  时 needs_confirm=False （action级指定为True→实际False）
→ 无action参数  时 needs_confirm=True  （fallback工具级）
```

#### 示例2：file默认不需确认，action=delete需确认

```
file_operation:
  needs_confirmation=False
  action_confirmation={"delete": True}

→ action=delete 时 needs_confirm=True  （action级指定）
→ action=move   时 needs_confirm=False （action级未配置→fallback工具级）
→ 无action参数  时 needs_confirm=False （fallback工具级）
```

## 3 check_fn — 动态参数检查

### 3.1 什么时候需要check_fn

`needs_confirmation` 和 `action_confirmation` 都是**静态规则**（注册时就定死了），它们只回答 **"这个工具/这个action要不要确认"**。

但有时候是否安全取决于**具体的参数值**，注册时根本不知道：

```
同一个 file_operation action=write：
  params: {file_path: "/tmp/test.txt", content: "hello"}
    → 写临时文件，安全

  同样的 action=write：
  params: {file_path: "C:/Windows/system.ini", content: "malicious"}
    → 写系统文件，危险
```

上面的例子，**静态规则无法区分**，必须运行时**看参数内容**来判断。这就是 `check_fn` 的作用：

| | needs_confirmation | action_confirmation | check_fn |
|--|-------------------|-------------------|----------|
| **类型** | 静态 bool | 静态 Dict | **动态函数** |
| **判断依据** | 只看工具名 | 看 action 字段 | **看 params 具体内容** |
| **注册时就知道** | ✅ 是 | ✅ 是 | ❌ 不知道，运行时才知道 |
| **典型用途** | 这个工具整体危不危险 | 这个action危不危险 | **这个参数值危不危险** |

### 3.2 一个简单例子

```python
def my_check_fn(params: Dict) -> Dict[str, Any]:
    """检查参数内容是否危险"""
    path = params.get("file_path", "")
    # 动态判断：写系统目录就阻止
    if path.startswith("C:/Windows"):
        return {"is_safe": False, "blocked": True, "message": "禁止写入系统目录"}
    # 没发现问题 → 让后面的流程继续判定
    return {"is_safe": True}
```

**执行流程**：

```
check_fn(params)  ← 先执行check_fn，看参数内容是否危险
  │
  ├─ 返回 blocked=True → 立即阻止，标记 dangerous
  │
  └─ 返回 is_safe=True → 继续走静态规则判定
       │
       ├─ needs_confirm=True  → 弹确认窗（destructive）
       └─ needs_confirm=False → 直接执行（safe）
```

### 3.3 三个安全判定机制的关系图

```
┌─────────────────────────────────────────────────────────┐
│                    check_before_execute()                │
├─────────────────────────────────────────────────────────┤
│  第一步：_check_known_risks()  ← 内置的固定风险检测       │
│    ├─ 路径越权检查（file工具                              │
│    ├─ 写入污染检查                                       │
│    └─ 代码注入检查                                       │
│      └─ 命中 → 直接阻断, dangerous                       │
├─────────────────────────────────────────────────────────┤
│  第二步：check_fn(params)  ← 注册时传入的自定义检查       │
│    └─ 动态检查参数内容                                    │
│      └─ 返回 blocked → 阻断, dangerous                   │
├─────────────────────────────────────────────────────────┤
│  第三步：_get_needs_confirmation()  ← 静态规则判定       │
│    ├─ action级覆盖（有action且配置了）                    │
│    └─ 工具级默认（无action或未配置）                      │
│      └─ needs_confirm=True  → destructive 弹确认窗       │
│      └─ needs_confirm=False → safe 直接执行              │
└─────────────────────────────────────────────────────────┘
```

**三步的关系**：

| 步骤 | 叫什么 | 作用 | 谁写的 |
|------|--------|------|--------|
| 第一步 | 已知风险检测 | 系统内置的通用安全检查（路径、写入、注入） | 框架自带 |
| 第二步 | check_fn | **查看具体参数值**判断是否危险 | **开发者自定义** |
| 第三步 | 静态规则 | 根据工具名/action决定要不要弹确认窗 | 注册时配置 |

### 3.4 签名与返回值

```python
def my_check_fn(params: Dict) -> Dict[str, Any]:
```

| 返回值 | 含义 | 后续行为 |
|--------|------|---------|
| `{"is_safe": False, "blocked": True, "message": "..."}` | 判定为危险，阻止执行 | 标记 dangerous，前端显示阻断 |
| `{"is_safe": True}` | 没发现问题 | 继续走第三步静态规则 |
| 返回格式不对（缺 is_safe） | 异常 | 忽略，继续走静态规则 |
| 抛出异常 | 代码出错 | 按危险处理，阻止执行 |

### 3.5 注册示例

```python
tool_registry.register(
    name="my_tool",
    ...
    check_fn=my_check_fn,  # 传入自定义检查函数
)

## 4 后端返回格式

### 4.1 check_before_execute() 返回值

```python
{
    "is_safe": bool,              # True=可安全执行, False=不安全
    "requires_confirmation": bool, # True=需用户确认
    "blocked": bool,               # True=已阻止执行
    "message": str,                # 提示信息
    "safety_level": str,           # 前端显示用: "safe"/"destructive"/"dangerous"
}
```

### 4.2 safety_level三值含义

| 值 | 含义 | 触发条件 |
|-----|------|---------|
| `"safe"` | 安全，可执行 | `needs_confirm=False`，无风险命中 |
| `"destructive"` | 破坏性，需确认 | `needs_confirm=True`，未阻断 |
| `"dangerous"` | 危险，已阻断 | 工具未注册 / 已知风险命中 / check_fn阻断 |

## 5 前后端safetyLevel数据流

```
backend: check_before_execute → safety_level (string)
    │
    ▼ SSE stream (onAuthorizationRequired)
action_handler: MetaStep.data.safety_level
    │
    ▼ CustomEvent("authorization_required")
NewChatContainer: rawData.safety_level → setAuthorizationPending({safetyLevel})
    │
    ▼ prop
AuthorizationModal: request.safetyLevel → SAFETY_LEVEL_CONFIG 映射 → 显示
```

const safetyConfig = SAFETY_LEVEL_CONFIG[request.safetyLevel] || {

  toolName: string;                 // 工具名称

## 8 安全性分析

| 场景 | 前端处理 | 安全性 |
|------|---------|--------|
| safetyLevel缺失 | 显示"default"灰色标签 | 弹窗仍正常显示和操作 |
| safetyLevel未知值 | 以原值作为标签文字 | 用户仍能看到实际级别 |
| 后端safety_level变更 | 无需修改前端代码 | SAFETY_LEVEL_CONFIG自动映射 |
| 恶意篡改safety_level | 仅影响显示颜色/标签 | 确认流程由后端控制，前端只做展示 |
