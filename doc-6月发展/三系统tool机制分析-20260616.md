## 3 注册机制详细对比

### 3.1 OmniAgentAs-desk 注册机制

**实际代码路径**: `backend/app/services/tools/registry.py`

**注册方式**：纯函数式 `registry.register()` + 废弃的 `@register_tool()` 装饰器

**`register()` 方法签名**（共 15 个参数）：

```python
def register(
    self,
    name: str,                                              # 必填
    description: str,                                       # 必填
    category: ToolCategory,                                 # 必填
    implementation: Callable,                               # 必填
    version: str = "1.0.0",                                 # 可选
    dependencies: Optional[List[str]] = None,               # 可选
    input_model: Optional[Type[BaseModel]] = None,          # 可选
    input_schema: Optional[Dict] = None,                    # 可选
    output_schema: Optional[Dict] = None,                   # 可选
    examples: Optional[List[Dict]] = None,                  # 可选
    expose_to_llm: bool = True,                             # 可选
    next_actions: Optional[Dict[str, Any]] = None,          # 可选
    failure_hint_fn: Optional[Callable] = None,             # 可选
    needs_confirmation: bool = False,                       # 可选（二元默认，替代5级枚举）
    action_confirmation: Optional[Dict[str, bool]] = None,  # 可选（action级覆盖）
    check_fn: Optional[Callable] = None,                    # 可选（动态参数检查）
) -> Dict[str, Any]:
```

**关键修正**: 15个参数（4必填+11可选）。

**`register()` 内部实现**（SRP拆分）：

```python
# register() 拆分为两个私有方法（遵守SRP）
register() → _update_existing_tool()  # 已存在时更新
           → _register_new_tool()     # 新工具注册
```

**遗留的 `@register_tool()` 装饰器**（第 244-290 行）：仍然定义但**不再使用**，最后一批工具在 2026-05-02 重构期间已迁移至显式调用。

### 3.3 OpenCode 注册机制

**实际代码路径**: `packages/opencode/src/tool/`（TypeScript 版本，anomalyco/opencode）

**注册方式**：`tool()` 工厂函数 + Plugin Hooks

```typescript
// 实际 API（@opencode-ai/plugin/dist/tool.d.ts）
export function tool<Args extends z.ZodRawShape>(input: {
    description: string;
    args: Args;
    execute(args: z.infer<z.ZodObject<Args>>, context: ToolContext): Promise<string>;
})
```

**关键特性**:
- `tool()` 是简单的恒等函数（`return input`）
- **没有** `name` 字段 — 工具名由文件名决定
- **没有** `hides` 或 `permit` 字段 — 权限通过 `opencode.json` 的 `permission` 配置

**Plugin Hooks**（已确认存在）：

| Hook | 类型 | 功能 |
|------|------|------|
| `tool` | 属性映射 | `{ [key: string]: ToolDefinition }` |
| `tool.execute.before` | 函数 | 执行前拦截 |
| `tool.execute.after` | 函数 | 执行后处理（title/output/metadata） |
| `tool.definition` | 函数 | 修改工具定义（description/parameters） |

**内置工具数量**: **13个**（bash, edit, write, read, grep, glob, lsp, apply_patch, skill, todowrite, webfetch, websearch, question）

## 5 安全体系对比

### 5.1 OmniAgentAs-desk 安全体系（二元安全 + check_fn）

**历史**：原使用5级安全枚举 `ToolSafetyLevel`（READ_ONLY/SAFE/DESTRUCTIVE/DANGEROUS_SANDBOX/DANGEROUS），于 2026-06-16 重构为二元安全机制（见 `doc-6月发展/5级安全系统清理设计方案-20260616.md`）。

**当前安全机制**（定义于 `tool_types.py:87-89`）：

| 属性 | 类型 | 说明 | 优先级 |
|------|------|------|--------|
| `needs_confirmation` | `bool` | 工具级默认确认策略 | 低（全工具基调） |
| `action_confirmation` | `Dict[str, bool]` | action级覆盖确认策略 | 高（指定action时优先） |
| `check_fn` | `Callable` | 动态参数安全检查函数 | 最高（运行时间检查参数内容） |

**判定流程**（`safety/tool_safety_checker.py`）：

```
check_before_execute()
  ├─ 第一步：_check_known_risks()     ← 内置固定检测（路径越权/写入污染/代码注入）
  ├─ 第二步：check_fn(params)         ← 自定义动态检查（看参数具体内容）
  └─ 第三步：_get_needs_confirmation() ← 静态规则判定
       ├─ action级覆盖（有action且配置了）
       └─ 工具级默认（无action或未配置）
```

**返回格式**（前端显示用）：

```python
{
    "is_safe": bool,               # True=可安全执行
    "requires_confirmation": bool,  # True=需HITL确认
    "blocked": bool,                # True=已阻止执行
    "message": str,                 # 提示信息
    "safety_level": str,            # "safe"/"destructive"/"dangerous"
}
```

**前/后端safety_level映射**（`frontend/src/components/AuthorizationModal/index.tsx`）：

| 后端值 | 前端Tag颜色 | 标签文字 | 说明 |
|--------|-----------|---------|------|
| `"safe"` | blue | 安全 | 直接执行，不弹窗 |
| `"destructive"` | orange | 破坏性 | 需HITL确认 |
| `"dangerous"` | red | 系统危险 | 已阻断 |

### 5.2 Hermes 安全体系

无内置安全等级。通过 `check_fn` 参数自定义检查函数，每个工具单独实现安全逻辑。

### 5.3 OpenCode 安全体系

通过 `opencode.json` 的 `permission` 配置块控制工具权限。Plugin Hooks 中有 `permission.ask` 钩子。无显式安全等级系统。

## 7 工具加载时机对比

### 7.1 OmniAgentAs-desk 懒加载

**流程**：
```
请求到达 → Agent.__init__() → _init_tools_and_executor() → ensure_tools_registered()
                                                                    ↓
                                                         懒加载器按需导入7个模块
                                                                    ↓
                                                         每个模块执行注册函数
                                                                    ↓
                                                         标记 _registered_categories
```

**关键特征**：
- 首次触发时全量加载（非按需加载具体工具）
- 使用 `_registered_categories: set` 防止重复注册
- 启动时不加载工具，仅加载 registry 本身

### 7.2 OpenCode 按需加载

**流程**：
```
CLI 启动 → 初始化 Effect Runtime
              ↓
   用户输入 → intent 分析
              ↓
   确定需要的工具 → 加载对应 tool module
              ↓
   构建 ToolDefinition 列表
              ↓
   发送给 LLM → 执行工具
```

**关键特征**：
- 真正的按需加载（只加载用户需要的工具）
- 插件工具在插件启用时注册

### 10.2 注册机制深度对比

#### 10.2.1 OmniAgentAs-desk — 显式调用 + Pydantic自动Schema

**模式**: 每个分类的 `_register.py` 调用 `tool_registry.register()`

```python
# file_register.py（重构前示例，当前已改用 binary bool 参数）
tool_registry.register(
    name="read_text_file",
    description="读取文本文件内容",
    category=ToolCategory.FILE,
    implementation=FileTools().read_text_file,
    input_model=ReadTextFileInput,   # ← Pydantic V2 BaseModel
)
```

**关键特性**:

- **SRP满足**: register() 拆分为 `_update_existing_tool()` + `_register_new_tool()`，单一职责
- **Pydantic自动Schema**: `input_model → model_json_schema() → _fix_schema_types()` 自动修复Union/Optional
- **惰性加载**: `lazy_loader.py` 按分类逐个加载，失败不影响其他分类
- **装饰器语法**: `@register_tool(...)` 适配器模式，对装饰器友好

#### 10.2.2 Hermes — 模块级自注册 + AST发现

**模式**: 每个工具文件在模块级调用 `registry.register()`

```python
# file_tools.py — 模块级(导入时执行)
registry.register(name="read_file", toolset="file",
    schema=READ_FILE_SCHEMA,        # ← 手工JSON字典
    handler=_handle_read_file,
    check_fn=_check_file_reqs)
```

```python
# registry.py — 自动发现
def discover_builtin_tools():
    for path in tools_dir.glob("*.py"):
        if _module_registers_tools(path):  # AST解析检查
            importlib.import_module(f"tools.{path.stem}")
```

**关键特性**:

- **模块级副作用**: 导入即执行 `registry.register()`，有副作用
- **AST发现**: 不用显式列表，用 `ast.parse` 检测文件是否含 `registry.register()` 调用
- **多个参数**: `register()` 有 ~13 个参数(含 emoji, check_fn, dynamic_schema_overrides 等)
- **手工Schema**: 每个工具的手写 JSON Schema 字典

#### 10.2.3 OpenCode — tool()工厂 + Plugin Hooks

**模式**: 插件通过 `tool()` 函数声明工具，通过 Hooks 钩子扩展

```typescript
// 实际 API（@opencode-ai/plugin/dist/tool.d.ts）
tool({
    description: "Execute shell command",
    args: { command: z.string() },
    execute: async (args, context) => {
        return result;
    },
})
```

```typescript
// Plugin Hooks（@opencode-ai/plugin/dist/index.d.ts）
export interface Hooks {
    tool?: { [key: string]: ToolDefinition }
    "tool.execute.before"?: (input, output) => Promise<void>
    "tool.execute.after"?: (input, output) => Promise<void>
    "tool.definition"?: (input, output) => Promise<void>
}
```

**关键特性**:

- **恒等函数**: `tool()` 只是 `return input`，不做任何处理
- **文件名即工具名**: 无 name 字段，由文件名决定
- **Plugin生命周期**: 注册、执行前、执行后、定义修改四个钩子
- **Vercel AI SDK**: 底层使用 AI SDK 处理 LLM 调用和工具调度

### 10.3 加载时机与生命周期对比

| 方面 | OmniAgentAs-desk | Hermes | OpenCode |
|------|-----------------|--------|----------|
| **首次加载时机** | 首次Agent创建 | 启动时(discovers调用) | Boot时(插件注册) |
| **工具选择** | 分类懒加载 + BM25动态注入 | 工具集过滤(toolset) | Agent权限规则 |
| **运行时新增** | ✅ BM25检索自动注入新分类 | ✅ tool_search桥梁 | ❌ 需重启/重加载 |
| **缓存策略** | 300s + 60s | 30s(check_fn) | 无(实时生成) |
| **失败处理** | 分类失败不影响其他，重试 | check_fn控制可用性 | 插件异常隔离 |

**核心差异**:

- **OmniAgent** 是唯一支持运行时动态发现并注入新工具的系统（BM25）
- **Hermes** 是唯一使用 AST 扫描自动发现的系统
- **OpenCode** 是唯一具有完整插件生命周期钩子的系统

### 10.4 Schema 定义方式对比

### 10.5 工具调度与执行对比

#### 10.5.1 完整调度链路

**OmniAgentAs-desk**:

**Hermes**:


**OpenCode**:
```

```

| **已知风险检测** | ✅ safety/模块检查 | ❌ 无 | ❌ 无 |
| **Action级覆盖** | ✅ action_confirmation + check_fn动态检查 | ❌ 无 | ❌ 无 |
| **重试机制** | ✅ 指数退避(默认3次) | ❌ 无 | ❌ 无 |

| 特性 | OmniAgentAs-desk | Hermes | OpenCode |
|------|-----------------|--------|----------|

### 10.6 10大编码规范符合度

| 规则 | OmniAgentAs-desk | Hermes | OpenCode |

| **SRP** | ✅ register()拆3个私方法 | ⚠️ register()参数过多 | ✅ 纯函数拆分 |

| **ISP** | ✅ 注册接口职责分离 | ⚠️ register参数过多 | ✅ tool()简洁 |
| **复用优先** | ✅ ToolHelper 14个复用模块 | ❌ | ✅ |
| **禁止backward** | ✅ 旧parsers标记deprecated | ✅ 无 | ✅ 重写新版 |

### 10.7 关键结论与建议

#### 10.7.1 核心优势对比

| 系统 | 最优势 | 最劣势 |
|------|-------|-------|
| **OmniAgentAs-desk** | 安全体系完整(二元安全+HITL) + 动态注入(BM25) + Pydantic自动Schema | 仅有OpenAI格式输出，无跨Provider适配 |
| **Hermes** | 工具数量最多(~68) + AST自动发现 + 多Provider适配 | 手写Schema维护成本高 + 安全机制弱 |
| **OpenCode** | 类型安全(Zod编译时) + Plugin生命周期 + Vercel AI SDK集成 | 工具数量少(13) + TypeScript学习曲线 |

#### 10.7.2 8项重大差距

| # | 差距项 | 领先系统 | 建议 |
|---|-------|---------|------|
| 1 | 跨Provider协议适配 | Hermes/OpenCode | OmniAgent应考虑多Provider支持 |
| 2 | AST自动发现 | Hermes | 可减少CATEGORY_MODULES手动维护 |
| 3 | Effect并发 | OpenCode | 可用asyncio.gather改进并发 |
| 4 | Schema清理 | Hermes | 可引入schema清理机制 |
| 5 | 插件生命周期钩子 | OpenCode | 执行前/后/定义修改 |
| 6 | 渐进式公开(工具搜索) | Both | OmniAgent的BM25已是最优方案 |
| 7 | 参数类型强制 | Hermes | coerce_tool_args 值得引入 |
| 8 | MCP动态发现 | Hermes/OpenCode | OmniAgent缺少MCP支持 |

#### 10.7.3 OmniAgentAs-desk 独有且应该保持的优势

- **二元安全+HITL**: 独此一家，绝不可删除
- **Pydantic自动Schema生成**: 零维护成本
- **BM25动态注入**: 运行时按需发现，智能高效
- **3层响应格式**: 同时满足前端展示和LLM上下文控制
- **ToolHelper 14个复用模块**: 高复用，符合DRY
- **llm_data字段**: 上下文精简，控制token消耗
