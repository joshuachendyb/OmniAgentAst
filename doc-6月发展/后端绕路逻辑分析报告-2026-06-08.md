### 2.2 核心问题

**问题1: AIServiceFactory是纯空壳类**

86行代码，11个方法全是单函数转发。类本身无任何逻辑，纯粹是独立函数的命名空间包装。更严重的是存在**双重状态**：类变量`_instance`/`_current_provider`与`get_service.py`模块级变量重复定义。

```python
# 现状: 调用要经过空壳类
service = AIServiceFactory.get_service(config_path)
  → AIServiceFactory.get_service()        # 空壳
    → _get_service(config_path)           # 实际逻辑

# 简化: 直接调独立函数
service = get_service(config_path)
```

**问题2: ChatStreamMixin是纯空壳Mixin**

```python
class ChatStreamMixin:
    async def chat_stream(self, message, history):
        async for chunk in self.chat_with_tools_stream(message, history):  # 只去掉tools参数!
            yield chunk
```

这个Mixin存在的唯一理由是提供`chat_stream`方法名，但它的实现就是调`chat_with_tools_stream`传`tools=None`。

**问题3: CapabilityDetector是死代码**

`BaseAIService._detect_reasoning_support()`硬编码返回False（llm_core.py:126），`CapabilityDetector`的完整探测逻辑从未被调用。所有推理模型走普通路径，reasoning_content支持探测完全失效。

**问题4: Mixin伪装线性继承**

```
MRO: BaseAIService → ChatStreamMixin → ChatWithToolsStreamMixin → ToolCallerMixin → object
```

三个Mixin之间强耦合：ChatStream依赖ChatWithToolsStream，ToolCaller依赖ChatWithToolsStream。这不是真正的Mixin组合，是**线性继承链用Mixin语法伪装**。

**问题5: 两套429重试实现**

- `_post_with_retry`：手动调`engine.execute(_do_post)`
- `_stream_with_retry`：用`engine.execute_async_context()`

两套重试逻辑并存，违反DRY。

### 2.3 简化建议

| 动作 | 说明 | 预期收益 |
|------|------|---------|
| 删除AIServiceFactory类 | 直接用`get_service()`等独立函数 | 消除11个空壳+双重状态 |
| 删除ChatStreamMixin | 在BaseAIService上直接定义chat_stream | 消除1个空壳Mixin |
| 删除build_messages空壳 | 直接调build_llm_messages | 消除1个空壳 |
| 接入CapabilityDetector | 替代硬编码False | 恢复reasoning探测 |
| 统一429重试 | 删除_post_with_retry | 消除重复重试逻辑 |
| Mixin改为直接方法 | 三个Mixin强耦合，直接写在BaseAIService | 消除伪装继承 |

## 三、★★★★★ SSE数据流（序列化地狱）

### 3.1 数据格式转换链（7次转换，3次冗余）

```
Step 1: ChunkStep对象        (Agent内部创建)
   ↓  ChunkStep.to_dict()
Step 2: dict                  (sse_formatter.py:65)
   ↓  json.dumps(base)
Step 3: SSE字符串             ("data: {...}\n\n")
   ↓  emit_only内部: json.loads(sse_data[6:])
Step 4: dict                  ★冗余! 刚序列化就反序列化
   ↓  run_sse_stream.py:53-56: json.loads(sse_data[6:])
Step 5: dict                  ★又做一次! 和emit_only内部重复
   ↓  current_execution_steps.append(step_dict)
Step 6: list[dict]
   ↓  save → update_message_fields → json.dumps(execution_steps)
Step 7: json字符串            ★每个chunk都全量序列化整个列表
   ↓  SQLite cursor.execute()
Step 8: SQLite BLOB
```

### 3.2 核心问题

**问题1: 序列化-反序列化死循环**

```
Step对象 → to_dict() → json.dumps() → SSE字符串 → json.loads() → dict → json.dumps() → SQLite
```

同一个数据被序列化3次、反序列化1次。`SSE字符串 → json.loads() → dict`做了**两次**（emit_only内部一次，run_sse_stream又一次）。

**问题2: 每个chunk全量DB写入**

每收到一个chunk，整个`execution_steps`列表被`json.dumps`一次写入SQLite。假设50步对话，第50步时序列化50个步骤的完整数据，而前49步数据完全没变却被重复序列化49次。

每个chunk触发**5次SQL操作**：
1. `ensure_session_exists` — SQL SELECT验证session
2. `allocate` 内 — 2次SQL SELECT
3. `update_message_fields` — SQL UPDATE (含json.dumps全量序列化)
4. `update_session_message_count` — SQL UPDATE

其中`ensure_session_exists`和`allocate`的结果在第一次后不会变，但每个chunk都重复查询。

**问题3: 断头导入BUG**

```python
# run_sse_stream.py:13
from app.services.react_sse_wrapper.emit_only import emit_only  # ← emit_only.py不存在!
```

函数`emit_only`实际定义在`emit_and_save.py`中。运行时会抛出`ModuleNotFoundError`。

**问题4: save散落5处4层**

| save位置 | 文件 | 层级 |
|---------|------|------|
| start时save | start_step.py:68 | chat_stream层 |
| 每个chunk时save | run_sse_stream.py:57 | services层 |
| 中断时save | chat_stream_v2.py:68 | API层 |
| 取消时save | task_cancel_check.py:32 | services层 |
| error时save | yield_error_sse.py:28 | services层 |

### 3.3 简化建议

| 动作 | 说明 |
|------|------|
| 消除序列化循环 | emit_only返回(step_dict, sse_data, current_content)三元组，run_sse_stream直接用step_dict |
| 增量DB写入 | 只在final/error步骤时写入DB，中间chunk只保存在内存 |
| 修复断头导入 | 改为`from app.services.react_sse_wrapper.emit_and_save import emit_only` |
| 统一save策略 | save只在start(创建消息)和final/error(更新内容)两个时机执行 |
| 消除Pydantic中转 | ExecutionStepsUpdate模型没做验证，直接传参数 |
| 缓存session查询 | ensure_session_exists和allocate结果缓存，不重复查询 |

## 四、★★★★ 工具执行链

### 4.1 三层委托链

```
BaseAgent._execute_tool()                          ← 第1层: 空壳，纯转发
  └─ execute_tool_with_unified_retry()             ← 第2层: finish短路+not-found+别名查找
       └─ ToolRetryEngine.execute_tool_with_retry() ← 第3层: 参数验证+重试循环+执行
            └─ _execute_tool_once()                ← 第4层: 实际执行(asyncio.wait_for)
```

### 4.2 核心问题

**问题1: 第2层和第3层重复工具查找**

第2层`execute_tool_with_unified_retry`已做not-found判断和别名查找（行51-64），确保`tools.get(action)`有值。但第3层`execute_tool_with_retry`又做了一次完整查找（行127-142）——在第2层调用路径下**永远不会触发**，是死代码。

**问题2: normalize_params静默删除参数**

```python
for key in list(params.keys()):
    if key not in valid_params:
        logger.warning(...)  # 只打warning
        del params[key]      # 静默删除！调用方无感知
```

LLM传了多余参数，只打warning就删掉。如果LLM传了关键参数但schema未声明，会导致静默失败。

**问题3: 全局单例线程不安全**

```python
_tool_retry_engine: Optional[ToolRetryEngine] = None
def _get_tool_retry_engine() -> ToolRetryEngine:
    global _tool_retry_engine
    if _tool_retry_engine is None:  # 无锁保护!
        _tool_retry_engine = ToolRetryEngine()
    return _tool_retry_engine
```

多线程同时调用可能创建多个实例。且`self.available_tools`全局共享，任何Agent动态加载工具都影响全局。

**问题4: 有RetryEngine却手动写while循环**

`ToolRetryEngine.execute_tool_with_retry`内部手动写`while engine.attempt_count <= max_retries`循环，而`RetryEngine`通用重试引擎已有`execute()`方法——违反DRY。

### 4.3 简化建议

| 动作 | 说明 |
|------|------|
| 删除BaseAgent._execute_tool空壳 | 直接调execute_tool_with_unified_retry |
| 合并第2层和第3层工具查找 | 第2层完成查找后直接传tool函数，第3层不再重复查找 |
| 参数验证改为报错 | 多余参数返回错误让LLM修正，而非偷偷删掉 |
| 使用RetryEngine.execute() | 替代手动while循环 |
| 全局单例改为实例变量 | 每个Agent持有自己的ToolRetryEngine实例 |
