## 一、旋智体需求分析与架构范围

### 1.0 旋智体说明

**旋智体**是本系统的核心设计概念：以task为**螺旋**核心（旋），LLM+Agent分层协作（智），经验沉淀与工具自动合成实现自我演进（体）。整个系统的需求分析和架构设计都围绕实现旋智体展开。

| 维度 | 含义 | 覆盖章节 |
|------|------|---------|
| **旋**（螺旋演进） | 每一圈task完成不是回到原点，而是上升一个台阶。task执行的正确性、稳定性是基础能力——决定了底座牢不牢 | §1.1-§1.5 |
| **智**（智能决策） | LLM是大脑，Agent是中枢。第2层做编排决策，第3层做执行决策，第4层做服务决策。分层协作、各司其职 | §1.2-§1.3 |
| **体**（体系化积累） | 每转一圈沉淀一圈：经验→模式识别→自动合成工具（ArtTool→FlowTool）→能力螺旋上升。螺旋越高系统越强——这是核心差异化 | §2.10 |

三个核心架构已锁定：

task-base — 以完成任务为目标，以task为螺旋核心演进
task-Agent薄转发器 — 只做ReAct循环调度，不做编排决策
AST注册 — 工具代码自注册，零人工维护
#### 1.0.1 task-base 旋智体开发总体需求

旋智体是方向，但开发必须分阶段递进，每阶段打好基础再进入下一阶段。

**三阶段开发路线图**：

| 阶段 | 名称 | 核心能力 | 工具模式 | 目标 |
|------|------|---------|---------|------|
| **Phase 1** | 基础task能力 | 单task-Agent + ~20内置tool | 仅内置tool（无ArtTool/FlowTool） | 单Agent通过内置tool稳定完成task，打好"轮"的基础 |
| **Phase 2** | 工具演进 | 引入ArtTool合成 | 内置tool + ArtTool | 即兴代码→经验库→自动合成原子工具，实现"体"的初步演进 |
| **Phase 3** | 流程抽象 | 引入FlowTool合成 | 内置tool + ArtTool + FlowTool | 编排模式→经验库→自动合成工作流，实现"体"的完整闭环 |

**核心技术决策**：

| 决策项 | 内容 | 依据 |
|-------|------|------|
| **task-Agent命名与定位** | 核心ReAct循环Agent命名为**task-Agent**，是**薄转发器**——只做接收task→ReAct循环→调度工具→输出结果的标准流程，不做编排决策（类似OpenCode CodeAgent / Hermes AIAgent） | 薄转发器原则保证职责单一、可测试、可替换 |
| **内置tool规模** | 约**20个**，精细粒度保证LLM准确选择 | 选项过多导致LLM选择错误率上升 |
| **tool注册方式** | **AST方法**——扫描工具代码自动提取定义（参数签名、返回值类型、功能描述），无需人工维护注册表 | 新增工具只写代码即可自动注册，零维护负担 |

**Phase 1（当前阶段）的核心目标**：单Agent + 内置tool稳定完成task，不引入ArtTool/FlowTool。这个阶段只做"轮"的基础，做到"不是烂货"。Phase 2和Phase 3在Phase 1稳定后再推进。

### １.1 需求分析

#### 1.1.1 交付模式（结果如何到达用户）

| 模式 | 用户等待方式 | 传输方式 |
|------|-------------|---------|
| **同步** | 在线等待 | SSE流式（内容逐段到达） |
| **异步** | 不等待，事后查结果 | 整包（用户轮询查询 或 服务端推送完成事件） |

注：同步模式下始终走SSE流式，内容自动逐段到达，用户不需要选择传输方式。异步模式下用户提交后可以离开，完成后通过查询接口或推送通知获取结果。

**同步和异步不是开机二选一的模式**。用户在同一个会话中可以在两种模式间灵活切换：先聊天交流（同步）确定需求，然后提交耗时任务到后台（异步），任务完成后回来继续讨论结果（同步）。两者并行不悖，互不干扰。

#### 1.1.2 用户场景 × 交付模式

| 场景 | 用户行为 | 通常采用的交付模式 |
|------|---------|------------------|
| 对话问答 | 发消息 → 等回复 → 追问 | 同步（也可用异步，后台执行后通知） |
| 任务执行 | 下任务 → 离开 → 回来查结果 | 异步（短任务也可用同步，在线等结果） |

用户发消息后 建立一个task_id，任务类型的可以凭这个id随时回来查进度。

#### 1.1.2a 用户操作流程说明（通俗版）

**同步模式——聊天式操作**（默认，最常见）

用户打开页面，看到的是一个聊天输入框。打字发消息，AI实时逐段回复，像微信聊天一样：

适用场景：日常问答、写代码、改文章、翻译等短平快的交互。
---
**异步模式——任务式操作**（用于耗时操作）--自动识别+主动操作 两种模式

页面上除了聊天输入框，还有一个"新建任务"的入口（按钮或菜单）。用户点开后：
```
第1步：填写任务描述（比如"分析项目代码生成架构图"）
第2步：点"提交任务"
第3步：页面显示"任务已提交，编号：T-20260531-001"
第4步：用户可以关掉页面去喝茶/开会/睡觉
第5步：回来后打开"任务列表"看到任务状态
     ├─ 进行中 → 显示进度（已分析10个文件...）
     ├─ 已完成 → 点开看结果
     └─ 已取消 → 显示取消原因
第6步（可选）：如果用户没关页面，任务完成时页面会弹通知
```

适用场景：分析整目录库、批量处理文件、导出大型报告等耗时1分钟以上的操作。

---

**同步vs异步，前端怎么区分？**

| | 同步（聊天） | 异步（任务） |
|------|-------------|-------------|
| **入口** | 聊天输入框 | "新建任务"按钮 /自动识别|
| **用户能不能走开** | 不能，在线等 | 能，走后回来查 |
| **能不能中断** | 能（暂停/停止两个按钮） | 能（取消任务） |
| **页面会不会关** | 不关 | 可以关 |
| **结果怎么拿** | 直接看到 | 打开任务列表查看 |

**特别注意**：同步和异步不是开机二选一的死板模式，用户在同一会话中可以灵活交替使用。典型场景：先聊天沟通（同步）确定需求细节，然后提交分析任务（异步），任务跑完了回来继续讨论结果（同步）。两者可以同时存在，互不干扰。

---
#### 1.1.2b 关于SSE（两种用途，用户不用管）

SSE（Server-Sent Events）在本架构中有两种不同用途：

- **同步模式 — 流式推送（逐块传输回复内容）**：AI回复的内容逐段显示出来（像打字效果一样），这是后台用SSE逐块推送的。**用户不需要做任何设置或选择**。
- **异步模式 — 事件推送（单次完成通知）**：异步任务完成后，服务端通过SSE推送一条完成事件，前端收到后向用户弹通知。异步模式下不传输回复内容，只发一个"任务完成了"的标志。

两种SSE用途完全不同：同步是长流式（多个事件块），异步是单次事件。**对终端用户**两种都透明——同步用户看到内容逐段出现，异步用户看到弹窗通知。**对前端开发者**两种都是需要对接的SSE事件协议。

有的系统让用户选传输方式——本架构不这么做。原因：
- 用户只管发消息、看回复，不需要知道底层用什么技术
- SSE流式由第1层自动处理，对用户完全透明
- 无论AI回复多长，用户看到的效果都是"内容逐段出现"
- 如果用户想等全部内容出来再看，不操作就行——流式推送结束后自然就是完整结果

#### 1.1.3 系统必须支持的能力

| 能力 | 说明 |
|------|------|
| 接收用户消息并生成回复 | 用户发消息，系统处理后返回内容（同步模式下内容逐段显示，如同打字效果；异步模式下通过整包查询获取） |
| 回复到达（同步模式：SSE流式逐段到达；异步模式：整包查询） | 同步模式下内容通过SSE逐段到达，如同打字效果；异步模式下通过查询接口一次性获取完整结果 |
| **输出格式无关** | AI回复内容以原始文本流形态推送，一段回复中可能含纯文本、Markdown标记（代码块/表格/列表/标题）、结构化数据等混合内容。系统不假设、不限制内容格式。第1层透传原始内容不做格式转换，前端收到后自行判断如何渲染 |
| 使用外部资源 | 处理过程中可读文件、执行命令、查询网络等 |
| **可复用的链式执行机制** | 单个任务可能需要多次"调LLM→解析响应→执行工具→回传结果→再调LLM"循环，直到任务结束。这个执行机制是独立于同步/异步模式的**内核**——同步模式下它向第1层逐块推送输出，异步模式下它向任务记录追加执行步骤。不论前端什么模式，执行内核的行为是一致的 |
| 上下文保持 | 同一会话中，前面的对话影响后面的回复 |
| 用户可中断（暂停 + 停止） | 支持两种中断方式：①**暂停**——暂时中止回复生成，已出内容保留，用户之后可点击"继续"从停下的位置恢复生成（仅同步模式）；②**停止**——终止回复生成或任务，已出内容保留，不可恢复。同步模式下暂停/停止后会话不销毁，用户可以继续发送新消息。异步任务仅支持停止，中断后状态标记为已取消，已执行步骤记录保留 |
| 失败自动重试 | 通信层（第4层）：网络闪断、服务超时等临时故障自动重试；执行层（第3层）：AI返回空响应、格式错误时自动重试；编排层（第2层）：任务整体失败时重试整个任务。三层重试配合，不丢数据 |
| 多会话并发 | 多个对话/任务同时运行，互不干扰 |
| 模型可切换 | 支持用户手动切——用户可在界面上主动切换当前会话使用的模型。切换范围。第4层屏蔽各模型接口差异，上层无需感知切换 |
| 敏感操作确认 | 执行潜在危险操作前先征得用户同意。同步模式下用户在线可选择同意/拒绝；异步模式下任务暂停，标记为"待用户确认"，用户回来后处理 |
| **多个内部模块共用AI调用** | 对话等都需要调用AI，调用入口统一 |
| **逐段输出过程中可插入工具调用** | AI在逐段输出中可随时下发工具调用指令。第3层（执行引擎层）收到指令后调度工具执行，执行结果回传LLM继续生成后续文本。多个工具调用按LLM返回的顺序串行执行 |
| **不同模型能力兼容** | 不同AI模型的工具调用能力、输出格式支持程度不同，系统需统一处理差异 |
| **任务追踪** | 每次操作的全生命周期（创建→执行→完成/失败/回滚）可追溯。编排层管理任务级状态，执行引擎层管理步骤级状态 |
| **异步任务列表管理** | 用户可查看所有异步任务列表，按状态（进行中/已完成/已取消/待确认）筛选，获取每个任务的基本信息（task_id、名称、状态、创建时间、完成时间） |
| **可信可靠可预期（异步模式核心保障）** | 异步模式下用户不需要人工盯守。三条保障：①**可信**——每一步操作有日志可追溯，不会"悄悄失败"，用户随时可查任务执行细节；②**可靠**——通过自动重试+工具幂等性设计+状态持久化，确保任务不会因网络抖动或临时故障丢失或重复；③**可预期**——进度透明，用户知道任务进行到哪一步、处于什么状态、需不需要自己介入 |
| **旋智体演进** | 系统以**螺旋**模式运行：每一圈task执行完成后沉淀经验工具（ArtTool/FlowTool），下一圈task在前一圈的基础上执行，能力螺旋上升、越积越厚。**task执行是基础功能**（做不好就是烂货），**演进是核心差异化**——完成后不是结束，而是提炼经验、合成工具、持续改善的起点 |

#### 输出内容在各层的处理方式

内容从LLM生成到前端显示，经过各层的处理方式如下：

| 层 | 对内容做的事 | 不做的事 |
|---|-------------|---------|
| 第4层 LLM服务层 | 原始透传——LLM返回的内容逐块上传 | 不修改内容，不格式转换 |
| 第3层 执行引擎层 | 从LLM原始响应中提取用户可见的文本（剥离function call元数据、reasoning token等非内容部分），生成clean text chunks向上传 | 不转换格式（如MD→纯文本）、不增删改内容实质 |
| 第2层 编排层 | 完全透传，不识别内容 | 不做任何格式判断或转换 |
| 第1层 接入层 | 将文本块原样放入SSE事件的content字段推送 | 不做格式转换、截断、过滤 |
| 前端 | 收到原始内容后，自行判断内容格式并选择渲染方式 | — |

> 核心原则：各层只做自己职责内的事。唯一算得上"规范化"的操作在第3层——剥离LLM响应中非内容的元数据。除此之外，内容格式原封不动向上传递，不做任何格式假设或转换。系统对AI输出内容的格式不做保证——纯文本、Markdown标记（代码块/表格/列表/标题）、结构化数据等都可以出现，均由前端自行处理。

####  1.1.4 架构必须提供的底层设施

| 设施 | 说明 |
|------|------|
| 多模型接入 | 接入不同AI模型提供商，调用方式统一 |
| tool扩展 | 新增读写文件、执行命令、网络查询等能力时，不改核心处理流程 |
| 会话隔离 | 不同会话的执行上下文各自独立 |
| 可观测性 | 每次处理、每次外部调用都有日志可追溯 |
| 配置管理 | 模型参数、超时时间、重试策略等可配置 |

###   1.2 架构原则（设计约束）

| 原则 | 说明 | 违反后果 |
|------|------|---------|
| **SRP** | 每层只做一件事（详见章节一的分层定义） | 职责混杂，改A影响B |
| **严格调用** | 接入层→编排层→执行引擎层→LLM服务层，不可跳过或越层（详见章节一） | 逻辑不可控，重试/状态管理遗漏 |
| **接口契约** | 层间通过定义接口通信，不直接依赖具体实现 | 层耦合，换实现牵一发动全身 |
| **编排与执行分离** | 业务决策（做什么模式、什么策略）和执行机制（怎么做循环、怎么调工具）是两类不同职责，必须分属不同层次。编排层不控制循环细节，执行引擎层不做业务决策 | 第2层膨胀为混沌层，改编排影响执行逻辑 |
| **不向后兼容** | 新设计只对需求负责，现有代码后续按新设计重构 | 被旧代码绑架，设计变形 |

### 1.3 设计决策

以下设计决策影响第2层（编排层）、第3层（执行引擎层）、第4层（LLM服务层）的具体形态。分两类：已确定的和待定的。

#### 1.3.1 已确定的设计决策

| 决策项 | 最终选择 | 理由 |
|-------|---------|------|
| **第2层与传输协议的关系** | 第2层输出**事件对象**，不关心传输格式。第3层产生输出块，第2层完全透传，第1层负责序列化为SSE事件 | 四层架构决定了职责边界：第2层只做业务决策，不接触传输格式 |
| **整包（非流式）路径** | 同步始终流式无整包选项；异步整包查询走标准调用链（第1层→第2层→第3层），不设跳层快捷路径 | 同步/异步模式决策已在1.1.1中确定，且无实际需求支持跳层路径。严格调用原则适用全部场景，没有例外 |
| **LLM输出内容格式与前端渲染** | LLM输出内容以原始文本流透传，不做任何格式转换。第3层只剥离非内容元数据（function call定义、reasoning token等），第2层和第1层完全不碰内容。纯文本、Markdown、代码片段等混合格式均由前端收到后自行判断渲染 | 各层只做自己职责内的事。格式转换是前端的职责，不在本架构范围内。详见1.1.3"输出格式无关"和"输出内容在各层的处理方式" |
| **重试职责分配** | 分层重试：第2层=任务级重试，第3层=步骤级重试（空响应/格式错误），第4层=通信级重试（超时/限流/断连） | 已在1.1.3需求中明确为分层协作机制，无需再讨论 |
| **工具基础设施归属** | 作为**第3层内部组件**，只有执行引擎层可访问。经典Agent模式，工具是执行引擎的核心组成部分 | 当前无需求支持多层直接调用工具 |
| **暂停状态存储** | **持久化到数据库**，恢复时从DB载入 | 适应短时间暂停和长时间暂停（如用户离开几小时后再回来恢复）等多样场景 |
| **持久化经验学习** | 第3层执行引擎新增**经验库**内部模块：执行经验（任务类型/执行步骤/成功失败/耗时/错误模式/工具效能/用户偏好）结构化持久化存储到DB；新任务启动前检索相似历史经验注入Prompt指导执行方向；执行完成后提取经验特征更新经验库 | 必须实现的核心功能模块，已确认优先级为"必须"，概要设计2.10已完成设计要点（存储结构/检索机制/更新策略） |
| **工具模式选择** | 第3层工具体系分为三层：**内置tool**（系统预注册，始终可用）+ **ArtTool**（即兴代码经经验库自动合成）+ **FlowTool**（编排模式经经验库自动合成）。三层是递进关系而非替代关系，共同构成第3层的完整工具能力。外部MCP/Skill协议作为兼容生态备选通道，不做主力 | ArtTool即兴代码执行补充了内置tool无法覆盖的灵活场景；FlowTool编排抽象补充了内置tool+ArtTool手动组合的重复劳动。新增工具优先从即兴代码中自动沉淀，无需人工预注册 |
| **task-Agent命名与定位** | 核心ReAct循环Agent命名为**task-Agent**。它是一个**薄转发器**——只做"接收task→ReAct循环→调度工具→输出结果"的标准流程，不做复杂的编排决策。类似OpenCode的CodeAgent（只做tool+LLM的调度循环）和Hermes的AIAgent（conversation_loop中的薄调度层） | 薄转发器原则保证了task-Agent职责单一、可测试、可替换。编排决策属于第2层，执行循环属于第3层，task-Agent是第3层ReAct循环的具体载体 |
| **tool规模与注册方式** | 内置tool总量控制在**20个左右**，不宜多。注册方式采用**AST方法**——通过AST扫描工具代码自动提取工具定义（参数签名、返回值类型、功能描述），无需人工维护注册表 | 20个左右的精细粒度保证LLM能准确选择内置tool，不至于选项过多导致选择错误。AST注册避免了手工注册表的维护负担，新增工具只需写代码即可自动注册 |
| **实施阶段划分** | **第一阶段**（基础能力）：单task-Agent + 约20个内置tool，专注完成task。task-Agent是薄转发器（类似OpenCode CodeAgent或Hermes AIAgent），只做ReAct循环调度，不做复杂编排。**第二阶段**（工具进化）：在task-Agent基础上引入ArtTool，即兴代码→经验库→自动合成原子工具。**第三阶段**（流程抽象）：在ArtTool基础上引入FlowTool，编排模式→经验库→自动合成工作流 | 三阶段逐级递进：先做好基础（单agent+内置tool），再用ArtTool实现工具演进，最后用FlowTool实现业务流抽象。每阶段必须稳定后再推进下一阶段 |

### 1.5 四层架构各层职责定义

以下定义四层架构中每一层**必须承担的职责**和**禁止承担的职责**，是架构设计的硬约束。

---

#### 第1层：接入层

| 维度 | 说明 |
|------|------|
| **定位** | 用户与系统的唯一交互界面，所有请求的起点。极薄，只做协议适配 |

**对前端的接口（前端调用第1层）**：

| 接口 | 接口类型 | 请求参数 | 返回值 | 说明 |
|------|---------|---------|--------|------|
| 发送消息 | 同步请求 | 用户消息文本 | AI回复内容（SSE流式） | 用户发消息后在线等结果。回复内容通过SSE通道自动逐块到达，用户无需任何设置 |
| 发送任务 | 异步请求 | 任务描述文本 | task_id | 用户发任务后可离开 |
| 查询任务进度 | 查询接口 | task_id | 任务状态+已输出内容 | 用户查看任务执行状态 |
| 获取任务结果 | 查询接口 | task_id | 最终结果 | 任务完成后查看结果 |
| 获取任务列表 | 查询接口 | 筛选条件（状态、时间范围等） | 任务列表（含task_id/名称/状态/创建时间/完成时间） | 用户查看所有异步任务的概览列表，可按状态筛选 |
| 取消任务 | 中断接口 | task_id | 中断确认 | 用户点击取消按钮 |
| 暂停回复 | 中断接口 | 会话ID | 暂停确认 | 用户点击暂停按钮，暂时中止AI回复生成，已出内容保留（仅同步模式） |
| 继续回复 | 中断接口 | 会话ID | 继续确认 | 用户点击继续按钮，从暂停位置恢复AI回复生成（仅同步模式） |
| 接收异步完成通知 | SSE推送 | 会话ID | 完成事件 | 异步任务完成后，服务端通过SSE推送完成通知 |

> 说明：SSE（Server-Sent Events）是第1层对外（前端）的技术接口。对**终端用户**透明（用户看到的是内容逐段出现，不需要知道SSE），但对**前端开发者**是需要对接的技术协议。前端建立一次SSE连接，后续所有同步回复和异步完成通知都通过此连接推送。事件类型包括：text（文本块）、tool（工具调用）、final（完成标志）。
> 
> **注意**：text事件中的content字段为原始文本流，纯文本、Markdown、JSON、代码片段等格式均可出现，取决于LLM本次输出的形态。第1层不做任何格式转换或假设，前端需自行判断内容格式并选择合适的渲染方式。

**对下的接口（第1层调用第2层）**：

| 接口 | 调用方向 | 请求参数 | 返回值 | 说明 |
|------|---------|---------|--------|------|
| 处理同步请求 | 第1层→第2层 | 用户消息 | AI回复（通过回调接收流式块） | 转发同步请求给编排层，编排层通过回调逐块返回AI回复内容，由第1层通过SSE推送给前端 |
| 处理异步请求 | 第1层→第2层 | 任务描述 | task_id | 转发异步请求给编排层，编排层创建任务记录后返回task_id，后台执行 |
| 转发中断信号 | 第1层→第2层 | task_id+中断类型 | 中断确认 | 将用户取消/暂停/继续指令转发给编排层 |
| 查询任务状态 | 第1层→第2层 | task_id | 任务状态+已输出内容 | 向编排层查询单个任务进度 |
| 查询任务列表 | 第1层→第2层 | 筛选条件（状态、时间范围等） | 任务列表 | 向编排层查询所有异步任务概览 |

**禁止承担的职责**：

| 禁止 | 原因 |
|------|------|
| 不做业务编排 | 编排层的职责 |
| 不做执行引擎 | 执行引擎层的职责 |
| 不调用AI模型 | LLM服务层的职责 |
| 不做工具调度 | 执行引擎层的职责 |
| 不做内容格式转换 | 第1层不识别、不转换、不限制内容格式。纯文本/Markdown/JSON/代码混合等格式均由透传，不做MD→HTML或任何形式的渲染处理 |
| 不做重试 | 编排层+执行引擎层+LLM服务层分别负责 |
| 不做状态管理 | 编排层+执行引擎层的职责 |

| 维度 | 说明 |

#### 1.7 本需求总结的业务场景说明

以下场景说明本架构支持哪些具体业务场景，每种场景对应需求中的哪些能力。

**场景1：日常对话问答**（同步+流式+偶发toolcall）

用户发一条消息，AI逐段回复。这是最常见的场景，不需要任何前置配置，打开就用。有时AI会在回复过程中调用工具——比如用户问"今天天气怎么样"，AI先调天气API拿到数据，再继续回复。toolcall对用户完全透明，用户看到的就是"AI思考了一下，然后给出答案"。

涉及需求：同步模式、流式传输、用户可中断、上下文保持、逐段输出可插入工具调用。

**场景2：深度分析任务**（异步+整包+密集toolcall）

用户提交一个任务（如"分析整个代码库的结构"），关上页面去喝茶。AI在后台反复调用文件读取、代码搜索等工具，一步步分析代码结构。每一次工具调用的结果和推理步骤都记录在任务日志中。用户回来后打开任务列表找到已完成的任务，不仅能看最终结论，还能追溯AI每一步的分析过程。

涉及需求：异步模式、任务列表管理、任务追踪。

**场景3：一边聊天一边跑任务**（同步+异步混合）

用户正在聊天问问题，突然想到要跑一个耗时分析，点"新建任务"提交，转头继续聊天。两个互不干扰。

涉及需求：多会话并发、同步+异步并行。

**场景4：AI用工具查资料做多步推理**（同步+流式+工具调用）

用户问"帮我查一下这周的天气，再对比上周的"。AI先搜索天气API拿到这周数据，再搜索上周数据，最后汇总对比。整个过程用户在线看着AI一步步做。

涉及需求：可复用的链式执行机制、逐段输出可插入工具调用、使用外部资源。

**场景5：AI要干危险操作（同步版）**（同步+流式+敏感toolcall）

用户说"帮我把无用的临时文件都删了"。AI生成回复过程中触发了删除文件的toolcall指令，第2层（编排层）识别出这是敏感操作类型，不直接执行，先通知第1层向用户发送确认消息。用户在界面上看到"要删除临时文件，请确认"，点"同意"后第2层指示第3层（执行引擎层）执行该toolcall。

涉及需求：敏感操作确认、同步模式、逐段输出可插入工具调用。

**场景6：AI要干危险操作（异步版）**（异步+整包+敏感toolcall）

用户提交了一个异步任务"批量处理文件"，AI在后台一步步执行。执行到某一步AI发出删除文件的toolcall指令，第2层（编排层）识别为敏感操作，不执行该toolcall而是暂停任务，标记为"待用户确认"。用户回来后看到待确认提示和具体操作描述，选择"继续"后第2层指示第3层执行该toolcall，选择"取消"则跳过。

涉及需求：敏感操作确认、异步模式、任务状态管理、使用外部资源。

**场景7：切换模型处理不同任务**（系统自动+用户手动+toolcall能力差异）

用户问简单问题时系统自动用轻量快速模型回应（该模型可能不支持toolcall，系统对其只输出纯文本），问代码分析时自动切到支持toolcall的强模型。用户也可以在界面上手动切换当前会话使用的模型。不同模型的toolcall能力和输出格式差异由第4层（LLM服务层）统一屏蔽，第2、3层不感知。

涉及需求：模型可切换（两级切换）、不同模型能力兼容。

**场景8：AI中途卡住自动重试**（失败自动恢复+toolcall超时重试）

网络突然断了一下，第4层（LLM服务层）自动重试连回来，继续接收AI的流式输出。另一种情况：AI发出了一个toolcall指令，但工具调用超时了（比如外部API没响应），第3层（执行引擎层）识别为工具执行失败，自动重新请求AI生成新的toolcall或给出替代方案。如果工具执行后返回空结果或错误，AI如实告知用户"查不到"或"执行失败"，由用户决定后续操作。用户看到的是"AI停顿了一下，又继续了"。

涉及需求：失败自动重试（通信层+业务层）、使用外部资源。

**场景9：多轮对话持续改进**（上下文保持+toolcall结果回传）

用户先说"帮我写个排序函数"，AI写完。用户再说"改成降序"。AI知道前面写了排序函数，直接改参数，不会从零开始。更复杂的场景：第一轮AI调用了代码搜索工具找到了某个函数的位置，第二轮用户说"在这个函数旁边加一个注释"，AI带着第一轮的工具结果继续处理，不需要重复搜索。

涉及需求：上下文保持、同步模式、逐段输出可插入工具调用。

总结：本架构通过**同步/异步两种交付模式**、**message/toolcall两种响应类型**的正交组合，覆盖从日常对话到复杂任务的各种场景。两种模式的信任机制不同：同步模式靠用户"盯着看"保证信任，用户能看到AI一步步输出，随时可以干预；异步模式靠系统**可信可靠可预期**保证信任——用户不需要盯着，因为系统保证任务不会丢、过程可查、结果可预期。用户只需要在"聊天"和"建任务"之间做选择，其余由系统自动处理。

##  二、正确的架构层次模型－概要设计

**四层架构是旋智体的结构实现**。接入层→编排层→执行引擎层→LLM服务层，四层协作完成"task执行→沉淀经验→工具进化→下一轮更强"的演进闭环。

| 旋智体维度 | 对应层 | 承担职责 |
|-----------|-------|---------|
| **旋**（task螺旋） | 第1层（对外接口）+ 第2层（任务管理）+ 第3层（执行实施）+ 第4层（通信底座） | 四层协作完成一个task，每圈上升一个台阶 |
| **智**（智能决策） | 第2层（编排决策：做什么模式）+ 第3层（执行决策：ReAct循环）+ 第4层（服务决策：选哪个模型） | 分层决策，各司其职 |
| **体**（演进能力） | 第3层-经验库（记忆）+ 第3层-ArtTool/FlowTool合成（进化） | 执行完成后沉淀经验，模式识别后合成工具 |

###   2.2 四层各层的职责定义

以下各层职责定义与1.5节保持一致，此处从概要设计角度列出各层的接口视图和功能清单。

#### 第1层：接入层

| 维度 | 说明 |
|------|------|
| **逻辑功能** | 接收用户请求，转发到第2层，将第2层返回的结果以SSE/查询等协议形式推送给前端 |
| **边界** | 只做协议适配，不做任何业务逻辑 |

**对前端的接口（前端调用第1层）**：

| 接口 | 接口类型 | 请求参数 | 返回值 | 说明 |
|------|---------|---------|--------|------|
| 发送消息 | 同步请求 | 用户消息 | AI回复（SSE流式） | 用户发消息后在线等结果，回复通过SSE通道逐块到达 |
| 发送任务 | 异步请求 | 任务描述 | task_id | 用户发任务后可离开 |
| 查询任务进度 | 查询接口 | task_id | 任务状态+已输出内容 | 用户查看任务执行状态 |

| 维度 | 说明 |

#### 四层调用关系速查

| 调用方向 | 允许 | 禁止 |
|---------|------|------|
| 第1层 → 第2层 | ✅ | 第1层直接调用第3层或第4层 |
| 第1层 → 第3层/第4层 | ❌ | 必须经过第2层编排出 |
| 第2层 → 第3层 | ✅ | 第2层直接调用第4层或LLM API |
| 第2层 → 第4层/LLM | ❌ | 必须通过第3层执行 |
| 第3层 → 第4层 | ✅ | 第3层直接调用LLM API |
| 第4层 → 第3层/第2层 | ❌ | LLM服务层不能反调用上层 |
| 第4层 → LLM API | ✅ | — |

**三个维度的关系说明（职责归属）**：

| 维度 | 定义 | 谁负责 | 取值 |
|------|------|--------|------|
| **响应类型** | LLM返回什么 | 第3层（执行引擎层） | 纯文本 / 工具调用 |
| **传输方式** | 数据怎么到达 | 第3层（执行引擎层） | 流式(逐块) / 整包(一次性) |
| **交付模式** | 用户是否等待 | 第1层+第2层（接入层+编排层） | 同步(在线等) / 异步(可离开) |

三者正交，互不影响。第3层只关心**响应类型**和**传输方式**，不关心交付模式。第1层和第2层共同决定交付模式（第2层决策，第1层用SSE/查询等协议转换给前端）。

注：流式模式下，纯文本块和工具调用信号可在同一流中交错出现（如先吐文本、再下工具调用指令、继续吐文本）。第3层按LLM返回的原始顺序逐块传递，不重新排序。

**多会话并发与隔离**：第2层按会话ID维护任务记录，第3层按会话ID独立维护执行上下文。不同会话的执行引擎实例互不共享上下文，确保会话隔离。同步模式下每个会话独立接收SSE流，异步模式下每个任务独立记录步骤日志。第2层和第3层均不设全局锁或跨会话共享状态，天然支持多会话并发。

                   │                   │ 【对下】           │                    │
                   │                   │ 转发执行中断       │                    │
                   │                   │ ────────────────→│ 终止ReAct循环       │
  ←── 中断确认 ───│ ←── 中断确认 ────│                   │                    │
```

### 2.6 同步/异步模式对比

| 维度 | 同步模式 | 异步模式 |
|------|---------|---------|
| **用户是否等待** | 是，一直等到结果返回 | 否，提交后可离开 |
| **第1层行为** | 调用第2层后接收流式输出，SSE推送给前端 | 调用第2层后立即返回task_id |
| **第2层行为** | 决策同步交付→启动第3层执行→透传输出块 | 创建任务记录→启动第3层后台执行 |
| **第3层行为** | ReAct循环，逐块输出到第2层 | ReAct循环，步骤记录到任务日志 |
| **结果交付** | SSE流式逐段到达（始终流式，无整包选项） | 用户查询时获取，或完成后通知 |
| **用户可中断** | ✅ 暂停（可恢复）+ 停止（不可恢复） | ✅ 停止（不可恢复） |
| **适用场景** | 对话问答（用户在线等） | 任务执行（用户可离开） |

| 前端功能 | 接口类型 | 所属层次 | 接口说明 | 请求参数 | 返回值 |

| 发送任务 | 异步请求 | 第1层 | 接收任务，后台执行 | 任务描述 | task_id |
| 查询任务进度 | 查询接口 | 第1层 | 返回任务当前状态和已执行步骤 | task_id | 任务状态+已输出内容 |
| 获取任务结果 | 查询接口 | 第1层 | 任务完成后返回最终结果 | task_id | 最终结果 |

| 切换模型 | 配置接口 | 第1层 | 用户手动切换当前会话使用的模型，第1层记录到会话配置 | 模型标识 | 切换确认 |

### 2.9 工具体系三层结构：内置tool → ArtTool → FlowTool

第3层（执行引擎层）的工具体系分为三个层级，层层递进：

| 层级 | 名称 | 来源 | 生命周期 | 示例 |
|------|------|------|---------|------|
| **L0** | **内置tool** | 系统预注册的稳定工具 | 始终可用，不依赖经验库 | 文件读写、网络请求、shell执行 |
| **L1** | **ArtTool** | 即兴代码经经验库识别重复模式后自动合成 | 随经验积累动态增长 | 格式转换、数据过滤、文本处理 |
| **L2** | **FlowTool** | ArtTool+内置tool的编排模式经经验库识别后自动合成 | 随编排模式积累动态增长 | 代码审查、批量文件处理 |

三级工具均作为第3层内部组件，由第3层统一调度。

#### 2.9.1 ArtTool（原子工具）

| 维度 | 说明 |
|------|------|
| **定义** | LLM即兴生成的Python代码片段，经经验库识别重复模式后抽象为带参数签名的可复用原子工具 |
| **来源** | 第3层ReAct循环中LLM产出代码块 → 沙箱执行 → 记录到经验库 → 重复N次后自动抽象 |
| **运行方式** | 第3层代码执行引擎沙箱隔离运行 |
| **接口形态** | `def art_tool_name(param1: type, param2: type) -> dict`，带参数签名和返回值 |
| **存储位置** | 经验库工具槽位（第3层内部） |
| **适用场景** | 小型独立任务：文件搜索、格式转换、数据过滤、文本处理 |
| **实现阶段** | **第一阶段实现**（完整功能） |

#### 2.9.2 FlowTool（工作流）

| 维度 | 说明 |
|------|------|
| **定义** | 多个ArtTool与内置tool按固定顺序编排的工作流定义，经经验库识别重复编排模式后自动抽象为可复用工作流 |
| **来源** | 第2层编排 + 第3层执行 → 记录调用序列到经验库 → 重复N次后自动抽象为FlowTool定义 |
| **运行方式** | 第3层执行引擎按FlowTool定义依次调度各步骤（ArtTool或内置tool），步骤间传递数据 |
| **接口形态** | FlowTool定义含：步骤列表（工具名+参数映射）、数据依赖关系、条件分支、超时设置 |
| **存储位置** | 经验库工作流槽位（第3层内部） |
| **适用场景** | 中型多步骤任务：代码审查（拉代码→搜索→分析→汇总）、批量文件处理（遍历→转换→合并） |
| **实现阶段** | **第二阶段实现**（当前仅做占位，预留接口+简要代码，待经验积累和方法论成熟后正式实现） |

**三级关系**：

```
内置tool（L0，系统预置）——始终可用，不依赖经验积累
    ↓
即兴代码执行（运行中动态生成，一次性）
    ↓ N次同类模式
ArtTool（L1，原子工具，自动合成）——可复用，不依赖编排层
    ↓ N次同类编排模式
FlowTool（L2，工作流，自动合成）——跳过编排层逐次决策

内置tool ──┐
              ├── 组合编排 ──→ FlowTool（工作流）
ArtTool ──────┘
```

三级工具的演进关系：**内置tool**是系统预置的基础能力，始终可用的原子原语。**ArtTool**从即兴代码中自动合成，是L0的进化——将一次性代码抽象为可复用工具。**FlowTool**从ArtTool+内置tool的编排模式中自动合成，是L1的进化——将多步编排抽象为工作流。每级都在上一级基础上积累，螺旋越高、工具越多。

### 2.10 旋智体：持久化经验学习与工具进化设计要点

**旋智体**是本架构的系统级演进概念：以task为**螺旋**核心（旋），LLM指挥+Agent协调（智），体系化持久化与自动合成工具（体）。旋智体将原本零散的"持久化经验学习""ArtTool合成""FlowTool合成"等模块统一为一个完整的演进闭环：

```
task(n) 执行 → 沉淀经验 → 经验驱动 task(n+1) 更优
          ↓                   ↓
    基础功能（烂货/好东西的分水岭）    核心差异化
          ↓
    重复模式N次 → ArtTool/FlowTool 自动合成 → 系统能力永久提升
```

**核心公式**：
- **旋智体 = task(n) + 经验积累 → task(n+1) + 经验深化 → 循环演进**
- 做task不是终点，task结束后会**沉淀经验工具**
- 下一个task不是从零开始，而是**站在上一轮肩膀上**
- 螺旋越高，系统越强

#### 2.10.1 设计定位

| 维度 | 说明 |
|------|------|
| **所属层次** | 第3层（执行引擎层），内部核心模块。各子操作的层归属见下表 |
| **职责边界** | 只做经验的结构化存储、检索与更新，不做经验内容的应用决策（应用决策由ReAct循环在运行时做出） |
| **与外部关系** | 依赖DB持久化存储（与暂停状态存储共用DB基础设施）。DB调用由第3层经验库模块发起 |

**各操作的层归属**：

| 操作 | 所属层次 | 说明 |
|------|---------|------|
| **经验总结**（从执行数据中提取特征） | 第3层（经验库模块） | 执行完成后读取步骤日志，提取任务类型/步骤序列/耗时/成功失败/错误模式等特征 |
| **经验持久化**（写入DB） | 第3层→外部DB | 第3层将经验特征写入数据库，DB不在四层架构范围内 |
| **经验检索**（按特征查询DB） | 第3层→外部DB | 第3层按任务特征查询数据库，取回top-k相似经验 |
| **经验注入**（检索结果怎么用） | 第3层（Prompt组装模块） | 将检索结果格式化后注入系统Prompt或用户Prompt，供ReAct循环的初始规划使用 |
| **ArtTool合成**（模式识别→抽象原子工具） | 第3层（经验库模块+代码执行） | 识别重复代码模式，LLM自动生成带参数签名工具代码，写入工具槽位 |
| **FlowTool合成**（ArtTool编排→工作流） | 第3层（经验库模块+ArtTool） | 识别多次按固定顺序调用多个ArtTool的模式，LLM自动生成工作流定义 |

#### 2.10.2 经验存储与更新

| 类别 | 存储内容 | 用途 | 更新时机 | 更新方式 |
|------|---------|------|---------|---------|
| **任务经验** | 任务类型标识、执行步骤序列、成功/失败标记、总耗时、步骤级耗时、错误原因与位置 | 同类任务参考历史步骤，避免重复踩坑 | 执行完成后 | 追加新记录，同类合并更新统计指标 |
| **工具效能** | 工具名、调用场景特征、成功率、平均耗时、常见失败模式 | 帮助ReAct循环在多个可选工具中选择最优方案 | 执行完成后 | 更新已有记录的成功率/耗时等统计指标 |
| **用户偏好** | 用户常用设置、行为模式偏好、特定任务场景的习惯 | 个性化执行，减少不必要的确认交互 | 用户反馈时 | 正向+1、负向-1，累积达±3后生效 |
| **错误模式** | 错误类型（空响应/格式错误/超时/工具失败）、触发条件、解决方式记录 | 快速识别同类错误，自动应用已知解决方案 | 错误发生时 | 新增错误记录，相同错误合并计数 |

同类经验合并原则：同类型任务的多条经验按执行时间加权汇总，保留统计指标（执行次数、成功率、平均耗时），不保留单次原始记录。

> **数据归属**：用户偏好数据归用户所有，实施时需提供数据清除接口，支持用户查看和删除自己的偏好数据。

#### 2.10.3 经验检索与注入

经验检索由第3层经验库模块按任务特征查询DB；检索结果由第3层Prompt组装模块格式化后注入初始Prompt。

```
新任务 → 提取任务特征（意图+领域+复杂度） → 第3层查询DB检索 top-k 相似经验
    ↓
检索结果注入初始Prompt（格式化为历史参考段落）
    ↓
Prompt组装完成 → ReAct循环启动
    ↓
执行过程中遇错 → 增量检索同类错误经验 → 注入重试prompt
```

| 检索维度 | 匹配方式 | 用途 |
|---------|---------|------|
| **任务类型** | 精确匹配 | 同类任务历史步骤参考 |
| **意图+领域** | 特征相似度 | 跨任务但同领域的经验迁移 |
| **错误特征** | 精确匹配+模糊匹配 | 遇到错误时快速定位已知解决方案 |

#### 2.10.4 与ReAct循环的集成

经验库的最终价值体现在它对ReAct循环**三个时点**的赋能上：

```
新任务到达
    ↓
[启动前] 检索历史经验 → LLM生成执行计划（有向步骤图，含条件分支）
    ↓                           ← 经验指导：同类任务做过什么、哪种方案成功率更高
ReAct循环按计划执行
    ├─ 每步成功 → 记录执行数据
    ├─ 遇到错误 → [增量检索]同类错误经验 → 指导重试方向
    └─ 计划偏了 → 重新调LLM修正计划
    ↓
[完成后] 提取执行特征 → 更新经验库
```

| 时点 | 做什么 | 输入 | 输出 |
|------|--------|------|------|
| **启动前** | 检索 top-k 相似经验 → 注入Prompt → LLM生成执行计划 | 任务特征 | 执行计划（步骤列表+条件分支）+ 经验参考提示 |
| **执行中** | 每步记录执行数据到步骤日志；遇错时增量检索同类错误经验 | 执行上下文+错误特征 | 重试策略建议 |
| **完成后** | 提取本次经验特征 → 合并更新到经验库对应类别 | 步骤日志+最终结果 | 经验库更新 |

**执行计划的含义**：ReAct循环不是盲目试错，而是在启动前先让LLM产出一个**简短的执行计划**。计划不是形式文法的AST，而是带条件分支的操作步骤列表（通常不超过10-20步），节点是原子操作（调工具/执行代码/调LLM/读文件等），边是数据依赖和控制流。Agent按图执行，每步完成标记节点状态，遇到异常就修正剩余路径。

经验库在这个阶段提供**历史参考**——同类任务之前怎么做、哪种方案成功率更高——帮助LLM制定出更精准的初始计划。

> **经验检索超时策略**：启动前经验检索设置超时阈值（建议5秒），超时后跳过经验注入直接启动ReAct循环，避免因DB查询慢增加首次响应延迟。执行中遇错的增量检索同理，超时后直接重试不使用经验指导。

#### 2.10.5 经验驱动的工具合成（三层递进：内置tool → ArtTool → FlowTool）

当经验库积累到一定程度，系统能力会在内置tool基础上出现**两级跃迁**：

| 级别 | 产物 | 来源 | 定义 | 适用场景 |
|------|------|------|------|---------|
| **L0 → 内置tool** | 基础工具 | 系统预注册 | 文件读写、网络请求、shell执行等稳定操作，始终可用，不依赖经验积累 | 通用基础操作 |
| **L1 → ArtTool** | 原子工具 | 即兴代码经经验库自动合成 | LLM即兴生成的代码片段，经模式识别后抽象为带参数签名的可复用工具 | 小型独立任务（文件搜索、格式转换、数据过滤） |
| **L2 → FlowTool** | 工作流 | ArtTool+内置tool的编排模式经经验库自动合成 | 多个ArtTool与内置tool按固定顺序编排的工作流定义，自动识别并抽象 | 中型多步骤任务（代码审查=拉代码+搜索+分析+汇总） |

**L1：ArtTool 合成**（从即兴代码到可复用原子工具）

```
经验库积累阶段（2.10.2 × 2.10.4持续运行）
    ↓ 同类代码模式出现N次（默认3次）
模式识别触发（经验库模块）
    ↓ 经验库发现多次任务结构相同，只是参数不同
提取公共逻辑 + 参数化变化部分
    ↓
LLM生成带参数签名的ArtTool代码
    ↓
写入经验库工具槽位 → 新ArtTool下次自动可用
```

| 阶段 | 行为 | 依赖 |
|------|------|------|
| **前N次** | 代码执行即兴生成并执行代码，记录到经验库 | 经验库（记忆）+ 代码执行（实现能力） |
| **第N次后** | 经验库发现重复模式 → LLM自动抽象为ArtTool | 经验库（模式识别） |
| **之后** | 新任务直接调用ArtTool，无需再即兴生成 | 经验库工具槽位 |

**L2：FlowTool 合成**（从ArtTool编排到可复用工作流）

```
多个ArtTool + 内置tool经过多次按固定顺序组合调用后
    ↓
经验库发现编排模式重复N次（默认3次）
    ↓ 步骤序列、数据依赖、条件分支结构相同
LLM生成FlowTool定义（哪些ArtTool/内置tool、调用顺序、数据传递、分支条件）
    ↓
写入经验库工作流槽位 → 新FlowTool下次自动可用
```

| 阶段 | 行为 | 依赖 |
|------|------|------|
| **前N次** | 编排层（第2层）+ ArtTool和内置tool按需组合执行，记录调用序列到经验库 | 经验库（记忆）+ ArtTool（原子能力）+ 内置tool（基础能力） |
| **第N次后** | 经验库发现编排模式重复 → LLM自动抽象为FlowTool | 经验库（模式识别） |
| **之后** | 新任务直接按FlowTool定义执行，跳过编排层的逐次决策 | 经验库工作流槽位 |

**三级递进关系**：

```
内置tool（L0，系统预置）——始终可用，不依赖经验积累
    ↓
即兴代码执行（一次性，运行中动态生成）
    ↓ N次同类模式
ArtTool（L1，原子工具，自动合成）
    ↓ N次同类编排模式
FlowTool（L2，工作流，自动合成）
```

**关键原则**：工具合成不是凭空设计，而是执行经验规律化的自然结果。没有经验库，每步都是新的，无从识别模式。没有代码执行，只能调内置tool，无法动态创造。ArtTool和FlowTool在**内置tool基础上**逐级递进，从原子到编排逐层抽象。所有操作均在**第3层**内部完成。

## 三  详细设计的编写规范

本节定义第 4/5/6/7章（分别对应第4/3/2/1层）详细设计的编写方法和输出要求。

### 3.1 设计方法

| 项目 | 说明 |
|------|------|
| **层间关系** | 四层之间采用调用关系，不采用继承关系。层间接口通过函数调用实现 |
| **层内结构** | 每层内部可以使用类和对象，但类之间是组合/调用关系，不是继承关系 |
| **设计来源** | 严格基于第2章概要设计展开，每层的职责、接口、禁止项必须与概要设计一致 |
| **设计方向** | 自顶向下：先定义模块划分，再定义函数清单，最后定义函数签名和调用关系 |

#### 3.2 设计必须遵守的原则

| 原则 | 说明 | 在详细设计中的体现 |
|------|------|------------------|
| **SRP** — 单一职责 | 一个函数只做一件事 | 每个函数功能说明只描述一件事 |
| **DRY** — 不重复 | 相同逻辑只写一次 | 新增函数前先查是否已有类似功能 |
| **KISS** — 保持简单 | 能简单不复杂 | 函数签名和调用链尽量简洁 |
| **SLAP** — 同一抽象层 | 一个函数不混搭高层编排和底层细节 | 函数说明中明确其抽象层级 |
| **YAGNI** — 不要过度设计 | 不加用不上的接口 | 只设计概要设计要求的功能，不提前设计"可能用到"的功能 |
| **复用优先** | 有公用则复用 | 新增函数前必须先查公用函数清单 |

与2.2节"三个维度的关系说明"一致。此处再列一次方便参考：

| 维度 | 定义 | 谁负责 | 取值 |
|------|------|--------|------|
| **响应类型** | LLM返回什么 | 第3层（执行引擎层） | 纯文本 / 工具调用 |
| **传输方式** | 数据怎么到达 | 第3层（执行引擎层） | 流式(逐块) / 整包(一次性) |

注：流式模式下，纯文本块和工具调用信号可在同一流中交错出现，第3层按LLM返回的原始顺序逐块传递，不重新排序。

#### 4.3.1 流式调用

```python
async def call_stream(
    messages: List[Dict[str, str]],
    tools: Optional[List[Dict]] = None,
    config: Optional[LLMConfig] = None
) -> AsyncIterator[LLMChunk]:
    """
    流式调用LLM，原始SSE响应逐块到达。

    第3层（执行引擎层）通过此接口获取LLM的流式响应。
    先校验参数（messages不能为空/None，tools不能为None），
    再进入流式处理。

    错误报告策略（统一规则）：
    - 流开始前（第一个chunk yield之前）的错误：直接抛异常
    - 流开始后的错误：yield LLMChunk(type="error") 作为最后一个chunk后结束
    - call_complete() 全程抛异常（无流式上下文）

    参数:
        messages: 对话消息列表，格式 [{"role": "user", "content": "..."}]
        tools: 工具定义列表（可选），传入则LLM可调用工具
        config: 模型配置（可选，传None则取模块级默认配置——启动时从config.yaml加载的LLMConfig实例）

    返回:
        异步迭代器，每次yield一个LLMChunk:
        - LLMChunk(type="text", content="...")     # 文本片段
        - LLMChunk(type="toolcall", name="...", arguments="...", toolcall_id="...", index=N)  # 工具调用信号
        - LLMChunk(type="done")                      # 结束标记
        - LLMChunk(type="error", error="...")        # 流中错误（最后一个chunk）

    异常（流开始前）:
        ValueError: messages为空或None、tools包含无效格式
        LLMConnectionError: 网络连接失败（重试后仍失败）
        LLMRateLimitError: 请求频率超限（重试后仍超限）
        LLMResponseError: 响应格式错误
    """

async def call_complete(
    messages: List[Dict[str, str]],
    tools: Optional[List[Dict]] = None,
    config: Optional[LLMConfig] = None
) -> LLMResponse:
    """
    整包调用LLM，一次性返回完整响应。

        ValueError: messages为空或None
        LLMConnectionError: 网络连接失败
        LLMRateLimitError: 请求频率超限
        LLMResponseError: 响应格式错误

### 4.4 调用关系

> **各层的分工**：传输方式（流式/整包）由第3层决策选择，第4层实现两种方式供第3层按需调用。第3层对主循环用流式（实时反馈），对单次调LLM用整包。下图为第3层选定方式后在第4层内部的具体调用链。

#### 4.4.1 流式(stream)调用链

```
第3层调用 call_stream()
    │
    ├── _rate_limit_check()            # 限流检查，控制请求频率
    │
    ├── _adapt_model_capabilities()    # 根据模型调整参数
    │
    ├── _build_request_body()          # 构建OpenAI格式请求体（含tools定义，若有）
    │
    ├── _retry_with_backoff()          # 网络重试封装
    │   └── _http_stream_request()     # HTTP SSE流式请求
    │
    ├── _parse_stream_chunk()          # 解析原始SSE块
    │   └── yield LLMChunk             # 自动识别文本/工具调用/结束
    │
    └── yield LLMChunk(type="done")    # 结束标记
```

```
第3层调用 call_complete()
    │
    ├── _rate_limit_check()            # 限流检查，控制请求频率
    │
    ├── _adapt_model_capabilities()    # 根据模型调整参数
    │
    ├── _build_request_body()          # 构建OpenAI格式请求体
    │
    ├── _retry_with_backoff()          # 网络重试封装
    │   └── _http_complete_request()   # HTTP整包请求
    │
    └── _parse_complete_response()     # 解析完整响应，提取文本和工具调用
        └── 返回 LLMResponse
```

### 4.5 错误处理

| 异常类型 | 触发条件 | 处理方式 | 重试策略 |
|---------|---------|---------|---------|
| `LLMConnectionError` | 网络连接失败、DNS解析失败 | 重试3次，指数退避+jitter（base_delay×2^n ±随机30%）；异常消息必须脱敏，不含api_key | 自动重试 |
| `LLMRateLimitError` | API返回429限流 | 重试3次，等待时间按API返回的Retry-After；异常消息必须脱敏 | 自动重试 |
| `LLMTimeoutError` | 请求超时（默认值取LLMConfig.timeout） | 重试2次，每次增加超时时间 | 自动重试 |
| `LLMResponseError` | 响应格式错误、解析失败 | 流开始前抛异常；流开始后yield LLMChunk(type="error")为最后一个chunk | 不重试 |
| `LLMAuthError` | API密钥无效、认证失败 | 流开始前抛异常；流开始后视为响应错误处理；异常消息不得泄漏api_key | 不重试 |

### 4.6 数据类型定义

```python

class LLMChunk:
    """流式响应块（传输级，不做语义分类）"""

    content: str = ""   # 文本片段（type="text"时）
    name: str = ""      # 工具名称（type="toolcall"时）
    arguments: str = "" # 工具参数JSON片段（type="toolcall"时，可能跨多个chunk拼接）
    toolcall_id: str = "" # 工具调用ID（type="toolcall"时）
    index: int = 0      # 工具调用索引（type="toolcall"时，标识并行多工具调用中的第几个）
    error: str = ""     # 错误信息（type="error"时）

@dataclass
class LLMResponse:
    """整包响应"""
    type: str                                    # "message" | "toolcall"
    content: str                                 # 文本内容
    tool_calls: List[Dict[str, str]]            # 工具调用列表（type="toolcall"时），每个元素含 name/arguments/toolcall_id
    usage: Dict[str, int]                        # token使用量
    model: str                                   # 使用的模型名

@dataclass
class LLMConfig:
    """LLM配置（从系统 config.yaml 加载，运行时只读——修改需在任务间切换时更新）
    
    模块级默认配置：模块启动时从 config.yaml 加载一个 LLMConfig 实例作为默认值，
    各接口的 config=None 即使用此默认实例。上层也可传入自定义 LLMConfig 覆盖。
    
    实施注意：config.yaml 当前缺少多模态开关和上下文窗口大小两项配置，
    后续实施时需补充 llm.multimodal.enabled 和 llm.context_window 字段。
    """
    model: str                                      # 模型名（必填，从config.yaml获取）
    api_key: str                                    # API密钥（必填，从config.yaml获取）
    base_url: str                                   # API地址（必填，从config.yaml获取）
    timeout: int = 120                              # 超时时间（秒，默认2分钟）
    max_retries: int = 3                            # 最大重试次数
    max_rpm: int = 0                                # 每分钟最大请求数（0=不限流）

```

### 3.2 问题1：BaseAIService类跨越两个层次

| 维度 | 说明 |
|------|------|
| **问题描述** | `BaseAIService` 类同时包含用户入口层方法（chat）和LLM服务层方法（chat_stream） |

**精准分析**：
- `chat()` 方法（`llm_core/chat.py:15`）是用户入口，接收用户消息，返回ChatResponse（从原`llm_core.py:219`拆出）

- 同样 `chat_with_tools()`（第1层，`llm_core/tool_caller.py:58`）和 `chat_with_tools_stream()`（第3层，`llm_core/chat_with_tools_stream.py:21`）也存在同样问题

```
当前结构（SRP拆文件但未拆类继承）：

| **问题描述** | `ChatRouter` 类已不存在（SRP已拆为独立函数），但第1层入口函数 `route()` 和第2层编排函数仍在同一目录下 |
| **代码位置** | `chat_router/` 目录（`route.py:19` + `step_start.py:15` + `step_react_loop.py:13` + `detect_intent.py:13` + `init_route_context.py:15`） |
| **涉及方法** | `route()`（第1层）+ `step_start()`/`step_react_loop()`（第2层） |
| **风险等级** | P2-中 |

**精准分析**：
- `route()` 函数（`chat_router/route.py:19`，原 `chat_router.py:279`）是用户入口，接收请求，编排4步流程
- `step_start()` 函数（`chat_router/step_start.py:15`，原 `chat_router.py:255` 的 `_step_start()`）是第2层start步骤编排
- `step_react_loop()` 函数（`chat_router/step_react_loop.py:13`，原 `chat_router.py:267` 的 `_step_react_loop()`）是第2层ReAct循环编排
- `detect_intent()` 函数（`chat_router/detect_intent.py:13`）是第2层意图检测
- `init_route_context()` 函数（`chat_router/init_route_context.py:15`）是第2层初始化
- SRP已做了方法到独立文件的拆分，但第1层和第2层仍在同一个 `chat_router/` 目录下，入口 `route()` 仍直接调用编排函数

**改进措施**：
```
当前结构（已拆为独立文件但同目录混合）：
chat_router/ 目录
├── route.py:19                    ← 第1层：用户入口
├── step_start.py:15               ← 第2层：start步骤
├── step_react_loop.py:13          ← 第2层：ReAct循环
├── detect_intent.py:13            ← 第2层：意图检测
├── init_route_context.py:15       ← 第2层：初始化
├── chat_stream_v2.py              ← FastAPI端点
└── ...

正确结构：
ChatEntry (chat_entry.py)          ← 第1层：用户入口
└── route()

StepOrchestrator (step_orchestrator.py)  ← 第2层：业务编排
├── detect_intent()
├── step_start()
└── step_react_loop()
```

### 3.6 问题5：层与层之间没有清晰的接口边界

| 维度 | 说明 |
|------|------|
| **问题描述** | 层与层之间直接依赖具体实现，没有定义清晰的接口 |
| **涉及模块** | 全局 |
| **风险等级** | P2-中 |

**精准分析**：
- 第1层直接调用第2层/第3层的具体方法，没有接口抽象
- 第2层直接调用第3层的具体方法，没有接口抽象
- 层与层之间耦合度过高，修改一层会影响其他层

**具体案例**：
```
step_react_loop() [chat_router/step_react_loop.py:13]
  └── generate_sse_stream(ai_service=ai_service, ...)  ← 直接传具体对象
        └── run_sse_stream(llm_client=ai_service, ...)  ← 直接传具体对象
              └── AgentFactory.create(...)  ← 直接依赖AgentFactory
                    └── agent.run_stream() [base_react/run_stream.py:17] ← 直接调用

```

**改进措施**：
```
当前结构：
第1层 → 直接调用第2层/第3层的具体方法

正确结构：
第1层 → 定义接口 → 第2层实现接口
第2层 → 定义接口 → 第3层实现接口
```

## 四、改进措施方案

### 4.1 重构建议汇总

| 重构方向 | 具体措施 | 预期效果 | 工作量 |
|---------|---------|---------|--------|
| **拆分BaseAIService** | 将chat()/chat_with_tools()移到第1层入口模块，chat_stream()/chat_with_tools_stream()保留在第3层LLM服务模块 | 消除层次跨越（问题1） | 中 |
| **拆分generate_sse_stream** | 将入口分发逻辑与编排执行逻辑分离到不同模块 | 消除层次跨越（问题2） | 小（文件已拆，职责未分） |
| **拆分BaseAgent** | 将run()移到第1层入口模块，run_stream()保留在第2层编排模块 | 消除层次跨越（问题3） | 中（文件已拆，类继承未拆） |
| **拆分ChatRouter** | 将route()入口与step_*()编排分离到不同模块 | 消除层次混杂（问题4） | 小（文件已拆，同目录未分） |
| **层间接口定义** | 第1层↔第2层↔第3层之间定义清晰接口 | 降低耦合度（问题5） | 中 |
| **重试机制收敛** | 统一使用RetryEngine，SSE路径补充重试保护 | 消除重试分散（问题6） | 小 |

> **注**：2026-05-31 SRP目录拆分后，问题1-4的文件级拆分已完成，但职责级分离（入口vs编排）和类继承解耦仍未完成。
