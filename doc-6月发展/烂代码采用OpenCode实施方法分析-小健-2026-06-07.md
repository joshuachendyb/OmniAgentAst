## 一、总体结论：OpenCode做对了什么，我们做错了什么

### 1.1 OpenCode的核心设计哲学

| 设计原则 | OpenCode做法 | 我方现状 | 差距 |
|---------|-------------|---------|------|
| **一个功能一个入口** | 每个子系统只有1个公开接口（Provider.Send/Stream、Agent.Run/Cancel、Tool.Info/Run） | 同一功能3-4个入口（重试3套、任务追踪3处、SSE保存3处） | **严重** |
| **注入依赖+直接调用** | 全部同步顺序代码：`a.messages.List()` → `provider.Stream()` → `tool.Run()`，无Hook/PubSub/回调 | Mixin透传层、兼容壳函数、多层委托 | **严重** |
| **调度核心薄** | Agent调度核心仅176行，职责单一：`for { stream→tool→collect→判断继续 }` | base_react 421行 + mixin 372行 + sse_wrapper 398行 = 1191行，职责混乱 | **严重** |
| **重试统一** | 1个`shouldRetry()`函数，所有重试走同一逻辑 | 3套重试引擎各自实现，互不复用 | **严重** |
| **事件驱动流式** | 10种ProviderEvent，Provider层创建，Agent层消费 | SSE事件类型混乱，创建和消费边界模糊 | **中等** |
| **工具接口统一** | `BaseTool`接口：`Info()` + `Run(ctx, ToolCall) → ToolResponse` | 工具注册有`get()`和`get_tool()`两个入口，返回格式不同 | **轻微** |

### 1.2 OpenCode不做什么（比做什么更重要）

| OpenCode不做 | 谁在做 | 我方错误做法 | 应改为 |
|-------------|--------|-------------|--------|
| ❌ 不做编排 | 上层app.go | 意图分类+模式切换+Agent分发混在routes.py | 采用OpenCode主Agent+子Agent模式 |
| ❌ 不做Prompt拼接 | config→provider层 | PromptBuildMixin透传PromptAssembler，增加调用链深度 | 直接调PromptAssembler，删除Mixin透传层 |
| ❌ 不做安全过滤 | 每个tool内permission.Request() | SafetyManager两个入口：check_and_execute + execute_with_safety | 统一为1个入口，参考permission.Request模式 |
| ❌ 不做多Agent协调 | AgentTool（SubAgent机制） | 多个Agent子类各自重复初始化逻辑 | 采用SubAgent模式：子Agent+子Session+成本归父 |

## 二、逐项问题分析（按严重度排序）

### 问题1：重试逻辑三分天下 — 最严重的DRY违反

**当前烂代码**：
- `utils/retry_engine.py`（301行）— 通用RetryEngine
- `services/agent/tool_retry_engine.py`（277行）— 工具ToolRetryEngine
- `services/llm_core/llm_core.py:106-118` — `_post_with_retry`独立实现

**违反原则**：DRY（不重复）、SRP（单一职责）、YAGNI（死代码：`execute_context`未使用）

**问题根因**：三个重试引擎各自实现while循环+sleep+退避，互不复用。`ToolRetryEngine`内部自实现了重试循环，未复用`RetryEngine.execute()`。`_post_with_retry`的兜底逻辑绕过了RetryEngine。

**OpenCode的做法**：
```go
// OpenCode只有1个shouldRetry函数（openai.go:337-363）
func shouldRetry(attempts int, err error) (bool, int64, error) {
    // 仅重试 429(限流) 和 500(服务端错误)
    if attempts > maxRetries (8) → return error
    // 优先使用服务端 Retry-After 头
    if Retry-After header exists: retryMs = headerValue * 1000
    else: backoffMs = 2000 * 2^(attempts-1) + jitter
    return true, retryMs
}
// 所有重试场景（LLM调用、工具执行）都走这个函数
```

**采用OpenCode的实施方案**：
1. **删除`ToolRetryEngine`中的自实现重试循环**，改为调用`RetryEngine.execute()`
2. **删除`_post_with_retry`中的兜底逻辑**，429重试统一走`RetryEngine`
3. **删除`RetryEngine.execute_context`死代码**（零调用）
4. **统一退避算法**：采用OpenCode的`2000ms * 2^(attempt-1) + 20%jitter`，优先使用Retry-After头
5. **统一重试条件**：仅429和500可重试，最大8次

**修改后架构**：
```
RetryEngine（唯一重试引擎）
  ├── LLM调用重试：_post_with_retry → RetryEngine.execute()
  ├── 流式429重试：_StreamRetryContext → RetryEngine.execute()
  └── 工具执行重试：ToolRetryEngine → RetryEngine.execute()
```

### 问题2：Agent调度核心过厚 — 1191行 vs OpenCode 176行

**当前烂代码**：
- `base_react/` 421行 — Agent基类
- `react_agent_mixin/` 372行 — Mixin层
- `react_sse_wrapper/` 398行 — SSE包装器
- 合计 **1191行**，而OpenCode调度核心仅**176行**

**违反原则**：SRP（单一职责）、SLAP（同一抽象层）、DRY（Mixin透传冗余）

**问题根因**：
1. PromptBuildMixin只是PromptAssembler的透传层，增加了调用链深度但无附加逻辑
2. `_init_task_tracking`在ReactAgentMixin和AgentInitializer两处实现，行为不一致
3. SSE包装器混入了DB保存、日志记录、错误处理等非调度职责

**OpenCode的做法**：
```go
// Agent调度核心 ≈ 176行，做的事情：
// 1. 起goroutine（20行）           ← 并发执行
// 2. while(needTool) { LLM→tool }（35行）  ← ReAct循环
// 3. for event := range stream（10行）     ← 流式事件输入
// 4. for _, tc := range toolCalls（72行）  ← 工具派发
// 5. switch event.Type（39行）             ← 事件类型分发
// 关键：全部是同步顺序代码，注入依赖+直接调用，无Hook/PubSub/回调
```

**采用OpenCode的实施方案**：
1. **删除PromptBuildMixin透传层**：`_build_candidates_hint`和`_build_cross_tool_hint`直接改为调用`PromptAssembler`的公有方法，Mixin层不再需要
2. **统一`_init_task_tracking`**：删除ReactAgentMixin中的版本，统一使用AgentInitializer版本
3. **SSE包装器瘦身**：将DB保存、日志记录移出调度核心，参考OpenCode的"先写内存，流式完成后一次性批量写入DB"
4. **删除`_init_llm_strategies`空方法**：base_react.py:57的死代码

**修改后架构**：
```
调度核心（参考OpenCode 176行模式）
  ├── ReAct循环：while(needTool) { stream→tool→collect→判断继续 }
  ├── 事件分发：switch event.Type { 10种事件 }
  ├── 工具派发：findTool → tool.Run → collect result
  └── 取消机制：dict[session_id] → cancel_func

基础设施（与调度核心分离）
  ├── DB操作：先写内存，完成时批量写入
  ├── Prompt构建：直接调PromptAssembler（无Mixin透传）
  └── 日志/费用：独立模块，调度核心不沾
```

### 问题3：chat与chat_with_tools聚合逻辑DRY违反

**当前烂代码**：
- `llm_core/chat.py:17-49` — `chat()`聚合逻辑
- `llm_core/tool_caller.py:66-91` — `chat_with_tools()`聚合逻辑
- 两处代码几乎完全相同：`full_content += chunk.content`、reasoning fallback等

**违反原则**：DRY（不重复）

**问题根因**：两个函数各自实现了"逐chunk消费→聚合content→处理reasoning"的逻辑，代码复制粘贴。

**OpenCode的做法**：
```go
// OpenCode只有1个streamAndHandleEvents函数（agent.go:322-438）
// 无论有无工具，都走同一个事件处理循环
for event := range eventChan {
    processEvent(assistantMsg, event)  // 统一处理10种事件
}
// tool calls在EventComplete时从accumulator提取
```

**采用OpenCode的实施方案**：
1. **提取公共聚合逻辑**：创建`_consume_stream()`函数，处理"逐chunk消费→聚合content→reasoning fallback"
2. **chat()和chat_with_tools()都调用`_consume_stream()`**，差异仅在参数（是否传tools）
3. **参考OpenCode的Accumulator模式**：流式过程中用accumulator拼接tool call arguments，结束时一次性提取完整tool calls

### 问题4：SafetyManager双入口 — check_and_execute vs execute_with_safety

**当前烂代码**：
- `safety/manager.py` — `check_and_execute()`（async版，含before/after hook）
- `safety/manager.py` — `execute_with_safety()`（sync版，委托给hook）
- 调用方需知道选哪个

**违反原则**：SLAP（同一抽象层）、SRP（单一职责）

**问题根因**：async和sync两个版本并存，调用方需要知道选哪个。且`check_and_execute`中`operation_func`不支持async函数。

**OpenCode的做法**：
```go
// OpenCode每个工具内部做权限检查，只有1个入口
p := b.permissions.Request(permission.CreatePermissionRequest{
    SessionID: sessionID, ToolName: b.Info().Name, Action: "execute",
})
if !p { return tools.NewTextErrorResponse("permission denied"), nil }
// 执行
result := tool.Run(ctx, toolCall)
```

**采用OpenCode的实施方案**：
1. **统一为1个入口**：`check_and_execute()`，同时支持sync和async函数
2. **删除`execute_with_safety()`**，所有调用方统一走`check_and_execute()`
3. **参考OpenCode的permission.Request模式**：`(session_id, tool_name, action) → bool`矩阵判断
4. **修复async支持**：`check_and_execute`中检测`operation_func`是否为coroutine，是则await

### 问题5：ToolRegistry双查询入口 — get() vs get_tool()

**当前烂代码**：
- `registry.py` — `get()`返回dict格式
- `registry.py` — `get_tool()`返回dataclass
- 两者功能相同仅返回类型不同

**违反原则**：DRY（不重复）

**问题根因**：历史遗留，两个方法为不同调用方提供不同返回格式。

**OpenCode的做法**：
```go
// OpenCode只有1个工具查找方式
for _, availableTool := range a.tools {
    if availableTool.Info().Name == toolCall.Name {
        tool = availableTool
        break
    }
}
result, err := tool.Run(ctx, toolCall)
```

**采用OpenCode的实施方案**：
1. **统一为1个方法**：`get_tool()`返回dataclass（标准格式）
2. **删除`get()`**，需要dict格式的调用方自己调`.to_dict()`或直接访问属性
3. **或者**：保留1个方法，用参数控制返回格式：`get(name, as_dict=False)`

## 四、OpenCode方法 vs 我方方法的对照表

| 维度 | OpenCode方法 | 我方当前烂做法 | 应改为 |
|------|-------------|---------------|--------|
| **重试** | 1个shouldRetry无状态函数 | 3套重试引擎各自实现 | 统一为RetryEngine单入口 |

## 五、OpenCode绝对不能借鉴的部分（我方差异点）

| OpenCode做法 | 我方不采用原因 | 我方替代方案 |
|-------------|---------------|-------------|
| 每delta写一次DB | 过于频繁，SQLite并发写入会锁 | 先写内存，流式完成后批量写入 |
| 静态工具预注册写死列表 | 第二阶段要AST动态生成 | 第一阶段静态注册，第二阶段AST |
| Prompt硬编码在Go源码常量中 | 老系统方法效果差，不能实现复杂任务 | 文本文件方式（prompt.md），运行时加载 |
| 针对特殊provider做适配 | 我们只有OpenAI兼容模式 | 统一接口，厂商差异在adapter层消化 |
| Go的goroutine+channel | Python没有原生等价物 | asyncio.Task + SSE长连接 |
| 单层Agent（无编排层） | 我们有意图分类+多Agent分发 | 主Agent+子Agent模式（已决策采用） |
