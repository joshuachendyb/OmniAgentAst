# task-Agent 概要设计（第2层 + 第3层）修订版 v3

> **修订日期：** 2026-06-02  
> **修订范围：** 第2层（task任务编排 + Agent前置处理功能模块）、第3层（Agent 执行中心 task-Agent）  
> **设计原则：** 第3层对标 OpenCode agent.go，薄调度器模式；第2层只管前置，后置归第3层循环外  

---

## 一、第3层和第2层的设计决策说明

### 1.1 第三层的设计决策

| 决策 | 理由 |
|------|------|
| 采用薄调度器模式 | OpenCode agent.go 的薄调度器只有 ~176 行，只做 5 件事。职责单一、可测试、可替换。成功的软件验证了这个方法是可行的 |
| 实现方式：inline + 构造函数注入 | agent.go 薄调度器的实现用了代码 inline 和构造函数依赖注入两种方法，没有用函数调用的方法 |
| Agent 的紧要功能模块 | 依赖从构造函数传入，Agent 循环里直接调注入对象的接口方法。没有 Hook、没有回调注册、没有 Pub/Sub。全部是同步的顺序代码 |
| 必须参考 OpenCode 的实现 | 薄调度的 Agent 实现方法是经过验证的，不另起炉灶 |

### 1.2 为什么第3层分"循环内"和"循环外"

| 决策 | 理由 |
|------|------|
| 循环内 = 同步薄调度器 | Agent 的核心工作是调 LLM → 执行工具 → 循环。这是关键路径，只做必须做的紧要事情 |
| 循环外 = 异步后置模块 | 持久化、经验提取、工具合成不是关键路径。通过异步方法实现（事件、队列等合适的方式）。后置事件的结果不需要 Agent 即时拿到——比如存入数据库等持久化操作，或者为以后的新任务准备数据 |
| 循环内数据传递 | 如果一个循环步骤产出的数据需要被下一个循环步骤使用，在循环体内增加函数来合成前一个循环的数据，供下一个循环使用 |

### 1.3 为什么第2层只管前置

| 决策 | 理由 |
|------|------|
| 第2层不碰 Agent 执行结果 | Agent 同步返回后，第2层只负责把结果交付给第1层。持久化、经验提取、工具合成由第3层循环外的异步模块处理 |
| 第2层不控制 Agent 循环 | 第2层不控制 ReAct 循环细节、不调度工具、不维护上下文。这些是 Agent 循环内的职责 |
| 第2层执行 Agent 的前置事项 | 同步/异步决策、模型选择、工具集选择、上下文准备——全部是 Agent 启动前的准备工作 |

### 1.4 关键取舍

| 取舍 | 说明 |
|------|------|
| Agent 不存储数据 | Agent 循环内只通过 StepRecorder 记录步骤（同步调用），实际的 DB 写入在循环外的异步模块中完成。Agent 不碰 DB |
| Agent 不做经验 | 经验检索在第2层前置处理的"上下文准备"中完成（检索结果注入 Prompt）。经验更新在第3层循环外异步完成。Agent 不知道经验库的存在 |
| 第2层不调 LLM | 所有 LLM 调用都通过第3层的 LLMService |

### 1.5 待研究的事项

| 事项 | 说明 |
|------|------|
| Prompt 机制 | OpenCode 和 Hermes 各自的 Prompt 注入和 conversation history 的机制流程是什么。弄清楚后再完善第2层和第3层的概要设计 |

## 二、第2层：task任务编排+Agent前置处理功能模块

### 2.1 定位

> **只管 Agent 启动前的一切准备。不管 Agent 结束后的事。**

```
                用户请求
                    │
                    ▼
          ┌──────────────────┐
          │   第2层 前置处理   │
          │   准备 Agent 输入  │
          └────────┬─────────┘
                   │
                   ▼
          ┌──────────────────┐
          │   第3层 Agent     │
          │   ┌───────────┐  │
          │   │ 循环内：    │  │
          │   │ 薄调度器    │  │
          │   │ 5件事(同步) │  │
          │   └─────┬─────┘  │
          │         │        │
          │   ┌─────▼─────┐  │
          │   │ 循环外：    │  │
          │   │ 后置模块    │  │
          │   │ (异步)     │  │
          │   └───────────┘  │
          └──────────────────┘
```

### 2.2 职责

| 职责 | 说明 |
|------|------|
| 同步/异步决策 | 判断本次请求是同步（用户在线等）还是异步（用户可离开） |
| 模型选择 | 确定本次执行使用哪个 LLM 模型 |
| 工具集选择 | 确定 Agent 本次可用的工具范围 |
| 上下文准备 | 加载对话历史、检索相关经验、选择 Prompt 策略、注入项目上下文文件 |
| 组装 Agent 输入 | 将上述准备结果打包为 Agent 需要的输入格式 |
| 创建 Agent 依赖 | 创建 LLMService、ToolRegistry、StepRecorder 实例 |

**前置处理的产出 = Agent 启动所需的全部输入 + 依赖注入完成。**

### 2.3 禁止

| 禁止 | 原因 |
|------|------|
| 不处理 Agent 执行后的任何事 | 后置归第3层循环外 |
| 不控制 ReAct 循环 | 第3层循环内的职责 |
| 不调度工具 | 第3层循环内的职责 |
| 不调 LLM | 第3层通过注入的 LLMService 调用 |

### 2.4 与第3层的接口

**第2层给第3层：**

| 给什么 | 方式 | 说明 |
|--------|------|------|
| LLMService | 构造函数注入 | Agent 调 LLM |
| ToolRegistry | 构造函数注入 | Agent 查找和执行工具 |
| StepRecorder | 构造函数注入 | Agent 记录步骤 |
| 对话消息列表 | 输入参数 | 组装好的完整消息序列（含系统提示词） |
| 运行时配置 | 输入参数 | 模型参数（temperature/max_tokens 等） |

**第3层返回给第2层：**

| 返回 | 方式 | 说明 |
|------|------|------|
| 执行结果 | 同步返回 | Agent 同步返回后，第2层将结果交付给第1层 |

## 三、第3层：Agent 执行中心 (task-Agent)

### 3.1 定位

> **对标 OpenCode agent.go。分为两部分：循环内（同步薄调度器）+ 循环外（异步后置模块）。**

OpenCode agent.go 的结构：

| 部分 | 行数 | 做的事 |
|------|------|--------|
| **薄调度器（核心）** | ~176行 | 5件事（见3.2） |
| **周边基础设施** | ~582行 | DB操作、Provider创建、Summarize、费用核算 |

task-Agent 采用同一结构。区别在于：OpenCode 的周边基础设施里包含消息存储和会话管理——task-Agent 把持久化相关的移到循环外做。

### 3.2 循环内：薄调度器——5 件事（同步顺序）

| # | 事 | 说明 |
|---|-----|------|
| 1 | **启动执行** | 接收输入，初始化循环状态 |
| 2 | **ReAct 循环** | `while 未完成: 调 LLM → 有工具就执行 → 结果回传 → 继续` |
| 3 | **流式事件处理** | 接收 LLM 的流式响应，逐块解析（文本增量 / 工具调用增量） |
| 4 | **工具派发** | 遍历 LLM 返回的工具调用列表，逐个执行，结果回填消息 |
| 5 | **事件类型分发** | 根据响应类型（文本/工具调用/结束/错误）走不同处理分支 |

### 3.3 循环内：构造函数注入的依赖

| 依赖 | 做什么 | Agent 怎么用 |
|------|--------|-------------|
| **LLMService** | 调用 LLM 模型 | `self.llm.call(messages, tools)` |
| **ToolRegistry** | 查找和执行工具 | `self.tools.get(name).run(args)` |
| **StepRecorder** | 记录每一步执行 | `self.recorder.record(thought, action, result)` |

### 3.4 循环外：后置功能模块（异步事件）

Agent 循环同步返回后，以下模块在循环外异步执行。不影响用户拿到结果。

| 功能 | 说明 | 执行方式 |
|------|------|---------|
| **结果持久化** | 对话历史写入 DB、步骤记录写入 DB、用量/费用更新 | 异步任务 |
| **经验提取** | 从执行步骤中提取特征，更新经验库 | 异步任务 |
| **工具合成** | 同类模式出现 N 次后，自动合成 ArtTool 或 FlowTool | 异步任务 |
| **任务状态流转** | 完成 → 标记完成；失败 → 判断是否任务级重试 | 回调/消息 |
| **任务级重试** | 系统故障导致任务失败时，创建新的后台任务重新调度 | 异步任务 |
| **结果交付** | 同步模式：流式推送给第1层；异步模式：更新任务记录 + 推送通知 | 回调/消息 |

**三种异步方式：**

| 方式 | 适用 | 例子 |
|------|------|------|
| **回调 (Callback)** | "做完了通知我一下" | 经验提取完成 → 触发工具合成检查 |
| **消息/事件 (Pub/Sub)** | "谁关心谁自己来取" | 异步任务完成 → 推送通知 |
| **异步任务** | "不着急，后台慢慢做" | 经验提取、工具合成、任务重试 |

### 3.5 循环内不做的（循环外做）

| 循环内不做 | 谁做 |
|-----------|------|
| 不持久化到 DB | 循环外：结果持久化 |
| 不提取经验 | 循环外：经验提取 |
| 不合成工具 | 循环外：工具合成 |
| 不管理任务生命周期 | 循环外：任务状态流转 |
| 不选择模型 | 第2层前置处理 |
| 不选择工具集 | 第2层前置处理 |
| 不加载上下文 | 第2层前置处理 |

### 3.6 工具安全

工具安全是 Agent 循环内机制。工具注册时声明安全级别，Agent 在派发工具时自动检查：

```
Agent 派发 tool_call：
    ├─ 安全级别 = READ_ONLY / SAFE → 直接执行
    └─ 安全级别 = DANGEROUS → Agent 暂停，通知第2层确认
```

## 四、完整流程

```
                    第2层 前置处理
                    ┌──────────────────────────────────────┐
                    │ 1. 同步/异步决策                       │
                    │ 2. 选模型 → 创建 LLMService            │
                    │ 3. 选工具集 → 创建 ToolRegistry        │
                    │ 4. 准备上下文 → 组装 messages[]         │
                    │ 5. 创建 StepRecorder                   │
                    │                                        │
                    │ 产出：注入依赖 + 输入参数                │
                    └──────────┬───────────────────────────┘
                               │
                               ▼
                    第3层 Agent 执行中心
                    ┌──────────────────────────────────────┐
                    │                                      │
                    │  ┌── 循环内（同步顺序）─────────────┐ │
                    │  │                                    │ │
                    │  │  agent = TaskAgent(llm, tools,    │ │
                    │  │                    recorder)       │ │
                    │  │                                    │ │
                    │  │  result = agent.run(msgs, config)  │ │
                    │  │                                    │ │
                    │  │  1. 初始化                          │ │
                    │  │  2. while 未完成:                   │ │
                    │  │       llm.call(msgs, tools)        │ │
                    │  │  3.   for chunk in stream:         │ │
                    │  │         解析 chunk                  │ │
                    │  │  4.   for tc in tool_calls:        │ │
                    │  │         tools.get.run(tc)           │ │
                    │  │         recorder.record(tc, result) │ │
                    │  │  5.   switch type → 继续/结束      │ │
                    │  │                                    │ │
                    │  │  同步返回 final_result              │ │
                    │  └────────────┬───────────────────────┘ │
                    │               │                         │
                    │  ┌────────────▼───────────────────────┐ │
                    │  │  循环外（异步事件/模块）             │ │
                    │  │                                    │ │
                    │  │  ┌─ 结果交付 → 返回第2层/第1层 ─┐  │ │
                    │  │  │                                │  │ │
                    │  │  └─ 异步任务 ──────────────────┐  │  │ │
                    │  │      ├─ 结果持久化 → DB          │  │ │
                    │  │      ├─ 经验提取 → 经验库        │  │ │
                    │  │      ├─ 工具合成检查             │  │ │
                    │  │      ├─ 任务状态流转             │  │ │
                    │  │      └─ 任务级重试               │  │ │
                    │  └──────────────────────────────────┘ │
                    └──────────────────────────────────────┘
```

## 七、概要设计结论

1. **第2层 = 只管前置**——准备 Agent 输入 + 创建依赖注入。不管后置
2. **第3层 循环内 = OpenCode 薄调度器**——5件事，~176行等效，同步顺序
3. **第3层 循环外 = 后置功能模块**——持久化/经验/合成/状态/重试。异步事件驱动
4. **循环内的依赖全是构造函数注入**——LLMService、ToolRegistry、StepRecorder
5. **Agent 不知道 LLM 是谁、不知道结果存哪里、不知道状态怎么流转**
