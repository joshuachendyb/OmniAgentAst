## 1 背景

当前系统有 **23个** 多功能组合工具（通过 `action`/`mode`/`language` 等参数分发多个操作，含内部路由）。在 ToolSearch 机制可用之前，合并是为了减少工具总数、降低 LLM 选择负担。现在 ToolSearch 支持 BM25 全文检索按需发现工具，LLM 可以精准搜索到具体工具名。此时是否应该拆分这些组合工具，让每个工具职责单一（SRP），值得分析。

## 3 逐个评估

### 3.1 评估维度（量化标准）

| 维度 | 说明 | 量化标准 | 权重 |
|------|------|---------|------|
| **参数差异度** | 不同action的参数是否完全不同 | 🔴 高=参数集正交（交集为空）<br>🟡 中=部分重叠（交集非空且非全集）<br>🟢 低=基本相同（交集≈全集） | 高 |
| **概念耦合度** | action之间是否天然属于同一概念 | 🟢 高=天然成对（如compress/extract）<br>🟡 中=同领域不同意图（如create/delete）<br>🔴 低=完全独立 | 高 |
| **ToolSearch收益** | 拆分后LLM能否通过搜索精确找到 | ✅ 高=搜索词与工具名强相关<br>⚠️ 中=搜索词需补充上下文<br>❌ 低=搜索词与工具名弱相关 | 中 |
| **拆分成本** | 代码改动量、实现层是否已独立 | 🟢 低=实现层已独立<br>🟡 中=需抽取共享逻辑<br>🔴 高=需重构核心逻辑 | 中 |

**结论**：`拆` = 建议拆分；`留` = 保持合并；`可拆` = 可拆可不拆，收益不大

### 3.2 高优先级 — 务必拆分

#### 1. `file_operation`（4个action：move/copy/delete/rename）

| 评估维度 | 分析 |
|---------|------|
| **参数差异度** | 🔴 高：delete不需要destination，rename不需要recursive/force，每个action各有不同参数 |
| **概念耦合度** | 🟡 中：都是文件操作，但move/copy/delete/rename是**独立操作**，并不需要同时调用 |
| **ToolSearch收益** | ✅ 高：LLM可搜"delete file"→`delete_file`，搜"rename file"→`rename_file` |
| **拆分成本** | 🟢 低：实现层已经是4个独立的私有方法 `_move_file`/`_copy_file`/`_delete_file`/`_rename_file` |

**结论**：✅ **拆** → `move_file`, `copy_file`, `delete_file`, `rename_file`

**当前参数对比**：

```
file_operation(action="move",   source, destination)         ← 2个必填参数
file_operation(action="copy",   source, destination, recursive, preserve_metadata)  ← 4个参数
file_operation(action="delete", source, recursive, force)     ← 3个参数，无destination！
file_operation(action="rename", source, destination)          ← 2个必填参数
```

LLM 很容易把 delete 的 destination 参数填上，或把 rename 的 recursive 填上。

---

#### 2. `mouse_control`（4个action：click/move/scroll/position）

| 评估维度 | 分析 |
|---------|------|
| **参数差异度** | 🔴 极高：4个action的参数集完全正交——click需要x/y/button；move只需要x/y；scroll需要direction/amount；position不需要任何参数 |
| **概念耦合度** | 🟡 中：都是鼠标操作，但click和position是完全不同的意图 |
| **ToolSearch收益** | ✅ 高：LLM搜"click"→`mouse_click`，搜"scroll"→`mouse_scroll` |
| **拆分成本** | 🟢 低：每个action逻辑独立 |

**结论**：✅ **拆** → `mouse_click`, `mouse_move`, `mouse_scroll`, `mouse_get_position`

**当前参数对比**：

```
mouse_control(action="click",    x, y, button)          ← 2~3个参数
mouse_control(action="move",     x, y)                  ← 2个参数
mouse_control(action="scroll",   direction, amount)     ← 2个完全不同含义的参数
mouse_control(action="position")                        ← 无参数！
```

4个action的参数集几乎正交，LLM 极易混淆（比如给 scroll 传 x/y 坐标）。

---

#### 3. `window_control`（7个action：focus/resize/maximize/minimize/restore/topmost/unpin）

| 评估维度 | 分析 |
|---------|------|
| **参数差异度** | 🔴 高：只有resize需要width/height，其他6个action只需要window_title |
| **概念耦合度** | 🟡 中：都是窗口控制，但focus和resize是独立意图 |
| **ToolSearch收益** | ✅ 高：7个action中6个是简单操作，LLM直接搜"maximize window" |
| **拆分成本** | 🟢 低：逻辑简单，每个action几行代码 |

**结论**：✅ **拆** → `focus_window`, `resize_window`, `maximize_window`, `minimize_window`, `restore_window`, `set_window_topmost`, `unpin_window`

**当前参数对比**：

```
window_control(action="focus",    window_title)            ← 1个参数
window_control(action="resize",   window_title, width, height)  ← 3个参数
window_control(action="maximize", window_title)            ← 1个参数
...（其他5个也都是1个参数）
```

LLM 很容易给 maximize 误传 width/height。

**结论**：✅ **拆** → `create_task`, `delete_task`, `list_tasks`

**当前参数对比**：

```
task_control(action="create", task_name, command, schedule, start_time?, interval?)  ← 3必填+2可选
task_control(action="delete", task_name)                                              ← 1个参数
task_control(action="list",   state?)                                                 ← 0~1个参数
```

### 3.3 中优先级 — 可拆但不急

#### 6. `keyboard_control`（3个action：type/shortcut/combo）

| 评估维度 | 分析 |
|---------|------|
| **参数差异度** | 🟡 中：type需要text+interval，shortcut/combo只需要text_or_keys。且text_or_keys的含义因action而异（type=文本, shortcut=快捷键名, combo=按键列表） |
| **概念耦合度** | 🟡 中：都是键盘操作，但type和shortcut是不同意图 |
| **拆分成本** | 🟢 低 |

**结论**：⚠️ **可拆** → `keyboard_type`, `keyboard_shortcut`, `keyboard_combo`  
（但参数差异没到"完全正交"的程度，收益中等）

#### 7. `archive_tool`（2个action：compress/extract）

| 评估维度 | 分析 |
|---------|------|
| **参数差异度** | 🟡 中：compress需要format/compression_level/exclude_patterns，extract不需要 |
| **概念耦合度** | 🟢 高：compress和extract是成对操作，天然属于同一个工具 |
| **ToolSearch收益** | ✅ 高：LLM搜"compress"→compress_files，搜"extract"→extract_archive，比搜"archive"更精准 |
| **拆分成本** | 🟢 低：_compress_files/_extract_archive已独立 |

**结论**：✅ **拆** → `compress_files`, `extract_archive`  
（原评估"留"，v1.3改为"拆"：ToolSearch下拆分收益明确，LLM可直接搜"compress"或"extract"精准命中）

#### 8. `service_control`（4个action：start/stop/restart/list）

| 评估维度 | 分析 |
|---------|------|
| **参数差异度** | 🟡 中：list不需要service_name，start/stop/restart需要 |
| **概念耦合度** | 🟢 高：都是服务生命周期管理，start/stop/restart天然成组 |

| **拆分成本** | 🟡 中：涉及统一的状态管理 |

**结论**：🟢 **留**  
原因：start/stop/restart是服务生命周期管理，天然成组；list虽参数不同，但概念同属服务管理，拆分收益不大。

### 4.3 实施阶段划分（重新评估版 v3）

#### 划分原则

1. **Phase 1 = FILE+DOCUMENT类集中拆分**：file_operation + archive_tool + read_document + write_document，同属文件/文档操作，无GUI依赖，测试环境简单
2. **Phase 2 = 非SCREEN系统管理类**：task_control + registry_control，同属系统管理类，无GUI依赖
3. **Phase 3 = SCREEN类集中**：mouse_control + window_control + keyboard_control，全部SCREEN分类工具集中处理

```
Phase 1 — FILE+DOCUMENT类（4个工具 → 14个单工具，净增11）
  ├─ file_operation → move_file, copy_file, delete_file, rename_file
  │   理由：
  │   ① 实现层已独立（_move_file/_copy_file/_delete_file/_rename_file），拆分成本最低
  │   ② 4个action参数差异大，LLM混淆风险高
  │   ③ 用它验证完整拆分SOP：schema拆分→tools拆分→register更新→旧代码删除→测试→ToolSearch索引更新
  ├─ archive_tool → compress_files, extract_archive
  │   理由：
  │   ① compress需format/compression_level/exclude_patterns，extract不需要，参数差异中等
  │   ② 原评估"留"是因为概念耦合度高（成对操作），但ToolSearch下LLM搜"compress"→compress_files更精准
  │   ③ 属FILE分类，与file_operation同批处理
  ├─ read_document → read_pdf, read_docx, read_pptx, read_xlsx
  │   理由：
  │   ① 按后缀路由5种解析器，但参数差异大：pdf需pages/extract_tables，xlsx需sheet_name/max_rows/header，csv需encoding/delimiter，pptx无额外参数
  │   ② 拆分后LLM搜"read pdf"→read_pdf，搜"read excel"→read_xlsx，精准度大幅提升
  │   ③ 实现层已独立（_read_pdf/_read_docx/_read_pptx/_read_xlsx），拆分成本低
  │   ④ csv/tsv/json归入read_xlsx（表格类数据读取），不单独拆分
  └─ write_document → write_docx, write_xlsx, write_pdf, write_pptx
      理由：
      ① 按后缀路由4种写入器，参数差异大：docx需content/paragraphs/title/table_data，xlsx需data/sheet_name，pdf需title/content/paragraphs/table_data，pptx需title/slides
      ② 拆分后LLM搜"write excel"→write_xlsx，搜"create pptx"→write_pptx，精准度大幅提升
      ③ 实现层已独立（_write_docx/_write_xlsx/_write_pdf/_write_pptx），拆分成本低

Phase 2 — 非SCREEN系统管理类（2个工具 → 6个单工具，净增4）
  ├─ task_control → create_task, delete_task, list_tasks
  │   理由：
  │   ① 参数差异极大（create 3必填+2可选 vs delete 1个 vs list 0~1个），所有工具中差异最大
  │   ② 属SYSTEM分类，无GUI依赖，测试环境简单
  │   ③ create/delete/list是独立操作，不存在生命周期耦合
  └─ registry_control → registry_read, registry_write, registry_delete
      理由：
      ① write需4个参数 vs read/delete各2个，参数差异大
      ② read/write/delete是独立操作，概念独立
      ③ 属REGISTRY分类，无GUI依赖，与task_control同属系统管理类，可同步处理

Phase 3 — SCREEN类集中（3个工具 → 14个单工具，净增11）
  ├─ mouse_control → mouse_click, mouse_move, mouse_scroll, mouse_get_position
  │   理由：参数集完全正交，LLM最易混淆（scroll传x/y），拆分收益最高
  ├─ window_control → focus_window, resize_window, maximize_window,
  │                   minimize_window, restore_window, set_window_topmost, unpin_window
  │   理由：7个action中6个只需1个参数，resize需3个，LLM极易给maximize误传width/height
  └─ keyboard_control → keyboard_type, keyboard_shortcut, keyboard_combo
      理由：
      ① 参数差异中等（text_or_keys含义因action而异），非完全正交
      ② 概念耦合度中等（都是键盘操作，但type和shortcut意图不同）
      ③ ToolSearch收益中等（搜"type text"→keyboard_type，但搜"keyboard"也能找到原工具）
      ④ 放SCREEN批次末尾，与mouse/window统一处理

  SCREEN类放最后的原因：
  ① 3个工具都属SCREEN分类，共享desktop_tools.py文件，集中处理避免反复切换文件
  ② GUI操作需要桌面环境才能测试，与非SCREEN工具的测试环境要求不同
  ③ mouse_control/window_control高优 + keyboard_control中优，一次拆完SCREEN类
  ④ 拆分总量最大（净增11个工具），放在SOP最成熟的阶段执行
```

## 5 风险与注意事项
