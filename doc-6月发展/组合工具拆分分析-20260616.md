## 1 背景

当前系统有 **23个** 多功能组合工具（通过 `action`/`mode`/`language` 等参数分发多个操作，含内部路由）。在 ToolSearch 机制可用之前，合并是为了减少工具总数、降低 LLM 选择负担。现在 ToolSearch 支持 BM25 全文检索按需发现工具，LLM 可以精准搜索到具体工具名。此时是否应该拆分这些组合工具，让每个工具职责单一（SRP），值得分析。

---

## 3 逐个评估

### 3.1 评估维度（量化标准）

| 维度 | 说明 | 量化标准 | 权重 |
|------|------|---------|------|
| **参数差异度** | 不同action的参数是否完全不同 | 🔴 高=参数集正交（交集为空）<br>🟡 中=部分重叠（交集非空且非全集）<br>🟢 低=基本相同（交集≈全集） | 高 |
| **概念耦合度** | action之间是否天然属于同一概念 | 🟢 高=天然成对（如compress/extract）<br>🟡 中=同领域不同意图（如create/delete）<br>🔴 低=完全独立 | 高 |
| **ToolSearch收益** | 拆分后LLM能否通过搜索精确找到 | ✅ 高=搜索词与工具名强相关<br>⚠️ 中=搜索词需补充上下文<br>❌ 低=搜索词与工具名弱相关 | 中 |
| **拆分成本** | 代码改动量、实现层是否已独立 | 🟢 低=实现层已独立<br>🟡 中=需抽取共享逻辑<br>🔴 高=需重构核心逻辑 | 中 |

**结论**：`拆` = 建议拆分；`留` = 保持合并；`可拆` = 可拆可不拆，收益不大

### 3.2 高优先级 — 务必拆分

#### 1. `file_operation`（4个action：move/copy/delete/rename）

| 评估维度 | 分析 |
|---------|------|
| **参数差异度** | 🔴 高：delete不需要destination，rename不需要recursive/force，每个action各有不同参数 |
| **概念耦合度** | 🟡 中：都是文件操作，但move/copy/delete/rename是**独立操作**，并不需要同时调用 |
| **ToolSearch收益** | ✅ 高：LLM可搜"delete file"→`delete_file`，搜"rename file"→`rename_file` |
| **拆分成本** | 🟢 低：实现层已经是4个独立的私有方法 `_move_file`/`_copy_file`/`_delete_file`/`_rename_file` |

**结论**：✅ **拆** → `move_file`, `copy_file`, `delete_file`, `rename_file`

**当前参数对比**：

```
file_operation(action="move",   source, destination)         ← 2个必填参数
file_operation(action="copy",   source, destination, recursive, preserve_metadata)  ← 4个参数
file_operation(action="delete", source, recursive, force)     ← 3个参数，无destination！
file_operation(action="rename", source, destination)          ← 2个必填参数
```

LLM 很容易把 delete 的 destination 参数填上，或把 rename 的 recursive 填上。

---

#### 2. `mouse_control`（4个action：click/move/scroll/position）

| 评估维度 | 分析 |
|---------|------|
| **参数差异度** | 🔴 极高：4个action的参数集完全正交——click需要x/y/button；move只需要x/y；scroll需要direction/amount；position不需要任何参数 |
| **概念耦合度** | 🟡 中：都是鼠标操作，但click和position是完全不同的意图 |
| **ToolSearch收益** | ✅ 高：LLM搜"click"→`mouse_click`，搜"scroll"→`mouse_scroll` |
| **拆分成本** | 🟢 低：每个action逻辑独立 |

**结论**：✅ **拆** → `mouse_click`, `mouse_move`, `mouse_scroll`, `mouse_get_position`

**当前参数对比**：

```
mouse_control(action="click",    x, y, button)          ← 2~3个参数
mouse_control(action="move",     x, y)                  ← 2个参数
mouse_control(action="scroll",   direction, amount)     ← 2个完全不同含义的参数
mouse_control(action="position")                        ← 无参数！
```

4个action的参数集几乎正交，LLM 极易混淆（比如给 scroll 传 x/y 坐标）。

#### 3. `window_control`（7个action：focus/resize/maximize/minimize/restore/topmost/unpin）

| 评估维度 | 分析 |
|---------|------|
| **参数差异度** | 🔴 高：只有resize需要width/height，其他6个action只需要window_title |
| **概念耦合度** | 🟡 中：都是窗口控制，但focus和resize是独立意图 |
| **ToolSearch收益** | ✅ 高：7个action中6个是简单操作，LLM直接搜"maximize window" |
| **拆分成本** | 🟢 低：逻辑简单，每个action几行代码 |

**结论**：✅ **拆** → `focus_window`, `resize_window`, `maximize_window`, `minimize_window`, `restore_window`, `set_window_topmost`, `unpin_window`

**当前参数对比**：

```
window_control(action="focus",    window_title)            ← 1个参数
window_control(action="resize",   window_title, width, height)  ← 3个参数
window_control(action="maximize", window_title)            ← 1个参数
...（其他5个也都是1个参数）
```

LLM 很容易给 maximize 误传 width/height。

**当前参数对比**：

```
task_control(action="delete", task_name)                                              ← 1个参数
task_control(action="list",   state?)                                                 ← 0~1个参数
```

### 3.3 中优先级 — 可拆但不急

#### 6. `keyboard_control`（3个action：type/shortcut/combo）

|---------|------|
| **参数差异度** | 🟡 中：type需要text+interval，shortcut/combo只需要text_or_keys。且text_or_keys的含义因action而异（type=文本, shortcut=快捷键名, combo=按键列表） |
| **概念耦合度** | 🟡 中：都是键盘操作，但type和shortcut是不同意图 |
| **拆分成本** | 🟢 低 |

**结论**：⚠️ **可拆** → `keyboard_type`, `keyboard_shortcut`, `keyboard_combo`

#### 7. `archive_tool`（2个action：compress/extract）

| 评估维度 | 分析 |
|---------|------|
| **参数差异度** | 🟡 中：compress需要format/compression_level/exclude_patterns，extract不需要 |
| **概念耦合度** | 🟢 高：compress和extract是成对操作，天然属于同一个工具 |

**结论**：🟢 **留**  
原因：压缩和解压是成对出现的操作，概念耦合度高。拆分后LLM需要两次调用才能完成压缩→解压流程，无收益。

## 4 拆分方案汇总

### 4.1 拆分清单

**高优先级**（5个工具 → 18个单工具）：

| 原工具 | 拆分为 | 数量 |
|--------|-------|------|
| `file_operation` | `move_file`, `copy_file`, `delete_file`, `rename_file` | 4 |
| `mouse_control` | `mouse_click`, `mouse_move`, `mouse_scroll`, `mouse_get_position` | 4 |
| `window_control` | `focus_window`, `resize_window`, `maximize_window`, `minimize_window`, `restore_window`, `set_window_topmost`, `unpin_window` | 7 |
| `task_control` | `create_task`, `delete_task`, `list_tasks` | 3 |
| `registry_control` | `registry_read`, `registry_write`, `registry_delete` | 3 |
| | **小计** | **+13个净增** |

**中优先级**（1个工具 → 3个单工具）：

| 原工具 | 拆分为 | 数量 |
|--------|-------|------|

## 5 风险与注意事项

| 风险 | 说明 | 缓释措施 |
|------|------|---------|
| **会话历史失效** | 已有会话历史引用旧工具名，拆分后无法重放 | 接受此代价，旧会话无法重放，新会话使用新工具 |
| **前端显示** | 工具列表变长，用户可能需要适应 | ToolSearch正好解决此问题，更多工具反而更精准 |
| **实现耦合** | 部分组合工具有共享逻辑（如data_file_format的格式检测） | 共享逻辑抽成helper，各独立工具复用 |
| **BM25权重调整** | ToolSearch需要为新工具名重新计算权重 | 拆分后同步更新tool_search索引 |
| **性能风险** | 工具数51→66，ToolSearch索引增29% | 压测BM25检索耗时，确保<100ms |
| **测试风险** | 18个新工具需编写测试用例 | 按Phase分批测试，每个Phase测试通过后再继续 |
| **文档风险** | 工具文档、用户手册、API文档需更新 | 同步更新doc/目录下的工具文档 |

### 7.3 会话历史处理

**接受会话历史失效**:
- 旧会话引用旧工具名（如`file_operation`），拆分后无法重放
- 这是禁止向后兼容的必然代价
- 新会话使用新工具名（如`delete_file`）

**清理工作**:
- ❌ 不添加任何向后兼容代码
- ❌ 不保留旧工具注册
- ✅ 接受旧会话无法重放
