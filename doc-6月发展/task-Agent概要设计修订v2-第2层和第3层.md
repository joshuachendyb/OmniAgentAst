# task-Agent 概要设计（第2层 + 第3层）修订版 v2

> **修订日期：** 2026-06-02  
> **修订范围：** 第2层（Agent 前置/后置处理功能模块）、第3层（Agent 执行中心 task-Agent）  
> **设计原则：** 第3层对标 OpenCode agent.go，功能不多不少；概要设计只定义职责边界和依赖关系  

---

## 一、依赖关系模式说明

OpenCode agent.go 的依赖方式是：**构造函数依赖注入 + 直接接口调用**。Agent 所有依赖从构造函数传入，Agent 循环里直接调注入对象的接口方法。没有 Hook，没有回调注册，没有 Pub/Sub。全部是同步的顺序代码。

task-Agent 采用同一模式：

| 依赖 | 传入方式 | Agent 怎么用 |
|------|---------|-------------|
| LLMService | 构造函数注入 | `self.llm.call(messages, tools)` |
| ToolRegistry | 构造函数注入 | `self.tools.get(name)` |
| StepRecorder | 构造函数注入 | `self.recorder.record(step)` |

**全部是注入，不存在 Hook。Agent 循环里直接调，全是同步的顺序代码。**

## 二、第2层：Agent 前置/后置处理功能模块

### 2.1 定位

> Agent 的"管家"。Agent 只负责执行，执行前的一切准备和执行后的一切处理都在第2层。

```
同步模式：                        异步模式：

用户消息                           用户任务
    │                                  │
    ▼                                  ▼
┌──────────────┐                ┌──────────────┐
│ 第2层 前置处理 │                │ 第2层 前置处理 │
│ 准备输入       │                │ 准备输入       │
└──────┬───────┘                └──────┬───────┘
       │                               │
       ▼                               ▼
┌──────────────┐                ┌──────────────┐
│ 第3层 Agent  │                │ 第3层 Agent  │
│ 执行         │                │ 后台执行      │
└──────┬───────┘                └──────┬───────┘
       │                               │
       ▼                               ▼
┌──────────────┐                ┌──────────────┐
│ 第2层 后置处理 │                │ 第2层 后置处理 │
│ 流式推结果    │                │ 存结果+通知   │
└──────┬───────┘                └──────────────┘
       │
       ▼
   用户看到结果                     用户回来查结果
```

### 2.2 前置处理（Agent 执行前）

| 职责 | 说明 |
|------|------|
| 同步/异步决策 | 判断本次请求是同步（用户在线等）还是异步（用户可离开） |
| 模型选择 | 确定本次执行使用哪个 LLM 模型 |
| 工具集选择 | 确定 Agent 本次可用的工具范围 |
| 上下文准备 | 加载对话历史、检索相关经验、选择 Prompt 策略、注入项目上下文文件 |
| 组装 Agent 输入 | 将上述准备结果打包为 Agent 需要的输入格式 |
| 创建 Agent 依赖 | 创建 LLMService、ToolRegistry、StepRecorder 实例 |

**前置处理的产出 = Agent 启动所需的全部输入 + 依赖注入完成。**

### 2.3 后置处理（Agent 执行后）

Agent 同步返回结果后，第2层后置处理启动。分同步路径和异步路径：

**同步路径（用户等结果，必须完成才能返回）：**

| 职责 | 说明 |
|------|------|
| 结果持久化 | 对话历史写入 DB、步骤记录写入 DB、Token 用量/费用更新 |
| 结果交付 | 同步模式：流式推送给第1层；异步模式：更新任务记录 |

**异步路径（后台执行，不阻塞用户拿到结果）：**

| 职责 | 说明 | 执行方式 |
|------|------|---------|
| 经验提取 | 从执行步骤中提取特征，更新经验库 | 异步任务/goroutine |
| 工具合成 | 同类模式出现 N 次后，自动合成 ArtTool 或 FlowTool | 异步任务/goroutine |
| 任务状态流转 | 完成 → 标记完成；失败 → 判断是否任务级重试 | 回调/消息 |
| 任务级重试 | 系统故障导致整个任务失败时，创建新的后台任务重新调度 | 异步任务 |

**三种异步执行方式的使用场景：**

| 方式 | 适用 | 例子 |
|------|------|------|
| **回调 (Callback)** | "做完了通知我一下" | 经验提取完成 → 触发工具合成检查 |
| **消息/事件 (Pub/Sub)** | "谁关心谁自己来取" | 异步任务完成 → 推送通知给用户 |
| **异步任务 (Goroutine/线程)** | "不着急，后台慢慢做" | 经验提取本身、工具合成、任务重试 |

**与第3层的区别：第3层全部是同步的顺序代码（对标 OpenCode）。第2层后置处理的异步路径采用回调/消息/异步任务，因为经验提取和工具合成不是用户等待的关键路径。**

### 2.4 与第3层的接口

**第2层给第3层的东西（构造函数注入 + 输入参数）：**

| 给什么 | 方式 | 说明 |
|--------|------|------|
| LLMService | 构造函数注入 | Agent 调 LLM。第2层根据模型选择结果创建具体实例 |
| ToolRegistry | 构造函数注入 | Agent 查找和执行工具。第2层根据工具集选择结果填充 |
| StepRecorder | 构造函数注入 | Agent 记录步骤。第2层提供实现（内部写 DB） |
| 对话消息列表 | 输入参数 | 第2层组装好的完整消息序列（含系统提示词） |
| 运行时配置 | 输入参数 | 模型参数（temperature/max_tokens 等） |

**第3层返回给第2层：**

| 返回 | 方式 | 说明 |
|------|------|------|
| 执行结果 | 同步返回 | `result = agent.run(...)` 阻塞直到 Agent 完成 |
| 工具安全确认 | 同步返回/抛异常 | DANGEROUS 工具触发时，Agent 暂停并通知第2层 |

## 三、第3层：Agent 执行中心 (task-Agent)

### 3.1 定位

> **对标 OpenCode agent.go。薄调度器模式。全部是同步的顺序代码。**

OpenCode agent.go 的结构：

| 部分 | 行数 | 做的事 |
|------|------|--------|
| **薄调度器（核心）** | ~176行 | 5件事（见3.2） |
| **周边基础设施** | ~582行 | DB操作、Provider创建、Summarize、费用核算 |

薄调度器和周边基础设施的关系：**全部通过构造函数注入的依赖来访问，Agent 循环里直接调注入对象的接口方法。没有 Hook，没有回调注册。**

task-Agent 采用同一结构。区别在于：OpenCode 的周边基础设施里包含消息存储和会话管理——task-Agent 把这些移到第2层，所以 task-Agent 的周边基础设施比 OpenCode 更少。

### 3.2 薄调度器——5 件事

task-Agent 的薄调度器和 OpenCode 一样，只做 5 件事：

| # | 事 | 说明 |
|---|-----|------|
| 1 | **启动执行** | 接收输入，初始化循环状态 |
| 2 | **ReAct 循环** | `while 未完成: 调 LLM → 有工具就执行 → 结果回传 → 继续` |
| 3 | **流式事件处理** | 接收 LLM 的流式响应，逐块解析（文本增量 / 工具调用增量） |
| 4 | **工具派发** | 遍历 LLM 返回的工具调用列表，逐个执行，结果回填消息 |
| 5 | **事件类型分发** | 根据响应类型（文本/工具调用/结束/错误）走不同处理分支 |

### 3.3 周边基础设施（构造函数注入的依赖）

这些是 Agent 循环里直接调用的注入对象。全部通过构造函数传入：

| 依赖 | 做什么 | Agent 怎么用 |
|------|--------|-------------|
| **LLMService** | 调用 LLM 模型 | `self.llm.call(messages, tools)` |
| **ToolRegistry** | 查找和执行工具 | `self.tools.get(name).run(args)` |
| **StepRecorder** | 记录每一步执行 | `self.recorder.record(thought, action, result)` |

**全部是同步的顺序代码。Agent 循环里直接调这些注入对象的接口方法。**

### 3.4 明确不做的（全部交给第2层）

| 不做 | 交给谁 |
|------|--------|
| 不存储对话历史 | 第2层后置处理 |
| 不管理任务生命周期 | 第2层后置处理 |
| 不提取经验 | 第2层后置处理 |
| 不合成工具 | 第2层后置处理 |
| 不选择模型 | 第2层前置处理（作为依赖注入） |
| 不选择工具集 | 第2层前置处理（作为依赖注入） |
| 不加载上下文 | 第2层前置处理（组装好传入） |
| 不组装 Prompt | 第2层前置处理（组装好传入） |
| 不做同步/异步决策 | 第2层前置处理 |

### 3.5 工具安全

工具安全是 Agent 内部机制。工具注册时声明安全级别，Agent 在派发工具时（第4件事）自动检查：

```
Agent 派发 tool_call：
    ├─ 安全级别 = READ_ONLY / SAFE → 直接执行
    └─ 安全级别 = DANGEROUS
        ├─ 同步模式 → Agent 暂停，通过同步返回通知第2层确认
        └─ 异步模式 → Agent 暂停，标记"待确认"，由第2层后续处理
```

Agent 不关心确认 UI——只负责在 DANGEROUS 时暂停并通知第2层。

## 四、第2层 → 第3层 → 第2层 完整流程

```
                    第2层 前置处理
                    ┌──────────────────────────────────────┐
                    │ 1. 同步/异步决策                       │
                    │ 2. 选模型 → 创建 LLMService 实例       │
                    │ 3. 选工具集 → 创建 ToolRegistry 实例   │
                    │ 4. 准备上下文 → 组装 messages[]         │
                    │ 5. 创建 StepRecorder                   │
                    │                                        │
                    │ 产出：                                  │
                    │  llm_service = AnthropicService(...)    │
                    │  tool_registry = ToolRegistry([...])    │
                    │  messages = [...]                      │
                    │  config = {model, temperature, ...}     │
                    │  recorder = StepRecorder(db)            │
                    └──────────┬───────────────────────────┘
                               │
                               │ 构造函数注入 + 输入参数传入
                               ▼
                    第3层 Agent 执行中心 (task-Agent)
                    ┌──────────────────────────────────────┐
                    │                                      │
                    │  agent = TaskAgent(                  │
                    │      llm=llm_service,      ← 注入    │
                    │      tools=tool_registry,  ← 注入    │
                    │      recorder=recorder,    ← 注入    │
                    │  )                                   │
                    │                                      │
                    │  result = agent.run(messages, config) │
                    │                                      │
                    │  薄调度器 5 件事（全部同步顺序）：      │
                    │  1. 初始化循环状态                     │
                    │  2. while 未完成:                     │
                    │       llm.call(msgs, tools)  ← 注入   │
                    │  3.   for chunk in stream:            │
                    │         解析 chunk                    │
                    │  4.   for tc in tool_calls:           │
                    │         tools.get.run(tc)  ← 注入     │
                    │         recorder.record()  ← 注入     │
                    │       msgs.append(result)             │
                    │  5.   switch type → 文本/工具/结束    │
                    │                                      │
                    │  同步返回 final_result                │
                    └──────────┬───────────────────────────┘
                               │
                               │ result = agent.run(...)
                               ▼
                    第2层 后置处理
                    ┌──────────────────────────────────────┐
                    │ 1. 对话历史写入 DB                     │
                    │ 2. 步骤记录写入 DB（recorder 已收集）   │
                    │ 3. 经验提取 → 经验库                   │
                    │ 4. 工具合成检查（模式识别 × N）        │
                    │ 5. 任务状态 → 完成 / 失败重试 / 暂停    │
                    │ 6. 结果交付                           │
                    └──────────────────────────────────────┘
```

## 六、与 OpenCode agent.go 的对照

| OpenCode agent.go 的... | 我们的做法 |
|--------------------------|-----------|
| `Run()` — 入口 | `agent.run(messages, config)` |
| `processGeneration()` — 主循环 | 薄调度器：第2、5件事 |
| `streamAndHandleEvents()` — 流式处理 | 薄调度器：第3件事 |
| `processEvent()` — 事件分发 | 薄调度器：第5件事 |
| `a.provider` — LLM 调用 | 注入的 LLMService |
| `a.tools` — 工具列表 | 注入的 ToolRegistry |
| `a.activeRequests` — 取消控制 | 中断信号 |
| `a.messages.Create/Update` — 消息存储 | 注入的 StepRecorder（记录，不存储） |
| `a.messages.Create/Update` — 持久化 | **不做 → 第2层后置处理** |
| `a.sessions.Save` — 会话管理 | **不做 → 第2层后置处理** |
| `generateTitle()` — 标题生成 | **不做 → 第2层前置/后置处理** |
| `Summarize()` — 上下文压缩 | **不做 → 第2层后置处理** |
| `TrackUsage()` — 费用核算 | **不做 → 第2层后置处理** |

**所有依赖都是构造函数注入 + 直接接口调用。全部同步的顺序代码。**

---

**概要设计结论：**

1. **第3层 = OpenCode agent.go 的薄调度器**——5件事，~176行等效范围
2. **第3层只做薄调度**——存储、会话、标题、摘要、经验全部在第2层
3. **依赖全部是构造函数注入**——LLMService、ToolRegistry、StepRecorder
4. **全部是同步的顺序代码**——Agent 循环里直接调注入对象的接口方法
5. **Agent 不知道 LLM 是谁、不知道工具结果存哪里、不知道任务状态怎么流转**
