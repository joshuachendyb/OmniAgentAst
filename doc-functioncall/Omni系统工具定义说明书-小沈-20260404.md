## 一、工具定义核心（精要）

### 1.1 工具定义四要素

每个工具定义包含以下4个核心要素：

| 要素 | 来源 | 说明 |
|------|------|------|
| **name** | 手动指定 | 工具名称，LLM 通过此名称调用 |
| **description** | 手动指定 | 工具用途说明，告诉 LLM 何时使用 |
| **parameters** | Pydantic模型自动生成 | 参数类型、必选/可选、参数描述 |
| **examples** | 手动指定 | 使用示例（可选，帮助 LLM 理解） |

### 1.2 一个函数的完整定义包含

| 序号 | 字段 | 来源 | 说明 |
|------|------|------|------|
| 1 | name | @register_tool 手动指定 | 工具名称 |
| 2 | description | @register_tool 手动指定 | 工具用途说明 |
| 3 | input_schema | Pydantic 模型自动生成 | 参数类型、必选/可选、描述 |
| 4 | input_examples | @register_tool 手动指定 | 使用示例（可选） |

### 1.2 工具定义代码结构

```python
@register_tool(
    name="工具名",                                    # ← 1. name
    description="工具用途说明...",                     # ← 2. description
    input_model=工具参数模型,                         # ← 3. Pydantic模型 → 自动生成parameters
    examples=[{...}]                                  # ← 4. examples（可选）
)
async def 工具函数(self, 参数...):
    ...
```

### 1.3 Pydantic 模型定义参数

```python
class 工具名Input(BaseModel):
    参数名: 类型 = Field(..., description="参数说明")           # 必填参数
    参数名: 类型 = Field(default=默认值, description="参数说明")  # 可选参数
```

### 1.4 最终输出格式（OpenAI兼容）

```python
{
    "type": "function",
    "function": {
        "name": "工具名",
        "description": "工具用途...",
        "parameters": {
            "type": "object",
            "properties": {
                "参数名": {
                    "type": "类型",
                    "description": "参数说明"
                }
            },
            "required": ["必填参数"]
        }
    }
}
```

## 二、详细说明与示例

### 2.1 @register_tool 装饰器

**作用**：注册工具，将其添加到工具注册表

**参数**：
- `name`: 工具名称（必填）
- `description`: 工具描述，说明使用场景和参数说明（必填）
- `input_model`: Pydantic 模型类，用于生成参数 Schema（必填）
- `examples`: 使用示例列表，可选，帮助 LLM 理解参数格式

### 2.2 Pydantic 模型

**作用**：用 Pydantic 定义工具参数的结构、类型、验证规则
 
类型是什么？（str、int、bool）
必填还是可选？（... 必填，default=xxx 可选）
参数描述是什么？

**关键特性**：
- `...` 表示必填参数
- `default=xxx` 表示可选参数
- `Field(description="xxx")` 添加参数描述
- `ge`、`le`、`gt`、`lt` 用于数值范围限制

### 2.3 自动 Schema 生成- Pydantic 自动转换

**原理**：通过 `model_json_schema()` 将 Pydantic 模型转换为 JSON Schema

```
Pydantic 模型                          JSON Schema
─────────────────────────────────────────────────────────
str                                  → "type": "string"
int                                  → "type": "integer"
bool                                 → "type": "boolean"
... 必填                             → "required": ["参数名"]
Field(description="xxx")              → "description": "xxx"
Field(default=1)                      → 省略（默认值的参数不在 required 中）
```

### 2.4 最终 Schema 格式 —— 给 LLM 看的"说明书"
```python
{
    "type": "function",
    "function": {
        "name": "read_file",
        "description": "读取文件的内容。使用场景：...",
        "parameters": {
            "type": "object",
            "properties": {
                "file_path": {
                    "type": "string",
                    "description": "文件的完整路径"
                },
                "offset": {
                    "type": "integer",
                    "description": "起始行号，从1开始"
                },
                "limit": {
                    "type": "integer", 
                    "description": "最大读取行数"
                },
                "encoding": {
                    "type": "string",
                    "description": "文件编码"
                }
            },
            "required": ["file_path"]   # 只file_path必填
        }
    }
}
```
LLM 看到后就知道：
- "有个 read_file 工具"
- "必须传 file_path 参数"
- "可以选填 offset、limit、encoding"

### 2.5 一句话总结

| 步骤 | 比喻 |
|------|------|
| 1. 工具注册 | 告诉系统"我有一个工具" |
| 2. Pydantic模型 | 定义工具的参数长什么样 |
| 3. 自动生成Schema | 自动转换成LLM能看懂的格式 |
| 4. 最终Schema | 给 LLM 的完整说明书 |

## 三、示例：list_directory 工具

### 3.1 Pydantic 模型（file_schema.py）

```python
class ListDirectoryInput(BaseModel):
    """list_directory 工具的输入参数"""
    dir_path: str = Field(
        description="目录的完整路径（必须是绝对路径，如 D:/项目代码）"
    )
    recursive: bool = Field(
        default=False,
        description="是否递归列出所有子目录，默认为False（不递归）"
    )
    max_depth: int = Field(
        default=10,
        ge=1,
        le=50,
        description="最大递归深度，仅当 recursive=True 时有效，默认为10"
    )
```

### 3.2 工具注册（file_tools.py）

```python
@register_tool(
    name="list_directory",
    description="""列出指定目录中的所有文件和子目录。

使用场景：
- 当用户想要查看某个文件夹里有什么文件时使用此工具
- 当需要了解目录结构时使用
- 当需要获取文件列表进行进一步操作时使用

参数说明：
- dir_path: 目录的完整路径（必须是绝对路径，如 D:/项目代码）
- recursive: 是否递归列出子目录内容，默认为False
- max_depth: 最大递归深度，仅当 recursive=True 时有效

【重要】必须使用 dir_path 作为参数名""",
    input_model=ListDirectoryInput,
    examples=[
        {"dir_path": "C:/Users/用户名/Documents", "recursive": False},
        {"dir_path": "D:/项目代码", "recursive": True, "max_depth": 3}

)
async def list_directory(
    self,
    dir_path: str,
    recursive: bool = False,
    max_depth: int = 100000,
) -> Dict[str, Any]:
    """列出目录内容"""
    ...
```

### 3.4 Tool Examples 编写指南

**examples 的作用**：
- 帮助 LLM 理解参数的格式和组合方式
- 展示必填参数和可选参数的典型用法
- 覆盖不同的使用场景
- 提高 LLM 调用工具的准确率

**编写原则**：
1. **覆盖典型场景**：至少包含 2-3 个不同场景的示例
2. **展示参数组合**：展示必填参数单独使用、与可选参数组合使用
3. **真实路径**：使用真实的路径格式（Windows 绝对路径）
4. **简洁明了**：每个示例只展示一种用法，不要过于复杂

```python
    input_model=ReadFileInput,
    examples=[
        # 示例1：只传必填参数（最常用场景）
        {"file_path": "D:/OmniAgentAs-desk/backend/app/main.py"},

        {"file_path": "D:/OmniAgentAs-desk/backend/app/services/agent.py", "offset": 1, "limit": 100},

        {"file_path": "D:/OmniAgentAs-desk/doc/Function-Call参考和新设计-小沈-2026-04-03.md", "offset": 500, "limit": 50}

)
```

## 4.1.1 examples 的作用方式

**说明**：`examples` 是在工具注册阶段（`@register_tool`）使用的，帮助 Agent（LLM）理解参数的格式、组合方式和典型用法，提高 LLM 调用工具的准确率。

**注意**：OpenAI API 的 `tools` 格式不支持独立的 `examples` 字段，因此系统在自动生成 Schema 时，会将 `examples` 拼入 `description` 中，让 LLM 能看到使用示例。

### examples 在系统中的处理流程

```
注册阶段：@register_tool(examples=[...])
              ↓
系统存储：examples 保存在工具注册表中
              ↓
生成 Schema 时：examples 拼入 description 末尾
              ↓
发送给 LLM：LLM 从 description 中看到使用示例
              ↓
LLM 调用：参考示例中的参数组合方式，正确调用工具
```

### 拼入 description 后的效果

系统自动将 examples 转换为 description 的追加内容：

```python
# 原始 description

# 拼入 examples 后的 description（实际发送给 LLM 的内容）
"列出指定目录中的所有文件和子目录。\n\n使用场景：...\n\n使用示例：\n- 只传必填参数：{\"dir_path\": \"C:/Users/用户名/Documents\"}\n- 递归列出子目录：{\"dir_path\": \"D:/项目代码\", \"recursive\": true, \"max_depth\": 3}"
```

这样 LLM 就能从 description 中看到具体的参数组合示例，提高调用准确率。

## 4.2 给LLM的message构成时包括tool的介绍方法

**说明**：本节以兼容OpenAI模式为例，展示如何将注册好的tool组装到API请求中发送给LLM，以及LLM返回tool_calls后的完整交互流程。

**交互流程**：
```
用户消息 → 系统组装tools+messages → 调用LLM API
                                      ↓
                              LLM返回tool_calls
                                      ↓
                          系统执行工具获取结果
                                      ↓
                          将结果返回LLM继续对话
                                      ↓
                              LLM返回最终回复
```

### 4.2.1 第一步：组装tools数组

## 五、定义一个 Function 的核心事项

### 5.1 什么是"定义"

**"定义"** = 在系统中**创建并注册**一个工具，让 LLM 能够发现并调用它。

核心动作：
1. 告诉系统工具的名称
2. 告诉系统工具的用途
3. 告诉系统工具需要什么参数
4. （可选）给出使用示例

### 5.2 定义核心事项总结

| 核心事项 | 说明 | 必须？ |
|---------|------|--------|
| **name** | 工具叫什么（LLM 通过此名称调用） | ✅ 必填 |
| **description** | 工具干什么用（告诉 LLM 何时使用） | ✅ 必填 |
| **parameters** | 工具需要什么参数（类型、必填/可选、说明） | ✅ 必填 |
| **examples** | 使用示例（帮助 LLM 理解参数格式） | ❌ 可选 |

### 5.3 定义流程

```
1. 定义 Pydantic 模型
   ↓
2. 使用 @register_tool 注册
   ↓
3. 自动生成 OpenAI 格式 Schema
   ↓
4. LLM 调用时使用
```

### 5.4 核心定义工作（name 确定后）

在 **name** 确定后，**定义一个 Function 的核心工作就是 2 件事**：

| 核心工作 | 说明 | 关键点 |
|----------|------|--------|
| **1. 写准确的 description** | 告诉 LLM 何时使用这个工具 | 使用场景、用途说明 |
| **2. 写完整和准确的 parameters** | 告诉 LLM 需要传什么参数 | 类型、必填/可选、参数说明 |

#### 5.4.1 核心工作如何实现到 2.1-2.3 步骤中
|----------|----------|----------|
| **description** | 2.1 @register_tool | 在装饰器中通过 `description` 参数手动编写 |
| **parameters** | 2.2 Pydantic模型 | 通过 Pydantic 模型类定义参数（类型、必填/可选） |
| | 2.3 自动生成 | 通过 `model_json_schema()` 自动转换为 JSON Schema |


#### 5.4.2 实现流程
```
编写 description → @register_tool(description="...")
                    ↓
定义 parameters → Pydantic 模型类
                    ↓
自动生成 → model_json_schema() → 最终 parameters Schema
```
> **定义一个 Function = name + 准确的 description + 完整准确的 parameters**
（其中 name 确定后，核心就是 description 和 parameters）

### 5.5 LLM的function call tool及参数必填（Required）与可选（Optional）的定义

#### 5.5.1 正确理解和定义：Function Call 与参数的必填和可选【最高原则-定稿版，只有老陈才可以修改】

**说明**：Function Call中参数的"必填"和"可选"，不是传统编程中"必须"和"可选"的意思，而是针对 LLM 返回的时候给出的参数的约束，没有LLM调用tool这个概念，LLM是不可能调用tool的，是Agent调用， 只是LLM给出Agent调用tool的一些必要参数而已

**定义 1：Function Call** ——LLM 必须了解 Agent 的能力，即 Agent 能够执行什么 Function Call Tool。通过 LLM 给出执行的 Tool 和 Tool 的参数，简称为 LLM Tool。 
**定义 2：必填（Required）** ——模型LLM必须提供**的参数。LL返回的时候必须提供参数值。 
**定义 3：可选（Optional）** —— 模型LLM没有必须填写；但是Agent执行tool的时是，由 Agent根据场景需要补充相应的参数的合理的值 或者使用合乎场景的默认值

**关键理解**：

1. `required` 数组中的参数 = **必填**，即 LLM 返回时**必须提供**的参数。不在 `required` 数组中的参数 = **可选**，即 LLM 返回时**不需要提供**的参数。
2. 不在 `required` 数组中的参数 = **可选**，是 LLM 没有必须填写，但是 Agent 执行 Tool 时**必须有**的参数（由 Agent 内部根据场景需要补充相应的参数的合理的值 或者使用合乎场景的默认值|）。

#### 5.5.2 【错误理解】——几种很常见的错误理解和错误分析说明——必须彻底抛弃

**错误一：认为"LLM 调用 Tool"**

| 错误理解 | 正确理解 |
|---------|---------|
| "LLM 调用工具时..." | **LLM 根本不调用 Tool**。LLM 只是给出要执行的 Tool 和参数。是 **Agent** 接收后调用 Tool。 |

**错误分析**：
- **错误根源**：被"Function Calling"这个名字误导，以为 LLM 在执行函数。
- **正确理解**：LLM 只是**决策者**（决定用什么 Tool，给什么参数），Agent 才是**执行者**（接收参数，补全缺失，真正调用 Tool）。

---

**错误二：用"工具执行时是否需要"来判断必填/可选**

| 错误理解 | 正确理解 |
|---------|---------|
| "这个参数工具执行时必须知道吗？是→必填" | 必填/可选是**针对 LLM 返回的约束**。工具执行时**所有参数都必须有值**，区别只在于**值是谁给的**。 |

**错误分析**：
- **错误根源**：从工具实现的角度判断参数是否必填。认为"工具执行时需要这个参数就是必填，不需要就是可选"。
- **正确理解**：必填参数的值**来自 LLM**。可选参数的值**来自 Agent 兜底**（根据场景补充或使用默认值）。工具执行时，参数必须有值，只是来源不同。

---

**错误三：把可选参数理解为"工具执行时不需要"**

| 错误理解 | 正确理解 |
|---------|---------|
| "可选参数就是工具执行时不需要的参数" | 可选参数是 LLM 不需要提供，但 Agent 执行 Tool 时**必须有**的参数。 |

**错误分析**：
- **错误根源**：认为可选参数 = 工具用不上的参数。
- **正确理解**：可选参数是 LLM **没有必须填写**的参数，但 Agent 执行 Tool 时**必须有这个值**。由 Agent 内部根据场景需要补充相应的参数的合理的值，或者使用合乎场景的默认值。

---

**错误四：把必填/可选等同于传统编程的必需/可选参数**

| 错误理解 | 正确理解 |
|---------|---------|
| "必填就是函数调用时必须传的参数，可选就是可以不传的参数" | 传统编程是函数调用，Function Call 是 **LLM 生成参数 -> Agent 接收并兜底 -> Agent 调用 Tool**。 |

**错误分析**：
- **错误根源**：用传统编程思维理解 Function Call。在传统编程中，可选参数意味着函数可以不使用该参数。
- **正确理解**：Function Call 的必填/可选是**针对 LLM 的约束**。LLM 是参数的"提供者"，Agent 是参数的"执行者"。必填 = LLM 必须提供。可选 = LLM 可以不填，Agent 负责兜底。

---

**本质总结**（死守 5.5.1 原文）：

1. **必填参数**：LLM 返回时**必须提供**参数值。
2. **可选参数**：LLM **没有必须填写**，但 Agent 执行 Tool 时**必须有**（由 Agent 内部根据场景需要补充相应的参数的合理的值 或者使用合乎场景的默认值）。

#### 5.5.3 两种表达形式

**方式 1：Pydantic 写法（代码定义）**：
```python
# 必填：使用 Field(...)，省略 default
file_path: str = Field(..., description="文件的完整路径")

# 可选：必须指定 default 值
head: int = Field(default=None, description="读取文件的前 N 行")
```

**方式 2：JSON Schema 表现（发送给 LLM 的格式）**：
```json
{
  "required": ["file_path"],
  "properties": {
    "file_path": { "type": "string", "description": "文件的完整路径" },
    "head": { "type": "integer", "description": "读取文件的前 N 行" }
  }
}
```
| 设置 | 行为 |
|------|------|
| `strict: true` | 模型**必须严格遵循** JSON Schema，必填参数一个都不能少 |
| 不设置 strict | 模型尽力遵循，但可能偶尔遗漏必填参数 |

**实践建议**：生产环境务必设置 `strict: true`。

#### 5.5.4 设计注意事项

| 注意事项 | 说明 |
|---------|------|
| **必填参数尽量少** | 必填参数过多会导致 LLM 提取困难，反复追问用户，中断执行流程。只保留 Agent 无法自行推断的核心参数。 |
| **可选参数定义要留白** | **严禁在 Tool 定义中写死业务默认值**（如 `format="zip"`）。应设为 `None`，将决策权交给 Agent，由 Agent 根据场景动态补充合理值。 |
| **Agent 补全策略要智能** | Agent 必须根据上下文（用户意图、文件类型等）动态计算可选参数的值，而不是盲从 Tool 的预设值。 |
| **避免可选参数过多** | 可选参数过多会增加 LLM 的理解负担和幻觉风险。只保留 Agent 确实需要干预的控制参数。 |
| **必填参数不可有默认值** | 必填参数代表 LLM 必须提供的核心信息，绝不能有默认值，否则 LLM 会认为可以不传。 |
