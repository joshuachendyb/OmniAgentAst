## 一、核心概念

本文档分析**普通用户的日常场景**与**Function Call（工具调用）**之间的对应关系。

用户说的是**日常用语（人话）**，系统执行的是**工具调用序列（机器指令）**。中间通过**意图翻译（Intent Translation）**机制进行转换。

---

## 二、从日常场景到 Function Call 的转换逻辑

```
【用户日常场景】：日常用语（非结构化）
      ↓
【意图翻译层】（中间处理机制）：
  1. 意图识别（用户想干嘛？）
  2. 步骤拆解（需要分几步做？）
  3. 参数提取（从话里抠出关键信息）
      ↓
【Function Call 层】：生成执行计划（Plan）
  Step 1: Tool A(参数)
  Step 2: Tool B(参数)
  ...
```

---

## 三、典型日常场景与 Function Call 对应关系

以下梳理日常最高频的 7 类场景，展示从"日常场景"到"Function Call 调用序列"的转换过程：

### 场景 1：找文件（跨类别组合）

*   **用户日常场景**："帮我找下上周下载的那个合同"
*   **意图翻译**：
    1.  **意图**：搜索文件，并通知用户结果。
    2.  **拆解**：算时间范围 -> 搜文件 -> 筛时间 -> 确认内容 -> 通知用户。
    3.  **参数**：目录=`Downloads`，关键词=`合同`，时间=`上周`。
*   **对应 Function Call 序列（跨类别）**：
    1.  `get_current_time()` -> **【5 类：时间/日期】**获取当前时间，计算"上周"范围
    2.  `search_files(search_dir="Downloads", pattern="*合同*")` -> **【1 类：文件操作】**搜索文件
    3.  `get_file_info(file_path="找到的文件 1")` -> **【1 类：文件操作】**检查修改时间是否在上周范围内
    4.  `read_text_file(file_path="确认的文件")` -> **【1 类：文件操作】**确认内容是合同
    5.  `send_notification(title="找到文件", message="已找到上周下载的合同：xxx.pdf")` -> **【28 类：通知】**通知用户结果

### 场景 2：改内容（文件编辑类）

*   **用户日常场景**："把这个文档里的旧地址改成新地址"
*   **意图翻译**：
    1.  **意图**：修改文件内容。
    2.  **拆解**：先读文件 -> 再替换内容 -> 自动保存。
    3.  **参数**：文件=`当前文档`，旧内容=`旧地址`，新内容=`新地址`。
*   **对应 Function Call 序列**：
    1.  `read_text_file(file_path="当前文档")`
    2.  `precise_replace_in_file(file_path="当前文档", old_string="旧地址", new_string="新地址")`

### 场景 3：整理文件（文件管理类）

*   **用户日常场景**："帮我把桌面上的照片整理到一个文件夹里"
*   **意图翻译**：
    1.  **意图**：分类移动文件。
    2.  **拆解**：看桌面有什么 -> 建文件夹 -> 把照片移进去。
    3.  **参数**：源=`Desktop`，目标=`照片整理`，类型=`jpg/png`。
*   **对应 Function Call 序列**：
    1.  `list_directory(dir_path="Desktop")`
    2.  `create_directory(dir_path="Desktop/照片整理")`
    3.  `move_file(source="Desktop/照片.jpg", destination="Desktop/照片整理/照片.jpg")` *(循环执行)*

### 场景 4：看文档（文档阅读类）

*   **用户日常场景**："帮我看看这个 PDF 合同写了什么，重点看金额"
*   **意图翻译**：
    1.  **意图**：阅读并提取信息。
    2.  **拆解**：读 PDF -> 提取文本 -> 分析总结。
    3.  **参数**：文件=`合同.pdf`，关注点=`金额`。
*   **对应 Function Call 序列**：
    1.  `read_pdf(file_path="合同.pdf")`
    2.  *(LLM 内部处理：总结文本，提取金额信息)*

### 场景 5：查信息（信息查询类）

*   **用户日常场景**："现在几点了？30 天后是几号？"
*   **意图翻译**：
    1.  **意图**：获取时间 + 计算日期。
    2.  **拆解**：查当前时间 -> 算偏移日期。
    3.  **参数**：偏移=`+30 天`。
*   **对应 Function Call 序列**：
    1.  `get_current_time()`
    2.  `calculate_date(date="今天", days=30)`

### 场景 6：网络诊断（系统维护类）

*   **用户日常场景**："帮我看看网络通不通，能不能上百度"
*   **意图翻译**：
    1.  **意图**：测试网络连通性。
    2.  **拆解**：Ping 目标网站。
    3.  **参数**：目标=`www.baidu.com`。
*   **对应 Function Call 序列**：
    1.  `ping(host="www.baidu.com")`

### 场景 7：打包文件（文件处理类）

*   **用户日常场景**："帮我把这些文件打包，我要发邮件"
*   **意图翻译**：
    1.  **意图**：压缩文件。
    2.  **拆解**：打包指定文件 -> 生成压缩包。
    3.  **参数**：源=`这些文件`，输出=`压缩包.zip`。
*   **对应 Function Call 序列**：
    1.  `compress_archive(source_path="这些文件", output_path="压缩包.zip")`

---

## 四、意图翻译的三大核心能力

| 能力 | 说明 | 例子 |
|------|------|------|
| **1. 意图识别** | 听懂用户想干嘛 | "找下"=搜索，"改成"=编辑，"打包"=压缩 |
| **2. 步骤拆解** | 把一件事拆成几步 | 找文件 = 搜索 + 过滤 + 确认 |
| **3. 参数提取** | 从话里抠出关键信息 | "上周"=时间过滤，"合同"=关键词 |

---

## 五、总结

**日常场景与 Function Call 的对应关系**：

*   **输入**：日常场景（一句话，人话）
*   **中间处理**：意图翻译（Intent Translation）
*   **输出**：Function Call 调用序列（机器指令列表）
*   **本质**：AI 的**规划能力（Planning）**，将非结构化需求转化为结构化工具调用。
