## 1 流式对话完整流程梳理

### 1.1 当前实现的数据流

```
用户点击发送
    ↓
handleSend()
    ↓
executeStreamSend(userMessage)
    ↓
【步骤 1】添加用户消息到 messages
    setMessages((prev) => [...prev, userMessage])
    ↓
【步骤 2】创建空的 assistant 消息占位
    assistantMessage = {
      content: "",
      isStreaming: true,
      executionSteps: []
    }
    setMessages((prev) => [...prev, assistantMessage])
    ↓
【步骤 3】调用 SSE Hook
    sendStreamMessage(userMessage.content)
    ↓
SSE 连接建立，开始接收数据
    ↓
onStep(执行步骤) → 更新最后一条消息的 executionSteps
onChunk(内容片段) → 追加到 content
onComplete(完成) → 设置 isStreaming=false
onError(错误) → 设置 isStreaming=false
```

## 3 问题详细分析

### 3.1 问题 1：错误消息没有标识（P0-紧急）

**位置**: `NewChatContainer.tsx:245-268`

**问题描述**:
- `onError` 回调没有设置 `isError: true` 标记
- 错误消息和正常消息样式完全一样
- 用户无法从视觉上区分错误消息

**当前代码**:
```typescript
// onError - 流式错误
useCallback((error: string) => {
  console.error("SSE 流式错误:", error);
  message.error({
    content: `AI 响应失败：${error}`,
    duration: 5,
  });

  setMessages((prev) => {
    const lastMessage = prev[prev.length - 1];
    if (lastMessage && lastMessage.role === "assistant") {
      const updated = [...prev];
      updated[updated.length - 1] = {
        ...lastMessage,
        content: lastMessage.content || "抱歉，发生了错误",
        isStreaming: false,
        // ❌ 缺少：isError: true
      };
      return updated;
    }
    return prev;
  });
  setLoading(false);
}, [])
```

**影响**:
- ❌ 用户不知道哪条消息出错了
- ❌ 错误消息没有视觉警示
- ❌ 用户体验差

**修复方案**:
```typescript
setMessages((prev) => {
  const lastMessage = prev[prev.length - 1];
  if (lastMessage && lastMessage.role === "assistant") {
    const updated = [...prev];
    updated[updated.length - 1] = {
      ...lastMessage,
      content: lastMessage.content || `**错误**: ${error}`,
      isError: true,  // ✅ 新增
      isStreaming: false,
    };
    return updated;
  }
  return prev;
});
```

**UI 效果**:
- 错误消息使用红色背景、红色边框、红色文字
- 角色名称显示"AI 助手【错误】"

### 3.2 问题 2：空消息立即显示（P0-紧急）

**位置**: `NewChatContainer.tsx:826-835`

**问题描述**:
- 在发送请求前就创建空的 assistant 消息
- 空消息立即显示在界面上
- 用户看到空白气泡在等待

**当前代码**:
```typescript
const executeStreamSend = async (userMessage: Message) => {
  setLoading(true);
  clearSteps();

  const assistantMessage: Message = {
    id: (Date.now() + 1).toString(),
    role: "assistant",
    content: "",  // ❌ 空内容
    timestamp: new Date(),
    executionSteps: [],
    isStreaming: true,
    model: undefined,
  };
  setMessages((prev) => [...prev, assistantMessage]);  // ❌ 立即添加到列表

  setPendingMessage(userMessage);
  sendStreamMessage(userMessage.content);  // ❌ 然后才发送请求
};
```

**影响**:
- ❌ 空消息立即显示，用户体验差
- ❌ 如果请求失败，空消息已经显示在列表中
- ❌ 消息列表中出现空白气泡

**修复方案**:
```typescript
const executeStreamSend = async (userMessage: Message) => {
  setLoading(true);
  clearSteps();

  // ✅ 不立即创建空消息，在 onChunk 中收到第一个 chunk 再创建
  setPendingMessage(userMessage);
  sendStreamMessage(userMessage.content);
};

// onChunk 回调中
useCallback((chunk: string) => {
  setMessages((prev) => {
    const lastMessage = prev[prev.length - 1];
    // 如果最后一条是 assistant 且 isStreaming，追加内容
    if (lastMessage && lastMessage.role === "assistant" && lastMessage.isStreaming) {
      const updated = [...prev];
      updated[updated.length - 1] = {
        ...lastMessage,
        content: lastMessage.content + chunk,
      };
      return updated;
    }
    // ✅ 如果没有 assistant 消息，创建第一条
    if (!lastMessage || lastMessage.role !== "assistant") {
      const newAssistantMessage: Message = {
        id: (Date.now() + 1).toString(),
        role: "assistant",
        content: chunk,  // ✅ 使用收到的第一个 chunk
        timestamp: new Date(),
        executionSteps: [],
        isStreaming: true,
        model: undefined,
      };
      return [...prev, newAssistantMessage];
    }
    return prev;
  });
}, [])
```

### 3.3 问题 3：执行步骤和内容分离（P1-高）

**位置**: `NewChatContainer.tsx:152-189`

**问题描述**:
- `onStep` 只更新 `executionSteps`
- `onChunk` 只更新 `content`
- 用户看不到思考和回复的对应关系

**当前代码**:
```typescript
// onStep 回调 - 只更新 executionSteps
useCallback((step: ExecutionStep) => {
  setMessages((prev) => {
    const lastMessage = prev[prev.length - 1];
    if (lastMessage && lastMessage.role === "assistant" && lastMessage.isStreaming) {
      const updatedSteps = [...(lastMessage.executionSteps || []), step];
      const updated = [...prev];
      updated[updated.length - 1] = {
        ...lastMessage,
        executionSteps: updatedSteps,
      };
      return updated;
    }
    return prev;
  });
}, [])

// onChunk 回调 - 只更新 content
useCallback((chunk: string) => {
  setMessages((prev) => {
    const lastMessage = prev[prev.length - 1];
    if (lastMessage && lastMessage.role === "assistant" && lastMessage.isStreaming) {
      const updated = [...prev];
      updated[updated.length - 1] = {
        ...lastMessage,
        content: lastMessage.content + chunk,
      };
    }
    return prev;
  });
}, [])
```

**影响**:
- ❌ 无法追溯某段内容是哪个思考步骤产生的
- ❌ 用户看不到 AI 决策过程的关联
- ❌ 调试困难，无法分析 AI 的思考链路

**修复方案**:
```typescript
interface ExecutionStep {
  type: "thought" | "action" | "observation" | "final" | "error" | "interrupted" | "chunk";
  content?: string;
  tool?: string;
  params?: Record<string, any>;
  result?: string;
  timestamp: number;
  stepNumber?: number;
  // ✅ 新增：关联的内容范围
  contentStart?: number;  // 内容起始位置
  contentEnd?: number;    // 内容结束位置
}
```

### 3.4 问题 7：start 事件没有被利用（P1-高）

**位置**: `sse.ts:370-376` 和 `chat.py:369-375`

**问题描述**:
- 后端发送 start 事件，包含 display_name、provider、model、task_id
- 前端只保存 task_id，没有利用其他信息
- start 到 chunk 之间（1-3 秒）用户看到空白

**当前代码**:
```typescript
// 后端发送 start 事件
yield f"data: {json.dumps({
    'type': 'start',
    'display_name': display_name,
    'provider': ai_service.provider,
    'model': ai_service.model,
    'task_id': task_id
})}\n\n"

// 前端接收 start 事件
case "start":
    console.log("[SSE] 收到 start 事件:", rawData);
    if (rawData.task_id && setServerTaskId) {
        setServerTaskId(rawData.task_id);  // ✅ 只保存 task_id
    }
    break;  // ❌ 没有创建 assistant 消息占位
```

**影响**:
- ❌ 浪费 start 事件 - 后端返回了 display_name、provider、model，但前端没有使用
- ❌ 消息创建时机不对 - start 事件是创建 assistant 消息的最佳时机，但没有利用
- ❌ 用户等待时间长 - 要等到第一个 chunk 才显示消息，start 到 chunk 之间用户看到空白

**修复方案**:
```typescript
// 收到 start 事件时
case "start":
    // ✅ 创建占位消息，显示"AI 思考中..."
    const placeholderMessage: Message = {
        id: (Date.now() + 1).toString(),
        role: "assistant",
        content: "🤔 AI 正在思考...",  // ✅ 有内容
        isStreaming: true,
        isPlaceholder: true,  // ✅ 标记为占位
        model: rawData.model,  // ✅ 保存模型信息
        provider: rawData.provider,  // ✅ 保存提供商
        taskId: rawData.task_id,  // ✅ 保存任务 ID
    };
    setMessages((prev) => [...prev, placeholderMessage]);
    if (rawData.task_id && setServerTaskId) {
        setServerTaskId(rawData.task_id);
    }
    break;

// 收到第一个 chunk 时
case "chunk":
    // ✅ 更新占位消息为真实消息
    setMessages((prev) => {
        const lastMessage = prev[prev.length - 1];
        if (lastMessage && lastMessage.isPlaceholder) {
            const updated = [...prev];
            updated[updated.length - 1] = {
                ...lastMessage,
                content: rawData.content,
                isPlaceholder: false,  // ✅ 移除占位标记
            };
            return updated;
        }
        // ...
    });
    break;
```

### 3.5 问题 9：错误处理不一致（P2-中）

**位置**: `sse.ts:444-453` 和 `chat.py:520-521`

**问题描述**:
- 后端直接返回 `str(e)`，可能包含敏感信息
- 前端没有区分网络错误、解析错误、后端错误
- 无法根据错误类型采取不同处理策略

**当前代码**:
```typescript
// 前端错误处理
case "error": {
    setIsReceiving(false);
    setIsConnected(false);
    disconnect();
    const errorMsg = step.content || "未知错误";
    message.error(`流式传输错误：${errorMsg}`);
    onError?.(errorMsg);
    console.error("[SSE] 错误:", errorMsg);
    break;
}

// 后端错误处理
except Exception as e:
    yield f"data: {json.dumps({'type': 'error', 'content': f'流式响应出错：{str(e)}'})}\n\n"
```

**影响**:
- ❌ 错误信息泄露 - 后端直接返回 `str(e)`，可能包含敏感信息
- ❌ 前端没有区分 - 网络错误、解析错误、后端错误都混在一起
- ❌ 没有错误分类 - 无法根据错误类型采取不同处理策略

**修复方案**:
```python
# 后端
except Exception as e:
    # ✅ 记录详细错误日志
    logger.error(f"流式响应异常：{e}", exc_info=True)
    # ✅ 返回用户友好的错误信息
    yield f"data: {json.dumps({'type': 'error', 'content': f'AI 处理异常：{type(e).__name__}'})}\n\n"
```

```typescript
// 前端
const errorInfo = {
  error: error.name === "AbortError" ? "请求超时" : error.message,
  taskId: taskId,  // ✅ 传递 taskId
  type: error.name,
};
onError?.(errorInfo);
```

### 3.7 问题 A：AI 思考中状态不是可选的（P0-紧急）

**位置**: 文档第 259 行

**问题描述**:
- 流式模式下，"AI 思考中..."状态是必须的，不是可选的
- 当前实现没有这个状态，直接显示空白 assistant 消息
- 用户体验差

**影响**:
- ❌ 用户看到空白消息，不知道发生了什么
- ❌ 没有加载提示，用户体验差
- ❌ 可能被误认为系统卡死

**显示位置（明确说明）**:
```
┌─────────────────────────────────┐
│ 消息列表：                       │
│                                 │
│ ┌─────────────────────────┐     │
│ │ [用户] 请帮我分析文件     │     │  ← 用户消息（右侧）
│ └─────────────────────────┘     │
│                                 │
│ ┌─────────────────────────┐     │
│ │ [AI] 🤔 AI 正在思考...   │     │  ← AI 占位消息（左侧）
│ └─────────────────────────┘     │
```

**明确结论**:
- **"AI 思考中..."显示在 AI 侧（左侧）的消息气泡位置**
- **作为一条临时占位消息**
- **收到第一个 chunk 后替换为真实 AI 回复**

**修复方案**: 与问题 7 合并，在 start 事件处理中创建占位消息。

### 3.8 问题 C：start 到 chunk 之间的空白（P1-高）

**修复方案**: 与问题 7 合并，在 start 事件处理中创建占位消息。

### 4.2 影响范围

| 问题 | 影响功能 | 影响用户 | 修复难度 |
|------|---------|---------|---------|
| 错误消息无标识 | 流式输出错误显示 | 所有用户 | ⭐ 简单 |
| 空消息立即显示 | 消息列表显示 | 所有用户 | ⭐⭐ 中等 |
| 执行步骤和内容分离 | 执行过程可视化 | 高级用户 | ⭐⭐⭐ 复杂 |

## 5 正确的流式对话显示流程设计

### 5.1 理想的流程设计

```
用户发送消息
    ↓
显示用户消息
    ↓
【必须】显示"AI 思考中..."状态（带加载动画）
    ↓
收到 start 事件 → 创建占位消息，显示"AI 正在思考..."
    ↓
收到第一个 chunk → 替换占位消息为真实 AI 回复
    ↓
继续接收 chunk，追加到 content
    ↓
同时接收 step，追加到 executionSteps
    ↓
收到 final 或 error
    ↓
设置 isStreaming = false
设置 isError（如果有错误）
设置 model 信息
    ↓
消息完成显示
```

### 5.2 关键改进点对比

| 改进点 | 当前做法 | 正确做法 | 优势 |
|--------|---------|---------|------|
| **消息创建时机** | 立即创建空消息 | start 事件时创建占位消息 | 避免空消息显示 |
| **空消息显示** | 显示空白气泡 | 显示"AI 思考中..."提示 | 用户体验更好 |
| **错误标识** | 无 isError 标记 | 添加 isError 和红色样式 | 一眼识别错误 |
| **模型信息** | 不保存 | 保存 model 字段 | 可追溯模型 |
| **状态管理** | 只有 isStreaming | 添加 isError、isPlaceholder 等 | 状态完整 |
| **执行追溯** | 步骤和内容分离 | 步骤和内容关联 | 可追溯思考过程 |
| **错误处理** | 前后端不统一 | 统一错误处理逻辑 | 用户友好 |
| **消息保存** | 静默失败 | 重试 + 用户提示 | 可靠性高 |
