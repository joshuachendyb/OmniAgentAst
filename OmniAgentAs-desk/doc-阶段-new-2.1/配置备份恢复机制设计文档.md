## 一、设计背景

### 1.1 问题描述

在更换模型配置时，如果新配置不可用（如 API Key 无效、网络问题等），会导致系统无法正常使用 AI 服务。之前的问题：

1. **配置更新后无回滚机制**
   - 更新配置直接写入文件
   - 验证失败无法恢复
   - 用户需要手动修改配置文件

2. **备份文件累积**
   - 每次更新都备份
   - 成功后不删除备份
   - 产生大量无用备份文件

3. **验证流程分离**
   - 配置更新和服务验证是两个独立调用
   - 没有关联备份管理
   - 验证失败后配置已损坏

### 1.2 设计目标

1. **配置可回滚**
   - 验证失败自动恢复到旧配置
   - 保证系统始终可用

2. **备份自动化**
   - 更新前自动备份
   - 成功后自动删除
   - 失败后自动恢复

3. **用户无感知**
   - 前端流程不变
   - 后端自动处理
   - 错误友好提示

## 二、方案设计

### 2.1 核心思路

**事务性备份恢复机制**：

```
更新配置 → 备份 → 验证 → 成功删除/失败恢复
```

### 2.2 完整流程

#### 时序图

```
前端                    后端 (config.py)         后端 (chat.py)         配置文件
 |                           |                        |                     |
 |--updateConfig----------->|                        |                     |
 |                           |                        |                     |
 |                           | 1. 备份 config.yaml     |                     |
 |                           | 2. 更新配置            |                     |
 |                           | 3. 验证配置完整性      |                     |
 |                           | 4. 写入文件            |                     |
 |<--成功 + backup_path------|                        |                     |
 |                           |                        |                     |
 | 显示"已切换到 XXX"         |                        |                     |
 |                           |                        |                     |
 |--validateService--------->|                        |                     |
 |                           |                        |                     |
 |                           |                        | 5. 测试 AI 服务      |
 |                           |                        |                     |
 |                           |                        | 6a. 成功→删除备份   |
 |<--服务正常----------------|                        |                     |
 | 显示绿色对勾               |                        |                     |
 |                           |                        |                     |
 |                           |                        | 6b. 失败→恢复备份   |
 |<--服务不可用--------------|                        |                     |
 | 显示红色错误               |                        |                     |
```

#### 阶段说明

| 阶段 | 调用接口 | 操作 | 备份状态 |
|------|---------|------|---------|
| **1. 配置更新** | `PUT /config` | 备份→更新→返回 | 保留备份 |
| **2. 服务验证** | `GET /chat/validate` | 测试→删除/恢复 | 处理备份 |

### 2.3 关键设计

#### 设计 1：备份保留策略

**之前（有问题）**：
```python
# config.py
update_config():
    备份
    更新配置
    删除备份 ← 立即删除
    返回成功
```

**问题**：
- 验证失败无法恢复
- 备份无意义

**现在（正确）**：
```python
# config.py
update_config():
    备份
    更新配置
    保留备份 ← 暂时保留
    返回 backup_path

# chat.py
validate_ai_service():
    测试服务
    if 成功:
        删除备份 ✅
    else:
        恢复备份 ❌
```

#### 设计 2：备份查找逻辑

```python
def _find_latest_backup():
    """查找最新的备份文件"""
    backup_dir = config_path.parent
    backup_files = sorted(
        backup_dir.glob("config.yaml.backup.*"),
        key=lambda x: x.stat().st_mtime,
        reverse=True
    )
    return backup_files[0] if backup_files else None
```

**原理**：
- 备份文件名包含时间戳：`config.yaml.backup.20260301_163000`
- 按修改时间排序
- 最新的就是当前更新的备份

#### 设计 3：异常处理

```python
try:
    # 更新配置
    update_config()
except Exception as e:
    # 异常时恢复备份
    if backup_path and backup_path.exists():
        restore_backup(backup_path)
        delete_backup(backup_path)
    raise
```

**保证**：
- 任何异常都不会损坏配置
- 备份文件最终会被清理

### 3.1 config.py 修改

#### update_config 函数

**文件位置**: `backend/app/api/v1/config.py`

**关键修改**:

```python
@router.put("/config")
async def update_config(config_update: ConfigUpdate):
    """更新系统配置（带备份恢复机制）"""
    
    backup_path = None  # ⭐ 初始化
    config_path = None
    
    try:
        config_path = _get_config_path()
        
        # 1. 备份
        backup_path = _backup_config_file(config_path)
        
        # 2. 更新配置
        # ...
        
        # 3. 写入文件
        with open(config_path, 'w') as f:
            yaml.dump(config_data, f)
        
        # 4. ⭐ 保留备份（不删除）
        logger.info(f"配置更新成功，备份文件保留：{backup_path}")
        
        # 5. 返回 backup_path
        return {
            "success": True,
            "backup_path": str(backup_path)  # ⭐ 保留路径
        }
        
    except Exception as e:
        # ⭐ 异常时恢复备份
        if backup_path and config_path and backup_path.exists():
            shutil.copy2(str(backup_path), str(config_path))
            backup_path.unlink()
        raise
```

**变更点**：
- ✅ 保留备份文件
- ✅ 返回 backup_path
- ✅ 异常时恢复备份

### 3.2 chat.py 新增

#### 辅助函数

**文件位置**: `backend/app/api/v1/chat.py`

**新增函数**:

```python
async def _delete_backup():
    """验证成功后删除备份"""
    config_path = _get_config_path()
    backup_dir = config_path.parent
    
    # 查找最新备份
    backup_files = sorted(
        backup_dir.glob("config.yaml.backup.*"),
        key=lambda x: x.stat().st_mtime,
        reverse=True
    )
    
    if backup_files:
        backup_files[0].unlink()
        logger.info(f"验证成功，已删除备份")


async def _restore_backup_and_delete():
    """验证失败后恢复备份并删除"""
    config_path = _get_config_path()
    backup_dir = config_path.parent
    
    # 查找最新备份
    backup_files = sorted(
        backup_dir.glob("config.yaml.backup.*"),
        key=lambda x: x.stat().st_mtime,
        reverse=True
    )
    
    if backup_files:
        # 恢复备份
        shutil.copy2(str(backup_files[0]), str(config_path))
        logger.info(f"验证失败，已恢复备份")
        
        # 删除备份
        backup_files[0].unlink()
        logger.info(f"已删除备份")
```

#### validate_ai_service 修改

```python
@router.get("/chat/validate")
async def validate_ai_service():
    """验证 AI 服务配置"""
    
    try:
        ai_service = AIServiceFactory.get_service()
        
        # 检查 API Key
        if not ai_service.api_key:
            # ⭐ 验证失败：恢复备份
            await _restore_backup_and_delete()
            return ValidateResponse(success=False, ...)
        
        # 验证服务
        is_valid = await ai_service.validate()
        
        if is_valid:
            # ⭐ 验证成功：删除备份
            await _delete_backup()
            return ValidateResponse(success=True, ...)
        else:
            # ⭐ 验证失败：恢复备份
            await _restore_backup_and_delete()
            return ValidateResponse(success=False, ...)
            
    except Exception as e:
        # ⭐ 异常：恢复备份
        await _restore_backup_and_delete()
        raise
```

**变更点**：
- ✅ 验证成功删除备份
- ✅ 验证失败恢复备份
- ✅ 异常时恢复备份

### 5.1 测试场景

| 场景 | 操作 | 预期结果 | 备份状态 |
|------|------|---------|---------|
| **正常切换** | 切换到有效模型 | 成功，服务正常 | 已删除 ✅ |

## 六、运维说明

### 6.1 备份文件管理

**正常情况**：
- ✅ 不会有备份文件累积
- ✅ 每次更新后自动清理

**异常情况**：
- ❌ 如果进程被强制杀死
- ❌ 备份文件可能残留

**清理方法**：
```bash
# 手动清理备份文件
rm config/config.yaml.backup.*
```

### 6.2 日志监控

**关键日志**：

| 日志内容 | 级别 | 说明 |
|---------|------|------|
| `配置文件已备份：{path}` | INFO | 备份成功 |
| `配置更新成功，备份文件保留：{path}` | INFO | 更新成功，等待验证 |
| `验证成功，已删除备份：{path}` | INFO | 验证成功 |
| `验证失败，已恢复备份：{path}` | WARNING | 验证失败 |
| `删除备份失败：{error}` | ERROR | 清理失败 |
| `恢复备份失败：{error}` | ERROR | 恢复失败 |

**监控命令**：
```bash
# 查看备份相关日志
tail -f logs/app.log | grep "备份"
```
