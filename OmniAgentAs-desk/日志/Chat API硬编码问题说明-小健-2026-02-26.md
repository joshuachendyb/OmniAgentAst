**文档编号**: ITBC-MD-001  
**问题类型**: 🔴 被测代码问题  
**严重程度**: P1-高  
**创建时间**: 2026-02-26 12:43:28

## 一、问题现象

在前端 `api.ts` 和后端 `chat.py` 中，多处存在对 AI 提供商类型的硬编码，导致：
1. 新增提供商时需要修改多处代码
2. 代码不符合后端 `config.py` 中明确规定的"禁止硬编码Provider名称"原则
3. 前后端支持的提供商不一致，存在兼容性问题

---

## 三、问题严重程度

**严重程度**: P1-高

**影响范围**: 
- 影响新增 AI 提供商的功能扩展
- 影响代码的可维护性和可扩展性
- 前后端支持的提供商不一致，可能导致运行时错误
- 违反了项目明确的"禁止硬编码Provider名称"的规范

**后果**: 
- 新增提供商时需要修改多处代码
- 不符合后端 `config.py` 中明确规定的"禁止硬编码Provider名称"原则
- 代码与设计规范冲突
- 现有的 `longcat` 提供商在某些接口中不被支持

---

## 四、为什么必须修复

### 4.1 设计规范冲突
后端 `config.py` 第48-64行明确规定了：
```
【绝对禁止！硬编码Provider名称 - 所有代码编写人员必须遵守！】
禁止事项：
1. 绝对禁止在代码中硬编码具体的provider名称
2. 所有provider必须从配置文件中动态遍历，不能写死
3. 配置文件里有什么provider，代码就处理什么provider
4. 这是通用程序，不是只给这几个provider用的！
```

当前代码明显违反了这一规范。

### 4.2 实际支持的提供商不一致
- 后端 `PROVIDER_DISPLAY_NAMES` 定义了 3 个提供商：`longcat`, `opencode`, `zhipuai`
- 但 `switch_ai_provider` 仅支持 2 个
- 前端各接口支持的提供商也不一致

### 4.3 可扩展性问题
硬编码会导致：
- 新增提供商需要修改多处代码
- 容易遗漏某些位置
- 代码维护成本高

```python
# 修复前：
# 验证提供商名称
if provider not in ["zhipuai", "opencode"]:
    raise HTTPException(
        status_code=400,
        detail=f"不支持的提供商: {provider}，支持的选项: zhipuai, opencode"
    )

# 修复后（参考config.py的方式）：
# 从配置中动态验证提供商

config = get_config_instance()

if provider not in ai_config:
    raise HTTPException(
        status_code=400,
        detail=f"不支持的提供商: {provider}"
    )
```
