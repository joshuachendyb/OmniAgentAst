## 一、分析背景

### 1.1 项目背景

OmniAgentAs-desk 是一款基于AI的智能桌面助手应用，提供聊天对话、任务执行、数据分析等多种功能。其中，流式API消息显示功能是核心交互体验的重要组成部分，直接影响用户对AI响应的理解和使用体验。

### 1.2 使用场景分类

根据消息类型和内容特点，将流式API消息分为以下3种主要场景：

| 场景分类 | 特点 | 需要显示的内容 |
|---------|------|--------------|
| **简单问答** | 无思考过程，直接输出 | 只显示最终答案 |
| **复杂问答** | 思考过程（连续文本）+ 答案 | 思考过程（可折叠）+ 最终答案 |
| **动作操作类** | 思考过程和操作步骤交替 | 思考过程 + 执行动作（带参数/结果）+ 最终答案 |

## 二、问题描述

### 2.1 核心问题

在当前版本中，流式API消息显示存在严重的时序颠倒和布局问题：

**问题1：消息显示时序颠倒**
- **现象**：AI思考过程（thinking字段）显示在下方，而最终结果显示在上方
- **影响**：不符合用户阅读和理解的逻辑顺序，导致混淆
- **严重程度**：P1-高

**问题2：字段获取错误**
- **现象**：前端错误地获取了 `rawData.observation` 字段而非 `rawData.result` 字段
- **影响**：导致显示大片JSON数据而非用户友好的文本内容
- **严重程度**：P0-紧急

**问题3：数据库未保存执行步骤**
- **现象**：前端未保存 `execution_steps` 到数据库
- **影响**：刷新页面后执行步骤信息消失，用户无法回顾完整会话历史
- **严重程度**：P1-高

**问题4：布局层次不合理**
- **现象**：消息内容和执行步骤的视觉层次关系不正确
- **影响**：降低了信息的可读性和用户体验
- **严重程度**：P2-中

## 三、分析方法和原则

为了解决流式界面UI问题，我们采用以下逻辑逐步分析和解决：

1. **界面显示要求分析**：从用户角度分析流式输出的AI侧UI应该如何优化
2. **布局实现讨论**：确定几层布局、使用什么控件类型
3. **API字段使用分析**：后端SSE返回的信息在界面上如何准确显示
4. **结构化处理决策**：前端处理还是后端处理，或各自处理一部分
5. **需求形成**：要求后端API做对应的修改来支持前端需求

## 四、界面显示要求分析

### 4.1 优化后布局方案（已采纳用户建议）

**核心规则**：
- 思考和执行可能交替出现多次，不一定是一对一的关系
- 思考不折叠，用户可直接查看
- 执行可折叠（结果可能很长）
- 有执行步骤时才显示折叠区域

#### 场景1：简单问答（只有思考或无执行）
```
┌─────────────────────────────────────┐
│ 思考①：我需要分析这个问题...     【黄色】│
│ 思考②：现在我应该...            【黄色】│
├─────────────────────────────────────┤
│ 最终答案content                    │
└─────────────────────────────────────┘
→ 不显示折叠区域
```

#### 场景2：有思考 + 执行（交替出现）
```
┌─────────────────────────────────────┐
│ 思考①：分析问题...              【黄色】│
│ 思考②：需要执行操作...          【黄色】│
│ ▼ 执行详情（可折叠）                                 │
│   工具①：read_file              【蓝色】│
│   结果①：文件内容...            【绿色】│
│   工具②：analyze               【蓝色】│
│   结果②：分析结果...            【绿色】│
├─────────────────────────────────────┤
│ 最终答案content                    │
└─────────────────────────────────────┘
→ 显示折叠区域，包含执行步骤
```

#### 场景3：思考和执行多次交替
```
┌─────────────────────────────────────┐
│ 思考①：先读取配置...            【黄色】│
│ 思考②：配置读取成功...          【黄色】│
│ ▼ 执行详情1                                    │
│   工具：read_file            【蓝色】│
│   结果：配置内容            【绿色】│
│ 思考③：现在分析数据...          【黄色】│
│ 思考④：分析完成...             【黄色】│
│ ▼ 执行详情2                                    │
│   工具：analyze             【蓝色】│
│   结果：分析报告            【绿色】│
├─────────────────────────────────────┤
│ 最终答案content                    │
└─────────────────────────────────────┘
```

### 4.2 边界情况处理

#### 4.2.1 只有思考（无执行）
- **处理**：不显示折叠区域，思考直接显示
- **场景**：简单问答、快速回复

#### 4.2.2 无思考无执行
- **处理**：只显示content
- **场景**：直接返回答案、无需思考

#### 4.2.3 任务中断
- **处理**：显示已接收的内容，标记"（任务已中断）"
- **场景**：用户点击停止、网络中断

#### 4.2.4 错误场景
- **处理**：显示错误信息
- **场景**：API失败、网络超时、参数错误

#### 4.2.5 执行结果很长
- **处理**：折叠收起，保持界面整洁
- **原因**：JSON结果可能很长，用户可选择展开查看

### 5.2 消息渲染的数据流

#### 5.2.1 消息对象结构

```typescript
interface Message {
  id: string;                    // 消息唯一标识
  role: "user" | "assistant";    // 消息角色
  content: string;               // 消息文本内容
  timestamp: Date;              // 时间戳
  executionSteps?: ExecutionStep[];  // 执行步骤数组
  isStreaming?: boolean;        // 是否正在流式输出
  isError?: boolean;            // 是否为错误消息
  displayName?: string;         // AI显示名称
  model?: string;              // 模型名称
  provider?: string;           // 提供商
}
```

#### 5.2.2 SSE流式数据处理

在NewChatContainer中，通过useSSE hook接收流式数据：

```typescript
// useSSE配置
const {
  isReceiving,        // 是否正在接收
  executionSteps,     // 执行步骤数组
  currentResponse,    // 当前响应内容
  sendMessage,        // 发送消息方法
} = useSSE({ baseURL, sessionId });
```

**数据流转过程**：
1. 用户输入消息 → sendMessage发送请求
2. SSE建立连接，后端推送事件
3. onStep回调接收ExecutionStep，更新messages状态
4. 消息渲染到界面

### 5.3 MessageItem组件结构详解

MessageItem是核心渲染组件，其内部结构如下：

#### 5.3.1 外层布局容器

```typescript
<div style={{ display: "flex", alignItems: "flex-start" }}>
  {/* 布局逻辑：AI消息左对齐，用户消息右对齐 */}
</div>
```

**对齐规则**：
- 用户消息：justifyContent = "flex-end"（右侧）
- AI消息：justifyContent = "flex-start"（左侧）
- 系统消息：justifyContent = "center"（居中）

#### 5.3.2 AI消息的内部元素

AI消息从左到右依次包含：

1. **Avatar（头像）**
   - 组件：antd Avatar
   - 图标：RobotOutlined（机器人）
   - 位置：消息左侧

2. **消息内容区域**
   - 包含：角色名称 + 时间戳 + 消息气泡 + 执行面板
   - 最大宽度：calc(100% - 60px)

3. **角色名称和时间戳**
   - 角色名称：如"AI 助手【GPT-4】"
   - 时间戳：相对时间（如"3分钟前"），悬浮显示具体时间
   - 位置：消息气泡上方

4. **消息气泡（content）**
   - 显示message.content的内容
   - 复制按钮悬浮在右上角

5. **执行过程面板（ExecutionPanel）**
   - 通过Collapse组件包裹
   - header显示"AI 思考过程"
   - 流式输出时自动展开

#### 5.3.3 用户消息的结构

用户消息与AI消息结构相同，但有以下差异：
- Avatar在消息右侧（而不是左侧）
- 背景色为蓝色渐变
- 无执行过程面板

### 5.4 ExecutionPanel组件结构详解

ExecutionPanel用于展示AI执行步骤，包含以下元素：

#### 5.4.1 外层Collapse容器

```typescript
<Collapse>
  <Panel header="执行详情">
    {/* 步骤内容 */}
  </Panel>
</Collapse>
```

**Collapse属性**：
- activeKey：控制展开状态，流式输出时自动展开
- size="small"：紧凑尺寸

#### 5.4.2 步骤类型与渲染

ExecutionPanel通过step.type判断步骤类型，渲染不同样式：

| 步骤类型 | 类型标识 | 显示内容 |
|---------|---------|---------|
| thought | 思考 | 黄色Tag + 思考内容 |
| action | 工具调用 | 蓝色Tag + 工具名称 + 参数 + 结果 |
| observation | 观察结果 | 绿色内容区 + 结果数据 |
| final | 最终答案 | 绿色勾选图标 + 最终内容 |
| error | 错误 | 红色错误图标 + 错误信息 |

#### 5.4.3 步骤渲染逻辑

```typescript
// 伪代码：步骤渲染流程
switch (step.type) {
  case "thought":
    return <Tag>思考</Tag> + <内容区>;
  case "action":
    return <Tag>工具</Tag> + <工具名> + <参数> + <结果>;
  case "observation":
    return <观察结果区>;
  case "final":
    return <最终答案区>;
  case "error":
    return <错误信息区>;
}
```

#### 5.6.4 当前显示顺序问题

**问题核心**：执行面板（思考过程）位于消息内容（最终答案）之后

**代码体现**（MessageItem.tsx 第355-478行）：

```typescript
<div style={{ ...getMessageStyle(), position: "relative" }}>
  {/* 1. 复制按钮 - 位置无关紧要 */}
  <Button ... />

  {/* 2. 消息内容（最终答案）- 先显示 */}
  <div>{message.content}</div>

  {/* 3. 执行面板（思考过程）- 后显示 */}
  <Collapse>
    <Panel>
      <ExecutionPanel steps={message.executionSteps} />
    </Panel>
  </Collapse>
</div>
```

**视觉呈现**：

```
┌─────────────────────────────────────────┐
│ AI 助手【GPT-4】        3分钟前         │
├─────────────────────────────────────────┤
│ │                                       │ │
│ │  这是最终答案内容（先显示）           │ │  ← 步骤2：message.content

// StepRow组件：单行步骤显示

const StepRow = ({ step }) => {
  // 步骤类型对应的颜色
  const colorMap = {
    thought: "#faad14",     // 黄色 - 思考
    action: "#1890ff",      // 蓝色 - 工具调用
    observation: "#52c41a", // 绿色 - 结果
    final: "#52c41a",       // 绿色 - 最终答案
    error: "#cf1322",       // 红色 - 错误
  };

  // 步骤类型对应的显示文字
  const labelMap = {
    thought: "思考",
    action: "工具",
    observation: "结果",
    final: "答案",
    error: "错误",
  };

  const color = colorMap[step.type] || "#666";
  const label = labelMap[step.type] || "步骤";

  return (
    <div style={{ marginBottom: 4 }}>
      {/* 步骤名称：带颜色 */}
      <span style={{ color, fontWeight: 500, marginRight: 8 }}>
        {label}：
      </span>

      {/* 步骤内容：根据类型显示不同内容 */}
      <span style={{ color: "#333" }}>
        {step.type === "action" && (
          <>
            {step.tool}
            {step.params && (
              <span style={{ color: "#999", marginLeft: 8 }}>
                参数：{JSON.stringify(step.params)}
              </span>
            )}
          </>
        )}
        {step.type === "observation" && (
          <>{typeof step.result === "string" ? step.result : JSON.stringify(step.result)}</>
        )}
        {step.type === "thought" && step.content}
        {step.type === "final" && step.content}
        {step.type === "error" && step.content}
      </span>
    </div>
  );
};
```

### 7.1 流式接口调用方式说明

**问题**：前端使用流式接口是循环调用的吗？

**结论**：**不是循环调用**，而是**单次请求-响应模式**。

**技术实现**（sse.ts 第248-323行）：
```typescript
// 使用 fetch API + ReadableStream 实现流式接收
const response = await fetch(url, {
  method: "POST",
  body: JSON.stringify({
    messages: [{ role: "user", content: content }],
    stream: true,  // 告诉后端需要流式输出
  }),
});

const reader = response.body.getReader();
const decoder = new TextDecoder("utf-8");

// while循环读取流数据，直到.done为true
while (true) {
  const { done, value } = await reader.read();
  if (done) {
    // 数据接收完成，断开连接
    break;
  }
  // 处理接收到的数据块
  buffer += decoder.decode(value, { stream: true });
}
```

**工作流程**：
```
用户发送消息 → 建立SSE连接 → 流式接收数据 → 数据处理 → 连接断开
                                    ↓
                              每个数据块触发 onStep/onChunk 回调
```

**特点**：
- 一次请求建立一个SSE连接
- 数据以流的方式持续推送过来
- 接收完成后连接自动断开
- 不是循环重复调用同一个接口

#### 7.3.4 后端8种type类型的前端处理对照

根据小沈的分析文档，后端SSE接口发送8种type类型，前端在sse.ts中分别处理如下：

| 序号 | type类型 | 说明 | 前端处理文件位置 | 前端处理逻辑 |
|------|---------|------|-----------------|-------------|
| 1 | `start` | AI开始处理 | sse.ts 第481-500行 | 创建startStep对象，显示"🤔 AI正在思考..." |
| 2 | `thought` | 思考过程 | sse.ts 第502-510行 | 创建step对象，添加到executionSteps数组 |
| 3 | `action` | 执行动作 | sse.ts 第502-510行 | 创建step对象，添加到executionSteps数组 |
| 4 | `observation` | 执行结果 | sse.ts 第502-510行 | 创建step对象，添加到executionSteps数组 |
| 5 | `chunk` | 流式内容块 | sse.ts 第512-517行 | 拼接content到buffer，更新currentResponse |
| 6 | `final` | 最终答案 | sse.ts 第519-538行 | 调用onComplete回调，标记接收完成 |
| 7 | `error` | 错误信息 | sse.ts 第540-546行 | 调用onError回调，显示错误信息 |
| 8 | `interrupted` | 任务中断 | sse.ts 第548-553行 | 调用onComplete回调，标记中断 |

**详细处理逻辑**：

```typescript
// sse.ts 第480-554行 switch处理
switch (rawData.type) {
  case "start": {
    // 1. start - AI开始处理
    const startStep: ExecutionStep = {
      type: "start",
      content: "🤔 AI 正在思考...",
      timestamp: Date.now(),
      model: rawData.model,
      provider: rawData.provider,
      display_name: displayName,
    };
    setExecutionSteps((prev) => [...prev, startStep]);
    onStep?.(startStep);
    break;
  }

  case "thought":
  case "action":
  case "observation": {
    // 2/3/4 - 思考/动作/观察：统一处理
    step.contentStart = responseBufferRef.current.length;
    step.contentEnd = step.contentStart;
    setExecutionSteps((prev) => [...prev, step]);
    onStep?.(step);
    break;
  }

  case "chunk": {
    // 5. chunk - 流式内容块
    responseBufferRef.current += rawData.content || "";
    setCurrentResponse(responseBufferRef.current);
    onChunk?.(rawData.content || "");
    break;
  }

  case "final": {
    // 6. final - 最终答案
    if (step.content) {
      if (!responseBufferRef.current) {
        responseBufferRef.current = step.content;
        setCurrentResponse(responseBufferRef.current);
        onChunk?.(step.content);
      }
    }
    onComplete?.(responseBufferRef.current, { model, provider, displayName });
    setIsReceiving(false);
    setIsConnected(false);
    break;
  }

  case "error": {
    // 7. error - 错误信息
    const errorMsg = rawData.content || rawData.error || "未知错误";
    onError?.(errorMsg);
    setIsReceiving(false);
    setIsConnected(false);
    break;
  }

  case "interrupted": {
    // 8. interrupted - 任务中断
    onComplete?.(responseBufferRef.current, undefined);
    setIsReceiving(false);
    setIsConnected(false);
    break;
  }
}
```

**关键问题**：根据小沈的分析，`type=observation`（第4类）的处理存在以下问题：
- 前端使用了 `rawData.observation` 作为 `result`，但后端发的是 `result` 字段
- 应该使用 `result` 字段（已简化），而不是 `observation`（原始JSON对象）

###### 字段1: `type` （P0-核心）

| 属性 | 说明 |
|------|------|
| **字段名称** | type |
| **数据类型** | string |
| **取值范围** | `"start" \| "thought" \| "action" \| "observation" \| "chunk" \| "final" \| "error" \| "interrupted"` |
| **来源** | 后端发送 |
| **出现阶段** | 所有阶段 |

**内涵定义**：
- `type`是消息步骤的类型标识符，用于区分当前这条消息在整个AI执行流程中所处的阶段和性质
- 它是ExecutionStep中最核心的字段，决定了前端如何渲染该步骤以及如何处理其包含的数据

**外延说明**：

| type值 | 含义 | 典型场景 | 前端渲染方式 |
|-------|------|---------|-------------|
| `start` | AI开始处理任务 | 用户发送消息后 | 显示"正在思考"提示 |
| `thought` | 推理/思考过程 | Agent分析问题时 | 显示思考内容（可折叠） |
| `action` | 执行动作 | Agent调用工具时 | 显示动作名称+参数 |
| `observation` | 执行结果 | 工具返回结果后 | 显示结果内容 |
| `chunk` | 流式内容块 | 最终答案输出时 | 逐字拼接显示 |
| `final` | 最终答案 | 任务完成时 | 显示完整答案 |
| `error` | 错误信息 | 执行失败时 | 显示错误详情 |
| `interrupted` | 任务中断 | 用户主动中断时 | 显示中断提示 |

**与其他字段的关系**：
- `type`决定了哪些其他字段会存在
- 例如：`type=action`时必有`action`和`action_input`字段；`type=observation`时必有`result`字段

**常见误区**：
- ❌ 不要把`thought`误认为是最终答案——它是中间推理过程
- ❌ 不要把`action`误认为是结果——它只是动作名称
- ❌ `chunk`和`final`都包含内容，但`chunk`是累积的中间状态，`final`是最终状态

###### 字段2: `step` （P0-核心）

| 属性 | 说明 |
|------|------|
| **字段名称** | step |
| **数据类型** | number |
| **取值范围** | 正整数（1, 2, 3, ...） |
| **来源** | 后端发送（type=action/observation时） |
| **出现阶段** | type=action, observation |

**内涵定义**：
- `step`是执行步骤的序号，从1开始递增，用于标识当前操作是第几步
- 它帮助用户理解AI执行任务的完整流程和进度
- 与Agent内部的step计数器对应，每执行一个动作+观察循环，step+1

**外延说明**：
- **用途1**：显示执行进度，例如"第3步（共5步）"
- **用途2**：帮助用户追踪AI的完整思考-执行链条
- **用途3**：用于日志和调试，定位特定步骤的问题

**与其他字段的关系**：
- `step`与`action`、`observation`是强关联的
- 通常：`step=N`对应一个`action`（第N步动作）和紧随其后的`observation`（第N步结果）
- `thought`可能没有step（因为thought是推理过程，不一定对应具体步骤）

**常见误区**：
- ❌ 不要认为每个`thought`都有step——thought是思考，不是具体动作
- ❌ step是后端维护的，前端不应该修改

###### 字段3: `thought` （P0-核心）

| 属性 | 说明 |
|------|------|
| **字段名称** | thought |
| **数据类型** | string |
| **取值范围** | 任意字符串 |
| **来源** | 后端发送（type=observation时） |
| **出现阶段** | type=observation |

**内涵定义**：
- `thought`是Agent在执行某个动作之前的推理过程，是Agent的"思考"
- 它解释了"为什么我要执行这个动作"
- 这是ReAct（Reasoning + Acting）模式中的"Reasoning"部分
- 类似于人类在执行操作前的思考和决策过程

**外延说明**：
- **典型内容**：例如"我需要先读取这个文件来了解其内容"、"让我分析一下这个错误的原因"
- **作用**：让用户理解AI的决策过程，增加透明度和可解释性
- **显示方式**：通常显示在对应`action`之前或作为上下文

**与action的关系**：
- `thought` → `action`：先有思考，决定要做什么
- `thought` → `action` → `observation`：完整的一个执行循环
- 一个`observation`通常包含一个`thought`（为什么做）和一个`action`（做了什么）

**与content的区别**：
- `thought`：Agent的推理过程（为什么做）
- `content`：消息的实际文本内容（可能是thought的内容，也可能是其他）

**常见误区**：
- ❌ 不要把`thought`和`action`混淆——thought是"为什么"，action是"做什么"
- ❌ 不要把`thought`和`result`混淆——thought是推理，结果是observation

| 属性 | 说明 |

| **字段名称** | action |

**内涵定义**：
- `action`是Agent要执行的工具或操作的名称
- 它回答"AI要做什么"这个问题
- 这是ReAct模式中的"Acting"部分

**外延说明**：

### 7.4 字段处理决策原则

#### 7.4.1 字段处理核心原则

**基于我们确定的原则**：

1. **字段名一致性原则**
   - 前端接收接口的字段名称与后端返回的字段名称和含义完全一致
   - 不能出现字段名差异，导致调试困难

2. **后端优先处理原则**
   - 对相同字段的信息处理，坚持"能够后端处理就后端处理"
   - 后端最清楚数据来源，可以确保字段命名准确

3. **前端处理场景**
   - 如果后端处理不利于前端渲染，由前端处理
   - 或者后端处理一部分，前端处理一部分

4. **目的**
   - 减少前端字段映射逻辑
   - 所有前端（包括移动端）都能直接使用正确字段
   - 便于前后端调试对照

#### 7.4.2 采用方案

**采用方案**：后端处理为主，前端处理为辅

**原因**：
1. **准确性高**：后端最清楚数据来源
2. **减少前端负担**：前端不需要做复杂的字段映射逻辑
3. **统一规范**：所有前端（包括移动端）都能直接使用正确字段

#### 7.4.3 对后端的需求（精准版）

**核心原则**：前后端字段名和含义完全一致，不能出现差异。

**需求确认**：根据字段对照分析，后端当前发送的字段已经与前端需求一致，只需确认以下几点：

| 序号 | 需求项 | 现状确认 | 动作 |
|------|--------|---------|------|

#### 问题2：result vs observation区别（P0）

**小沈回复**：

| 字段 | 类型 | 含义 | 获取方式 |
|------|------|------|---------|
| observation | object | **原始结果**，可能包含嵌套JSON | rawData.observation |
| result | string | **处理后的文本**，人类可读 | rawData.result |

**生成过程**：
```
observation（原始对象） → simplify_observation()函数 → result（处理后的文本）
```

**前端应该使用**：result字段显示给用户 ❗

**与前端期望是否一致**：✅ 一致（前端已意识到应该用result）

#### 问题3：action vs tool关系（P1）

**小沈回复**：

| 字段 | 含义 | 关系 |
|------|------|------|
| action | Agent执行的动作名称 | 主字段 |
| tool | action的**兼容字段**，两者值相同 | 兼容字段 |

**代码体现**（chat.py:723-737）：
```python
step_data = {
    'action': step_action,      # 主字段
    'tool': step_action,         # 兼容字段，值相同
    'result': step_result        # 处理后的文本
}
```

**前端应该使用**：action字段（主字段），tool字段已不推荐

**与前端期望是否一致**：✅ 一致

#### 问题4：content在不同type中的含义（P1）

**小沈回复**：

详细说明见小沈文档1.3.5.2章节，这里简述：

| Type | content含义 | 类型 |
|------|------------|------|
| thought | 提示信息（固定文本） | 静态文本 |
| action | 动作描述 | 静态文本 |
| observation | = thought（兼容字段） | 静态文本 |
| chunk | **AI回复片段**（会累积） | 动态内容 |
| final | **AI最终回复** | 动态内容 |
| error | 错误信息 | 静态文本 |
| interrupted | 中断信息 | 静态文本 |

**⚠️ 重要**：content在chunk/final中是**需要拼接的AI回复内容**，在其他type中是**直接显示的描述文本**

**与前端期望是否一致**：✅ 一致（前端已意识到需要根据type判断）

#### 问题5：chunk中is_reasoning字段（P2）

**小沈回复**：

⚠️ **需要进一步讨论**

**问题背景**：
- 当前chunk事件只返回content，没有区分是"思考过程"还是"最终答案"
- 小新希望能够区分LLM的reasoning_content（思考）和content（答案）

**当前状态**：
- 后端**没有**区分reasoning和answer的逻辑
- 这需要后端在收到LLM响应时，解析OpenAI的`reasoning_content`字段（如果有）

**需要讨论的问题**：
1. 是否需要后端解析LLM的reasoning_content字段？
2. 如果需要，是新增一个type（如reasoning）还是扩展chunk？
3. 前端显示时如何区分思考和答案（两个区域显示？）

**建议**：如果LLM支持reasoning_content（如o1、o3系列），可以新增type来区分

#### 问题6：暂停/继续API（P2）

**小沈回复**：

❌ **暂不实现**

**原因**：
1. 当前LLM流式API（如OpenAI）**不支持真正的暂停**
2. 暂停只是前端UI状态切换，后端无法真正暂停LLM调用
3. 如果用户需要"取消"当前任务，可以使用现有的**中断功能**（点击停止按钮）

**当前可用的替代方案**：
- 用户点击"停止"按钮 → 调用 `/chat/stream/cancel/{task_id}` → 后端中断任务

**如果未来要实现**：
- 需要LLM提供商支持pause/resume API
- 实现成本较高，优先级较低
