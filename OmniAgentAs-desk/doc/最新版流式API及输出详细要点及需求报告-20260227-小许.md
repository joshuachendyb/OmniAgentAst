## 三、后端及API功能需求

### 3.1 流式API接口

#### API-STREAM-001：流式消息发送
- **接口路径**: POST /api/v1/chat/stream
- **功能描述**: 发送流式消息请求，返回SSE格式响应
- **请求格式**:
  - Content-Type: application/json
  - Body:
    ```json
    {
      "messages": [
        {
          "role": "user",
          "content": "消息内容"
        }
      ],
      "stream": true,
      "task_id": "可选，前端生成的任务ID（UUID格式）"
    }
    ```
- **请求参数验证**:
  - messages不能为空
  - messages每个对象必须有role和content字段
  - task_id必须是有效的UUID格式（如果提供）
- **响应格式**: SSE（Server-Sent Events）流式响应
  - media-type: text/event-stream
  - 事件类型:
    - `start`: 开始事件，返回task_id
    - `thought`: 思考过程，包含思考内容
    - `action`: 工具调用，包含工具名称和参数
    - `observation`: 观察结果，包含工具执行结果
    - `chunk`: 内容片段，逐token返回
    - `final`: 最终结果，包含完成状态
    - `interrupted`: 任务中断通知
    - `error`: 错误信息，包含错误详情
- **业务逻辑**:
  1. 验证请求参数
  2. 生成或使用前端提供的taskId
  3. 注册任务到running_tasks
  4. 处理消息并生成流式响应
  5. 清理任务状态
- **来源文档**: 《会话流式API及输出功能需求详细说明-小查-20260227.md》
- **优先级**: P1-高
- **实现状态**: 已实现

#### API-STREAM-002：任务中断
- **接口路径**: POST /api/v1/chat/stream/cancel/{task_id}
- **功能描述**: 取消正在执行的流式任务
- **路径参数**:
  - task_id（必填）：任务ID（UUID格式）
- **请求格式**:
  - Content-Type: application/json
- **响应格式**:
  ```json
  {
    "success": true,
    "message": "任务已取消"
  }
  ```
- **业务逻辑**:
  1. 验证task_id格式
  2. 检查任务是否存在于running_tasks
  3. 设置任务的取消标志
  4. 任务在下一次检查时检测到取消标志并停止
- **错误处理**:
  - 404：任务不存在
  - 400：task_id格式无效
- **线程安全**:
  - 使用asyncio.Lock保护running_tasks的访问
  - 所有访问running_tasks的地方都需要加锁
- **来源文档**: 《会话流式API及输出功能需求详细说明-小查-20260227.md》
- **优先级**: P1-高
- **实现状态**: 已实现

### 3.2 会话管理API

#### API-SESSION-001：获取会话消息
- **接口路径**: GET /api/v1/sessions/{session_id}/messages
- **功能描述**: 获取会话的消息列表和详细信息
- **路径参数**:
  - session_id（必填）：会话ID
- **响应格式**:
  ```json
  {
    "messages": [
      {
        "id": "message_id",
        "role": "user",
        "content": "消息内容",
        "created_at": "2026-02-27 23:18:45"
      }
    ],
    "title": "会话标题",
    "version": 1,
    "title_locked": false,
    "title_source": "auto"
  }
  ```
- **响应字段说明**:
  - messages: 消息数组，按创建时间升序排列
    - id: 消息ID
  - role: 消息角色（user/assistant）
  - content: 消息内容
  - created_at: 创建时间
  - title: 会话标题
  - version: 乐观锁版本号
  - title_locked: 标题锁定状态
  - title_source: 标题来源（user=用户手动设置，auto=自动生成）
- **业务逻辑**:
  1. 查询会话信息
  2. 查询会话所有消息
  3. 返回完整数据
- **错误处理**:
  - 404：会话不存在
- **来源文档**: 《前端功能深度分析与问题修复报告-2026-02-26.md》
- **优先级**: P1-高
- **实现状态**: 已实现

**结论**: 不存在功能冲突，所有已识别的问题已在2026-02-26完成修复。

## 五、交互流程需求

### 5.1 流式消息发送流程

**流程描述**:
1. 用户在输入框输入消息
2. 用户点击发送按钮
3. 前端生成唯一taskId（UUID格式）
4. 前端设置currentTaskId为taskId
5. 前端发送POST请求到/api/v1/chat/stream
6. 请求参数包含messages、stream=true、task_id
7. 后端接收请求，验证参数
8. 后端返回start事件（包含task_id）
9. 后端处理消息，返回thought/action/observation事件
10. 后端返回chunk事件（逐token）
11. 前端接收chunk事件，逐token显示内容
12. 前端显示光标提示
13. 前端展开思考过程
14. 后端返回final事件，表示完成
15. 前端清理流式状态

**状态流转**:
- 空闲 → 发送中 → 流式接收中 → 完成
- currentTaskId：null → taskId → null
- isStreaming：false →开始true →完成false

**关键界面元素**:
- 消息输入框
- 发送按钮
- AI消息气泡（显示流式内容）
- 思考过程折叠面板
- 光标提示

**来源文档**: 《会话流式API及输出功能需求详细说明-小查-20260227.md》

### 5.2 任务中断流程

**流程描述**:
1. 用户点击但中断按钮
2. 前端获取当前taskId（serverTaskId || currentTaskId）
3. 前端发送POST请求到/api/v1/chat/stream/cancel/{taskId}
4. 后端接收请求，验证taskId
5. 后端设置任务取消标志
6. 后端在执行步骤中检查cancelled状态
7. 如果检测到取消，抛出CancelledError
8. 后端返回interrupted事件
9. 前端接收interrupted事件
10. 前端显示中断提示
11. 前端停止流式显示
12. 前端清理taskId

**状态流转**:
- 流式接收中 → 中断中 → 已中断
- currentTaskId：taskId → null
- isStreaming：true → false

**关键界面元素**:
- 中断按钮（红色危险样式）
- 任务加载状态指示器
- 中断提示消息

**来源文档**: 《会话流式API及输出功能需求详细说明-小查-20260227.md》

### 5.3 会话标题更新流程

**流程描述**:
1. 新会话创建，自动生成智能标题
2. 前端显示标题为"X月X日 [时间段]会话 HH:MM"
3. 用户点击标题编辑按钮
4. 用户修改标题内容
5. 前端调用PUT请求到/api/v1/sessions/{session_id}
6. 请求参数包含title和version
7. 后端执行乐观锁检查
8. 如果版本一致，更新标题和版本号
9. 后端返回新的版本号
10. 前端更新本地version
11. 前端显示保存成功图标
12. 如果版本冲突，显示冲突提示

**状态流转**:
- 标题状态：自动生成 → 用户修改 → 保存中 → 已保存
- titleLocked：false（可自动更新）或true（用户锁定）
- version：当前版本号

**关键界面元素**:
- 会话标题显示
- 标题编辑按钮
- 锁定/解锁图标
- 保存状态图标（加载/成功）

**来源文档**: 《前端功能深度分析与问题修复报告-2026-02-26.md》
