### 问题1：🔴 乐观锁并发安全问题（P0-紧急）

**位置**: `sessions.py` 第696-721行，`update_session()`函数

**问题描述**:
```python
# 当前实现
cursor.execute(
    '''SELECT id, title, version, COALESCE(title_locked, 0) as title_locked
       FROM chat_sessions 
       WHERE id = ? AND is_deleted = FALSE AND version = ?''',
    (session_id, update_data.version)
)
session = cursor.fetchone()

# ... 后续UPDATE操作
cursor.execute(f'UPDATE chat_sessions SET ... WHERE id = ?', ...)
```

**问题根因**:
1. **SELECT和UPDATE不是原子的**：SQLite不支持`SELECT ... FOR UPDATE`，当前的"检查再执行"模式不是原子的
2. **竞态条件风险**：两个并发请求可能同时通过version检查，然后都执行UPDATE，导致丢失更新
3. **缺乏事务保护**：没有显式开启事务，SELECT和UPDATE操作不在同一个事务中

**影响范围**:
- 所有并发更新标题的场景
- 可能导致标题更新丢失或版本号不一致
- 数据一致性风险

**修复建议**:
```python
# 修复方案：使用显式事务和原子性UPDATE
@router.put("/sessions/{session_id}")
async def update_session(session_id: str, update_data: SessionUpdate):
    try:
        conn = _get_db_connection()
        cursor = conn.cursor()
        
        # ⭐ 修复：显式开启事务
        cursor.execute("BEGIN")
        
        fields_exist = check_db_fields_exist(conn)
        utc_time = get_utc_timestamp()
        
        if fields_exist['version']:
            if update_data.version is not None:
                # ⭐ 修复：原子性UPDATE（避免SELECT）
                # 尝试直接更新，检查受影响行数
                cursor.execute(
                    '''UPDATE chat_sessions 
                       SET title = ?, updated_at = ?, 
                           title_locked = ?, 
                           title_updated_at = ?, 
                           version = version + 1
                       WHERE id = ? AND is_deleted = FALSE AND version = ?''',
                    (update_data.title, utc_time, 1, utc_time, 
                     session_id, update_data.version)
                )
                
                # 检查是否更新成功（乐观锁验证）
                if cursor.rowcount == 0:
                    # ⭐ 修复：版本冲突
                    conn.rollback()
                    conn.close()
                    raise HTTPException(
                        status_code=409,
                        detail="会话已被其他用户修改，请刷新后重试"
                    )
                
                # 更新成功，获取更新后的数据
                cursor.execute(
                    '''SELECT id, title, version FROM chat_sessions WHERE id = ?''',
                    (session_id,)
                )
                session = cursor.fetchone()
            else:
                # 兼容模式：直接更新
                cursor.execute(
                    '''UPDATE chat_sessions 
                       SET title = ?, updated_at = ?,
                           title_locked = ?,
                           title_updated_at = ?
                       WHERE id = ?''',
                    (update_data.title, utc_time, 1, utc_time, session_id)
                )
                cursor.execute(
                    '''SELECT id, title, COALESCE(version, 1) as version FROM chat_sessions WHERE id = ?''',
                    (session_id,)
                )
                session = cursor.fetchone()
```

**优先级**: P0-紧急  
**预计修复时间**: 2小时

### 问题2：🔴 事务边界不完整（P0-紧急）

**位置**: `sessions.py` 第666-803行，`update_session()`函数

**问题描述**:
```python
# 当前代码
cursor.execute(f'UPDATE chat_sessions SET ... WHERE id = ?', update_values)

# 插入标题历史记录（在UPDATE之后）
cursor.execute(
    '''INSERT INTO chat_session_title_history ...''',
    (...)
)

conn.commit()  # ⚠️ 问题：没有显式开启事务
```

**问题根因**:
1. **缺乏显式事务**：没有`BEGIN`语句，依赖SQLite的自动事务
2. **部分更新风险**：如果历史记录插入失败，UPDATE已经提交了
3. **数据不一致风险**：会话表更新成功，但历史表记录失败

**修复建议**:
```python
# 修复方案：显式事务和错误处理
try:
    conn = _get_db_connection()
    cursor = conn.cursor()
    
    # ⭐ 修复：显式开启事务
    cursor.execute("BEGIN")
    
    # ... 执行UPDATE
    
    # 插入标题历史记录
    cursor.execute(
        '''INSERT INTO chat_session_title_history ...''',
        (...)
    )
    
    # ⭐ 修复：提交事务
    cursor.execute("COMMIT")
    conn.close()
    
except Exception as e:
    # ⭐ 修复：回滚事务
    try:
        cursor.execute("ROLLBACK")
    except:
        pass
    conn.close()
    raise HTTPException(status_code=500, detail=f"更新会话失败: {str(e)}")
```

**优先级**: P0-紧急  
**预计修复时间**: 30分钟

if fields_exist['version']:

### 问题5：🟡 向后兼容性字段检查错误（P1-高）

**位置**: `sessions.py` 第32-73行，`check_db_fields_exist()`函数

**问题描述**:
```python
# 问题：PRAGMA table_info返回的row['name']是Unicode字符串
cursor.execute("PRAGMA table_info(chat_sessions)")
columns = {row['name'] for row in cursor.fetchall()}

# 问题：新字段可能不存在，但代码认为存在

```

1. **Unicode字符串比较问题**：row['name']可能包含不可见字符
2. **大写敏感问题**：某些数据库配置可能影响大小写
3. **默认值错误**：新字段不存在时应该返回False，而不是假设存在

**修复建议**:
```python
# 修复方案：更健壮的字段检查
def check_db_fields_exist(conn) -> dict:
    cursor = conn.cursor()
    fields_exist = {
        'title_locked': False,
        'title_updated_at': False,
        'version': False
    }
    
    try:
        cursor.execute("PRAGMA table_info(chat_sessions)")
        rows = cursor.fetchall()
        
        # ⭐ 修复：标准化字段名
        columns = {str(row['name']).strip().lower() for row in rows}
        
        # ⭐ 修复：使用标准化比较
        fields_exist['title_locked'] = 'title_locked' in columns
        fields_exist['title_updated_at'] = 'title_updated_at' in columns
        fields_exist['version'] = 'version' in columns
        
        # ... 其余代码不变
```

**优先级**: P1-高

CREATE TABLE IF NOT EXISTS chat_session_title_history (

    session_id TEXT NOT NULL,  -- ⭐ 修复：正确语法
    title TEXT NOT NULL,     -- ⭐ 修复：正确语法
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_by TEXT,
    change_reason TEXT,
    FOREIGN KEY (session_id) REFERENCES chat_sessions(id) ON DELETE CASCADE
);
