### 0.2 需求理解

#### 0.2.1 重构原则

在现有系统已实现的模型配置使用流程和方法的基础上，增补或修改导致前后端模型使用不一致的问题。

#### 0.2.2 模型配置文件结构

配置文件支持多个provider和同一provider下多个model。

```yaml
ai:
  provider: opencode
  model: kimi-k2.5-free   # 当前使用的模型
  opencode:
    models:
      - minimax-m2.5-free
      - glm-5-free
      - kimi-k2.5-free
```

"模型"指 `provider+model` 的组合。

#### 0.2.3 当前模型（Current Model）

- `ai.provider` 和 `ai.model` 是当前选定的使用模型（current model）
- 其他模型为候选模型
- 如果配置文件中没有明确设置 `ai.provider` 和 `ai.model` 的值，选择模型列表的第一个模型作为当前选定使用模型

#### 0.2.4 模型数组字段

后端从配置文件读取后构建模型数组，建议字段包括：

| 字段 | 说明 |
|------|------|
| 序号 | 模型序号 |
| provider | provider名称 |
| model | 模型名称 |
| display_name | 显示名称 |
| current_model | 是否为当前模型 |

#### 0.2.5 用户端显示位置

模型下拉列表显示在标题栏，具体要求：

- 标题栏标题名称右侧显示当前使用模型
- 用颜色显示模型的可用状态（沿用现在的设计和实现方法）
- 标题栏头像左侧显示模型列表的下拉框

#### 0.2.6 系统初次启动流程

1. Web页面 `current_model` 位置显示"未配置"
2. 客户端（web页面和未来的APP页面）每次启动，从服务端获取模型列表，构建下拉框列表
3. 下拉列表的模型顺序必须与配置文件中的模型列表顺序完全一致
4. 客户端自动启动"检查服务"功能

#### 0.2.7 检查服务功能

- 调用后端的"检查服务"对应功能
- 后端取 `ai.provider` 和 `ai.model` 进行检查（采用现在的检查方法和逻辑)
- 检查完成后返回客户端：
  - `current_model` 是否可用
  - `current_model` 的 `display_name`

前端显示处理（沿用现在的显示逻辑），后端返回的检查信息分为3类（需要与前端开发小新确定，实际与小新的设计要一致）：

| 类型 | 说明 | 显示 |
|------|------|------|
| error | 完全不可用（网络不通、模型API错等） | 红色字体，提醒用户检查配置文件或更换模型 |
| warning | 能够联通模型，但可能有限速、费用或信用分不够 | 黄色色字体 |
| 正常 | 正常 | 绿色字体 |

#### 0.2.8 用户选择其他模型流程

客户端支持用户从列表中选择新模型的 `display_name`。

逻辑流程：
1. 从模型下拉框选择选定模型后，发送这个模型信息给后端
2. 让后端检查这个选定的模型是不是可用
3. 如果可用：
   - 后端更新 `ai.provider` 和 `ai.model` 的值
   - 反馈的后端：在前端用新模型名称信息替换上一次的模型名称信息
4. 否则：
   - 提示新模型信息有问题（沿用现在的处理逻辑）

#### 0.2.9 保存和使用当前模型

客户端退出时，web退出后要保存最后一次选择的可用模型。下一次启动时，从配置文件读取并使用上次最后可用的模型进行检查服务，并显示这个模型的可用状态信息。

说明：前端不用特别保存当前可用模型，因为这个当前可用模型都在模型配置文件的 `ai.provider` 和 `ai.model` 参数中（如果模型配置文件假定都是正常的情况下）。

#### 0.2.10 AI对话中的模型使用

AI对话中，使用前端选择的也应该是模型配置文件中的 `ai.provider` 和 `ai.model` 进行大模型调用。

然后在每一次AI对话中携带 `ai.provider` 和 `ai.model` 信息组装成 `display_name` 反馈给前端，前端在每一次对话中在"AI助手"后显示这个 `display_name`。

#### 0.3 需求分析

基于0.1原始需求和0.2需求理解，对现有系统进行深度分析，提出重构方案。

**0.3.1 现有系统现状分析**

**后端现状**（backend/app/api/v1/config.py）：

| 问题点 | 现状 | 影响 |
|--------|------|------|
| ModelInfo字段 | 只有id、provider、model、display_name | 缺少current_model字段 |
| get_model_list | 只返回models数组和default_provider | 无法标记当前模型 |
| update_config | 只更新ai.provider | 没有更新ai.model |
| 读取配置 | 只读取ai.{provider}.model | 没有读取顶层ai.model |

**前端现状**（frontend/src/components/Layout/index.tsx）：

| 问题点 | 现状 | 影响 |
|--------|------|------|
| _currentProvider | 字符串格式"provider-modelname" | 无法准确记录provider和model |
| 初始化匹配 | 用config.ai_provider和config.ai_model匹配 | config.ai_model可能为空 |
| 切换模型 | 只传provider给updateConfig | 后端无法知道具体模型 |

**配置文件现状**（config/config.yaml）：

| 问题点 | 现状 | 影响 |
|--------|------|------|
| ai.provider | 有值（opencode） | 只记录了provider |
| ai.model | 无 | 无法记录当前使用的具体模型 |

**0.3.2 根本问题分析**

**核心问题**：配置文件缺少顶层`ai.model`字段

导致的问题链：
```
配置文件缺少 ai.model
  ↓
后端 get_model_list 无法标记当前模型
  ↓
前端初始化时匹配不到正确的模型
  ↓
用户切换模型时，后端只更新了provider，没有更新model
  ↓
后端读取模型时，用ai.{provider}.model（默认模型）而不是用户选择的模型
  ↓
前后端模型使用不一致
```

**0.3.3 重构方案**

**设计原则**：
1. 最小化前端修改量
2. 后端承担主要修改
3. 保持现有使用方式和显示格式
4. 向后兼容（优先读取ai.model，fallback到provider默认模型）

**重构方案**：

**修改点1：配置文件增加ai.model字段**

**修改点2：后端get_model_list增加current_model标记**

```python
class ModelInfo(BaseModel):
    id: int
    provider: str
    model: str
    display_name: str
    current_model: bool = False  # 【新增】是否为当前模型

@router.get("/config/models")
async def get_model_list():
    config = get_config_instance()
    ai_config = config.get("ai", {})
    
    # 【新增】读取当前模型
    current_provider = ai_config.get("provider", "")
    current_model = ai_config.get("model", "")  # 【新增】
    
    models = []
    model_id = 1
    
    for provider in SUPPORTED_PROVIDERS:
        provider_config = ai_config.get(provider, {})
        models_list = provider_config.get("models", [])
        
        for model_name in models_list:
            models.append(ModelInfo(
                id=model_id,
                provider=provider,
                model=model_name,
                display_name=f"{provider_display_name} ({model_name})",
                current_model=(  # 【新增】标记当前模型
                    provider == current_provider and 
                    model_name == current_model
                )
            ))
            model_id += 1
    
    return {"models": models}
```

**修改点3：后端update_config同时更新ai.model**

```python
@router.put("/config")
async def update_config(config_update: ConfigUpdate):
    config_path = _get_config_path()
    
    with open(config_path, 'r+', encoding='utf-8') as f:
        config_data = yaml.safe_load(f) or {}
    
    # 更新 ai.provider
    if config_update.ai_provider:
        config_data['ai']['provider'] = config_update.ai_provider
    
    # 【新增】更新 ai.model
    if config_update.ai_model:
        config_data['ai']['model'] = config_update.ai_model
    
    # 保存配置
    with open(config_path, 'w', encoding='utf-8') as f:
        yaml.dump(config_data, f, allow_unicode=True, default_flow_style=False)
    
    # 重新加载
    config = get_config_instance()
    config.reload()
    
    return {"success": True, ...}
```

## 三、问题根本原因

### 3.1 配置文件缺少 `ai.model`

**现状**：
- 配置文件只有 `ai.provider: opencode`
- 没有 `ai.model` 字段
- 后端 `getConfig()` 返回的 `ai_model` 是空字符串

**导致的问题**：
```
用户选择 "OpenCode (kimi-k2.5-free)"
  ↓
前端更新：_currentProvider = "opencode-kimi-k2.5-free"
  ↓
前端调用 updateConfig，但只传了 ai_provider
  ↓
后端只更新 config['ai']['provider'] = "opencode"
  ↓
后端无法知道用户选择的是哪个具体模型
  ↓
用户以为用了 kimi-k2.5-free，但后端不知道！
```

### 3.2 前后端数据不一致

| 位置 | 当前模型 | 格式 |
|------|---------|------|
| 前端状态 | `"opencode-kimi-k2.5-free"` | `"provider-modelname"` |
| 后端读取 | `""`（空字符串） | 无 |
| 配置文件 | 无 | 缺少 ai.model |

## 四、设计方案（保持现有使用方式）

### 4.1 配置文件修改

**增加顶层 `ai.model` 字段**：

```yaml
ai:
  provider: opencode
  model: kimi-k2.5-free   # 【新增】当前使用的模型
  opencode:
    models:
      - minimax-m2.5-free
      - glm-5-free
      - kimi-k2.5-free
```

**字段含义**：
| 字段 | 含义 | 何时修改 |
|------|------|---------|
