**前端调用详细代码**:
```typescript
// frontend/src/utils/sse.ts 第248行
const url = `${config.baseURL}/chat/stream`;

// 实际发送代码 (sse.ts:248-290)
const sendMessage = useCallback(
  async (content: string) => {
    // 1. 创建 AbortController 用于取消请求
    const controller = new AbortController();
    
    // 2. 发送POST请求
    const response = await fetch(url, {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({
        messages: [{ role: "user", content: content }],
        stream: true,
        task_id: taskId || undefined,  // 用于中断追踪
      }),
      signal: controller.signal,  // 绑定取消信号
    });
    
    // 3. 使用 ReadableStream 读取流式数据
    const reader = response.body?.getReader();
    const decoder = new TextDecoder();
    
    // 4. 循环读取并解析SSE数据
    while (true) {
      const { done, value } = await reader.read();
      if (done) break;
      const text = decoder.decode(value);
      // 按行分割，处理 "data: {...}" 格式
      const lines = text.split('\n');
      for (const line of lines) {
        if (line.startsWith('data: ')) {
          const data = JSON.parse(line.slice(6));
          // 5. 根据type调用不同回调
          switch (data.type) {
            case 'start': onStart?.(data); break;
            case 'thought': onStep?.(data); break;
            case 'action': onStep?.(data); break;
            case 'observation': onStep?.(data); break;
            case 'chunk': onChunk?.(data); break;
            case 'final': onComplete?.(data); break;
            case 'error': onError?.(data); break;
            case 'interrupted': onComplete?.(data); break;
          }
        }
      }
    }
  },
  [url, taskId, config, callbacks]
);
```

**前端回调处理**:
- `onStart` - 接收start事件，显示AI名称
- `onStep` - 接收thought/action/observation事件，显示执行步骤
- `onChunk` - 接收chunk事件，流式显示AI回复
- `onComplete` - 接收final/interrupted事件，任务完成
- `onError` - 接收error事件，显示错误

**中断机制**: 前端通过 `controller.abort()` 发送取消信号，后端通过 `check_and_yield_if_interrupted()` 检查并返回interrupted事件

**使用现状**: 前端主流程正在使用，是本次分析的重点

   - chunk用于普通对话，体验好但前端处理复杂
   - 文件操作不需要chunk，因为是步骤式执行，最后一次性返回结果

**补充说明：中断机制（界面按钮和SSE中断是同一个机制）**

前端中断按钮和后端SSE中断是**同一个机制**，流程如下：
```
用户点击"停止"按钮
       ↓
前端调用 /chat/stream/cancel/{task_id} 接口
       ↓
后端将该task_id标记为 cancelled=True
       ↓
前端继续读取SSE流
       ↓
后端循环中 check_and_yield_if_interrupted() 检测到 cancelled
       ↓
发送 type=interrupted 事件
       ↓
前端收到 interrupted，停止显示，任务结束
```

**代码对应关系**：

| 环节 | 位置 | 代码 |

---

**字段1: type**

| 属性 | 说明 |
|------|------|
| 必返回 | 全部8种type |
| 含义 | 固定值，标识当前SSE事件的类型 |

```json

```

**字段2: content（⚠️ 重点：多含义字段）**

| 出现的Type | 含义 | 说明 |
|-----------|------|------|
| thought | AI分析提示 | 非实际推理内容，只是提示信息如"正在分析任务..." |
| action | 动作描述 | AI即将执行的动作描述，如"检测到文件操作意图..." |
| observation | **等同于thought** | 兼容字段，值与thought相同 |
| chunk | AI回复片段 | 逐token返回的最终回复内容片段 |
| final | 完整回复 | 完整的AI回复内容 |

| interrupted | 中断信息 | 中断原因描述 |

| Type | content含义 | 类型 |
|------|------------|------|
| thought | 提示信息 | 静态文本 |
| action | 动作描述 | 静态文本 |
| observation | **= thought** | 兼容字段 |
| chunk | **AI回复内容** | 动态内容（会累积） |
| final | **AI最终回复** | 动态内容 |
| error | 错误信息 | 静态文本 |
| interrupted | 中断信息 | 静态文本 |

**建议**：前端使用content时，需要根据type判断其含义：
- chunk/final：需要拼接的AI回复内容
- 其他type：直接显示的提示/描述文本

### 4.3 补充：非文件操作时关闭步骤UI的时机

#### 4.3.1 场景分析

**当前非File操作流程（chat.py 第780-858行）**：

```
1. start事件 → display_name
2. action事件 → content="正在调用AI服务..."（第782行）
3. chunk事件 → content="AI回复片段1"，content="片段2"...（循环，第836行）
4. final事件 → content="完整AI回复"（第853行）
```

**问题**：
- 非文件操作时，后端会发送action（"正在调用AI服务..."）和chunk（流式回复）
- 前端收到action后显示步骤面板，但收到chunk后没有关闭步骤面板
- 导致步骤和最终回复同时显示，界面混乱

#### 4.3.2 方案对比

**方案A：你的建议 - 用action代替chunk**

```
1. start事件 → display_name
2. action事件 → content="完整AI回复"（把AI回复放这里）
3. final事件 → content="完整AI回复"
```

| 优点 | 缺点 |
|------|------|
| 前端处理简单 | ❌ 失去流式显示效果 |
| 不需要拼接buffer | ❌ action语义改变 |
| | ❌ 用户体验下降 |

**方案B：收到chunk时关闭步骤UI（推荐）**

```
1. start事件 → display_name
2. action事件 → content="正在调用AI服务..."
3. chunk事件 → content="AI片段"（保留流式效果）
   → 同时关闭步骤UI ✅
4. final事件 → content="完整AI回复"
```

| 优点 | 缺点 |
|------|------|
| ✅ 保留流式效果 | 需要处理时序 |
| ✅ 用户体验好 | 需要添加关闭回调 |
| ✅ 步骤UI正确关闭 | |

**原因**：INSERT 语句只保存了 session_id, role, content, timestamp, display_name，没有保存 execution_steps

**现象**：步骤显示空白

2. 但 ReAct Agent 的 `thought` 字段可能为空
3. LLM 并不总是返回 thought
4. 导致 content 为空，前端显示空白

### 问题3：前端result获取错误

**现象**：清洗后的result文本没有正确获取

**原因**：前端获取的是 `rawData.observation` 而不是 `rawData.result`

### 问题4：LLM解析错误

**现象**：有时候能显示，有时候不能

**原因**：LLM 返回非JSON格式，解析失败，但代码有重试机制

## 八、综合处理方法

基于日志分析，总结以下处理方法：

### 8.1 content为空的处理

| 情况 | 处理方法 |
|------|---------|
| result.message默认值为空 | 使用observation.result.result提取 |
| observation为空 | 使用默认文本"（无结果）" |
| 多层嵌套都为空 | 回退到 thought 内容 |

### 8.2 result清洗规则（综合版）

| 工具/情况 | 原始 result | 清洗结果 | 显示示例 |
|-----------|------------|---------|---------|
| list_directory | `{"entries": [...], "total_count": 41}` | "📁 列出了 41 个文件/目录" | 列出了 41 个文件/目录 |
| read_file | 文件内容文本 | "📄 文件内容（前100字）" | 文件内容... |
| search_files | 搜索结果列表 | "🔍 找到 X 个结果" | 找到 5 个结果 |
| finish | `{"result": "回复文本"}` | 直接文本 | 回复文本 |
| 执行失败 | `{"success": false, "error": "..."}` | "❌ 错误信息" | ❌ 文件不存在 |
| 解析失败 | LLM返回非JSON | "⚠️ 解析失败，将重试..." | ⚠️ 解析失败 |
| 空数据 | 空或null | "（无结果）" | （无结果） |

### 8.3 LLM异常情况处理

| 情况 | 处理方法 | 显示 |
|------|---------|------|
| JSON解析失败 | 记录错误，重试 | "⚠️ AI思考中..." |
| 超时 | 返回超时错误 | "❌ 请求超时" |
| 网络错误 | 返回网络错误 | "❌ 网络错误" |
| token耗尽 | 返回配额错误 | "❌ API配额已用完" |
