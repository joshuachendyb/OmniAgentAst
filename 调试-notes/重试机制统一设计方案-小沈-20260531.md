## 一、回答用户3个核心问题

### 问题1：重试模块功能和流程

系统中有 **3种完全不同的失败模式**，需要 **3种完全不同的重试策略**：

| 失败类型 | 失败层 | 重试内容 | 重试策略 |
|---------|--------|---------|---------|
| **HTTP 429限流** | 网络层 | 重新发同一个HTTP请求 | 指数退避（1s→2s→4s），等服务端冷却 |
| **Parse格式错误** | 语义层 | 重新发一次LLM调用，prompt里加"格式不对请重来" | 指数退避（1s→2s→4s），最多3次 |
| **空响应** | 响应层 | 截断历史，重新发一次LLM调用 | 固定退避1s，最多2次 |

**为什么不能合并成一种重试**：这3种失败发生在不同的层，需要不同的修复手段：
- 429只需要**等一等再试**（HTTP层重试）
- Parse错误需要**改prompt重新问**（Agent循环重试）
- 空响应需要**截断历史重新问**（Agent循环重试，但策略不同）

### 问题2：在哪里调用，需要几个地方

**3个调用点，不可能更少。** 原因：3种失败发生在不同的层。

```
调用点1 ─── 第3层（LLM服务）─── HTTP 429重试
    ↑
调用点2 ─── 第2层（Agent循环）─── Parse重试 + 空响应重试
    ↑
调用点3 ─── 第1.5层（SSE会话）─── 网络异常/超时 重试整个流
```

| 调用点 | 层 | 代码位置 | 重试什么 | 为什么必须在这 |
|--------|-----|---------|---------|---------------|
| **调用点1** | 第3层 | `BaseAIService` + `LLMClient` | HTTP 429 | 只有HTTP层能拿到status_code |
| **调用点2** | 第2层 | `BaseAgent.run_stream()` 循环内 | Parse错误 + 空响应 | 需要修改prompt/截断历史，重新调LLM |
| **调用点3** | 第1.5层 | `react_sse_wrapper.py` | 网络异常、超时 | 需要重建整个Agent，重发请求 |

**辅助路径的重试也要收拢**：当前 `IntentClassifier` 和 `CapabilityDetector` 绕过 `BaseAIService` 直接用 `LLMClient`，各自手写重试。应统一收拢到调用点1。

### 问题3：如果只在一个地方调用，在哪？

**不可能。** 如果硬要只放一个地方，放到哪都不对：

| 方案 | 放在哪 | 会漏什么 | 后果 |
|------|--------|---------|------|
| 只放第3层 | `BaseAIService` | 漏掉Parse和空响应 | LLM返回乱码时无限循环（格式错→重试→还是格式错→再重试…） |
| 只放第2层 | Agent循环 | 漏掉429 | 429时Agent循环重试，但HTTP连接已经断了，重试也白搭 |
| 只放SSE层 | SSE wrapper | 漏掉所有应用层错误 | SSE流正常但LLM返回空/乱码时无保护 |

**正确答案**：3个调用点，各管各的层，职责清晰。

## 三、统一方案

### 3.1 总体架构（3个调用点）

```
┌──────────────────────────────────────────────────────┐
│  调用点1：第3层（LLM服务）— HTTP 429 重试              │
│  ├─ BaseAIService._post_with_retry()   [非流式]       │
│  ├─ BaseAIService._stream_with_retry() [流式]         │
│  └─ 新增：IntentClassifier / CapabilityDetector      │
│     改为通过 BaseAIService 调用（统一入口）             │
├──────────────────────────────────────────────────────┤
│  调用点2：第2层（Agent循环）— Parse + 空响应重试        │
│  ├─ BaseAgent.run_stream() 循环内                     │
│  ├─ Parse错误 → 截断prompt重发                        │
│  └─ 空响应   → 截断历史重发                           │
├──────────────────────────────────────────────────────┤
│  调用点3：第1.5层（SSE会话）— 网络异常重试（新增）      │
│  └─ generate_sse_stream_with_retry()                 │
│     包裹整个 agent.run_stream()                       │
│     网络/超时异常 → 重建Agent → 重发                   │
└──────────────────────────────────────────────────────┘
```

### 3.2 三个重试函数（`retry_engine.py` 新增）

每个调用点对应一个工厂函数，调用方不需要关心内部配置。

```python
def create_network_retry_engine(max_retries=3, backoff_factor=2.0) -> RetryEngine:
    """调用点1：第3层 LLM服务 — HTTP 429指数退避重试

    唯一调用点：BaseAIService.__init__()
    _post_with_retry 和 _stream_with_retry 共用同一个engine实例。

    原理：HTTP 429时等待退避后重发同一个请求，等服务端冷却。
    """
    return RetryEngine(
        max_retries=max_retries,
        backoff_strategy=BackoffStrategy.EXPONENTIAL,
        backoff_factor=backoff_factor,
    )

def create_agent_retry_engine(
    parse_max_retries=3, parse_backoff_factor=2.0,
    empty_max_retries=2, empty_backoff_factor=1.0,
) -> Tuple[RetryEngine, RetryEngine]:
    """调用点2：第2层 Agent循环 — Parse + 空响应重试

    唯一调用点：BaseAgent.__init__()
    一次调用返回两个engine（parse_engine, empty_engine），
    分别供 _handle_parse_error 和 _handle_empty_response 使用。

    原理：LLM返回格式错误或空响应时，修改prompt/截断历史后重新调LLM。
    """
    parse_engine = RetryEngine(
        max_retries=parse_max_retries,
        backoff_strategy=BackoffStrategy.EXPONENTIAL,
        backoff_factor=parse_backoff_factor,
    )
    empty_engine = RetryEngine(
        max_retries=empty_max_retries,
        backoff_strategy=BackoffStrategy.FIXED,
        backoff_factor=empty_backoff_factor,
    )
    return parse_engine, empty_engine

def create_sse_retry_engine(max_retries=3, backoff_factor=2.0) -> RetryEngine:
    """调用点3：第1.5层 SSE会话 — 网络异常/超时重试

    唯一调用点：generate_sse_stream_with_retry()

    原理：网络断开或超时时，重建Agent，重新发起整个SSE流。
    """
    return RetryEngine(
        max_retries=max_retries,
        backoff_strategy=BackoffStrategy.EXPONENTIAL,
        backoff_factor=backoff_factor,
    )
```

**设计原则：每个函数只有一个调用点。**

| 函数 | 唯一调用点 | 调用位置 | 返回 |
|------|-----------|---------|------|
| `create_network_retry_engine()` | `BaseAIService.__init__()` | `llm_core.py` | 一个engine实例，`_post_with_retry` 和 `_stream_with_retry` 共用 |
| `create_agent_retry_engine()` | `BaseAgent.__init__()` | `base_react.py` | 两个engine实例（parse + 空响应），一次调用全部创建 |
| `create_sse_retry_engine()` | `generate_sse_stream_with_retry()` | `react_sse_wrapper.py` | 一个engine实例，包裹整个SSE流 |

### 3.3 调用点1 改造：第3层统一

**当前问题**：`IntentClassifier` 和 `CapabilityDetector` 绕过 `BaseAIService`，直接用 `LLMClient`，各自手写重试。

**改造方案**：辅助路径改为通过 `BaseAIService` 调用，统一获得429重试保护。

| 文件 | 改造内容 |
|------|---------|
| `llm_core.py` | `_StreamRetryContext` 改用 `create_network_retry_engine()` |
| `intent_classifier.py` | 改为通过 `BaseAIService.chat()` 调用（不再直接用 `LLMClient`） |
| `capability_detector.py` | 改为通过 `BaseAIService.chat()` 调用（不再直接用 `LLMClient`） |

### 3.4 调用点2 改造：第2层统一

**当前问题**：
- `_parse_retry_engine` 调用私有方法 `_calculate_delay()`
- 手动管理 `parse_retry_count`，与 RetryEngine 状态两份并存
- 重试复位点分散在5处

**改造方案**：

`base_react.py` 初始化（一次调用，两个engine）：
```python
# 当前
from app.utils.retry_engine import RetryEngine, BackoffStrategy
self._parse_retry_engine = RetryEngine(max_retries=3, backoff_strategy=BackoffStrategy.EXPONENTIAL, backoff_factor=2.0)
self._empty_response_retry_engine = RetryEngine(max_retries=2, backoff_strategy=BackoffStrategy.FIXED, backoff_factor=1.0)

# 改造后
from app.utils.retry_engine import create_agent_retry_engine
self._parse_retry_engine, self._empty_response_retry_engine = create_agent_retry_engine()
```

`react_handler_mixin.py` 使用（Parse重试）：
```python
# 当前（react_handler_mixin.py:219）
_retry_delay = self._parse_retry_engine._calculate_delay(self.parse_retry_count + 1)
self.parse_retry_count += 1
if self.parse_retry_count >= self.max_parse_retries:
    yield self._exit_with_error(...)

# 改造后
# delay 必须在 record_attempt() 之前取，否则退避偏移一位
_retry_delay = self._parse_retry_engine.current_delay  # 基于当前attempt_count（未递增）
await asyncio.sleep(_retry_delay)
self._parse_retry_engine.record_attempt()
if self._parse_retry_engine.exhausted:  # attempt_count >= max_retries
    yield self._exit_with_error(...)
```

5处 `parse_retry_count = 0` 全部改为 `self._parse_retry_engine.reset_attempts()`：
- `_handle_chunk_type` (line 101)
- `_handle_completion_type` (line 160)
- `_handle_thought_only` (line 188)
- `_handle_action_type` (line 253)
- `base_react.py` 空响应分支 (line 326)

**注意**：`base_react.py:326` 的 `parse_retry_count = 0` 是**解析重试计数器复位**，不是空响应计数器冗余，**不可移除**。

### 3.5 调用点3 新增：第1.5层SSE重试

`react_sse_wrapper.py` 新增模块级函数：

```python
async def generate_sse_stream_with_retry(
    messages, intent_type, confidence, candidates, provider, model,
    temperature, task_id, session_id, ai_service, next_step,
    running_tasks, running_tasks_lock, current_execution_steps
) -> AsyncGenerator[str, None]:
    """带重试保护的SSE流（调用点3：第1.5层）"""
    retry_engine = create_sse_retry_engine()
    while True:
        try:
            async for event in generate_sse_stream(
                messages, intent_type, confidence, candidates, provider,
                model, temperature, task_id, session_id, ai_service,
                next_step, running_tasks, running_tasks_lock,
                current_execution_steps
            ):
                yield event
            return  # 正常结束
        except asyncio.CancelledError:
            return  # 用户中断，直接退出，不重试
        except (asyncio.TimeoutError, ConnectionError, httpx.RemoteProtocolError,
                httpx.ConnectError, httpx.ReadTimeout) as e:
            if retry_engine.exhausted:
                error_data = {"type": "error", "error": {"type": "connection_error",
                    "message": "SSE连接失败，重试耗尽", "detail": str(e)}}
                yield f"data: {json.dumps(error_data, ensure_ascii=False)}\n\n"
                return
            delay = retry_engine.current_delay
            logger.warning(f"[SSE重试] 第{retry_engine.attempt_count}次，等待{delay:.0f}s: {e}")
            await asyncio.sleep(delay)
            retry_engine.record_attempt()
        except Exception as e:
            error_data = {"type": "error", "error": {"type": "unknown_error",
                "message": str(e), "detail": traceback.format_exc()}}
            yield f"data: {json.dumps(error_data, ensure_ascii=False)}\n\n"
            return
```

**SSE重试状态恢复**：`_run_sse_stream` 每次调用都会创建新的 Agent 实例（`AgentFactory.create(intent_type)`），重试时 Agent 状态天然重建。conversation_history 会丢失前一轮的部分对话历史，这是可接受的——SSE重试等同于重新发起请求。

**SSE幂等性**：重试时已 yield 给客户端的步骤事件不会被撤回。当前设计接受此行为，不做去重。

`ChatRouter` 中调用点从 `generate_sse_stream(...)` 改为 `generate_sse_stream_with_retry(...)`。

### 3.6 RetryEngine 新增语义属性

```python
@property
def exhausted(self) -> bool:
    """重试是否已耗尽（必须用 >=，不能用 >）"""
    return self._attempt_count >= self._max_retries

@property
def current_delay(self) -> float:
    """当前退避延迟（在 record_attempt() 之前调用）"""
    return self._calculate_delay(self._attempt_count + 1)
```
