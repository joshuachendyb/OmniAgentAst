### 一、架构设计问题（8个问题）

#### 问题1: NewChatContainer组件职责过重
**来源文档**: 《Omni-第二次UX深度分析-详细报告-老杨-20260227.md》第四章，4.1节
**问题原因**: 
- 组件承担了过多职责：会话管理、消息管理、SSE流式输出、安全检测、任务中断、标题管理、状态持久化
- 代码行数达1394行，严重违反单一职责原则
- 所有功能耦合在一个组件中，难以测试和维护

**影响**: 
- 修改一个功能可能影响其他功能
- 难以进行单元测试和集成测试
- 修改时难以确定影响范围
- 代码复用性差

**改进方法**: 
1. 拆分组件职责，创建独立组件：
   - `ChatSessionManager` - 会话管理
   - `MessageManager` - 消息管理
   - `SSEConnection` - SSE连接管理
   - `SecurityChecker` - 安全检测
   - `TaskController` - 任务中断控制
   - `TitleManager` - 标题管理
2. 使用React Context或状态管理库（如Zustand）来管理共享状态
3. 使用props和回调来实现组件间通信

**预计工作量**: 大型重构，需要5-7天

#### 问题3: Layout组件包含过多业务逻辑

#### 问题4: 项目结构划分不清晰
**来源文档**: 《Omni-第二次UX深度分析-详细报告-20260227.md》第一章，1.2节
**问题原因**: 
- `components/`和`pages/`目录划分不清晰，部分组件功能边界模糊
- 缺少 `hooks/` 目录，自定义Hook散落在各个组件中
- 缺少 `assets/` 目录，静态资源管理混乱
- `tests/` 目录结构与源代码不匹配

**影响**: 
- 代码组织混乱，难以查找和维护
- 自定义Hook难以复用
- 静态资源难以管理
- 测试文件难以组织

**改进方法**: 
1. 重新组织目录结构：
   ```
   frontend/src/
   ├── components/      # 通用组件
   │   ├── layout/       # 布局组件
   │   ├── chat/          # 聊天组件
   │   ├── settings/     # 设置组件
   │   └── shared/        # 共享组件
   ├── pages/             # 页面组件
   │   ├── Chat/
   │   ├── History/
   │   └── Settings/
   ├── hooks/              # 自定义Hook
   ├── services/           # API服务
   ├── utils/              # 工具函数
   ├── contexts/            # React Context
   ├── types/              # TypeScript类型
   ├── assets/             # 静态资源
   └── tests/              # 测试文件
   ```
2. 将散落的自定义Hook移到hooks/目录
3. 将静态资源移到assets/目录
4. 重新组织测试文件结构

**预计工作量**: 中型重构，需要2-3天
