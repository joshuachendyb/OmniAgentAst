#### 1.2.1 ToolsStrategy 添加重试逻辑

**修改文件**: `backend/app/services/agent/llm_strategies.py`

- 添加 MAX_RETRIES = 3（最大重试次数）
- 添加 RETRY_DELAY = 2（重试等待时间秒）

- 添加 `_format_error_hint()` 方法，根据错误类型生成友好提示：

  - 超时：显示"请求超时，请检查网络后重试"
  - 连接错误：显示"网络连接失败，请检查网络后重试"
  - 认证错误：显示"API认证失败，请检查API密钥是否有效"

```
ToolsStrategy 调用 LLM
    ↓
收到 429 限流错误
    ↓
重试 1/3，等待 2 秒
    ↓
收到 429 限流错误
    ↓
重试 2/3，等待 2 秒
    ↓
收到 429 限流错误
    ↓
重试 3/3，等待 2 秒
    ↓
仍然失败 → 降级到 TextStrategy
    ↓
TextStrategy 调用 LLM
    ↓
返回空内容 + error="429"
    ↓
生成友好提示："[错误] 模型访问量过大（429限流），请稍后再试或更换模型"
```

## 1. TextStrategy分层处理架构C+A+B+增强中文提取 - 小沈-2026-03-29

### 1.1 问题背景

早期版本P8的修复错误：TextStrategy无论LLM返回什么，都包装成 `action_tool: "finish"`，导致LLM返回的工具调用意图被丢弃。

### 1.2 分层处理架构（C+A+B）

| 方案 | 作用 | 必要性 |
|------|------|--------|
| 方案C | 预处理层：ToolParser.parse_response() | ⭐⭐⭐ 必须 |
| 方案A | 正则提取层：增强_extract_from_text中文支持 | ⭐⭐ 推荐 |
| 方案B | 保底层：工具名匹配 | ⭐ 可选 |

### 1.3 处理流程

```
LLM 返回 content
    ↓
情况0: 空内容 → 返回 finish, content=""
    ↓
方案C: ToolParser.parse_response()
    ├─ JSON 解析成功 → 返回 action
    └─ 失败 → _extract_from_text()
        ├─ 提取成功 → 返回 action
        └─ 失败 → ValueError
    ↓
方案A: _extract_from_text() 增强版
    ├─ 英文纯文本含 action → ✅ 提取
    └─ 中文纯文本含 action → ✅ 提取
    ↓
方案B: _extract_by_known_tools()
    ├─ 找到已知工具名 → 返回匹配工具
    └─ 未找到 → 继续
    ↓
无法提取 → 返回 finish，保留完整 content
```

### 1.4 支持的场景

| 场景 | 处理方式 |
|------|---------|
| JSON 标准格式 | 方案C: ToolParser 解析 |
| Markdown JSON | 方案C: ToolParser 解析 |
| 英文纯文本含 action | 方案C: _extract_from_text |
| 中文纯文本含 action | 方案A: 增强正则提取 |
| 工具名直接出现 | 方案B: 已知工具名匹配 |
| 纯文本不含 action | 返回 finish，保留 content |
| JSON 数组格式 | 方案C: ToolParser 解析 |

## 2. 彻底修复 step 计数器统一问题 - 小沈-2026-03-23

### 1.1 问题描述

文件操作异常时，error 步骤编号跳号。例如：
- start: step=1
- error: step=3 (跳过了 step=2)

### 1.2 根因分析

`ver1_run_stream` 内部使用独立的 `step_count` 变量，与 chat2.py 的 `next_step()` 函数不共享状态，导致两个计数器不同步。

### 1.3 历史问题

| 版本 | 问题 | 当时的修复 |
|------|------|-----------|
| v0.7.39 | step_counter 被定义两次 | 删除重复定义 |
| v0.7.41 | ver1_run_stream step 计数器独立 | step_start=next_step()+1（将错就错） |
| v0.7.53 | error 步骤跳号 | max_step 计算（还是将错就错） |

### 1.4 本次修复（真正统一）

**方案**：传入 `next_step` 函数给 `ver1_run_stream`，让所有步骤共用同一个计数器。

```python
# chat2.py 传入
agent.ver1_run_stream(get_next_step=next_step)

# agent.py ver1_run_stream 使用
step = get_next_step()  # 与 chat2.py 共用同一个计数器
```

## 1. Structured Outputs 自适应兼容方案研究

更新时间：2026-03-20 10:05:00

### 1.1 研究背景

**问题**：原有方案依赖硬编码模型名称，新模型需要手动添加，无法处理未知模型。

**解决方案**：基于特征检测的自适应兼容方案，根据模型反馈自动适配。

### 1.2 三种 Structured Outputs 模式

| 模式 | 是否需要解析 | 可靠性 | 支持模型数 | 推荐度 |
|------|------------|--------|----------|--------|
| **tools** | ❌ 不需要 | ✅ 高（结构化返回） | ~50个 | ⭐⭐⭐ 强烈推荐 |
| **response_format** | ✅ 需要解析 | ⚠️ 中（部分非标准） | ~45个 | ⭐⭐ 可用 |
| **prompt** | ✅ 需要解析 | ❌ 低（需正则提取） | 所有模型 | ⭐ 不推荐 |

**返回格式对比**：

```
【tools 模式】- 推荐
{
  "tool_calls": [{
    "function": {
      "name": "search_files",
      "arguments": "{\"path\": \"C:/\", \"pattern\": \"*.py\"}"
    }
  }]
}
✅ 参数已是结构化，直接解析 arguments 即可

【response_format 模式】
{
  "content": "{\"action\": \"search_files\", \"path\": \"C:/\", \"pattern\": \"*.py\"}"
}
⚠️ content 是 JSON 字符串，需要 json.loads() 解析一次

【prompt 模式】
{
  "content": "我需要搜索 C:/ 下的 Python 文件"
}
❌ 纯文本，需要正则提取，容易出错
```

**根据实测结果修正**：

|--------|------|----------|------|
| **1** | **tools** | ~50个 | ✅ 支持最广 |
| **2** | response_format | ~45个 | 部分返回非标准JSON |
| **3** | prompt | 所有模型 | 降级方案 |

## 1. LLM工具调用参数命名规范 - 核心实现

更新时间：2026-03-20 08:23:00

### 1.1 问题背景

| 问题 | 系统预期 | LongCat返回 | 后果 |
|------|---------|------------|------|
| 参数名不一致 | `dir_path` | `directory_path` | 工具调用失败 |

**问题根因**：
| 层级 | 问题 | 影响 |
|-----|------|------|
| **工具定义层** | description过于简单 | LLM不知道如何正确调用 |
| **Prompt层** | 没有参数命名规则 | LLM随意命名 |
| **执行层** | 依赖参数映射 | 治标不治本 |

### 1.2 解决方案设计

**三层防御策略**：
```
第一层：增强System Prompt（全局规则）
      ↓
第二层：详细工具Description（每个工具）
      ↓
第三层：添加Examples（正确用法示范）
```

**技术选型**：
| 组件 | 选择 | 理由 |
|------|------|------|
| 参数定义 | Pydantic模型 | 自动生成Schema，类型安全 |
| 装饰器 | 自定义register_tool | 支持Pydantic模型 |
| 白名单 | 动态生成 | 自动添加所有存在的盘符 |
| 示例 | input_examples | Claude官方推荐 |

```python
            if drive.exists():
                paths.append(drive)
    return paths
```

### 1.6 路径验证改进

```python
def _validate_path(self, file_path: str) -> tuple[bool, Optional[str]]:
    real_path = Path(os.path.realpath(os.path.expanduser(file_path)))
    for allowed in self.allowed_paths:
        allowed_real = Path(os.path.realpath(allowed))
        if str(real_path).startswith(str(allowed_real)):
            return True, None
    return False, f"路径 '{file_path}' 不在允许的操作范围内..."
```

**改进点**：
1. 使用 `os.path.realpath` 规范化路径
2. 使用 `os.path.expanduser` 处理 `~`
3. 使用前缀匹配而非精确匹配
