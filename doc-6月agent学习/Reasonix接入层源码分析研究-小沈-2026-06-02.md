## 一、接入层架构：三层事件驱动

```
┌─────────────────────────────────────┐
│         前端 (Frontend)               │
│  TUI(channel)  TextSink  SSE/Browser │
│        │           │          │      │
│        └──── event.Sink ──────┘      │  ← 单方法接口
├─────────────────────────────────────┤
│       Controller (control/)          │  ← 传输无关的会话驱动
│   Send/Cancel/Approve/Compact       │
│   拥有 agent.Runner                  │
├─────────────────────────────────────┤
│        Agent (Runner)                │
│   sink.Emit(typed event)            │
└─────────────────────────────────────┘
```

**唯一抽象**：`Sink` 接口 `Emit(Event)`——整个前后端通信就这一个方法，没有回调、没有确认、没有背压。

## 三、Controller（`internal/control/controller.go`，~1300行）

### 3.1 设计定位

Controller是传输无关的会话驱动器。**不依赖任何前端**，接收 `event.Sink` 和 `agent.Runner`。

**关键字段**（`controller.go:42-132`）：

| 字段 | 用途 |
|------|------|
| `runner agent.Runner` | 执行turn的Runner（Agent/Coordinator） |
| `sink event.Sink` | 事件输出目标（前端注入） |
| `policy permission.Policy` | 权限策略 |
| `approvals map[string]chan approvalReply` | 待审批map（ID→channel） |
| `asks map[string]chan []event.AskAnswer` | 待回答问题map（ID→channel） |
| `autoApprove bool` | 计划审批后自动审批模式 |

### 3.2 命令接口

### 3.3 异步执行（`runGuarded`, 第273-292行）

```go
func (c *Controller) runGuarded(body func(ctx context.Context) error) {
    go func() {
        err := body(ctx)
        c.sink.Emit(event.Event{Kind: event.TurnDone, Err: err})
    }()
}
```

核心保证：
- **TurnDone总是最后一个事件**——前端据此知道turn结束
- **Err != nil表示失败**；Err == nil也表示用户取消（非错误）
- **保证串行**：turn执行中后续调用是no-op

### 3.4 审批系统（`controller.go:1563-1618`）

审批流程：
```
Gate → Controller.requestApproval(tool, subject)
   1. 检查 session grant（已批准缓存）
   2. 检查 autoApprove（计划模式）
   3. 检查 bypass（YOLO模式）
   4. 发出 ApprovalRequest 事件
   5. 阻塞等待 channel 响应
   6. 前端调用 Approve(id, allow, session) → channel回复
   7. 继续执行
```

**计划审批流**：批准计划后自动开启 `autoApprove`，执行轮的写入工具不再逐个审批。

## 四、端到端数据流

```
Agent.Run()
  ├── sink.Emit(TurnStarted)
  │
  ├── stream() ← Provider.Stream()
  │   ├── ChunkReasoning → sink.Emit(Reasoning)
  │   ├── ChunkText     → sink.Emit(Text)
  │   ├── ChunkToolCallStart → sink.Emit(ToolDispatch{Partial:true})
  │   └── 流结束 → sink.Emit(Message)
  │
  ├── sink.Emit(Usage)
  │
  ├── executeBatch()
  │   ├── 每个调用 → sink.Emit(ToolDispatch)
  │   ├── 执行（并行/串行）
  │   └── 每个结果 → sink.Emit(ToolResult)
  │
  └── 循环直到无tool_call
      → Controller.runGuarded → sink.Emit(TurnDone)
```

**关键设计**：ToolDispatch在流式阶段就发送（`Partial: true`），前端在LLM流式生参数时就开始渲染"正在调工具"。

```
POST /compact   → 压缩上下文
POST /new       → 新会话
```

### 5.2 Broadcaster（`broadcaster.go:15-65`）

```go
type Broadcaster struct {
    mu   sync.Mutex
    subs map[chan []byte]struct{}
}
```

- `Emit(event)`：序列化JSON → 分发到每个订阅者channel
- 缓冲满则丢弃（非阻塞，无回压）
- `Subscribe()`：创建带缓冲channel（64），返回unsubscribe函数

### 5.3 Wire格式（`wire.go:115-158`）

```go
type wireEvent struct {
    Kind       string        `json:"kind"`        // "turn_started", "text"等
    Text       string        `json:"text,omitempty"`
```

Kind到wire字符串映射：
`event.TurnStarted` → `"turn_started"`  
`event.Text` → `"text"`

## 六、对我方参考价值

| Reasonix做法 | 我方现状 | 可借性 |
|-------------|---------|--------|
| **单方法Sink接口** `Emit(Event)` | SSE+yield混合，回调满天飞 | **⭐⭐⭐⭐直接引入** |
| **14种结构化Event** | step类型模糊，前后端耦合 | **⭐⭐⭐⭐引入Event类型系统** |
| **Controller传输无关** | 事件产生和推送耦合在一起 | **⭐⭐⭐⭐Controller+Sink分离** |
| **审批即事件** | 无审批系统 | **⭐⭐⭐参考审批流设计** |
| **TurnDone总在最后** | 无明确turn结束标记 | **⭐⭐⭐⭐直接引入** |
| **ToolDispatch Partial标记** | 流式参数未知，前端等待 | **⭐⭐⭐参考partial标记** |
| **Broadcaster扇出** | 单SSE连接 | **⭐⭐参考多客户端** |
| **channel 1024缓冲** | 无缓冲 | **⭐⭐⭐引入缓冲防回压** |
