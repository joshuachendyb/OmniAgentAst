**创建时间**: 2026-05-07 19:55:01  
**创建人**: 小沈  
**问题现象**: LLM调用time工具时传错参数名，导致`unexpected keyword argument`错误

## 一、问题现象

### 1.1 典型错误案例
**问题1** 

| 工具 | LLM传的参数 | 正确参数名 | 错误信息 |
|------|-----------|----------|---------|
| `time_format` | `{"date":"2026-06-19"}` | `timestamp`, `pattern` | `got an unexpected keyword argument 'date'` |
| `time_is_weekend` | `{"path":"//www.baidu.com/..."}` | `date` | `got an unexpected keyword argument 'path'` |

**问题2** 
整个能力探测系统是完备的：

CapabilityDetector.detect()
    → LLMFeature（模型能力特征）
        → StrategySelector.select()
            → SelectedStrategy(method="response_format"|"tools"|"text")
但没有人触发它。adapter 从来没人注入，全系统 10 个 Agent 全部走 text_strategy。

### 1.2 参数校验结果

```python
# Schema定义（正确）
time_format: properties = ['timestamp', 'pattern']
time_is_weekend: properties = ['date']

# 函数签名（正确）
time_format(timestamp: Optional[Any] = None, pattern: Optional[str] = None)
time_is_weekend(date: Optional[Any] = None)

# LLM实际传入（错误）
time_format(date="2026-06-19")           # 参数名错误
time_is_weekend(path="http://...")       # 完全错误（把URL当参数）
```

## 三、问题根因分析

### 3.1 详细描述 vs Parameter Reminder 对比

**详细描述（L32-37）**：
```
2. time_format - Format timestamp or date string
   - Parameters:
     - timestamp: Unix timestamp (int/float), date string, or datetime. None = current time
     - pattern: Format string like "%Y年%m月%d日"
   - Example: time_format(timestamp=1777103094, pattern="%Y年%m月%d日")
```

**Parameter Reminder（L136）**：
```
- time_format: timestamp, pattern    ← 只有两个单词！无类型/示例
```

### 3.2 LLM为何传错参数

| 因素 | 说明 | 影响 |
|------|------|------|
| **位置距离** | 详细描述L32-37，Parameter Reminder L136，相隔100行 | LLM调用时记忆模糊 |
| **LLM记忆机制** | 调用时凭"最近记忆"，Parameter Reminder靠后更近 | 简略信息覆盖详细描述 |
| **信息密度** | Parameter Reminder太简略，`timestamp, pattern`只有名字 | 无法区分参数类型 |
| **语义偏差** | `time_format`语义是"格式化时间"，LLM直觉认为有`date`参数 | 语义理解错误 |

### 3.3 两个错误的差异

| 错误 | LLM思考过程 | 根因 |
|------|-----------|------|
| `time_format(date=...)` | "格式化日期应该有date参数" | **语义偏差**：直觉认为时间工具应该叫`date` |
| `time_is_weekend(path=...)` | 把搜索上下文的URL参数带进来 | **上下文污染**：前序对话的参数名干扰 |

| 层次 | 信息量 | 位置 | LLM可见性 |
|------|--------|------|----------|

## 五、改进方案

### 5.1 方案A：增强Parameter Reminder（✅ v0.12.19已实施）

**已实施**：所有8个Prompt类已增强Parameter Reminder，统一包含类型说明+FORBIDDEN段落。

```python
# 当前：
"- time_format: timestamp, pattern\n"
"- time_is_weekend: date\n"

# 改进后：
"- time_format: timestamp(时间戳/日期串如2026-04-25), pattern(格式串如%Y年%m月%d日)\n"
"- time_is_weekend: date(日期串/时间戳，如2026-04-25)\n"
```

**优点**：
- 保持Parameter Reminder简洁性
- 加入类型说明和示例，LLM不会混淆
- 修改量小

### 5.2 方案B：加强制性约束提醒（已融入FORBIDDEN段落）

v0.12.19已在各Parameter Reminder中统一加入FORBIDDEN段落，替代单独的强制性约束提醒。

### 5.3 方案C：执行层兜底（✅已实施）

修改 `tool_executor.py` 的 `_normalize_params`：检测到非法参数后**删除**，防止传给函数崩溃。

```python
# 当前（只WARNING不删除）：
if key not in valid_params:
    logger.warning(...)

# 改进（删除非法参数）：
invalid_keys.append(key)
for key in invalid_keys:
    del params[key]
```

**优点**：防止崩溃  
**缺点**：治标不治本，LLM仍传错参数

#### 实现步骤

**步骤1**：在 `registry.py` 新增方法：

```python
def to_openai_tools(self, category: Optional[ToolCategory] = None) -> List[Dict]:
    """
    生成OpenAI API格式的tools定义
    
    Args:
        category: 工具分类，None=全部
    
    Returns:
        [{"type": "function", "function": {...}}, ...]
    """
    tools = []
    tool_list = self.list_tools(category=category, include_metadata=True)
    
    for meta in tool_list:
        if not meta.expose_to_llm:
            continue
        tools.append({
            "type": "function",
            "function": {
                "name": meta.name,
                "description": meta.description,
                "parameters": meta.input_schema
            }
        })
    
    return tools
```

**步骤2**：在 `BaseAgent.__init__()` 或 `react_agent_mixin._init_llm_strategies()` 中：

```python
def _init_llm_strategies(self):
    """初始化LLM调用策略"""
    self.text_strategy = TextStrategy()
    
    # Adapter + Tools策略（带异常降级）
    try:
        from app.services.agent.llm_adapter import LLMAdapter
        from app.services.agent.llm_strategies import ToolsStrategy, ResponseFormatStrategy
        from app.services.tools.registry import tool_registry
        
        if self.llm_client and hasattr(self.llm_client, 'api_base'):
            self.adapter = LLMAdapter(
                api_base=self.llm_client.api_base,
                api_key=self.llm_client.api_key,
                model=self.llm_client.model,
                auto_detect=False  # 延迟探测，首次调用时触发
            )
            
            openai_tools = tool_registry.to_openai_tools(self.tool_category)
            self.tools_strategy = ToolsStrategy(tools=openai_tools)
            self.response_format_strategy = ResponseFormatStrategy()
        else:
            self.adapter = None
            self.tools_strategy = None
            self.response_format_strategy = None
    except Exception as e:
        logger.warning(f"[LLM策略初始化失败，降级到文本模式: {e}")
        self.adapter = None
        self.tools_strategy = None
        self.response_format_strategy = None
```

**设计依据**: 根因分析——参数信息通过3个独立通道、4个源文件到达LLM，各通道信息不同步导致LLM传参错误

### 13.1 问题根源回顾

参数信息通过 **3条独立通道** 从 **4个源文件** 到达 LLM：

```
① time_schema.py  ──→  Pydantic 模型（参数名+类型+description）
      （✅ 唯一真相源，自动生成 JSON Schema）
          │
          ├──→ 通道 3：tools JSON Schema（结构化，API tools 参数）
          │   （✅ 已自动生成，不需手写）
          │
          ├──→ ② time_register.py：TIME_TOOL_DESCRIPTIONS
          │    的"参数说明"段落（❌ 手写，重复 Pydantic Field desc）
          │    也进入通道 3 的 description 字段
          │
          ├──→ ③ time_prompts.py:get_system_prompt()
          │    的 "- Parameters:" 块（❌ 手写）
          │    └──→ 通道 1：conversation_history[0] 自然语言描述
          │
          └──→ ④ time_prompts.py:get_parameter_reminder()
                的手写参数列表（❌ 手写）
                └──→ 通道 1：conversation_history[0] 自然语言描述

  另有 ──→ registry.py:get_all_tools_summary()
            └──→ 通道 2：_call_llm_with_summary() 每轮注入 system 消息
                （❌ 半自动，含参数名重复）
```

**3条通道详解**：

| 通道 | 传递方式 | 出现在哪 | 内容 | 源文件 | 当前状态 |
|------|---------|---------|------|--------|---------|
| **通道 1** | `conversation_history[0]` system message | system prompt 段落② (`- Parameters:`块) + 段落⑥ (Parameter Reminder) | 自然语言：参数名、类型、示例 | ③ `time_prompts.py:get_system_prompt()` + ④ `time_prompts.py:get_parameter_reminder()` | ❌ 手写 |
| **通道 2** | 每轮 ReAct 循环追加一条 system 消息 | `_call_llm_with_summary()` 注入到 `history_dicts` | "【当前可用工具列表】time_format(timestamp): 描述..." | `registry.py:get_all_tools_summary()` | ❌ 半自动（含参数名） |
| **通道 3** | API `tools` 参数（JSON Schema 结构化） | **⚠ 当前未启用！** 设计路径：`registry.py` → `ToolsStrategy(tools=...)` → `llm_core.chat_with_tools(tools=...)` → `request_json["tools"]`。另有 `react_schema.py:get_tools_schema_for_function_calling()` 也生成tools格式，但 `self._tools_schema` 只赋值从未读取，是死代码 | `function.name`（工具名）+ `function.description`（来自 register）+ `function.parameters.properties`（来自 Pydantic）+ `function.parameters.required`（来自 Pydantic） | ① `time_schema.py`（Pydantic 模型 → `input_schema`）+ ② `time_register.py`（`TIME_TOOL_DESCRIPTIONS` → `description`）→ `registry.py` | ❌ 未启用；`properties/required` 由 Pydantic 自动生成正确，但②的 `description` 含重复参数说明 |

**核心问题**：同一份参数名/类型/描述在 4 个文件中独立手写，修改不同步 → LLM 收到矛盾信息 → 传参错误。

- ① `time_schema.py`: `timestamp: Optional[Union[int, float, str]]`（✅ 唯一真相源）

- ③ `time_prompts.py:get_system_prompt()`: "- timestamp: Unix timestamp (int/float)..."（❌ 重复）
- ④ `time_prompts.py:get_parameter_reminder()`: "time_format: timestamp(optional, int/float/str)..."（❌ 重复）

```python
        return "string"
```

**Step A**：在 `registry.py` 中新增 `generate_param_reminder()`。

**关键设计**：直接从 Pydantic `input_schema` 的 `default` 字段读取默认值（比正则解析 description 更可靠）。

验证示例：

```python
def _format_default(val) -> str:
    """将 Pydantic 默认值格式化为字符串，跳过 None"""
    if val is None:
        return ""          # Optional 标记，无语义信息
    if isinstance(val, str):
        return "default=" + val
    if isinstance(val, bool):
        return "default=" + ("true" if val else "false")
    if isinstance(val, (int, float)):
        return "default=" + str(val)
    return ""


def generate_param_reminder(self, category=None) -> str:
    """从 input_schema 自动生成 Parameter Reminder 文本
    
    参数信息完全来自 Pydantic 模型：
    - 参数名：properties 的 key
    - 参数类型：properties[field].type
    - 必填/可选：是否在 required 数组中
    - 默认值：properties[field].default（跳过 None）
    """
    lines = ["Parameter Reminder (auto-generated from Pydantic):", ""]
    for name, meta in sorted(self._tools.items(), key=lambda x: x[0]):
        if not meta.expose_to_llm:
            continue
        if category and meta.category != category:
            continue
        schema = meta.input_schema
        if not schema or "properties" not in schema:
            continue
        
        required_set = set(schema.get("required", []))
        param_parts = []
        for pname, pinfo in schema.get("properties", {}).items():
            ptype = pinfo.get("type", "any")
            req_str = "required" if pname in required_set else "optional"
            default_str = _format_default(pinfo.get("default"))
            if default_str:
                param_parts.append("{}({}, {}, {})".format(pname, req_str, ptype, default_str))
            else:
                param_parts.append("{}({}, {})".format(pname, req_str, ptype))
        
        if param_parts:
            lines.append("- " + name + ": " + ", ".join(param_parts))
    
    return "\n".join(lines)
```

## 十四、三策略适配方案

**问题发现**: 当前第12章和第13章的设计都以ToolsStrategy（Function Calling）为目标，
但未处理模型仅支持ResponseFormatStrategy或仅支持TextStrategy的场景。

### 14.1 三种策略的LLM消息构建差异

| 策略 | API参数 | LLM收到什么参数信息 | 当前实现状态 |
|------|---------|-------------------|------------|
| **ToolsStrategy** | `request_json["tools"] = [...]` | 完整的Pydantic Schema（参数名+类型+必填+description） | ✅ `to_openai_tools()` 已实现 |
| **ResponseFormatStrategy** | `request_json["response_format"] = {...}` | **仅有 `tool_params: object` 泛型约束**，无具体参数名 | ❌ **当前只传泛型schema，LLM不知道参数名** |
| **TextStrategy** | 无附加参数 | **仅靠prompt文本**（通道1⑥ Parameter Reminder + 通道②工具描述） | ⚠ 依赖手写reminder，G6改为自动生成后依赖Pydantic |

**三种策略下 LLM 看到的参数信息来源总结**：

```
TextStrategy:        messages.system = [段落②工具描述] + [段落⑥Parameter Reminder]
                     ↑ 参数名的唯一来源

ResponseFormat:      messages.system = [段落②工具描述] + [段落⑥Parameter Reminder]
                     response_format.tool_name.enum = [工具名列表]（改造后）
                     ↑ 参数名的唯一来源仍是段落⑥ Reminder

ToolsStrategy:       messages.system = [段落②工具描述] + [段落⑥Parameter Reminder]
                     tools[].function.parameters.properties = [完整参数定义]
                     ↑ 参数名的第一来源，段落⑥ Reminder可简化
```
通道1 段落②	工具用途 + 使用场景 + 示例	去掉参数行（G5）	❌ 不再重复
通道1 段落⑥	参数名 + 类型 + required/optional + 语义默认值	Pydantic 自动生成（G6+G1）	✅ 唯一文本源
通道2	工具名: 简短描述（无参数）	registry（G7去掉参数名）	❌ 不再重复
通道3 tools	结构化 JSON Schema（properties + required）	Pydantic model_json_schema()	✅ 结构化源，与文本源同根


**ToolsStrategy 发给 LLM 的完整请求**：

```json
{
    "model": "deepseek-v3.1",
    "messages": [
        {
            "role": "system",
            "content": "【当前系统】Windows...\n\n"
                       "2. time_format - Format timestamp or date string\n"
                       "   When to use: \"格式化时间\"\n"
                       "   Example: time_format(timestamp=1777103094, pattern=\"%Y年%m月%d日\")\n\n"
                       "【Response Format】: ...\n"
                       "Parameter Reminder:\n"
                       "- time_format: timestamp(optional, int/float/str), pattern(optional, str)\n"
                       ...
        },
        {"role": "user", "content": "Task: 查询今天是星期几\nCurrent time: 2026-05-09 12:00:00\n..."}
    ],
    "tools": [                                              ← 结构化的工具定义数组
        {
            "type": "function",
            "function": {
                "name": "get_current_time",
                "description": "获取当前系统时间...",
                "parameters": {                             ← 来自 Pydantic input_schema
                    "type": "object",
                    "properties": {
                        "timezone": {"type": "string", "description": "..."},
                        "format": {"type": "string", "description": "..."}
                    }
                }
            }
        },
        ...其他工具
    ],
    "tool_choice": "auto"
}
```

LLM 从 `tools[].function.parameters.properties` 中直接看到每个工具的参数名、类型、描述。
LLM 从 `tools[].function.parameters.required` 中知道哪些参数必填。

### 14.2 ResponseFormatStrategy改造：带工具参数的response_format

#### 14.2.1 ToolsStrategy vs ResponseFormatStrategy 的 LLM 请求对比

核心问题：**ResponseFormatStrategy当前传的response_format是泛型的，不包含任何工具参数的定义**。

```python
# 当前 ResponseFormatStrategy.response_format（llm_strategies.py:631-645）
{
    "type": "json_object",
    "properties": {
        "thought": {"type": "string"},
        "tool_name": {"type": "string"},
        "tool_params": {"type": "object", "description": "工具参数"}
        # ↑ 泛型 object，没有具体的参数名和类型！
    }
}
```


**ResponseFormatStrategy 发给 LLM 的完整请求**（改造前）：

```json
{
    "model": "deepseek-v3.1",
    "messages": [
        {
            "role": "system",
            "content": "【当前系统】Windows...\n\n"
                       "2. time_format - Format timestamp or date string\n"
                       "   When to use: \"格式化时间\"\n"
                       "   Example: time_format(timestamp=1777103094, pattern=\"%Y年%m月%d日\")\n\n"
                       "【Response Format】: ...\n"
                       "Parameter Reminder:\n"
                       "- time_format: timestamp(optional, int/float/str), pattern(optional, str)\n"
                       ...
        },
        {"role": "user", "content": "Task: 查询今天是星期几\n..."}
    ],
    "response_format": {                                    ← 只有输出格式约束
        "type": "json_schema",
        "json_schema": {
            "name": "execute_tool",
            "schema": {
                "type": "object",
                "properties": {
                    "thought": {"type": "string"},
                    "tool_name": {"type": "string"},        ← 没有 enum！LLM不知道有哪些工具
                    "tool_params": {"type": "object"}       ← 泛型！LLM不知道参数名
                }
            }
        }
    }
}
```

LLM 从 `response_format` 中只能知道输出必须是 `{thought, tool_name, tool_params}` 这个结构。
**LLM 从哪里知道有哪些工具？** → 从 messages 中的系统提示（段落②工具描述）知道。
**LLM 从哪里知道参数名？** → **仅靠** 系统提示中的 Parameter Reminder（段落⑥）。

**关键差异**：

| 对比项 | ToolsStrategy | ResponseFormatStrategy |
|--------|--------------|----------------------|
| **告诉LLM有哪些工具** | `tools[].function.name` ✅ 明确 | **仅靠** 系统提示段②的文字描述 ❌ 模糊 |
| **告诉LLM参数名** | `tools[].function.parameters.properties` ✅ | **仅靠** 系统提示段⑥的 Parameter Reminder |
| **告诉LLM参数类型** | `properties[param].type` ✅ | 系统提示段⑥的文字 |
| **告诉LLM必填/可选** | `required` 数组 ✅ | 系统提示段⑥的 `(required)/(optional)` |
| **保证输出是JSON** | `tool_choice="auto"` ❌ 不保证 | `response_format` ✅ 保证 |
| **保证参数名正确** | `properties` 约束 ✅ 模型级 | ❌ 仅靠prompt文本提醒，LLM可能传错 |

**改造目标（14.2）**：在 response_format 中加入工具名枚举，让 LLM 至少知道有哪些工具可选：
```json
"tool_name": {
    "type": "string",
    "enum": ["get_current_time", "time_format", "timer_set", ...]
}
```

但 `tool_params` 的约束没法通过 response_format 实现——因为 OpenAI 的 response_format 不支持"tool_name=A时params={...}，tool_name=B时params={...}"这种条件式 schema。`tool_params` 仍将是 `type: object` 泛型，参数名的约束仍然依赖 prompt 文本。

---


#### 14.2.2 ResponseFormatStrategy改造：带工具参数的response_format

需要将 `response_format` 从泛型改为带具体工具参数的schema。

**改造方法**：在 `_init_llm_strategies()` 初始化 `ResponseFormatStrategy` 时，传入动态生成的response_format schema：

```python
def _build_response_format_schema(tools: List[Dict]) -> Dict:
    """根据openai_tools生成response_format schema"""
    return {
        "type": "json_schema",
        "json_schema": {
            "name": "execute_tool",
            "strict": True,
            "schema": {
                "type": "object",
                "properties": {
                    "thought": {"type": "string"},
                    "reasoning": {"type": "string"},
                    "tool_name": {
                        "type": "string",
                        "enum": [t["function"]["name"] for t in tools]  # 枚举所有工具名
                    },
                    "tool_params": {
                        "type": "object",
                        "description": "工具参数"
                        # 注意：response_format中的tool_params不能像tools那样分工具定义。
                        # 工具参数的约束仍依赖通道1⑥的Parameter Reminder。
                    }
                },
                "required": ["thought", "tool_name", "tool_params"]
            }
        }
    }
```

**限制说明**：OpenAI的response_format不支持"条件式"的tool_params（即不同tool_name对应不同params）。这意味着：
- 即使用response_format，LLM仍需从通道1⑥的Parameter Reminder获取参数名
- response_format只能保证输出是合法JSON，不能保证参数名正确
- **G6（自动生成Parameter Reminder）在ResponseFormatStrategy下仍然是必要的**

**结论**：ResponseFormatStrategy仅保证JSON格式合法，不能替代ToolsStrategy的参数约束能力。

造后 ResponseFormatStrategy 只能做到：

response_format.tool_name.enum = ["get_current_time", "time_format", ...]
让 LLM 知道有哪些工具名可选。但做不到的：

能力	ToolsStrategy	ResponseFormatStrategy
properties 详细参数名+类型	✅ tools[].function.parameters.properties	❌ 做不到
required 必填标记	✅ tools[].function.parameters.required	❌ 做不到
参数级约束	✅ 模型级	❌ 只能靠prompt文本（通道1⑥ Reminder）

文档 2262-2267 行已经明确写了这个限制，14.1 的总结表也对比了。这个局限是 OpenAI API 本身的限制——response_format 不支持条件式 schema（tool_name=A 时 params 要含 timestamp，tool_name=B 时 params 要含 delay），所以不可能通过 response_format 达到 ToolsStrategy 的约束力。


### 14.3 TextStrategy 的参数准确性改进

TextStrategy 下 LLM 没有任何 API 级约束，参数名错误的唯一防线是通道1⑥的 Parameter Reminder。
G6 已将该提醒改为从 Pydantic 自动生成，但格式仍然是 "human-readable list" 风格。
以下方案可进一步提升参数准确性，按效果/成本排序。

#### 方案A：代码风格函数签名（推荐）

将 Parameter Reminder 格式从自然语言改为 LLM 更熟悉的**代码函数签名**风格。

```python
# 当前（G6自动生成，人类可读风格）：
"- time_format: timestamp(optional, int/float/str), pattern(optional, str)"

# 方案A（代码函数签名风格）：
"def time_format(timestamp?: int|float|string, pattern?: string)"
```

**理由**：LLM 的训练数据中包含大量函数定义代码，代码风格的参数描述比自然语言更"像回事"，LLM 更容易严格按照定义传参。

**实现**：修改 `generate_param_reminder()` 中的格式，提供两种风格选项：

```python
def generate_param_reminder(self, category=None, style="code") -> str:
    """
    style="code":  def time_format(timestamp?: int|float|string, pattern?: string)
    style="text":  - time_format: timestamp(optional, int/float/str), pattern(optional, str)
    """
    ...
    if style == "code":
        # 代码风格
        required_set = set(schema.get("required", []))
        params = []
        for pname, pinfo in schema["properties"].items():
            ptype = pinfo.get("type", "any")
            default_str = self._format_default(pinfo.get("default"))
            optional_mark = "" if pname in required_set else "?"
            if default_str:
                params.append(f"{pname}{optional_mark}: {ptype}={default_str.split('=')[1]}")
            else:
                params.append(f"{pname}{optional_mark}: {ptype}")
        if params:
            lines.append(f"def {name}({', '.join(params)})")
    else:
        # 文本风格（当前），省略...
```

**输出对比**：
```
当前文本风格:
- time_format: timestamp(optional, int/float/str), pattern(optional, str)

方案A代码风格:
def time_format(timestamp?: int|float|string, pattern?: string)
def timer_set(delay: number, callback: string, callback_data?: object)
def time_is_weekend(date?: int|float|string)
```

#### 方案B：`response_format: json_object` 兜底

TextStrategy 的 `llm_client.chat()` 默认不传任何参数。但如果模型支持 `response_format: {"type": "json_object"}`，即使不支持 tools，也可以加上这个约束——至少保证 LLM 输出是合法 JSON，`parse_react_response()` 不需要容错。

```python
# 当前 TextStrategy.call()
response = await llm_client.chat(message, history_dicts)

# 改造后 TextStrategy.call()
response = await llm_client.chat_with_response_format(
    message, history_dicts,
    response_format={"type": "json_object"}
)
```

**成本**：极低，只需判断模型是否支持 `response_format`。
**效果**：保证 JSON 格式正确，但参数名约束仍需 Parameter Reminder。
**依赖**：需在 `CapabilityDetector` 中判断模型能力，TextStrategy 按能力级别调整。

#### 方案C：动态注入Schema JSON示例

在每轮 system prompt 末尾注入当前工具的完整 JSON Schema 作为参考。虽然 ToolsStrategy 下这个信息通过 `tools` 参数传递，但 TextStrategy 下也可以把 Schema 以文本形式放在 prompt 中。

```
参考 Schema（仅作参考，实际调用仍以 JSON 格式返回）:
get_current_time: {"timezone": {"type": "string", "required": false}, "format": {"type": "string", "required": false}}
time_format: {"timestamp": {"type": "string|integer", "required": false}, "pattern": {"type": "string", "required": false}}
```

**成本**：低，直接从 `input_schema` 生成。
**效果**：让 LLM 看到完整的参数类型定义，减少直觉性传参错误。

#### 方案D：参数名纠正循环

当 `_normalize_params` 检测到非法参数名时，不直接返回错误，而是调用 LLM 重试（带纠正提示）。

```python
# 当检测到非法参数名时（当前行为）
return {"success": False, "error": f"unexpected keyword argument 'path'"}

# 纠正循环（方案D）
attempts = 0
while attempts < max_retries:
    result = await self._execute_tool(action, params)
    if "unexpected keyword argument" in str(result.get("error", "")):
        # 提取错误的参数名，映射到正确参数名
        wrong_param = extract_wrong_param(result["error"])
        correct_param = alias_map.get(wrong_param, wrong_param)
        # 在 observation 中告诉 LLM 正确参数名
        observation += f"\n⚠️ 参数名 '{wrong_param}' 错误，应使用 '{correct_param}'。请重新调用。"
        attempts += 1
        continue
    break
```

**成本**：中（额外 LLM 调用），但大幅降低参数错误率。

#### 方案对比总结

| 方案 | 效果 | 成本 | 复杂度 | 推荐 |
|------|------|------|--------|------|
| **A: 代码风格签名** | 中（减少参数直觉性错误） | 极低 | 低 | ✅ **推荐** |
| **B: json_object 兜底** | 中（保证 JSON 格式） | 极低 | 低 | ✅ **推荐** |
| **C: 注入 Schema** | 中（让 LLM 看到完整类型） | 低 | 低 | ⚠ 可选 |
| **D: 纠正循环** | 高（兜底最强） | 高（额外 LLM 调用） | 中 | ⏸ 暂缓 |


TextStrategy下，LLM没有任何API层面的参数约束，完全依赖prompt文本。

**【重要】方案C与G6互斥——选择方案C，去掉G6**：
- G6（自动生成Parameter Reminder）和方案C（动态注入Schema文本）功能重叠，**两者不可并存**
- **选择方案C，去掉G6**：方案C直接复用`self.openai_tools`生成完整参数定义，比G6的手写Parameter Reminder更准确、更自动
- 实施时：删除`build_full_system_prompt()`中对`get_parameter_reminder()`的调用，参数信息由`_tools_to_schema_text()`提供（见第16章16.2.5步骤5和16.3）

改造后各通道在TextStrategy下的作用：

| 通道 | 内容 | TextStrategy下 | 是否必要 |
|------|------|---------------|---------|
| 通道1② get_system_prompt() | 工具用途+场景+示例 | G5去掉-Parameters:块后，保留When to use+Example | ✅ 必要 |
| 通道1⑥ Parameter Reminder | ~~已去掉G6~~，由方案C替代 | **已删除**，参数信息由方案C提供 | ❌ 已去掉 |
| 方案C Schema文本注入 | 完整参数名+类型+required/optional | **唯一参数信息来源**（从Pydantic自动生成） | ✅ **关键** |
| 通道2 工具概要 | 工具名: 简短描述（无参数） | 告诉LLM有哪些工具可用 | ✅ 辅助 |
| 通道3 tools JSON Schema | 结构化参数定义 | **不使用**（走TextStrategy时不发送） | ❌ |

**关键结论**：TextStrategy下，**方案C（动态注入Schema文本）是必须项**，替代G6成为LLM获取参数名的唯一渠道。

### 14.4 三种策略下参数信息完整性对比

| 维度 | TextStrategy | ResponseFormatStrategy | ToolsStrategy |
|------|-------------|----------------------|--------------|
| 知道有哪些工具 | ✅ 通道2概要 | ✅ 通道2概要 | ✅ tools[name] |
| 知道参数名 | ✅ **仅靠方案C Schema注入** | ✅ 方案C Schema注入（response_format只有object泛型） | ✅ tools[].parameters.properties |
| 参数必填/可选 | ✅ 方案C Schema注入 | ✅ 方案C Schema注入 | ✅ required数组 |
| 参数类型 | ✅ 方案C Schema注入 | ✅ 方案C Schema注入 | ✅ type字段 |
| 默认值 | ✅ 方案C Schema注入 | ✅ 方案C Schema注入 | ✅ default字段 |
| JSON格式保证 | ❌ 需parse_react_response容错 | ✅ response_format保证 | ✅ tool_calls原生 |

**核心发现**：**方案C替代G6后，无论哪种策略，参数信息都有来源**。ToolsStrategy下tools参数提供强约束，TextStrategy和ResponseFormatStrategy下方案C提供文本级约束。

### 14.5 策略选择对实施的指导

| 当前能力 | 推荐策略 | 需要做什么 |
|----------|---------|-----------|
| 支持tools（function calling） | **ToolsStrategy** | 按第12章FC方案激活，走最优通道 |

#### 15.0.3.1 支持tool_calls模式时的OUTPUT_FORMAT修改方案

**创建时间**: 2026-05-10 18:04:40
**创建人**: 小健

**背景**：当前OUTPUT_FORMAT（15.0.4节）只考虑了text/response_format两种策略，都是LLM在content中返回JSON。
当使用tools模式（chat_with_tools）时，LLM可能走tool_calls通道，此时返回格式完全不同。

| 策略 | LLM调用方式 | OUTPUT_FORMAT的作用 | 优先级 |

#### 15.0.4 ResponseFormatStrategy的返回约束方案

```json
            "dir_path": {"type": "string", "description": "目录路径"},
            "recursive": {"type": "boolean", "description": "是否递归"}
          },
          "required": ["dir_path"]
        }
      }
    }
  ],
  "tool_choice": "auto"
}
```

**LLM返回有两种可能**：

###### 2a. 走tool_calls通道（理想情况，模型规范遵守OpenAI FC协议）

```json
{
  "choices": [{
    "message": {
      "role": "assistant",
      "content": null,
      "tool_calls": [{
        "id": "call_abc123",
        "type": "function",
        "function": {
          "name": "list_directory",
          "arguments": "{\"dir_path\": \"D:/\", \"recursive\": true}"
        }
      }]
    }
  }]
}
```

**行为特征**：
- ✅ **参数名和类型完全受Schema约束**：API校验，不合法会被拒绝
- ✅ **tool_name必然是已定义工具**：不在tools列表中的名字不会被返回
- ❌ **没有thought字段**：tool_calls结构中不存在thought
- ❌ **没有reasoning字段**：tool_calls结构中不存在reasoning
- ❌ **content为null**：LLM的思考过程完全丢失

**信息完整度**：⭐⭐⭐（缺thought和reasoning，需要_format_tool_calls硬造假的thought）

###### 2b. 走content通道（模型不完全遵守FC协议，实际常见）

```json
{
  "choices": [{
    "message": {
      "role": "assistant",
      "content": "{\"thought\": \"查看D盘目录\", \"reasoning\": \"用list_directory\", \"tool_name\": \"list_directory\", \"tool_params\": {\"dir_path\": \"D:/\"}}",
      "tool_calls": []
    }
  }]
}
```

**行为特征**：
- ✅ **thought和reasoning完整**：LLM在content中按OUTPUT_FORMAT写了
- ⚠️ **JSON可能破损**：thought中的引号/特殊字符可能导致JSON不合法（如`"名为"穆里奇"的文件"`）
- ⚠️ **tool_name不在Schema约束中**：LLM可能返回不存在的工具名
- ⚠️ **参数名不受Schema约束**：可能用非标准参数名

**信息完整度**：⭐⭐⭐⭐（信息完整但JSON可能破损，需正则兜底解析）

---

##### 3. 两种模式对比总结

| 维度 | response_format | tools(走tool_calls) | tools(走content) |
|------|----------------|--------------------|--------------------|
| **JSON合法性** | ✅ API强制保证 | ✅ 结构化不需JSON解析 | ⚠️ 可能破损 |
| **thought/reasoning** | ✅ 完整 | ❌ 丢失（需造假） | ✅ 完整 |
| **tool_name约束** | ⚠️ 无约束，可能返回不存在的工具 | ✅ 必然是已定义工具 | ⚠️ 无约束 |
| **参数名约束** | ⚠️ 无约束，可能用非标准名 | ✅ Schema强约束 | ⚠️ 无约束 |
| **参数值类型** | ⚠️ 无约束 | ✅ Schema强约束 | ⚠️ 无约束 |
| **模型兼容性** | ⚠️ 部分模型不支持 | ⚠️ 很多模型不稳定走content | ✅ 所有模型都能写 |
| **并行调用** | ❌ 一次一个 | ✅ 支持多tool_calls | ❌ 一次一个 |
| **解析统一性** | ✅ 走parse_react_response | ❌ 需额外_format_tool_calls | ✅ 走parse_react_response |

---

##### 4. 结论与建议

**核心矛盾**：tools模式的tool_calls通道**参数约束最强但信息最不完整**（丢失thought/reasoning），而content通道**信息最完整但约束最弱**。

**建议方案**：**以OUTPUT_FORMAT为统一契约，content返回为优先通道**

1. **所有策略统一注入OUTPUT_FORMAT**：不管哪种策略，prompt里都要求LLM在content中返回`{thought, reasoning, tool_name, tool_params}`格式
2. **ToolsStrategy下，content有内容时优先用content**：因为content里包含完整的thought/reasoning，信息更完整
3. **tool_calls作为参数安全的补充通道**：当content为空、tool_calls有值时，走tool_calls（参数结构化保证），但需要补造thought/reasoning
4. **解析统一走parse_react_response**：无论哪个策略、哪条通道返回，最终都交给parse_react_response统一解析（含正则兜底）

**优先级排序**：
```
content中的OUTPUT_FORMAT格式 > tool_calls结构化格式 > 破损JSON+正则兜底 > 关键词匹配 > implicit
```

---

**约束设计的分层结构**：

| 约束层 | 约束内容 | 约束来源 | 约束力 |
|--------|---------|---------|--------|
| **第1层：格式约束** | tool_calls数组格式 | tools参数 | ✅ 强（API校验） |
| **第2层：工具名约束** | function.name必须是已定义工具 | tools参数 | ✅ 强（API校验） |
| **第3层：参数约束** | 参数名、类型、必填性 | tools参数Schema | ✅ 强（API校验） |

**设计要点**：
- tools参数承担主要约束职责（强约束）
- OUTPUT_FORMAT仅提供说明性内容（弱约束）
- Prompt的重要性最低

```python
def _tools_to_schema_text(self) -> str:
    """将openai_tools转换为文本格式（方案C）- 小沈 2026-05-09
    
    直接复用tools结构，转换为LLM可读的文本格式
    只在text和response策略下使用
    
    Returns:
        Schema文本（如："timer_set: delay(number, required), callback(string, required)"）
    """
    if not self.openai_tools:
        return ""
    
    lines = ["【Tools Schema参考（仅作参考，实际调用仍以JSON格式返回）】:"]
    
    for tool in self.openai_tools:
        func = tool.get("function", {})
        name = func.get("name", "")
        params = func.get("parameters", {})
        properties = params.get("properties", {})
        required = params.get("required", [])
        
        if not properties:
            lines.append(f"{name}: 无参数")
            continue
        
        # 转换为简洁格式：参数名(类型, required/optional)
        params_list = []
        for pname, pinfo in properties.items():
            ptype = pinfo.get("type", "any")
            pdefault = pinfo.get("default")
            is_required = pname in required
            
            if pdefault is not None:
                params_list.append(f"{pname}({ptype}, default={pdefault})")
            elif is_required:
                params_list.append(f"{pname}({ptype}, required)")
            else:
                params_list.append(f"{pname}({ptype}, optional)")
        
        lines.append(f"{name}: {', '.join(params_list)}")
    
    return "\n".join(lines)
```

### 18.4 _call_llm_with_summary()：每轮LLM调用的消息变换核心

**此方法是理解整个messages机制的关键**。每轮LLM调用，`conversation_history`不直接发送给LLM，而是经过此方法变换后组装为`assembled_messages`。

#### 18.4.1 变换流程（react_agent_mixin.py L294-457）

```

    ↓

    - 反向查找最后一条role=user的消息作为last_message
    - 其余消息(含system/assistant/system[Observation])作为history_dicts
    ↓
步骤2: 合并 temp_history (L324-326)
    - 若有temp_history(流式chunk临时历史)，追加到history_dicts尾部
    ↓
步骤3: 策略选择 (L328-337)
    - adapter.ensure_capability() → strategy_method = "text"/"tools"/"response_format"
    ↓
步骤4: 工具信息注入 — 仅非FC模式 (L339-363)
    - strategy_method != "tools" 时:
      - 构建工具内容: "【已加载工具（完整）】...\n\n【其他可用工具（概要）】..."
      - 插入到history_dicts中第一个非system消息之前
    ↓

    - "【已执行工具(勿重复)】tool1→success; tool2→success..."
    - 以role=system追加到history_dicts尾部
    ↓

    - strategy_method == "text" 时:

      - 以role=system追加到history_dicts尾部
    ↓
```

#### 18.7.4 response_format的约束力分析

| 约束项 | response_format能做到 | response_format做不到 |
|--------|---------------------|---------------------|
| 输出必须是JSON | ✅ `type: json_schema` | - |
| thought/reasoning必须有 | ✅ `required: [...]` | - |
| tool_name必须是枚举值之一 | ✅ `enum: [...]` | - |
| tool_params必须是object | ✅ `type: object` | - |
| 不同tool_name对应不同params | - | ❌ **OpenAI API不支持条件式schema** |
| 参数名正确性保证 | - | ❌ 仍依赖注入的prompt文本(【已加载工具】) |

**关键限制**：`tool_params`在response_format中只能是`type: object`泛型，无法按tool_name区分不同参数结构。参数名约束仍然依赖注入的【已加载工具（完整）】等文本。

#### 19.1.3 核心问题定位

**问题1：conversation_history操作散落（最严重）**

```
base_react.py 中的 9 处 conversation_history.append:
  ① L368: 初始化 → append(system: sys_prompt)
  ② L369: 初始化 → append(user: task_prompt)
  ③ L520: chunk无工具Agent → append(assistant: chunk_buffer)
  ④ L535: chunk连续阈值 → append(assistant: chunk_buffer)
  ⑤ L560: answer/implicit → append(assistant: chunk_buffer)
  ⑥ L619: thought_only → append(assistant: response)
  ⑦ L691: action前flush → append(assistant: chunk_buffer)
  ⑧ L827: action后 → append(assistant: response)
  ⑨ L1465: _add_observation_to_history → append(system: [Observation])
```

**无统一入口**：每个分支直接操作 `self.conversation_history`，没有统一的 `add_message(role, content)` 方法，导致：
- 截断/前缀/去重逻辑只在 `_add_observation_to_history` 中做，assistant消息追加时没做
- 无法在统一入口做日志/审计/校验
- **trim_history()只在3处调用**（L622 thought_only后、L1020 action末尾、L1466 _add_observation_to_history内），其他assistant append(①③④⑤⑦⑧)均未调trim，存在历史无限增长风险

**问题2：_call_llm_with_summary()职责过载（164行，6种职责）**

```

  职责1: conversation_history拆分(L306-322)   17行

  职责3: 策略选择(L328-337)                   10行

  职责5: Schema注入(L381-390)                 8行

  = 9种职责，164行
```

**问题3：observation_text构建嵌在action分支中（82行）**

observation_text的构建逻辑（success/warning/error + next_actions + quality_check + 缓存/失败计数 + 已执行汇总更新）全嵌在 `run_stream()` 的action分支中，与循环控制流混杂，难以独立测试。

**问题4：三策略的响应解析重复模式**

```

```

三者共享 `dict_list_to_messages` 转换，但各自独立实现响应解析，没有统一的解析接口。

### 19.3 重构方案：新增 `message_builder.py`（推荐方案）

```python
# Author: 小沈 - 2026-05-20

class MessageBuilder:
    """
    Prompt/Message组装的统一入口
    
    将分散在 base_react.py 和 react_agent_mixin.py 中的
    conversation_history操作、observation构建、消息注入逻辑集中管理。
    
    设计原则：
    - 所有conversation_history的写操作都通过此类
    - observation_text构建独立可测试
    - 消息注入(工具/Schema/汇总)独立可测试
    
    【生命周期与会话绑定说明 — 小沈 2026-05-20 基于评审建议修正】：
    - MessageBuilder 实例生命周期必须与 Agent 实例（或单次 run_stream 会话）强绑定（作为 self.message_builder 实例属性）。
    - 严禁将其设计为模块级全局单例或类静态属性，以防止多会话并发执行时发生状态交叉污染。
    """
    
    def __init__(self, max_context_chars: int = 80000):
        self.conversation_history: List[Dict] = []
        self.temp_history: List[Dict] = []
        self._executed_tool_summary: List[str] = []
        self._failed_attempts: Dict[str, int] = {}
        self._executed_cache: Dict[str, Any] = {}
        self._cache_timestamps: Dict[str, float] = {}
        self.MAX_CONTEXT_CHARS = max_context_chars
    
    # ===== 第一组：conversation_history写操作（统一入口）=====
    
    def init_history(self, sys_prompt: str, task_prompt: str) -> None:
        """初始化conversation_history — 替代base_react.py L368-369"""
        self.conversation_history = [
            {"role": "system", "content": sys_prompt},
            {"role": "user", "content": task_prompt}
        ]
    
    def add_assistant(self, content: str) -> None:
        """追加assistant消息 — 替代8处散落的conversation_history.append
        
        【注意】当前代码中只有thought_only(L622)和observation间接(L1466)调了_trim_history，
        其他assistant append均未调trim。此处统一调trim_history()是**新增行为**，
        修复了当前代码中assistant消息可能导致历史无限增长的隐患。
        """
        self.conversation_history.append({"role": "assistant", "content": content})
        self.trim_history()
    
    def add_observation(self, observation_text: str) -> None:
        """追加observation消息 — 替代_add_observation_to_history"""
        # 智能截断
        budget = self._get_observation_budget()
        if len(observation_text) > budget:
            observation_text = self._smart_truncate(observation_text, budget=budget)
        # 统一[Observation]前缀
        observation_text = self._normalize_observation_prefix(observation_text)
        # 追加(role=system)
        self.conversation_history.append({"role": "system", "content": observation_text})
        self.trim_history()
    
    def add_parse_error(self, error_msg: str) -> None:
        """追加解析错误到history — 替代base_react.py L643"""
        self.add_observation(f"Parse Error: {error_msg}. Please ensure your response follows the ReAct format.")
    
    def flush_temp_to_history(self, chunk_buffer: str) -> None:
        """将chunk_buffer从temp_history刷入正式history — 替代L559-562/L689-693"""
        self.temp_history.clear()
        if chunk_buffer:
            self.add_assistant(chunk_buffer)
    
    # ===== 第二组：observation_text构建（独立可测试）=====
    
    def build_observation_text(
        self,
        tool_name: str,
        tool_params: dict,
        execution_result: dict,
        observation_prefix: str = "",
        alt_hint: str = ""
    ) -> str:
        """
        根据工具执行结果构建observation_text — 替代base_react.py L829-910
        
        独立于ReAct循环，可单独测试。
        """
        exec_status = execution_result.get('status', 'unknown')
        
        if exec_status == 'success':
            text = self._build_success_observation(execution_result)
        elif exec_status == 'warning':
            text = self._build_warning_observation(execution_result)
        else:
            text = self._build_error_observation(tool_name, tool_params, execution_result, alt_hint)
        
        # next_actions附加
        text = self._append_next_actions(text, execution_result)
        # 前缀(缓存命中/失败拦截)
        if observation_prefix:
            text = observation_prefix + text
        
        return text
    
    def _build_success_observation(self, result: dict) -> str:
        """success状态observation构建"""
        text = f"Observation: success - {result.get('summary', '')}"
        llm_data = result.get('llm_data')
        if llm_data:
            text += "\n" + self._format_llm_data(llm_data)
        elif result.get('data'):
            text += "\n" + self._format_data(result.get('data'))
        return text
    
    def _build_warning_observation(self, result: dict) -> str:
        """warning状态observation构建"""
        text = f"Observation: warning - {result.get('summary', '')}"
        if result.get('data'):
            text += f"\n部分数据: {result.get('data')}"
        return text
    
    def _build_error_observation(self, tool_name: str, tool_params: dict, result: dict, alt_hint: str) -> str:
        """error/timeout/permission_denied状态observation构建"""
        text = f"Observation: {result.get('status', 'error')} - {result.get('summary', '')}"
        error_detail = result.get('error', '')
        if error_detail and str(error_detail) != str(result.get('summary', '')):
            text += f"\n异常详情: {str(error_detail)[:300]}"
        if tool_params:
            text += f"\n调用参数: {str(tool_params)[:200]}"
        if alt_hint:
            text += f"\n{alt_hint}"
        return text
    
    # ===== 第三组：每轮LLM调用的消息组装（从_call_llm_with_summary拆出）=====
    
    def split_history_for_llm(self) -> Tuple[str, List[Dict]]:
        """
        将conversation_history拆分为last_message和history_dicts
        — 替代react_agent_mixin.py L306-322
        """
        last_user_idx = -1
        for i in range(len(self.conversation_history) - 1, -1, -1):
            if self.conversation_history[i].get("role") == "user":
                last_user_idx = i
                break
        
        if last_user_idx >= 0:
            last_message = self.conversation_history[last_user_idx]["content"]
            history_dicts = (self.conversation_history[:last_user_idx] 
                           + self.conversation_history[last_user_idx+1:])
        else:
            last_message = self.conversation_history[-1]["content"]
            history_dicts = self.conversation_history[:-1]
        
        return last_message, history_dicts
    
    def merge_temp_history(self, history_dicts: List[Dict]) -> List[Dict]:
        """合并temp_history — 替代react_agent_mixin.py L324-326"""
        if self.temp_history:
            return list(history_dicts) + list(self.temp_history)
        return history_dicts
    
    def inject_tools_info(
        self, 
        history_dicts: List[Dict], 
        tools_content: str
    ) -> List[Dict]:
        """
        注入工具信息到history_dicts — 替代react_agent_mixin.py L339-363
        在第一个非system消息前插入
        """
        if not tools_content:
            return history_dicts
        tools_msg = {"role": "system", "content": tools_content}
        insert_pos = 0
        for i, msg in enumerate(history_dicts):
            if msg.get("role") != "system":
                insert_pos = i
                break
        else:
            insert_pos = len(history_dicts)
        return list(history_dicts[:insert_pos]) + [tools_msg] + list(history_dicts[insert_pos:])
    
    def inject_executed_summary(self, history_dicts: List[Dict]) -> List[Dict]:
        """注入已执行工具汇总 — 替代react_agent_mixin.py L365-371"""
        if not self._executed_tool_summary:
            return history_dicts
        done_tools = [s for s in self._executed_tool_summary if '→success' in s]
        if not done_tools:
            return history_dicts
        progress = ("【已执行工具(勿重复)】" + "; ".join(done_tools[-8:]) 
                   + "\n注意：上述工具已成功执行，结果已在Observation中，禁止再次调用！")
        return list(history_dicts) + [{"role": "system", "content": progress}]
    
    def inject_schema_text(self, history_dicts: List[Dict], schema_text: str) -> List[Dict]:
        """注入Schema文本 — 替代react_agent_mixin.py L381-390"""
        if not schema_text:
            return history_dicts
        return list(history_dicts) + [{"role": "system", "content": schema_text}]
    
    def assemble_final_messages(
        self,

        last_message: str
    ) -> List[Dict]:
        """组装最终发给LLM的messages — 替代react_agent_mixin.py L394-395"""
        return list(history_dicts) + [{"role": "user", "content": last_message}]

    def trim_history(self) -> None:
```

```python
# ===== 在 base_react.py 中新增，约80行 =====

def _build_observation_text(execution_result: dict, tool_name: str = "") -> str:
    """
    根据工具执行结果构建LLM observation文本。
    替代 run_stream() action分支中的 observation 构建逻辑（原L831-910）。
    
    工具层统一返回格式后，observation构建简化为：
      - SUCCESS → message + （llm_data优先于data）
      - WARNING_ → message + 部分数据
      - ERR_*   → [code] + message
    """
    import json
    code = execution_result.get("code", "SUCCESS")
    message = execution_result.get("message", "")
    display_data = execution_result.get("llm_data") or execution_result.get("data")
    
    if code == "SUCCESS":
        text = f"Observation: success - {message}"
        if execution_result.get("warning"):
            text += f"\n⚠ 警告: {execution_result['warning']}"
        if display_data:
            text += f"\n数据: {json.dumps(display_data, ensure_ascii=False)}"
        return text
    elif code.startswith("WARNING_"):
        text = f"Observation: warning - {message}"
        if display_data:
            text += f"\n部分数据: {json.dumps(display_data, ensure_ascii=False)}"
        return text
    else:
        return f"Observation: error [{code}] - {message}"


def _add_to_history(history: list, role: str, content: str, trim_fn=None) -> list:
    """
    统一的消息追加入口。
    替代9处散落的 conversation_history.append。
    
    调用方传入 trim_fn 回调，由调用方决定裁剪策略（不自动trim）。
    """
    history.append({"role": role, "content": content})
    if trim_fn:
        trim_fn()
    return history


def _split_last_user(history: list) -> tuple:
    """
    从 conversation_history 中分离最后一条 user 消息。
    替代 _call_llm_with_summary 中的拆分逻辑（L310-322）。
    
    返回: (last_message, history_dicts)
    """
    last_user_idx = -1
    for i in range(len(history) - 1, -1, -1):
        if history[i].get("role") == "user":
            last_user_idx = i
            break
    
    if last_user_idx >= 0:
        last_message = history[last_user_idx]["content"]
        history_dicts = history[:last_user_idx] + history[last_user_idx+1:]
    else:
        last_message = history[-1]["content"]
        history_dicts = history[:-1]
    
    return last_message, history_dicts
```

### 19.7 方案总结（更新版 — 小沈 2026-05-20）

**结论：prompt/message组装逻辑确实分散严重，有必要集中化。**

**推荐方案：新增 `message_builder.py`，将8组逻辑集中管理**（已融入19.8审查修正）：

1. **conversation_history写操作**（9处append → 统一入口 init_history/add_assistant/add_observation，**add_assistant不自动trim**）
2. **observation_text构建**（主工具80行+并行工具25行 → 统一 `build_observation_text(is_parallel)` 消除重复）
3. **observation截断辅助**（常量+budget+smart_truncate → MessageBuilder内聚）
4. **历史裁剪**（_trim_history + 6个辅助方法共约155行 → MessageBuilder内聚，集成延迟裁剪，**由_call_llm_with_summary每轮统一调度**）
5. **每轮LLM调用的消息组装**（9种职责163行 → 6个独立方法+精简调度约65行）
6. **Schema文本生成+缓存管理**（_tools_to_schema_text + invalidate_cache → MessageBuilder内聚）
7. **缓存/失败计数管理**（主工具+并行工具重复代码 → 统一入口）
8. **格式化辅助**（_format_llm_data → MessageBuilder内聚）

**前置修复**：`ResponseFormatStrategy.call()` return缺失 + `load_tools_by_intent()` 缓存清除不完整（均已修复）。

**不改的部分**：BasePromptTemplate.py（模板层）、llm_core.py（传输层）、llm_strategies.py（策略层保持独立）、各xxx_prompts.py（内容层）。

**实施方式**：渐进式3步走，每步含精确变更清单+行为变化分析+验证方案+回滚方案，每步可独立验证，不破坏现有功能。

**3项审查修正已融入**：
- **修正1**：add_assistant不自动trim → 改为_call_llm_with_summary每轮统一调度（第1步1-A/1-B/1-C + 第2步2-B）
- **修正2**：验证方案补全 → 第1步1-D增加6项验证+安全网grep
- **修正3**：conversation_history按4类分类替换+属性透传安全网 → 第1步1-B⑨
