## 一、🔴 HIGH 严重问题

### H1. `parse_react_response` JSON预解析路径缺chunk处理

| 字段 | 值 |
|------|-----|
| **文件** | `react_output_parser.py:154-202` |
| **发现时间** | 2026-05-13 |
| **引入者** | 小健发现（小沈Phase 1代码） |

**问题描述**：
`TextStrategy.call()` 遇到chunk类型时返回 JSON `{"type": "chunk", "tool_name": null, ...}`。该JSON进入 `parse_react_response()` 的JSON预解析路径（第146-227行）。该路径显式处理了 `type="answer"`（第167行）和 `type="parse_error"`（第156行），但**没有处理 `type="chunk"`**。当 `explicit_type` 不是上述两种时，代码走到第179行检查 `"tool_name" in data`——由于 `data` 中有 `tool_name: null`，此检查为 True，于是返回 `type="action"` 且 `tool_name=None`。`base_react.py` 收到 `type="action"` 后尝试执行 `_execute_tool(None, {})`，进入死循环或崩溃。

**影响路径**：`_GenericAgent`（通用兜底Agent）→ `TextStrategy.call()` → `parse_react_response()` → JSON预解析路径→ 错误识别为action

**修复方案**：
在JSON预解析路径中增加 `type="chunk"` 的显式处理：

```python
if explicit_type == "chunk":
    return {
        "type": "chunk",
        "thought": data.get("thought", ""),
        "content": data.get("content", ""),
        "reasoning": data.get("reasoning", ""),
        "tool_name": None,
        "tool_params": None,
        "response": data.get("response", data.get("content", "")),
        "error": None
    }
```

**风险**：低。新增分支不影响已有行为，只让chunk类型不被误判为action。

### H2. `chat_router.py` 与 `react_sse_wrapper.py` 循环导入

| 字段 | 值 |
|------|-----|
| **文件** | `chat_router.py:330,380,420` + `react_sse_wrapper.py:47` |
| **发现时间** | 2026-05-13 |
| **引入者** | 既有代码 |

**问题描述**：
- `react_sse_wrapper.py:47` 从 `chat_router.py` import `CRSS_CONFIDENCE_THRESHOLD`
- `chat_router.py:330` 从 `react_sse_wrapper` import `running_tasks`, `running_tasks_lock`
- `chat_router.py:380` 从 `react_sse_wrapper` import `generate_sse_stream`
- `chat_router.py:420` 从 `react_sse_wrapper` import `cancel_task`, `pause_task`, `resume_task`

两个模块互相导入，形成循环依赖。目前能工作是因为Python模块加载顺序恰好让 `CRSS_CONFIDENCE_THRESHOLD` 在 `chat_router.py` 执行到第330行之前就被定义了。但如果改变import顺序或重构模块，会直接引发 `ImportError`。

**修复方案（三个选项）**：

**方案A**：将 `CRSS_CONFIDENCE_THRESHOLD` 移回 `crss_scorer.py`（它本来就是从那儿来的），`react_sse_wrapper.py` 改为从 `crss_scorer` 导入，切断循环。

**方案B**：将 `chat_router.py` 中对 `react_sse_wrapper` 的导入改为**函数内部延迟导入**（lazy import），避免模块加载时的循环依赖。

**方案C**：将任务控制函数（cancel_task/pause_task/resume_task）移出 `react_sse_wrapper.py`，放到独立的模块中。

**建议**：方案A最简单，且`CRSS_CONFIDENCE_THRESHOLD`本来就定义在`crss_scorer.py`中，只需要改import来源。

### H3. `_INTENT_DEFINITIONS` key `"env"` 与 `ToolCategory.ENVIRONMENT.value` 不匹配

| 字段 | 值 |
|------|-----|
| **文件** | `intent_classifier.py:68` |
| **发现时间** | 2026-05-13 |
| **引入者** | 小沈（既有代码） |

**问题描述**：
`_INTENT_DEFINITIONS` 中使用 `"env"` 作为key（第68行），但 `ToolCategory.ENVIRONMENT.value = "environment"`。`_build_intent_prompt()` 检查 `if label in _INTENT_DEFINITIONS` 时，label `"environment"` 不匹配key `"env"`。结果是环境变量的意图描述**从未出现在LLM分类prompt中**，LLM永远不会将"查看PATH"这类请求分类为environment。

**修复方案**：
将 `_INTENT_DEFINITIONS` 第68行的 `"env"` 改为 `"environment"`：

```python
# 修改前
"env": "环境变量，包括查看/设置PATH、HOME、TEMP等系统环境变量、系统变量配置等",

# 修改后
"environment": "环境变量，包括查看/设置PATH、HOME、TEMP等系统环境变量、系统变量配置等",
```

**风险**：极低，只影响LLM分类器prompt的标签匹配。
