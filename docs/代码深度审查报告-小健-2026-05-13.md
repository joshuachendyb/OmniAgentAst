### H1. `parse_react_response` JSON预解析路径缺chunk处理

| 字段 | 值 |
|------|-----|
| **文件** | `react_output_parser.py:154-202` |
| **发现时间** | 2026-05-13 |
| **引入者** | 小健发现（小沈Phase 1代码） |

**问题描述**：
`TextStrategy.call()` 遇到chunk类型时返回 JSON `{"type": "chunk", "tool_name": null, ...}`。该JSON进入 `parse_react_response()` 的JSON预解析路径（第146-227行）。该路径显式处理了 `type="answer"`（第167行）和 `type="parse_error"`（第156行），但**没有处理 `type="chunk"`**。当 `explicit_type` 不是上述两种时，代码走到第179行检查 `"tool_name" in data`——由于 `data` 中有 `tool_name: null`，此检查为 True，于是返回 `type="action"` 且 `tool_name=None`。`base_react.py` 收到 `type="action"` 后尝试执行 `_execute_tool(None, {})`，进入死循环或崩溃。

**影响路径**：`_GenericAgent`（通用兜底Agent）→ `TextStrategy.call()` → `parse_react_response()` → JSON预解析路径→ 错误识别为action

**修复方案**：
在JSON预解析路径中增加 `type="chunk"` 的显式处理：

```python
if explicit_type == "chunk":
    return {
        "type": "chunk",
        "thought": data.get("thought", ""),
        "content": data.get("content", ""),
        "reasoning": data.get("reasoning", ""),
        "tool_name": None,
        "tool_params": None,
        "response": data.get("response", data.get("content", "")),
        "error": None
    }
```

**风险**：低。新增分支不影响已有行为，只让chunk类型不被误判为action。

### H2. `chat_router.py` 与 `react_sse_wrapper.py` 循环导入

| 字段 | 值 |
|------|-----|
| **文件** | `chat_router.py:330,380,420` + `react_sse_wrapper.py:47` |
| **发现时间** | 2026-05-13 |
| **引入者** | 既有代码 |

**问题描述**：
- `react_sse_wrapper.py:47` 从 `chat_router.py` import `CRSS_CONFIDENCE_THRESHOLD`
- `chat_router.py:330` 从 `react_sse_wrapper` import `running_tasks`, `running_tasks_lock`
- `chat_router.py:380` 从 `react_sse_wrapper` import `generate_sse_stream`
- `chat_router.py:420` 从 `react_sse_wrapper` import `cancel_task`, `pause_task`, `resume_task`

两个模块互相导入，形成循环依赖。目前能工作是因为Python模块加载顺序恰好让 `CRSS_CONFIDENCE_THRESHOLD` 在 `chat_router.py` 执行到第330行之前就被定义了。但如果改变import顺序或重构模块，会直接引发 `ImportError`。

**修复方案（三个选项）**：

**方案A**：将 `CRSS_CONFIDENCE_THRESHOLD` 移回 `crss_scorer.py`（它本来就是从那儿来的），`react_sse_wrapper.py` 改为从 `crss_scorer` 导入，切断循环。

**方案B**：将 `chat_router.py` 中对 `react_sse_wrapper` 的导入改为**函数内部延迟导入**（lazy import），避免模块加载时的循环依赖。

**方案C**：将任务控制函数（cancel_task/pause_task/resume_task）移出 `react_sse_wrapper.py`，放到独立的模块中。

**建议**：方案A最简单，且`CRSS_CONFIDENCE_THRESHOLD`本来就定义在`crss_scorer.py`中，只需要改import来源。

### H3. `_INTENT_DEFINITIONS` key `"env"` 与 `ToolCategory.ENVIRONMENT.value` 不匹配

| 字段 | 值 |
|------|-----|
| **文件** | `intent_classifier.py:68` |
| **发现时间** | 2026-05-13 |
| **引入者** | 小沈（既有代码） |

**问题描述**：
`_INTENT_DEFINITIONS` 中使用 `"env"` 作为key（第68行），但 `ToolCategory.ENVIRONMENT.value = "environment"`。`_build_intent_prompt()` 检查 `if label in _INTENT_DEFINITIONS` 时，label `"environment"` 不匹配key `"env"`。结果是环境变量的意图描述**从未出现在LLM分类prompt中**，LLM永远不会将"查看PATH"这类请求分类为environment。

**修复方案**：
将 `_INTENT_DEFINITIONS` 第68行的 `"env"` 改为 `"environment"`：

```python
# 修改前
"env": "环境变量，包括查看/设置PATH、HOME、TEMP等系统环境变量、系统变量配置等",

# 修改后
"environment": "环境变量，包括查看/设置PATH、HOME、TEMP等系统环境变量、系统变量配置等",
```

**风险**：极低，只影响LLM分类器prompt的标签匹配。

### H4. CORS配置冲突（`allow_origins=["*"]` + `allow_credentials=True`）

| 字段 | 值 |
|------|-----|
| **文件** | `main.py:62-68` |
| **发现时间** | 2026-05-13 |
| **引入者** | 小沈补充发现 |

**问题描述**：
CORS规范明确规定：当 `allow_credentials=True` 时，`allow_origins` 不能为 `"*"`。浏览器遇到这种配置会**直接拒绝**携带凭据（cookies/Authorization头）的跨域请求。实际效果是：如果后端有认证机制，前端所有需要认证的API调用全部失效。

```python
# 当前代码（错误）
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],
    allow_credentials=True,  # 与allow_origins=["*"]冲突
    allow_methods=["*"],
    allow_headers=["*"],
)
```

**影响**：所有需要携带凭据的跨域请求被浏览器拒绝，认证功能完全失效。

**修复方案**：
将 `allow_origins=["*"]` 改为明确的前端URL列表，或根据环境配置动态设置：

```python
app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:5173", "http://127.0.0.1:5173"],
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

### H5. 数据库工具SQL注入风险

| 字段 | 值 |
|------|-----|
| **文件** | `database/database_tools.py:121,226` |
| **发现时间** | 2026-05-13 |
| **引入者** | 小沈补充发现 |

**问题描述**：
`query_sql()` 和 `execute_sql()` 直接将用户传入的原始SQL传给 `cursor.execute(sql)`。`query_sql()` 的只读检查（行101）仅检查SQL是否以SELECT/SHOW/DESCRIBE等开头——在白名单模式下这是一个有效防护，但sqlite3的 `cursor.execute()` 不支持参数化查询的WHERE子句注入防护。对于MySQL/PostgreSQL（行114,221），使用 `conn.execute(sql)` 同样存在注入风险。`execute_sql()` 的DROP/TRUNCATE黑名单（行190）可被绕过（如 `DROP   TABLE` 含多余空格）。

**影响路径**：LLM生成的SQL → 直接执行 → 可读取/篡改/删除数据库
- `query_sql`：可以读取任意表数据（通过UNION、子查询等）
- `execute_sql`：可以INSERT/UPDATE/DELETE任意数据

**修复方案**：
1. `query_sql` 保留SQL白名单检查（当前实现已相对安全）
2. `execute_sql` 增加更严格的危险操作过滤，使用正则而非子串匹配
3. **长期方案**：对 `execute_sql` 增加用户确认机制（如二次确认）

```python
# execute_sql 改进：使用正则严格检查危险关键字
import re
DANGEROUS_PATTERN = re.compile(r'\b(DROP|TRUNCATE|ALTER|CREATE|GRANT|REVOKE)\b', re.IGNORECASE)
if not dry_run and DANGEROUS_PATTERN.search(sql):
    return {"code": "WARNING", ...}
```

### H6. Shell工具命令注入风险

| 字段 | 值 |
|------|-----|
| **文件** | `shell/shell_tools.py:77-84,104-112` |
| **发现时间** | 2026-05-13 |
| **引入者** | 小沈补充发现 |

**问题描述**：
`execute_shell_command()` 使用 `subprocess.Popen(command, shell=True)` 和 `subprocess.run(command, shell=True)` 执行命令（行77-84,104-112）。当 `command` 参数包含shell元字符（如 `;`, `|`, `&&`, `` ` `` 等）时，可执行非预期的附加命令。虽然当前命令由LLM生成（风险可控），但这是设计层面的安全隐患。

**修复方案**：
1. **最小化 shell=True 的使用**：优先使用 `shlex.split()` 分割命令并以列表形式传递（避免shell解释）
2. **添加命令安全检查**：在执行前检测并拦截包含管道/链接操作的命令
3. 使用 `subprocess.Popen` 时执行前做参数白名单校验

```python
import shlex

# 对简单命令尝试使用列表形式（避免shell解释）
try:
    cmd_parts = shlex.split(command)
    # 仅当不包含shell元字符时才使用列表形式
    if not any(c in command for c in [';', '|', '&&', '||', '`', '$(']):
        result = subprocess.run(cmd_parts, capture_output=True, timeout=timeout_sec)
        # 处理结果...
        return ...
except Exception:
    pass  # fallback到shell=True
```
