# 中断问题分析报告

## 问题概述
用户反馈中断功能不正常，点击中断按钮后任务无法立即停止。

## 后端问题分析

### 1. 中断检查时机问题
**文件**: `backend/app/services/react_sse_wrapper.py:444`
```python
async with running_tasks_lock:
    if running_tasks.get(task_id, {}).get("cancelled", False):
        interrupted_data = create_incident_data('interrupted', '任务已被中断', step=next_step())
        yield f"data: {json.dumps(interrupted_data)}\n\n"
        break
```

**问题**: 中断检查只在文件操作循环的每次迭代开始时进行。如果LLM调用或工具执行耗时很长，中断请求无法及时响应。

### 2. LLM流式响应中断检查间隔太长
**文件**: `backend/app/services/llm_core.py:197-205`
```python
while True:
    try:
        line = await asyncio.wait_for(response.aiter_lines().__anext__(), timeout=5.0)
    except asyncio.TimeoutError:
        # 超时了，检查取消标志
        if self._cancelled:
            logger.info("[chat_stream] 检测到取消标志（5秒超时检查），中断流式响应")
            yield StreamChunk(content="", model=self.model, is_done=True, stream_error="任务已取消", stream_error_type="cancelled")
            return
```

**问题**: 5秒的检查间隔太长，用户需要等待最多5秒才能中断。

### 3. 中断标志传播问题
**文件**: `backend/app/services/react_sse_wrapper.py:615`
```python
if "ai_service" in task_info and task_info["ai_service"]:
    ai_service = task_info["ai_service"]
    try:
        ai_service.cancel()
```

**问题**: `ai_service` 可能不是同一个实例，或者 `cancel()` 方法没有正确传播到所有相关组件。

### 4. 任务记录过早删除
**文件**: `backend/app/services/react_sse_wrapper.py:625`
```python
# 从 running_tasks 中移除任务（避免内存泄漏）
del running_tasks[task_id]
```

**问题**: 如果任务还在执行中，删除任务记录会导致后续的中断检查失败。

## 前端问题分析

### 1. 竞态条件问题
**文件**: `frontend/src/components/Chat/NewChatContainer.tsx:1867-1869`
```typescript
// ✅【方案1】立即更新UI状态，给用户即时反馈
setLoading(false);
setIsPaused(false);
if (setIsReceiving) setIsReceiving(false);
```

**问题**: 在调用 `cancel` API 之前就更新UI状态，如果API调用失败，UI状态已经改变但任务仍在运行。

### 2. 中断事件检测逻辑缺陷
**文件**: `frontend/src/components/Chat/NewChatContainer.tsx:1889-1899`
```typescript
let retries = 0;
while (retries < 3) {
    await new Promise(resolve => setTimeout(resolve, 500));
    // 检查是否收到了interrupted事件
    if (hasReceivedInterruptEventRef.current) {
        console.log("[中断] 已收到 interrupted 事件");
        break;
    }
    retries++;
    console.log("[中断] 等待 interrupted 事件... 重试", retries);
}
```

**问题**:
- 硬编码等待时间：500ms × 3次 = 最多等待1.5秒
- 没有考虑网络延迟
- 没有超时机制

### 3. hasReceivedInterruptEventRef 重置时机问题
**文件**: `frontend/src/components/Chat/NewChatContainer.tsx:1902`
```typescript
// 重置中断事件标记，为下一次中断准备
hasReceivedInterruptEventRef.current = false;
```

**问题**: 在断开连接**之前**就重置了标记，如果在断开连接过程中收到了新的 `interrupted` 事件，会被错误地忽略。

### 4. SSE事件处理问题
**文件**: `frontend/src/utils/sse.ts:1358-1364`
```typescript
case "interrupted":
    // 【小强修复 2026-04-10】添加 onShowSteps?.(true)，确保中断时步骤列表显示
    onShowSteps?.(true);
    onComplete?.(responseBufferRef.current, undefined);
    setIsReceiving(false);
    setIsConnected(false);
    break;
```

**问题**:
- 调用 `onComplete` 可能不合适
- 没有清理 `hasReceivedInterruptEventRef`
- 没有通知前端中断已完成

**文件**: `frontend/src/components/Chat/NewChatContainer.tsx`

```typescript
// 添加等待函数
const waitForInterruptEvent = async (maxWaitTime = 3000, checkInterval = 200) => {
    const startTime = Date.now();
    let hasReceivedEvent = false;
    
    while (Date.now() - startTime < maxWaitTime) {
        if (hasReceivedInterruptEventRef.current) {
            console.log("[中断] 已收到 interrupted 事件");
            hasReceivedEvent = true;
            break;
        }
        await new Promise(resolve => setTimeout(resolve, checkInterval));
    }
    
    if (!hasReceivedEvent) {
        console.warn(`[中断] 在 ${maxWaitTime}ms 内未收到 interrupted 事件，继续执行`);
    }
    
    return hasReceivedEvent;
};

// 在中断处理中使用
const receivedEvent = await waitForInterruptEvent(3000, 200);
```
