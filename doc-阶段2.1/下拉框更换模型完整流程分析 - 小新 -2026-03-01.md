## 一、流程概述

下拉框更换模型功能涉及前后端多个组件的协作，包含 **13 个关键步骤**，每一步都必须正确执行才能确保模型切换成功。

---

## 二、完整流程图

```
用户点击下拉框选择模型
   ↓
【第 1 步】前端 Select onChange 触发
   ↓
【第 2 步】从 modelList 查找 selectedModel
   ↓
【第 3 步】调用 API 切换模型
   ↓
【第 4 步】后端处理配置更新（7 个子步骤）
   ↓
【第 5 步】前端更新选中状态
   ↓
【第 6 步】前端自动调用 handleCheckService
   ↓
【第 7 步】handleCheckService 刷新配置验证
   ↓
【第 8 步】handleCheckService 刷新模型列表
   ↓
【第 9 步】后端 getModelList 确定 current_model
   ↓
【第 10 步】前端更新当前选中的模型
   ↓
【第 11 步】handleCheckService 验证服务可用性
   ↓
【第 12 步】后端 validate_ai_service 验证（9 个子步骤）
   ↓
【第 13 步】前端显示验证结果
   ↓
完成
```

## 三、详细步骤分析

### 第 1 步：前端 Select onChange 触发

**文件**: `frontend/src/components/Layout/index.tsx`  
**行号**: 610-640

**代码**:
```typescript
<Select
  value={_currentProvider}
  onChange={async (value: string) => {
    // value 格式："${provider}-${model}"
    // 示例："opencode-kimi-k2.5-free"
  }}
>
  {modelList.map((model) => (
    <Option
      key={model.id}
      value={`${model.provider}-${model.model}`}
    >
      {model.display_name}
    </Option>
  ))}
</Select>
```

**关键点**:
- ✅ Option value 必须使用 `${provider}-${model}` 格式
- ✅ 不能只使用 model.id 或 model.name
- ✅ 原因：同一个 model 名称可能属于多个 provider

## 五、常见问题诊断

### 问题 1：切换模型后配置文件未更新

**症状**:
- 调用 PUT /config 返回成功
- 但配置文件中的 ai.model 未更新

**可能原因**:
1. 后端缓存未清空
2. 配置文件写入失败

**解决方法**:
```python
# 确保清空缓存
AIServiceFactory._instance = None
AIServiceFactory._config = None
```

---

### 问题 2：前端显示的选中模型与后端不一致

**症状**:
- 前端切换到 kimi-k2.5-free
- 后端返回的 current_model 仍是 minimax-m2.5-free

**可能原因**:
1. 配置文件未正确更新
2. getModelList 读取旧配置
3. current_model 标记逻辑错误

**解决方法**:
1. 检查配置文件是否更新
2. 检查缓存是否清空
3. 检查 final_provider 和 final_model 是否正确

---

### 问题 3：备份文件未删除

**症状**:
- 验证成功后，备份文件仍然存在

**可能原因**:
1. validate_ai_service 未获取到 backup_path
2. _delete_backup_by_path 未正确执行

**解决方法**:
```python
# 确保设置全局备份路径
AIServiceFactory.set_backup_paths(str(backup_path), str(config_path))

# 确保验证成功后删除备份
if is_valid and backup_path:
    await _delete_backup_by_path(backup_path)
```

## 六、设计原则

### 原则 1：provider 和 model 必须成对使用

**原因**:
- 同一个 model 名称可能属于多个 provider
- 只有 provider+model 组合才能唯一确定一个模型

**应用**:
- 发送时：ai_provider + ai_model
- 接收时：ai_provider + ai_model
- 使用时：两个字段必须同时使用

---

### 原则 2：配置更新必须备份

**原因**:
- 配置更新可能失败
- 需要回滚到更新前的状态

**流程**:
1. 更新前备份
2. 更新配置
3. 验证成功 → 删除备份
4. 验证失败 → 恢复备份

---

### 原则 3：缓存必须及时清空

**原因**:
- 配置更新后，缓存可能仍是旧数据
- 导致后续读取错误

**时机**:
- 配置更新后
- 切换模型后
