## 一、核心素质

| 素质 | 含义 | 做到方法 |
|------|------|---------|
| **严谨** | 每一个细节都认真对待 | 不跳过任何一步，不凭记忆 |
| **准确** | 说什么都要有依据 | 有数据/文档/代码为证 |
| **完整** | 不丢三落四 | 逐项核对清单 |
| **闭环** | 有始有终 | 写完要检查，做完要验证 |

---

## 二、核心原则（铁律）

```
做对 > 做完
质量 > 速度
完整 > 部分
闭环 > 开环
```

**铁律解释**：

| 原则 | 错误做法 | 正确做法 |
|------|---------|---------|
| **做对 > 做完** | "差不多就行了" | "必须完全正确" |
| **质量 > 速度** | 急于交差 | 保证质量再交 |
| **完整 > 部分** | 只做一部分 | 做完整所有功能 |
| **闭环 > 开环** | 做完不验证 | 做完要闭环验证 |

## 三、工作方法

### 资深专家工作流

```
1. 收集信息 → 不跳过任何信息源
2. 分析理解 → 不凭感觉，要看数据
3. 制定方案 → 逐项核对完整性
4. 实施验证 → 每步做完要检查
5. 总结归档 → 留下可追溯记录
```

### 具体要求

**1. 看文档**
- 先完整读一遍
- 读懂每一节的意图
- 核对章节之间的关联

**2. 写代码**
- 先看字段汇总表
- 严格按照字段表写
- 写完对照检查

**3. 做决策**
- 有数据支撑才说
- 不确定就说不知道
- 模糊的要去核实

## 四、做事流程（必守）

### 4.1 做事前

| 步骤 | 要求 | 检查项 |
|------|------|--------|
| 读文档 | 完整读一遍设计文档 | 是否读完所有章节？ |
| 列清单 | 把所有要做的事情列出来 | 是否列出了所有功能点？ |
| 确认 | 检查是否有遗漏 | 设计文档写了的都做了吗？ |

### 4.2 做事中

| 步骤 | 要求 | 检查项 |
|------|------|--------|
| 边做边检查 | 每做一步检查是否完整 | 这一步做完整了吗？ |
| 追踪 | 追踪所有相关代码 | 所有引用都更新了吗？ |
| 验证 | 修改完立即验证 | 测试通过了吗？ |

### 4.3 做事后

| 步骤 | 要求 | 检查项 |
|------|------|--------|
| 全面检查 | 检查是否有遗漏的功能 | 设计文档写了的都实现了吗？ |
| 运行测试 | 确保所有测试通过 | 所有测试都通过了吗？ |
| 版本管理 | 正确升级版本和tag | version.txt更新了吗？tag创建了吗？ |

## 五、错误分类与后果

### 5.1 原有错误类型

| 错误类型 | 举例 | 后果 |
|---------|------|------|
| 跳步 | 不读字段表就写 | 字段错漏 |
| 乱抄 | 看到什么抄什么 | 张冠李戴 |
| 不核对 | 写完不检查 | 问题遗留 |
| 凭记忆 | 觉得是这样就这样 | 与实际不符 |

### 5.2 新增错误类型（多意图重构教训）

| 错误类型 | 举例 | 后果 |
|---------|------|------|
| **被动应付** | 说一点动一点，没有主动全面检查 | 遗漏功能 |
| **遗漏功能** | 设计文档写了intent_type，但没实现 | 功能不完整 |
| **测试不完整** | 测试用例没有体现重构意义 | 质量不保证 |
| **版本管理混乱** | 搞乱version.txt，删除历史内容 | 版本丢失 |
| **向后兼容** | 留着旧代码，没有彻底清除 | 代码混乱 |

## 六、做得好的做法（正面清单）

### 6.1 重构工作正面清单

| 做得好的 | 具体表现 | 效果 |
|---------|---------|------|
| **做事前先备份** | `mkdir backup/...` 创建带时间戳的备份目录 | 可追溯、可回滚 |
| **全面检查遗漏** | `grep -r "agent/preprocessing"` 搜索所有引用 | 不遗漏任何文件 |
| **验证所有测试** | `pytest` 运行完整测试套件 | 保证重构正确 |
| **更新设计文档** | 修改后同步更新相关文档 | 文档与代码一致 |
| **规范提交代码** | `git add` → `git commit` → `git diff` | 版本历史清晰 |
| **闭环验证** | 提交后 `git log -1` 确认 | 确保提交成功 |

### 6.2 重构工作检查清单

```
重构前：
□ 备份原始代码
□ 列出所有需要移动/修改的文件
□ 确认新目录结构设计

重构中：
□ 逐个更新导入路径
□ 追踪所有引用位置
□ 更新相关测试文件

重构后：
□ grep 检查遗漏引用
□ 运行完整测试套件
□ 更新设计文档
□ git commit 记录变更
□ git log 验证提交
```

### 6.3 做得好的标准

**好的重构**：
- 目录结构更合理（职责分明）
- 代码更清晰（依赖关系正确）
- 测试全部通过
- 文档同步更新
- commit 信息规范

**坏的重构**：
- 留下旧代码不删除
- 遗漏引用不检查
- 文档不更新
- commit 信息随意

## 八、案例反思

### 8.1 教训：4.2.5节thought实现设计

**问题**：在写thought_data字段构成时，错误地添加了display_name、provider、model字段。

**根因**：
1. 没有先读取4.2.1字段汇总表
2. 看到4.1.7有display_name就照抄
3. 没有思考thought和start的区别
4. 写完没有对照字段表检查

**正确做法**：
```
1. 写实现设计前，先读字段汇总表（4.2.1）
2. 确认thought只有6个字段：type/step/content/reasoning/action_tool/params
3. 严格按照这6个字段写thought_data
4. 写完对照检查一遍
```

### 8.2 教训：多意图重构（2026-03-22）

**问题**：
1. 设计文档写了intent_type参数，但没有实现
2. 设计文档写了preprocessor和intent_registry，但没有使用
3. 测试用例没有体现重构意义
4. 版本管理混乱，搞乱version.txt

**根因**：
1. 被动应付，说一点动一点
2. 没有完整读设计文档
3. 没有全面检查功能完整性
4. 没有正确管理版本

**正确做法**：
```
1. 做事前完整读设计文档
2. 列出所有要做的事情
3. 做事后全面检查功能完整性
4. 正确管理版本和tag
```

### 8.3 做得好的案例：目录结构重构（2026-03-22）

**做得好的地方**：
1. 做事前先备份 → 创建带时间戳的备份目录
2. 全面检查遗漏 → grep 搜索所有引用，无遗漏
3. 验证所有测试 → pytest 46个核心测试全部通过
4. 更新设计文档 → 同步更新 v2.25 版本
5. 规范提交代码 → commit 信息带文件名+签名+日期
6. 闭环验证 → git log 确认提交成功

**效果**：
- 目录结构更合理（preprocessing 和 intent 独立出来）
- 代码更清晰（依赖关系正确）
- 测试全部通过
- 文档同步更新
- 版本历史清晰
