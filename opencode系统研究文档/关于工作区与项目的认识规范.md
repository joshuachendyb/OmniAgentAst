## 三、AI 的理解和来源

### 3.1 工作区

#### 来源
```
<env>
  Working directory: D:\2bktest\MDview
</env>
```
**来源依据**：工具执行命令时的默认目录

#### 理解定义
- **技术视角**：Bash 命令默认执行的物理目录
- **特性**：
  - 工具执行命令时有默认目录 → 我叫它"工作区"
  - 如果未指定 `workdir` 参数，命令默认在此目录执行
  - 可能包含多个项目

#### 示例
```
命令1: ls                    → 在 D:\2bktest\MDview 执行
命令2: ls -la MermaidReader → 在 D:\2bktest\MDview 执行，列出子目录
命令3: ls workdir="D:\2bktest\MDview\MermaidReader" → 在项目目录执行
```

### 3.2 项目

#### 来源
```
用户描述：
- "MermaidReader 项目"
- 文档中 "# MermaidReader 应用程序架构设计"
```
**来源依据**：用户将某个目录视为完整的软件工程

#### 理解定义
- **逻辑视角**：一个完整的、独立的软件工程或功能单元
- **特性**：
  - 有明确的业务目标和技术栈
  - 包含源代码、配置文件、文档等完整资源
  - 可以独立开发、测试、部署

#### 示例
```
项目示例：
- MermaidReader/     → MD 文件阅读器（独立的软件工程）
- （待定）MermaidEditor/ → MD 文件编辑器（如果存在，是另一个项目）
```

## 四、AI 提供的判断标准

### 4.1 判断新建项目 vs 旧项目的标准

#### 1. 技术栈是否兼容
```
技术栈相同 → 旧项目
技术栈不同 → 新项目

例：
- 现有项目：React + TypeScript
- 新需求：React + TypeScript → 旧项目
- 新需求：Vue + JavaScript → 新项目
```

#### 2. 业务领域是否相关
```
业务相关 → 旧项目
业务独立 → 新项目

例：
- MermaidReader（阅读MD文件）
- 编辑器功能 → 旧项目（属于MD工具生态）
- 游戏应用 → 新项目（完全无关）
```

#### 3. 技术架构是否支持
```
架构支持 → 旧项目
不兼容 → 新项目

例：
- 当前：纯前端应用
- 需求：添加后端API
  - 微服务架构 → 新建后端项目
  - 全栈应用 → 扩展现有项目
```

#### 4. 功能耦合度
```
强耦合 → 旧项目
弱耦合/独立 → 新项目

例：
- MermaidReader 添加深色模式 → 旧项目（UI功能）
- 独立的MD转PDF工具 → 新项目（独立工具）
```

#### 5. 维护成本
```
旧项目扩展：需要修改核心代码 → 考虑新建
旧项目扩展：只需添加新模块 → 旧项目
```

### 4.2 判断流程

```
开始
  ↓
技术栈兼容？─否→新建项目
  ↓ 是
业务相关？─否→新建项目
  ↓ 是
架构支持？─否→新建项目
  ↓ 是
功能耦合？─弱→考虑新项目
  ↓ 强
维护成本？─高→考虑新项目
  ↓ 低
→ 扩展现有项目
```

### 4.3 快速决策三问
1. **技术栈一样吗？** 不一样 → 新项目
2. **业务相关吗？** 不相关 → 新项目
3. **能做成模块吗？** 太耦合 → 新项目

三个答案都是 "可以" → 旧项目
