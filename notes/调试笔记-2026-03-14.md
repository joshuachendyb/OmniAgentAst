## 2 历史教训：函数签名修改必须同步更新所有调用方 【小查追加】2026-03-14 20:48:20

### 2.1 问题发生经过

**时间线**：
- **2026-03-12 15:54:52**：小沈提交commit `800f0fd27`，修改`onComplete`回调参数
- **2026-03-12 - 2026-03-13**：功能看似正常（因为executionStepsRef有数据）
- **2026-03-14**：用户发现刷新后思考过程丢失，问题暴露

### 2.2 错误修改详情

**修改前（正常工作）**：
```typescript
// 参数定义
executionStepsFromSSE?: ExecutionStep[]

// 使用
const stepsToSave = executionStepsFromSSE || executionStepsRef.current || [];
```

**修改后（引入BUG）**：
```typescript
// 参数定义被改为对象
sseData?: {
  execution_steps?: ExecutionStep[];
  [key: string]: any;
}

// 使用
const stepsFromSSE = sseData?.execution_steps;  // ❌ 永远是undefined！
```

### 2.3 为什么调用方没改？

**sse.ts第716行调用代码**：
```typescript
onComplete?.(
  responseBufferRef.current, 
  metadata, 
  [...handlers.getCurrentExecutionSteps(), step]  // ← 传递的是ExecutionStep[]数组！
);
```

**问题**：调用方仍然是传递数组，但接收方期望的是对象，导致类型不匹配。

### 2.4 为什么现在才暴露？

| 场景 | 结果 | 原因 |
|------|------|------|
| 实时显示 | ✅ 正常 | executionStepsRef.current有数据 |
| 导出JSON | ✅ 正常 | 从内存导出，有完整数据 |
| 刷新后恢复 | ❌ 丢失 | 数据库保存时execution_steps为undefined |

**根本原因**：`stepsFromSSE`为`undefined`时，fallback到`executionStepsRef.current`，但保存到数据库时可能时机不对，导致数据未保存。

### 2.5 核心教训

**铁律：修改函数签名时必须同步修改所有调用方！**

**检查清单**：
- [ ] 修改函数参数前，先找到所有调用位置
- [ ] 使用IDE的"Find Usages"功能确保找到所有调用
- [ ] 同步修改所有调用方的传参方式
- [ ] 修改后进行端到端测试，验证数据流完整
- [ ] 特别关注数据库保存/加载的数据完整性

**预防措施**：
1. 使用TypeScript严格模式，类型不匹配会编译报错
2. 函数修改后进行全局搜索，确保没有遗漏
3. 建立代码审查机制，函数签名修改需要双人确认
4. 修改后必须测试数据持久化（保存/加载）
