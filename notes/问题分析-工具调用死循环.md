# 问题分析：工具调用死循环（"检查E盘"重复调同个工具）

**创建时间**: 2026-05-23 12:52:34  
**编写人**: 小沈  
**问题现象**: 用户说"检查E盘目录情况"，模型第1次正常调 `list_directory("E:/")` 拿到结果，
后第2~6次持续调同一个工具+参数，直到客户端断开。中间每次都是"缓存命中"被跳过执行。

---

## 一、根因

**Observation 工具执行结果用 `role: system` 注入，不是标准的 `role: tool`。**

### 1.1 当前对话结构

发给模型的 messages 如下（取第3轮为例）：

```
system: [系统提示词]                              ← 指令
system: [已加载工具完整信息]                        ← 指令
assistant: {"tool_name":"list_directory",...}     ← 模型第1次回答
system: [Observation] success - 7项               ← 第一次执行结果
assistant: {"tool_name":"list_directory",...}     ← 模型第2次回答（又调一次）
system: [缓存命中] 禁止再次调用...                  ← 第二次被缓存
system: 【已执行工具(勿重复)】                      ← 指令：别重复
system: [Tools Schema参考]                         ← 指令：参数格式
user: Task: 检查E盘目录情况...                      ← 任务
```

### 1.2 问题在哪

`role: system` 的语义是 **指令/上下文**，但工具执行结果不是指令——它是 **数据**。

模型看到的是一堆 `system`（指令）+ 一条 `user`（任务）。模型会想：
> "一堆指令之后，用户又发了同样的任务——说明任务还没完成，我得继续调工具。"

如果 observation 用 **`role: tool`**（OpenAI 标准角色，专门表示工具执行结果）：

```
assistant: {"tool_name":"list_directory",...}
tool: [Observation] success - 7项                 ← 数据来了
```

模型天然知道"数据已经有了，该总结了"——不会再重复调。

---

## 二、变更历史

| 时间 | 提交者 | 变更 | observation role |
|------|--------|------|-----------------|
| 2026-03-21 | 小沈 | 初始实现 `base.py` | `role: user` |
| 2026-05-13 | 小沈 | 提交 `4c4c2158`，改为 `role: system` | `role: system` |
| 2026-05-21 | 小沈 | Phase1 搬入 `message_builder.py`，保持 `system` | `role: system` |

**原始（`role: user`）的问题**：`split_history_for_llm()` 会把最后一条 `user` 消息当成"用户新输入"传给 LLM，让模型以为用户又发了新消息。

**改 `role: system` 时的权衡**（2026-05-13）：小沈意识到不能用 `role: user`，但当时没考虑 `role: tool`。`role: system` 在当时是合理的折中——不被当成用户输入，但那时系统只用 qwen2.5 等模型，没遇到死循环问题。

**现在的场景变了**：换了 minimax-m2.1 等模型后，这些模型更严格遵循 OpenAI Chat API 的角色语义，`role: system` 不再被识别为"工具结果"，导致死循环。

---

## 三、影响面分析

### 3.1 直接影响
- 模型持续调同一工具直到 `max_steps=100` 或客户端断开
- 缓存命中时虽不执行工具，但浪费 LLM API 调用（每轮 2-6s）
- 用户看到"执行中"状态卡死

### 3.2 触发条件
1. 工具执行结果被缓存命中
2. Observation 用 `role: system` 注入，模型不识别为工具结果

**注**：此问题和模型走 TextStrategy 还是 ToolsStrategy **无关**。prompt_136（20轮死循环）走的是 ToolsStrategy，prompt_146（6轮死循环）走的是 TextStrategy，observation 都通过 `message_builder.py:87` 统一写作 `role: system`。两种策略下 bug 表现完全一致。

### 3.3 修复涉及文件

| 文件 | 行 | 修改内容 |
|------|----|---------|
| `message_builder.py` | 87 | `role: system` → `role: tool` |
| `message_builder.py` | 452 | `_is_observation_role()` 加 `role == "tool"` 判断 |
| `message_builder.py` | 228-237 | `trim_history()` 分支处理 `role: tool` |

---

## 四、其他 role 分配对照

| role | 当前用途 | 是否合理 |
|------|---------|---------|
| `system` | 系统提示词、工具定义、Schema、执行汇总（别重复） | ✅ 全部合理（指令/上下文类） |
| `user` | 初始任务、每轮重新组装的任务 | ✅ 全部合理（用户输入类） |
| `assistant` | 模型返回的JSON、流式文本 | ✅ 全部合理（模型回答类） |
| `tool` | **从未使用** | ❌ 应该用于 Observation |

**结论**：只有 observation（`message_builder.py:87`）这一处不合理，其余 7 处 role 分配全部正确。

## 五、parse_error 分析（"介绍自己吧" 3次 parse_error）

**创建时间**: 2026-05-23 13:10:25  
**编写人**: 小沈  
**问题现象**: 用户问"介绍自己吧"，系统连续3次返回 parse_error（总共耗时约3秒），最终用户看到的回答是错误提示。

### 5.1 根因

**`chat_with_tools()` 非流式版本缺少 `self.reset_cancel()`，导致取消状态污染后继续请求。**

### 5.2 完整链路

从 applog 2026-05-23 确认的证据链：

```
11:58:01  prompt_136 被用户取消
         → [TaskControl] 取消任务: task_id=397b9898
         → [BaseAIService.cancel] _cancelled = True  ← 设置取消标志
         → [Task Cancelled] 任务已取消，保留记录

11:58:48  用户发送"介绍自己吧" (prompt_138)
         → [ChatRouter] route() 复用传入的 ai_service  ← 关键：同一实例

11:58:57  探测完成 → tools ✅ 支持 → 策略: tools

11:58:58  [chat_with_tools] 检测到取消，中断请求       ← 确凿证据
          [Function Calling] API Error: 任务已取消
          → 返回 parse_error

11:58:58  第2轮 → [chat_with_tools] 检测到取消，中断请求
11:58:59  第3轮 → [chat_with_tools] 检测到取消，中断请求
11:59:00  第4轮 → [chat_with_tools] 检测到取消，中断请求
          → 重试次数用尽 → 返回 parse_error 给用户
```

**关键日志行**（`app_2026-05-23.log`）：
| 行号 | 时间 | 内容 |
|------|------|------|
| 25526 | 11:58:50 | `[ChatRouter] route() 复用传入的 ai_service` |
| 25561 | 11:58:58 | `[chat_with_tools] 检测到取消，中断请求` |
| 25562 | 11:58:58 | `[Function Calling] API Error: 任务已取消` |

### 5.3 每次 parse_error 耗时约 1 秒

round 2: 11:58:57.213 → 11:58:58.213 = **1000ms**  
round 3: 11:58:58.348 → 11:58:59.352 = **1004ms**  
round 4: 11:58:59.508 → 11:59:00.520 = **1012ms**

每个恰好 1 秒——这是心跳检查间隔。`chat_with_tools()` 后台任务每 1 秒检测 `_cancelled=True`，检测到后立即中断。

### 5.4 为什么探测能成功而实际调用失败

探测（capability detector）直接调用 API 进行 probe，**不走 `chat_with_tools()`**，所以不受 `_cancelled` 影响。

### 5.5 修复确认

`llm_core.py:500` 已添加 `self.reset_cancel()`：

```python
async def chat_with_tools(self, ...):
    self.reset_cancel()    # ✅ 修复：新请求前重置取消状态
    try:
        ...
```

流式版本 `chat_with_tools_stream()` 和 `chat_stream()` 之前已经有 `self.reset_cancel()`，只有非流式 `chat_with_tools()` 遗漏了，导致这个 bug。

## 六、session 404 分析（"会话不存在"）

**创建时间**: 2026-05-23 13:10:25  
**编写人**: 小沈  
**问题现象**: 用户发送消息时，后端返回 404: 会话不存在。涉及 session ID：`f2244428`、`1081a064`、`a0346b81`、`522f2dbe`。

### 6.1 根因

**前端缓存的 sessionId 在 DB 中不存在 → 跳过 `createSession()` → 直接发消息 → 后端 404。**

### 6.2 完整链路

DB 确认：153 个 session 中，上述 4 个 ID **全都不存在**。applog 2026-05-23 中仅有 1 次 `创建会话成功`（09:55:30 session `402039bb`）。

前端流程（`useChatSend.ts:116-126`）：

```typescript
let currentSessionId = sessionId;
if (!currentSessionId) {
    // 只有 sessionId 为 null 时才创建
    const newSession = await sessionApi.createSession(...);
    currentSessionId = newSession.session_id;
} else {
    // sessionId 不为 null → 跳过创建 → 直接用
    currentSessionIdRef.current = currentSessionId;
}
```

**sessionId 的来源**（`useChatSession.ts` 初始化流程）：
1. URL 参数 `?session_id=xxx`
2. sessionStorage 缓存（`restoreState()`）
3. 后端加载最近会话

### 6.3 applog 确认的证据

```
12:14:31  [Preprocessing] [session=f2244428...]  input: '你好，请问你是谁？'
12:14:33  [ChatRouter] route() 复用传入的 ai_service
12:14:33  [Save] 保存失败: 404: 会话不存在: f2244428       ← 立即 404
```

没有 `创建会话成功`、没有 `保存用户消息ID`——session 完全不存在于 DB。

同时，`ai_service` 仍被 `_cancelled` 污染（见 五），导致 LLM 调用也失败：

```
12:14:41  [chat_with_tools] 检测到取消，中断请求
12:14:41  API Error: 任务已取消
```

### 6.4 待确认

- 这些 session ID 具体是从 sessionStorage 还是 URL 恢复的
- `restoreState()` 的 sessionStorage 何时写入
- 是否可能存在 DB 重建/路径变更导致 session 丢失
