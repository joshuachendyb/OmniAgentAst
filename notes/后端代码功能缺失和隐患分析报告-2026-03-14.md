### 缺失1：会话中断记录内存泄漏风险（代码TODO已标记，待小沈实现）

**位置**：`chat.py:652-656`

**代码证据**：
```python
# 【小沈-2026-03-13修复】会话级别中断记录（防止重连循环）
# 记录已中断的会话ID和最后一次中断时间
# TODO【小健-2026-03-13深度检查】：需要添加定期清理机制，否则长期运行会内存泄漏
# TODO【小健-2026-03-13深度检查】：需要添加锁保护，否则并发写入可能有竞态条件
interrupted_sessions: dict[str, datetime] = {}
INTERRUPTED_SESSION_TIMEOUT = timedelta(minutes=5)
```

**处理状态**：📋 **代码中已保留TODO标记，交由小沈后续处理**

**问题分析**：
1. **功能缺失**：`interrupted_sessions` 字典添加了记录，但**没有实现清理机制**
2. **内存泄漏**：每次会话中断都添加记录，永不清理，长期运行会导致内存溢出
3. **线程安全**：字典在多个协程中访问，没有锁保护，存在竞态条件

**实际影响**：
- 后端长期运行一天，可能积累数千条中断记录
- 内存占用持续增长，可能导致OOM崩溃

**建议修复方案**：
```python
# 添加定期清理任务
import asyncio

async def cleanup_interrupted_sessions():
    """定期清理过期的中断记录"""
    while True:
        await asyncio.sleep(300)  # 每5分钟清理一次
        now = datetime.now()
        async with running_tasks_lock:  # 需要添加锁
            expired = [
                sid for sid, ts in interrupted_sessions.items()
                if now - ts > INTERRUPTED_SESSION_TIMEOUT
            ]
            for sid in expired:
                del interrupted_sessions[sid]
            if expired:
                logger.info(f"清理过期中断记录: {len(expired)}条")

# 在 startup 事件中启动清理任务
```

## 🔧 三、代码质量问题

### 问题1：数据库连接异常时可能泄漏（需改用finally确保关闭）

**位置**：`sessions.py:126-150`（会话创建函数）

**代码证据**：
```python
@router.post("/sessions", response_model=SessionResponse)
async def create_session(request: CreateSessionRequest):
    conn = None
    try:
        conn = _get_db_connection()
        cursor = conn.cursor()
        # ... 数据库操作
        conn.commit()
        conn.close()  # ✅ 正常路径有关闭
        return SessionResponse(...)
    except HTTPException:
        raise  # ❌ 直接抛出异常，conn已关闭但cursor可能未清理
    except Exception as e:
        logger.error(f"创建会话失败: {e}")
        raise HTTPException(status_code=500, detail="创建会话失败")
        # ❌ 异常分支下conn.close()未执行，连接可能泄漏
```

**逐路径分析**：

| 路径 | 是否关闭连接 | 问题 |
|------|------------|------|
| 正常流程 | ✅ `conn.close()` 执行 | 无问题 |
| HTTPException异常 | ❌ 直接抛出 | 连接已关闭，但cursor未显式关闭 |
| 通用Exception异常 | ❌ 未执行close | **连接泄漏**，数据库连接池耗尽 |

**问题细化**：
1. **异常分支未使用finally**：`try-except` 结构中，异常分支未执行 `conn.close()`
2. **cursor未显式关闭**：虽然Python会自动关闭cursor，但在异常时可能导致资源未及时释放
3. **其他函数存在相同问题**：`get_sessions()`, `update_session()`, `delete_session()` 等函数也存在类似问题

**实际影响**：
- 数据库连接池配置有限制（默认SQLite无连接池，但网络数据库如PostgreSQL有连接池限制）
- 异常频繁发生时，连接池可能耗尽，导致后续请求全部失败

**建议修复**：
```python
@router.post("/sessions", response_model=SessionResponse)
async def create_session(request: CreateSessionRequest):
    conn = None
    cursor = None
    try:
        conn = _get_db_connection()
        cursor = conn.cursor()
        # ... 数据库操作
        conn.commit()
        return SessionResponse(...)
    except HTTPException:
        raise  # 重新抛出，由外层处理
    except Exception as e:
        logger.error(f"创建会话失败: {e}")
        if conn:
            conn.rollback()  # 异常时回滚
        raise HTTPException(status_code=500, detail="创建会话失败")
    finally:
        if cursor:
            cursor.close()  # 确保cursor关闭
        if conn:
            conn.close()  # 确保连接关闭
```

### 问题2：异常处理不够细化（缺少特定异常类型捕获）

**位置**：`chat.py:626-639`

**代码证据**：
```python
# 当前异常处理结构
try:
    # ... AI调用等操作
except HTTPException:
    raise  # 直接抛出HTTP异常
except IndexError as e:
    # 消息列表索引错误
    raise HTTPException(status_code=400, detail="消息列表格式错误")
except Exception as e:
    # 所有其他异常统一处理
    logger.error(f"聊天请求服务端错误: {e}", exc_info=True)
    raise HTTPException(status_code=500, detail="服务器内部错误，请稍后重试")
```

**问题细化分析**：

当前异常处理结构过于笼统，将所有非 `IndexError` 的异常统一作为500错误处理，**无法区分不同类型的客户端错误**：

| 异常类型 | 当前处理 | 正确处理 | 影响 |
|---------|---------|---------|------|
| `json.JSONDecodeError` | 500错误 | 应为400错误 | 用户无法定位JSON格式问题 |
| `KeyError` | 500错误 | 应为400错误 | 用户无法定位消息结构缺失字段 |
| `TypeError` | 500错误 | 应为400错误 | 用户无法定位消息类型错误 |
| `ValueError` | 500错误 | 应为400错误 | 用户无法定位参数值错误 |
| `httpx.RequestError` | 500错误 | 应为503错误 | 用户无法区分网络问题和服务错误 |
| `httpx.TimeoutException` | 500错误 | 应为504错误 | 用户无法区分超时和服务崩溃 |

**实际影响**：
1. **客户端调试困难**：所有错误都返回500，前端无法给用户明确提示
2. **日志分析困难**：无法区分客户端问题和服务端问题
3. **问题定位困难**：开发人员需要从日志中排查具体异常类型

**建议的异常处理层级**：
```python
try:
    # ... AI调用等操作
except HTTPException:
    raise  # 直接抛出，保持原有状态码

# 第一层：客户端错误（4xx）
except json.JSONDecodeError as e:
    logger.warning(f"JSON解析错误: {e}")
    raise HTTPException(status_code=400, detail=f"请求JSON格式错误: {str(e)}")

except KeyError as e:
    logger.warning(f"消息结构缺失字段: {e}")
    raise HTTPException(status_code=400, detail=f"消息缺少必需字段: {str(e)}")

except (TypeError, ValueError) as e:
    logger.warning(f"参数类型或值错误: {e}")
    raise HTTPException(status_code=400, detail=f"参数错误: {str(e)}")

except IndexError as e:
    logger.warning(f"消息列表索引错误: {e}")
    raise HTTPException(status_code=400, detail="消息列表格式错误")

# 第二层：网络/外部服务错误（5xx）
except httpx.TimeoutException as e:
    logger.error(f"AI服务请求超时: {e}")
    raise HTTPException(status_code=504, detail="AI服务响应超时，请稍后重试")

except httpx.RequestError as e:
    logger.error(f"AI服务网络错误: {e}")
    raise HTTPException(status_code=503, detail="AI服务暂时不可用，请稍后重试")

# 第三层：未知服务端错误（500）
except Exception as e:
    logger.error(f"聊天请求服务端错误: {e}", exc_info=True)
    raise HTTPException(status_code=500, detail="服务器内部错误，请稍后重试")
```

**优先级建议**：P3（优化项，不影响核心功能，但可提升用户体验和问题排查效率）
