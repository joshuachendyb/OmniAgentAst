### 隐患一：硬编码API地址（两处不一致）

**严重级别**: P3（低）

**问题位置**: 
- `NewChatContainer.tsx` 第175行
- `api.ts` 第16行

**代码证据**:

**位置1** - `NewChatContainer.tsx:175`:
```typescript
const {
  ...
} = useSSE(
  {
    baseURL: "http://localhost:8000/api/v1",  // ← 硬编码
    sessionId: sessionId || "default-session",
  },
```

**位置2** - `api.ts:16`:
```typescript
const API_BASE_URL = "http://localhost:8000/api/v1";
```

**问题分析**:
1. API地址硬编码在**两处不同文件**
2. 没有使用环境变量或配置文件
3. 开发/生产环境切换需要改两处代码（容易遗漏）

**实际影响**:
- 部署到生产环境时可能忘记修改其中一个地址
- 导致API请求失败

**建议修复方案**:
```typescript
// 建议创建配置文件：src/config/api.ts
export const API_CONFIG = {
  baseURL: import.meta.env.VITE_API_BASE_URL || "http://localhost:8000/api/v1",
};

// 然后在两个文件中统一导入使用
import { API_CONFIG } from '@/config/api';
// useSSE({ baseURL: API_CONFIG.baseURL, ... })
```

### 隐患二：类型断言绕过TypeScript检查

**严重级别**: P2（中）

**问题位置**: `sse.ts` 第713-741行

**代码证据**:
```typescript
// 第714-741行（共8处使用 as any）
if (rawData.code) {
  (step as any).code = rawData.code;           // ← 第1处
}
if (rawData.error_type) {
  (step as any).error_type = rawData.error_type; // ← 第2处
}
if (rawData.model) {
  (step as any).model = rawData.model;         // ← 第3处
}
if (rawData.provider) {
  (step as any).provider = rawData.provider;   // ← 第4处
}
if (rawData.details) {
  (step as any).details = rawData.details;     // ← 第5处
}
if (rawData.stack) {
  (step as any).stack = rawData.stack;         // ← 第6处
}
if (rawData.retryable !== undefined) {
  (step as any).retryable = rawData.retryable; // ← 第7处
}
if (rawData.retry_after !== undefined) {
  (step as any).retry_after = rawData.retry_after; // ← 第8处
}
```

**问题分析**:
1. 使用`as any`绕过TypeScript类型检查
2. 如果后端API字段名变更（如`retry_after`改为`wait_time`），无法及时发现
3. 字段名拼写错误不会被发现

**实际影响**:
- 代码重构时无法追踪这些字段的使用
- API字段变更导致静默失败

**建议修复方案**:
```typescript
// 方案1：扩展 ExecutionStep 接口（推荐）
// 在 sse.ts 第49-106行的 ExecutionStep 接口中添加：
export interface ExecutionStep {
  // ... 现有字段
  // 新增可选字段
  code?: string;
  error_type?: string;
  model?: string;
  provider?: string;
  details?: string;
  stack?: string;
  retryable?: boolean;
  retry_after?: number;
  wait_time?: number;
}

// 然后直接使用，不需要 as any
if (rawData.code) {
  step.code = rawData.code;
}
```

### 隐患三：可选字段空值检查不完整

**严重级别**: P2（中）

**问题位置**: `api.ts` 第264-269行、`api.ts` 第346-367行

**代码证据**:

**位置1** - `api.ts:264-269`:
```typescript
const [modelData, status] = await Promise.all([
  configApi.getModelList(),
  chatApi.validateService(),
]);

if (modelData.models) {  // ← 没有检查 modelData 本身是否为 null
  setModelList(modelData.models);
}
```

**位置2** - `api.ts:356-367`:
```typescript
getModelList: async (): Promise<{
  models: {...}[];
  default_provider: string;
}> => {
  const response = await api.get<{...}>("/config/models");
  return response.data;  // ← 如果 response.data 是 null 怎么办？
}
```

**问题分析**:
1. `if (modelData.models)` 只检查了 models 是否存在
2. 如果 `modelData` 本身是 `null` 或 `undefined`，会直接抛出错误
3. 如果后端返回 `{ models: null }`，判断会通过但后续使用会报错

**实际影响**:
- API返回异常数据时导致前端崩溃

**建议修复方案**:
```typescript
// 更安全的检查方式
const [modelData, status] = await Promise.all([
  configApi.getModelList(),
  chatApi.validateService(),
]);

// 检查 modelData 存在且 models 是数组
if (modelData?.models && Array.isArray(modelData.models)) {
  setModelList(modelData.models);
} else {
  console.error("getModelList 返回数据格式异常:", modelData);
  setModelList([]);  // 设置默认值
}
```

### 隐患四：异常处理未区分错误类型

**严重级别**: P2（中）

**问题位置**: `NewChatContainer.tsx` 第406-473行

**代码证据**:
```typescript
// 第406-473行
} catch (saveError) {
  console.error("保存AI回复或标题失败:", saveError);
  
  // 所有错误都走同样的重试逻辑
  let retryCount = 0;
  const maxRetries = 3;
  
  const retrySave = async () => {
    while (retryCount < maxRetries) {
      retryCount++;
      message.warning(`保存失败，正在重试 (${retryCount}/${maxRetries})...`);
      // ... 重试
    }
  };
  
  retrySave();
}
```

**具体问题**:

错误类型未细分，不同错误应该不同处理：

| 错误类型 | 当前处理 | 应该的处理 |
|---------|---------|-----------|
| **409版本冲突** | 重试3次（无意义） | 通知用户刷新页面，不重试 |
| **网络断开** | 重试3次 | 合理，但应该等待网络恢复 |
| **服务器500** | 重试3次 | 合理 |
| **业务错误（会话不存在）** | 重试3次（无意义） | 直接提示用户，不重试 |

**实际影响**:
- 409冲突会触发3次无意义重试
- 用户看到3次"保存失败"警告
- 最终缓存到localStorage的数据可能已过时

**建议修复方案**:
```typescript
} catch (saveError: any) {
  console.error("保存AI回复或标题失败:", saveError);
  
  // 分类处理不同错误
  const errorCode = saveError?.response?.status;
  const errorType = saveError?.response?.data?.error_type;
  
  // 情况1：版本冲突 - 不重试，直接提示
  if (errorCode === 409) {
    message.error("会话已被其他用户修改，请刷新页面");
    // 从服务器获取最新数据
    try {
      const sessionData = await sessionApi.getSessionMessages(currentSessionId);
      if (sessionData.version) setSessionVersion(sessionData.version);
      if (sessionData.title) setSessionTitle(sessionData.title);
    } catch (syncError) {
      console.error("同步失败:", syncError);
    }
    return;  // 不重试
  }
  
  // 情况2：业务错误（会话不存在等） - 不重试
  if (errorCode === 404 || errorCode === 400) {
    message.error(saveError?.response?.data?.detail || "保存失败");
    return;  // 不重试
  }
  
  // 情况3：网络或服务器错误 - 重试
  let retryCount = 0;
  const maxRetries = 3;
  // ... 重试逻辑
}
```

### 隐患五：setTimeout清理不完整（内存泄漏）

**严重级别**: P2（中）

**问题位置**: `sse.ts` 第220-228行、第472-478行

**代码证据**:

**定义处** - `sse.ts:220-228`:
```typescript
const reconnectConfigRef = useRef<ReconnectConfig>({
  enabled: true,
  maxAttempts: 3,
  baseDelay: 1000,
  maxDelay: 10000,
});
const reconnectAttemptsRef = useRef(0);
const reconnectTimeoutRef = useRef<number | null>(null);  // ← setTimeout返回值
const pendingMessageRef = useRef<{...} | null>(null);
```

**清理处** - `sse.ts:472-478`:
```typescript
useEffect(() => {
  return () => {
    disconnect();
    pendingMessageRef.current = null;  // ← 清理了 ref
    // ⚠️ 遗漏：没有清理 reconnectTimeoutRef
  };
}, [disconnect]);
```

**同时查看 disconnect 函数** - `sse.ts:234-257`:
```typescript
const disconnect = useCallback((manualDisconnect: boolean = false) => {
  if (reconnectTimeoutRef.current) {
    clearTimeout(reconnectTimeoutRef.current);
    reconnectTimeoutRef.current = null;
  }
  // ... 其他清理
}, []);
```

**问题分析**:
- `disconnect()` 函数会清理 `reconnectTimeoutRef`
- 但是组件卸载时调用的 `disconnect()` 依赖外部传入的 `manualDisconnect` 参数
- 清理 useEffect 中直接调用 `disconnect()` 没有传递参数，默认 `false`
- 这个场景下可能会遗漏对某些清理逻辑的触发

**实际影响**:
- 组件卸载后，setTimeout 可能仍然触发回调
- 回调中访问已卸载组件的状态会导致内存泄漏或控制台警告

**建议修复方案**:
```typescript
useEffect(() => {
  return () => {
    // 组件卸载时应该手动断开，停止重连
    disconnect(true);  // ← 传递 true，表示手动断开
    pendingMessageRef.current = null;
    
    // 额外确保 timeout 被清理
    if (reconnectTimeoutRef.current) {
      clearTimeout(reconnectTimeoutRef.current);
      reconnectTimeoutRef.current = null;
    }
  };
}, [disconnect]);
```
