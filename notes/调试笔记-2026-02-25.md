## 1 小新代码检查发现taskId前后端交互问题 【taskId接口完整性问题】2026-02-25 00:10:24

### 1.1 问题描述

在前端代码深度扫描过程中，小新发现了严重的 taskId 前后端交互不完整问题，导致任务中断功能无法正常工作。

### 1.2 前端实现情况

**前端已完成的工作**：
- ✅ `SSEConfig` 接口已添加 `taskId` 字段
- ✅ `UseSSEReturn` 接口已添加 `setTaskId` 方法
- ✅ 请求 body 已包含 `task_id: taskId || undefined`
- ✅ `NewChatContainer.tsx` 已调用 `setTaskId(taskId)`
- ✅ `ChatContainer.tsx` 已添加服务状态初始化

**前端数据流**：
```
用户发送消息
    ↓
NewChatContainer.executeStreamSend()
    ├─ 生成 taskId = crypto.randomUUID()
    ├─ setCurrentTaskId(taskId)  [保存到状态]
    └─ setTaskId(taskId)          [传给 SSE Hook]
            ↓
       useSSE Hook
            ↓
       sendMessage()
            ↓
       fetch 请求
            └─ body: { ..., task_id: taskId }
```

### 1.3 后端实现情况

**后端已有的实现**：
- ✅ 后端已有 `running_tasks` 字典管理任务
- ✅ 后端已有 `cancel_stream_task` 接口 `/chat/stream/cancel/{task_id}`
- ✅ 后端会检查 `running_tasks.get(task_id, {}).get("cancelled", False)` 来中断任务
- ✅ 后端会在流式响应的关键位置检查中断标志

**后端问题**：
- ❌ `ChatRequest` 模型**缺少 `task_id` 字段**
- ❌ 后端**自己生成 `task_id = str(uuid.uuid4())`**（第289行）
- ❌ 后端**没有使用前端传的 `task_id`**
- ❌ 后端**没有把 `task_id` 返回给前端**（在 start 事件里）

### 1.4 问题根源

**数据流断层**：

```
前端传了 task_id → 后端不用，自己生成 → 后端不返回 task_id → 前端不知道用哪个 task_id 中断
```

**正确的数据流应该是**：

```
前端生成 task_id
    ↓
前端请求：body={ ..., task_id: "xxx-xxx" }
    ↓
后端接收 task_id，用这个而不是自己生成
    ↓
后端 start 事件：{ "type": "start", "task_id": "xxx-xxx", ... }
    ↓
前端接收并保存 task_id
    ↓
用户点击中断
    ↓
前端请求：/chat/stream/cancel/{保存的task_id}
```

### 1.5 需要修改的内容

**后端需要修改**：

| 文件 | 修改项 | 说明 |
|------|--------|------|
| `chat.py` | `ChatRequest` 添加 `task_id` 字段 | 接收前端传的 task_id |
| `chat.py` | 使用前端传的 task_id 而不是自己生成 | 第289行 |
| `chat.py` | start 事件里添加 `task_id` 返回给前端 | 第309行 |

### 1.6 影响评估

## 3 资深专家发现流式输出显示严重问题 【流式显示+UI布局】2026-02-25 00:10:24

### 3.1 问题描述

北京老陈要求深度检查流式输出显示，资深专家（小查）发现严重问题：

### 3.2 发现的严重问题

| 编号 | 问题类型 | 严重程度 | 说明 |
|------|---------|---------|------|
| FRC-FLOW-001 | 流式显示完全没有实现 | 🔴 P0-紧急 | MessageItem只显示message.content，不显示currentResponse，流式数据浪费了！ |
| FRC-FLOW-002 | UI布局位置错误 | 🔴 P0-紧急 | AI思考过程面板位置完全错误，应该在AI消息旁边，而不是割裂！ |
| FRC-FLOW-003 | 消息结构混乱 | 🟠 P1-高 | 先创建assistant消息占位但不更新content，用户看到的消息不正确！ |

### 3.3 问题详细分析

#### FRC-FLOW-001：流式显示完全没有实现

**现象**：
- sse.ts中，currentResponse在实时更新（逐token）
- NewChatContainer.tsx中，有currentResponse状态
- 但是！MessageItem.tsx中，只显示message.content，**完全没有使用currentResponse！**

**后果**：
- 用户看不到逐token生成的过程
- 流式API等于白做了，用户体验极差
- 违背常识：流式就是要让用户看到逐步生成

**正确做法**：
```typescript
// NewChatContainer.tsx应该传递currentResponse给MessageItem
<MessageItem 
  message={item} 
  currentResponse={currentResponse}  // 新增！
  isStreaming={item.isStreaming}  // 新增！
  showExecution={showExecution}
/>

// MessageItem.tsx应该根据isStreaming显示不同内容
if (message.isStreaming) {
  // 显示动态流式内容（逐token）
  return <div>{currentResponse}</div>;
} else {
  // 显示静态消息
  return <div>{message.content}</div>;
}
```

#### FRC-FLOW-002：UI布局位置错误

**当前错误布局**：
```
消息列表（上面）
  ↓
AI思考过程面板（Collapse）❌ 错误位置！割裂了！
  ↓
消息列表（下面）
输入区域
```

**正确布局（方案1-推荐）**：
```
消息列表
  ↓
用户消息
  ↓
AI消息（内部包含可折叠的思考过程）✅ 正确！
  ↓
输入区域
```

**正确布局（方案2-备选）**：
```
消息列表（左） | 思考过程面板（右）
  ↓
输入区域（全宽）
```

#### FRC-FLOW-003：消息结构混乱

**现象**：
- 先创建assistant消息占位：content为空
- 然后，currentResponse在更新
- 但是，assistant消息的content没有被更新！
- 最终，用户看到的AI消息是空的，或者显示不正确

**正确做法**：
- 创建assistant消息时，content=''，isStreaming=true
- 每次收到chunk时，更新currentResponse
- MessageItem根据isStreaming，显示currentResponse而不是message.content
