### 1.1 调查背景

用户反馈：在前端下拉框中选择不同的大模型后，希望在AI回复的消息中显示当前使用的是哪个模型。

### 1.2 数据流路径分析

```
前端下拉框选择 → chatApi.switchProvider() → 后端更新配置文件 → 每次对话调用 get_service()
```

### 1.3 关键代码位置

### 1.4 发现的严重问题

**问题：AIServiceFactory.get_service() 使用单例缓存模式**

代码位置：`backend/app/services/__init__.py` 第77-78行

```python
# 如果实例已存在，直接返回缓存的实例，不会重新读取配置！
if cls._instance is not None:
    return cls._instance
```

**问题影响**：

| 对话次数 | 使用的模型 | 说明 |
|---------|-----------|------|
| 第1次 | ✅ 新模型 | 切换后首次对话，instance为None，重新读取配置 |
| 第2次及以后 | ❌ 旧模型 | instance已缓存，直接返回，不会再读取配置文件 |

**代码逻辑分析**：

```python
@classmethod
def get_service(cls, config_path: Optional[str] = None) -> BaseAIService:
    # 第一次检查（无锁，快速路径）
    if cls._instance is not None:
        return cls._instance  # 直接返回缓存的实例！
    
    with cls._lock:
        # 第二次检查（有锁，防止重复创建）
        if cls._instance is not None:
            return cls._instance
        
        # 只有第一次会执行到这里，创建新实例
        config = cls.load_config(config_path)
        provider = ai_config.get("provider", "zhipuai")
        
        # 创建新实例并缓存
        cls._instance = ZhipuAIService(...)  # 或 OpenCodeService
    
    return cls._instance
```

### 1.5 结论与建议

#### 1.5.1 结论

| 功能 | 状态 | 说明 |
|------|------|------|
| 前端下拉框选择 | ✅ 正常 | 选择不同provider时，currentProvider状态正确更新 |
| 切换模型API | ✅ 正常 | chatApi.switchProvider() 能正确调用后端接口 |
| 配置文件更新 | ✅ 正常 | switch_provider() 能正确更新config.yaml |
| **实际对话使用的模型** | ❌ **异常** | 只有切换后第一次对话有效，之后一直使用同一个模型 |

#### 1.5.2 修复建议

**方案一：在 get_service() 中强制每次重新加载配置**

修改 `backend/app/services/__init__.py` 的 `get_service()` 方法：

```python
@classmethod
def get_service(cls, config_path: Optional[str] = None) -> BaseAIService:
    # 【修复】每次都重新加载配置，忽略缓存
    cls._instance = None  # 清空缓存实例
    
    # 然后按原有逻辑创建新实例...
```

**方案二：在 chat.py 的 /chat/stream API 中处理**

在每次调用AI服务前，重新加载配置并创建新的service实例。

## 4 修复完成：流式输出+模型名称显示 【小新】2026-02-23 05:45:04

### 4.1 问题发现

**问题1：流式输出不工作**
- 现象：AI回复时一次性显示全部内容，没有实时流式输出
- 原因：前端使用EventSource只支持GET，但后端是POST请求

**问题2：模型名称不显示**
- 现象：AI消息只显示"AI助手"，不显示具体使用的模型名称
- 原因：后端返回了model字段，但前端onComplete回调未正确处理

### 4.2 修复方案

#### 修复1：sse.ts - fetch + ReadableStream

```typescript
// 文件：frontend/src/utils/sse.ts
// 问题：EventSource只支持GET，无法向后端POST message参数
// 解决：改用fetch + ReadableStream

const response = await fetch(url, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    messages: [{ role: 'user', content: content }],
    stream: true
  })
});

const reader = response.body.getReader();
const decoder = new TextDecoder('utf-8');

while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  
  buffer += decoder.decode(value, { stream: true });
  const lines = buffer.split('\n');
  buffer = lines.pop() || '';
  
  for (const line of lines) {
    processSSEData(line, handlers);
  }
}
```
