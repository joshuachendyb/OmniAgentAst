## 问题分析

### 问题点1：URL会话加载失败后继续执行

**代码**: 第920-974行

```typescript
if (sessionData.messages && sessionData.messages.length > 0) {
  // 加载会话成功
  setSessionId(urlSessionId);
  currentSessionIdRef.current = urlSessionId;
  setMessages(...);
  return;  // ← 只有加载成功才 return
}
// ← 如果加载失败（没有消息），不会 return，继续往下执行
```

**问题**:
- 如果 `sessionData.messages` 为空或不存在，不会进入 if 块
- 不会执行 return
- 继续往下执行到第1005行和第1017行

---

### 问题点2：第1005行的判断条件

**代码**: 第1005-1014行

```typescript
// 🔴 修复3: 只有在没有URL参数时才考虑sessionStorage
if (!urlSessionId) {
  const restored = restoreState();
  if (restored) {
    return;
  }
}
```

**分析**:
- 这个 if 判断的是 `!urlSessionId`（urlSessionId 为假值）
- 但是，即使 urlSessionId 有值，如果加载失败，也会跳过这个 if
- 继续往下执行到第1017行加载最近会话

**问题**:
- URL会话有值，但加载失败（比如会话没有消息）
- 跳过 sessionStorage 恢复（因为 urlSessionId 不为空）
- 加载最近会话（第1017行）

## 根本原因

**问题**: URL会话加载失败后，没有 return，继续执行到加载最近会话的逻辑

**流程**:
```
用户点击历史会话 A
  ↓
URL参数: session_id=A
  ↓
调用 getSessionMessages(A)
  ↓
返回: messages=[]（会话A没有消息）
  ↓
if (messages.length > 0) → False
  ↓
不执行 return
  ↓
if (!urlSessionId) → False（urlSessionId=A）
  ↓
跳过 sessionStorage 恢复
  ↓
加载最近会话 B
  ↓
setSessionId(B) ← ❌ 错误：会话ID被替换为B
```

## 修复方案

### 方案1：加载URL会话失败时也 return

**修改**: 第974行后添加 else 分支

```typescript
if (sessionData.messages && sessionData.messages.length > 0) {
  // 加载会话成功
  setSessionId(urlSessionId);
  currentSessionIdRef.current = urlSessionId;
  setMessages(...);
  return;
} else {
  // 加载失败，也 return，避免加载最近会话
  console.warn("🔴 URL会话没有消息，跳过加载:", urlSessionId);
  setSessionJumpLoading(false);
  message.destroy("session-load");
  return;
}
```

## 结论

**根本原因**: URL会话加载失败（没有消息）后，没有 return，继续执行到加载最近会话的逻辑，导致会话ID被替换。

**修复建议**: 采用方案1，在URL会话加载失败时也 return。
