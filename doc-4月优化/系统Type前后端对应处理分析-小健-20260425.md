## 一、总体说明

### 1.1 分析目的

系统正在从"chat/file双路径"架构升级为"统一ReAct循环"架构。在统一架构下，所有请求都走ReAct循环，LLM的每一次输出都通过解析器转化为Step类型，经SSE传到前端渲染。**type体系是前后端通信的契约**，必须确保：

1. 后端产生的每个type，前端都能正确处理
2. 前端处理的每个type，后端都能正确产生
3. type的语义在前后端之间一致

### 1.2 Type传递全链路

```
LLM原始输出
    ↓
parse_react_response() → parse type (answer/implicit/action/thought_only/parse_error/chunk)
    ↓
base_react.py run_stream() → yield Step对象 (thought/action_tool/observation/final/error/chunk)
    ↓
react_sse_wrapper.py _format_sse_event() → SSE event type (thought/action_tool/observation/final/error/chunk/incident)
    ↓
SSE传输 (data: {"type": "xxx", ...})
    ↓
前端 sse.ts processSSEData() → ExecutionStep对象
    ↓
StepRow/StepContent/ExecutionPanel → 渲染UI
```

## 二、后端Type体系

### 2.1 Parse Type（解析器输出）

**文件**: `react_output_parser.py` — `parse_react_response()`

| type | 含义 | 触发条件 |
|------|------|---------|
| `action` | 工具调用 | LLM输出包含Action/工具名/JSON tool_name |
| `answer` | 显式最终回答 | LLM输出包含Answer/finish/tool_name=="finish" |
| `implicit` | 隐式最终回答 | 纯文本无关键词但长度>=5，或JSON中tool_name为None且含完成关键词 |
| `thought_only` | 纯思考 | 有Thought标记但无Action/Answer |
| `parse_error` | 解析失败 | 空响应/短文本/所有解析层失败 |
| `chunk` | 流式文本片段 | （当前未在ReAct流程中产生，仅定义） |

### 2.2 Step Type（ReAct循环输出）

**文件**: `reasoning_steps.py` — 各Step类

| type | Step类 | is_done() | 产生场景 |
|------|--------|-----------|---------|
| `start` | (裸字典) | - | 请求开始时，`start_step.py` 发送 |
| `thought` | ThoughtStep | False | action/answer/implicit/thought_only时 |
| `action_tool` | ActionToolStep | False | action时执行工具后 |
| `observation` | ObservationStep | return_direct | action时工具执行结果 |
| `chunk` | ChunkStep | False | （当前ReAct流程不产生） |
| `final` | FinalStep | True | answer/implicit/return_direct时 |
| `error` | ErrorStep | True | max_steps/空响应/parse_error耗尽/异常时 |
| `interrupted` | (裸字典) | - | 任务被取消时 |

### 2.3 Incident Type（中断/暂停/恢复）

**文件**: `incident_handler.py` — `create_incident_data()`

```python
# incident_handler.py 第17-41行
def create_incident_data(incident_value: str, message: str, step: Optional[int] = None) -> dict:
    data = {
        'type': 'incident',              # 固定为incident
        'incident_value': incident_value, # interrupted/paused/resumed/retrying
        'content': message,
        'message': message,
        'timestamp': create_timestamp(),
        'is_reasoning': False,
        'reasoning': ''
    }
    if step is not None:
        data['step'] = step
    return data
```

| incident_value | 含义 | 后端产生位置 |
|---------------|------|------------|
| `interrupted` | 任务被中断 | `base_react.py` yield裸字典 → `_format_sse_event()` 转换；`react_sse_wrapper.py` 直接调用 |
| `paused` | 任务被暂停 | `react_sse_wrapper.py` 中断检查 |
| `resumed` | 任务被恢复 | `react_sse_wrapper.py` 中断检查 |
| `retrying` | 正在重试 | chat路径：`chat_stream_query.py` 第111-116行 ✅；**ReAct路径：不产生** ❌ |

## 五、发现的问题及处理方法

### 问题1：`chunk` type在ReAct流程中不产生，SSE层丢弃

**严重程度**: 高（阻塞统一ReAct重构）

**现状**：
- 后端 `reasoning_steps.py` 已定义 `ChunkStep`（type="chunk"）
- 前端 `sse.ts` 已有 `case "chunk"` 处理逻辑（累加到responseBufferRef）
- 前端 `stepStyles.ts` 已有chunk样式（📝 内容，紫色系）
- 前端 `stepConstants.ts` 已有chunk标签（"回复"）和图标（"💬"）
- **但**：`base_react.py` 的 `run_stream()` 从不yield ChunkStep
- **但**：`react_sse_wrapper.py` 的 `_format_sse_event()` 对chunk返回空字符串（丢弃）

**影响**：统一ReAct后，纯对话场景的LLM输出无法作为chunk传递到前端，前端收不到流式文本。

**处理方法**：

1. **`base_react.py`**：在ReAct循环中新增chunk类型处理（详见重构文档3.3节）
2. **`react_sse_wrapper.py`**：`_format_sse_event()` 新增chunk分支

```python
# react_sse_wrapper.py _format_sse_event() 中新增
elif event_type == 'chunk':
    return format_chunk_sse(event, step, model, provider)

# 新增函数
def format_chunk_sse(event: dict, step: int, model: str, provider: str) -> str:
    """格式化chunk类型的SSE事件"""
    chunk_data = {
        "type": "chunk",
        "step": step,
        "content": event.get("content", ""),
        "thought": event.get("thought", ""),
        "reasoning": event.get("reasoning", ""),
        "timestamp": event.get("timestamp", ""),
        "is_reasoning": event.get("is_reasoning", False)
    }
    return f"data: {json.dumps(chunk_data, ensure_ascii=False)}\n\n"
```

3. **前端**：`ExecutionPanel.tsx` 新增chunk渲染分支

```typescript
// ExecutionPanel.tsx renderStepContent 中新增
case 'chunk':
    return (
        <div className="step-chunk">
            <Typography.Paragraph>{step.content}</Typography.Paragraph>
        </div>
    );
```

### 问题2：`interrupted` 在base_react.py中用裸字典，违反Step封装统一性

**严重程度**: 中

**现状**：
- `base_react.py` 中有 **3处** yield 裸字典：
  - 第219-223行（循环开始中断检查）：`yield {"type": "interrupted", "step": step_count, "message": "用户取消了任务"}`
  - 第392行（工具执行前中断检查）：`yield {"type": "interrupted", "step": step_count, "message": "用户取消了任务"}`
  - 第407行（工具执行后中断检查）：`yield {"type": "interrupted", "step": step_count, "message": "用户取消了任务"}`
- `_format_sse_event()` 捕获 `event_type == "interrupted"`，调用 `create_incident_data("interrupted", ...)` 转换为 `type="incident"` 格式
- 前端最终收到 `type="incident"` + `incident_value="interrupted"`，处理正确

**影响**：
1. **违反Step封装统一性**：同文件中其他所有步骤（thought、action_tool、observation、final、error）都已使用 `StepFactory.create_*_step()` + `step.to_dict()` 封装，唯独 interrupted 用裸字典
2. **type字段格式不一致**：裸字典 `type="interrupted"` 与标准格式 `type="incident"` + `incident_value="interrupted"` 不一致，依赖 `_format_sse_event()` 中间层做格式转换（补丁式兼容）
3. **缺少timestamp等标准字段**：裸字典没有 `timestamp`、`is_reasoning`、`reasoning` 等标准字段，而 `create_incident_data()` 会自动填充

**处理方法**：

方案A（推荐）：使用 `create_incident_data()` 替代裸字典，从源头产生标准格式

```python
# base_react.py 修改：3处裸字典全部替换
# 需要在文件头部添加导入：
from app.chat_stream.incident_handler import create_incident_data

# 第219-223行（循环开始中断检查）：
# 修改前：
# yield {
#     "type": "interrupted",
#     "step": step_count,
#     "message": "用户取消了任务"
# }
# 修改后：
interrupted_data = create_incident_data(
    incident_value="interrupted",
    message="用户取消了任务",
    step=step_count
)
yield interrupted_data

# 第392行（工具执行前中断检查）：
# 修改前：
# yield {"type": "interrupted", "step": step_count, "message": "用户取消了任务"}
# 修改后：
interrupted_data = create_incident_data(
    incident_value="interrupted",
    message="用户取消了任务",
    step=step_count
)
yield interrupted_data

# 第407行（工具执行后中断检查）：
# 修改前：
# yield {"type": "interrupted", "step": step_count, "message": "用户取消了任务"}
# 修改后：
interrupted_data = create_incident_data(
    incident_value="interrupted",
    message="用户取消了任务",
    step=step_count

```

同时修改 `_format_sse_event()` 新增 `incident` 类型处理（因为 base_react.py 现在 yield 的是 `type="incident"` 的字典，而非 `type="interrupted"`）：

```python
# react_sse_wrapper.py _format_sse_event() 中新增
# 在 elif event_type == 'interrupted' 之前添加：
elif event_type == 'incident':
    # incident类型直接格式化为SSE（base_react.py已使用create_incident_data产生标准格式）
    return f"data: {json.dumps(event, ensure_ascii=False)}\n\n"
```

**注意**：原有的 `elif event_type == 'interrupted'` 分支（第143-149行）可保留作为兼容层，处理其他可能仍发送 `type="interrupted"` 的代码路径（如旧版agent），但应标注为兼容代码。

方案B（备选）：创建 IncidentStep 类，纳入 StepFactory 封装体系

```python
# reasoning_steps.py 中新增 IncidentStep 类
class IncidentStep(ReasoningStep):
    """事件步骤（interrupted/paused/resumed/retrying）"""
    def __init__(self, step: int, incident_value: str, message: str,
                 timestamp: Optional[int] = None, wait_time: Optional[int] = None):
        super().__init__(step, timestamp)
        self._incident_value = incident_value
        self._message = message
        self._wait_time = wait_time

    def get_type(self) -> str:
        return "incident"

    def get_content(self) -> str:
        return self._message

    def is_done(self) -> bool:
        return self._incident_value == "interrupted"

    def to_dict(self) -> Dict[str, Any]:
        result = {
            "type": "incident",
            "incident_value": self._incident_value,
            "step": self._step,
            "timestamp": self._timestamp,
            "content": self._message,
            "message": self._message,
            "is_reasoning": False,
            "reasoning": ""
        }
        if self._wait_time is not None:
            result["wait_time"] = self._wait_time
        return result

# StepFactory 新增
@staticmethod
def create_incident_step(step: int, incident_value: str, message: str,
                         wait_time: Optional[int] = None) -> IncidentStep:
    return IncidentStep(step=step, incident_value=incident_value,
                        message=message, wait_time=wait_time)
```

方案B更彻底（纳入Step封装体系），但改动范围更大。方案A改动最小且能解决核心问题。

### 问题3：`retrying` incident_value 后端不产生，是死type

**严重程度**: 中

**现状**：
- 前端 `sse.ts` 第1385行有 `case "retrying"` 处理（调用 `onRetry` 回调）
- 前端 `stepStyles.ts` 有 `retrying` 样式（🔄 重试，蓝紫色系）
- 前端 `stepConstants.ts` 有 `retrying` 标签（"重试"）和图标（"🔄"）
- 前端 `StepContent.tsx` 有 `retrying` 渲染逻辑
- **chat路径**：`chat_stream_query.py` 第111-116行在请求超时重试时调用 `create_incident_data('retrying', ...)` ✅
- **ReAct路径**：`base_react.py` 的 `parse_error` 重试逻辑（第330-353行）**不产生** retrying 事件 ❌
- **react_sse_wrapper.py**：`_format_sse_event()` 中没有 retrying 相关处理 ❌

**影响**：
1. **ReAct路径重试不可见**：`base_react.py` 中 `parse_error` 重试时（第338行 `self.parse_retry_count += 1`），前端没有任何提示，用户看不到"正在重试"信息
2. **chat路径与ReAct路径行为不一致**：chat路径超时重试会显示retrying提示，ReAct路径解析失败重试却静默进行
3. **前端retrying处理链路形同虚设**：在ReAct路径下，前端的 `onRetry` 回调、retrying样式、retrying渲染逻辑永远不会被触发

**处理方法**：

方案A（推荐）：在 `base_react.py` 的 parse_error 重试逻辑中产生 retrying 事件

```python
# base_react.py parse_error 分支中，重试前发送retrying事件
# 修改前（第330-353行）：
if parsed["type"] == "parse_error":
    error_msg = parsed.get("error", "Unknown parse error")
    logger.warning(f"[parse_react_response] 情况4: 解析错误: {error_msg}, 重试次数={self.parse_retry_count}")
    self._add_observation_to_history(f"Parse Error: {error_msg}. ...")
    self.parse_retry_count += 1
    if self.parse_retry_count >= self.max_parse_retries:
        # yield ErrorStep
        ...
    continue

# 修改后：
if parsed["type"] == "parse_error":
    error_msg = parsed.get("error", "Unknown parse error")
    logger.warning(f"[parse_react_response] 情况4: 解析错误: {error_msg}, 重试次数={self.parse_retry_count}")
    self._add_observation_to_history(f"Parse Error: {error_msg}. ...")
    self.parse_retry_count += 1
    if self.parse_retry_count >= self.max_parse_retries:
        # yield ErrorStep
        ...
    else:
        # 新增：发送retrying事件，让前端显示重试提示
        retrying_data = create_incident_data(
            incident_value="retrying",
            message=f"解析失败，正在重试（第{self.parse_retry_count}次）",
            step=step_count
        )
        yield retrying_data
    continue
```

同时确保 `react_sse_wrapper.py` 的 `_format_sse_event()` 能处理 incident 类型（已在问题2方案A中解决）。

方案B（备选）：如果确定ReAct路径不需要重试UI，从前端移除retrying相关代码（不推荐，统一ReAct后应保持两条路径行为一致）

### 问题5：incident类型在ExecutionPanel中不渲染

**严重程度**: 中

**现状**：
- `ExecutionPanel.tsx` 的 `renderStepContent` switch（第269-482行）中没有 `case "incident"` 分支
- incident/interrupted/paused/resumed/retrying 走 `default: return null`，在步骤面板中不显示
- 但 `StepContent.tsx` 有完整的incident渲染逻辑（interrupted/paused/resumed/retrying 双重匹配）
- `StepRow/index.tsx` 第34行有 effectiveType 计算：`step.type === 'incident' ? step.incident_value || 'incident' : step.type`
- `stepStyles.ts` 有 interrupted/paused/resumed/retrying 的样式定义（但**没有** incident 容器类型的样式）
- `stepConstants.ts` 有所有incident子类型的标签和图标

**影响**：
1. **中断/暂停/恢复/重试事件在ExecutionPanel步骤列表中不可见**：用户在折叠的步骤面板中看不到这些事件
2. **StepContent与ExecutionPanel渲染不一致**：StepContent能渲染incident，但ExecutionPanel不能，导致展开步骤面板时能看到incident步骤，但折叠时看不到
3. **ExecutionPanel样式查找fallback**：第265-267行 `STEP_STYLES[step.type]` 对 `type="incident"` 会 fallback 到 `STEP_STYLES.thought`（因为stepStyles中没有incident的colorScheme），但由于switch走default返回null，样式问题被掩盖

**处理方法**：

方案A（推荐）：在ExecutionPanel中新增incident渲染，使用 effectiveType 统一处理

```typescript
// ExecutionPanel.tsx renderStepContent 中新增
// 在 case "error" 之前添加：

case "incident": {
  // incident容器类型：根据incident_value分发到具体渲染
  const incidentValue = (step as ExecutionStep & Record<string, unknown>).incident_value as string;
  const incidentMessage = step.content || step.message || '';
  
  switch (incidentValue) {
    case "interrupted":
      return (
        <div className="step-item">
          <div className="step-incident" style={{
            borderLeft: "2px solid #ffbb96",
            padding: "6px 8px",
            borderRadius: 4,
            marginTop: 8,
          }}>
            <Tag color="warning">⚠️ 中断</Tag>
            <Typography.Text type="secondary">{incidentMessage || '任务已中断'}</Typography.Text>
          </div>
        </div>
      );
    case "paused":
      return (
        <div className="step-item">
          <div className="step-incident" style={{
            borderLeft: "2px solid #d9d9d9",
            padding: "6px 8px",
            borderRadius: 4,
            marginTop: 8,
          }}>
            <Tag color="default">⏸️ 暂停</Tag>
            <Typography.Text type="secondary">{incidentMessage || '任务已暂停'}</Typography.Text>
          </div>
        </div>
      );
    case "resumed":
      return (
        <div className="step-item">
          <div className="step-incident" style={{
            borderLeft: "2px solid #52c41a",
            padding: "6px 8px",
            borderRadius: 4,
            marginTop: 8,
          }}>
            <Tag color="success">▶️ 恢复</Tag>
            <Typography.Text type="secondary">{incidentMessage || '任务已恢复'}</Typography.Text>
          </div>
        </div>
      );
    case "retrying":
      return (
        <div className="step-item">
          <div className="step-incident" style={{
            borderLeft: "2px solid #1890ff",
            padding: "6px 8px",
            borderRadius: 4,
            marginTop: 8,
          }}>
            <Tag color="processing">🔄 重试</Tag>
            <Typography.Text type="secondary">{incidentMessage || '正在重试...'}</Typography.Text>
          </div>
        </div>
      );
    default:
      return null;
  }
}
```

**注意**：由于 sse.ts 第1347行已将 `step.type` 设置为 `"incident"`（而非直接设置为 `"interrupted"`/`"paused"` 等），ExecutionPanel 的 switch 只需处理 `case "incident"`，通过 `incident_value` 内嵌 switch 分发。不需要为 `case "interrupted"` / `case "paused"` 等添加单独分支（这些直接type值在当前SSE流中不会出现）。

方案B（备选）：如果确认incident不需要在步骤列表中显示，保持现状，但在代码中添加注释说明设计意图：

```typescript
// ExecutionPanel.tsx renderStepContent 的 default 分支中
default:
  // incident类型（interrupted/paused/resumed/retrying）由全局回调处理，
  // 不在步骤面板中渲染（设计意图：这些是控制流事件，不是推理步骤）
  return null;
```

## 六、统一ReAct后的完整Type体系

统一ReAct重构完成后，type体系应为：

### 6.1 核心业务Type（7个）

| type | 产生场景 | 前端处理 | 前端渲染 |
|------|---------|---------|---------|
| `start` | 请求开始 | 构建startStep | ExecutionPanel ✅ / StepContent ✅ |
| `thought` | LLM思考/推理 | 设置thought字段 | ExecutionPanel ✅ / StepContent ✅ |
| `action_tool` | 执行工具 | 设置tool_name/params/result | ExecutionPanel ✅ / StepContent ✅ |
| `observation` | 工具执行结果 | 设置observation/return_direct | ExecutionPanel ✅ / StepContent ✅ |
| `chunk` | **LLM流式文本片段** | **累加到responseBuffer** | **ExecutionPanel 待补 / StepContent 待补** |
| `final` | 对话完成 | 调用onComplete | ExecutionPanel ✅ / StepContent ✅ |
| `error` | 错误 | 调用onError | ExecutionPanel ✅ / StepContent ✅ |

### 6.2 Incident Type（1个容器 + 4个子类型）

| type | incident_value | 产生场景 | 前端处理 | 问题 |
|------|---------------|---------|---------|------|
| `incident` | `interrupted` | 任务取消 | sse.ts → onComplete / StepContent ✅ / ExecutionPanel ❌ | 问题5：ExecutionPanel不渲染 |
| `incident` | `paused` | 任务暂停 | sse.ts → onPaused / StepContent ✅ / ExecutionPanel ❌ | 问题5：ExecutionPanel不渲染 |
| `incident` | `resumed` | 任务恢复 | sse.ts → onResumed / StepContent ✅ / ExecutionPanel ❌ | 问题5：ExecutionPanel不渲染 |
| `incident` | `retrying` | 解析重试 | sse.ts → onRetry / StepContent ✅ / ExecutionPanel ❌ | 问题3：ReAct路径不产生 + 问题5：ExecutionPanel不渲染 |

**前端处理详细检查**：

| 环节 | interrupted | paused | resumed | retrying |
|------|------------|--------|---------|----------|
| sse.ts case "incident" 内嵌switch | onComplete + onShowSteps ✅ | onPaused ✅ | onResumed ✅ | onRetry ✅ |
| sse.ts step对象构建 | type="incident", incident_value="interrupted", content=message ✅ | 同左 ✅ | 同左 ✅ | 同左 ✅ |
| StepContent.tsx 渲染 | 双重匹配(interrupted/incident+incident_value) ✅ | 双重匹配 ✅ | 双重匹配 ✅ | 双重匹配 ✅ |
| StepRow effectiveType | "interrupted" ✅ | "paused" ✅ | "resumed" ✅ | "retrying" ✅ |
| ExecutionPanel.tsx 渲染 | **不渲染(default→null) ❌** | **不渲染 ❌** | **不渲染 ❌** | **不渲染 ❌** |
| 后端ReAct路径产生 | base_react.py yield裸字典(问题2) ✅ | react_sse_wrapper.py ✅ | react_sse_wrapper.py ✅ | **不产生(问题3) ❌** |

**结论**：4个incident事件中，**2个有问题**：
1. **retrying**（问题3）：ReAct路径后端不产生retrying事件，前端处理链路形同虚设
2. **所有4个incident子类型**（问题5）：ExecutionPanel不渲染，步骤面板中不可见
