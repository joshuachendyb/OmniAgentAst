## 1. 问题A：加载慢的问题

### 1.1 完整加载流程

**前端代码位置**：NewChatContainer.tsx:1543-1609

```typescript
loadSession()
  → loadLatestHistoryMessages() 
    → listSessions(1, 1, undefined, true)  ← 获取最近会话ID
    → loadHistoryMessages(sessionId)
      → getSessionMessages(sessionId)  ← 获取所有消息（无分页）
  → setMessages(result.messages)  ← 触发React重渲染
```

### 1.2 性能瓶颈点

| 序号 | 瓶颈点 | 位置 | 说明 |
|------|-------|------|------|
| 1 | **list_sessions查询** | sessions.py:392-489 | page_size=1，但ORDER BY需要扫描全表 |
| 2 | **getSessionMessages无分页** | sessions.py:595-603 | 无LIMIT，一次返回所有消息 |
| 3 | **JSON解析** | sessions.py:616 | 每条消息的execution_steps都需要json.loads() |
| 4 | **前端渲染** | NewChatContainer.tsx:1570 | setMessages()触发大量MessageItem重渲染 |

### 1.3 后端SQL分析

**sessions.py:595-603**：
```python
cursor.execute(
    '''SELECT id, session_id, role, content, timestamp, execution_steps, display_name
       FROM chat_messages 
       WHERE session_id = ?
       ORDER BY timestamp ASC''',  # ← 注意：无LIMIT限制！
    (session_id,)
)
```

**问题**：如果会话消息很多（如100+条），每次加载都返回全部消息，每条消息还要解析JSON。

### 1.4 前端渲染瓶颈

**NewChatContainer.tsx:1570**：
```typescript
setMessages(result.messages);  // ← 触发React重渲染，渲染所有MessageItem组件
```

**问题**：大量MessageItem组件同时渲染，导致UI卡顿。

## 2. 问题B：发送后UI更新慢

### 2.1 handleSend完整流程

**代码位置**：NewChatContainer.tsx:1802-1935

```
handleSend(messageContent)
  ↓ 第1822行：network检查
  checkNetworkConnection()  ← 同步调用，可能延迟
  ↓ 第1844行：创建会话（如果没有sessionId）
  sessionApi.createSession()  ← 同步调用
  ↓ 第1653行：保存用户消息
  sessionApi.saveMessage()  ← 同步调用，等待数据库返回
  ↓ 第1878行：安全检测
  securityApi.checkCommand()  ← 同步调用
  ↓ 第1894行：发送消息
  executeStreamSend()
    → 创建assistant占位消息
    → sendStreamMessage()  ← SSE流式返回
```

### 2.2 关键延迟点

| 序号 | 延迟点 | 行号 | 原因 |
|------|-------|------|------|
| 1 | **网络连接检查** | 1822-1836 | checkNetworkConnection()额外网络往返 |
| 2 | **创建新会话** | 1844-1851 | 如果没有sessionId，需要API调用 |
| 3 | **保存用户消息** | 1653-1681 | sessionApi.saveMessage()同步调用，必须等数据库返回 |
| 4 | **安全检测** | 1878-1881 | securityApi.checkCommand()同步调用 |
| 5 | **SSE响应等待** | 1620-1722 | sendStreamMessage，等待后端流式返回 |

### 2.3 代码分析

**第1653-1681行：先保存用户消息到后端**：
```typescript
const saveResult = await sessionApi.saveMessage(currentSessionId, {
  role: "user",
  content: userMessage.content,
  client_os: clientInfo.client_os,
  browser: clientInfo.browser,
  device: clientInfo.device,
  network: clientInfo.network,
});
```

**问题**：这是同步调用，必须等待数据库返回message_id后才能创建assistant消息。

**第1878-1881行：安全检测**：
```typescript
setCheckingDanger(true);
const checkResult = await securityApi.checkCommand(userMessage.content);
```

**问题**：每次发送消息都要经过安全检测，增加额外网络往返。

### 2.4 SSE回调链

**sse.ts:966-1106**：
```
onStep(start)  → 更新executionSteps → UI显示步骤
onStep(chunk) → onChunk回调 → UI显示内容  
onStep(final) → onComplete → setLoading(false)
```

**结论**：handleSend的每个步骤是**串联**执行，等待上一个完成才执行下一个，整体延迟是累加的。

```python
async def update_session(session_id: str, update_data: SessionUpdate):
    # 使用原子性UPDATE，检查version
    cursor.execute(
        '''UPDATE chat_sessions 
           SET title = ?, updated_at = ?, title_locked = ?, version = version + 1
           WHERE id = ? AND is_deleted = FALSE AND version = ?''',
        (update_data.title, utc_time, 1, utc_time, session_id, update_data.version)
    )
    if cursor.rowcount == 0:
        # 版本冲突，返回409
        raise HTTPException(status_code=409, detail="会话已被其他用户修改")
```
