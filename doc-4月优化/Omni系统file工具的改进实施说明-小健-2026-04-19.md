## 2. 逐项修复方案

### P1: read_file — Schema与函数签名默认值不一致

**现状**：
- `file_schema.py:35`：`ReadFileInput.limit` 默认2000，le=10000
- `file_tools.py:359`：`read_file(limit=READ_FILE_DEFAULT_LIMIT)` 即500
- LLM拿到Schema说"默认2000行"，但Python不传limit时用500

**根因**：`READ_FILE_DEFAULT_LIMIT=500` 是后来小沈加的分页方案（line 36），但没同步更新Schema。

**修复方案**：**改Schema，使其与函数签名一致**

修改 `file_schema.py` 第34-38行：
```python
# 修改前
limit: int = Field(
    default=2000,
    ge=1,
    le=10000,
    description="最大读取行数，默认为2000行，最大10000行"
)

# 修改后
limit: int = Field(
    default=500,
    ge=1,
    le=10000,
    description="最大读取行数，默认为500行，最大10000行"
)
```

**不修改函数签名的原因**：`READ_FILE_DEFAULT_LIMIT=500` 是有意设计——大文件默认读500行避免内存问题，这个值是合理的。应该让Schema反映真实行为，而不是改行为去迁就Schema。

**验证**：LLM不传limit时，Schema默认500，函数签名默认500，行为一致。

### P2: list_directory — Schema与函数签名默认值不一致

**现状**：
- `file_schema.py:69-73`：`ListDirectoryInput.max_depth` 默认10，ge=1，le=50
- `file_tools.py:566`：`list_directory(max_depth=100000)`
- LLM拿到Schema说"默认10，最大50"，实际默认递归100000层

**根因**：小沈把函数签名的max_depth改成了100000（与search_files保持一致），但没同步改Schema。

**修复方案**：**改函数签名，使其与Schema一致**

修改 `file_tools.py` 第566行：
```python
# 修改前
max_depth: int = 100000,

# 修改后

```

**不修改Schema的原因**：list_directory的max_depth=10是合理的默认值。用户通常只想看一层目录，递归10层已经覆盖绝大多数场景。100000层递归对大目录（如E盘49万文件）是灾难性的。

### P3: search_file_content — while True死循环

**现状**（`file_tools.py:1006-1091`）：
```python
seen_count = 0
while True:
    batch_results = []
    for root, dirs, files in os.walk(search_path):  # 每次从头遍历！
        for filename in files:
            if seen_count < start_offset:  # 跳过已处理的
                seen_count += 1
                continue
            # ... 搜索逻辑 ...
            if len(batch_results) >= BATCH_SIZE:
                break
    all_results.extend(batch_results)
    if not batch_results:
        break  # 唯一退出条件
    # 【删除 max_results 限制判断】← 删除后无退出！
```

**问题分析**：
1. `os.walk` 每次循环都从头遍历目录树
2. `seen_count` 跳过已处理文件，但 `batch_results` 每次都会收集到新结果（因为BATCH_SIZE=10000，目录中通常有足够多文件）
3. 删除max_results后，`while True` 没有退出条件——只要目录中有匹配文件，`batch_results` 永远不为空
4. 实际上 `seen_count` 机制让每次循环跳过更多文件，最终 `batch_results` 会为空，循环会退出。**但前提是目录中文件总数有限**。如果目录非常大（如49万文件），这个循环会运行非常多次，每次都从头os.walk再跳过seen_count个文件，**时间复杂度是O(n²)**。

**修复方案**：**去掉while True循环，单次遍历即可**

`while True` 循环的原始意图是分批获取结果（每批BATCH_SIZE），但删除max_results后，目标是"返回全部结果"。既然要全部结果，**一次遍历就够了**，不需要分批循环。

修改 `file_tools.py` 的 `_search_sync` 函数（line 992-1095）：
```python
def _search_sync():
    import os
    
    all_results = []
    search_term = pattern.strip()
    start_offset = decode_page_token(page_token) if page_token else 0
    
    seen_count = 0
    for root, dirs, files in os.walk(search_path):
        # 【新增】尊重 recursive 参数
        if not recursive:
            dirs.clear()  # 不递归：清空子目录列表，os.walk不会进入子目录
            # 注意：仅对第一层生效，因为os.walk第一层dirs清空后不会递归
        
        for filename in files:
            # file_pattern 过滤
            if file_pattern and file_pattern != "*":
                # 【修改】用fnmatch替代手工正则（见P10）
                import fnmatch
                if not fnmatch.fnmatch(filename, file_pattern):
                    continue
            
            file_path = Path(root) / filename
            file_str = str(file_path.relative_to(search_path))
            
            # page_token 偏移跳过
            if seen_count < start_offset:
                seen_count += 1
                continue
            
            seen_count += 1
            
            # 读取文件内容
            try:
                with open(file_path, 'r', encoding='utf-8', errors='replace') as f:
                    content = f.read()
            except (PermissionError, OSError):
                continue
            
            # 搜索内容
            matches = []
            if use_regex and regex is not None:
                for match in regex.finditer(content):
                    start = max(0, match.start() - 50)
                    end = min(len(content), match.end() + 50)
                    context = content[start:end]
                    matches.append({
                        "start": match.start(),
                        "end": match.end(),
                        "matched": match.group(),
                        "context": context
                    })
            else:
                idx = content.find(search_term)
                while idx != -1:
                    start = max(0, idx - 50)
                    end = min(len(content), idx + len(search_term) + 50)
                    context = content[start:end]
                    matches.append({
                        "start": idx,
                        "end": idx + len(search_term),
                        "matched": search_term,
                        "context": context
                    })
                    idx = content.find(search_term, idx + 1)
            
            if matches:
                all_results.append({
                    "file": file_str,
                    "matches": matches,
                    "match_count": len(matches)
                })
    
    # 排序：匹配多的文件在前
    all_results.sort(key=lambda x: x["match_count"], reverse=True)
    
    return all_results
```

**关键改动**：
1. 去掉 `while True` 循环，改为单次 `for root, dirs, files in os.walk()`
2. 去掉 `BATCH_SIZE` 限制（已无意义）
3. 顺便修复P6（加 `dirs.clear()` 尊重recursive参数）
4. 顺便修复P10（用 `fnmatch.fnmatch` 替代手工正则）
5. 顺便修复P11（`errors='ignore'` → `errors='replace'`）

### P4: search_files — while True死循环

**现状**（`file_tools.py:1244-1320`）：与P3同样的 `while True` + `os.walk` 从头遍历模式。

**修复方案**：**同P3，去掉while True循环，单次遍历**

修改 `file_tools.py` 的 `_search_sync` 函数（line 1231-1322）：
```python
def _search_sync():
    import os
    import fnmatch
    
    all_matches = []
    seen_files = set()
    start_offset = decode_page_token(page_token) if page_token else 0
    
    for root, dirs, files in os.walk(search_path):
        # 深度限制
        if recursive:
            rel_root = Path(root).relative_to(search_path)
            depth = len(rel_root.parts) if str(rel_root) != "." else 0
            if depth >= max_depth:
                dirs.clear()  # 不再深入此目录的子目录
                continue
        else:
            dirs.clear()  # 不递归
        
        for filename in files:
            # 【修改】用fnmatch替代手工正则（见P10）
            if not fnmatch.fnmatch(filename, file_pattern):
                continue
            
            file_path = Path(root) / filename
            file_str = str(file_path.relative_to(search_path))
            
            if file_str in seen_files:
                continue
            
            # page_token 偏移跳过
            current_idx = len(seen_files)
            if current_idx < start_offset:
                seen_files.add(file_str)
                continue
            
            seen_files.add(file_str)
            
            try:
                size = file_path.stat().st_size
            except (PermissionError, OSError):
                size = 0
            
            all_matches.append({
                "name": filename,
                "path": file_str,
                "size": size
            })
    
    return all_matches
```

**关键改动**：
1. 去掉 `while True` 循环，改为单次 `os.walk`
2. 去掉 `BATCH_SIZE` 和 `batch_matches`/`batch_seen`（已无意义）
3. 去掉无用的 `after = last_file` 变量（line 1317）
4. 顺便修复P10（用 `fnmatch.fnmatch`）
5. `recursive=False` 时用 `dirs.clear()` 而非深度判断

**修复方案**：**改Schema默认值为用户主目录**

修改 `file_schema.py`：

```python
# SearchFileContentInput
path: str = Field(
```

- 比当前工作目录 `"."` 更可预测

### P9: move_file — 目标文件已存在时静默覆盖

**现状**（`file_tools.py:874-877`）：
```python
def _move_sync():
    dst.parent.mkdir(parents=True, exist_ok=True)
    shutil.move(str(src), str(dst))
    return True
```

`shutil.move` 在目标文件已存在时的行为：
- 同一文件系统：`os.rename`，目标文件被**静默覆盖**
- 跨文件系统：先copy再delete，目标文件被**静默覆盖**

**修复方案**：移动前检查目标是否存在，存在则报错

修改 `_move_sync`：
```python
def _move_sync():
    dst.parent.mkdir(parents=True, exist_ok=True)
    # 【新增】检查目标是否已存在
    if dst.exists():
        raise FileExistsError(f"目标路径已存在: {dst}，移动操作已取消。请先删除目标文件或指定其他路径。")
    shutil.move(str(src), str(dst))
    return True
```

**为什么不加overwrite参数**：
- move_file是LLM调用的工具，静默覆盖对LLM来说太危险
- LLM无法像人类一样确认"是的，覆盖"
- 报错后LLM可以先delete_file再move_file，流程更安全
- 如果未来确实需要overwrite，可以再加参数，但默认必须是False

### P10: 通配符转正则bug

**现状**（`file_tools.py:1018` 和 `1250`）：
```python
fp = file_pattern.replace(".", r"\.").replace("*", ".*").replace("?", ".")
fp = f"^{fp}$"
```

**Bug示例**：
- `*.py` → `^.*\.py$` → 匹配 `test.py.bak`（`.*`贪婪匹配`test`，`.py`匹配中间）
- `config.*` → `^config\..*$` → 匹配 `config.yaml.bak`（同上）
- `test?.py` → `^test.\.py$` → 正确

**修复方案**：**用标准库 `fnmatch.fnmatch` 替代手工正则**

Python的 `fnmatch` 模块专门处理文件名通配符，语义与shell一致：
- `*.py` 只匹配 `test.py`，不匹配 `test.py.bak`
- `config.*` 只匹配 `config.yaml`，不匹配 `config.yaml.bak`

已在P3和P4的修复中一并替换。具体改动：

`search_file_content` 中（原line 1016-1024）：
```python
# 修改前

    import re

    fp = f"^{fp}$"
    try:
        if not re.match(fp, filename):
            continue
    except:
        pass

# 修改后
if file_pattern and file_pattern != "*":
    import fnmatch
    if not fnmatch.fnmatch(filename, file_pattern):
        continue
```

`search_files` 中（原line 1248-1255）：
```python

regex_pattern = f"^{regex_pattern}$"
try:
    file_regex = re.compile(regex_pattern)
except re.error:
    file_regex = None
...

    continue

# 修改后

...

    continue
```

### P11: read_file — errors='ignore'静默丢弃解码错误

**现状**（`file_tools.py:391`）：
```python
with open(path, 'r', encoding=encoding, errors='ignore') as f:
```

**问题**：`errors='ignore'` 静默跳过无法解码的字节。如果文件是GBK编码但用UTF-8打开，中文字符会悄悄消失，用户完全不知道内容被截断。

**修复方案**：改为 `errors='replace'`

```python
# 修改前
with open(path, 'r', encoding=encoding, errors='ignore') as f:

# 修改后
with open(path, 'r', encoding=encoding, errors='replace') as f:
```

`errors='replace'` 会用 `\ufffd`（�）替代无法解码的字节，用户能看到有内容丢失，比静默丢弃好。

**进一步优化**（可选）：在返回结果中加 `encoding_warning` 字段，如果内容中包含�，提示用户可能编码错误：
```python
content_str = "".join(f"{i}: {line}" for i, line in enumerate(selected_lines, start=offset))
encoding_warning = None
if "\ufffd" in content_str and encoding == "utf-8":
```

### P12: write_file — 非原子写入

**现状**（`file_tools.py:497-500`）：
```python
def _write_sync():
    path.parent.mkdir(parents=True, exist_ok=True)
    with open(path, 'w', encoding=encoding) as f:
        f.write(content)
    return True
```

**问题**：直接 `open('w')` 会先清空文件再写入。如果写入过程中断（进程崩溃、磁盘满），原内容已丢失，新内容不完整。

**修复方案**：先写临时文件，再原子重命名

```python
def _write_sync():
    path.parent.mkdir(parents=True, exist_ok=True)
    # 写入临时文件
    import tempfile
    fd, tmp_path = tempfile.mkstemp(
        dir=path.parent,
        prefix='.~',
        suffix='.tmp'
    )
    try:
        with os.fdopen(fd, 'w', encoding=encoding) as f:
            f.write(content)
        # 原子重命名（os.replace在Windows和Linux都是原子操作）
        os.replace(tmp_path, path)
    except Exception:
        # 写入失败，删除临时文件，原文件不受影响
        try:
            os.unlink(tmp_path)
        except OSError:
            pass
        raise
    return True
```

**原理**：
- `tempfile.mkstemp` 创建临时文件（同目录，保证同一文件系统）
- `os.replace` 在POSIX和Windows上都是原子操作
- 写入失败时原文件完好，不会出现半写状态

**注意**：需要在文件顶部加 `import os`（已有）。

### P13: _validate_path — 字符串前缀匹配漏洞

**现状**（`file_tools.py:309`）：
```python
if str(real_path).startswith(str(allowed_real)):
    return True, None
```

**Bug示例**：
- 白名单允许 `/home/user`
- `/home/userbackdoor` 的字符串以 `/home/user` 开头 → **通过验证！**
- `/home/user/` 的子目录 → 正确通过

**修复方案**：用 `Path.is_relative_to()` 替代字符串前缀匹配

```python
# 修改前
for allowed in self.allowed_paths:
    allowed_real = Path(os.path.realpath(allowed))
    if str(real_path).startswith(str(allowed_real)):
        return True, None

# 修改后
for allowed in self.allowed_paths:
    allowed_real = Path(os.path.realpath(allowed))
    try:
        if real_path.is_relative_to(allowed_real):
            return True, None
    except (ValueError, AttributeError):
        # Python < 3.9 fallback
        if str(real_path).startswith(str(allowed_real) + os.sep) or real_path == allowed_real:
            return True, None
```

`Path.is_relative_to()` 是Python 3.9+的方法，语义正确：`/home/userbackdoor` 不相对于 `/home/user`，但 `/home/user/docs` 相对于 `/home/user`。

**兼容性**：如果需要支持Python 3.8，用fallback逻辑：`startswith(allowed + os.sep) or path == allowed`。
