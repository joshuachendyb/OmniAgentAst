## 一、引言：三个问题，一个病根

本方案针对当前系统中三个最核心的功能模块进行彻底重构：

| 序号 | 模块 | 代码位置 | 作用 |
|------|------|---------|------|
| 1 | **预处理 (PreprocessingPipeline)** | `preprocessing/pipeline.py` | 文本矫正 + 意图检测 → 实际上**仅strip** |
| 2 | **意图检测 (route_with_fallback)** | `chat_router.py:66` | CRSS+LLM两阶段意图路由 |
| 3 | **安全检测 (check_command_safety)** | `command_security.py:637` | 危险命令黑名单匹配 |

**病根只有一个：管线断裂（Pipeline Fragmentation）**

这三个功能被实现为**顺序排列但互不关联**的独立步骤。每一步的输出被丢弃，不被下一步使用。数据不流动，价值不叠加。

```
步骤1: 预处理 → corrected(丢弃)  intent="unknown"(丢弃)
步骤2: 意图检测 → intent(仅用于分发)  corrected(丢弃)
步骤3: 安全检测 → blocked(仅日志，不中断)
步骤4: Agent执行 → 在原文上执行
                              ← 断裂！每一步都在独立的战场上作战
```

## 二、逐个功能深度诊断

### 2.1 预处理（PreprocessingPipeline）

#### 2.1.1 代码真相

```python
# pipeline.py — 全部有效逻辑（2026-04-30后被掏空）
async def process(self, user_input, session_id=""):
    corrected = user_input.strip()    # 仅strip
    return {
        "original": user_input,
        "corrected": corrected,        # = user_input.strip()
        "intent": "unknown",          # 永远unknown
        "confidence": 0.0,            # 永远0.0
        "all_intents": {},            # 永远空
    }
```

#### 2.1.2 为什么会变成这样？

**历史演变**：
1. **2026-03-20**：原始设计意图检测+文本矫正都在`PreprocessingPipeline`中
2. **2026-04-30**：意图检测逻辑被剥离到`route_with_fallback()`（两阶段路由）
3. **2026-04-30**：LLM调用从`PreprocessingPipeline`中移除
4. **之后**：没有人清理这个空壳，它一直活着但只做`strip()`

**问题根因**：**不是设计问题，是重构不完整**。原始设计（连贯的预处理→矫正→意图检测）是有价值的，但在2026-04-30重构时，把意图检测抽走了却**没有清理空壳**，也没有**让矫正结果继续流向意图检测**。

#### 2.1.3 是否有价值？

| 方面 | 价值判断 |
|------|---------|
| **文本矫正** | ✅ **非常有价值**。用户输入"删处所有文件"（错别字"处"→"除"），CRSS无法识别，安全检测无法拦截。矫正后"删除所有文件"→SHELL意图→安全检测→拦截 |
| **预处理流水线概念** | ✅ **有价值**。把矫正→意图→安全串联成流水线，数据自动流转 |
| **当前实现** | ❌ **零价值**。仅做`strip()`，浪费一次`async`调用和一次日志打印 |

**结论**：保留"文本矫正"功能，抛弃"空壳PreprocessingPipeline"形式。

---

### 2.2 意图检测（route_with_fallback，CRSS+LLM两阶段）

#### 2.2.1 设计评价

**框架设计：优秀**。两阶段路由（规则快速匹配→LLM兜底）是业界成熟的意图分类模式，比单一规则或单一LLM都更优。

- 优点：80%+的请求在CRSS阶段完成（毫秒级），只有模糊输入才走LLM（秒级）
- 优点：CRSS双维度评分（类型×动作）设计合理，有层次感

#### 2.2.2 致命缺陷

| 缺陷 | 代码位置 | 影响 | 严重性 |
|------|---------|------|--------|
| **无chat意图** | `chat_router.py:121` | 无匹配意图fallback到`network`，纯对话"你好"会触发网络搜索 | 🔴 **高频故障** |
| **CRSS在原文上执行** | `route_with_fallback():73` 调用`detect_intent_v2(user_input)` | `user_input`是原始未矫正文本，错别字导致CRSS失效 | 🟠 **功能失效** |
| **LLM矫正结果未回传** | `classify_intent()`返回`corrected`，但调用方不用 | 白花LLM调用的矫正能力，安全检测在错误文本上执行 | 🟠 **资源浪费** |
| **置信度阈值硬编码** | `crss_scorer.py:15` `CRSS_CONFIDENCE_THRESHOLD = 0.3` | 无法动态调整，无匹配场景一刀切 | 🟡 **灵活性差** |
| **意图与安全脱钩** | 安全检测不知道意图结果 | `chat`意图也走安全检测浪费资源；`shell`意图安全检测应加严 | 🟡 **效率低** |

#### 2.2.3 是否有价值？

| 方面 | 价值判断 |
|------|---------|
| **两阶段路由框架** | ✅ **非常值得保留**。CRSS+LLM是正确方向 |
| **CRSS双维度评分** | ✅ **值得保留**。分层设计好，增量改进即可 |
| **chat意图缺失** | ❌ **必须补上**。这是当前最影响用户体验的缺陷 |
| **矫正结果未回传** | ❌ **必须修复**。这是管线断裂的核心表现之一 |

### 2.3 安全检测（check_command_safety）

#### 2.3.1 设计评价

**黑名单+正则匹配：正确的方向，但在错误的位置**。

`CommandSafetyChecker` 内部实现质量不错：
- 两级检测：黑名单精确匹配 + 正则模式匹配
- 四级风险等级：safe/low/medium/high/critical
- 支持中文危险命令

但**部署方式有严重问题**。

#### 2.3.2 两大致命缺陷

**缺陷一：第一道防线是纸老虎**

```python
# chat_router.py 步骤4 — 虚设的防线
security_check_result = check_command_safety(user_input)
if security_check_result.get('blocked', False):
    logger.warning(...)  # ← 仅记录日志！继续执行！
    # 没有return！没有yield error！没有任何中断！
```

而真正的拦截在react_sse_wrapper中：
```python
# react_sse_wrapper.py — 真正的防线
if not security_check_result.get('is_safe', True):
    yield error_response  # ← 在这里真正中断
    return
```

**后果**：
- 如果有人修改`react_sse_wrapper.py`（比如加一个功能时不小心移除了安全检查），整个安全体系就崩了
- `chat_router.py`的安全检测完全是装饰性的
- 维护者会误以为"步骤4做了安全检查"，实际上它什么都没做

**缺陷二：只检测用户输入层，不检测工具执行层**

```
用户: "帮我清空磁盘" → Agent(LLM)生成 execute_shell_command("format C: /q")
                                                          ↑
                                                   安全检测漏检！
                                                      （只检查了"帮我清空磁盘"，没检查"format C: /q"）
```

这是最危险的场景：**用户用自然语言描述危险意图，安全检测不识别，Agent的LLM生成危险命令，无拦截执行**。

#### 2.3.3 是否有价值？

| 方面 | 价值判断 |
|------|---------|
| **黑名单+正则检测** | ✅ **非常值得保留**。实现质量好，准确率高 |
| **用户输入层检测** | ✅ **必须保留**。第一道防线，拦截直接输入的危险命令 |
| **工具执行层检测** | ❌ **缺失**。这是当前最危险的安全漏洞 |
| **chat_router的虚设调用** | ❌ **必须删除或修复**。纸老虎比没有更危险 |
| **chinese危险命令支持** | ✅ **有价值**。中文Agent场景下的独特优势 |

## 三、核心设计决策：保留还是重构？

| 模块 | 决策 | 理由 |
|------|------|------|
| PreprocessingPipeline类 | ❌ **删除空壳** | 仅strip，意图永远unknown，无存在价值 |
| 文本矫正功能 | ✅ **保留并强化** | 前置为管线第一步，供所有后续步骤使用 |
| route_with_fallback | ✅ **保留并强化** | 两阶段路由框架优秀，补上缺陷即可 |
| CRSS双维度评分 | ✅ **保留** | 设计好，微调即可 |
| chat意图 | ✅ **新增** | 无匹配时走chat而非network |
| check_command_safety | ✅ **保留并强化** | 保留用户输入层检测，新增工具执行层检测 |
| chat_router的虚设检测 | ❌ **修复为真实拦截** | blocked=True必须中断流程 |

## 四、完美方案：数据驱动的完整管线

### 4.1 核心理念

**管线化而非步骤化**。每一步的输出自动成为下一步的输入，数据在管线中流动：

```
用户输入
   │
   ▼
┌──────────────────────────────────────────────────────┐
│  管线 (RequestPipeline)                              │
│                                                      │
│  步骤1: 文本矫正 ←─────────────────────────┐         │
│      ↓ corrected_text                       │         │
│  步骤2: 意图检测 (在corrected上)             │         │
│      ↓ intent + confidence + corrected_llm   │         │
│  步骤3: 安全策略 (在corrected上, 结合intent)    │         │
│      ↓ safety_decision + risk_info           │         │
│  步骤4: Agent分发 ──────────────────────────┘         │
│                                                      │
│  步骤5: Agent执行层 ─── 第二层安全守卫                │
│      ↓ (工具调用时实时检查)                           │
│  步骤6: 结果返回                                      │
└──────────────────────────────────────────────────────┘
```

### 4.2 核心数据结构：PipelineContext

```python
# backend/app/services/pipeline/context.py

from dataclasses import dataclass, field
from typing import Optional, Dict, List, Any
from enum import Enum
from app.services.tools.registry import ToolCategory


class SafetyDecision(Enum):
    ALLOW = "allow"               # 安全，直接放行
    CONFIRM = "confirm"           # 中风险，需用户确认
    BLOCK = "block"               # 高风险，直接阻止


@dataclass
class PipelineContext:
    """
    管线上下文 — 数据在管线中流动的核心载体
    
    设计原则：
    1. 每个步骤只读/写自己的字段，不越界
    2. 后续步骤永远通过 `ctx.effective_input` 获取最佳可用的文本
    3. 每个步骤可以修改上游步骤的输出（如LLM矫正更新corrected）
    """
    # ── 原始输入 ──
    raw_input: str
    
    # ── 步骤1: 文本矫正（由Corrector写入） ──
    corrected_input: str = ""
    corrections: List[Dict[str, Any]] = field(default_factory=list)
    correction_source: str = ""  # "rule" / "llm" / "none"
    
    # ── 步骤2: 意图检测（由IntentDetector写入） ──
    intent: Optional[ToolCategory] = None
    confidence: float = 0.0
    candidates: List[ToolCategory] = field(default_factory=list)
    all_intents: Dict[str, float] = field(default_factory=dict)
    intent_source: str = ""  # "crss" / "llm" / "chat_heuristic" / "default"
    is_chat_intent: bool = False  # 是否为纯对话意图
    
    # ── 步骤3: 安全策略（由SafetyAnalyzer写入） ──
    safety_decision: SafetyDecision = SafetyDecision.ALLOW
    risk_level: str = "none"
    risk_score: float = 0.0
    risk_detail: Dict[str, Any] = field(default_factory=dict)
    blocked_reason: str = ""
    confirm_message: str = ""
    
    # ── 步骤4: Agent信息（由Router写入） ──
    agent_intent_type: str = ""
    agent_category: Optional[ToolCategory] = None
    
    # ── 元信息 ──
    session_id: str = ""
    task_id: str = ""
    pipeline_version: str = "3.0"
    
    @property
    def effective_input(self) -> str:
        """
        最佳可用文本
        
        优先级：corrected_input (如有) > raw_input
        所有后续步骤（意图检测、安全检测、Agent执行）都应使用此属性
        """
        return self.corrected_input if self.corrected_input else self.raw_input
    
    @property
    def has_correction(self) -> bool:
        """是否有矫正发生"""
        return self.corrected_input != self.raw_input and bool(self.corrected_input)
```

### 4.3 管线调度器

```python
# backend/app/services/pipeline/request_pipeline.py

from app.services.pipeline.context import PipelineContext, SafetyDecision
from app.services.pipeline.corrector import TextCorrectorV2
from app.services.pipeline.intent_detector import IntentDetectorV2
from app.services.pipeline.safety_analyzer import SafetyAnalyzerV2
from app.utils.logger import logger


class RequestPipeline:
    """
    请求处理管线 — 统一调度 矫正→意图→安全→分发
    
    使用方式：
        pipeline = RequestPipeline()
        ctx = await pipeline.process("帮我删处所有文件", session_id="xxx")
        # ctx.effective_input → "帮我删除所有文件"
        # ctx.intent → SHELL
        # ctx.safety_decision → BLOCK
    """
    
    def __init__(self):
        self._corrector = TextCorrectorV2()
        self._intent_detector = IntentDetectorV2()
        self._safety_analyzer = SafetyAnalyzerV2()
    
    async def process(
        self,
        user_input: str,
        session_id: str = ""
    ) -> PipelineContext:
        """
        执行完整管线
        
        管线顺序不可变：矫正必须在意图检测之前，意图检测必须在安全策略之前
        因为：
        - 矫正错误 → 意图检测基于错误文本 → 路由错误
        - 意图检测结果 → 安全策略差异化（chat跳过，shell加严）
        """
        ctx = PipelineContext(
            raw_input=user_input,
            session_id=session_id,
        )
        
        # 步骤1: 文本矫正（在前！）
        # 矫正结果通过 ctx.effective_input 供后续所有步骤共享
        ctx = await self._corrector.correct(ctx)
        
        # 步骤2: 意图检测（在矫正后文本上执行）
        # 此时 ctx.effective_input 已经是矫正后的文本
        # 如果LLM兜底返回了更优的矫正文本，IntentDetector会更新ctx.corrected_input
        ctx = await self._intent_detector.detect(ctx)
        
        # 步骤3: 安全策略分析（在矫正后文本上执行，结合意图信息）
        # chat意图跳过安全检测（闲聊不执行工具）
        ctx = await self._safety_analyzer.analyze(ctx)
        
        # 步骤4: Agent分发决策
        ctx = self._resolve_agent(ctx)
        
        logger.info(
            f"[Pipeline] 管线完成 → "
            f"input='{ctx.raw_input[:30]}', "
            f"corrected='{ctx.corrected_input[:30] if ctx.has_correction else '(same)'}', "
            f"intent={ctx.intent}, "
            f"safety={ctx.safety_decision.value}, "
            f"source={ctx.intent_source}"
        )
        
        return ctx
    
    def _resolve_agent(self, ctx: PipelineContext) -> PipelineContext:
        """确定Agent分发目标"""
        if ctx.safety_decision == SafetyDecision.BLOCK:
            ctx.agent_intent_type = "blocked"
            ctx.agent_category = None
            return ctx
        
        if ctx.is_chat_intent:
            # chat意图 → 通用对话Agent（轻量工具集，不包含shell/file等危险工具）
            ctx.agent_intent_type = "chat"
            ctx.agent_category = ToolCategory.META
            return ctx
        
        if ctx.intent is not None:
            ctx.agent_intent_type = ctx.intent.value
            ctx.agent_category = ctx.intent
        else:
            # 极端情况：所有检测都无匹配 → 走chat
            ctx.agent_intent_type = "chat"
            ctx.agent_category = ToolCategory.META
            ctx.intent_source = "default_chat"
            ctx.is_chat_intent = True
        
        return ctx
```

### 4.4 步骤1：文本矫正器（TextCorrectorV2）

#### 4.4.1 设计要点

**为什么矫正必须第一个执行？**

```
错误顺序：先检测意图，再矫正
  原文 "删处文件" → CRSS("删处文件") → 无匹配 → LLM兜底("删处文件") → 猜错意图
  
正确顺序：先矫正，再检测意图
  原文 "删处文件" → 矫正("删除文件") → CRSS("删除文件") → FILE(1.5) → 命中！
```

**矫正策略两级**：

| 级别 | 方法 | 延迟 | 适用场景 |
|------|------|------|---------|
| L1: 规则矫正 | 字典映射+正则 | <1ms | 常见错别字（删处→删除、格式花→格式化） |
| L2: LLM矫正 | LLM分类器返回 | ~1s | CRSS无匹配时触发，LLM同时返回corrected+intent |

**注意**：L2矫正不由本类触发，而是由`IntentDetectorV2`在CRSS失败时自然触发（LLM同时返回corrected和intent），避免额外的LLM调用。

```python
    'abcdefghijklmnopqrstuvwxyz/@.-_+=,.!?'
)


class TextCorrectorV2:
    """
    文本矫正器 V2 — 管线第一步
    
    职责：
    1. 基本清理：strip + 全角半角转换 + 多余空格压缩
    2. 安全关键词错别字矫正（优先！确保安全检测在正确文本上执行）
    3. 常见错别字字典映射矫正
    4. LLM级矫正不由本类触发，由IntentDetectorV2在CRSS失败时自然触发
    
    不做什么：
    - 不做LLM调用（那是IntentDetector的职责）
    - 不做pycorrector调用（太重，延迟不可控）
    """
    
    async def correct(self, ctx: PipelineContext) -> PipelineContext:
        """
        执行规则级矫正（L1）
        
        只在有明显错别字时标记矫正，轻微差异不触发
        """
        text = ctx.raw_input
        if not text:
            ctx.corrected_input = ""
            return ctx
        
        corrected = text
        
        # 1. 基本清理
        corrected = corrected.strip()
        
        # 2. 全角→半角（数字字母符号）
        corrected = corrected.translate(_FULLWIDTH_MAP)
        
        # 3. 多余空格压缩
        corrected = re.sub(r' +', ' ', corrected)  # 连续空格→单空格
        corrected = re.sub(
            r'([\u4e00-\u9fff])\s+([\u4e00-\u9fff])',
            r'\1\2', corrected
        )  # 中文间空格→无空格
        
        # 4. 错别字矫正（安全关键词优先）
        corrections = []
        for typo, fixed in TYPO_MAP.items():
            if typo in corrected:
                corrected = corrected.replace(typo, fixed)
                corrections.append({
                    "from": typo,
                    "to": fixed,
                    "type": "security" if typo in SECURITY_TYPO_MAP else "common"
                })
        
        # 是否安全相关矫正
        has_security_correction = any(
            c["type"] == "security" for c in corrections
        )
        
        # 5. 更新上下文
        is_corrected = corrected != text
        ctx.corrected_input = corrected
        ctx.corrections = corrections
        ctx.correction_source = "rule"
        
        if is_corrected:
            logger.info(
                f"[Corrector] 规则矫正: '{text}' → '{corrected}', "
                f"corrections={corrections}"
            )
            if has_security_correction:
                logger.warning(
                    f"[Corrector] ⚠ 安全关键词矫正: {corrections}, "
                    f"原文可能含有危险操作意图但被错别字掩盖"
                )
        
        return ctx
```

#### 4.4.3 关键设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 安全关键词错别字优先级 | 最高（覆盖普通映射） | 安全第一：确保安全关键词不错过 |
| 全角半角转换 | 做 | 中文用户常输入全角字符，影响CRSS匹配 |
| 中文间空格压缩 | 做 | 中文空格影响关键词匹配（"删 除"不匹配"删除"） |
| pycorrector | 不用 | 太重（加载模型~1s），效果不稳定 |
| LLM矫正 | 不在此触发 | 避免额外LLM调用，由IntentDetector在CRSS失败时自然触发 |

### 4.5 步骤2：意图检测器（IntentDetectorV2）

#### 4.5.1 设计要点

**关键改进**：
1. 所有检测在 `ctx.effective_input`（矫正后文本）上执行
2. 新增chat启发式：CRSS无匹配时先判断是否为闲聊，再调LLM兜底
3. LLM矫正结果更新到PipelineContext（`ctx.corrected_input`）
4. chat意图走轻量Agent（无shell/file等危险工具）

#### 4.5.2 代码设计

```python
# backend/app/services/pipeline/intent_detector.py

from typing import Optional
from app.services.pipeline.context import PipelineContext, SafetyDecision
from app.services.intents.crss_scorer import detect_intent_v2, CRSS_CONFIDENCE_THRESHOLD
from app.services.tools.registry import ToolCategory, resolve_category
from app.utils.logger import logger


# ═══════════════════════════════════════════════════════════════
# Chat意图判定关键词
# CRSS无匹配时，用这些关键词判断是否为纯对话
# ═══════════════════════════════════════════════════════════════
_CHAT_PATTERNS = [
    "你好", "您好", "谢谢", "感谢", "再见", "拜拜", "晚安",
    "请问", "讲个", "什么是", "为什么", "怎么样", "如何理解",
    "解释一下", "说说", "聊聊", "你觉得", "你认为",
    "hello", "hi", "thanks", "bye", "good morning",
    "?", "？",  # 单独的疑问词（"？"单独使用）
]

# ═══════════════════════════════════════════════════════════════
# 危险操作否定关键词
# 即使看起来像chat，包含这些词也不能走chat
# ═══════════════════════════════════════════════════════════════
_DANGEROUS_INDICATORS = [
    "删除", "格式化", "清除", "卸载", "杀死", "终止",
    "rm", "format", "del", "kill", "drop",
]


class IntentDetectorV2:
    """
    意图检测器 V2 — 管线第二步
    
    改进点（vs route_with_fallback）：
    1. 在矫正后的文本(ctx.effective_input)上执行CRSS
    2. 新增chat意图：CRSS无匹配且无危险关键词时走chat
    3. LLM矫正结果更新到PipelineContext
    4. candidates按置信度排序
    """
    
    async def detect(self, ctx: PipelineContext) -> PipelineContext:
        """执行两阶段意图检测"""
        text = ctx.effective_input  # ★ 使用矫正后的文本！
        
        if not text:
            ctx.is_chat_intent = True
            ctx.intent_source = "empty_input"
            return ctx
        
        # ===== 阶段1: CRSS快速匹配 =====
        primary, candidates, confidence = detect_intent_v2(text)
        
        if primary is not None and confidence >= CRSS_CONFIDENCE_THRESHOLD:
            ctx.intent = primary
            ctx.confidence = confidence
            ctx.candidates = candidates
            ctx.intent_source = "crss"
            logger.info(
                f"[IntentDetector] CRSS ✓ → {primary.value} "
                f"(conf={confidence:.4f})"
            )
            return ctx
        
        # ===== CRSS无匹配或低置信度：先判断是否为闲聊 =====
        if self._is_chat_query(text):
            ctx.intent = None
            ctx.confidence = 0.4
            ctx.candidates = []
            ctx.intent_source = "chat_heuristic"
            ctx.is_chat_intent = True
            logger.info(
                f"[IntentDetector] Chat启发式 ✓ → chat意图 "
                f"(input='{text[:30]}')"
            )
            return ctx
        
        # ===== 阶段2: LLM语义分类（兜底）=====
        try:
            from app.services.preprocessing.intent_classifier import classify_intent
            
            intent_labels = [c.value for c in ToolCategory]
            llm_result = await classify_intent(text, intent_labels)
            
            intent_str = llm_result.get("intent", "")
            llm_confidence = float(llm_result.get("confidence", 0.5))
            intent_enum = resolve_category(intent_str) if intent_str else None
            
            # ★★★ 关键改进：LLM矫正结果更新到上下文 ★★★
            llm_corrected = llm_result.get("corrected", "")
            if llm_corrected and llm_corrected != text:
                ctx.corrected_input = llm_corrected
                ctx.correction_source = "llm"
                ctx.corrections.append({
                    "from": text,
                    "to": llm_corrected,
                    "type": "llm"
                })
                logger.info(
                    f"[IntentDetector] LLM矫正: "
                    f"'{text}' → '{llm_corrected}'"
                )
            
            if intent_enum:
                ctx.intent = intent_enum
                ctx.confidence = llm_confidence
                ctx.candidates = [intent_enum]
                ctx.all_intents = llm_result.get("all_intents", {})
                ctx.intent_source = "llm"
            else:
                # LLM也没识别出来 → 走chat
                ctx.is_chat_intent = True
                ctx.intent_source = "llm_fallback_chat"
            
            logger.info(
                f"[IntentDetector] LLM → {intent_str}({intent_enum}), "
                f"conf={llm_confidence}"
            )
            
        except Exception as e:
            logger.warning(f"[IntentDetector] LLM兜底失败: {e}")
            # LLM失败时：如果CRSS有低置信度结果就用它，否则走chat
            if primary is not None:
                ctx.intent = primary
                ctx.confidence = confidence * 0.5  # 降权
                ctx.intent_source = "crss_llm_fallback"
            else:
                ctx.is_chat_intent = True
                ctx.intent_source = "default_chat"
        
        return ctx
    
    def _is_chat_query(self, text: str) -> bool:
        """
        启发式判断是否为纯对话
        
        规则：
        1. 文本短（<40字）且包含chat关键词
        2. 不包含任何危险操作关键词
        3. 纯疑问句（以"？/?/"结尾且不包含操作型动词）
        """
        if len(text) > 60:
            return False  # 长文本不太可能是纯闲聊
        
        # 包含危险关键词 → 不走chat
        text_lower = text.lower()
        for indicator in _DANGEROUS_INDICATORS:
            if indicator.lower() in text_lower:
                return False
        
        # 检查chat模式
        for pattern in _CHAT_PATTERNS:
            if pattern in text:
                return True
        
        # 纯标点/表情/单字
        cleaned = re.sub(r'[\s\.,!?，。！？、~～]+', '', text)
        if len(cleaned) <= 3:
            return True
        
        return False
```

#### 4.5.3 与route_with_fallback的关键差异

| 方面 | 当前(route_with_fallback) | 新设计(IntentDetectorV2) |
|------|--------------------------|------------------------|
| 输入文本 | 原始user_input | ctx.effective_input（矫正后） |
| chat意图 | ❌ 无，fallback到network | ✅ 启发式判定，走轻量chat Agent |
| LLM矫正结果 | ❌ 丢弃 | ✅ 更新到ctx.corrected_input |
| 危险关键词否定 | ❌ 无 | ✅ 即使像chat，含危险词也不走chat |
| CRSS低置信度处理 | 直接调LLM | 先判断chat，再调LLM |

### 4.6 步骤3：安全策略分析器（SafetyAnalyzerV2）

#### 4.6.1 设计要点

**核心改进**：
1. 在矫正后文本上执行安全检测（`ctx.effective_input`）
2. 结合意图信息做差异化安全策略（chat跳过，shell加严）
3. 输出`SafetyDecision`决策（不是仅日志！）
4. `BLOCK`直接中断管线流程

#### 4.6.2 代码设计

```python
# backend/app/services/pipeline/safety_analyzer.py

from app.services.pipeline.context import PipelineContext, SafetyDecision
from app.services.command_security import check_command_safety, calculate_risk_score_v2
from app.services.tools.registry import ToolCategory
from app.utils.logger import logger


class SafetyAnalyzerV2:
    """
    安全策略分析器 V2 — 管线第三步
    
    差异化策略：
    - chat意图：跳过安全检测（闲聊不执行任何工具）
    - shell意图：加严检测（低风险也需确认）
    - file/network/desktop：标准检测
    - 其他：宽松检测
    
    输出决策（不再是blocked bool）：
    - ALLOW：安全，放行
    - CONFIRM：中风险，需用户确认
    - BLOCK：高风险/极高风险，直接阻止
    """
    
    # 各意图类型的加严系数（>1.0 = 更严格，<1.0 = 更宽松）
    INTENT_SENSITIVITY = {
        ToolCategory.SHELL: 1.5,      # shell命令→加严50%
        ToolCategory.FILE: 1.2,       # 文件操作→加严20%
        ToolCategory.DESKTOP: 1.0,    # 桌面操作→标准
        ToolCategory.NETWORK: 1.2,    # 网络操作→加严20%
        ToolCategory.SYSTEM: 1.3,     # 系统操作→加严30%
        ToolCategory.DOCUMENT: 0.8,   # 文档操作→降低20%
        ToolCategory.META: 0.5,       # 元信息→降低50%
    }
    
    # 低风险级别在哪些意图下视为安全
    LOW_RISK_SAFE_INTENTS = {
        ToolCategory.META,
        ToolCategory.DOCUMENT,
    }
    
    async def analyze(self, ctx: PipelineContext) -> PipelineContext:
        """执行安全策略分析"""
        text = ctx.effective_input  # ★ 使用矫正后的文本！
        
        # ── chat意图 → 跳过安全检测 ──
        # chat意图不执行任何工具，无需安全检查
        if ctx.is_chat_intent:
            ctx.safety_decision = SafetyDecision.ALLOW
            ctx.risk_level = "none"
            logger.info(
                f"[SafetyAnalyzer] Chat意图 → 跳过安全检测"
            )
            return ctx
        
        # ── 执行安全检测 ──
        security_result = check_command_safety(text)
        
        is_safe = security_result.get('is_safe', True)
        risk_level = security_result.get('risk_level', 'none')
        blocked = security_result.get('blocked', False)
        is_need_confirm = security_result.get('is_need_confirm', False)
        
        # ── 意图感知的决策调制 ──
        sensitivity = self.INTENT_SENSITIVITY.get(ctx.intent, 1.0) if ctx.intent else 1.0
        
        # 低风险 + 该意图下低风险视为安全 → ALLOW
        if is_safe or (
            risk_level == "low" 
            and ctx.intent in self.LOW_RISK_SAFE_INTENTS
        ):
            ctx.safety_decision = SafetyDecision.ALLOW
            ctx.risk_level = risk_level or "safe"
            logger.info(
                f"[SafetyAnalyzer] ALLOW: intent={ctx.intent}, "
                f"risk={risk_level}"
            )
        
        # 中风险 × 加严系数 >= 1.5 → 升级为BLOCK
        elif risk_level == "medium":
            if sensitivity >= 1.5:
                ctx.safety_decision = SafetyDecision.BLOCK
                ctx.risk_level = "high"  # 升级
                ctx.blocked_reason = (
                    f"意图({ctx.intent.value})下中风险操作被加严拦截: "
                    f"{security_result.get('risk', '')}"
                )
                logger.warning(
                    f"[SafetyAnalyzer] BLOCK(升级): intent={ctx.intent}, "
                    f"sensitivity={sensitivity}, reason={ctx.blocked_reason}"
                )
            else:
                ctx.safety_decision = SafetyDecision.CONFIRM
                ctx.risk_level = "medium"
                ctx.confirm_message = (
                    f"检测到中风险操作：{security_result.get('risk', '')}。"
                    "是否继续执行？"
                )
                logger.info(
                    f"[SafetyAnalyzer] CONFIRM: intent={ctx.intent}, "
                    f"risk={risk_level}"
                )
        
        # 高风险/极高风险 → BLOCK
        elif blocked or risk_level in ("high", "critical"):
            ctx.safety_decision = SafetyDecision.BLOCK
            ctx.risk_level = risk_level
            ctx.blocked_reason = (
                f"安全拦截({risk_level}): "
                f"{security_result.get('risk', '未知风险')}"
            )
            logger.warning(
                f"[SafetyAnalyzer] BLOCK: intent={ctx.intent}, "
                f"risk={risk_level}, reason={ctx.blocked_reason}"
            )
        
        # 低风险 → ALLOW
        else:
            ctx.safety_decision = SafetyDecision.ALLOW
            ctx.risk_level = risk_level or "safe"
            logger.info(
                f"[SafetyAnalyzer] ALLOW: intent={ctx.intent}, "
                f"risk={risk_level}"
            )
        
        # 获取CRSS风险评分详情（供前端展示）
        try:
            risk_detail = calculate_risk_score_v2(text)
            ctx.risk_score = risk_detail.get('score', 0)
            ctx.risk_detail = risk_detail
        except Exception:
            pass
        
        return ctx
```

### 4.8 第二层安全：工具执行层守卫（ToolExecutionGuard）

#### 4.8.1 为什么需要第二层？

**用户输入层检测（第一层）的盲区**：

```
用户: "帮我清理一下系统缓存"
  → 管线矫正: "帮我清理一下系统缓存"（无错别字，无黑名单匹配）
  → 意图检测: SHELL(0.6) → 安全策略: ALLOW（低风险）
  → Agent分发: shell Agent
  
Agent(LLM)执行:
  Thought: 用户要清理系统缓存，我应该执行 rm -rf /tmp/*
  Action: execute_shell_command(command="rm -rf /tmp/*")
                                         ↑
                                    第一层安全漏检！
                                    （"帮我清理一下系统缓存"不匹配任何黑名单）
```

**第二层守卫**在工具执行时检查实际的工具调用参数，拦截危险的Shell命令。

#### 4.8.2 代码设计

```python
# backend/app/services/agent/tool_execution_guard.py

from typing import Dict, Any, Tuple, Set
from app.services.command_security import check_command_safety
from app.utils.logger import logger


# ═══════════════════════════════════════════════════════════════
# 需要安全守卫的工具
# ═══════════════════════════════════════════════════════════════
GUARDED_TOOLS: Dict[str, str] = {
    "execute_shell_command": "command",     # Shell命令执行
    "execute_python": "code",               # Python代码执行
    "execute_javascript": "code",           # JS代码执行
    "shell_session": "command",             # Shell会话
    "run_command": "command",               # 通用命令执行
}


class ToolExecutionGuard:
    """
    工具执行安全守卫 — 第二层安全
    
    在ToolExecutor.execute()中调用，拦截Agent LLM生成的危险工具调用。
    
    与第一层（用户输入层）的区别：
    - 第一层：检测"用户说了什么"
    - 第二层：检测"Agent要执行什么"
    
    配合效果：
    - 用户直接输入"rm -rf /" → 第一层拦截（黑名单匹配）
    - 用户说"清空系统" → Agent生成"rm -rf /" → 第二层拦截
    """
    
    def __init__(self, enabled: bool = True):
        self._enabled = enabled
        self._stats = {"blocked": 0, "allowed": 0, "skipped": 0}
    
    def check(
        self,
        tool_name: str,
        tool_params: Dict[str, Any]
    ) -> Tuple[bool, str]:
        """
        检查工具调用是否安全
        
        Returns:
            (is_safe, reason): 是否安全 + 不安全的原因
        """
        if not self._enabled:
            self._stats["skipped"] += 1
            return True, ""
        
        # 非守卫工具 → 放行
        if tool_name not in GUARDED_TOOLS:
            self._stats["skipped"] += 1
            return True, ""
        
        # 提取命令参数
        param_name = GUARDED_TOOLS[tool_name]
        command = tool_params.get(param_name, "")
        
        if not command or not isinstance(command, str):
            self._stats["skipped"] += 1
            return True, ""
        
        # 执行安全检测（复用check_command_safety！）
        security_result = check_command_safety(command)
        
        is_safe = security_result.get("is_safe", True)
        risk_level = security_result.get("risk_level", "safe")
        blocked = security_result.get("blocked", False)
        risk = security_result.get("risk", "")
        
        if not is_safe:
            if blocked:
                # 高风险 → 阻止
                self._stats["blocked"] += 1
                logger.warning(
                    f"[ToolGuard] 🚫 BLOCKED: tool={tool_name}, "
                    f"risk={risk_level}, cmd='{command[:80]}'"
                )
                return False, f"工具安全拦截: {risk}"
            else:
                # 低/中风险 → 记录警告但放行
                self._stats["allowed"] += 1
                logger.warning(
                    f"[ToolGuard] ⚠ WARNING: tool={tool_name}, "
                    f"risk={risk_level}, cmd='{command[:80]}'"
                )
                return True, f"警告: {risk}"
        
        self._stats["allowed"] += 1
        return True, ""
    
    def get_stats(self) -> Dict[str, int]:
        return dict(self._stats)
```

## 六、为什么不直接用现有方案？

### 6.1 与v1.0方案（2026-05-20）的差异

| 方面 | v1.0方案 | v3.0本方案 | 差异说明 |
|------|---------|-----------|---------|
| 矫正策略 | 弱矫正（主要靠LLM） | 强矫正（字典+安全关键词优先+LLM） | 安全关键词矫正优先确保安全检测不遗漏 |
| chat意图 | heuristic关键词匹配 | heuristic+危险词否定+纯标点检测 | 更安全：含危险词的"聊天"不走chat |
| 安全决策 | 静态（ALLOW/CONFIRM/BLOCK） | 意图感知动态调制（sensitivity系数） | shell意图下中风险升级为BLOCK |
| Agent执行层安全 | 有（简要设计） | 有（详细实现+复用check_command_safety） | 复用现有检测逻辑，保持代码一致性 |
| 数据流 | 矫正→意图→安全 | 矫正→意图→安全→**Agent分发+第二层安全** | 增加了完整的安全闭环 |
| LLM矫正回传 | 有（简要提及） | 有（明确写入PipelineContext + 日志记录） | 更完整的数据流处理 |

### 6.2 为什么不是增量的微调？

三个功能的问题是**系统性**的（管线断裂），不是独立的。如果逐个微调：
- 只修复chat意图 → 矫正问题仍存在，安全检测仍虚设
- 只修复安全检测 → 矫正问题仍存在，chat意图仍缺失
- 只修复矫正 → 矫正结果仍不被后续使用

需要一次**系统级重构**将三者串联成完整管线，一次解决所有断裂点。

## 八、总结

| 维度 | 结论 |
|------|------|
| **三个功能是否有价值** | ✅ 全部有价值——文本矫正、意图检测、安全检测都是核心能力 |
| **为什么没起作用** | 不是单个功能的设计问题，而是**管线断裂**：数据不流动、安全不执行、矫正不共享 |
| **保留还是删除** | 保留功能、删除空壳（PreprocessingPipeline类）、重构架构 |
| **最佳方案** | 数据驱动的完整管线（矫正→意图→安全）+ 双层安全防御 |
| **核心改变** | 从"三个独立步骤"到"一条完整管线"，从"纸老虎安全"到"真拦截" |

**一句话总结**：功能本身都对，但需要从"各自为战"变成"协同作战"。
