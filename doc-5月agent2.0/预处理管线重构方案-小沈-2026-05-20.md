## 一、三个功能的现状诊断

### 1.1 功能一：PreprocessingPipeline（预处理）

**设计意图**：对用户输入做文本矫正（错别字修正、编码修复、白名单过滤）+ 意图检测，为后续Agent提供干净的输入。

**实际代码**：

```python
# preprocessing/pipeline.py 第15-48行 — 全部有效逻辑
async def process(self, user_input, session_id=""):
    corrected = user_input.strip()      # 仅做strip
    errors = []                         # 空列表
    return {
        "original": user_input,
        "corrected": corrected,          # = user_input.strip()
        "errors": errors,               # 永远为[]
        "intent": "unknown",            # 永远为"unknown"
        "confidence": 0.0,             # 永远为0.0
        "all_intents": {},             # 永远为{}
    }
```

**诊断结论**：**已名存实亡**。`corrected`仅是strip后的原文，`intent`永远是"unknown"。意图检测逻辑已在2026-04-30被完全移除，移交给了`route_with_fallback()`。PreprocessingPipeline现在的唯一作用是一次async调用 + 一次日志打印。

**问题根因**：不是设计问题，是**实现被掏空后未清理**。原始设计（LLM矫正+意图检测）有价值，但被逐步剥离后只剩空壳。

**是否有价值**：**有价值，但当前实现完全无效**。文本矫正和安全增强是真实需求——用户输入"删处所有文件"（错别字"处"应为"除"），安全检测无法识别为危险操作。

---

### 1.2 功能二：意图检测（route_with_fallback，CRSS+LLM两阶段）

**设计意图**：快速准确地将用户输入路由到正确的Agent，避免所有请求走同一个Agent导致工具集过大。

**实际代码运行效果**：

| 用户输入 | CRSS结果 | LLM兜底 | 最终路由 | 是否正确 |
|----------|---------|---------|---------|---------|
| "帮我截图保存到桌面" | DESKTOP(0.98) | 不调 | desktop | 正确 |
| "下载一个文件" | NETWORK(0.85) | 不调 | network | 正确 |
| "今天星期几" | META(0.92) | 不调 | meta | 正确 |
| "你好" | None(0.0) | 调LLM | network(硬编码fallback) | **错误** |
| "讲个笑话" | None(0.0) | 调LLM | network(硬编码fallback) | **错误** |
| "帮我写一个删除所有文件的脚本" | SHELL(0.6) | 不调 | shell | **风险：应先安全审查** |
| "删处所有文件"（错别字） | None(0.0) | 调LLM→shell | shell | **矫正后才能安全检测** |

**诊断结论**：**有价值且大部分场景有效，但有三类缺陷**：

1. **无匹配意图fallback到network不合理**：纯对话走network会触发网络搜索
2. **CRSS在错别字输入上失效**：关键词匹配无法识别"删处"="删除"
3. **意图检测与安全检测顺序错误**：安全检测在意图检测之后，且在未矫正的原文上执行

**问题根因**：**设计缺陷 + 实现不完整**。两阶段路由框架设计合理，但：
- CRSS阶段缺少否定检测的深度处理
- 缺少"chat/通用对话"意图分类
- 矫正结果未反馈给安全检测
- LLM阶段2的矫正结果未替换原始输入

---

### 1.3 功能三：安全检测（check_command_safety）

**设计意图**：拦截危险命令（rm -rf /、format c:等），防止Agent执行破坏性操作。

**实际代码运行效果**：

```
用户输入 "rm -rf /"
  ├─ chat_router步骤4: check_command_safety() → blocked=True → 仅日志，继续执行 ← 虚设！
  └─ react_sse_wrapper:    check_command_safety() → is_safe=False → 发error并终止 ← 真正拦截
```

**诊断结论**：**有价值，第一道防线设计有严重缺陷**：

1. **chat_router步骤4的blocked=True不中断流程**：注释说"如果被阻止，记录警告但继续执行"，这是一道虚设的防线
2. **真正的拦截依赖react_sse_wrapper的冗余检查**：如果有人修改wrapper移除检查，安全将完全失效
3. **安全检测在未矫正的原文上执行**：用户输入"删处所有文件"→安全检测不识别→Agent执行后由LLM生成"删除所有文件"→危险操作漏检
4. **仅检测用户输入层，不检测Agent执行的工具调用层**：用户说"写一个脚本"→安全检测通过→Agent执行`execute_shell_command("rm -rf /")`→工具调用层无拦截
5. **无法检测间接指令**：用户用自然语言描述危险操作意图（如"帮我清空整个磁盘"），安全检测不识别

**问题根因**：**设计缺陷**。安全检测放在了错误的位置（用户输入层而非工具执行层），且两道防线逻辑不一致。

---

### 1.4 三个功能的根本问题：管线断裂

当前三个功能是**顺序排列但互不关联**的独立步骤：

```
步骤1: 预处理(仅strip) → corrected_text（丢弃，后续不用）
步骤2: 意图检测(在原文上) → intent（与步骤1无关）
步骤3: 初始化
步骤4: 安全检测(在原文上) → blocked（仅日志，不中断）
步骤5: start步骤
步骤6: Agent执行(在原文上)
```

**断裂点**：
- 步骤1的`corrected`未被步骤2/4/6使用
- 步骤2的意图信息未影响步骤4的检测策略
- 步骤4的`blocked`未中断步骤6
- 步骤4仅检测用户输入，不检测Agent的工具调用

---

## 二、完美方案设计

### 2.1 设计理念

**核心理念：管线化而非步骤化**

当前的6步骤是松散排列的独立函数调用，数据不流转。完美方案将这三个功能重构为一条**数据驱动的管线**，每一步的输出自动成为下一步的输入：

```
用户输入 → [矫正] → [意图检测] → [安全策略] → [Agent分发] → [工具执行安全]
              ↓           ↓            ↓             ↓              ↓
          corrected    intent+conf   safety_policy   agent          tool_guard
```

### 2.2 RequestPipeline 管线架构

```python
# backend/app/services/pipeline/request_pipeline.py

from dataclasses import dataclass, field
from typing import Optional, Dict, List, Any
from enum import Enum

from app.services.tools.registry import ToolCategory


class SafetyDecision(Enum):
    ALLOW = "allow"               # 安全，直接放行
    CONFIRM = "confirm"           # 中风险，需用户确认
    BLOCK = "block"               # 高风险，直接阻止
    DEFER_TO_AGENT = "defer"      # 低风险但需Agent执行时再检查


@dataclass
class PipelineContext:
    """
    管线上下文 — 每一步的输出自动追加到此上下文，供后续步骤使用
    
    这是整个管线的核心数据结构，所有步骤读写此上下文。
    """
    # 原始输入
    raw_input: str
    
    # 步骤1: 矫正
    corrected_input: str = ""
    corrections: List[Dict[str, Any]] = field(default_factory=list)
    is_corrected: bool = False
    
    # 步骤2: 意图检测
    intent: Optional[ToolCategory] = None
    confidence: float = 0.0
    candidates: List[ToolCategory] = field(default_factory=list)
    all_intents: Dict[str, float] = field(default_factory=dict)
    intent_source: str = ""  # "crss" / "llm" / "default"
    
    # 步骤3: 安全策略
    safety_decision: SafetyDecision = SafetyDecision.ALLOW
    risk_level: str = "none"
    risk_score: float = 0.0
    risk_detail: Dict[str, Any] = field(default_factory=dict)
    blocked_reason: str = ""
    confirm_message: str = ""
    
    # 步骤4: Agent信息
    agent_intent_type: str = ""
    agent_category: Optional[ToolCategory] = None
    
    # 元信息
    session_id: str = ""
    task_id: str = ""
    pipeline_version: str = "2.0"
    
    @property
    def effective_input(self) -> str:
        """有效输入 — 优先使用矫正后的文本，回退到原始文本"""
        return self.corrected_input if self.is_corrected else self.raw_input


class RequestPipeline:
    """
    请求处理管线 — 统一调度矫正→意图→安全→分发
    
    设计原则：
    1. 数据驱动：每一步读写PipelineContext，数据自动流转
    2. 短路执行：安全拦截时立即返回，不进入Agent
    3. 矫正优先：矫正结果供意图检测和安全检测共用
    4. 双层安全：用户输入层 + 工具执行层
    """
    
    def __init__(self):
        self._corrector = TextCorrectorV2()
        self._intent_detector = IntentDetectorV2()
        self._safety_analyzer = SafetyAnalyzerV2()
    
    async def process(self, user_input: str, session_id: str = "") -> PipelineContext:
        """
        执行完整管线
        
        Returns:
            PipelineContext: 管线上下文（包含所有步骤的输出）
        """
        ctx = PipelineContext(
            raw_input=user_input,
            session_id=session_id,
        )
        
        # 步骤1: 文本矫正（在意图检测和安全检测之前！）
        ctx = await self._corrector.correct(ctx)
        
        # 步骤2: 意图检测（在矫正后的文本上执行）
        ctx = await self._intent_detector.detect(ctx)
        
        # 步骤3: 安全策略分析（结合矫正后文本 + 意图信息）
        ctx = await self._safety_analyzer.analyze(ctx)
        
        # 步骤4: 确定Agent分类（结合意图+安全决策）
        ctx = self._resolve_agent(ctx)
        
        return ctx
    
    def _resolve_agent(self, ctx: PipelineContext) -> PipelineContext:
        """确定Agent分类"""
        if ctx.safety_decision == SafetyDecision.BLOCK:
            # 安全拦截时，不需要确定Agent
            ctx.agent_intent_type = "blocked"
            return ctx
        
        if ctx.intent is not None:
            ctx.agent_intent_type = ctx.intent.value
            ctx.agent_category = ctx.intent
        else:
            # 未匹配意图 → chat通用对话（而非network）
            ctx.agent_intent_type = "chat"
            ctx.agent_category = ToolCategory.META
            ctx.intent_source = "default"
        
        return ctx
```

### 2.3 步骤1：文本矫正（TextCorrectorV2）

#### 2.3.1 设计思路

当前矫正功能有三层但全部无效：
- L1（pycorrector）：未使用
- L2（PreprocessingPipeline）：仅strip
- L3（LLM intent_classifier）：矫正结果未替换原始输入

**重构方案**：将矫正前置为管线第一步，矫正结果供所有后续步骤使用。矫正策略分两级：

| 级别 | 方法 | 延迟 | 适用场景 |
|------|------|------|---------|
| L1: 规则矫正 | 正则+字典 | <1ms | 常见错别字、全角半角转换、多余空格 |
| L2: LLM矫正 | LLM调用 | ~1s | CRSS无法匹配时（模糊输入、口语化表达） |

#### 2.3.2 代码设计

```python
# backend/app/services/pipeline/corrector.py

import re
from typing import List, Dict, Any
from app.services.pipeline.request_pipeline import PipelineContext
from app.utils.logger import logger


# 常见中文错别字映射（高频+安全相关优先）
_TYPO_MAP = {
    "删处": "删除", "册除": "删除", "山除": "删除",   # 删除-安全关键词
    "格式花": "格式化", "格试化": "格式化",           # 格式化-安全关键词
    "关毕": "关闭", "官闭": "关闭",                   # 关闭-安全关键词
    "重起": "重启", "从起": "重启",                   # 重启-安全关键词
    "文建": "文件", "文件夹夹": "文件夹",             # 文件
    "命今": "命令", "命令": "命令",                   # 命令
    "网落": "网络", "网路": "网络",                   # 网络
    "系通": "系统", "细统": "系统",                   # 系统
    "时间日期": "时间日期",                           # 时间
    "桌目": "桌面", "卓面": "桌面",                   # 桌面
    "截土": "截图", "结图": "截图",                   # 截图
}

# 全角→半角转换表（数字+字母+符号）
_FULLWIDTH_MAP = str.maketrans(
    '０１２３４５６７８９ＡＢＣＤＥＦＧＨＯＰＱＲＳＴＵＶＷＸＹＺａｂｃｄｅｆｇｈｉｋｌｍｎｏｐｑｒｓｔｕｖｗｘｙｚ／＠．－＿',
    '0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZabcdefghijklmnopqrstuvwxyz/@.-_'
)


class TextCorrectorV2:
    """
    文本矫正器 V2 — 管线第一步
    
    策略：
    1. 基本清理：strip + 全角半角转换 + 多余空格压缩
    2. 错别字矫正：基于字典的常见错别字替换
    3. LLM矫正：仅在CRSS无法匹配时由IntentDetector触发
    
    注意：LLM矫正不再由本类执行，而是由IntentDetectorV2在CRSS失败时
    调用LLM意图分类器（LLM同时返回corrected+intent）。
    """
    
    async def correct(self, ctx: PipelineContext) -> PipelineContext:
        """执行规则级矫正（L1）"""
        text = ctx.raw_input
        
        # 1. 基本清理
        corrected = text.strip()
        
        # 2. 全角→半角
        corrected = corrected.translate(_FULLWIDTH_MAP)
        
        # 3. 多余空格压缩（中文间空格、连续空格）
        corrected = re.sub(r' +', ' ', corrected)
        corrected = re.sub(r'([\u4e00-\u9fff])\s+([\u4e00-\u9fff])', r'\1\2', corrected)
        
        # 4. 错别字矫正
        corrections = []
        for typo, fixed in _TYPO_MAP.items():
            if typo in corrected:
                corrected = corrected.replace(typo, fixed)
                corrections.append({"from": typo, "to": fixed})
        
        # 5. 更新上下文
        ctx.corrected_input = corrected
        ctx.is_corrected = corrected != text
        ctx.corrections = corrections
        
        if ctx.is_corrected:
            logger.info(
                f"[Corrector] 矫正: '{text}' → '{corrected}', "
                f"corrections={corrections}"
            )
        
        return ctx
```

**关键设计决策**：
- 规则级矫正（L1）在管线第一步执行，**所有后续步骤使用`ctx.effective_input`**
- LLM级矫正（L2）由IntentDetectorV2在CRSS失败时自然触发（LLM同时返回corrected+intent），无需单独调用
- 安全关键词错别字（"删处"→"删除"）优先级最高，确保安全检测在矫正后的文本上执行

### 2.4 步骤2：意图检测（IntentDetectorV2）

#### 2.4.1 设计思路

当前两阶段路由（CRSS+LLM）框架设计合理，但有以下缺陷：
1. CRSS在矫正前的文本上执行
2. 无"chat/通用对话"意图
3. LLM矫正结果未替换原始输入
4. candidates不触发跨分类工具预加载

**重构方案**：CRSS和LLM均在矫正后的文本上执行，新增chat意图，LLM矫正结果更新到PipelineContext。

#### 2.4.2 代码设计

```python
# backend/app/services/pipeline/intent_detector.py

from typing import Optional
from app.services.pipeline.request_pipeline import PipelineContext, SafetyDecision
from app.services.intents.crss_scorer import detect_intent_v2, CRSS_CONFIDENCE_THRESHOLD
from app.services.tools.registry import ToolCategory, resolve_category
from app.utils.logger import logger


# CRSS置信度阈值下的chat意图判定关键词
_CHAT_PATTERNS = [
    "你好", "谢谢", "再见", "请问", "讲个", "什么是", "为什么",
    "怎么样", "如何理解", "解释一下", "说说", "聊聊",
    "hello", "hi", "thanks", "bye",
]


class IntentDetectorV2:
    """
    意图检测器 V2 — 管线第二步
    
    改进点：
    1. 在矫正后的文本(effective_input)上执行CRSS
    2. 新增chat意图：CRSS无匹配且无危险关键词时走chat
    3. LLM矫正结果更新到PipelineContext
    4. candidates置信度分布保留在all_intents中
    """
    
    async def detect(self, ctx: PipelineContext) -> PipelineContext:
        """执行意图检测"""
        text = ctx.effective_input  # 使用矫正后的文本！
        
        # ===== 阶段1: CRSS快速匹配 =====
        primary, candidates, confidence = detect_intent_v2(text)
        
        if primary is not None and confidence >= CRSS_CONFIDENCE_THRESHOLD:
            ctx.intent = primary
            ctx.confidence = confidence
            ctx.candidates = candidates
            ctx.intent_source = "crss"
            logger.info(
                f"[IntentDetector] CRSS → intent={primary.value}, "
                f"conf={confidence:.4f}, candidates={[c.value for c in candidates]}"
            )
            return ctx
        
        # ===== CRSS低置信度：判断是否为chat意图 =====
        if self._is_likely_chat(text):
            ctx.intent = ToolCategory.META  # chat意图使用META分类（轻量工具）
            ctx.confidence = 0.5
            ctx.candidates = [ToolCategory.META]
            ctx.intent_source = "chat_heuristic"
            ctx.agent_intent_type = "chat"
            logger.info(f"[IntentDetector] chat启发式 → 走chat意图")
            return ctx
        
        # ===== 阶段2: LLM语义分类（兜底）=====
        try:
            from app.services.preprocessing.intent_classifier import classify_intent
            
            intent_labels = [c.value for c in ToolCategory]
            llm_result = await classify_intent(text, intent_labels)
            
            intent_str = llm_result.get("intent", "")
            llm_confidence = float(llm_result.get("confidence", 0.5))
            intent_enum = resolve_category(intent_str)
            
            # ★ LLM矫正结果更新到上下文 ★
            llm_corrected = llm_result.get("corrected", "")
            if llm_corrected and llm_corrected != text:
                ctx.corrected_input = llm_corrected
                ctx.is_corrected = True
                ctx.corrections.append({
                    "from": text, "to": llm_corrected, "source": "llm"
                })
                logger.info(f"[IntentDetector] LLM矫正: '{text}' → '{llm_corrected}'")
            
            ctx.intent = intent_enum
            ctx.confidence = llm_confidence
            ctx.all_intents = llm_result.get("all_intents", {})
            ctx.intent_source = "llm"
            
            if intent_enum:
                ctx.candidates = [intent_enum]
            
            logger.info(
                f"[IntentDetector] LLM → intent={intent_str}({intent_enum}), "
                f"conf={llm_confidence}, source=llm"
            )
        except Exception as e:
            logger.warning(f"[IntentDetector] LLM兜底失败: {e}")
            # LLM失败：保持CRSS结果（如有），否则走chat
            if ctx.intent is None:
                ctx.intent = ToolCategory.META
                ctx.confidence = 0.3
                ctx.candidates = [ToolCategory.META]
                ctx.intent_source = "default_chat"
                ctx.agent_intent_type = "chat"
        
        return ctx
    
    def _is_likely_chat(self, text: str) -> bool:
        """
        启发式判断是否为纯对话/闲聊
        
        条件：
        1. 文本中包含chat关键词
        2. 文本较短（<20字）
        3. 不包含危险操作关键词
        """
        if len(text) > 50:
            return False  # 长文本不太可能是纯闲聊
        
        text_lower = text.lower()
        for pattern in _CHAT_PATTERNS:
            if pattern in text_lower:
                # 二次检查：不包含危险操作关键词
                from app.services.command_security import DANGEROUS_COMMANDS
                for dangerous in DANGEROUS_COMMANDS:
                    if dangerous.lower() in text_lower:
                        return False  # 虽然像闲聊但包含危险词，不走chat
                return True
        
        return False
```

**关键改进**：
1. **所有检测在`ctx.effective_input`（矫正后文本）上执行**
2. **新增chat启发式**：CRSS无匹配时先判断是否为闲聊，再调LLM兜底
3. **LLM矫正结果更新到PipelineContext**：后续安全检测在矫正后文本上执行
4. **未匹配意图fallback到chat而非network**：纯对话不会触发网络搜索

### 2.5 步骤3：安全策略分析（SafetyAnalyzerV2）

#### 2.5.1 设计思路

当前安全检测的三大缺陷：
1. 第一道防线（chat_router）不中断流程
2. 安全检测在未矫正的原文上执行
3. 仅检测用户输入层，不检测工具执行层

**重构方案**：安全检测重构为**安全策略分析**，输出决策而非简单的blocked标志。

#### 2.5.2 代码设计

```python
# backend/app/services/pipeline/safety_analyzer.py

from app.services.pipeline.request_pipeline import PipelineContext, SafetyDecision
from app.services.command_security import check_command_safety, calculate_risk_score_v2
from app.services.tools.registry import ToolCategory
from app.utils.logger import logger


class SafetyAnalyzerV2:
    """
    安全策略分析器 V2 — 管线第三步
    
    改进点：
    1. 在矫正后的文本(effective_input)上执行安全检测
    2. 结合意图信息做差异化安全策略
    3. 输出SafetyDecision决策（ALLOW/CONFIRM/BLOCK/DEFER）
    4. chat意图降低安全检测敏感度（闲聊无需严格检测）
    """
    
    async def analyze(self, ctx: PipelineContext) -> PipelineContext:
        """执行安全策略分析"""
        text = ctx.effective_input  # 使用矫正后的文本！
        
        # chat意图 → 跳过安全检测（闲聊不执行任何工具）
        if ctx.agent_intent_type == "chat":
            ctx.safety_decision = SafetyDecision.ALLOW
            ctx.risk_level = "none"
            ctx.risk_score = 0.0
            logger.info(f"[SafetyAnalyzer] chat意图 → 跳过安全检测")
            return ctx
        
        # 执行安全检测
        security_result = check_command_safety(text)
        
        is_safe = security_result.get('is_safe', True)
        risk_level = security_result.get('risk_level', 'none')
        blocked = security_result.get('blocked', False)
        is_need_confirm = security_result.get('is_need_confirm', False)
        
        # ★ 根据检测结果做决策（不再仅记录日志！）★
        if blocked:
            # 高风险/极高风险 → 直接阻止
            ctx.safety_decision = SafetyDecision.BLOCK
            ctx.risk_level = risk_level
            ctx.blocked_reason = security_result.get('risk', '检测到高风险操作')
            logger.warning(
                f"[SafetyAnalyzer] BLOCKED: risk_level={risk_level}, "
                f"reason={ctx.blocked_reason}, input='{text[:50]}'"
            )
        elif is_need_confirm:
            # 中风险 → 需用户确认
            ctx.safety_decision = SafetyDecision.CONFIRM
            ctx.risk_level = risk_level
            ctx.confirm_message = (
                f"检测到{risk_level}风险操作：{security_result.get('risk', '')}。"
                "是否继续执行？"
            )
            logger.info(
                f"[SafetyAnalyzer] CONFIRM: risk_level={risk_level}, "
                f"input='{text[:50]}'"
            )
        else:
            # 安全或低风险 → 放行，但Agent执行工具时仍需检查
            ctx.safety_decision = SafetyDecision.ALLOW
            ctx.risk_level = risk_level
            logger.info(f"[SafetyAnalyzer] ALLOW: risk_level={risk_level}")
        
        # 获取CRSS风险评分详情（供前端展示）
        try:
            risk_detail = calculate_risk_score_v2(text)
            ctx.risk_score = risk_detail.get('score', 0)
            ctx.risk_detail = risk_detail
        except Exception:
            pass
        
        return ctx
```

**关键改进**：
1. **chat意图跳过安全检测**：闲聊不会执行任何工具，无需检测
2. **在矫正后文本上执行安全检测**：`ctx.effective_input`已是矫正后的文本
3. **BLOCK决策直接阻止**：不再仅记录日志，管线返回后chat_router根据决策中断
4. **CONFIRM决策需用户确认**：中风险操作需用户确认后才继续

### 2.7 工具执行层安全守卫（第二层安全）

#### 2.7.1 设计思路

当前安全检测仅覆盖用户输入层。但最危险的场景是：用户用自然语言描述意图 → 安全检测通过 → Agent的LLM生成危险工具调用 → 无拦截执行。

**例如**：
- 用户："帮我清理一下系统" → 安全检测通过（无黑名单匹配）→ Agent执行 `execute_shell_command("rm -rf /tmp/*")` → 无拦截

**方案**：在 `ToolExecutor.execute()` 中增加工具调用层安全守卫。

#### 2.7.2 代码设计

```python
# backend/app/services/agent/tool_execution_guard.py

from typing import Dict, Any, Optional, Tuple
from app.services.command_security import check_command_safety, calculate_risk_score_v2
from app.utils.logger import logger


# 需要安全守卫的工具列表（仅Shell类和可能执行命令的工具）
_GUARDED_TOOLS = {
    "execute_shell_command",    # Shell命令执行
    "execute_python",           # Python代码执行
    "execute_javascript",       # JavaScript代码执行
    "shell_session",            # Shell会话
}

# 需要提取命令参数名的映射
_COMMAND_PARAM_NAMES = {
    "execute_shell_command": "command",
    "execute_python": "code",
    "execute_javascript": "code",
    "shell_session": "command",
}


class ToolExecutionGuard:
    """
    工具执行安全守卫 — 第二层安全
    
    在ToolExecutor.execute()中调用，拦截Agent LLM生成的危险工具调用。
    
    与用户输入层安全检测的区别：
    - 用户输入层：检测用户原始输入是否包含危险命令
    - 工具执行层：检测Agent LLM生成的工具调用参数是否包含危险命令
    
    两层配合：
    - 第一层拦截用户直接输入的危险命令（"rm -rf /"）
    - 第二层拦截Agent间接生成的危险命令（用户说"清理系统"→Agent生成"rm -rf"）
    """
    
    def __init__(self, enabled: bool = True):
        self._enabled = enabled
        self._blocked_count = 0
        self._allowed_count = 0
    
    def check(self, tool_name: str, tool_params: Dict[str, Any]) -> Tuple[bool, str]:
        """
        检查工具调用是否安全
        
        Args:
            tool_name: 工具名称
            tool_params: 工具参数
            
        Returns:
            (is_safe, reason): 是否安全 + 原因
        """
        if not self._enabled:
            return True, ""
        
        # 非守卫工具 → 直接放行
        if tool_name not in _GUARDED_TOOLS:
            self._allowed_count += 1
            return True, ""
        
        # 提取命令/代码参数
        param_name = _COMMAND_PARAM_NAMES.get(tool_name)
        if not param_name:
            return True, ""
        
        command = tool_params.get(param_name, "")
        if not command or not isinstance(command, str):
            return True, ""
        
        # 执行安全检测
        security_result = check_command_safety(command)
        
        if not security_result.get('is_safe', True):
            risk_level = security_result.get('risk_level', 'unknown')
            risk = security_result.get('risk', '未知风险')
            
            # 高风险 → 阻止
            if security_result.get('blocked', False):
                self._blocked_count += 1
                logger.warning(
                    f"[ToolGuard] BLOCKED: tool={tool_name}, "
                    f"risk_level={risk_level}, risk={risk}, "
                    f"command='{command[:100]}'"
                )
                return False, f"工具调用安全拦截: {risk} (风险等级: {risk_level})"
            
            # 中风险 → 记录警告但放行（Agent已做过安全评估）
            logger.warning(
                f"[ToolGuard] WARNING: tool={tool_name}, "
                f"risk_level={risk_level}, risk={risk}, "
                f"command='{command[:100]}'"
            )
        
        self._allowed_count += 1
        return True, ""
    
    @property
    def stats(self) -> Dict[str, int]:
        return {
            "blocked": self._blocked_count,
            "allowed": self._allowed_count,
        }
```

## 六、核心价值总结

### 6.1 预处理（PreprocessingPipeline）→ 文本矫正

| 维度 | 重构前 | 重构后 |
|------|--------|--------|
| 价值 | 无（仅strip） | **高**：矫正错别字使安全检测和意图检测更准确 |
| 执行时机 | 步骤1但结果被丢弃 | **管线第一步**，结果供所有后续步骤使用 |
| 矫正能力 | 无 | 规则级（<1ms）+ LLM级（CRSS失败时自然触发） |

### 6.2 意图检测 → IntentDetectorV2

| 维度 | 重构前 | 重构后 |
|------|--------|--------|
| 未匹配fallback | network（触发搜索） | **chat**（轻量对话，不搜索） |
| 矫正联动 | 无（CRSS在原文上） | **有**（CRSS在矫正后文本上） |
| LLM矫正结果 | 丢弃 | **更新到PipelineContext**，供安全检测使用 |
| chat意图 | 无 | **有**（启发式识别闲聊） |

### 6.3 安全检测 → SafetyAnalyzerV2 + ToolExecutionGuard

| 维度 | 重构前 | 重构后 |
|------|--------|--------|
| 第一道防线 | 虚设（仅日志） | **真正拦截**（BLOCK→直接阻止） |
| 安全层数 | 1层（用户输入层） | **2层**（用户输入层+工具执行层） |
| 矫正联动 | 无（在原文上检测） | **有**（在矫正后文本上检测） |
| 间接指令 | 无法检测 | **工具执行层可检测** |
| 确认机制 | 有接口但未实现 | **CONFIRM决策**需用户确认 |

### 6.4 管线化 vs 步骤化

| 维度 | 重构前 | 重构后 |
|------|--------|--------|
| 数据流转 | 断裂（各步骤独立） | **PipelineContext贯穿**（数据自动流转） |

| 矫正引入错误（如把文件名中"删处"矫正为"删除"） | 错别字字典仅包含**安全关键词**和**常见高频错别字**，文件名/路径不会被矫正（全角转半角除外） |
| chat启发式误判（把"删除文件"当闲聊） | `_is_likely_chat()` 二次检查DANGEROUS_COMMANDS，包含危险词的不走chat |
| ToolExecutionGuard增加执行延迟 | 仅对_GUARDED_TOOLS中的4个工具检查，其他工具零开销 |
| LLM矫正失败 | LLM矫正仅在CRSS失败时触发，失败后回退到规则级矫正结果 |
| 管线化改造影响现有流程 | 分阶段实施，Phase 1-2可与现有代码并存，Phase 3才切换入口 |
