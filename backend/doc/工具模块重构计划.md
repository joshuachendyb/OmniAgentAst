## 一、当前架构问题分析

### 1.1 当前结构

```
backend/app/
├── services/tools/
│   ├── registry.py                    # 有ToolRegistry类
│   ├── file/
│   │   ├── file_tools.py              # 17个工具实现
│   │   ├── file_register.py           # [NEW] 注册点
│   │   └── ...
│   ├── time/
│   │   ├── time_tools.py              # 9个工具（已迁移）
│   │   ├── time_register.py          # [NEW] 注册点
│   │   └── ...
│   └── (system/network/database/desktop) # 待创建
└── tools/
    └── (已删除time_tools.py)           # 已清理
```

### 1.2 问题

| 问题 | 说明 |
|------|------|
| **两套注册系统** | `file_tools._TOOL_REGISTRY` 和 `registry.tool_registry` 互不通信 |
| **注册点分散** | time_tools用registry.py的装饰器，file_tools用自己的装饰器 |
| **难以扩展** | 新增工具不知道该用哪个注册方式 |

### 1.3 目标

统一注册到 `registry.tool_registry`，各分类只负责实现和注册自己的工具。

## 三、registry.py 核心设计

### 3.1 主要功能

```python
# 全局单例
tool_registry = ToolRegistry()

# 注册工具（通过装饰器）
@register_tool(
    name="工具名",
    description="工具描述",
    category=ToolCategory.FILE,
    examples=[...]
)
def tool_function(params):
    """实现"""
    pass

# 获取所有工具
get_registered_tools()

# 按分类获取
tool_registry.list_tools(category=ToolCategory.FILE)
```

### 3.2 架构注释（将写入registry.py头部）

```python
"""
工具注册表模块 - 统一入口

【架构规范】2026-04-26 小沈

## 一、总体架构
- registry.py: 总注册表，统一入口，所有工具通过 @register_tool 装饰器注册到这里
- 各分类目录: {category}_register.py（注册点）+ {category}_tools.py（实现）

## 二、新增工具流程
1. 在对应分类的 {category}_tools.py 中编写实现函数
2. 在对应分类的 {category}_register.py 中使用 @register_tool 装饰器注册
3. 在分类的 __init__.py 中导入注册模块（触发注册）

## 三、目录组织规范
| 分类 | 目录 | 注册文件 | 实现文件 |
|------|------|---------|---------|
| 文件操作 | file/ | file_register.py | file_tools.py |
| 时间日期 | time/ | time_register.py | time_tools.py |
| Shell命令 | shell/ | shell_register.py | shell_tools.py |
| 网络通信 | network/ | network_register.py | network_tools.py |
| 环境变量 | env/ | env_register.py | env_tools.py |
| 系统信息 | system/ | system_register.py | system_tools.py |
| 数据库 | database/ | database_register.py | database_tools.py |
| 桌面功能 | desktop/ | desktop_register.py | desktop_tools.py |

## 四、命名规范
- 注册装饰器: @register_tool
- 注册函数: {分类}tool.register (如 filetool.register, timetool.register)
- 实现文件: {分类}_tools.py
- 注册文件: {分类}_register.py

## 五、必须遵守的规则
1. 工具必须注册到 registry.py 的 tool_registry
2. 各分类只负责实现，注册在 __init__.py 导入时自动触发
3. 文件名必须具体，不能用笼统名称如 tools.py
4. 空目录保留，用于未来扩展
"""
```

## 四、重构执行计划

### 4.1 阶段划分

| 阶段 | 内容 | 风险 |

### 4.2 Phase 1: 完善 registry.py

**目标**: 添加架构注释，完善装饰器

**操作**:
1. 在 registry.py 头部添加架构注释
2. 确保 register_tool 装饰器正��工作
3. 添加分类别名支持

### 4.3 Phase 2: 迁移 time_tools.py

**目标**: 将 app/tools/time_tools.py 迁移到 services/tools/time/

**操作**:
1. 创建 `services/tools/time/` 目录
2. 创建 `time_register.py`（注册点）
3. 移动 `time_tools.py` 到 `time/`
4. 移除 app/tools/ 目录（如果空了）

**风险**: time_tools.py 用的是 registry.py 的装饰器，需要确认导入路径正确

### 4.4 Phase 3: 创建 file_register.py

**目标**: 将 file_tools.py 的注册逻辑独立出来

**操作**:
1. 创建 `file_register.py`
2. 导入 file_tools.py 中的工具函数
3. 使用 @register_tool 重新注册
4. 移除 file_tools.py 中的旧注册逻辑

### 4.5 Phase 4: 更新 __init__.py

**目标**: 确保导入链正确

**操作**:
1. 更新 services/tools/__init__.py
2. 更新各分类的 __init__.py
3. 移除 app/tools/ 的导入（如果存在）

### 4.6 Phase 5: 验证测试

**目标**: 确保功能不丢失

**操作**:
1. 运行单元测试
2. 测试工具注册
