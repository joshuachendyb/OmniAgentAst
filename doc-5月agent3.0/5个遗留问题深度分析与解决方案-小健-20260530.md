### 1.2 核心问题：_StreamRetryContext

**当前代码**（`core.py:74-93`）：

```python
async def __aenter__(self):
    self.service._ensure_client()
    for attempt in range(self.max_retries):                    # L77: 手写重试循环
        self._response_ctx = self.service._llm_sdk.stream(     # L78: 创建流式请求
            "POST", self.url, headers=self.headers, json=self.json_body
        )
        response = await self._response_ctx.__aenter__()       # L81: 进入上下文
        if self.service._is_rate_limit_status(response.status_code):  # L82: 429判断
            await self._response_ctx.__aexit__(None, None, None)     # L83: 关闭连接
            if attempt < self.max_retries - 1:
                delay = self._engine._calculate_delay(attempt + 1)   # L85: 调用私有方法!
                logger.warning(...)
                await asyncio.sleep(delay)
                continue
            else:
                logger.error(...)
                return response
        return response
    return response
```

**3个违规点**：
1. **封装违反**：第85行调用`self._engine._calculate_delay()`（私有方法，`_`前缀）
2. **DRY违反**：手写重试循环，与RetryEngine的execute逻辑重复
3. **未用已设计接口**：RetryEngine.execute_context()正是为此场景设计但未使用

### 1.3 RetryEngine.execute_context()为何不能直接用

当前`execute_context()`（`retry_engine.py:125-160`）有4个结构性缺陷，无法直接替换：

| 差异点 | _StreamRetryContext(当前) | execute_context()(现有) | 阻碍程度 |
|--------|--------------------------|------------------------|----------|
| 重试触发 | 基于response.status_code判断(非异常) | 基于Exception捕获 | **致命** |
| 上下文管理 | 手动__aenter__/__aexit__ | 无上下文管理 | **致命** |
| 工厂函数 | 返回异步上下文管理器 | 同步工厂直接return | **致命** |
| 失败时行为 | 返回最后一个429 response(不抛异常) | 抛出最后一次异常 | **中等** |

### 1.4 RetryCounter分析

**文件**: `app/utils/retry_counter.py`

| 维度 | RetryCounter | RetryEngine |
|------|-------------|-------------|
| 职责 | 纯计数器(can_retry/increment/reset) | 完整重试引擎(退避策略+执行循环) |
| 退避计算 | 无 | 有_calculate_delay |
| 执行循环 | 无 | 有execute/execute_context |
| 回调机制 | 无 | 有on_retry |

RetryCounter是RetryEngine功能的**严格子集**，但它的使用场景（`chat_stream_query.py`）需要在外部循环中yield SSE事件，不适合用RetryEngine.execute()包裹。这是**合理的抽象层次差异**，不是DRY违反。

**使用者**: 仅`app/chat_stream/chat_stream_query.py`一处（第15/31/160/205/306行）

### 1.5 解决方案：新增execute_async_context()

**在`RetryEngine`中新增方法**，支持异步上下文管理器+基于返回值判断重试：

```python
# app/utils/retry_engine.py 新增方法

async def execute_async_context(
    self,
    ctx_factory: Callable,
    result_check: Callable[[Any], bool],
) -> Any:
    """异步上下文管理器模式执行（用于429流式重试场景）

    Args:
        ctx_factory: 创建异步上下文管理器的工厂函数
                     调用后返回ctx，需有__aenter__/__aexit__
        result_check: 结果检查函数，返回True表示需要重试
                     如: lambda response: response.status_code == 429

    Returns:
        第一个不需要重试的结果，或重试耗尽后的最后一个结果
    """
    self._attempt_count = 0
    last_result = None
    ctx = None

    while self._attempt_count <= self._max_retries:
        try:
            ctx = ctx_factory()
            result = await ctx.__aenter__()
            if result_check(result):
                # 需要重试：关闭当前连接
                await ctx.__aexit__(None, None, None)
                ctx = None
                last_result = result
                self._attempt_count += 1
                if self._attempt_count >= self._max_retries:
                    # 重试耗尽，返回最后结果（不抛异常）
                    return result
                delay = self._calculate_delay(self._attempt_count)
                if self._on_retry:
                    self._on_retry(self._attempt_count, None)
                await asyncio.sleep(delay)
                continue
            # 不需要重试：直接返回
            return result
        except Exception as e:
            # 异常路径：确保上下文关闭
            if ctx:
                await ctx.__aexit__(type(e), e, e.__traceback__)
                ctx = None
            self._attempt_count += 1
            if not self._is_retryable(e):
                raise
            if self._attempt_count >= self._max_retries:
                raise
            delay = self._calculate_delay(self._attempt_count)
            if self._on_retry:
                self._on_retry(self._attempt_count, e)
            await asyncio.sleep(delay)

    return last_result
```

**然后`_StreamRetryContext.__aenter__`简化为**：

```python
async def __aenter__(self):
    self.service._ensure_client()
    response = await self._engine.execute_async_context(
        ctx_factory=lambda: self.service._llm_sdk.stream(
            "POST", self.url, headers=self.headers, json=self.json_body
        ),
        result_check=lambda r: self.service._is_rate_limit_status(r.status_code),
    )
    self._response_ctx = response  # 供__aexit__使用
    return response
```

### 1.7 风险评估

| 风险项 | 级别 | 说明 |
|--------|------|------|
| 改动RetryEngine公共接口 | **低** | 新增方法，不改已有execute()/execute_context() |
| 异步上下文管理器生命周期 | **中** | __aexit__必须正确调用，否则httpx连接泄漏。但execute_async_context已在try/except/正常3条路径都处理了__aexit__ |
| 重试语义变更 | **低** | 当前range(max_retries)是0..2(3次请求)，execute_async_context用while<=max_retries也是0..2(3次请求)，语义一致 |
| on_retry回调 | **低** | 新增的异步上下文路径也触发on_retry，与现有行为一致 |

### 1.8 RetryCounter结论

**保留RetryCounter，不删除**。理由：
1. chat_stream_query.py需要在外部循环yield SSE事件，RetryEngine.execute()无法包裹这种场景
2. RetryCounter是RetryEngine的简化版（纯计数器），适用于"需要外部循环控制"的场景
3. 两者是合理抽象层次差异，不是DRY违反
4. 建议在RetryCounter类docstring中标注："RetryEngine的简化版，适用于需要外部循环控制的重试场景（如流式SSE）"

### 3.2 tool_executor.py为何是薄壳

`ToolExecutor.execute()`的核心逻辑：

```python
async def execute(self, action: str, action_input: dict) -> dict:
    # 1. finish短路（5行）
    if action == "finish":
        return action_input.get("result", "")

    # 2. not-found判断（8行）
    if action not in self.available_tools:
        return build_error(error_code="ERR_TOOL_NOT_FOUND", ...)

    # 3. 委托（1行）—— 核心逻辑全在tool_retry_engine.py
    return await execute_tool_with_unified_retry(action, action_input, self.available_tools)
```

112行文件中，真正有价值的逻辑只有上面3步（14行），其余都是类定义/初始化/import。`finish短路`和`not-found判断`完全可以下沉到`execute_tool_with_unified_retry()`开头。

### 3.3 解决方案：薄壳下沉，删除中间层

**Step 1**: `execute_tool_with_unified_retry()`开头增加finish和not-found判断

```python
# app/services/agent/tool_retry_engine.py

async def execute_tool_with_unified_retry(
    action: str, action_input: dict, available_tools: dict, **kwargs
) -> dict:
    # 下沉自tool_executor.py的2个前置判断
    if action == "finish":
        return action_input.get("result", "")
    if action not in available_tools:
        return build_error(error_code="ERR_TOOL_NOT_FOUND", ...)

    # 原有逻辑不变
    ...
```

**Step 2**: `tool_init_mixin.py`不再创建ToolExecutor实例

```python
# 改前
from app.services.agent.tool_executor import ToolExecutor
self.executor = ToolExecutor(self._tools_dict)

# 改后
from app.services.agent.tool_retry_engine import execute_tool_with_unified_retry
self._execute_tool_func = lambda action, params: execute_tool_with_unified_retry(
    action, params, self._tools_dict

```

**Step 3**: 删除`tool_executor.py`

**Step 4**: 更新`app/services/agent/__init__.py`中的重导出
