## 二、逐层分析

### 2.1 app/db/config.py — 工厂函数

**功能**: `create_connection(db_path)` 统一创建DB连接，设置WAL + busy_timeout。  
**被使用**: chat_db.py、operations_db.py 内部调用。

### 2.2 app/db/chat_db.py — chat_history.db 连接

**导出**:
- `get_connection()` → 调用 `config.create_connection(CHAT_DB_PATH)`，裸连接
- `init_database()` → 建表

### 2.3 app/db/operations_db.py — operations.db 连接

**导出**:
- `get_connection()` → 调用 `config.create_connection(OPERATIONS_DB_PATH)`，裸连接
- `init_database()` → 建表

### 2.4 app/db/__init__.py — 统一导出

**导出**:
- `create_connection`（工厂函数）
- `get_chat_connection`（chat_db.get_connection 别名）
- `get_operations_connection`（operations_db.get_connection 别名）
- `init_chat_database` / `init_operations_database`

### 2.5 app/api/v1/db_helpers.py — API层共享工具

**位置**: `api/v1/db_helpers.py`（注意：在 api/v1 下，不是在 db/ 下）  
**功能**: 封装 API 层共用的 DB 工具函数

| 函数 | 用途 | 来源 |
|------|------|------|
| `get_db_connection()` | 上下文管理器，包装 chat_db.get_connection + 自动关闭 | 原创 |
| `check_db_fields_exist(conn)` | 检查 chat_sessions 表字段 | 原创 |
| `_convert_to_utc(time_value)` | 任意时间→UTC ISO 字符串 | 原创 |
| `_get_utc_timestamp()` | 获取当前 UTC ISO 时间 | 原创 |
| `_ensure_ts_milliseconds(ts_value)` | 统一转毫秒整数 | 原创 |
| `_get_timestamp_ms()` | 获取当前毫秒时间戳 | 原创 |
| `_safe_parse_json(json_str)` | 安全解析 JSON | 原创 |

**依赖链**: `db_helpers → app.db.chat_db.get_connection`（仅依赖 chat_db）

### 2.6 使用方

| 文件 | 从 db_helpers 导入 | 额外 db 相关 |
|------|-------------------|-------------|
| `sessions.py` | `get_db_connection`, `check_db_fields_exist` | 也有 `from app.db.chat_db import get_connection`（直接裸连接） |
| `messages.py` | `check_db_fields_exist` | `from app.db.chat_db import get_connection as _get_connection`（直接裸连接） |
| `conversation.py` | `get_db_connection`, `check_db_fields_exist`, `_convert_to_utc` | ⚠️ 自己写了 `_get_timestamp_ms()` |

## 三、发现的问题

### 问题1（DRY违反）: conversation.py 重复定义了 _get_timestamp_ms

**文件**: `app/api/v1/conversation.py:38-40`

```python
def _get_timestamp_ms() -> int:
    """获取毫秒时间戳，避免时间戳存储为ISO字符串导致前端解析错误"""
    return int(datetime.now(timezone.utc).timestamp() * 1000)
```

与 `app/api/v1/db_helpers.py:153-155` 完全相同：

```python
def _get_timestamp_ms() -> int:
    """获取毫秒时间戳，避免时间戳存储为ISO字符串导致前端解析错误"""
    return int(datetime.now(timezone.utc).timestamp() * 1000)
```

conversation.py 已经导入了 `_convert_to_utc`，但没有导入 `_get_timestamp_ms`，而是自己又写了一遍。按理应该统一从 db_helpers 导入。

### 问题2（SLAP违反）: sessions.py 和 messages.py 同时使用两种连接方式

sessions.py 和 messages.py 都：
1. 从 `db_helpers` 导入 `get_db_connection()`（上下文管理器包装）
2. 也从 `app.db.chat_db` 直接导入 `get_connection`（裸连接）

同文件内存在两种不同的 DB 连接获取方式，违反了 SLAP 和"一个事情一个出入口"。

### 问题3: db_helpers 命名位置歧义

`db_helpers.py` 放在 `api/v1/` 下，但文件名暗示它属于 `db/` 层。实际它只是 v1 API 的内部工具模块，并非通用 DB 层。建议改名更清晰（如 `api/v1/_db_utils.py` 或 `api/v1/_api_db.py`）。

## 五、HTTP 客户端管理与 LLM 调用连接分析

**分析时间**: 2026-05-28 16:11

### 5.1 HTTP 客户端创建 — 与 DB 同样的问题模式

#### 5.1.1 辅助函数定义了没人用（和 _get_timestamp_ms 一模一样的模式）

`network_helper.py:200` 定义了 `_create_http_client()` 统一辅助函数（带连接池 max=100、代理、SSL），但 **network_tools.py 里 6 个网络工具函数没有一个用它**，全部自己内联写 `async with httpx.AsyncClient(...)`：

| 函数 | 文件行号 | 超时 | 连接池 |
|------|---------|------|--------|
| `http_request` | network_tools.py:205 | 内联 | 无 |
| `download_file` | network_tools.py:335 | 内联 | 无 |
| `fetch_webpage` | network_tools.py:469 | 内联 | 无 |
| `_search_mcp_engine` | network_tools.py:609 | 内联 | 无 |
| `_search_bing` | network_tools.py:738 | 内联 | 无 |

**模式完全一样**：定义了一个统一函数 → 调用方不导入 → 各自内联实现 → DRY 违反。

#### 5.1.2 多个无关的 HTTP 客户端各自独立创建

| 位置 | 文件行号 | 连接池 | 复用 |
|------|---------|--------|------|
| `BaseAIService.__init__` | llm_core.py:95 | 是（max=10） | 跨 LLM 调用共享 |
| `intent_classifier.classify_intent` | intent_classifier.py:151 | 否（每个请求新建） | 无 |
| `capability_detector._probe_api` | capability_detector.py:47 | 否 | 无 |
| `capability_detector.detect_strategy` | capability_detector.py:83 | 否 | 无 |
| `time_tools` 定时器回调 | time_tools.py:323 | 同步 httpx.get() | 无 |

#### 5.1.3 代理配置重复解析

至少 4 处各自内联解析代理配置：
- `network_tools.http_request()` 内联
- `network_tools.download_file()` 内联
- `network_tools.fetch_webpage()` 参数传递
- `_create_http_client()` 在 `network_helper.py`（唯一用统一方式，但无人调用）

### 5.3 LLM 调用路径重复

`BaseAIService` 有两种 LLM 调用方式：

### 5.4 重试逻辑仍有重叠

| 组件 | 机制 | 是否用 RetryEngine |
|------|------|-------------------|

## 六、DB / HTTP / LLM 对比总结

| 维度 | DB 连接管理 | HTTP 客户端管理 | LLM 调用管理 |
|------|-----------|---------------|-------------|
| 是否有统一工厂函数 | ✅ `config.create_connection()` | ⚠️ `network_helper._create_http_client()` **无人用** | ❌ `BaseAIService` 独立，intent_classifier 又另起炉灶 |
| 调用方是否统一使用 | ❌ sessions/messages 混用两种方式 | ❌ 6 个工具函数各自内联 | ❌ intent_classifier / capability_detector 各自新建 |
| 重复定义问题 | ⚠️ `conversation._get_timestamp_ms` | ⚠️ 6 处内联 `httpx.AsyncClient(...)` | ⚠️ 3 处独立的 LLM 客户端 |
| 生命周期管理 | ✅ 连接自动关闭（上下文管理器） | ❌ 内联客户端每个请求新建/销毁 | ❌ 替换时 close() 即发即弃；shutdown 未清理 |
| 重试统一 | — | ❌ 网络工具内联重试未走 RetryEngine | ⚠️ 大部分已统一，ToolRetryEngine 仍独立 |
| 根本原因 | 工具函数放错位置 + 调用方不使用统一入口 | 辅助函数无人调用 + 各函数各自为政 | 未统一 LLM 客户端创建模式 + 生命周期管理缺失 |

**所有三个领域都存在相同的问题模式**：有定义的统一入口 → 调用方绕过 → 各自内联实现 → DRY 违反。

    # 新表名建表（兼容新安装）
    cursor.execute('''
        CREATE TABLE IF NOT EXISTS task_operations (
            ...
