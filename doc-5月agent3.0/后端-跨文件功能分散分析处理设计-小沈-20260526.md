## 三、第二层次：关键路径深度分析

### 关键路径1：请求入口 → 路由分发

**路径**: `main.py` → `sessions.py` → `chat_router.py` → `react_sse_wrapper.py`

#### 发现的系统性问题

| 问题 | 严重度 | 描述 |
|------|--------|------|
| 安全检测双重执行 | **高** | `chat_router.py`L293和`react_sse_wrapper.py`L667各做一次`check_command_safety()`，且逻辑不一致（一个检查`blocked`，另一个检查`is_safe`） |
| `route_with_fallback()`位置错误 | 中 | 纯意图路由决策函数放在`chat_router.py`模块级(L67-149)，应属`intents/`包 |
| 服务层反向依赖API层 | **高** | `react_sse_wrapper` → `message_saver` → `sessions`，服务层不应依赖API层 |
| SSE yield闭包嵌在编排中 | 低 | `yield_sse`闭包(L299-300)的实现细节直接嵌在route()6步编排中，SLAP轻微违反 |

#### 数据流问题

`intent_info`中的`candidates`经历`List[ToolCategory]`→`List[str]`类型转换，信息传递有类型损失。

## 四、第三层次：功能分散深度分析

这是本报告的核心——找出哪些逻辑功能项被分散到多个文件中，七绕八绕。

### 功能项1：意图识别/分类 — 中度分散

**涉及文件**:
- `services/intents/crss_scorer.py`: `TYPE_KEYWORDS`(L37-87), `ACTION_DEFINITIONS`(L96-158), `detect_intent_v2()`(L299-324)
- `services/chat_router.py`: `route_with_fallback()`(L67-149), `_detect_intent()`(L245-256)
- `services/preprocessing/intent_classifier.py`: `classify_intent()`(L124-216), `_INTENT_DEFINITIONS`(L68-81)
- `services/tools/registry.py`: `resolve_category()`

**分散路径**: chat_router → crss_scorer(阶段1) → preprocessing/intent_classifier(阶段2) → registry.resolve_category(类型转换)

**根因**: 
1. 意图定义（关键词+语义描述）在crss_scorer和intent_classifier各维护一套
2. intent_classifier自行读YAML绕过Config单例
3. crss_scorer反向import command_security.DANGEROUS_COMMANDS

**整合方案**: 意图定义集中到`intents/definitions.py`；配置加载统一走Config；危险命令bonus通过参数注入

## 十三、功能分散根因分析

### 13.1 三个核心模式（都是Bug的温床）

| 模式 | 描述 | 涉及的Bug |
|------|------|-----------|
| **双重解析链** | LLM输出被TextStrategy解析一次，又被react_output_parser解析第二次 | 空工具名死循环、FC格式JSON未解析 |

### 13.2 为什么重构后Bug更多？

```
多轮重构的方式:
  第1轮: 拆分大函数 → 提取辅助函数，但保留原调用模式
  第2轮: 增加新功能 → 在辅助函数旁边加新逻辑，不做整合
  第3轮: 修改返回格式 → 只改一处，其他调用点加兼容层
  ...

结果:
  - 旧代码没删 (tool_parser.py 433行死代码)
  - 新旧桥接层膨胀 (file_tools 8个转发方法)
  - 统一化半途而废 (17处result dict各有不同默认值)
  - 每次重构都可能引入覆盖变量的Bug (current_content)
```

## 十、跨文件功能分散分析（新维度：功能完整性 Cohesion）

### 10.1 审查背景

上文的 5 原则分析是基于**结构视角**（一个类/文件/函数做了什么）。在用户要求下，补做一次**
功能视角**的分析：**一个业务逻辑功能，跨了多少个文件？**

> 核心判断标准：如果改一个功能需要读 N 个文件才能理解完整逻辑，那就违反了 SRP 的意图——不是因为一个类做太多，而是因为一个事分散太多。

### 10.2 追踪方法

4 路并行追踪 4 个核心业务功能的数据流：

| 追踪路 | 功能 | 追踪工具 |
|--------|------|---------|
| 路1 | LLM 响应被解析并路由到正确的处理分支 | `grep` + `read` 沿调用链追溯 |
| 路2 | 工具执行：解析出 `action_tool` 后到底发生了什么 | `grep` + `read` 沿 `_execute_tool_step` |
| 路3 | 错误处理：一个 LLM HTTP 错误如何传播到前端 | `grep` + `read` 沿异常栈 |
| 路4 | SSE 事件生成：一个 agent step 如何变成前端 SSE 事件 | `grep` + `read` 沿 yield 链 |

### 10.3 审查结果总览

| 功能 | 涉及文件数 | 涉及函数数 | 核心文件 | 风险等级 |
|------|-----------|-----------|---------|---------|
| **LLM 响应解析链** | 7 个 | ~50+ | `base_react.py` → `react_output_parser.py` → `tool_step_mixin.py` → `react_sse_wrapper.py` | ⚠️ 高 |
| **工具执行链** | 9 个 | ~20+ | `base_react.py` → `tool_step_mixin.py` → `registry.py` → 各 tool impl → 11 个分类文件 | ⚠️ 高 |
| **错误处理链** | 8 个 | ~10+ | `llm_core.py` → `llm_strategies.py` → `react_agent_mixin.py` → `base_react.py` → `react_sse_wrapper.py` → `chat_router.py` | ⚠️ 高 |
| **SSE 事件生成链** | 6 个 | ~15+ | `chat_router.py` → `react_sse_wrapper.py` → `base_react.py` → `tool_step_mixin.py` → `chunk_buffer.py` | ⚠️ 中 |

## 十二、路2：错误处理链 — 8 文件分散

### 12.1 数据流

```
错误发生点（LLM HTTP 错误 / 工具执行异常 / 解析失败 / 安全检查拦截）
  → (1) llm_core.py（HTTP 429/500/503 等原始错误）
  → (2) llm_strategies.py（策略层：restart/fallback/text 错误转换）
  → (3) react_agent_mixin.py（_call_llm：重试探测+错误包装）
  → (4) base_react.py（主循环：捕获并分发到 6 个错误分支）
  → (5) react_sse_wrapper.py（SSE 错误事件格式化）

  → (7) sessions.py（保存错误到历史）
  → (8) logging 各个文件（日志记录）
```

### 12.2 分散详情

| # | 文件 | 在错误链中的角色 | 捕获的错误类型 |
|---|------|----------------|---------------|
| 1 | `llm_core.py` | HTTP 异常抛出：429/500/503/连接超时 | `httpx.HTTPStatusError`, `httpx.TimeoutException` |
| 2 | `llm_strategies.py` | 策略层错误转换：restart→等待→重试 | `LLMResponseError` |
| 3 | `react_agent_mixin.py` | `_call_llm` 第 5 步「策略检测+重试探测」 | 探测 LLM 内部重试状态 |

### 12.3 修复方向

- 集中错误码定义：`errors.py` 定义 `AgentError`, `LLMError`, `ToolError`, `SafetyError` 等
- 消除 `base_react.py` 的 5 处统吃 `Exception`，改为类型化异常
- 消减路由层数：`react_agent_mixin.py` 不应自己做重试探测（应交给 `llm_core.py`）

```
  → (1) base_react.py:run_stream（主循环 yield 原始 step）
  → (2) react_sse_wrapper.py:generate_sse_stream（step 类型分发）
  → (3) react_sse_wrapper.py:_process_step（格式化+SSE 事件构建）
  → (4) chat_router.py:stream_chat（yield SSE 字符串到 HTTP 响应）
  → (5) chunk_buffer.py（chunk 累积+刷新生 SST 缓冲）
  → (6) sessions.py（保存 step 到数据库）
```

### 13.2 分散详情

| # | 文件 | 函数 | 角色 | 问题 |
|---|------|------|------|------|
| 1 | `base_react.py` | `run_stream` (L294-385) | 生成原始 step dict | yield 12 种事件类型，每种子类型格式不统一 |
| 2 | `react_sse_wrapper.py` | `generate_sse_stream` (L448-490) | 事件分发+生成 | L496/605 魔数 `sse_data[6:]` 解析重复 |
| 3 | `react_sse_wrapper.py` | `_process_step` (L198-605) | 精细事件格式化 | 440 行做 5+ 种事件格式化 |
| 4 | `chat_router.py` | `stream_chat` (L266-319) | HTTP SSE 响应 | yield 6 种 SSE 字符串到 Response |
| 5 | `chunk_buffer.py` | `append`, `flush`, `reset` | chunk 缓冲管理 | 3 个方法只在 base_react 中被调用 |
| 6 | `sessions.py` | `save_step_to_db` (L443-448) | 持久化 step | 手动 conn.close() |

**关键发现**: `react_sse_wrapper.py` 的 `_process_step` 440 行是 5 种事件格式化混合体，每个事件类型的字段构造规则不同，但没有统一的 `build_sse_event(type, data)` 函数。`sse_data[6:]` 魔数（去掉 `"data: "` 前缀）重复 2 次，应提取。

### 13.3 修复方向

- `react_sse_wrapper.py` 拆出 `SSEEventFormatter` 统一处理所有事件类型
- 消魔数：`sse_prefix = len("data: ")` 或 `_strip_sse_prefix()`

## 十四、跨文件分散汇总与修复建议

### 14.1 四条业务链对比

| 维度 | LLM 解析链 | 工具执行链 | 错误处理链 | SSE 生成链 |
|------|-----------|-----------|-----------|-----------|

### 14.2 核心结论

从**功能完整性 Cohesion**角度看：

1. **修改任意一个业务功能，需要读 6-9 个文件**才能理解完整逻辑
2. 核心瓶颈不是单个文件大（虽然 `react_output_parser.py` 2274 行确实大），而是**每个文件都沾了一点点不相干职责**
3. 最有修复价值的不是拆文件，而是**清理跨文件的功能碎片**——把分散在 6 个文件中的 LLM 响应解析相关零散逻辑收归 `react_output_parser.py`
