## 一、背景与问题

### 1.1 问题描述

`backend/app/services/tools/registry.py` 是工具注册表模块，当前 **884行**，承载了 **6种不同职责**，导致：

- 全项目 **28个文件** 依赖此模块，牵一发动全身
- 文件名 `registry.py` 与 `registration.py` 极易混淆（已识别的命名问题）
- 修改任何一处功能都可能影响其他无关功能

### 1.2 九大原则违反清单

| 原则 | 违反点 | 严重程度 |
|------|--------|---------|
| **SRP** | 884行文件承载6种职责：类型定义、Schema处理、注册表核心、查询辅助、格式转换、意图解析 | ❌ 严重 |
| **KISS** | 28个文件依赖同一模块，改名牵一发动全身 | ❌ 严重 |
| **DRY** | `get_implementation` 和 `get_exact_implementation` 逻辑100%相同 | ⚠️ 中等 |
| | `get_all_tools_summary` 和 `get_all_tools_detail` 约70%代码重叠（分组遍历、优先排序、分类输出三段完全一样） | ⚠️ 中等 |
| **SLAP** | Schema修复（底层）和工具查询（高层）混在同一层 | ⚠️ 中等 |
| **YAGNI** | `_extract_required_params` 和 `get_tool_meta` 未被外部调用 | ⚠️ 轻微 |
| **ISP** | 接口太胖，不同调用者只需其中一小部分 | ⚠️ 中等 |

## 三、拆分方案

### 3.1 目标文件结构

```
backend/app/services/tools/
├── __init__.py              # 统一导出（更新导入源）
├── tool_types.py            # 【新建】类型定义（~90行，含resolve_category/CATEGORY_ORDER/NAMES/INTENT_TO_CATEGORY）
├── schema_utils.py          # 【新建】Schema处理（~80行）
├── registry.py              # 【精简】核心注册表（~250行，从884行精简）
├── tool_queries.py          # 【新建】查询辅助函数（~60行）
├── tool_format.py           # 【新建】格式转换（~200行）
├── lazy_loader.py           # 【重命名】从 registration.py（~103行）
├── mixin.py                 # 不变
├── tool_config.py           # 不变
└── ...                      # 各分类目录不变
```

### 3.1.1 目录位置归位依据（v1.4补充）

按 AGENTS.md 分层规范，各文件均归入 `tools/` 目录：

| 层级 | 位置 | 职责 | 本方案是否涉及 |
|------|------|------|--------------|
| 全局层 | `app/utils/` | 跨层通用工具（如 time_utils、data_utils） | ✅ format_param_value（v1.3新增，复用优先规则） |
| Agent层 | `app/services/agent/agent_utils/` | Agent 专有逻辑 | ❌ |
| 工具层 | `app/services/tools/` | Tool 专有逻辑 | ✅ 5个新文件均在此 |

**归位判断依据**：

| 新文件 | 依赖的专有类型 | 归入 tools/ 理由 |
|--------|--------------|-----------------|
| `tool_types.py` | ToolCategory, ToolMetadata | 纯工具层类型定义，全局层和 Agent 层无此概念 |
| `schema_utils.py` | 无（纯 Pydantic 操作） | Schema 处理为工具注册服务，调用方仅 registry.py |
| `tool_queries.py` | tool_registry 单例, ToolCategory | 查询对象为工具层的 tool_registry |
| `tool_format.py` | tool_registry 单例, ToolCategory, format_param_value | 展示层数据来自工具层的 tool_registry |
| `lazy_loader.py` | tool_registry 单例 | 懒加载触发方为工具层 |

**唯一例外**：`format_param_value(val)` 放入 `app/utils/common.py`（全局层），因该函数纯值格式化，无工具层依赖，且 `message_utils.py`（Agent层）和 `tool_format.py`（工具层）均需调用，符合跨层公用函数定位。

### 3.1.2 文件/函数命名审查（v1.4新增）

#### 文件名审查

| 文件名 | 核心内容 | 名称是否准确 | 判断理由 |
|--------|---------|-------------|---------|
| `tool_types.py` | ToolCategory/ToolMetadata/常量/resolve_category | ✅ 准确 | "types"表达类型定义，是业界惯例 |
| `schema_utils.py` | _fix_schema_types/_generate_input_schema | ✅ 准确 | "schema_utils"明确表示 Schema 辅助工具 |
| `tool_queries.py` | get_implementations_from_registry 等 | ✅ 准确 | "queries"明确表示查询辅助函数 |
| `tool_format.py` | to_openai_tools/generate_param_reminder 等 | ✅ 准确 | "format"明确表示格式转换 |
| `lazy_loader.py` | ensure_tools_registered/is_tools_registered | ✅ 准确 | "lazy_loader"一眼看出惰性加载职责 |

#### 函数名审查

| 函数名 | 功能 | 名称是否准确 | 判断理由 |
|--------|------|-------------|---------|
| `format_param_value(val)` | None→""、bool→"true"/"false"、其他→str | ✅ 准确 | "format_param_value"清晰表达"格式化参数值" |
| `_group_tools_by_category(...)` | 按分类分组+优先排序，返回 (category_order, by_category) | ✅ 准确 | "group_by_category"是常见命名约定 |
| `_format_default(val)` | 调用 format_param_value 后加 "default=" 前缀 | ✅ 准确 | "format_default"明确表示"格式化默认值" |
| `_fix_schema_types(schema)` | 修复 Pydantic Schema 中缺失的 type 字段 | ✅ 准确 | "fix_schema_types"直接表达修复意图 |
| `_generate_input_schema(...)` | 从 Pydantic 模型生成 input_schema | ✅ 准确 | "generate_input_schema"清晰描述输入输出 |
| `resolve_category(intent_name)` | intent → ToolCategory 字典查表 | ✅ 准确 | "resolve"是"解析/确定"的业界常用词 |

### 3.2 各文件详细设计

#### 文件1: `tool_types.py`（~90行）

**职责**: 工具分类枚举、元数据定义、分类排序常量、意图→分类映射。**纯数据定义，无业务逻辑，无项目内部依赖。**

**包含内容**:

| 内容 | 说明 |
|------|------|
| `ToolCategory` 枚举 | 工具分类：FILE/SYSTEM/NETWORK/DESKTOP/DOCUMENT |
| `ToolMetadata` 数据类 | 工具元数据：name/description/category/version等字段 |
| `CATEGORY_ORDER` | 分类排序顺序（原在registry.py） |
| `CATEGORY_NAMES` | 分类显示名映射（原在registry.py） |
| `INTENT_TO_CATEGORY` | 意图→分类映射字典（原在registry.py） |
| `resolve_category()` | 意图→分类解析（独立函数，不依赖intent_mapper，纯字典查表） |

**依赖关系**:

| 依赖 | 说明 |
|------|------|
| `enum` | stdlib，枚举基类 |
| `dataclasses` | stdlib，数据类基类 |
| `typing` | stdlib，类型注解 |
| `datetime` | stdlib，时间字段 |

**无项目内部依赖**，是最底层的类型定义模块。

**被依赖方**: registry.py、tool_queries.py、tool_format.py、lazy_loader.py，以及全项目20+个文件。

**为什么独立**: `ToolCategory` 被28个文件导入，是最常被引用的符号。独立后，只依赖类型定义的文件不需要加载整个注册表（Schema处理、格式转换等）。

**resolve_category 迁入理由**（v1.1修正）:
- `resolve_category` 是纯字典映射查表，无业务逻辑，属于"类型定义"附属行为
- 迁入后 registry.py 不再依赖 intents 层，消除循环依赖风险
- 调用方改为 `from app.services.tools.tool_types import resolve_category`

### 3.4 拆分方法论（v1.5新增）

**核心原则：能复制就复制，不重写。**

拆分大型文件时，最安全的做法是**复制原代码逻辑，只改导入路径，不改业务逻辑**。重写会引入新错误，复制能保证行为不变。

#### 拆分步骤模板

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | **复制** | 从 registry.py 复制目标代码到新文件，**原样保留** |
| 2 | **修导入** | 在新文件中添加必要的 import，删除不需要的 import |
| 3 | **改函数签名** | 仅当方法→独立函数时，修改 self 参数为 registry 参数（如 tool_format.py） |
| 4 | **删旧代码** | 从 registry.py 删除已移出的代码 |
| 5 | **改调用方** | 更新所有调用方的导入路径 |

#### 严禁事项

| 禁止行为 | 原因 |
|---------|------|
| 拆分时顺便重写逻辑 | 引入新错误，无法对比验证 |
| 拆分时顺便优化算法 | 混入非拆分变更，增加回归风险 |
| 拆分时添加新功能 | 违反 SRP，拆分只做结构变更 |
| 拆分时修改函数行为 | 拆分后必须与拆分前行为100%一致 |

#### 验证方法

拆分前后对比验证：
1. **代码对比**：新文件中的函数逻辑与原 registry.py 中完全一致（仅改 self→registry 参数）
2. **导入验证**：grep 确认旧路径零残留，新路径正确
3. **测试验证**：pytest 全部通过，功能无变化
