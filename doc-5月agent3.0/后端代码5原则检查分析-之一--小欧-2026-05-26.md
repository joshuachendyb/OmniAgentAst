## 一、审查概述

### 1.1 审查背景

基于 5 大编码原则（**SRP**—单一职责、**DRY**—不重复、**KISS**—保持简单、**SLAP**—同一抽象层、**YAGNI**—不需要时不设计），对后端核心业务路径的所有 Python 文件进行深度审查。

### 1.2 审查范围（核心路径）

```
用户输入 → chat_router.py → preprocessing/(分类器+pipeline)
→ intents/(CRSS评分) → react_sse_wrapper.py
→ agent/(base_react+mixins+agent_factory+llm_strategies)
→ llm_adapter.py → llm_core.py → react_output_parser.py
→ tools/(registry+command_security)
→ prompts/BasePromptTemplate
→ api/v1/(sessions+routes)
```

### 1.3 审查方法

| 方法 | 说明 |
|------|------|
| **逐一完整读取** | 每个文件从头到尾完整阅读，不跳行 |
| **逐行分析** | 对照 5 原则逐行检查，标注违规位置和原因 |
| **交叉验证** | 多人独立审查（小欧/小沈/小健/小资），结果汇总 |
| **严重性分级** | P0=必须立即修复、P1=必须修复、P2=应该修复、P3=建议修复 |

### 3.1 base_react.py — BaseAgent 上帝类

**文件**: `services/agent/base_react.py` | **行数**: 762 行 | **审查人**: 小沈

#### 问题描述

`BaseAgent` 类承载了 **10 个以上正交职责**：

| 职责 | 行范围 | 说明 |
|------|--------|------|
| ReAct 循环编排 | 294–385 | `run_stream` 主循环 |
| 响应类型分发 | 346–380 | chunk/answer/thought_only/parse_error/action 分发 |
| 状态管理 | 多处 | `self.status`, `self.steps`, `self.llm_call_count` |
| 空响应重试+历史截断 | 416–456 | 含去重逻辑 |
| 解析错误处理 | 535–571 | 含网络错误判断+延迟 |
| 观察流编排 | 627–667 | 含 pending 调用分发 |
| 中断检查 | 706–724 | 4 处调用 |
| 动态工具加载 | 239–286 | intent → tools 映射 |
| 意图分类器实例 | 138 | `self._intent_classifier` |
| 块缓冲管理 | 460–487, 588–590, 633 | 通过 `ChunkBuffer` 实例 |

**违反原则**: SRP — 修改理由有 10+ 个方向。

#### 具体违规点

| # | 位置 | 问题 | 严重 |
|---|------|------|------|
| 3.1.1 | L72-762 全局 | 10+ 职责混一锅 | P0 |
| 3.1.2 | L311-656 | `_on_after_loop() + return` 模式重复 **12 次**（应在 `finally` 中处理）| P1 |
| 3.1.3 | L363-364 vs L580-581 | `tool_name/tool_params` 提取逻辑重复 | P2 |
| 3.1.4 | L522-524 vs L592-594 | `_thought_val` 合并逻辑重复 | P2 |

#### 修复方案

提取独立服务类：
- `ReActOrchestrator` — 专职 `while` 循环 + 分发
- `ResponseTypeDispatcher` — 专职解析类型分发
- `RunStateManager` — 专职 init/cleanup/interrupt
- 将 `_on_after_loop()` 放入 `try/finally` 消除 12 处重复

## 十、跨文件功能分散分析（新维度：功能完整性 Cohesion）

### 10.1 审查背景

上文的 5 原则分析是基于**结构视角**（一个类/文件/函数做了什么）。在用户要求下，补做一次**
功能视角**的分析：**一个业务逻辑功能，跨了多少个文件？**

> 核心判断标准：如果改一个功能需要读 N 个文件才能理解完整逻辑，那就违反了 SRP 的意图——不是因为一个类做太多，而是因为一个事分散太多。

### 10.2 追踪方法

4 路并行追踪 4 个核心业务功能的数据流：

| 追踪路 | 功能 | 追踪工具 |
|--------|------|---------|
| 路1 | LLM 响应被解析并路由到正确的处理分支 | `grep` + `read` 沿调用链追溯 |
| 路2 | 工具执行：解析出 `action_tool` 后到底发生了什么 | `grep` + `read` 沿 `_execute_tool_step` |
| 路3 | 错误处理：一个 LLM HTTP 错误如何传播到前端 | `grep` + `read` 沿异常栈 |
| 路4 | SSE 事件生成：一个 agent step 如何变成前端 SSE 事件 | `grep` + `read` 沿 yield 链 |

### 10.3 审查结果总览

|------|-----------|-----------|---------|---------|

## 十一、路1：LLM 响应解析链 — 7 文件分散

## 十二、路2：错误处理链 — 8 文件分散

### 12.1 数据流

```
错误发生点（LLM HTTP 错误 / 工具执行异常 / 解析失败 / 安全检查拦截）
  → (1) llm_core.py（HTTP 429/500/503 等原始错误）
  → (2) llm_strategies.py（策略层：restart/fallback/text 错误转换）
  → (3) react_agent_mixin.py（_call_llm：重试探测+错误包装）
  → (4) base_react.py（主循环：捕获并分发到 6 个错误分支）
  → (5) react_sse_wrapper.py（SSE 错误事件格式化）
  → (6) chat_router.py（错误→前端 SSE）
  → (7) sessions.py（保存错误到历史）
  → (8) logging 各个文件（日志记录）
```

### 12.2 分散详情

| # | 文件 | 在错误链中的角色 | 捕获的错误类型 |
|---|------|----------------|---------------|
| 1 | `llm_core.py` | HTTP 异常抛出：429/500/503/连接超时 | `httpx.HTTPStatusError`, `httpx.TimeoutException` |
| 2 | `llm_strategies.py` | 策略层错误转换：restart→等待→重试 | `LLMResponseError` |
| 3 | `react_agent_mixin.py` | `_call_llm` 第 5 步「策略检测+重试探测」 | 探测 LLM 内部重试状态 |
| 4 | `base_react.py` | **5 处** catch 块：parse_error/action_error/llm_error/obs_error/generic_error | `Exception`（统吃）|
| 5 | `react_sse_wrapper.py` | `_run_sse_stream` 外层 catch：错误→SSE `[DONE]` / `error` 事件 | `asyncio.CancelledError`, `Exception` |
| 6 | `chat_router.py` | `stream_chat` 外层 catch：SSE→前端 JSON error | `Exception` |
| 7 | `sessions.py` | 错误保存到 `error_history` 和 `steps_history` | 所有 agent 级别的错误 |
| 8 | 各文件 logging | `logger.error()` / `logger.exception()` | 各自 catch 的异常 |

**关键发现**: 一个 LLM 429 错误，至少要经过 7 个文件的 5 层 catch 才能到前端。每一层都做「错误包装→传递」，但没有任何集中错误码定义或错误类型转换规则。`base_react.py` 的 5 处统吃 `except Exception` 掩盖了具体错误类型。

### 12.3 修复方向

- 集中错误码定义：`errors.py` 定义 `AgentError`, `LLMError`, `ToolError`, `SafetyError` 等
- 消除 `base_react.py` 的 5 处统吃 `Exception`，改为类型化异常
- 消减路由层数：`react_agent_mixin.py` 不应自己做重试探测（应交给 `llm_core.py`）

| 维度 | LLM 解析链 | 工具执行链 | 错误处理链 | SSE 生成链 |
