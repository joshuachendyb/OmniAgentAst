## 一、需求分析与架构范围

### １.1 需求分析

#### 1.1.1 交付模式（结果如何到达用户）

| 模式 | 用户等待方式 | 内部可选方式 |
|------|-------------|-------------|
| **同步** | 在线等待 | 逐段到达 或 整包到达 |
| **异步** | 不等待，事后查结果 | 用户轮询查询 或 服务端推送完成事件 |

注：同步模式下"逐段到达"和"整包到达"是同一种模式下的**传输方式选项**，不是不同的交付模式。异步模式下用户提交后可以离开。

异步模式下，前端可通过查询任务进度轮询获取结果。如需实时通知，可增加SSE推送通道，在任务完成时推送完成事件。两种方式由第1层统一封装，第2层不感知。

#### 1.1.2 用户场景 × 交付模式

| 场景 | 用户行为 | 通常采用的交付模式 |
|------|---------|------------------|
| 对话问答 | 发消息 → 等回复 → 追问 | 同步（也可用异步，后台执行后通知） |
| 任务执行 | 下任务 → 离开 → 回来查结果 | 异步（短任务也可用同步，在线等结果） |

#### 1.1.2a 用户操作流程说明（通俗版）

**同步模式——聊天式操作**（默认，最常见）

用户打开页面，看到的是一个聊天输入框。打字发消息，AI实时逐段回复，像微信聊天一样：

```
用户打字 → 点发送 → 看到AI正在输出... → 输出完了 → 继续发下一条
      ↑______________________ 同一页面，全过程不离开 _______________________↓
```

用户不用做任何选择，打开就用。AI回复过程中用户随时可以点"停止"中断。

适用场景：日常问答、写代码、改文章、翻译等短平快的交互。

---

**异步模式——任务式操作**（用于耗时操作）

页面上除了聊天输入框，还有一个"新建任务"的入口（按钮或菜单）。用户点开后：

```
第1步：填写任务描述（比如"分析项目代码生成架构图"）
第2步：点"提交任务"
第3步：页面显示"任务已提交，编号：T-20260531-001"
第4步：用户可以关掉页面去喝茶/开会/睡觉
第5步：回来后打开"任务列表"看到任务状态
     ├─ 进行中 → 显示进度（已分析10个文件...）
     ├─ 已完成 → 点开看结果
     └─ 已取消 → 显示取消原因
第6步（可选）：如果用户没关页面，任务完成时页面会弹通知
```

用户提交后立刻拿到一个task_id，凭这个id随时回来查进度。

适用场景：分析整个代码库、批量处理文件、导出大型报告等耗时1分钟以上的操作。

---

**同步vs异步，前端怎么区分？**

| | 同步（聊天） | 异步（任务） |
|------|-------------|-------------|
| **入口** | 聊天输入框 | "新建任务"按钮 |
| **提交后** | 直接看到AI回复（逐段或整包） | 拿到task_id |
| **用户能不能走开** | 不能，在线等 | 能，走后回来查 |
| **能不能中断** | 能（点停止按钮） | 能（取消任务） |
| **页面会不会关** | 不关 | 可以关 |
| **结果怎么拿** | 直接看到 | 打开任务列表查看 |

**特别注意**：同一个会话里可以同时有同步对话和异步任务，互不干扰。比如用户一边聊天问问题，一边提交了一个耗时任务在后面跑着。

---

**关于SSE流式（用户不用管）**

同步模式下，AI回复的内容是逐段显示出来的（像打字效果一样）。这是后台自动用SSE技术推送的，**用户不需要做任何设置或选择**。

有的系统让用户选"流式"还是"整包"——本架构不这么做。原因：
- 用户只管发消息、看回复，不需要知道底层用什么技术
- SSE流式由第1层自动处理，对用户完全透明
- 无论AI回复多长，用户看到的效果都是"内容逐段出现"
- 如果用户想等全部内容出来再看，不操作就行——流式推送结束后自然就是完整结果

#### 1.1.3 系统必须支持的能力

| 能力 | 说明 |
|------|------|
| 接收用户消息并生成回复 | 用户发文本，系统处理后返回文本 |
| 回复到达（同步模式下支持逐段或整包） | 用户在线等待，内容可逐段到达或一次性到达 |
| 使用外部资源 | 处理过程中可读文件、执行命令、查询网络等 |
| 分多步完成复杂任务 | 单个任务可能需要多次"获取信息→处理→再获取"循环 |
| 上下文保持 | 同一会话中，前面的对话影响后面的回复 |
| 用户可中断 | 用户可随时停止进行中的回复生成或任务。中断后已生成的回复内容保留，正在执行的工具调用停止（或允许当前调用执行完毕），会话不销毁，用户可以继续发送新消息。异步任务中断后状态标记为已取消，已执行步骤记录保留 |
| 失败自动重试 | 通信层：网络闪断、服务超时等临时故障自动重试；业务层：AI返回空响应、格式错误时自动重试。两层重试配合，不丢数据 |
| 多会话并发 | 多个对话/任务同时运行，互不干扰 |
| 模型可切换 | 不同场景使用不同AI模型，也可由用户主动切换 |
| 敏感操作确认 | 执行潜在危险操作前先征得用户同意。同步模式下用户在线可选择同意/拒绝；异步模式下任务暂停，标记为"待用户确认"，用户回来后处理 |
| **多个内部模块共用AI调用** | 对话、意图分类、模型能力探测都需要调用AI，调用入口统一 |
| **逐段输出过程中可插入工具调用** | AI在逐段输出中可随时下发工具调用指令。第2层收到指令后调度工具执行，执行结果回传第3层，LLM继续生成后续文本。多个工具调用按LLM返回的顺序串行执行 |
| **不同模型能力兼容** | 不同AI模型的工具调用能力、输出格式支持程度不同，系统需统一处理差异 |
| **任务追踪** | 每次操作的全生命周期（创建→执行→完成/失败/回滚）可追溯 |

####  1.1.4 架构必须提供的底层设施

| 设施 | 说明 |
|------|------|
| 多模型接入 | 接入不同AI模型提供商，调用方式统一 |
| 外部资源扩展 | 新增读写文件、执行命令、网络查询等能力时，不改核心处理流程 |
| 会话隔离 | 不同会话的执行上下文各自独立 |
| 可观测性 | 每次处理、每次外部调用都有日志可追溯 |
| 配置管理 | 模型参数、超时时间、重试策略等可配置 |

###   1.2 架构原则（设计约束）

| 原则 | 说明 | 违反后果 |
|------|------|---------|
| **SRP** | 每层只做一件事（详见章节一的分层定义） | 职责混杂，改A影响B |
| **严格调用** | 入口层→编排层→LLM服务层，不可跳过或越层（详见章节一） | 逻辑不可控，重试/状态管理遗漏 |
| **接口契约** | 层间通过定义接口通信，不直接依赖具体实现 | 层耦合，换实现牵一发动全身 |
| **不向后兼容** | 新设计只对需求负责，现有代码后续按新设计重构 | 被旧代码绑架，设计变形 |

### 1.3 设计决策待定

以下决策影响第2层和第3层的具体形态：

| 决策 | 选项A | 选项B | 影响范围 |
|------|-------|-------|---------|
| **第2层vs传输协议** | 第2层输出**事件对象**，第1层负责序列化 | 第2层直接输出序列化后的数据 | 第2层是否与传输格式耦合 |
| **整包（非流式）路径** | 整包（非流式）**统一走第2层**编排 | 整包（非流式）跳过第2层直接调用底层 | 架构一致性 |
| **工具基础设施** | 作为**第2层内部组件** | 作为**独立公共层**，供多层使用 | 工具的访问范围 |
| **重试职责** | 第2层=业务重试（空响应/格式错），第3层=通信重试（超时/限流） | 全部集中在第2层 | 重试逻辑的归属 |

注：整包路径的决策会影响1.2"严格调用"原则的适用范围——若选B，则需在该原则中增加"整包快捷路径例外"条款。

###   1.4 不在范围内（不做的）

| 领域 | 原因 |
|------|------|
| 用户认证与授权 | 本架构不涉及，由上层系统处理 |
| 数据库持久化 | 本架构不涉及存储实现 |
| 前端UI渲染 | 本架构不涉及前端实现 |
| 工具内部实现 | 本架构只负责调度工具，不实现工具内部逻辑 |
| 部署与运维 | 本架构不涉及 |

### 1.5 三层架构各层职责定义

以下定义三层架构中每一层**必须承担的职责**和**禁止承担的职责**，是架构设计的硬约束。

---

#### 第1层：用户入口层

| 维度 | 说明 |
|------|------|
| **定位** | 用户与系统的唯一交互界面，所有请求的起点 |

**对前端的接口（前端调用第1层）**：

| 接口 | 接口类型 | 请求参数 | 返回值 | 说明 |
|------|---------|---------|--------|------|
| 发送消息 | 同步请求 | 用户消息文本 | AI回复内容（SSE流式） | 用户发消息后在线等结果。回复内容通过SSE通道自动逐块到达，用户无需任何设置 |
| 发送任务 | 异步请求 | 任务描述文本 | task_id | 用户发任务后可离开 |
| 查询任务进度 | 查询接口 | task_id | 任务状态+已输出内容 | 用户查看任务执行状态 |
| 获取任务结果 | 查询接口 | task_id | 最终结果 | 任务完成后查看结果 |
| 取消任务 | 中断接口 | task_id | 中断确认 | 用户点击取消按钮 |
| 接收异步完成通知 | SSE推送 | 会话ID | 完成事件 | 异步任务完成后，服务端通过SSE推送完成通知 |

> 说明：SSE（Server-Sent Events）流式推送是第1层的内部机制，用户无需干预。消息发出后，AI回复自动通过SSE通道逐块到达（事件类型：text/tool/final）。前端只需要建立一次SSE连接，后续所有同步回复都通过此连接推送。"流式"和"整包"是AI回复的到达方式，不是用户的选择项。

**对下的接口（第1层调用第2层）**：

| 接口 | 调用方向 | 请求参数 | 返回值 | 说明 |
|------|---------|---------|--------|------|
| 处理同步请求 | 第1层→第2层 | 用户消息 | 完整AI回复 | 转发同步请求给编排层，编排层处理完成后返回结果，由第1层通过SSE推送给前端 |
| 处理异步请求 | 第1层→第2层 | 任务描述 + task_id | 受理确认 | 转发异步请求给编排层，后台执行 |
| 转发中断信号 | 第1层→第2层 | task_id | 中断确认 | 将用户取消指令转发给编排层 |
| 查询任务状态 | 第1层→第2层 | task_id | 任务状态+已输出内容 | 向编排层查询任务进度 |

**禁止承担的职责**：

| 禁止 | 原因 |
|------|------|
| 不做业务编排（含推理循环） | 编排层的职责 |
| 不调用AI模型 | AI通信的职责 |
| 不做工具调度 | 编排层的职责 |
| 不做重试 | 编排层处理业务重试，LLM服务层处理网络重试 |
| 不做状态管理 | 编排层的职责 |

---

#### 第2层：业务编排层

| 维度 | 说明 |
|------|------|
| **定位** | 业务流程的大脑，管理多步推理的完整生命周期 |

**必须承担的职责**：

| 职责 | 说明 |
|------|------|
| 多步推理管理 | AI一次响应不够时，自动组织下一轮调用，直到得到最终结论 |
| 工具调度 | AI需要使用外部资源时，找到对应工具并执行 |
| 执行状态记录 | 每一步做了什么、结果是什么，全程可追溯 |
| 上下文维护 | 多轮对话时，将前面的对话内容带给AI |
| 业务级重试 | AI返回空响应或格式错误时，自动重试 |
| 异步任务生命周期 | 异步任务从创建到完成/失败/回滚的全过程管理 |
| 意图分类 | 用户消息进入编排后，由AI判断意图类型，决定处理策略 |
| 模型能力探测 | 调度模型前先探测模型支持能力（是否支持工具调用等） |

**禁止承担的职责**：

| 禁止 | 原因 |
|------|------|
| 不直接调用AI模型 | LLM服务层的职责 |
| 不构建HTTP请求 | LLM服务层的职责 |
| 不解析原始AI响应 | LLM服务层的职责 |
| 不做网络级重试 | LLM服务层的职责 |
| 不直接执行工具 | 工具自身实现业务逻辑，编排层只调度 |

---

#### 第3层：LLM服务层

| 维度 | 说明 |
|------|------|
| **定位** | 与外部AI模型的通信桥梁，屏蔽不同模型的接口差异 |

**必须承担的职责**：

| 职责 | 说明 |
|------|------|
| 流式通信 | 与AI模型进行逐块数据传输 |
| 整包通信 | 与AI模型进行完整请求-响应传输 |
| 模型能力适配 | 不同AI模型的能力差异由本层屏蔽，上层无需感知 |
| 网络级重试 | 网络超时、限流、连接断开时自动重试 |

**禁止承担的职责**：

| 禁止 | 原因 |
|------|------|
| 不做业务编排 | 编排层的职责 |
| 不做工具调度 | 编排层的职责 |
| 不做业务级重试 | 编排层的职责 |
| 不做状态管理 | 编排层的职责 |
| 不做上下文管理 | 编排层的职责 |
| 不判断工具是否应该执行 | 只负责从AI响应中提取工具调用参数 |

#### 前端对应功能与层次归属

前端所有功能**只与第1层交互**，不直接调用第2层和第3层。

| 前端功能 | 用户操作 | 所属层次 | 说明 |
|---------|---------|---------|------|
| 发送消息 | 用户输入文字后发送 | 第1层：用户入口层 | AI回复通过SSE通道自动逐块到达，用户无需选择传输方式 |

| 取消任务 | 点击取消按钮 | 第1层：用户入口层 | 中断信号转发给编排层 |

##  二、正确的架构层次模型

### 2.1 三层架构总览

```
┌─────────────────────────────────────────────────────────────────┐
│                    第1层：用户入口层                              │
│  职责：接收用户请求，转发到下一层，返回最终结果                      │

###   2.2 三层各层的职责定义

#### 第1层：用户入口层

| 维度 | 说明 |
|------|------|
| **逻辑功能** | 接收用户请求，转发到第2层，返回第2层的处理结果 |
| **边界** | 只做请求转发和响应封装 |

**对前端的接口（前端调用第1层）**：

| 接口 | 接口类型 | 请求参数 | 返回值 | 说明 |
|------|---------|---------|--------|------|
| 发送消息 | 同步请求 | 用户消息文本 | AI回复（流式或整包） | 用户发消息后在线等结果 |
| 接收流式回复 | SSE流式 | 用户消息文本 | 事件流（text/tool/final） | 逐块返回AI生成内容 |
| 接收整包回复 | 同步请求 | 用户消息文本 | 完整AI回复 | 等待完整回复后一次性返回 |
| 发送任务 | 异步请求 | 任务描述文本 | task_id | 用户发任务后可离开 |
| 查询任务进度 | 查询接口 | task_id | 任务状态+已输出内容 | 用户查看任务执行状态 |
| 获取任务结果 | 查询接口 | task_id | 最终结果 | 任务完成后查看结果 |
| 取消任务 | 中断接口 | task_id | 中断确认 | 用户点击取消按钮 |

**对下的接口（第1层调用第2层）**：

| 接口 | 调用方向 | 请求参数 | 返回值 | 说明 |
|------|---------|---------|--------|------|
| 处理同步请求 | 第1层→第2层 | 用户消息 | 完整AI回复 | 转发同步请求给编排层 |
| 处理异步请求 | 第1层→第2层 | 任务描述 | task_id | 转发异步请求给编排层，后台执行 |
| 转发中断信号 | 第1层→第2层 | task_id | 中断确认 | 将用户取消指令转发给编排层 |
| 查询任务状态 | 第1层→第2层 | task_id | 任务状态+已输出内容 | 向编排层查询任务进度 |

**禁止实现的功能**：

| 禁止 | 原因 |
|------|------|
| 不做业务编排 | 推理循环、工具调度由第2层负责 |
| 不调用LLM API | LLM通信由第3层负责 |
| 不做推理循环 | "思考→工具→观察→再思考"由第2层负责 |
| 不做工具调度 | 工具查找、执行由第2层负责 |
| 不做重试逻辑 | 网络重试由第3层，业务重试由第2层 |
| 不做状态管理 | 步骤记录、上下文由第2层负责 |

---

#### 第2层：业务编排层

| 维度 | 说明 |
|------|------|
| **逻辑功能** | 管理多步推理循环，调度工具，维护状态，处理业务级重试 |
| **边界** | 只做业务流程编排，不直接调用LLM API |

**对上的接口（第1层调用第2层）**：

| 接口 | 调用方向 | 请求参数 | 返回值 | 说明 |
|------|---------|---------|--------|------|
| 处理同步请求 | 第1层→第2层 | 用户消息 | 完整AI回复 | 接收同步请求，执行推理循环，返回最终结果 |
| 处理异步请求 | 第1层→第2层 | 任务描述 | task_id | 接收异步请求，后台执行推理循环 |
| 转发中断信号 | 第1层→第2层 | task_id | 中断确认 | 终止推理循环，返回中断结果 |
| 查询任务状态 | 第1层→第2层 | task_id | 任务状态+已输出内容 | 返回当前任务的执行状态和已生成内容 |

**对下的接口（第2层调用第3层）**：

| 接口 | 调用方向 | 请求参数 | 返回值 | 说明 |
|------|---------|---------|--------|------|
| 流式调用LLM | 第2层→第3层 | 消息列表+工具定义 | 流式响应（逐块） | 获取LLM的流式响应 |
| 整包调用LLM | 第2层→第3层 | 消息列表+工具定义 | 完整响应 | 获取LLM的完整响应 |

**必须实现的功能**：

| 功能 | 说明 |
|------|------|
| 推理循环编排 | 接收任务 → 调用第3层获取LLM响应 → 解析响应 → 如果需要工具则调度工具 → 将工具结果回传第3层 → 循环直到LLM给出最终结论 |
| 工具调度 | 从任务库中查找指定工具 → 执行工具 → 将工具执行结果回传给推理循环 |
| 状态管理 | 记录每个推理步骤（LLM调用、工具执行、中间结果） → 供后续步骤和最终结果使用 |
| 上下文管理 | 维护多轮对话历史 → 每次调用第3层时带上相关上下文 |
| 业务级重试 | 检测LLM返回是否有效（空响应、格式错误、工具失败） → 决定重试还是换策略 → 与第3层的网络重试配合 |
| 任务追踪 | 异步任务的生命周期管理（创建→执行中→完成/失败/回滚） |

**禁止实现的功能**：

| 禁止 | 原因 |
|------|------|
| 不直接调用LLM API | 所有LLM调用必须通过第3层 |
| 不构建HTTP请求 | 请求构建由第3层负责 |
| 不解析原始SSE流 | 原始流解析由第3层负责 |
| 不做网络级重试 | 网络超时/限流/断连的重试由第3层负责 |
| 不直接执行工具 | 工具的业务逻辑由工具自身实现，第2层只调度 |

---

#### 第3层：LLM服务层

| 维度 | 说明 |
|------|------|
| **逻辑功能** | 构建LLM API请求，发送请求，解析响应，返回结构化结果 |
| **边界** | 只负责LLM通信，不做任何业务逻辑 |


**对上的接口（第2层调用第3层）**：

第3层向上提供两种**响应类型**，每种类型支持两种**传输方式**：

| 响应类型 | 传输方式 | 说明 |
|---------|---------|------|
| **纯消息(message)** | 流式（逐块） | LLM返回文本内容，逐块到达 |
| **纯消息(message)** | 整包（一次性） | LLM返回文本内容，一次性到达 |
| **工具调用(toolcall)** | 流式（逐块） | LLM返回工具调用指令，逐块到达 |
| **工具调用(toolcall)** | 整包（一次性） | LLM返回工具调用指令，一次性到达 |

**对下的接口（第3层调用LLM API）**：

| 接口 | 调用方向 | 请求参数 | 返回值 | 说明 |
|------|---------|---------|--------|------|
| HTTP流式请求 | 第3层→LLM API | OpenAI格式请求体（stream=true） | SSE流式响应 | 与外部LLM进行流式通信 |
| HTTP整包请求 | 第3层→LLM API | OpenAI格式请求体 | JSON响应 | 与外部LLM进行整包通信 |

**必须实现的功能**：

| 功能 | 说明 |
|------|------|
| 纯消息响应解析 | 从LLM响应中提取文本内容，支持流式和整包两种传输方式 |
| 工具调用响应解析 | 从LLM响应中提取工具调用参数（工具名+参数），支持流式和整包两种传输方式 |
| 模型能力适配 | 不同模型的工具调用能力不同 → 统一接口屏蔽差异 → 第2层无需感知具体模型差异 |
| 网络级重试 | 网络超时、限流、连接断开时自动重试 |

**禁止实现的功能**：

| 禁止 | 原因 |
|------|------|
| 不做业务编排 | 推理循环由第2层负责 |
| 不做工具调度 | 工具调度由第2层负责 |
| 不做业务级重试 | 空响应/格式错误的重试由第2层负责 |
| 不做状态管理 | 步骤记录由第2层负责 |
| 不做上下文管理 | 对话历史管理由第2层负责 |
| 不判断工具是否应该执行 | 只负责从LLM响应中提取工具调用参数 |

---

#### 三层禁止关系速查

| 调用方向 | 允许 | 禁止 |
|---------|------|------|
| 第1层 → 第2层 | ✅ | 第1层直接调用第3层 |
| 第1层 → 第3层 | ❌ | 必须经过第2层 |
| 第2层 → 第3层 | ✅ | 第2层直接调用LLM API |
| 第3层 → 第2层 | ❌ | 第3层不能反调第2层 |
| 第3层 → LLM API | ✅ | - 

**三个维度的关系说明**：

| 维度 | 定义 | 谁负责 | 取值 |
|------|------|--------|------|
| **响应类型** | LLM返回什么 | 第3层 | 纯消息(message) / 工具调用(toolcall) |
| **传输方式** | 数据怎么到达 | 第3层 | 流式(逐块) / 整包(一次性) |
| **交付模式** | 用户是否等待 | 第1层 | 同步(在线等) / 异步(可离开) |

三者正交，互不影响。第3层只关心**响应类型**和**传输方式**，不关心交付模式

注：流式模式下，message块和toolcall块可在同一流中交错出现（如先吐文本、再下工具调用指令、继续吐文本）。第3层按LLM返回的原始顺序逐块传递，不重新排序。

### 2.6 同步/异步模式对比

| 维度 | 同步模式 | 异步模式 |
|------|---------|---------|
| **用户是否等待** | 是，一直等到结果返回 | 否，提交后可离开 |
| **第1层行为** | 调用第2层后阻塞等待 | 调用第2层后立即返回task_id |
| **第2层行为** | 在调用方线程内执行推理循环 | 在后台线程内执行推理循环 |
| **结果交付** | 逐段到达（流式）或整包到达 | 用户查询时获取，或完成后通知 |
| **用户可中断** | ✅ 随时发送中断指令 | ✅ 随时发送中断指令 |
| **适用场景** | 对话问答（用户在线等） | 任务执行（用户可离开） |

### 2.7 前端对应接口定义

前端所有功能**只与第1层交互**，不直接调用第2层和第3层。以下定义前端每个功能对应的接口。

| 前端功能 | 接口类型 | 所属层次 | 接口说明 | 请求参数 | 返回值 |
|---------|---------|---------|---------|---------|--------|
| 发送消息 | 同步请求 | 第1层 | 接收用户输入，调用编排层处理 | 用户消息文本 | AI回复（流式或整包） |
| 接收流式回复 | SSE流式 | 第1层 | 逐块返回AI生成内容 | 用户消息文本 | 事件流（text/tool/final） |
| 接收整包回复 | 同步请求 | 第1层 | 等待完整回复后一次性返回 | 用户消息文本 | 完整AI回复 |
| 发送任务 | 异步请求 | 第1层 | 接收任务，后台执行 | 任务描述文本 | task_id |
| 查询任务进度 | 查询接口 | 第1层 | 返回任务当前状态 | task_id | 任务状态+已输出内容 |
| 获取任务结果 | 查询接口 | 第1层 | 任务完成后返回最终结果 | task_id | 最终结果 |
| 取消任务 | 中断接口 | 第1层 | 转发中断指令给编排层 | task_id | 中断确认 |

### 2.8 详细设计的编写规范

本节定义第3/4/5章详细设计的编写方法和输出要求。

#### 2.8.1 设计方法

| 项目 | 说明 |
|------|------|
| **层间关系** | 三层之间采用调用关系，不采用继承关系。层间接口通过函数调用实现 |
| **层内结构** | 每层内部可以使用类和对象，但类之间是组合/调用关系，不是继承关系 |
| **设计来源** | 严格基于第2章概要设计展开，每层的职责、接口、禁止项必须与概要设计一致 |
| **设计方向** | 自顶向下：先定义模块划分，再定义函数清单，最后定义函数签名和调用关系 |

#### 2.8.2 设计必须遵守的原则

| 原则 | 说明 | 在详细设计中的体现 |
|------|------|------------------|
| **SRP** — 单一职责 | 一个函数只做一件事 | 每个函数功能说明只描述一件事 |
| **DRY** — 不重复 | 相同逻辑只写一次 | 新增函数前先查是否已有类似功能 |
| **KISS** — 保持简单 | 能简单不复杂 | 函数签名和调用链尽量简洁 |
| **SLAP** — 同一抽象层 | 一个函数不混搭高层编排和底层细节 | 函数说明中明确其抽象层级 |
| **YAGNI** — 不要过度设计 | 不加用不上的接口 | 只设计概要设计要求的功能，不提前设计"可能用到"的功能 |
| **复用优先** | 有公用则复用 | 新增函数前必须先查公用函数清单 |

#### 2.8.3 每章必须包含的内容

| 章节 | 内容 | 说明 |
|------|------|------|
| 代码结构 | 本层需要哪些代码文件 | 文件名、职责、与其他文件的关系 |
| 函数清单 | 本层有哪些函数 | 函数名、功能说明、输入参数、返回值 |
| 对外接口 | 本层对外暴露的接口 | 对应概要设计中的接口定义 |
| 对下接口 | 本层调用下层的接口 | 对应概要设计中的接口定义 |
| 调用关系 | 函数之间的调用链 | 从入口函数到最终执行的完整路径 |
| 错误处理 | 异常情况的处理策略 | 每个函数可能的异常和处理方式 |

#### 2.8.4 与概要设计的对应关系

```
概要设计（第2章）                详细设计（第3/4/5章）
─────────────────              ─────────────────────
三层架构总览         →          每层的代码结构图
每层职责定义         →          每层的函数清单

```

#### 2.8.5 命名规范

| 项目 | 规范 | 示例 |
|------|------|------|
| **函数命名** | 小写下划线，动词开头，体现动作 | `handle_sync_request()`、`call_llm_stream()` |
| **文件命名** | 小写下划线，体现职责 | `sync_handler.py`、`llm_client.py` |
| **类命名** | 大驼峰，体现职责 | `OrchestrationEngine`、`LLMAdapter` |
| **常量命名** | 大写下划线 | `MAX_RETRY_COUNT`、`DEFAULT_TIMEOUT` |
| **接口命名** | 体现调用方向和动作 | 第1层对下：`forward_to_orchestration()`；第2层对下：`call_llm_service()` |

#### 2.8.６　三个维度的关系说明［重复说明一下］

| 维度 | 定义 | 谁负责 | 取值 |
|------|------|--------|------|
| **响应类型** | LLM返回什么 | 第3层 | 纯消息(message) / 工具调用(toolcall) |
| **传输方式** | 数据怎么到达 | 第3层 | 流式(逐块) / 整包(一次性) |
| **交付模式** | 用户是否等待 | 第1层 | 同步(在线等) / 异步(可离开) |

三者正交，互不影响。第3层只关心**响应类型**和**传输方式**，不关心交付模式

注：流式模式下，message块和toolcall块可在同一流中交错出现，第3层按LLM返回的原始顺序逐块传递，不重新排序。

### 3.5 错误处理

| 异常类型 | 触发条件 | 处理方式 | 重试策略 |
|---------|---------|---------|---------|
| `LLMConnectionError` | 网络连接失败、DNS解析失败 | 重试3次，指数退避（1s/2s/4s） | 自动重试 |
| `LLMRateLimitError` | API返回429限流 | 重试3次，等待时间按API返回的Retry-After | 自动重试 |
| `LLMTimeoutError` | 请求超时（默认60s） | 重试2次，每次增加超时时间 | 自动重试 |
| `LLMResponseError` | 响应格式错误、解析失败 | 记录日志，向第2层报告错误 | 不重试，由第2层决定 |
| `LLMAuthError` | API密钥无效、认证失败 | 立即向第2层报告，不重试 | 不重试 |

### 3.2 问题1：BaseAIService类跨越两个层次

| 维度 | 说明 |
|------|------|
| **问题描述** | `BaseAIService` 类同时包含用户入口层方法（chat）和LLM服务层方法（chat_stream） |
| **代码位置** | `llm_core/llm_core.py:39` |
| **涉及方法** | `chat()`（第1层，已拆至`llm_core/chat.py:15`）+ `chat_stream()`（第3层，`llm_core/chat_stream.py:16`） |
| **风险等级** | P1-高 |

**精准分析**：
- `chat()` 方法（`llm_core/chat.py:15`）是用户入口，接收用户消息，返回ChatResponse（从原`llm_core.py:219`拆出）
- `chat_stream()` 方法（`llm_core/chat_stream.py:16`）是LLM服务层，真正调用LLM API（从原`llm_core.py:265`拆出）
- 两个方法虽已拆为独立文件，但仍通过类继承（ChatStreamMixin混合到BaseAIService）在同一个类中
- 同样 `chat_with_tools()`（第1层，`llm_core/tool_caller.py:58`）和 `chat_with_tools_stream()`（第3层，`llm_core/chat_with_tools_stream.py:21`）也存在同样问题

**改进措施**：
```
当前结构（SRP拆文件但未拆类继承）：
BaseAIService (llm_core/llm_core.py:39)
├── chat()                    → llm_core/chat.py:15   ← 第1层：用户入口
├── chat_stream()             → llm_core/chat_stream.py:16  ← 第3层：LLM服务
├── chat_with_tools()         → llm_core/tool_caller.py:58  ← 第1层：用户入口
└── chat_with_tools_stream()  → llm_core/chat_with_tools_stream.py:21  ← 第3层

ChatEntry (chat_entry.py)        ← 第1层：用户入口
├── chat()
└── chat_with_tools()

LLMService (llm_service.py)      ← 第3层：LLM服务
├── chat_stream()
└── chat_with_tools_stream()
```

- `route()` 函数（`chat_router/route.py:19`，原 `chat_router.py:279`）是用户入口，接收请求，编排4步流程
- `step_start()` 函数（`chat_router/step_start.py:15`，原 `chat_router.py:255` 的 `_step_start()`）是第2层start步骤编排
- `step_react_loop()` 函数（`chat_router/step_react_loop.py:13`，原 `chat_router.py:267` 的 `_step_react_loop()`）是第2层ReAct循环编排
- `detect_intent()` 函数（`chat_router/detect_intent.py:13`）是第2层意图检测
- `init_route_context()` 函数（`chat_router/init_route_context.py:15`）是第2层初始化
- SRP已做了方法到独立文件的拆分，但第1层和第2层仍在同一个 `chat_router/` 目录下，入口 `route()` 仍直接调用编排函数

**改进措施**：
```
当前结构（已拆为独立文件但同目录混合）：
chat_router/ 目录
├── route.py:19                    ← 第1层：用户入口
├── step_start.py:15               ← 第2层：start步骤
├── step_react_loop.py:13          ← 第2层：ReAct循环
├── detect_intent.py:13            ← 第2层：意图检测
├── init_route_context.py:15       ← 第2层：初始化
├── chat_stream_v2.py              ← FastAPI端点
└── ...

正确结构：
ChatEntry (chat_entry.py)          ← 第1层：用户入口
└── route()

StepOrchestrator (step_orchestrator.py)  ← 第2层：业务编排
├── detect_intent()
├── step_start()
└── step_react_loop()
```

### 3.6 问题5：层与层之间没有清晰的接口边界

| 维度 | 说明 |
|------|------|
| **问题描述** | 层与层之间直接依赖具体实现，没有定义清晰的接口 |
| **涉及模块** | 全局 |
| **风险等级** | P2-中 |

**精准分析**：
- 第1层直接调用第2层/第3层的具体方法，没有接口抽象
- 第2层直接调用第3层的具体方法，没有接口抽象
- 层与层之间耦合度过高，修改一层会影响其他层

**具体案例**：
```
step_react_loop() [chat_router/step_react_loop.py:13]
  └── generate_sse_stream(ai_service=ai_service, ...)  ← 直接传具体对象
        └── run_sse_stream(llm_client=ai_service, ...)  ← 直接传具体对象
              └── AgentFactory.create(...)  ← 直接依赖AgentFactory
                    └── agent.run_stream() [base_react/run_stream.py:17] ← 直接调用

```

**改进措施**：
```
当前结构：
第1层 → 直接调用第2层/第3层的具体方法

正确结构：
第1层 → 定义接口 → 第2层实现接口
第2层 → 定义接口 → 第3层实现接口
```

## 四、改进措施方案

### 4.1 重构建议汇总

| 重构方向 | 具体措施 | 预期效果 | 工作量 |
|---------|---------|---------|--------|
| **拆分BaseAIService** | 将chat()/chat_with_tools()移到第1层入口模块，chat_stream()/chat_with_tools_stream()保留在第3层LLM服务模块 | 消除层次跨越（问题1） | 中 |
| **拆分generate_sse_stream** | 将入口分发逻辑与编排执行逻辑分离到不同模块 | 消除层次跨越（问题2） | 小（文件已拆，职责未分） |
| **拆分BaseAgent** | 将run()移到第1层入口模块，run_stream()保留在第2层编排模块 | 消除层次跨越（问题3） | 中（文件已拆，类继承未拆） |
| **拆分ChatRouter** | 将route()入口与step_*()编排分离到不同模块 | 消除层次混杂（问题4） | 小（文件已拆，同目录未分） |
| **层间接口定义** | 第1层↔第2层↔第3层之间定义清晰接口 | 降低耦合度（问题5） | 中 |
| **重试机制收敛** | 统一使用RetryEngine，SSE路径补充重试保护 | 消除重试分散（问题6） | 小 |

> **注**：2026-05-31 SRP目录拆分后，问题1-4的文件级拆分已完成，但职责级分离（入口vs编排）和类继承解耦仍未完成。
