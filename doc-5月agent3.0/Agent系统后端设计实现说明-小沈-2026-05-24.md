## 1. 总体架构

### 1.1 架构层次

Agent系统采用**四层架构**：

```
┌─────────────────────────────────────────────────────────────┐
│  第1层：配置与工厂层                                         │
│  agent_config.py + agent_factory.py                         │
│  职责：声明式配置注册表 → 按意图类型创建正确的Agent实例         │
├─────────────────────────────────────────────────────────────┤
│  第2层：Agent实现层                                          │
│  universal_react.py + desktop_react.py                      │
│  职责：2个具体Agent子类，处理所有5种意图类型                    │
├─────────────────────────────────────────────────────────────┤
│  第3层：核心框架层                                            │
│  base_react.py (ReAct循环) + mixins/ (公用逻辑)              │
│  职责：ReAct循环核心 + 工具加载/LLM策略/回滚等可插拔能力         │
├─────────────────────────────────────────────────────────────┤
│  第4层：基础设施层                                            │
│  react_output_parser.py (解析器链)                           │
│  reasoning_steps.py (Step类体系 + StepFactory)               │
│  message_builder.py (消息管理)                               │
│  llm_strategies.py (TextStrategy + ToolsStrategy)             │
│  tool_executor.py (工具执行 + 重试)                          │
│  tool_result_formatter.py (结果格式化)                       │
│  types/ (类型定义)                                           │
│  职责：为上层提供所有基础设施能力                                │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 核心设计原则

| 原则 | 说明 |
|------|------|
| **声明式配置驱动** | 5种意图类型通过 `AGENT_REGISTRY` 声明式定义，无需子类 |
| **2个Agent子类** | `UniversalReactAgent` 处理 file/system/network/document；`DesktopReactAgent` 独立处理 desktop |
| **Mixin可插拔** | `ReactAgentMixin`（工具/LLM/追踪）+ `RollbackMixin`（回滚能力）通过多继承注入 |
| **解析器链模式** | 9个handler按优先级顺序尝试，每个独立处理一种输入格式 |
| **统一Step封装** | 所有yield输出通过 `StepFactory` → `ReasoningStep` 子类，消除裸字典 |
| **集中消息管理** | `MessageBuilder` 统一管理 `conversation_history` 的所有写操作 |

### 1.3 意图类型与Agent映射

| 意图类型 | 别名 | Agent类 | 工具分类(ToolCategory) | CRSS→ToolCategory | 回滚支持 |
|---------|------|---------|----------------------|-------------------|---------|
| file | 无 | UniversalReactAgent | FILE | FILE | ✅ |
| system | shell, meta, time, environment, env, code_execution | UniversalReactAgent | SYSTEM | SYSTEM | ❌ |
| network | 无 | UniversalReactAgent | NETWORK | NETWORK | ❌ |
| document | database | UniversalReactAgent | DOCUMENT | DOCUMENT | ❌ |
| desktop | 无 | DesktopReactAgent | DESKTOP | DESKTOP | ❌ |

> **重要**: shell工具通过shell_register.py注册到SYSTEM，meta/time工具通过meta_register.py注册到SYSTEM。
> ToolCategory.SHELL和ToolCategory.META枚举值保留用于兼容性，但无工具直接注册到它们。
> INTENT_TO_CATEGORY已修正(2026-05-24): shell→SYSTEM, meta→SYSTEM, time→SYSTEM, code_execution→SYSTEM。

## 2. 配置注册表（agent_config.py）

### 2.1 AgentConfig数据类

```python
@dataclass
class AgentConfig:
    intent_type: str            # 意图类型标识（唯一主键）
    category: ToolCategory      # 对应的工具分类枚举
    prompt_module: str          # Prompt模块路径（importlib懒加载）
    prompt_class_name: str      # Prompt类名（importlib懒加载）
    category_display_name: str  # 分类中文名（用于system prompt）
    rollback_enabled: bool      # 是否启用回滚
    max_steps: int              # 最大步数
    aliases: List[str]          # 别名列表（旧名到新名的映射）
```

**关键行为**：`prompt_class` 属性使用 `importlib` 懒加载，首次访问时动态导入Prompt模块。

### 2.2 AGENT_REGISTRY注册表

全局字典 `AGENT_REGISTRY: Dict[str, AgentConfig]`，以 `intent_type` 为键存储5个配置项。

### 2.3 resolve_agent_config函数

```
输入: intent_type (如 "time" → 搜索别名列表 → 找到 system 配置)
输出: AgentConfig
逻辑: 先精确匹配intent_type，再搜索所有配置的aliases列表
异常: ValueError (未知意图)
```

### 2.4 get_all_intent_types函数

返回所有 `intent_type + aliases` 的扁平列表，用于意图分类器候选意图集合。

### 4.3 run_stream方法（ReAct循环核心）

**整体结构**：

```
run_stream(task, context, max_steps, task_id, running_tasks)
  │
  ├─ 初始化：清空steps，重置message_builder，设置状态
  ├─ _on_session_init(task, context) Hook
  ├─ 获取sys_prompt + task_prompt
  ├─ _on_before_loop Hook
  ├─ init_history(sys_prompt, task_prompt)
  │
  └─ try:
       while True:
         ├─ 🅰 步数检查 (step_count >= max_steps → ErrorStep, return)
         ├─ 🅱 中断检查 (running_tasks.cancelled → interrupted事件, return)
         ├─ LLM调用 (await _get_llm_response())
         ├─ 🅲 LLM返回后中断检查
         ├─ 🅳 空响应检查 (response为空 → 空响应重试逻辑)
         ├─ 🅴 解析响应 (parse_react_response)
         │
         ├─ 根据 parsed["type"] 分支:
         │   ├─ "chunk" → 🅵 chunk处理
         │   ├─ "answer"/"implicit" → 🅶 正常完成
         │   ├─ "thought_only" → 🅷 纯思考
         │   ├─ "parse_error" → 🅸 解析错误重试
         │   └─ "action" → 🅹 工具调用
         │
    except Exception → 🅺 未捕获异常
```

#### 🅰 步数检查 (场景3)

```
触发条件: step_count >= max_steps (每次循环开始时检查)
行为:
  1. StepFactory.create_error_step(type="max_steps_exceeded", recoverable=False)
  2. 追加到 self.steps
  3. yield error_step.to_dict()
  4. _on_after_loop()
  5. return
```

#### 🅱 中断检查（循环开始前）

```
触发条件: task_id and running_tasks 且 running_tasks[task_id].cancelled == True
行为:
  1. 计算中断延迟时间
  2. create_incident_data(incident_value="interrupted")
  3. yield interrupted_data
  4. _on_after_loop()
  5. return
```

#### 🅲 LLM返回后中断检查

```
触发条件: LLM调用返回后，检查cancelled标志
用途: ai_service.cancel()取消了HTTP请求，防止把取消当成可重试错误
行为: 同上 interrupted 事件
```

#### 🅳 空响应检查 (场景2)

```
触发条件: not response
流程:
  计数器 empty_response_retry_count +1
  ├─ ≤ max_empty_response_retries(2) 且 history长度 > 4:
  │   ├─ 保留 system(前2条) + 最近2条，删除中间
  │   ├─ yield retrying事件
  │   └─ continue（重试）
  └─ 超过重试次数 或 history太短:
      ├─ ErrorStep(type="empty_response")
      ├─ yield + _on_after_loop + return
```

#### 🅴 解析响应

```
parsed = parse_react_response(response)
tool_name = parsed.get("tool_name") or parsed.get("action_tool") or "finish"
           # 修正后：空字符串/None都降级为"finish"
tool_params = parsed.get("tool_params", parsed.get("params", {}))
```

#### 🅵 chunk处理

```
触发条件: parsed["type"] == "chunk"
流程:
  1. 重置 parse_retry_count = 0
  2. 拼接 chunk_buffer += chunk_content
  3. consecutive_chunk_count += 1
  4. 追加到 temp_history
  5. StepFactory.create_chunk_step → yield
  6. 【分支】无工具Agent (tool_category is None):
     → 第一个chunk即为最终回答，flush buffer → FinalStep → return
  7. 【分支】连续chunk达阈值 (consecutive_chunk_count >= max_consecutive_chunks):
     → 提升为implicit，flush buffer → FinalStep → return
  8. 【兜底】continue（继续下一轮循环）
```

**chunk的语义**：表示"还没完，后面还有"，Agent追加到buffer后继续循环。不是完成信号。

#### 🅶 answer/implicit正常完成 (场景5)

```
触发条件: parsed["type"] in ["answer", "implicit"]
流程:
  1. flush chunk_buffer到正式会话历史（如果存在）
  2. 提取 answer_response（取值链）:
     response → tool_params.result → content → reasoning
  3. StepFactory.create_final_step(step, response, thought, model, provider)
  4. 追加到 self.steps → yield
  5. _on_after_loop() → return
```

**response取值链优先级**：
1. `parsed.get("response", "")`
2. `parsed.get("tool_params", {}).get("result", "")`
3. `parsed.get("content", "")`
4. `parsed.get("reasoning", "")`

#### 🅷 thought_only处理

```
触发条件: parsed["type"] == "thought_only"
流程:
  1. thought去重拼接：thought与thought_content相同则不拼
  2. StepFactory.create_thought_step(...)
  3. append到self.steps → yield
  4. message_builder.add_assistant(response)
  5. trim_history()
  6. continue（继续下一轮循环）
```

#### 🅸 parse_error处理

```
触发条件: parsed["type"] == "parse_error"
流程:
  1. 获取 error_msg
  2. 【分支】网络错误 vs LLM格式错误:
     ├─ 网络错误 (429/rate_limit/timeout/502等): 不注入history，指数退避等待
     └─ LLM格式错误: add_observation(error_msg) 注入history
  3. parse_retry_count += 1
  4. 【分支】重试次数 >= 3:
     → ErrorStep(type="parse_error", recoverable=False) → yield → return
  5. 【兜底】yield retrying事件 → continue
```

**429限流退避**：`2.0 * (2 ** parse_retry_count)` 秒（指数退避）。

#### 🅹 Action工具调用

```
触发条件: parsed["type"] == "action" (兜底，最后一个类型判断)
流程:
  ┌─ 1. flush chunk_buffer到正式history
  ├─ 2. thought去重拼接
  ├─ 3. ThoughtStep → yield（先输出思考步骤）
  ├─ 4. 状态设置为 EXECUTING
  ├─ 5. 【工具执行前中断检查】
  ├─ 6. await _execute_tool(tool_name, tool_params)
  ├─ 7. 【工具执行后中断检查】
  ├─ 8. 构建 execution_result_dict
  ├─ 9. ActionToolStep → yield（输出工具执行结果）
  ├─ 10. message_builder.add_assistant(response)（LLM响应加入历史）
  ├─ 11. build_observation_text → add_observation（observation加入历史）
  ├─ 12. ObservationStep → yield（输出观察步骤）
  ├─ 13. [return_direct检查] is_done() == True → FinalStep → return
  ├─ 14. trim_history()
  └─ 15. 【并行工具调用】处理_pending_calls
       ├─ 遍历每个pending call
       ├─ step_count递增
       ├─ execute_tool → ActionToolStep → yield
       ├─ ObservationStep → yield
       └─ 并行observation加入history（FC协议或标准格式）
```

**并行工具_pending_calls**：
- 来源：`parse_react_response` 透传的 `_pending_calls` 字段（来自ToolsStrategy多工具调用或JSON list）
- 执行顺序：主工具完成后，逐个执行（非真正并行，是顺序执行）
- 成对输出：每个并行工具输出 `ActionToolStep + ObservationStep`
- FC协议支持：如果 `_strategy == "tools"`，使用 `assistant(tool_calls) + tool(tool_call_id)` 注入

**return_direct检查**（第13步）：
```
observation_step.is_done() == True → 工具要求直接返回
  ├─ 取出 execution_result.data
  ├─ StepFactory.create_final_step(message + data)
  ├─ yield → _on_after_loop → return
```

#### 🅺 未捕获异常处理 (场景1)

```
except Exception:
  1. traceback.print_exc()
```

## 5. Mixin体系

### 5.1 ReactAgentMixin（react_agent_mixin.py）

`class ReactAgentMixin(ToolLoaderMixin)` — 提供所有Agent子类共享的公用逻辑。

#### 5.1.1 _init_tools_and_executor

```
流程: ensure_tools_registered() → load_tools_by_category() → ToolExecutor
用途: 加载指定分类的工具到_tools_dict，创建ToolExecutor实例
```

#### 5.1.2 _init_llm_strategies

```
流程:
  1. 创建 TextStrategy 实例
  2. 检查 llm_client 和 api_base 是否可用
  3. 【分支】可用:
     ├─ 创建 LLMAdapter(api_base, api_key, model)
     ├─ tool_registry.to_openai_tools(category) → openai_tools
     ├─ 创建 ToolsStrategy(tools=openai_tools)
     ├─ use_function_calling = True
     └─ _strategy = None（首次调用时懒确定）
  4. 【分支】不可用:
     ├─ adapter = None, tools_strategy = None
     ├─ use_function_calling = False
     └─ _strategy = "text"（固定文本模式）
```

#### 5.1.3 _init_task_tracking

```
输入: enable (默认True)
启用时: get_task_tracker() → _task_tracker + _task_created_by_agent = False
禁用时: _task_tracker = None
```

#### 5.1.4 工具提示生成

| 方法 | 用途 | 生成内容 |
|------|------|---------|
| `_get_tools_summary` | 跨分类工具概要 | registry.get_all_tools_summary(排除已加载分类) |
| `_get_tools_detail` | 已加载分类完整描述 | 多个load_categories的detail拼接 |
| `_build_candidates_hint` | 候选意图提示 | 候选意图列表文本 |
| `_build_cross_tool_hint` | 跨分类提示 | 提示可使用其他分类工具 |

#### 5.1.5 _build_system_prompt

```
流程:
  1. 检查 _custom_system_prompt → 有则直接返回
  2. prompts.build_full_system_prompt() → 基类公共规则
  3. + _build_candidates_hint() + _build_cross_tool_hint(category_name)
```

#### 5.1.6 _call_llm（LLM调用统一入口）

```
流程:
  1. llm_call_count += 1
  2. message_builder.prepare_messages_for_llm()
  3. 【首次调用】adapter.detect_strategy() → 确定策略 ("text" / "tools")
  4. _inject_tools_hint(messages, strategy)
     ├─ tools策略: 不注入（FC协议自带工具定义）
     └─ text策略: 注入工具描述detail + 概要summary（含缓存）
  5. 【text策略】_inject_schema(messages) → 注入Schema文本参考
  6. _log_prompt(messages, strategy)
  7. _dispatch_strategy(strategy, messages)
     ├─ tools: tools_strategy.call(llm_client, messages)
     └─ text: text_strategy.call(llm_client, messages)
  8. _log_response(response)
  9. return response
```

**策略确定时机**：首次LLM调用时通过 `adapter.detect_strategy()` 确定，后续使用缓存。

**注入顺序**：工具信息在第一个非system消息前插入 → Schema文本在末尾追加。

### 5.2 RollbackMixin（rollback_mixin.py）

`class RollbackMixin` — 可插拔的回滚能力，仅 `UniversalReactAgent` 包含。

**rollback方法**：
```
输入: step_number (None=回滚全部, 指定值=回滚到该步之后)
分支1: step_number is None:
  → executor.execute('rollback_session', {'task_id': self.task_id})
分支2: 指定step_number:
  → 收集 steps 中 step_number > 目标值的步骤
  → 按逆序逐个执行 rollback_operation(operation_id)
异常: ValueError(task_id缺失)
状态: 回滚完成后设置 AgentStatus.ROLLED_BACK
```

## 7. ReAct输出解析器（react_output_parser.py）

### 7.1 整体架构

采用**解析器链（Chain of Responsibility）模式**：9个handler按优先级顺序尝试，每个返回解析结果或None（交给下一个handler）。

```
parse_react_response(output)
  │
  └─ 遍历 _HANDLERS:
      ├─ Handler #1: dict输入
      ├─ Handler #2: list输入
      ├─ Handler #3: JSON数组字符串
      ├─ Handler #4: 空值/非字符串
      ├─ Handler #5: 标准JSON (json.loads)
      ├─ Handler #6: 非标准JSON (单引号)
      ├─ Handler #7: 混合文本JSON块 + 不完整JSON检测
      ├─ Handler #8: 正则兜底
      └─ Handler #9: 关键词匹配（最终兜底，始终返回结果）
```

### 7.2 返回统一格式

所有handler返回统一格式字典：

```python
{
    "type": str,           # action / answer / implicit / thought_only / parse_error / chunk
    "thought": str,        # 思考内容
    "content": str,        # 内容文本（兼容性字段）
    "reasoning": str,      # 推理过程
    "tool_name": str|None, # 工具名（action类型有值）
    "tool_params": dict|None, # 工具参数
    "response": str|None,  # 最终响应
    "error": str|None,     # 错误信息（parse_error类型有值）
    "_pending_calls": list # [可选] 并行工具调用列表
}
```

### 7.3 各Handler详细分析

#### Handler #1: dict输入

```
触发: isinstance(output, dict)
入口: _handle_dict_input → _create_action_result_from_dict
```

`_create_action_result_from_dict` 内部流程：
```
1. 检查 data 有效性 → 无效返回 parse_error
2. 显式type优先判断:
   ├─ type="parse_error" → _build_parse_error_result
   ├─ type="answer" → _build_answer_result
   ├─ type="chunk" → _build_chunk_result
3. 旧格式兼容: action字段存在且tool_name不存在 → _build_action_from_old_format
4. tool_name判读:
   ├─ "finish" → answer类型，result标准化
   ├─ None/空 → 降级为implicit（防止_execute_tool(None)崩溃）
   └─ 其他 → action类型，_process_tool_params处理参数
```

**关键行为**：
- tool_name为None时降级为implicit（而非action），防止 `_execute_tool(None, None)` 崩溃
- finish的result通过 `_normalize_result_to_str` 标准化为字符串

#### Handler #2: list输入 + Handler #3: JSON数组字符串

```
触发: isinstance(output, list) 或 str以"["开头
入口: _handle_list_input → _create_action_result_from_list
```

`_create_action_result_from_list` 内部流程：
```
1. 空数组 → parse_error
2. 提取所有有效dict元素
3. 取最后一个有效dict
4. 【Function Calling格式检测】item有function字段无tool_name:
   → 全部转换（function.name → name, function.arguments → args）
   → 最后一个作为主工具，之前的作为_pending_calls
5. 普通list: 委托给 _create_action_result_from_dict(last_item)
```

#### Handler #4: 空值/非字符串

```
触发: not output or not isinstance(output, str)
结果: parse_error ("Empty or non-string response from LLM")
```

#### Handler #5: 标准JSON

```
触发: json.loads(output) 成功且结果为dict
入口: _handle_standard_json → _process_json_result

_process_json_result 共享逻辑:
1. 显式type优先: parse_error → answer → chunk
2. 新字段格式 (tool_name in data):
   ├─ finish → _build_action_from_new_format(answer类型)
   └─ 其他 → _build_action_from_new_format(action类型)
3. 旧字段格式 (action in data):
   → _build_action_from_old_format
4. 无匹配 → None（交给下一个handler）
```

**`_build_action_from_new_format` 内部**：
```
1. is_finish = tool_name == "finish"
2. finish: tool_params.result → _normalize_result_to_str → response
3. 非finish: _process_tool_params(normalize → filter → supplement)
4. 返回 {type, thought, content, reasoning, tool_name, tool_params, response}
5. 透传 _pending_calls
```

**`_build_action_from_old_format` 内部**：
同上，但使用 action/action_input 字段名。

#### Handler #6: 非标准JSON

```
触发: _try_parse_non_standard_json(output) 成功且为dict
入口: _handle_non_standard_json → _process_json_result(与标准JSON路径相同)
```

**`_try_parse_non_standard_json`**：处理单引号JSON（LLM常用 `'content'` 而非 `"content"`）。

#### Handler #7: 混合文本JSON块

```
触发: 无法整体解析为JSON，但文本中包含"{...}" JSON块
入口: _handle_mixed_text_json

流程:
1. _extract_json_block(output) → 使用平衡括号算法提取JSON
2. 【不完整JSON检测】无JSON块且以{"thought":"开头:
   ├─ 先尝试正则兜底提取tool调用
   ├─ 成功 → 正则兜底结果
   └─ 失败 → 返回chunk（不完整JSON作为流式片段）
3. 有JSON块:
   ├─ tool_name == "finish" → answer类型（result标准化）
   ├─ tool_name 有值 → action类型（_process_tool_params处理）
   ├─ 无tool_name但有content/reasoning → implicit类型
   └─ 无匹配 → None
```

**`_extract_json_block` 多层降级解析**：
```
1. 平衡括号算法提取JSON字符串
2. json.loads → 成功直接返回
3. 编码修复 (utf-8 errors='replace')
4. 中文引号修复 (\\u201c\\u201d → 普通字符)
5. 未转义换行符替换
6. 尾随逗号修复
7. 【最后降级】手动提取 tool_name/tool_params/content/reasoning
   - 使用平衡括号算法逐个字段提取
   - 支持嵌套JSON对象（tool_params）
   - 支持长文本content字段
```

#### Handler #8: 正则兜底

```
触发: 前面所有JSON路径都失败
入口: _handle_regex_fallback → _try_regex_tool_call_fallback

流程:
1. 正则提取 tool_name "([^"]+)"
2. 【分支】tool_name == "finish":
   → answer类型（提取tool_params.result）
3. 【分支】其他工具名:
   → 验证工具是否在registry中注册
   → 提取 tool_params
   → action类型（含parse_warning="regex_tool_fallback"）
```

#### Handler #9: 关键词匹配（最终兜底）

```
入口: _handle_keyword_match → _determine_parse_type

优先级:
1. ```包裹JSON解析 → 解析```内的JSON
2. 关键词匹配:
   ├─ Action位置 < Answer位置 → _parse_action
   ├─ Answer存在 → _parse_answer
   └─ Thought存在 → _parse_thought_only
3. 长度兜底:
   ├─ ≥5字符 → implicit（非流式路径的完整文本回答）
   └─ <5字符 → parse_error
```

### 7.4 工具参数处理管道（_process_tool_params）

统一处理路径，消除4条路径（标准JSON/非标准JSON/混合文本/dict输入）的不一致性：

```
_process_tool_params(tool_params, tool_name, raw_output)
  ├─ 1. 非dict → 直接返回
  ├─ 2. _normalize_tool_params_content(tool_params)
  │    将tool_params中的content/text字段标准化
  ├─ 3. _filter_tool_params(tool_params)
  │    移除非法参数名（不在input_schema中的）
  └─ 4. _supplement_missing_params(tool_name, params, raw_output)
       从原始文本补充缺失的必需参数
```

### 7.5 关键词映射表

```python
REACT_KEYWORDS = {
    "thought":   r"(?:Thought|思考|推理):\s*",
    "action":    r"(?:Action|行动|工具调用|(?:调用|使用|执行)\s+|(?:工具|函数)\s*为):\s*",
    "action_input": r"(?:Action Input|工具参数|输入|参数):\s*",
    "answer":    r"(?:Answer|回答|最终答案|结论):\s*",
}
```

```
      │
      ├── ThoughtStep(ToolMixin, ReasoningStep)
```

### 9.4 add_observation详细流程

```
5. 【分支】text协议:
   └─ append role=system(content=observation_text)
6. trim_history()
```

#### observation预算计算

```python
budget = OBSERVATION_BUDGET_MIN + OBSERVATION_BUDGET_DECAY * max(0, 5 - llm_call_count)
# OBSERVATION_BUDGET_MIN = 20000
# OBSERVATION_BUDGET_DECAY = 10000
# 第1轮: 20000 + 10000*4 = 60000
# 第5轮: 20000 + 10000*0 = 20000
# 第10轮: 20000 + 10000*0 = 20000
# 上限: min(budget, 50000)
```

#### 智能截断

```
预算=60000, 内容=100000字符 →
  head(60%) = 前36000字符
  tail(40%-50) = 后23950字符
  → "前36000... [中间省略40000字符] ...后23950"
```

### 9.5 历史裁剪（trim_history）

```
触发条件: 总字符 > MAX_CONTEXT_CHARS * 0.8
         (120000字符阈值，MAX_CONTEXT_CHARS=150000)
流程:
  1. 按role分类: system(非observation), observation, assistant
  2. observation去重（MD5指纹）
  3. assistant保留最近10条
  4. observation保留最近30条
  5. 从最旧observation开始移除，直到满足预算(MAX_CONTEXT_CHARS * 0.7 = 105000)
  6. FC协议配对裁剪（确保assistant(tool_calls)与tool配对）
  7. 确保至少保留2条（system + user）
```

## 10. LLM调用策略（llm_strategies.py）

### 10.1 策略体系

```
LLMStrategy(ABC)
  ├── TextStrategy    # 文本模式
  └── ToolsStrategy   # Function Calling模式
```

**策略确定时机**：首次LLM调用时通过 `adapter.detect_strategy()` 确定，后续使用 `self._strategy` 缓存。

### 10.2 TextStrategy

#### call方法流程

```
1. await llm_client(message="", history=messages)
2. 【情况0】空内容:
   ├─ 有错误信息 → _format_error_hint() → finish(result=错误提示)
   └─ 无错误信息 → finish(result="⚠️ AI服务返回空响应")
3. 【P1】parse_react_response(content):
   ├─ type=answer → finish(result=response)
   ├─ type=chunk → 返回chunk JSON数据（含_thinking推理内容）
   ├─ type=implicit → finish(result=content)
   ├─ type=thought_only/parse_error → 继续下一层解析
   ├─ type=action:
      ├─ tool_name有值且raw_tool_params不是None → 直接返回
      └─ tool_name或raw_tool_params缺失 → 继续下一层解析
4. 【方案B】_extract_by_known_tools(content):
   ├─ 从文本中查找已注册的工具名
   ├─ 找到 → 返回 action(tool_name, tool_params)
   └─ 未找到 → 继续
5. 【兜底】parse_error("无法从LLM响应中提取工具调用")
```

#### _extract_by_known_tools

```
从content中搜索所有已注册工具名（不区分大小写）
找到第一个匹配后：
1. 尝试提取路径参数（Windows路径C:\... / Unix路径/...）
2. 提取text/content/file_name/search_query/keyword参数
```

### 10.3 ToolsStrategy

#### call方法流程

```
1. 检查 llm_client.chat_with_tools 是否存在 → 不存在降级到TextStrategy
2. await llm_client.chat_with_tools(message="", history=messages, tools=self.tools)
3. 保存原始FC响应到 llm_client._last_chat_response
4. 【错误处理】response.error → parse_error
5. 【JSON解析】response.content:
   ├─ json.loads(content) 为list:
   │   ├─ 非空 → _format_tool_calls(tool_calls)
   │   └─ 空 → finish("任务完成")
   ├─ json.loads失败:
   │   ├─ _extract_json_block 成功 → 返回提取的JSON
   │   └─ 完全失败 → 返回原始content（交给下游解析器）
   └─ 非list → 返回原始content
6. 【空响应】降级到TextStrategy
```

#### _format_tool_calls

```
输入: tool_calls列表（OpenAI格式）
单个工具调用:
  → {thought, tool_name, tool_params}
多个工具调用:
  → 第一个作为主工具（tool_name/tool_params）
  → 其余作为 _pending_calls
```

## 11. 工具执行器（tool_executor.py）

### 11.1 ToolExecutor

**构造函数**：
- 接收 `tools: Dict[str, Callable]` 参数
- 未传入时从 `tool_registry` 获取所有工具实现

### 11.2 execute方法流程

```
输入: action(工具名), action_input(参数字典)
流程:
1. 【废弃检查】is_deprecated_tool(action) → 已废弃直接返回 ERR_TOOL_DEPRECATED
2. 【别名转换】get_tool_name_alias(action) → 有别名则使用主名
3. 【finish处理】action == "finish":
   → {code: SUCCESS, data: action_input.result, message: "Task completed"}
4. 【工具查找】action in available_tools:
   ├─ 不在本地映射:
   │   ├─ 本地映射为空 → ensure_tools_registered()后重试
   │   ├─ 从全局registry查找 → 找到则加入本地映射
   │   └─ 未找到 → ERR_TOOL_NOT_FOUND
   └─ 在本地映射 → _execute_with_retry(action, action_input)
```

### 11.3 错误分类与重试

#### ErrorClassifier

```
classify(error) → ErrorType:
  ├─ asyncio.TimeoutError → TIMEOUT
  ├─ PermissionError → PERMISSION_DENIED
  ├─ FileNotFoundError → FILE_NOT_FOUND
  ├─ ValueError → INVALID_PARAMS
  ├─ KeyError/TypeError/AttributeError → INVALID_PARAMS
  ├─ OSError → FILE_NOT_FOUND
  ├─ "not found" / "does not exist" → TOOL_NOT_FOUND
  └─ 其他 → UNKNOWN
```

#### _execute_with_retry

```
循环 attempt_count = 0 to retry_policy.max_retries:
  1. _normalize_params(action, action_input) → 参数规范化
  2. 验证必需参数（inspect.signature）→ 缺少参数返回 ERR_MISSING_PARAM
  3. 执行工具:
     ├─ iscoroutinefunction(tool) → await tool(**params)
     └─ 同步工具:
         ├─ asyncio.to_thread(lambda: tool(**params)) 在线程池执行
         ├─ 返回coroutine → await执行异步逻辑
         └─ 返回普通值 → 直接返回
  4. 超时保护: asyncio.wait_for(tool, timeout=get_timeout(action))
  5. 异常捕获:
     ├─ 不可重试 → 立即返回 error 结果
     ├─ 可重试但超过最大次数 → 返回 error 结果
     └─ 可重试 → 指数退避等待后重试
```

**工具执行的三种路径**：
| 工具类型 | 检测方式 | 执行方式 |
|---------|---------|---------|
| 纯async def | `iscoroutinefunction=True` | 直接 `await` |
| lambda包装async | `iscoroutinefunction=False`，但返回coroutine | `to_thread`执行lambda，再`await` coroutine |
| 纯同步 | `iscoroutinefunction=False`，非coroutine返回值 | `to_thread`在线程池执行 |

### 11.4 参数规范化（_normalize_params）

```
从 tool_registry 获取 input_schema → 提取 valid_params 集合
删除不在 valid_params 中的参数（防止 unexpected keyword argument 错误）
```

## 12. 工具结果格式化（tool_result_formatter.py）

### 12.1 两条输出路径

| 方法 | 输出 | 消费方 |
|------|------|--------|
| `_format_llm_observation(result)` | 文本字符串 | LLM observation消息 |
| `_format_frontend_event(result)` | 字典 | 前端SSE事件 |

### 12.2 extract_status

```
输入: result dict
code == "SUCCESS":
  → "success" (无warning) 或 "warning" (有warning)
code.startswith("WARNING_"):
  → "warning"
其他:
  → "error"
```

### 12.3 _format_llm_observation

```
【SUCCESS】:
  text = "Observation: success - {message}"
  + warning文本 (如果有)
  + json.dumps(llm_data or data) (含_safe_truncate防OOM)
  + next_actions格式化 (最多前5条)
【WARNING_*】:
  text = "Observation: warning - {message}"
  + 部分数据
  + next_actions
【ERR_*】:
  text = "Observation: error [{code}] - {message}"
```

### 12.4 _format_frontend_event

```python
{
    "code": result.code,
    "message": result.message,
    "data": result.data,
    "retry_count": result.retry_count,
    "return_direct": result.return_direct,
    # 可选: warning, next_actions, attachment
}
```

### 12.5 _safe_truncate

仅防 `json.dumps` OOM，非业务截断。阈值为100000元素（dict取前100000键，list取前100000项）。

## 14. 完整数据流分析

### 14.1 一次Agent运行的全数据流

```
用户请求 → intent_classifier(CRSS) → intent_type
  │
  ▼
AgentFactory.create(intent_type)
  ├─ resolve_agent_config(intent_type) → AgentConfig
  └─ 创建 Agent 子类（UniversalReactAgent / DesktopReactAgent）
      ├─ __init__: tools → executor → strategies → prompts
      │
      ▼
  agent.run_stream(task, context)
      │
      ├─ init_history(sys_prompt, task_prompt)
      │   conversation_history = [
      │     {role: "system", content: sys_prompt},
      │     {role: "user", content: task_prompt}
      │   ]
      │
      ├─ LLM call 第1轮:
      │   ├─ _call_llm()
      │   │   ├─ prepare_messages → inject_tools → inject_schema
      │   │   ├─ detect_strategy() (首次) → "text" 或 "tools"
      │   │   ├─ TextStrategy.call / ToolsStrategy.call
      │   │   └─ return response (JSON string)
      │   │
      │   └─ parse_react_response(response)
      │       └─ handler chain → 统一结果字典
      │           └─ type = "action", tool_name = "list_directory"
      │
      ├─ Step 1: ThoughtStep → yield({type:"thought", ...})
      │
      ├─ ToolExecutor.execute("list_directory", {dir_path: "C:\\"})
      │   ├─ alias check → normalize params → validate params
      │   ├─ await tool(**params) → return result dict
      │   └─ {code: SUCCESS, data: [...], message: "成功列出目录"}
      │
      ├─ Step 2: ActionToolStep → yield({type:"action_tool", ...})
      ├─ add_assistant(response) → history追加
      ├─ build_observation_text(result) → observation文本
      ├─ add_observation(observation) → history追加（含截断）
      ├─ Step 3: ObservationStep → yield({type:"observation", ...})
      │
      ├─ [return_direct=False → 继续循环]
      │
      ├─ LLM call 第2轮: (同上)
      │   ├─ 现在 history 有 [sys, user, assistant, system(obs)]
      │   ├─ parse → type="answer", tool_name="finish"
      │   └─ FinalStep → yield({type:"final", response:..., ...})
      │
      └─ _on_after_loop() → return
```

### 14.2 SSE事件流

```
时间 →
  ┌────────┐ ┌────────┐ ┌──────────┐ ┌────────────┐ ┌───────┐
  │ thought │ │action_tool│ │observation│ │   ...     │ │ final │
  │ step=1  │ │ step=2  │ │  step=2  │ │  循环...   │ │step=N │
  └────────┘ └────────┘ └──────────┘ └────────────┘ └───────┘
```

每个Step通过 `yield step.to_dict()` 发送SSE事件，类型字段 `type` 标识事件类型，`step` 标识步骤编号。

### 14.3 工具执行结果在Agent内的流转

```
_execute_tool(action, params)
  │
  ▼
{code, data, message, retry_count, warning, attachment, next_actions, return_direct}
  │
  ├─ extract_status() → status (success/warning/error)
  ├─ execution_result_dict (含status/summary/data等)
  ├─ ActionToolStep(to_dict) → 前端action_tool事件
  │
  ├─ build_observation_text() → _format_llm_observation()
  │   └─ LLM observation文本
  │
  ├─ add_observation(observation) → history
  │
  └─ ObservationStep(to_dict) → 前端observation事件
```
