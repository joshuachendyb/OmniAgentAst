## 一、扫描方法论

| 层次 | 扫描范围 | 扫描深度 | 目标 |

## 四、6大功能块"单一功能分散"深度分析

### 4.1 功能A：消息/会话保存到数据库 🔴严重

**一句话描述**: 一次聊天中，用户消息和AI消息保存到SQLite的完整路径

**理想实现**: 一个`MessageRepository`类，`save_user_message()`和`save_assistant_message()`两个方法，统一管理DB连接、ID分配、字段更新。

**实际分散情况**:

| 实现点 | 文件 | 行号 | 做了什么 |
|--------|------|------|---------|
| 点1 | `react_sse_wrapper.py` | 43,114,133,426,502,549,607 | **7个位置**直接调save_execution_steps_to_db() |
| 点2 | `chat_stream/message_saver.py` | 38-92 | `save_execution_steps_to_db()`——中间层 |
| 点3 | `chat_stream/message_saver.py` | 95-121 | `add_step_and_save()`——另一个入口 |
| 点4 | `chat_stream/message_saver.py` | 129-154 | `create_add_step_and_save()`——工厂闭包 |
| 点5 | `api/v1/sessions.py` | 490-546 | `save_message()`——FastAPI路由，直接SQL INSERT |
| 点6 | `api/v1/sessions.py` | 549-594 | `AssistantMessageIdAllocator`——ID分配逻辑 |
| 点7 | `api/v1/sessions.py` | 630-671 | `_insert_assistant_message()` + `_update_message_fields()` |
| 点8 | `api/v1/sessions.py` | 713-729 | `save_execution_steps()`——UPSERT逻辑 |
| 点9 | `api/v1/sessions.py` | 13-16 | `_user_message_ids`/`_assistant_message_ids`——模块级字典缓存 |
| 点10 | `db/chat_db.py` | 11-24 | `get_connection()`——DB连接 |
| 点11 | `db/models/chat_models.py` | 1-67 | Pydantic模型定义 |

**七绕八绕路径**: SSE流中每次产生step → 手动append到list → message_saver.save_execution_steps_to_db() → 做无效session检查 → 跨模块调sessions.save_execution_steps()(**FastAPI路由函数**) → 创建Allocator → 分配ID(查缓存→查DB) → insert/update → 更新message_count。用户消息走**完全不同路径**：前端POST → sessions.save_message()路由 → 直接SQL INSERT。两条路径靠模块级字典`_user_message_ids`关联，**没有事务保证**。

**历史BUG风险**: "database is locked"并发BUG、ID缓存不一致导致消息错位、7个保存点散布易漏保存/重复保存、跨模块调路由函数（非标准用法）

## 五、跨文件功能分散分析（功能完整性Cohesion）

### 5.1 审查背景

上文的 5 原则分析是基于**结构视角**（一个类/文件/函数做了什么）。在用户要求下，补做一次**功能视角**的分析：**一个业务逻辑功能，跨了多少个文件？**

> 核心判断标准：如果改一个功能需要读 N 个文件才能理解完整逻辑，那就违反了 SRP 的意图——不是因为一个类做太多，而是因为一个事分散太多。

### 5.2 追踪方法

4 路并行追踪 4 个核心业务功能的数据流：

| 追踪路 | 功能 | 追踪工具 |
|--------|------|---------|
| 路1 | LLM 响应被解析并路由到正确的处理分支 | `grep` + `read` 沿调用链追溯 |
| 路2 | 工具执行：解析出 `action_tool` 后到底发生了什么 | `grep` + `read` 沿 `_execute_tool_step` |
| 路3 | 错误处理：一个 LLM HTTP 错误如何传播到前端 | `grep` + `read` 沿异常栈 |
| 路4 | SSE 事件生成：一个 agent step 如何变成前端 SSE 事件 | `grep` + `read` 沿 yield 链 |

### 5.3 审查结果总览

| 功能 | 涉及文件数 | 涉及函数数 | 核心文件 | 风险等级 |
|------|-----------|-----------|---------|---------|
| **LLM 响应解析链** | 7 个 | ~50+ | `base_react.py` → `react_output_parser.py` → `tool_step_mixin.py` → `react_sse_wrapper.py` | ⚠️ 高 |
| **工具执行链** | 9 个 | ~20+ | `base_react.py` → `tool_step_mixin.py` → `registry.py` → 各 tool impl → 11 个分类文件 | ⚠️ 高 |
| **错误处理链** | 8 个 | ~10+ | `llm_core.py` → `llm_strategies.py` → `react_agent_mixin.py` → `base_react.py` → `react_sse_wrapper.py` → `chat_router.py` | ⚠️ 高 |
| **SSE 事件生成链** | 6 个 | ~15+ | `chat_router.py` → `react_sse_wrapper.py` → `base_react.py` → `tool_step_mixin.py` → `chunk_buffer.py` | ⚠️ 中 |

### 5.5 路2：错误处理链 — 8 文件分散

#### 5.5.1 数据流

| # | 文件 | 在错误链中的角色 | 捕获的错误类型 |
|---|------|----------------|---------------|
| 1 | `llm_core.py` | HTTP 异常抛出：429/500/503/连接超时 | `httpx.HTTPStatusError`, `httpx.TimeoutException` |

从**功能完整性 Cohesion**角度看：

1. **修改任意一个业务功能，需要读 6-9 个文件**才能理解完整逻辑
2. 核心瓶颈不是单个文件大（虽然 `react_output_parser.py` 2274 行确实大），而是**每个文件都沾了一点点不相干职责**

## 六、功能分散根因分析

### 6.1 三个核心模式（都是Bug的温床）

| 模式 | 描述 | 涉及的Bug |
|------|------|-----------|
| **双重解析链** | LLM输出被TextStrategy解析一次，又被react_output_parser解析第二次 | 空工具名死循环、FC格式JSON未解析 |
| **双重重试/双重参数归一化** | tool_executor和retry_policy各自实现重试循环 | 错误分类不一致、循环依赖 |
| **分散的持久化点** | SSE事件在5个不同位置手动append+save | current_content变量覆盖Bug |

### 6.2 为什么重构后Bug更多？

```
多轮重构的方式:
  第1轮: 拆分大函数 → 提取辅助函数，但保留原调用模式
  第2轮: 增加新功能 → 在辅助函数旁边加新逻辑，不做整合
  第3轮: 修改返回格式 → 只改一处，其他调用点加兼容层
  ...

结果:
  - 新旧桥接层膨胀 (file_tools 8个转发方法)
  - 统一化半途而废 (17处result dict各有不同默认值)
  - 每次重构都可能引入覆盖变量的Bug (current_content)
```

### 6.3 完整根因链（从功能分散→重构累积→测试失守）

```
根因1: 功能分散
  ├── LLM解析: 3套独立实现分布在5个文件
  ├── 工具执行: 11个文件5层调用
  ├── SSE持久化: 5处手动append+save
  ├── 意图定义: 2套独立关键词集合
```
