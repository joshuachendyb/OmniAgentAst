## 一、气泡内部结构（HTML 渲染顺序）

要理解位置变化，首先要看代码里这三个部分是谁先谁后：

```html
<div class="message-bubble">
   <!-- 第 1 块：步骤列表 (Start, Thought, Action_tool) -->
   <StepsList />

   <!-- 第 2 块：说话内容 (Chunks) -->
   <ChunkContent />

   <!-- 第 3 块：思考提示 (就是用户问的这个) -->
   <ThinkingMessage>🤔 AI 正在思考... ▌</ThinkingMessage>
</div>
```

**关键点**：因为它排在**最后**（第 3 块），所以一旦前面有了内容，它就会被**挤到最下面**。

---

## 二、核心逻辑：IIFE 开关

这个 "🤔 AI 正在思考... ▌" 的生死，完全由一个 **IIFE（立即执行函数表达式）** 控制。

```tsx
{(() => {
  // 1. 检查是否有 action_tool
  for (const step of executionSteps) {
    if (step.type === 'action_tool') { hasAction = 1; break; }
  }
  
  // 2. 如果有 action_tool，直接返回 false（不渲染）
  if (hasAction === 1) return false;
  
  // 3. 流式模式：只有当没有 chunk 时才显示
  if (message.isStreaming) {
    return chunks.length === 0;
  }
})() && (
  <div>🤔 AI 正在思考... ▌</div>
)}
```

**简单说**：它就是一个"开关计算器"，算出 `true` 就显示，算出 `false` 就消失。

## 三、普通对话模式（Chunk 触发消失）

在普通模式下，AI 直接输出文字，**Chunk 一出现，它就消失了**。

### 数据流与视觉变化

```text
[1. Start 到达]
    │
    ├─ 数据变化：executionSteps = [start]
    │
    ├─ 计算：chunks = [] (长度 = 0)
    │
    ├─ IIFE 判断：
    │     if (message.isStreaming) {
    │        return chunks.length === 0;  // 0 === 0 -> TRUE
    │     }
    │
    └─ 视觉结果：渲染 DIV
          ┌──────────────────────────────┐
          │                              │
          │  🤔 AI 正在思考... ▌          │  <-- 显示在最上面
          │                              │
          └──────────────────────────────┘


          ↓ (AI 开始说话，Chunk 到达)


[2. Chunk 到达]
    │
    ├─ 数据变化：executionSteps = [start, chunk]
    │
    ├─ 计算：chunks = [chunk] (长度 = 1)
    │
    ├─ IIFE 判断：
    │     if (message.isStreaming) {
    │        return chunks.length === 0;  // 1 === 0 -> FALSE
    │     }
    │
    └─ 视觉结果：不渲染 DIV
          ┌──────────────────────────────┐
          │  实际说话内容...              │  <-- Chunk 内容在上面
          │                              │
          │  (DIV 已移除，字彻底没了)     │  <-- 消失
          └──────────────────────────────┘
```

## 四、ReAct 模式（Action_tool 触发消失）

在 ReAct 模式下，AI 不直接吐字，而是先思考、再调工具。**Thought 来了它还在，直到 Action_tool 来了才消失。**

### 数据流与视觉变化

```text
[1. Start 到达]
    │
    ├─ 数据变化：executionSteps = [start]
    │
    ├─ 计算：hasAction = 0 (无 action_tool)
    │
    ├─ IIFE 判断：
    │     if (hasAction !== 1) {  // 0 !== 1 -> TRUE
    │        if (message.isStreaming) {
    │           return chunks.length === 0;  // 0 === 0 -> TRUE
    │        }
    │     }
    │
    └─ 视觉结果：渲染 DIV
          ┌──────────────────────────────┐
          │                              │
          │  🤔 AI 正在思考... ▌          │  <-- 显示在最上面
          │                              │
          └──────────────────────────────┘


          ↓ (AI 正在思考，Thought 到达)


[2. Thought 到达]
    │
    ├─ 数据变化：executionSteps = [start, thought]
    │
    ├─ 计算：hasAction = 0 (thought 不是 action_tool)
    │
    ├─ IIFE 判断：
    │     if (hasAction !== 1) {  // 0 !== 1 -> TRUE
    │        if (message.isStreaming) {
    │           return chunks.length === 0;  // 0 === 0 -> TRUE
    │        }
    │     }
    │
    └─ 视觉结果：继续渲染 DIV（但被挤到底部）
          ┌──────────────────────────────┐
          │  💭 思考过程：正在分析...      │  <-- 新内容在上面
          │                              │
          │  🤔 AI 正在思考... ▌          │  <-- 被挤到最底部
          └──────────────────────────────┘


          ↓ (AI 决定调用工具，Action_tool 到达)


[3. Action_tool 到达]
    │
    ├─ 数据变化：executionSteps = [start, thought, action_tool]
    │
    ├─ 计算：hasAction = 1 (发现 action_tool!)
    │
    ├─ IIFE 判断：
    │     if (hasAction !== 1) {  // 1 !== 1 -> FALSE
    │        ... (逻辑不执行)
    │     }
    │     return false;  // 直接返回 FALSE
    │
    └─ 视觉结果：不渲染 DIV
          ┌──────────────────────────────┐
          │  💭 思考过程：正在分析...      │
          │  🔧 执行工具：search_web     │  <-- 内容继续增加
          │                              │
          │  (这里空了，字彻底没了)       │  <-- 消失！
          └──────────────────────────────┘
```

## 五、为什么位置会变化？

### 原因分析

| 阶段 | 位置 | 原因 |
| :--- | :--- | :--- |
| **Start** | 最上面 | 此时只有它一个内容 |
| **Chunk/Thought** | 最底部 | 被新内容**挤**下去了（代码排在最后） |
| **Action_tool** | **消失** | IIFE 返回 `false`，整个 DIV 被移除 |

### 通俗解释

1.  **为什么跑到底部？**
    *   因为它在代码里写在最后面。
    *   只要上面有内容（思考过程或说话内容），它就会自然地被**挤**到最底下。

2.  **为什么最后消失了？**
    *   因为代码里的"开关"（IIFE 函数）算出了 `false`。
    *   就像电灯开关被关掉一样，整个灯泡（DIV）从页面上拿掉了。

## 六、核心逻辑总结表

| 步骤 | 关键变量 | 判断条件 | 结果 | 视觉效果 |
| :--- | :--- | :--- | :--- | :--- |
| **Start** | `chunks.length = 0` | `0 === 0` | **显示** | 在最上面 |
| **Chunk** | `chunks.length = 1` | `1 === 0` | **消失** | 被移除 |
| **Thought** | `hasAction = 0` | `0 !== 1` | **继续显示** | 被挤到底部 |
| **Action_tool** | `hasAction = 1` | `1 !== 1` | **消失** | 被移除 |

| 2 | start 显示完成 | start | 🤔 | AI 正在思考 | 00:00 ▌ |

### 7.3 架构设计

#### 7.3.1 核心原则：混合设计

**状态来源**：从 `executionSteps` 派生，不单独维护 state（避免不同步）
**计时器**：独立 state，监听 step type 变化自动重置

```
executionSteps (唯一数据源)
       ↓
  派生 currentStatus
       ↓
  查询 statusConfig 映射表
       ↓
  渲染：图标 + 文字 + 计时器 + ▌
```

#### 7.3.2 数据结构

```tsx
// 状态配置表（完整版）
const statusConfig: Record<string, { icon: string; text: string; animate: boolean }> = {
  waiting:     { icon: '🚀', text: 'AI开始执行任务', animate: true },
  start:       { icon: '🤔', text: 'AI 正在思考', animate: true },
  thought:     { icon: '🛠️', text: 'Agent 正在执行"action_tool"', animate: true },
  action_tool: { icon: '👁️', text: 'Agent 正在执行"observation"', animate: true },
  observation: { icon: '🤔', text: 'AI 正在思考', animate: true },
  chunk:       { icon: '💬', text: 'AI 正在回复', animate: true },
  final:       { icon: '✅', text: 'AI任务执行完成', animate: false },
  error:       { icon: '✅', text: 'AI任务执行完成', animate: false },
  disconnected:{ icon: '⚠️', text: '连接已中断', animate: false },
};
```

#### 7.3.3 状态派生逻辑（完整版）

```tsx
// 从 executionSteps 派生当前状态
const lastStep = executionSteps[executionSteps.length - 1];
const hasChunk = executionSteps.some(s => s.type === 'chunk');

// 判断逻辑（优先级从高到低）：
// 1. 如果有 final/error → final
// 2. 如果 SSE 断线 → disconnected
// 3. 如果有 chunk → chunk（普通对话模式）
// 4. 否则取最后一个 step 的 type
// 5. 如果没有 steps → waiting
const currentStatus = !lastStep 
  ? 'waiting' 
  : lastStep.type === 'final' || lastStep.type === 'error'
    ? 'final'
    : hasChunk 
      ? 'chunk'  // 普通对话模式：AI 正在回复
      : isDisconnected  // 从 useSSE hook 获取断线状态
        ? 'disconnected'
        : lastStep.type;
```

#### 7.3.4 状态隔离设计

**重要**：每条消息独立维护自己的状态和计时器。

```tsx
// MessageItem.tsx - 每个消息实例独立的状态
const MessageItem = ({ message }: MessageItemProps) => {
  // 从当前消息的 executionSteps 派生状态
  const executionSteps = message.executionSteps || [];
  const [elapsedSeconds, setElapsedSeconds] = useState(0);
  
  // 派生逻辑只依赖当前消息的数据
  const lastStep = executionSteps[executionSteps.length - 1];
  const currentStatus = deriveStatus(lastStep, executionSteps);
  
  // ...
};
```

**为什么不会互相影响？**
- 每条消息有自己的 `executionSteps` 数组
- 每条消息有自己的 `elapsedSeconds` state
- 状态派生是纯函数，不依赖全局变量

### 7.4 计时器设计

#### 7.4.1 独立计时原理

```tsx
// 计时器 state（每个消息实例独立）
const [elapsedSeconds, setElapsedSeconds] = useState(0);

// 监听 currentStatus 变化，重置计时器
useEffect(() => {
  // 完成状态和断线状态不计时
  if (currentStatus === 'final' || currentStatus === 'disconnected') {
    return;
  }
  
  setElapsedSeconds(0);  // 切换状态时重置
  const timer = setInterval(() => {
    setElapsedSeconds(s => s + 1);
  }, 1000);
  
  // ✅ 组件卸载时清理计时器（防止内存泄漏）
  return () => clearInterval(timer);
}, [currentStatus]);  // 依赖 currentStatus
```

#### 7.4.2 SSE 断线处理

```tsx
// 从 useSSE hook 获取断线状态
const { isDisconnected } = useSSE(...);

// 断线时停止计时，显示警告
useEffect(() => {
  if (isDisconnected) {
    setCurrentStatus('disconnected');
    // 计时器自动停止（因为 currentStatus 变化）
  }
}, [isDisconnected]);
```

#### 7.4.2 格式化函数

```tsx
const formatTime = (seconds: number): string => {
  const mins = Math.floor(seconds / 60);
  const secs = seconds % 60;
  return `${String(mins).padStart(2, '0')}:${String(secs).padStart(2, '0')}`;
};
// 输出示例：00:05, 01:23, 12:05
```

### 7.6 与现有 IIFE 逻辑的集成方案

**现有逻辑**：IIFE 控制 "🤔 AI 正在思考... ▌" 的显示。
**新逻辑**：替换为动态状态提示。

```tsx
// MessageItem.tsx - 修改后的渲染逻辑
{(() => {
  // 1. 原有逻辑：判断是否应该显示底部状态提示
  const chunks = message.executionSteps?.filter(s => s.type === "chunk") || [];
  
  // 计算 hasAction（ReAct 模式判断）
  let hasAction = 0;
  for (const step of (message.executionSteps || [])) {
    if (step.type === 'action_tool') { hasAction = 1; break; }
    if (step.type === 'chunk') { hasAction = 0; }
  }
  
  // 原有判断：有 action_tool 时不显示
  if (hasAction === 1) return false;
  
  // 流式模式：始终显示（由 statusConfig 控制内容）
  if (message.isStreaming) return true;
  
  // 非流式：有 final 时显示完成
  if (message.executionSteps?.some(s => s.type === 'final')) return true;
  
  return false;
})() && (
  <div className="status-display">
    <DynamicStatusDisplay 
      executionSteps={message.executionSteps || []}
      isStreaming={message.isStreaming}
      isDisconnected={isDisconnected}
    />
  </div>
)}
```

**关键变化**：
- 原来：固定显示 "🤔 AI 正在思考... ▌"
- 现在：调用 `DynamicStatusDisplay` 组件，根据 step type 动态显示

### 7.8 边界情况处理

| 场景 | 处理方式 | 说明 |
|------|---------|------|
| **组件卸载** | `clearInterval(timer)` | 防止内存泄漏 |
| **SSE 断线** | 显示"⚠️ 连接已中断"，停止计时 | 避免无意义计时 |
| **多轮对话** | 每条消息独立 state | 状态不互相影响 |
| **加载历史消息** | 从 executionSteps 自动派生 | 显示正确状态 |
| **chunk 到达** | 显示"💬 AI 正在回复" | 普通对话模式 |
| **final/error** | 显示"✅ AI任务执行完成"，无动画 | 任务结束 |

### 7.9 优势分析

| 特性 | 方案 A（纯派生） | 方案 B（独立 state） | 混合设计（采用） |
|------|----------------|-------------------|----------------|
| 状态同步 | ✅ 永远同步 | ❌ 可能不同步 | ✅ 永远同步 |
| 计时器 | ❌ 难以实现 | ✅ 容易实现 | ✅ 容易实现 |
| 代码复杂度 | 中 | 低 | 中 |
| 可维护性 | 中 | 低 | ✅ 高 |
| 扩展性 | 中 | 高 | ✅ 高 |

**混合设计的核心优势**：
1. **状态不会不同步**：永远从 executionSteps 计算，不依赖手动更新
2. **计时器独立管理**：切换状态自动重置，逻辑清晰
3. **易于扩展**：新增 step type 只需改映射表，不用改核心逻辑
4. **支持历史消息**：加载历史消息时自动计算正确状态
