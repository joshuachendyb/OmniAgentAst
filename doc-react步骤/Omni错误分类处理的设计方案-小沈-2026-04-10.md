## 一、问题背景

### 1.1 为什么要区分错误类型

当前后端代码的错误处理存在以下问题：

1. **错误处理不统一**：分散在多处调用 `create_error_response()` + `create_error_step()`
2. **没有区分错误场景**：工具执行错误和会话级错误混为一谈
3. **杀死LLM继续的机会**：工具执行错误后直接yield error，相当于"杀死"了LLM继续思考的机会

**说明**：本设计方案基于第2、3、4章的设计要求，用于指导后续代码重构。第7、8、9章是具体的修改说明。

---

### 1.2 统一错误处理函数设计

为解决上述问题，设计了两个统一的错误处理函数，分别用于处理不同场景的错误：

#### 1.2.1 create_session_error_result()——会话级的错误信息进行统一的错误信息的标准的组装

**功能**：统一的会话级错误处理函数 - 接收原始错误信息，分类组装成用户友好的错误消息，然后创建 error_response（yield给前端）+ error_step（保存到数据库）

**核心逻辑**：
```
原始错误（异常/错误码）
    ↓
1. resolve_http_error_type() - 解析HTTP错误码（401/403/429/500/503/504等）
2. get_stream_error_info() - 根据错误类型获取用户友好的错误信息
3. create_error_response() - 创建SSE格式的错误响应（yield给前端）
4. create_error_step() - 创建错误步骤字典（保存到数据库）
```


**使用场景**：
| 场景 | 说明 |
|------|------|
| API超时耗尽 | 聊天API重试3次后仍然超时 |
| 认证/权限失败 | API key无效、权限不足（401/403） |
| 安全拦截 | 危险操作被安全模块拦截 |
| API完全不可用 | 服务器返回429/500/502/503/504等 |
| LLM返回空响应 | AI服务返回空内容 |
| 超过最大步数 | 达到迭代次数上限 |
| 未捕获异常 | 程序运行时的未知异常 |
| 功能未实现 | 调用的功能模块尚未实现 |

**特点**：
- 发生错误后就是**最后一步**
- yield给前端 + 保存到DB + 结束会话
- 不需要LLM再做判断

#### 1.2.3 yield erorr 过程说明

**重要说明**：error_handler.py 中的函数只负责**生成错误数据**，**yield 操作在 base_react.py 中执行**。

```
错误发生 → break 退出循环
    ↓
base_react.py 调用错误处理函数
    ↓
error_handler.py:
  - create_session_error_result() → 返回 (error_response, error_step) 元组
  - create_tool_error_result() → 返回 错误信息的dic；【在 action tool的字段里面进行统一的yield】
    ↓
base_react.py 执行 yield:
  - yield error_response  ← 字符串（SSE格式）
  - 或 yield action_tool_result  ← 字典
```

**设计优点**：
1. **职责分离**：error_handler.py 专注生成数据，base_react.py 专注 yield
2. **逻辑清晰**：循环内只做 break，循环外统一处理 yield
3. **统一入口**：所有错误都通过错误处理函数分类组装，保证错误信息一致性

---

#### 1.2.3 create_error_from_exception()——专门处理未捕获异常的函数（create_session_error_result的特例）

**功能**：统一的异常错误处理函数 - 接收 Exception 对象，解析异常获取错误信息，然后创建 error_response + error_step

**与 create_session_error_result() 的关系**：
- `create_error_from_exception()` 是 `create_session_error_result()` 的**特例**
- 专门接收 `Exception` 对象，内部调用 `get_function_call_error_info()` 解析异常
- 返回格式相同：(error_response, error_step) 元组
- 使用场景：`try...except Exception` 捕获的未知异常

**核心逻辑**：
```
Exception 对象
    ↓
base_react.py 调用 create_error_from_exception()
    ↓
error_handler.py 内部处理流程:
    1. get_function_call_error_info() - 解析异常，获取 code/message/error_type/retryable
    2. create_error_response() - 创建SSE格式的错误响应字符串
    3. create_error_step() - 创建错误步骤字典（保存到数据库）
    ↓
返回 (error_response, error_step) 元组
    ↓
base_react.py 执行: yield error_response  ← 字符串（SSE格式）
```

**使用场景**：
| 场景 | 说明 |
|------|------|
| 未捕获异常 | try...except Exception 捕获的未知错误 |

**特点**：
- 与 create_session_error_result() 使用场景不同，但返回格式相同
- 专门用于处理程序运行时的未知异常

---

#### 1.2.4 create_tool_error_result()-工具执行错误的信息的统一编制信息的服务函数

**功能**：统一的工具级错误处理函数 - 接收工具名、错误消息等参数，复用现有 action_tool 字段格式，生成可直接 yield 的错误字典

**核心逻辑**：
```
工具执行失败
    ↓
1. 根据 retry_count 判断是否可重试
2. 构建错误摘要（包含重试信息）
3. 复用 action_tool 字段格式
4. 返回可直接 yield 的 dict
```

**使用场景**：
| 场景 | 说明 |
|------|------|
| 文件操作失败 | 读取/写入文件失败，文件不存在等 |
| 网络请求失败 | HTTP请求超时、连接失败等 |
| 命令执行错误 | shell命令返回非0退出码 |
| 工具参数错误 | 工具参数不合法 |

**特点**：
- 发生错误 **≠** 最后一步
- LLM会收到这个作为 **observation**
- LLM决定：**继续循环** or **进入final**
- 复用 `format_action_tool_sse()` 的字段格式：
  - `execution_status='error'` - 标记为错误
  - `summary` - 放错误消息
  - `raw_data` - 放详细错误信息

---

#### create_session_error_result与create_tool_error_result两个函数的区别

| 对比项 | create_session_error_result() | create_tool_error_result() |
|--------|-------------------------------|-----------------------------|
| **返回格式** | (error_response, error_step) 元组 | 直接返回 dict |
| **yield方式** | `yield error_response`（字符串）| `yield action_tool_result`（字典）|
| **错误来源** | API/认证/安全/LLM层面的错误 | 工具执行层面的错误 |
| **后续处理** | 结束会话，不需要LLM判断 | 传给LLM，LLM决定是否继续 |
| **数据保存** | 保存 error_step 到数据库 | 无需单独保存（复用action_tool）|

## 二、错误分类设计

### 2.1 两种错误场景

```
错误发生后的两种情况：

┌─────────────────────────────────────────────────────────────┐
│ 场景A：会话级错误（Session-Level Error）                     │
├─────────────────────────────────────────────────────────────┤
│ - API超时耗尽、认证失败、API完全不可用                       │
│ - 发生错误后就是最后一步了                                   │
│ - 需要：yield error → 结束会话 → 保存DB                     │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ 场景B：工具级错误（Tool-Level Error）                       │
├─────────────────────────────────────────────────────────────┤
│ - 文件操作失败、网络请求失败、命令执行错误                     │
│ - 发生错误后可能不是最后一步，LLM会收到observation           │
│ - LLM判断：继续循环 or 进入final                           │
│ - 需要：复用action_tool字段 → 传给LLM观察                   │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 错误分类对照表

| 错误类型 | 分类 | 说明 | 处理方式 |
|---------|------|------|---------|
| API超时耗尽 | 会话级 | 重试3次后仍然超时 | 结束会话 |
| 认证/权限失败 | 会话级 | API key无效、权限不足 | 结束会话 |
| 安全拦截 | 会话级 | 危险操作被拦截 | 结束会话 |
| API完全不可用 | 会话级 | 429、500等 | 结束会话 |
| 工具执行失败 | 工具级 | 文件操作、网络请求失败 | 传给LLM决定 |
| 命令执行错误 | 工具级 | shell命令返回非0 | 传给LLM决定 |
| 工具未实现 | 会话级 | 功能模块尚未实现 | 结束会话 |

## 三、现有 action_tool 字段分析

### 3.1 现有字段结构

```python

def format_action_tool_sse(
```

|------|--------|---------|------|

| `step` | 数字 | 数字 | ✅ 保持不变 |
| `tool_name` | 工具名 | 工具名 | ✅ 保持不变 |
| `tool_params` | 参数 | 参数 | ✅ 保持不变 |

| `raw_data` | 原始数据 | 详细错误 | ✅ **可选：详细错误信息** |

| 需求 | 现有字段 | 是否满足 | 说明 |
|------|---------|---------|------|
| 标记错误 | `execution_status='error'` | ✅ 已满足 | 用 `'error'` 标记 |

| 工具参数 | `tool_params` | ✅ 已满足 | 保持不变 |
| 错误码 | - | ⚠️ 可扩展 | summary可包含错误码 |

### 3.4 可能需要新增的字段

**如果确实需要，可以考虑新增：**

| 字段名 | 类型 | 说明 | 优先级 |
|--------|------|------|--------|
| `error_code` | `str` | 错误码（如 `FILE_NOT_FOUND`） | 可选：`summary` 已包含 |
| `is_retryable` | `bool` | 是否可重试 | 可选：已有 `action_retry_count` 暗示 |
| `error_type` | `str` | 错误分类（如 `file_error`） | 可选：已有 `execution_status='error'` 暗示 |

**优先级原则**：
1. ✅ **优先使用现有字段**
2. ⚠️ **确实不够用时再新增**
3. ❌ **不要为了"完美"而新增字段**

## 四、核心设计

### 4.1 函数设计

```python
# ============================================================
# 会话级错误 - 统一的函数
# ============================================================

def create_session_error_result(
    error_type: str,           # 错误类型标识
    original_error: str,        # 原始错误消息
    step_num: int,
    model: str = None,
    provider: str = None,
    display_name: str = None,
    chat_timeout: int = None,
    max_retries: int = None,
    retryable: bool = True,
    retry_after: int = 3,
) -> tuple[str, dict]:
    """
    统一的会话级错误处理函数
    
    用于：
    - 聊天API超时耗尽
    - 认证/权限失败
    - 安全拦截
    - API完全不可用
    - 功能未实现
    
    特点：
    - 发生错误后就是最后一步
    - yield给前端 + 保存到DB + 结束会话
    """
    # 1. 解析错误码
    resolved_type = resolve_http_error_type(original_error) if original_error else None
    final_type = resolved_type if resolved_type else error_type
    
    # 2. 组装错误信息
    error_code, error_message = get_stream_error_info(final_type, original_error)
    
    # 3. 特殊情况处理
    if error_type == 'idle_timeout' and display_name and chat_timeout and max_retries:
        total_timeout = chat_timeout * max_retries
        error_message = f"请求超时：AI模型({display_name}) {chat_timeout}秒内未返回任何内容，已重试{max_retries}次，合计{total_timeout}秒"
    
    # 4. 创建 error_response（yield给前端）
    error_response = create_error_response(
        error_type=error_code,
        message=error_message,
        model=model,
        provider=provider,
        retryable=retryable,
        retry_after=retry_after,
        step=step_num
    )
    
    # 5. 创建 error_step（保存到DB）
    error_step = create_error_step(
        type='error',
        code=error_code,
        message=error_message,
        error_type=error_code,
        step_num=step_num,
        model=model,
        provider=provider,
        retryable=retryable,
        retry_after=retry_after
    )
    
    return error_response, error_step


# ============================================================
# 工具级错误 - 复用现有 format_action_tool_sse
# ============================================================

def create_tool_error_result(
    tool_name: str,            # 工具名称
    error_message: str,        # 错误消息
    step_num: int,
    tool_params: dict = None,  # 工具参数
    retry_count: int = 0,      # 当前重试次数
    max_retries: int = 3,     # 最大重试次数
    raw_data: Any = None,      # 详细错误信息（可选）
) -> dict:
    """
    统一的工具级错误处理函数
    
    【重要】必须复用现有 action_tool 字段，不另搞一套！
    
    用于：
    - 文件操作失败
    - 网络请求失败
    - 命令执行错误
    
    特点：
    - 发生错误 ≠ 最后一步
    - LLM会收到这个作为 observation
    - LLM决定：继续循环 or 进入final
    - 复用 format_action_tool_sse() 格式化
    """
    # 计算重试状态
    can_retry = retry_count < max_retries
    
    # 构建错误摘要
    if can_retry:
        summary = f"[错误] {tool_name} 执行失败: {error_message}，正在重试 ({retry_count + 1}/{max_retries})..."
    else:
        summary = f"[错误] {tool_name} 执行失败: {error_message}，已重试{max_retries}次"
    
    # 【关键】复用 format_action_tool_sse()，不需要新建函数
    # 返回 dict，由调用方用 format_action_tool_sse() 格式化
    return {
        'tool_name': tool_name,
        'tool_params': tool_params or {},
        'execution_status': 'error',     # 标记为错误
        'summary': summary,             # 错误摘要
        'raw_data': raw_data or error_message,  # 详细错误
        'action_retry_count': retry_count,
        'step': step_num,
        'timestamp': create_timestamp()
    }
```

## 十、核心原则

### 10.1 必须遵守的原则

1. **错误分类**：区分会话级错误和工具级错误
2. **工具级错误复用 action_tool**：`execution_status = 'error'` + `summary = 错误信息`

#### 12.4.4 实施4：解析失败处理（重试机制）

**目标**：解析失败时重试3次，仍失败则退出loop（遵循北京老陈规定）

**设计原则**：
- 解析失败是**可恢复错误**，应给LLM自我修正的机会
- 重试3次后仍失败，才退出loop
- 重试时添加错误提示到历史，让LLM重新输出正确格式

**计数器重置时机**（重要！）：
- **每次循环迭代开始时重置计数器**
- 这样每个step都有独立的3次重试机会
- 不会因为之前step的失败累积导致当前step提前退出

**当前代码**（base_react.py 第168-201行）：
```python
try:
    parsed = self.parser.parse_response(response)
except ValueError as e:
    error_result = ToolParser.handle_parse_error(response, e, logger)
    error_info = error_result["parsed_obs"]
    
    yield {
        "type": "thought",
        "step": step_count,
        "timestamp": create_timestamp(),
        "content": error_info["content"],
        ...
    }
    
    yield {
        "type": "final",
        "timestamp": create_timestamp(),
        "content": error_info["content"]
    }
    break
```

**修改后的流程**：
```
每次循环迭代开始
    ↓
重置计数器：parse_retry_count = 0
    ↓
解析响应
    ↓
解析成功？ → 继续正常流程
    ↓ 失败
添加错误到历史 → 重试计数器+1
    ↓
重试次数 < 3？ → 是 → 继续循环，让LLM重新尝试
    ↓ 否
重试次数 >= 3 → break退出循环 → 循环外 yield error
```

**修改后**（重试3次机制）：
```python
from app.chat_stream.error_handler import create_session_error_result

# 在类初始化中添加：
# self.max_parse_retries = 3

# 循环开始时重置计数器（重要！）
self.parse_retry_count = 0

try:
    parsed = self.parser.parse_response(response)
except ValueError as e:
    # 保存原始response到conversation_history
    self.conversation_history.append({"role": "assistant", "content": response})
    
    # 添加错误提示到历史，让LLM重新尝试
    self._add_observation_to_history(
        f"Parse error: {str(e)}. Please respond with valid JSON format."
    )
    
    # 重试计数器+1
    self.parse_retry_count += 1
    
    # 重试次数 < 3？继续循环； >= 3？退出循环
    if self.parse_retry_count >= self.max_parse_retries:
```

1. ~~直接break退出~~ → **重试3次机制**
