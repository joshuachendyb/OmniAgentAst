## 一、问题背景

### 1.1 为什么要区分错误类型

当前后端代码的错误处理存在以下问题：

1. **错误处理不统一**：分散在多处调用 `create_error_response()` + `create_error_step()`
2. **没有区分错误场景**：工具执行错误和会话级错误混为一谈
3. **杀死LLM继续的机会**：工具错误后直接yield error，相当于"杀死"了LLM继续思考的机会

**说明**：本设计方案基于第2、3、4章的设计要求，用于指导后续代码重构。第7、8、9章是具体的修改说明。

## 二、错误分类设计

### 2.1 两种错误场景

```
错误发生后的两种情况：

┌─────────────────────────────────────────────────────────────┐
│ 场景A：会话级错误（Session-Level Error）                     │
├─────────────────────────────────────────────────────────────┤
│ - API超时耗尽、认证失败、API完全不可用                       │
│ - 发生错误后就是最后一步了                                   │
│ - 需要：yield error → 结束会话 → 保存DB                     │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ 场景B：工具级错误（Tool-Level Error）                       │
├─────────────────────────────────────────────────────────────┤
│ - 文件操作失败、网络请求失败、命令执行错误                     │
│ - 发生错误后可能不是最后一步，LLM会收到observation           │
│ - LLM判断：继续循环 or 进入final                           │
│ - 需要：复用action_tool字段 → 传给LLM观察                   │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 错误分类对照表

| 错误类型 | 分类 | 说明 | 处理方式 |
|---------|------|------|---------|
| API超时耗尽 | 会话级 | 重试3次后仍然超时 | 结束会话 |
| 认证/权限失败 | 会话级 | API key无效、权限不足 | 结束会话 |
| 安全拦截 | 会话级 | 危险操作被拦截 | 结束会话 |
| API完全不可用 | 会话级 | 429、500等 | 结束会话 |
| 工具执行失败 | 工具级 | 文件操作、网络请求失败 | 传给LLM决定 |
| 命令执行错误 | 工具级 | shell命令返回非0 | 传给LLM决定 |
| 工具未实现 | 会话级 | 功能模块尚未实现 | 结束会话 |

### 3.3 现有字段覆盖情况

| 需求 | 现有字段 | 是否满足 | 说明 |
|------|---------|---------|------|
| 标记错误 | `execution_status='error'` | ✅ 已满足 | 用 `'error'` 标记 |
| 错误消息 | `summary` | ✅ 已满足 | summary放错误信息 |
| 详细错误 | `raw_data` | ✅ 已满足 | 可放完整错误堆栈 |
| 重试次数 | `action_retry_count` | ✅ 已满足 | 已有字段 |
| 工具名称 | `tool_name` | ✅ 已满足 | 保持不变 |
| 工具参数 | `tool_params` | ✅ 已满足 | 保持不变 |
| 错误码 | - | ⚠️ 可扩展 | summary可包含错误码 |

### 3.4 可能需要新增的字段

**如果确实需要，可以考虑新增：**

| 字段名 | 类型 | 说明 | 优先级 |
|--------|------|------|--------|
| `error_code` | `str` | 错误码（如 `FILE_NOT_FOUND`） | 可选：`summary` 已包含 |
| `is_retryable` | `bool` | 是否可重试 | 可选：已有 `action_retry_count` 暗示 |
| `error_type` | `str` | 错误分类（如 `file_error`） | 可选：已有 `execution_status='error'` 暗示 |

**优先级原则**：
1. ✅ **优先使用现有字段**
2. ⚠️ **确实不够用时再新增**
3. ❌ **不要为了"完美"而新增字段**

## 四、核心设计

### 4.1 函数设计

```python
# ============================================================
# 会话级错误 - 统一的函数
# ============================================================

def create_session_error_result(
    error_type: str,           # 错误类型标识
    original_error: str,        # 原始错误消息
    step_num: int,
    model: str = None,
    provider: str = None,
    display_name: str = None,
    chat_timeout: int = None,
    max_retries: int = None,
    retryable: bool = True,
    retry_after: int = 3,
) -> tuple[str, dict]:
    """
    统一的会话级错误处理函数
    
    用于：
    - 聊天API超时耗尽
    - 认证/权限失败
    - 安全拦截
    - API完全不可用
    - 功能未实现
    
    特点：
    - 发生错误后就是最后一步
    - yield给前端 + 保存到DB + 结束会话
    """
    # 1. 解析错误码
    resolved_type = resolve_http_error_type(original_error) if original_error else None
    final_type = resolved_type if resolved_type else error_type
    
    # 2. 组装错误信息
    error_code, error_message = get_stream_error_info(final_type, original_error)
    
    # 3. 特殊情况处理
    if error_type == 'idle_timeout' and display_name and chat_timeout and max_retries:
        total_timeout = chat_timeout * max_retries
        error_message = f"请求超时：AI模型({display_name}) {chat_timeout}秒内未返回任何内容，已重试{max_retries}次，合计{total_timeout}秒"
    
    # 4. 创建 error_response（yield给前端）
    error_response = create_error_response(
        error_type=error_code,
        message=error_message,
        model=model,
        provider=provider,
        retryable=retryable,
        retry_after=retry_after,
        step=step_num
    )
    
    # 5. 创建 error_step（保存到DB）
    error_step = create_error_step(
        type='error',
        code=error_code,
        message=error_message,
        error_type=error_code,
        step_num=step_num,
        model=model,
        provider=provider,
        retryable=retryable,
        retry_after=retry_after
    )
    
    return error_response, error_step


# ============================================================
# 工具级错误 - 复用现有 format_action_tool_sse
# ============================================================

def create_tool_error_result(
    tool_name: str,            # 工具名称
    error_message: str,        # 错误消息
    step_num: int,
    tool_params: dict = None,  # 工具参数
    retry_count: int = 0,      # 当前重试次数
    max_retries: int = 3,     # 最大重试次数
    raw_data: Any = None,      # 详细错误信息（可选）
) -> dict:
    """
    统一的工具级错误处理函数
    
    【重要】必须复用现有 action_tool 字段，不另搞一套！
    
    用于：
    - 文件操作失败
    - 网络请求失败
    - 命令执行错误
    
    特点：
    - 发生错误 ≠ 最后一步
    - LLM会收到这个作为 observation
    - LLM决定：继续循环 or 进入final
    - 复用 format_action_tool_sse() 格式化
    """
    # 计算重试状态
    can_retry = retry_count < max_retries
    
    # 构建错误摘要
    if can_retry:
        summary = f"[错误] {tool_name} 执行失败: {error_message}，正在重试 ({retry_count + 1}/{max_retries})..."
    else:
        summary = f"[错误] {tool_name} 执行失败: {error_message}，已重试{max_retries}次"
    
    # 【关键】复用 format_action_tool_sse()，不需要新建函数
    # 返回 dict，由调用方用 format_action_tool_sse() 格式化
    return {
        'tool_name': tool_name,
        'tool_params': tool_params or {},
        'execution_status': 'error',     # 标记为错误
        'summary': summary,             # 错误摘要
        'raw_data': raw_data or error_message,  # 详细错误
        'action_retry_count': retry_count,
        'step': step_num,
        'timestamp': create_timestamp()
    }
```

### 6.2 工具级错误 - BaseAgent

```python
# BaseAgent 中处理工具执行错误
async def run_stream(self, ...):
    for step in range(max_steps):
        try:
            # 调用工具
            result = await self.execute_tool(tool_name, tool_params)
            # 成功：yield action_tool
            yield format_action_tool_sse(
                step=step_num,
                tool_name=tool_name,
                tool_params=tool_params,
                execution_status='success',
                summary=f"{tool_name} 执行成功",
                raw_data=result
            )
            
        except ToolExecutionError as e:
            # 【关键】工具执行失败 = 不是最后一步！
            # 复用 action_tool 字段，不另搞一套！
            error_dict = create_tool_error_result(
                tool_name=tool_name,
                error_message=str(e),
                tool_params=tool_params,
                step_num=step_num,
                retry_count=retry_count,
                max_retries=max_retries,
            )
            
            # 格式化并yield
            error_sse = format_action_tool_sse(**error_dict)
            yield error_sse
            
            # 不return！继续循环！
            # LLM会收到这个action_tool(error)，决定：继续 or final
             continue
```

### 7.2 后续扩展：自动重试机制（可选）

**说明**：以下功能为可选扩展，当前不需要实现。如未来需要，可按此设计添加。

#### 7.2.1 什么时候需要重试

| 错误类型 | 是否可重试 | 原因 |
|---------|-----------|------|
| **临时错误** | ✅ 可重试 | 网络超时、连接断开、服务暂时不可用 |
| **永久错误** | ❌ 不重试 | 文件不存在、权限不足、参数错误 |

## 八、文件分布与函数关系

### 8.1 各文件的职责（重要！）

| 文件 | 职责范围 | 不要做什么 |
|------|---------|-----------|
| `error_handler.py` | **会话级错误**：API超时、认证失败、安全拦截、429/500等 | ❌ 不处理工具级重试 |
| `tool_executor.py` | **工具级错误**：文件操作、网络请求、命令执行失败 | ❌ 不处理会话级错误 |
| `sse_formatter.py` | **SSE格式化**：统一输出格式 | - |
| `base_react.py` | **ReAct循环**：调用tool_executor、组装LLM消息 | - |

### 8.2 现有代码函数确认

| 设计文档函数 | 现有代码函数 | 位置 | 关系 |
|-------------|-------------|------|------|
| `create_session_error_result()` | `create_error_result()` | error_handler.py:451 | **现有代码需要改名为** `create_session_error_result()`，与设计保持一致 |
| `create_error_from_exception()` | `create_error_from_exception()` | error_handler.py:535 | 已存在，用于处理Exception对象 |
| TRANSIENT_ERROR_PATTERNS | 无 | - | ❌ 待创建，放在 tool_executor.py |
| PERMANENT_ERROR_PATTERNS | 无 | - | ❌ 待创建，放在 tool_executor.py |

### 8.3 数据流向图

```
【会话级错误】 error_handler.py
                    │
                    ├─→ create_session_error_result() → 返回 (error_response, error_step)
                    │                                    ↓
                    │                              SSE格式 + DB存储
                    │                                    ↓
                    └───────────────────────────────→ yield给前端 → 结束会话

【工具级错误】 tool_executor.py
                    │
                    ├─→ execute() → 智能重试
                    │       ├─→ Transient错误 → 自动重试3次
                    │       └─→ Permanent错误 → 立即返回
                    │
                    └─→ 返回 execution_result {status, summary, retryable, retry_count}
                                          │
                                          ▼
                               create_tool_error_result() (error_handler.py)
                                          │
                                          ▼
                                format_action_tool_sse()
                                          │
                                          ▼
                                sse_formatter.py (复用现有字段!)
                                          │
                                          ▼
                               base_react.py 组装observation
                                          │
                                          ▼
                               yield action_tool → LLM决定 → 继续或final
```

### 8.4 核心原则

1. **错误分类明确**：
   - 会话级错误 → `error_handler.py`
   - 工具级错误 → `tool_executor.py`

2. **函数统一**：
   - 每个模块只做自己的事
   - 会话级错误不要往 tool_executor 里放
   - 工具级错误不要往 error_handler 里放

3. **字段复用**：
   - 工具级错误复用 `format_action_tool_sse()` 的字段
   - 不要新增字段

4. **LLM决策权**：
   - 工具级错误：LLM决定继续或final

```python
def create_tool_error_result(
    tool_name: str,            # 工具名称
    error_message: str,        # 错误消息
    step_num: int,
    tool_params: dict = None,  # 工具参数
    retry_count: int = 0,      # 当前重试次数（后续扩展用）
    raw_data: Any = None,     # 详细错误信息（可选）
) -> dict:
    """
    统一的工具级错误处理函数
    
    用途：文件操作失败、网络请求失败、命令执行错误
    特点：发生错误 ≠ 最后一步，LLM会收到这个作为observation
    
    返回：可直接yield的action_tool格式字典
    """
    # 构建错误摘要
    summary = f"[错误] {tool_name} 执行失败: {error_message}"
    
    # 返回dict，可直接yield
    return {
        'type': 'action_tool',
        'step': step_num,
        'timestamp': create_timestamp(),
        'tool_name': tool_name,
        'tool_params': tool_params or {},
        'execution_status': 'error',  # 标记为错误
        'summary': summary,
        'raw_data': raw_data or error_message,
        'action_retry_count': retry_count
    }
```

## 十、核心原则

### 10.1 必须遵守的原则

1. **错误分类**：区分会话级错误和工具级错误
2. **工具级错误复用 action_tool**：`execution_status = 'error'` + `summary = 错误信息`
3. **不新增字段**：工具级错误完全复用现有 `format_action_tool_sse()` 的字段
4. **统一处理**：每种错误类型有对应的统一函数
5. **保持 LLM 机会**：工具级错误要让 LLM 决定下一步
6. **文件职责清晰**：会话级→error_handler.py，工具级→tool_executor.py

### 10.2 设计目标

