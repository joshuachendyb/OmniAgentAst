## 二、SSE数据处理完整流程

### 2.1 流程概述

对于每个 `type`（如 start、thought、action_tool、observation、final），数据处理分为以下环节：

```
后端发送SSE事件 → 前端接收数据 → 数据解析 → 类型分发 → 
更新ref → 保存sessionStorage → 更新React state → 
触发回调 → 渲染显示
```

### 2.2 详细处理环节

#### 环节1：后端发送SSE事件

后端通过流式响应发送JSON格式的事件数据：

```json
data: {"type": "thought", "step": 2, "timestamp": 1775722431736, "content": "...", "tool_name": "list_directory"}
```

#### 环节2：前端接收数据

**文件**: `sse.ts`
**行号**: 454-509

```typescript
const reader = response.body.getReader();
const decoder = new TextDecoder("utf-8");
let buffer = "";

while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  
  buffer += decoder.decode(value, { stream: true });
  const lines = buffer.split("\n");
  buffer = lines.pop() || "";
  
  // 处理每一行数据
  for (const line of lines) {
    processSSEData(line, ...);
  }
}
```

#### 环节3：数据解析

**文件**: `sse.ts`
**行号**: 722-740

```typescript
const rawData = JSON.parse(jsonStr);

// 统一处理timestamp转换
let timestampValue = Date.now();
if (rawData.timestamp) {
  if (typeof rawData.timestamp === 'number') {
    timestampValue = rawData.timestamp;
  } else if (typeof rawData.timestamp === 'string') {
    const parsed = Date.parse(rawData.timestamp);
    timestampValue = isNaN(parsed) ? Date.now() : parsed;
  }
}

// 创建Step对象

  type: rawData.type as ExecutionStep["type"],

## 四、SSE连接生命周期

### 4.1 连接建立

**触发**: 用户发送消息，调用 `executeStreamSend` → `sendMessage` → `sendMessageInternal`

**日志**: 
```
🔌 [SSE连接] 开始建立连接 | 2026-04-09T08:13:48.000Z
```

### 4.2 空闲超时检测

**配置**: `IDLE_TIMEOUT = 60000` (60秒)

**逻辑**: 
- 每次收到数据更新 `lastDataTimeRef.current`
- 设置60秒超时定时器
- 超时后抛出异常，中断连接

### 4.3 连接关闭

**触发**: 
- 收到final事件（正常完成）
- 收到error事件（出错）
- 收到incident事件（中断）
- 空闲超时（长时间无数据）
- 用户主动断开

**日志**:
```
🔌 [SSE连接] 关闭连接 | manual=false | 2026-04-09T08:13:49.000Z
```

## 五、步骤类型说明

### 5.1 步骤类型列表

| type | 含义 | 说明 |
|------|------|------|
| start | 开始 | AI开始处理，显示模型信息 |
| thought | 思考 | AI的思考过程，可能包含工具调用 |
| action_tool | 工具执行 | 执行具体工具（list_directory等） |
| observation | 观察结果 | 工具执行返回的结果 |
| final | 完成 | AI生成最终回复 |
| error | 错误 | 处理出错 |

### 5.2 典型流程示例

```
用户: "查看D盘的目录"
       ↓
start: 显示模型信息 (dmxapi kimi-k2.5-free)
       ↓
thought: "用户要求查看D盘，我需要使用list_directory工具"
       ↓
action_tool: 执行 list_directory 工具
       ↓
observation: 工具返回结果 "成功读取目录，共28个项目"
       ↓
thought: "任务已完成，向用户汇报结果"
       ↓
final: 生成最终回复
```

### 8.4 关键理解点

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    关键理解点                                       │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  1. 【代码执行是同步的】                                            │
│     • while 循环中的所有操作（除 reader.read() 外）都是同步执行            │
│     • JSON.parse、switch、更新 ref、调用回调都是立即完成的              │
│                                                                     │
│  2. 【渲染是异步的】                                               │
│     • setExecutionSteps / setMessages 是同步调用 setState              │
│     • 但 React 的渲染周期是异步的（批量更新、Virtual DOM对比）      │
│     • 浏览器真正显示变化是在下一次渲染周期后                      │
│                                                                     │
│  3. 【为什么这样设计】                                             │
│     • 同步处理保证数据不错过                                        │
│     • 异步渲染保证性能（批量更新、合并多次setState）                   │
│     • 避免每次数据更新都触发一次完整渲染                             │
│                                                                     │
│  4. 【实际影响】                                                   │
