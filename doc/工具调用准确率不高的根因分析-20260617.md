## 二、根因分析（6大原因）

### 2.1 原因1：工具数量太多，选择困难

**问题**：

```
当前工具数: 65个
初始加载: 2个分类（FUND_RUNTIME + FILE）≈ 20个工具
动态注入: tool_search按需加载其他分类
```

**OpenAI的选择困境**：

1. **初始工具列表太长**：
   - 即使只加载2个分类，也有~20个工具
   - LLM需要从20个工具中选择正确的那个
   - 工具名相似度高（read_text_file vs read_media_file vs read_pdf）

2. **工具描述相似**：
   ```
   read_text_file: "读取文本文件。支持分页读取..."
   read_media_file: "读取图片、音频、视频文件..."
   read_pdf: "读取PDF文件内容..."
   ```
   LLM难以区分何时用哪个

3. **选择压力**：
   - 工具越多，选择难度指数增长
   - 65个工具 vs 10个工具，准确率差距明显

**证据**：

OpenAI官方建议：
- 工具数量控制在10-20个最佳
- 超过50个工具，准确率显著下降
- 建议用tool_choice强制指定

---

### 2.2 原因2：description质量不够好

**问题**：description写法影响LLM理解

**当前写法**（read_text_file）：

```
读取文本文件。支持分页读取(head/tail/offset/limit)。encoding默认utf-8,读取失败自动尝试gbk。适用场景:需要读取源代码、日志文件、配置文件等纯文本内容时使用。
```

**问题分析**：

1. **功能描述太泛**：
   - "读取文本文件" - 太笼统
   - 没说清楚什么文件算"文本文件"
   - 没说清楚什么场景必须用这个工具

2. **适用场景描述太弱**：
   - "适用场景:需要读取源代码、日志文件、配置文件等纯文本内容时使用"
   - 这只是举例，不是明确规则
   - LLM可能理解为"也可以用其他工具"

3. **缺少反面说明**：
   - 没说清楚"什么情况不要用这个工具"
   - 没说清楚"这个工具不能做什么"

**对比优秀写法**：

```
读取纯文本文件内容（.txt/.py/.js/.json/.md/.log等）。

【必须用此工具的场景】
- 读取源代码文件（.py/.js/.ts/.java等）
- 读取配置文件（.json/.yaml/.ini等）
- 读取日志文件（.log）
- 读取文档文件（.md/.txt）

【禁止用此工具的场景】
- 读取Word文档 → 用read_docx
- 读取Excel表格 → 用read_xlsx
- 读取PDF文档 → 用read_pdf
- 读取图片/音频/视频 → 用read_media_file

【关键特性】
- 自动检测编码（utf-8/gbk）
- 支持分页读取（head/tail/offset/limit）
- 返回文本内容，不返回二进制
```

**差异**：
- ✅ 明确什么情况必须用
- ✅ 明确什么情况禁止用
- ✅ 列出具体文件类型
- ✅ 区分与其他工具的边界

---

### 2.3 原因3：examples不够典型

**问题**：examples数量和质量影响LLM学习

**当前状态**：
- 平均2.4个examples
- examples大多是简单调用

**当前examples**（read_text_file）：

```json
[
  {"file_path": "D:/project/main.py"},
  {"file_path": "D:/project/main.py", "head": 10}
]
```

**问题分析**：

1. **场景覆盖不全**：
   - 只展示了读文件和分页读
   - 没展示读取日志文件、配置文件等场景
   - 没展示encoding参数的使用

2. **缺少错误示例**：
   - 没展示错误用法（如读取Word文档）
   - LLM不知道边界在哪里

3. **缺少对比示例**：
   - 没展示"这个场景用read_text_file，那个场景用read_pdf"
   - LLM难以区分工具边界

**对比优秀examples**：

```json
[
  {
    "comment": "读取Python源代码",
    "file_path": "D:/project/main.py"
  },
  {
    "comment": "读取日志文件前10行",
    "file_path": "D:/logs/app.log",
    "head": 10
  },
  {
    "comment": "读取JSON配置文件",
    "file_path": "D:/config/settings.json"
  },
  {
    "comment": "读取Markdown文档",
    "file_path": "D:/docs/README.md"
  },
  {
    "comment": "分页读取大文件（第100-200行）",
    "file_path": "D:/data/large.csv",
    "offset": 100,
    "limit": 100
  }
]
```

**注意**：OpenAI不支持comment字段，但可以在description里说明典型用法。

### 2.4 原因4：未启用strict模式

**问题**：未启用function.strict，arguments格式可能不规范

**当前状态**：

```python
func_def = {
    "name": meta.name,
    "description": meta.description,
    "parameters": meta.input_schema
    # ❌ 缺少 "strict": True
}
```

**后果**：

1. **LLM可能输出不符合Schema的arguments**：
   - 必填参数缺失
   - 类型错误
   - 枚举值错误

2. **运行时解析错误**：
   - arguments不是合法JSON
   - 参数类型不匹配
   - 导致工具执行失败

**OpenAI官方说明**：

> 启用strict模式后，模型保证输出的arguments严格符合JSON Schema，100%格式正确。

**建议**：

```python
func_def = {
    "name": meta.name,
    "description": meta.description,
    "parameters": meta.input_schema,
    "strict": True  # ← 启用严格模式
}
```

**注意**：strict模式要求parameters必须符合strict schema规范：
- `additionalProperties: false`
- 所有字段都要有type
- 嵌套object也要符合规范

---

### 2.5 原因5：未控制parallel_tool_calls

**问题**：默认允许并行调用，可能导致混乱

**当前状态**：
- 未传递parallel_tool_calls参数
- 默认true（允许并行调用）

**后果**：

1. **LLM一次返回多个tool_calls**：
   ```json
   {
     "tool_calls": [
       {"id": "call_1", "function": {"name": "read_text_file", ...}},
       {"id": "call_2", "function": {"name": "read_pdf", ...}},
       {"id": "call_3", "function": {"name": "read_xlsx", ...}}
     ]
   }
   ```

2. **问题**：
   - LLM不确定用哪个，索性全调用
   - 增加错误率
   - 浪费token

3. **文件操作场景更严重**：
   - 同时读多个文件可能冲突
   - 同时写多个文件更危险

**建议**：

```python
# 文件操作类工具强制串行
if has_file_operations(tools):
    body["parallel_tool_calls"] = False

# 查询类工具允许并行
else:
    body["parallel_tool_calls"] = True
```

---

### 2.6 原因6：system prompt不够明确

**问题**：system prompt对工具选择的指导不够强

**当前system prompt**（节选）：

```
<工具规则>
- 有专用工具时优先用专用工具，不能用 execute_code/execute_shell 替代
- 当前列表没有 → tool_search 搜一轮。搜索词用动词+对象描述你要做的事
- tool_search 返回空 → 确认确实无专用工具，再用 execute_code 自行实现
```

**问题分析**：

1. **"优先用专用工具"太弱**：
   - LLM可能理解为"也可以用其他工具"
   - 没有强制约束

2. **缺少工具选择规则**：
   - 没说清楚"读Word文档必须用read_docx"
   - 没说清楚"读PDF必须用read_pdf"
   - LLM可能选错

3. **缺少反面禁令**：
   - 没说清楚"禁止用read_text_file读Word"
   - 没说清楚"禁止用execute_shell读文件"

**对比优秀写法**：

```
<工具选择铁律>
【文件读取】
- 读.txt/.py/.js/.json/.md/.log → 必须用read_text_file
- 读.docx/.doc → 必须用read_docx，禁止用read_text_file
- 读.pdf → 必须用read_pdf，禁止用其他工具
- 读.xlsx/.xls/.csv → 必须用read_xlsx
- 读图片/音频/视频 → 必须用read_media_file

【文件写入】
- 写.txt/.py/.js/.json → 必须用write_text_file
- 写.docx → 必须用write_docx
- 写.xlsx → 必须用write_xlsx

【禁止行为】
- 禁止用execute_shell读取文件内容
- 禁止用execute_code读取文件内容
- 禁止用read_text_file读取Word/PDF/Excel
```

## 四、改进方案（优先级排序）

### 4.1 P0：立即改进（影响最大）

#### 改进1：启用strict模式

**修改**：`backend/app/services/tools/tool_description.py`

```python
def to_openai_tools(registry, categories=None, strict=True):
    tools = []
    for name, meta in sorted(registry._tools.items()):
        if not meta.expose_to_llm:
            continue
        func_def = {
            "name": meta.name,
            "description": meta.description,
            "parameters": meta.input_schema,
            "strict": strict,  # ← 新增
        }
        if meta.examples:
            func_def["examples"] = meta.examples
        tools.append({"type": "function", "function": func_def})
    return tools
```

**预期效果**：
- arguments格式100%正确
- 减少解析错误
- 提升准确率10-20%

#### 改进2：控制parallel_tool_calls

**修改**：`backend/app/services/llm/client_sdk.py`

```python
def _build_request_body(
    # ... 原有参数
    parallel_tool_calls: Optional[bool] = None,
) -> Dict:
    body = {"model": model, "messages": messages}
    # ...
    
    # 新增：根据工具类型控制并行/串行
    if tools:
        body["tools"] = tools
        if tool_choice:
            body["tool_choice"] = tool_choice
        
        # 自动判断：文件操作类工具强制串行
        if parallel_tool_calls is None:
            file_tools = {"read_text_file", "write_text_file", "edit_text_file", 
                         "move_file", "copy_file", "delete_file"}
            tool_names = {t["function"]["name"] for t in tools}
            parallel_tool_calls = not bool(tool_names & file_tools)
        
        body["parallel_tool_calls"] = parallel_tool_calls
    
    return body
```

**预期效果**：
- 文件操作强制串行，减少冲突
- 查询操作允许并行，提升效率
- 提升准确率5-10%

## 七、总结

### 7.1 核心问题

**工具调用准确率不高的根本原因**：

1. **工具数量太多**（65个），选择困难
2. **description质量不够好**，LLM难以区分
3. **examples不够典型**，场景覆盖不全
4. **未启用strict模式**，arguments格式不规范
5. **未控制parallel_tool_calls**，并行调用混乱
6. **system prompt不够明确**，缺少强制规则

---

### 7.2 关键发现

**OpenAI官方建议 vs 我们现状**：

| 建议 | 现状 | 差距 |
|------|------|------|
| 工具数10-20个 | 65个 | 超标3倍 |
| 启用strict模式 | 未启用 | 缺失 |
| 控制parallel_tool_calls | 未控制 | 缺失 |
| 明确工具选择规则 | 规则太弱 | 不足 |

---

### 7.3 最关键改进

**如果只改一处，改哪里？**

**答案**：优化system prompt

**原因**：
1. 不需要改代码，只改prompt
2. 立即生效，无需重启
3. 影响最大（15-25%提升）
4. 成本最低

**具体改法**：

在system prompt里加入工具选择铁律，明确说清楚：
- 什么场景必须用什么工具
- 什么场景禁止用什么工具
