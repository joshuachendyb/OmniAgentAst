## 二、🔴 严重问题（必须修复）

### 2.1 引用不存在的模块（2处，运行时崩溃风险）

#### BROKEN-1: `system_register.py` 引用了已迁移的 `reg_register`

**文件**: `app/services/tools/system/system_register.py:197`  
**代码**: `from app.services.tools.system.reg_register import _register_registry_tools`  
**根因**: 2026-05-18 小沈在 `system/` 下添加了注册表工具注册调用。2026-06-17 小健将 `reg_*` 整体迁入独立的 `win_registry/` 目录（`system/__init__.py` 注释中记录了迁移），但 `system_register.py:197` 的导入路径未同步更新。  
**现状**: `reg_register.py` 不存在于 `system/`，但 `win_registry/win_registry_register.py` 中存在同名的 `_register_registry_tools()` 函数。  
**影响**: 当 `_register_system_tools()` 被调用时，第197行的 `import` 会抛出 `ModuleNotFoundError`，**整个 system 工具注册流程崩溃**，注册表工具（registry_read/registry_write/registry_delete）彻底不可用。  
**修复**: 将导入路径改为 `from app.services.tools.win_registry.win_registry_register import _register_registry_tools`。

#### BROKEN-2: `desktop_tools.py` 12处引用不存在的 `gui_tools`

**文件**: `app/services/tools/desktop/desktop_tools.py` 第237/240/271/274/277/305/308/311/334/337/355/360行  
**代码**: `from app.services.tools.desktop.gui_tools import _focus_window` 等12个import  
**根因**: 实际文件名是 `desktop_gui_tools.py` 而非 `gui_tools.py`。这是遗留的命名不一致问题。  
**影响**: 所有12个函数（窗口聚焦/调整、鼠标点击/移动/滚动、键盘输入/快捷键、截图、剪贴板）在运行时均会触发 `ModuleNotFoundError`。  
**修复**: 将12处 `gui_tools` 改为 `desktop_gui_tools`。

## 三、🔴 架构反转问题（3处，违反分层原则）

### 3.1 `llm/` ← `agent/` 反向依赖

**文件**: `app/services/llm/base_service.py:21`  
**代码**: `from app.services.agent.agent_utils.message_utils import build_llm_messages`  
**调用**: 第256行 `chat()` 方法中使用  
**问题**: LLM服务层依赖于Agent层的消息构建函数。`build_llm_messages` 的功能是将用户消息+历史转换为LLM消息数组，属于 **LLM接口适配职责**，应位于 `llm/` 或 `utils/`。  
**违反原则**: SRP（LLM层做了Agent层的message格式化）、依赖方向（低层依赖高层）  
**修复建议**: 将 `build_llm_messages` 移至 `app/services/llm/` 或 `app/utils/`，agent层改为导入LLM层的函数。

### 3.2 `llm/` ← `task/` 反向依赖

**文件**: `app/services/llm/base_service.py:20`  
**代码**: `from app.services.task.task_registry import check_cancelled, check_paused`  
**问题**: LLM服务层直接依赖任务注册表的取消/暂停检查。这些检查应通过 **回调/事件机制** 或由 **调用方（agent层）注入**。  
**违反原则**: 依赖方向反转 + 耦合度偏高  
**修复建议**: 在 `BaseAIService.request_stream()` 中通过参数接收 `should_stop` 回调，而非直接调用 `task_registry`。

### 3.3 `visualization/` ← `safety/` 反向依赖

**文件**: `app/services/visualization/` 下6个文件  
| 文件 | 导入 |
|------|------|
| `tree_report.py:17` | `query_tree_operations` |
| `sankey_report.py:18` | `query_sankey_operations` |
| `text_report.py:110` | `query_file_operations` |
| `json_report.py:64` | `query_file_operations` |
| `animation_report.py:18` | `query_animation_operations` |
| `mermaid_report.py:24` | `query_mermaid_operations` |

**问题**: 可视化层直接依赖安全层的文件操作查询。可视化应通过 **数据仓库/DAO层** 获取数据，而非直接耦合到具体的安全模块查询函数。  
**违反原则**: 依赖方向反转 + 扩展不灵活  
**修复建议**: 引入统一的数据访问层，可视化只依赖数据接口而非具体的安全模块。

## 四、🔴 循环依赖风险

### 4.1 `task/` ↔ `agent/` 循环引用

| 方向 | 文件 | 导入内容 |
|------|------|---------|
| `task/` → `agent/` | `task_cancel_check.py:14`, `task_interrupt_check.py:14` | `MetaStep` from `agent/steps` |
| `agent/` → `task/` | `agent/core_agent/` (lazy import) | `hitl_confirmation`, `get_tracker` |

**问题**: `task/` 使用 `MetaStep`（agent步骤类型）构建中断/取消消息，`agent/` 使用 `task/` 的确认和追踪功能。目前是 **延迟导入（lazy import）** 避免了直接循环，但设计上仍有循环耦合。  
**修复建议**: 将 `MetaStep` 移到公共位置（如 `steps/base.py` 或 `types/`），使 `task/` 不依赖 `agent/` 包路径。
