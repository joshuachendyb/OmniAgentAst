## 二、问题根因分析

### 2.1 数据流程对比

#### 非流式模式（`/chat`）
```
后端处理
    ↓
返回 ChatResponse {
  content: "...",
  execution_steps: [...]
}
    ↓
前端从response获取
    ↓
saveMessage传递数据 ✅
```

#### 流式模式（`/chat/stream`）- 当前问题
```
后端 SSE 事件
    ↓
sse.ts: setExecutionSteps([step1, step2])
    ↓
useSSE 返回 executionSteps 数组
    ↓
onComplete 回调
    ↓
saveMessage时 ❌ 没有传递 executionSteps
```

## 四、修复方案建议

### 4.1 execution_steps保存（新增）

**修复位置**: NewChatContainer.tsx

**方案**: 在useSSE调用中获取executionSteps，在onComplete中传递给saveMessage

```typescript
// 1. useSSE调用时获取executionSteps
const {
  executionSteps,  // 需要添加
  // ...
} = useSSE(config, handleStep, handleChunk, handleComplete, handleError);

// 2. onComplete中传递（需要使用ref保存）
await sessionApi.saveMessage(currentSessionId, {
  role: "assistant",
  content: finalResponse,
  execution_steps: JSON.stringify(executionStepsRef.current),  // 新增
});
```

## 五、对报告4.4节的意见

### 5.1 统一显示区域方案

**报告建议**: 在步骤面板统一显示所有内容

**小新观点**: 不建议完全统一，建议保留两层结构

| 方案 | 优点 | 缺点 |
|------|------|------|
| 统一显示（报告方案） | 简单，不需要切换 | 内容多了要滚动很久，体验差 |
| **分层显示（建议）** | **结构清晰，职责分明** | **切换时机需要控制** |

**建议保留**：
```
✅ 步骤面板 - 执行步骤（工具调用过程）
✅ 回复区域 - AI最终回复
```
