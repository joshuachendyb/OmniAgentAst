## 🔴 一、准确性问题

### 问题1：关键词匹配策略不准确

**当前实现** (457-479行):
```python
for keyword in config["keywords"]:
    if keyword in message_lower:
        if keyword in message_lower.split() or len(keyword) >= 6:
            score += 0.3  # 完整词匹配
        else:
            score += 0.2  # 部分匹配
```

**问题**:
- 过于宽松的 `keyword in message_lower` 会导致误匹配
- 例如："文件" 会匹配 "文档"、"档案"、"文件名"等所有包含"文件"的词
- "list" 会匹配 "list_directory"、"listing"、"application"等

**场景示例**:
```python
message = "这个应用程序很有用"  # 应该不是文件操作
# 但 "list" in "应用程序" 为 True (匹配"应用")
# 可能被误判为 search 操作
```

**影响**:
- 增加误判率 (False Positive)
- 非文件操作被错误识别为文件操作
- 用户体验下降

### 问题2：路径特征检测不精确

**当前实现** (474-475行):
```python
if any(char in message for char in ['/', '\\', '.txt', '.md', '.py', '.json']):
    score += 0.1
```

**问题**:
- 简单的字符包含检测不够精确
- `.txt` 可能在 "text" 中出现
- `/` 可能在 "/" 或 "路径/" 中出现

**场景示例**:
```python
message = "告诉我关于文本编辑的技巧"  # 不是文件操作
# ".txt" in "文本" 为 True
# 会被错误加分
```

**影响**:
- 增加误判率
- 路径特征容易被误触发

### 问题3：置信度计算缺乏上下文理解

**当前实现** (459-480行):
```python
if keyword in message_lower.split() or len(keyword) >= 6:
    score += 0.3  # 完整词匹配
else:
    score += 0.2  # 部分匹配
```

**问题**:
- 没有考虑上下文语境
- 单纯的关键词匹配无法区分语义
- "删除" 在 "我不想删除" 和 "请删除" 中语义相反

**场景示例**:
```python
message = "我不想删除这个文件" 
# "删除" 被匹配，但用户实际不想删除
# 应该是反向意图，但被识别为删除操作

message = "请帮我查看一下文件内容"
# "查看" 被匹配，但上下文是 "帮我"，语义不同
```

**影响**:
- 无法处理否定意图 (不想、不要、别删等)
- 无法理解语气和上下文
- 识别准确性下降

### 问题4：多关键词匹配策略过激

**当前实现** (470-471行):
```python
if len(matched_keywords) >= 2:
    score += 0.2
```

**问题**:
- 简单的匹配数累加不够智能
- 可能导致无关词共同增强非文件操作的置信度
- 例如："我想删除文件中的一些文本" → "删除"+"文件"+"文本" 匹配3个关键词

**场景示例**:
```python
message = "我想删除文章中的一些文字描述"  # 这不是文件操作
# "删除" (delete) + "文件" (list) + "文字" (可能匹配read) = 3个关键词
# 置信度被错误提高
```

**影响**:
- 增加误判率
- 非文件操作被错误高置信度识别

## 🟡 二、性能问题

### 问题5：字符串操作效率低

**当前实现**:
```python
message_lower.split()  # 多次调用
keyword in message_lower.split()  # O(n*m) 复杂度
```

**问题**:
- 对于长消息，split() 和遍历操作效率低
- 每个关键词都要遍历整个词列表
- 时间复杂度 O(n*m*k) 其中 n=消息长度, m=关键词数, k=匹配次数

**影响**:
- 大文本消息处理缓慢
- 高频调用时性能瓶颈

### 问题6：正则表达式缺失

**当前实现**: 使用简单的 `in` 操作

**问题**:
- 无法处理复杂的模式匹配
- 无法使用正则表达式进行精确匹配
- 无法忽略标点符号、空格等干扰

**场景示例**:
```python
message = "请帮我看一下 C:\\Users\\test.txt 文件"
# "C:\\Users\\test.txt" 应该匹配路径特征
# 但由于特殊字符可能无法正确匹配
```

**影响**:
- 匹配精度不高
- 无法处理复合模式
