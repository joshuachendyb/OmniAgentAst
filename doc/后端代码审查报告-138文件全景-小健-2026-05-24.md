#### 13.1.1 当前结构拆解

#### 13.1.2 违反原则分析

- **SLAP 严重违反**: 同一函数内混合了 3 个抽象层次（循环控制、状态转换、buffer 操作）
- **SRP 严重违反**: 一个函数承担了 10 个独立职责
- **KISS 严重违反**: 498 行，阅读者无法在单一屏幕内理解完整逻辑
- **DRY 中度违反**: chunk_buffer flush 逻辑重复 3 次（chunk→final、answer→flush、action→flush）
- **YAGNI 轻度违反**: 注释掉的旧逻辑（如 `# self.parser = ToolParser()`）和大量注释保留

#### 13.1.4 重构方案

**目标**: 将 `run_stream` 从 498 行压缩到 ~80 行（高层循环骨架），其余逻辑下沉到 8 个私有方法。

```
run_stream(...)  # ~80 行：初始化 + while + 分发器 + 异常兜底
├── _initialize_run_state(task, context, max_steps, task_id)  # 40 行
├── _should_interrupt_and_yield(step_count, running_tasks, task_id)  # 15 行
├── _handle_llm_response(response, step_count, running_tasks, task_id)  # 30 行
│   ├── _handle_empty_response(response, step_count)  # 50 行（原 407-452）
│   ├── _dispatch_parsed_type(parsed, step_count, ...)  # 40 行
│   │   ├── _handle_chunk_type(parsed, step_count, ...)  # 60 行
│   │   ├── _handle_completion_type(parsed, step_count, ...)  # 40 行
│   │   ├── _handle_thought_only(parsed, step_count, ...)  # 35 行
│   │   ├── _handle_parse_error(parsed, step_count, ...)  # 50 行
│   │   └── _handle_action_type(parsed, step_count, ...)  # 80 行
│   └── _execute_action_flow(tool_name, tool_params, step_count, ...)  # 80 行
└── _handle_run_exception(e, step_count)  # 15 行
```

**关键设计决策**:

1. **提取 `ChunkBuffer` 类**（独立文件 `agent/chunk_buffer.py`）
   - 职责: 管理 chunk 拼接、阈值检测、flush 到 message_builder
   - 消除 3 处重复的 flush 逻辑
   - 接口: `append(content)`, `should_promote()`, `flush_to(builder)`, `clear()`

2. **提取 `_dispatch_parsed_type()` 分发器**
   - 将 6-way if/elif 链（chunk/answer/thought_only/parse_error/action/隐含兜底）提取为映射表
   - 使用 `HANDLER_MAP = {"chunk": _handle_chunk_type, ...}` 替代长 if/elif

3. **合并 4 处中断检查**
   - 当前 4 处中断检查代码几乎相同（仅日志前缀不同）
   - 提取为 `_should_interrupt_and_yield()`，接受 `check_point: str` 参数用于日志区分

4. **`_handle_action_type` 委托给 `_execute_action_flow()`**
   - 包含: ThoughtStep yield → 中断检查 → _execute_tool_step → 中断检查 → Observation → pending calls
   - 这是当前唯一还比较大的私有方法（~80行），但职责单一："执行完整 Action→Observation 周期"

#### 13.1.5 重构风险点与具体解决方案

| 风险 | 具体解决方案（含代码） |

| **5. tool_name 有效性检查** | `_valid_tool_names` 在循环外初始化（361-364行），作为参数向下传递：<br>```python<br># 原: 在 run_stream 内初始化 _valid_tool_names<br># 重构: 在 _initialize_run_state 中初始化并返回<br>def _initialize_run_state(self, ...) -> Tuple[ChunkBuffer, Set[str]]:<br>    ...<br>    try:<br>        valid_tool_names = {t["name"] for t in tool_registry.list_tools()} | {"finish"}<br>    except Exception:<br>        valid_tool_names = {"finish"}<br>    return chunk_buffer, valid_tool_names<br># run_stream 中:<br>chunk_buffer, valid_tool_names = self._initialize_run_state(...)<br># 传入 _dispatch_parsed_type:<br>result = _dispatch_parsed_type(parsed, step_count, chunk_buffer, valid_tool_names)<br>``` |
