### 13.1 `base_react.py:run_stream()` 重构设计

**当前规模**: 498 行 | **文件位置**: `backend/app/services/agent/base_react.py:297`

#### 13.1.1 当前结构拆解

`run_stream` 包含以下 10 个逻辑片段（抽象层次混杂）：

| 片段 | 行号 | 职责 | 抽象层次 |
|------|------|------|---------|
| 初始化状态 | 327-353 | 重置 steps/message_builder/status/counters | 高层 |
| 获取 Prompt | 343-352 | sys_prompt/task_prompt + init_history | 高层 |
| 最大步数检查 | 370-373 | if step_count >= max_steps → exit | 高层循环控制 |
| 中断检查 x4 | 378-388, 399-404, 700-705, 722-727 | 检查 running_tasks[task_id].cancelled | 高层循环控制 |
| 空响应处理 | 407-452 | 截断历史重试 / 报错退出 | 中层策略 |
| chunk 处理 | 467-520 | buffer 拼接、阈值检查、无工具Agent退出 | 低层数据操作 |
| answer/implicit | 524-562 | flush buffer、yield final、return | 中层状态转换 |
| thought_only | 565-598 | yield thought_step、trim_history、continue | 中层状态转换 |
| parse_error | 612-656 | 重试计数、429 退避、error/observation 注入 | 中层错误恢复 |
| action 执行 | 659-767 | ThoughtStep→工具执行→Observation→pending calls | 高层+中层+低层混合 |
| 异常兜底 | 782-794 | 未捕获异常 → ErrorStep | 高层错误处理 |

#### 13.1.2 违反原则分析

- **SLAP 严重违反**: 同一函数内混合了 3 个抽象层次（循环控制、状态转换、buffer 操作）
- **SRP 严重违反**: 一个函数承担了 10 个独立职责
- **KISS 严重违反**: 498 行，阅读者无法在单一屏幕内理解完整逻辑
- **DRY 中度违反**: chunk_buffer flush 逻辑重复 3 次（chunk→final、answer→flush、action→flush）
- **YAGNI 轻度违反**: 注释掉的旧逻辑（如 `# self.parser = ToolParser()`）和大量注释保留

#### 13.1.3 可复用性检查

搜索其他文件是否有可复用的逻辑片段：

| 当前片段 | 可复用来源 | 说明 |
|---------|-----------|------|
| chunk_buffer 管理 | `react_sse_wrapper.py` 也有 SSE chunk 累积逻辑 | 可提取为通用 `ChunkBuffer` 类 |
| 中断检查 `_check_interrupt` | 已是独立方法 ✅ | 无需改动 |
| 空响应截断历史 | 无直接复用 | 特定于 ReAct 循环 |
| 429 指数退避 | `llm_core.py` 有类似退避逻辑 | 可复用 `_calculate_retry_delay()` 工具函数 |
| `_execute_tool_step` | 已提取为独立方法 ✅ | 上一步修复已完成 |

#### 13.1.4 重构方案

**目标**: 将 `run_stream` 从 498 行压缩到 ~80 行（高层循环骨架），其余逻辑下沉到 8 个私有方法。

```
run_stream(...)  # ~80 行：初始化 + while + 分发器 + 异常兜底
├── _initialize_run_state(task, context, max_steps, task_id)  # 40 行
├── _should_interrupt_and_yield(step_count, running_tasks, task_id)  # 15 行
├── _handle_llm_response(response, step_count, running_tasks, task_id)  # 30 行
│   ├── _handle_empty_response(response, step_count)  # 50 行（原 407-452）
│   ├── _dispatch_parsed_type(parsed, step_count, ...)  # 40 行
│   │   ├── _handle_chunk_type(parsed, step_count, ...)  # 60 行
│   │   ├── _handle_completion_type(parsed, step_count, ...)  # 40 行
│   │   ├── _handle_thought_only(parsed, step_count, ...)  # 35 行
│   │   ├── _handle_parse_error(parsed, step_count, ...)  # 50 行
│   │   └── _handle_action_type(parsed, step_count, ...)  # 80 行
│   └── _execute_action_flow(tool_name, tool_params, step_count, ...)  # 80 行
└── _handle_run_exception(e, step_count)  # 15 行
```

**关键设计决策**:

1. **提取 `ChunkBuffer` 类**（独立文件 `agent/chunk_buffer.py`）
   - 职责: 管理 chunk 拼接、阈值检测、flush 到 message_builder
   - 消除 3 处重复的 flush 逻辑
   - 接口: `append(content)`, `should_promote()`, `flush_to(builder)`, `clear()`

2. **提取 `_dispatch_parsed_type()` 分发器**
   - 将 6-way if/elif 链（chunk/answer/thought_only/parse_error/action/隐含兜底）提取为映射表
   - 使用 `HANDLER_MAP = {"chunk": _handle_chunk_type, ...}` 替代长 if/elif

3. **合并 4 处中断检查**
   - 当前 4 处中断检查代码几乎相同（仅日志前缀不同）
   - 提取为 `_should_interrupt_and_yield()`，接受 `check_point: str` 参数用于日志区分

4. **`_handle_action_type` 委托给 `_execute_action_flow()`**
   - 包含: ThoughtStep yield → 中断检查 → _execute_tool_step → 中断检查 → Observation → pending calls
   - 这是当前唯一还比较大的私有方法（~80行），但职责单一："执行完整 Action→Observation 周期"

#### 13.1.5 重构风险点

| 风险 | 说明 | 缓解措施 |
|------|------|---------|
| **chunk 阈值逻辑丢失** | `consecutive_chunk_count >= max_consecutive_chunks` 的跨循环状态 | ChunkBuffer 类持久化计数 |
| **parse_error 后重置逻辑** | 多处 `consecutive_chunk_count = 0; chunk_buffer = ""` | 统一用 `ChunkBuffer.clear()` |
| **空响应截断历史的 dedup 逻辑** | `seen_ids = set()` 的 id() 去重 | 原样移到 `_handle_empty_response` |
| **pending calls 的 step_count 同步** | `self._pending_step_count` 属性穿透 | 保持 `_execute_pending_calls` 接口不变 |
| **tool_name 有效性检查** | `_valid_tool_names` 集合在循环外初始化 | 作为参数传入 `_dispatch_parsed_type` |

#### 13.1.6 验证标准

重构后需满足：
- `run_stream` 本体 ≤ 100 行
- 每个私有方法 ≤ 80 行
- `ChunkBuffer` 类可独立测试
- 所有 6 种 `parsed["type"]` 分支均有单元测试覆盖

#### 13.2.2 违反原则分析

- **SLAP 严重违反**: 低层初始化（UUID、默认值）与高层调度（Agent 分发）混在一起
- **SRP 严重违反**: 同时承担 prompt 日志、安全检查、任务生命周期管理、SSE 流调度 4 个职责
- **DRY 中度违反**: `_run_agent_sse_stream` 和 `_run_generic_sse_stream` 有 ~70% 重复（已在 P1-010 记录）
- **KISS 中度违反**: 安全检查部分（572-638）独自占 67 行，可提取

#### 13.2.3 可复用性检查

| 当前片段 | 可复用来源 | 说明 |
|---------|-----------|------|
| Prompt Logger 启动 | `chat_router.py` 也有类似启动逻辑 | 提取为 `_start_prompt_logging()` 通用函数 |
| 安全检查+CRSS | `command_security.py` 已有 `calculate_risk_score_v2()` | 复用，不要重复构建 risk_detail |
| 任务注册/清理 | `cancel_task`/`pause_task`/`resume_task` 也在操作 running_tasks | 提取为 `TaskLifecycleManager` 类 |
| 中断/暂停检查 | 独立函数 ✅ | `check_and_yield_if_interrupted`/`check_and_yield_if_paused` 已是独立函数 |

#### 13.2.4 重构方案

```
generate_sse_stream(...)  # ~60 行：参数初始化 + try/finally + 分发
defaults = _fill_defaults(kwargs)  # 15 行
try:
    task_id = _ensure_task_id(task_id)
    _register_task(task_id, ai_service, running_tasks, running_tasks_lock)  # 15 行
    
    # Prompt Logger（可选，非 chat 意图）
    if intent_type not in ("", "generic", "chat"):
        _log_prompts(messages, intent_type, confidence, session_id, task_id)  # 50 行
    
    # 安全检查
    security_result = _perform_security_check(messages, next_step, session_id, 
                                               current_execution_steps, ai_service)  # 60 行
    if security_result:
        yield security_result  # 已被拦截
        return
    
    # 中断/暂停检查
    if await _check_interrupt_and_yield(...):
        return
    if await _check_pause_and_yield(...):
        return
    
    # Agent 分发（核心调度）
    async for sse_chunk in _dispatch_agent_stream(...):
        yield sse_chunk
        
except asyncio.CancelledError:
    await _handle_client_disconnect(...)
except Exception as e:
    await _handle_stream_error(e, ...)
finally:
    await _cleanup_task(task_id, running_tasks, running_tasks_lock, agent_llm_holder, llm_call_count)
```

**关键设计决策**:

1. **提取 `TaskLifecycleManager` 类**
   - 职责: 管理 running_tasks 字典的注册、查询、清理
   - 消除 `generate_sse_stream`、`cancel_task`、`pause_task`、`resume_task` 中对 running_tasks 的直接操作重复
   - 接口: `register(task_id, ai_service)`, `cancel(task_id)`, `cleanup(task_id)`

2. **提取 `_perform_security_check()`**
   - 当前 572-638 行全部移出，返回 `Optional[str]`（error SSE）或 `None`（通过）
   - CRSS 评分报告构建逻辑独立，可复用于其他入口

3. **提取 `_log_prompts()`**
   - 当前 485-538 行移出
   - 复用 `chat_router.py` 中的相似逻辑

4. **统一 `_run_agent_sse_stream` + `_run_generic_sse_stream`**
   - 两个 runner 共享: 中断检查、SSE 格式化、StepFactory 错误处理、DB 保存
   - 提取为 `_run_sse_stream(agent: BaseAgent, ...)` 通用 runner
   - Agent 创建作为参数传入（`AgentFactory.create()` 或 `_GenericAgent`）

#### 13.2.5 重构风险点

| 风险 | 说明 | 缓解措施 |
|------|------|---------|
| **prompt_logger 生命周期** | start_request → log_system_prompt → log_task_prompt 必须在同一函数内完成 | `_log_prompts()` 内部保证顺序 |
| **running_tasks 并发安全** | async with running_tasks_lock | TaskLifecycleManager 内部加锁 |
| **CancelledError 后 yield** | 客户端已断开，yield 可能失败 | 保持 try/except 包裹 |
| **agent_llm_holder 回填** | `generate_sse_stream` 读取 `agent_llm_holder["n"]` | finally 中保留此逻辑 |

### 13.3 `sessions.py:save_execution_steps()` 重构设计

**当前规模**: 259 行 | **文件位置**: `backend/app/api/v1/sessions.py:766`

#### 13.3.1 当前结构拆解

| 片段 | 行号 | 职责 |
|------|------|------|
| 会话存在性检查 | 798-806 | SELECT chat_sessions WHERE id=? |
| AI 消息 ID 计算（双路径） | 814-894 | 路径1: 内存缓存(user_message_id+1) / 路径2: 数据库查询 |
| ID 冲突解决 | 896-909 | 若 expected_assistant_id 被占用，查找下一个可用 ID |
| Metadata 提取 | 911-920 | 从 execution_steps[0] (start) 提取 model/provider/display_name |
| 消息创建/更新判断 | 922-969 | 根据 should_create_new 判断 INSERT or UPDATE |
| 构建 UPDATE 字段 | 971-994 | 动态构建 execution_steps + content 更新语句 |
| message_count 更新 | 996-1016 | 首次创建时 +1，否则只更新 updated_at |
| 事务提交 | 1018 | conn.commit() |

#### 13.3.2 违反原则分析

- **SLAP 严重违反**: 低层 SQL 拼接与高层业务逻辑（消息 ID 计算策略）混在一起
- **SRP 严重违反**: 同时承担会话检查、消息 ID 分配、ID 冲突解决、metadata 提取、UPSERT 执行
- **KISS 中度违反**: AI 消息 ID 计算有 2 条路径（内存缓存 / 数据库回退），每条路径又分"ID 已存在"和"ID 不存在"，共 4 种情况

#### 13.3.3 可复用性检查

| 当前片段 | 可复用来源 | 说明 |
|---------|-----------|------|
| 会话存在性检查 | `save_message()` 也有相同检查 | 提取为 `_check_session_exists(session_id)` |
| AI 消息 ID 计算 | `save_message()` 有类似逻辑（576-746） | 提取为 `AssistantMessageIdAllocator` 类 |
| ID 冲突解决 | 无 | 特定于消息 ID 分配策略 |
| Metadata 提取 | 无 | 特定于 execution_steps 结构 |
| DB UPSERT | `save_message()` 也做 INSERT/UPDATE | 提取为 `_upsert_chat_message(...)` 通用函数 |

#### 13.3.4 重构方案

```
save_execution_steps(session_id, update_data)  # ~40 行
check_session_exists(session_id)  # 10 行（复用 save_message 的相同逻辑）

with get_db_connection() as conn:
    message_id = allocator.allocate(session_id, conn)  # 50 行
    metadata = extract_metadata(update_data.execution_steps)  # 10 行
    
    should_create = not message_exists(message_id, conn)
    if should_create:
        insert_assistant_message(message_id, session_id, metadata, 
                                  update_data.content, conn)  # 30 行
    
    update_message_fields(message_id, update_data, conn)  # 20 行
    update_session_message_count(session_id, should_create, conn)  # 15 行
    
    conn.commit()
```

**关键设计决策**:

1. **提取 `AssistantMessageIdAllocator` 类**
   - 职责: 为 assistant 消息分配唯一 ID
   - 策略: 优先内存缓存(user_message_id+1) → 数据库回退(last user msg id + 1) → 异常兜底(id=1)
   - 冲突检测: 若 ID 被占用，查找 max(id)+1
   - 该类可被 `save_execution_steps` 和 `save_message` 复用

2. **提取 `extract_metadata()`**
   - 从 execution_steps 列表中提取 start 步骤的 model/provider/display_name
   - 简单遍历，职责单一

3. **提取 `_upsert_chat_message()`**
   - 通用函数：根据 message_id 存在性决定 INSERT/UPDATE
   - `save_execution_steps` 和 `save_message` 均可复用

4. **使用上下文管理器 `get_db_connection()`**
   - 当前函数手动 `conn = get_connection()` + `finally: conn.close()`
   - 提取为上下文管理器，消除 10+ 处重复模式

#### 13.3.5 重构风险点

| 风险 | 说明 | 缓解措施 |
|------|------|---------|
| **message_count 只增不减** | 首次创建时 +1，避免重复计数 | `should_create` 标志必须准确 |
| **ID 冲突时的 max(id)+1** | 可能跳过可用 ID 造成空隙 | 保持当前逻辑（max+1）|
| **内存缓存 _user_message_ids 同步** | 多 worker 场景下可能不一致 | 当前已用 `_message_ids_lock`，保持 |
| **display_name 回退逻辑** | `f"{provider} ({model})"` | 提取到 `extract_metadata()` 中保留 |

        if getattr(config_update, field) is not None:
            handler(config_data, config_update, config_path)  # 各 10-20 行
    
    _auto_fix_and_validate(config_data, config_path, backup_path)  # 30 行
    _write_and_reload(config_data, config_path)                  # 15 行
    
    return _build_success_response(config_data, config_update)
    
except Exception:
    _restore_backup_if_needed(backup_path, config_path)  # 15 行（消除3处重复）
    raise
```

**关键设计决策**:

1. **策略模式处理字段更新**
   - 每个配置字段一个 `_update_xxx()` 函数
   - 消除长 if/elif 链，新增字段只需添加新 handler

2. **提取 `_restore_backup_if_needed()`**
   - 当前 3 处备份恢复逻辑完全相同（验证失败、HTTPException、Exception）
   - 提取为带 `backup_restored` 标志的通用函数

3. **提取 `_auto_fix_and_validate()`**
   - 包含: `_fix_config_common_issues()` + `_validate_config_integrity()` + 失败恢复
   - 单一职责：验证并处理失败

4. **提取 `_invalidate_ai_service_cache()`**
   - 当前多处 `AIServiceFactory._instance = None`
   - 统一调用此函数

#### 13.4.5 重构风险点

| 风险 | 说明 | 缓解措施 |
|------|------|---------|
| **备份文件清理时机** | 旧逻辑保留备份等待 validate_ai_service，新逻辑立即删除 | 当前代码已改为立即删除，保持 |
| **异常时备份恢复** | 需确保 config_path 已赋值后再恢复 | 初始化后立即赋值 |
| **Pydantic 手动转换** | security 配置手动转 dict 避免版本问题 | 保持当前转换逻辑 |
| **ai.provider/model 排序** | 必须确保在最前面 | `_ensure_ai_provider_model_first()` 原样保留 |

#### 13.5.4 重构方案

```

        file_path, regex, replacement, conflict_strategy,

    )  # 40 行
    operations.append(op)

**关键设计决策**:

1. **提取 `_collect_files()` 通用函数**
   - 当前嵌套在 `batch_rename_impl` 和 `file_statistics_impl` 内的文件收集逻辑几乎相同
   - 提取为模块级函数，支持 recursive/timeout/pattern_filter

2. **提取 `_process_single_rename()`**
   - 职责: 处理单个文件的重命名流程

   - 改为直接调用 `file_safety.py` 和 `registry` 的函数
   - 减少参数数量，提高可测试性

4. **提取 `@with_timeout(deadline)` 装饰器**
   - 当前 `_collect_sync` 内手动检查 `_time.monotonic() > _br_deadline`
   - 提取为装饰器，可被 `batch_rename_impl`、`file_statistics_impl`、`compress_files_impl` 复用

   - `chat_stream`、`chat_with_tools_stream`、`_parse_sse_stream` 均复用
