## 一、设计要求

### 1.1 thought阶段的处理流程

```
thought阶段（ReAct循环第1阶段）
  1. 判断前一阶段类型（start或observation）
       ↓
  2. 生成 reasoning 内容
       ↓
  3. 构建 thought_data 字典（含"分析中..."占位）
       ↓
  4. yield thought_data（第一次，发送给前端）
       ↓
  5. 保存到数据库
       ↓
  6. 组织LLM输入（reasoning + 工具列表 + 任务说明）
       ↓
  7. 调用LLM（流式响应）
       ↓ LLM返回
  8. LLM返回：content + reasoning + action_tool + params
       ↓ LLM流结束
  9. 更新 thought_data（替换占位为content）
       ↓
 10. yield thought_data（第二次，发送给前端）
       ↓
 11. 更新数据库
       ↓
 12. 判断 action_tool
       ↓
       ├─ finish → 进入 final（ReAct退出）
       └─ 其他工具 → 进入 action_tool 阶段
```

---

### 1.2 thought_data 字段构成

**版本1：第一次 yield thought_data（步骤4，不含action_tool和params）**

```python
thought_data = {
    'type': 'thought',                 # 固定值"thought"
    'step': next_step(),                # 步骤序号（递增）
    'timestamp': create_timestamp(),    # 时间戳（毫秒）
    'content': '分析中...',             # 占位符
    'reasoning': reasoning,             # 传导内容（start或observation）
    # 注意：不发空的 action_tool 和 params
}
```

**版本2：第二次 yield thought_data（步骤10，完整数据）**

```python
yield {
    'type': 'thought',                # 固定值
    'step': step,                      # 步骤序号（与第一次相同）
    'timestamp': timestamp,             # 时间戳（与第一次相同）
    'content': llm_content,            # LLM返回：推理结论
    'reasoning': reasoning,            # 传导内容（start或observation）
    'action_tool': action_tool,        # LLM返回：工具名称
    'params': params                   # LLM返回：工具参数
}
```

**两次yield的字段对比**：

| 字段 | 第一次yield | 第二次yield | 说明 |
|------|------------|------------|------|
| type | "thought" | "thought" | 固定值，不变 |
| step | 序号 | 序号 | 不变 |
| timestamp | 时间戳 | 时间戳 | 不变 |
| content | "分析中..." | llm_content | LLM返回后替换 |
| reasoning | 传导内容 | 传导内容 | 不变（从start/observation传入） |
| action_tool | **不发（为空）** | 工具名称 | LLM返回后填充，第二次才发 |
| params | **不发（为空）** | 工具参数 | LLM返回后填充，第二次才发 |

### 1.3 reasoning 字段传导逻辑

**传导来源对照**：

| 进入来源 | reasoning 内容 | 生成方式 |
|---------|---------------|---------|
| 从 start 进入 | start.content + start.is_clear | 字符串拼接 |
| 从 observation 进入 | observation.summary | 直接取值 |

**详细生成代码**：

## 五、注意事项

### 5.1 安全性

- LLM调用失败时，需要有fallback逻辑
- 如果LLM解析失败，需要使用默认值（action_tool='finish', params={}）

### 5.2 性能

- LLM调用应该使用流式接口，避免长时间阻塞
- content可以实时yield给前端，改善用户体验

### 5.3 兼容性

- 第二次yield的数据格式需要与前端协商
- 确保前端能正确解析action_tool和params字段

```python
    current_execution_steps,
    current_content,
    save_execution_steps_to_db,
    running_tasks,
    running_tasks_lock
):
    """Start1版本的thought处理（过渡用）"""
    thought_data = {
        'type': 'thought',
        'step': next_step(),
        'timestamp': create_timestamp(),
        'thinking_prompt': '正在分析任务...'
    }
    yield thought_data
    
    current_execution_steps.append(thought_data)
    await save_execution_steps_to_db(current_execution_steps, current_content)
    
    await asyncio.sleep(0.3)
    
    # 检查中断
    is_interrupted, interrupt_msg = await check_and_yield_if_interrupted(
        task_id, running_tasks, running_tasks_lock, next_step
    )
    if is_interrupted:
        yield interrupt_msg
        return
```
