## 二、文档设计核心要点

### 2.1 架构设计

**底层方法**（2个）：
| 方法 | 返回 | 说明 |
|------|------|------|
| `request()` | `ChatResponse` | 非流式，Agent用 |
| `request_stream()` | `AsyncGenerator[StreamChunk]` | 流式，SSE服务层用 |

**上层方法**（1个）：
| 方法 | 返回 | 说明 |
|------|------|------|
| `chat()` | `AsyncGenerator[StreamChunk]` | 前端SSE专用，是request_stream的别名 |

### 2.2 核心变化

从**4个方法按名字分模式** → **2个方法按参数分模式**

| 现有 | 新设计 | 说明 |
|------|--------|------|
| `chat()` | `chat()` | 保留，前端SSE入口 |
| `chat_stream()` | `request_stream(mode="text")` | 统一为request_stream |
| `chat_with_tools()` | `request(mode="tools")` | 统一为request |
| `chat_with_tools_stream()` | 删除 | 死代码 |
| 无 | `request_stream(mode="tools")` | 如未来需要FC流式，无需新增方法 |

### 2.3 关键特性

1. **mode参数**：通过`mode="text"`或`mode="tools"`控制模式
2. **统一构建**：`_build_request_body()`和`_build_messages()`集中处理
3. **request_tools**：动态加载工具机制（设计中）

## 五、具体差异详解

### 5.1 方法调用链对比

**文档设计**：
```
SSE服务层: chat(message, history) → request_stream(messages, mode="text")
Agent TextStrategy: request(messages, mode="text")
Agent ToolsStrategy: request(messages, mode="tools", tools=tools)
```

**当前实现**：
```
SSE服务层: run_sse_stream() → agent.run_react_cycle() → 内部调用LLM
Agent: self.llm_client.chat(messages, tools, tool_choice)  # 直接调用SDK层
```

### 5.2 模式控制方式

**文档设计**：通过`mode`参数统一控制
```python
# 纯文本
response = await service.request(messages=msgs, mode="text")
# Function Calling
response = await service.request(messages=msgs, mode="tools", tools=tools)
```

**当前实现**：通过不同方法名区分
```python
# 纯文本
response = await llm_client.chat(messages=msgs)
# Function Calling
response = await llm_client.chat(messages=msgs, tools=tools, tool_choice="auto")
```

### 5.3 流式处理

**文档设计**：统一使用`request_stream()`
```python
async for chunk in service.request_stream(messages=msgs, mode="text"):
    yield chunk
```

**当前实现**：分散在不同方法
```python
# 流式FC
async for chunk in self.chat_with_tools_stream(message, history, tools, tool_choice):
    yield chunk
# 流式文本（通过chat_with_tools聚合）
response = await self.chat_with_tools(message, history, tools=None, tool_choice=None)
```

## 七、差异原因分析

### 7.1 架构差异

1. **SDK层 vs Service层**：文档设计在Service层统一构建，当前实现在SDK层统一构建
2. **方法命名**：文档设计使用`request/request_stream`，当前实现使用`chat_with_tools/chat_with_tools_stream`
3. **模式控制**：文档设计通过参数控制，当前实现通过方法名区分

### 7.2 实现路径差异

1. **渐进式重构**：当前代码可能是渐进式重构的结果，未完全达到文档设计目标
2. **兼容性考虑**：保留了原有方法名，避免破坏现有调用链
3. **实际需求**：`chat_with_tools_stream()`实际在用，不是死代码

### 7.3 设计与实现的时间差

- 文档创建时间：2026-05-16
- 当前代码时间：2026-06-09
- 可能文档是设计目标，当前代码是中间状态
