## 三、P0 致命Bug清单

### 问题 P0-B1: `registry.py` — get_implementations_from_registry 永远返回空字典 ✅ 已修复

**位置**: `registry.py:873-881`

**现象**:
```python
def get_implementations_from_registry() -> Dict[str, Callable]:
    return {name: tool_registry.get_implementation(name)
            for name in tool_registry.list_tools()}
```
`list_tools()` 返回 `List[Dict[str, Any]]`（每条为 `{"name": "read_file", ...}` 格式），但循环变量 `name` 被当作字符串传给 `get_implementation()` → **dict当做key查找，永远返回None** → ToolExecutor永远拿到空工具字典。

**归属类别**: 🔴 P0 被测代码Bug

**根因分析**: 2026-05-10工具加载重构时，未同步更新`list_tools()`返回值格式与调用方的类型预期。

**修复代码**:
```python
def get_implementations_from_registry() -> Dict[str, Callable]:
    tools_list = tool_registry.list_tools()
    if tools_list and isinstance(tools_list[0], dict):
        tool_names = [t["name"] for t in tools_list if "name" in t]
    else:
        tool_names = tools_list
    return {name: tool_registry.get_implementation(name)
            for name in tool_names}
```

**验证**: 修复后ToolExecutor能正常加载工具 ✅ 2026-05-26 小沈

### 问题 P0-B3: `react_sse_wrapper.py` — current_content 变量覆盖Bug ✅ 已修复

**位置**: `react_sse_wrapper.py:499-501`

**现象**:
```python
# _run_sse_stream 函数签名:
async def _run_sse_stream(..., current_content: str, ...):
    ...
    if step_data.get('type') == 'final':
        current_content = step_data.get('response', '')  # 局部变量覆盖参数！
```

`current_content`作为参数传入，但函数体内对其赋值创建了一个**同名局部变量**，外层`generate_sse_stream`中的`current_content`变量永远不会被更新 → 只有第一个final事件之前的内容能持久化。

**归属类别**: 🔴 P0 被测代码Bug

**根因分析**: 2026-05-25重构拆分`generate_sse_stream`时，提取`_run_sse_stream`辅助函数，但参数传递方式有误。

**修复代码**: 将fallback改为保留旧值：
```python
current_content = step_data.get('response', current_content) or current_content
```

**文件**: `backend/app/services/react_sse_wrapper.py:499-501`

### 4.4 file_helpers.py (68.4KB, 1858行) — 与file_tools.py关系分析

#### 当前架构问题

```
LLM工具注册 (file_register.py)
    ↓

    ├── 工具端点实现 (read_file, write_text_file, etc.)
```

1. **双重验证**: 工具方法在FileTools中做一次validation → 传递给file_helpers → file_helpers内部再次通过回调验证

3. **BINARY_EXTENSIONS不一致**: 模块级25项 vs 类属性22项，相差至少10项

## 八、架构设计问题

### 8.1 循环依赖

| 依赖对 | 当前规避方式 | 风险 |
|--------|-------------|------|
| `tool_executor.py ↔ retry_policy.py` | 方法体内延迟导入 | 脆弱，两套重复的重试循环 |
| `react_sse_wrapper.py → sessions.py` | 方法体内导入`_assistant_message_ids` | service层不应导入api层 |

### 8.2 单例模式缺陷

`registry.py`的`tool_registry`模块级全局变量和`get_instance()`方法均不强制单例——任何人都可以`ToolRegistry()`创建新实例。`__init__`无守卫。

### 8.3 类别合并残留

`ToolCategory`枚举中仍保留已合并的`SHELL`和`META`值。`CATEGORY_ORDER`和`CATEGORY_NAMES`中包含逻辑上已废弃的枚举。`INTENT_TO_CATEGORY`映射是单向的。

### 8.4 MRO继承复杂性

```
UniversalReactAgent:
  ToolStepMixin → ReactAgentMixin(ToolLoaderMixin) → RollbackMixin → BaseAgent(ABC)

DesktopReactAgent:
  ToolStepMixin → ReactAgentMixin(ToolLoaderMixin) → BaseAgent(ABC)
```

`ReactAgentMixin`继承`ToolLoaderMixin`(from `tools/mixin.py`)，但又定义了`_init_tools_and_executor()`替代`_load_tools()`。两个mixin的tool加载路径相互覆盖。

## 十三、功能分散根因分析

### 13.1 三个核心模式（都是Bug的温床）

| 模式 | 描述 | 涉及的Bug |
|------|------|-----------|
| **双重解析链** | LLM输出被TextStrategy解析一次，又被react_output_parser解析第二次 | 空工具名死循环、FC格式JSON未解析 |
| **双重重试/双重参数归一化** | tool_executor和retry_policy各自实现重试循环 | 错误分类不一致、循环依赖 |
| **分散的持久化点** | SSE事件在5个不同位置手动append+save | current_content变量覆盖Bug |

### 13.2 为什么重构后Bug更多？

```
多轮重构的方式:
  第1轮: 拆分大函数 → 提取辅助函数，但保留原调用模式
  第2轮: 增加新功能 → 在辅助函数旁边加新逻辑，不做整合
  第3轮: 修改返回格式 → 只改一处，其他调用点加兼容层
  ...

结果:
  - 旧代码没删 (tool_parser.py 433行死代码)
  - 新旧桥接层膨胀 (file_tools 8个转发方法)
  - 统一化半途而废 (17处result dict各有不同默认值)
  - 每次重构都可能引入覆盖变量的Bug (current_content)
```
