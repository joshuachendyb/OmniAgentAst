#### P0-002: 消息数量计算错误（已部分修复）

**症状**: 数据库中 message_count 与实际消息数不一致

**根本原因**: 闭包导致的状态不同步

**已修复的代码**:
```typescript
// 第117-119行：新增 ref 追踪消息数量
const messagesCountRef = useRef<number>(0);

// 第436-438行：通过 useEffect 同步更新
useEffect(() => {
    messagesCountRef.current = messages.length;
}, [messages]);

// 第268行：使用 ref 获取准确值
const currentCount = messagesCountRef.current;
```

**仍存在的风险**:
- 如果 `useEffect` 还没执行完就调用 `onComplete`，ref 可能不是最新值
- 并发场景下可能出现竞态条件

**修复建议**:
```typescript
// 在发送消息时立即更新 ref，而不是依赖 useEffect
const handleSend = async () => {
    // ... 创建消息 ...
    setMessages(prev => [...prev, userMessage]);
    messagesCountRef.current += 1; // ⭐ 立即更新
}
```

### P1-高优先级问题（导致数据不一致）

#### P1-001: sessionId 四种状态存储的不一致

**四种状态存储**:
```
1. state.sessionId           - React state
2. currentSessionIdRef.current - useRef
3. sessionStorage            - 浏览器缓存
4. 后端数据库                  - 持久化存储
```

**不一致场景**:

| 场景 | state | ref | sessionStorage | 数据库 |
|------|-------|-----|----------------|--------|
| 新建会话 | ✅ 更新 | ✅ 更新 | ❌ 未更新 | ✅ 已保存 |
| 消息保存完成 | ❌ 未更新 | ✅ 可能是旧值 | ❌ 未更新 | ✅ 已保存 |
| 从历史页面跳转 | ✅ 更新 | ✅ 更新 | ⚠️ 可能是旧会话 | ✅ 已存在 |
| 页面刷新恢复 | ✅ 从缓存恢复 | ✅ 更新 | ✅ 读取 | ✅ 已存在 |

**风险场景**:
```typescript
// onComplete 回调中 (第256行)
const sessionIdToUse = currentSessionIdRef.current || sessionId;

// 问题：如果用户快速点击"新建会话" → 发送消息 → 再新建会话
// currentSessionIdRef 可能指向旧会话
```

**修复方案**:
```typescript
// 每次发送消息前强制同步
const handleSend = async () => {
    let currentSessionId = sessionId;
    if (!currentSessionId) {
        // 创建新会话
        const newSession = await sessionApi.createSession(title);
        currentSessionId = newSession.session_id;
        setSessionId(currentSessionId);
        currentSessionIdRef.current = currentSessionId; // ⭐ 同步更新
    } else {
        // ⭐ 强制同步，确保 ref 是最新值
        currentSessionIdRef.current = currentSessionId;
    }
}
```

## 🏗️ 三、架构缺陷分析

### 缺陷1: 状态管理分散

**问题**: 同一个数据（sessionId、messages）存储在多个地方
- React state
- useRef
- sessionStorage
- 后端数据库

**后果**:
- 状态同步复杂，容易出错
- 调试困难，不知道哪个存储是权威来源

**修复建议**:
```typescript
// 使用单一数据源
// 方案1：使用 React Context + useReducer
// 方案2：使用 Zustand 等状态管理库
// 方案3：定义明确的状态层次

// 建议的状态层次
type SessionState = {
    // 权威来源：后端数据库
    sessionId: string | null;
    messages: Message[];
    title: string;
    version: number;
    
    // UI 状态（仅前端）
    editingTitle: boolean;
    loading: boolean;
    
    // 缓存（可选）
    cached: boolean; // 标记数据来自缓存还是服务器
};
```

---

### 缺陷2: 异步操作缺乏事务保护

**问题**: 多个异步操作没有原子性保证

**示例**:
```typescript
// 第267-282行：消息保存是两个独立的 API 调用
await sessionApi.saveMessage(sessionIdToUse, {
    role: "user",
    content: currentPending.content,
    message_count: currentCount,
});

await sessionApi.saveMessage(sessionIdToUse, {
    role: "assistant",
    content: fullResponse,
    message_count: currentCount + 1,
});

// 问题：如果第一个成功，第二个失败
// 结果：用户消息保存了，AI 回复丢失
```

**修复建议**:
```typescript
// 后端提供批量保存 API
POST /api/v1/sessions/{id}/messages/batch
{
    "messages": [
        { "role": "user", "content": "..." },
        { "role": "assistant", "content": "..." }
    ]
}

// 后端使用事务保证原子性
@router.post("/sessions/{session_id}/messages/batch")
async def save_messages_batch(session_id: str, messages: List[MessageCreate]):
    conn = _get_db_connection()
    cursor = conn.cursor()
    cursor.execute("BEGIN")
    try:
        for msg in messages:
            cursor.execute(...)
        conn.commit()
    except:
        conn.rollback()
        raise
```

### 缺陷3: 标题管理缺乏统一入口

**问题**: 标题更新散落在多个地方
- 用户手动编辑 → `updateSession` API
- 消息保存完成 → ❌ 没有调用
- 新建会话 → `createSession` 时传入
- 历史页面 → 显示但不同步

**修复建议**:
```typescript
// 统一的标题管理函数
const updateSessionTitle = async (sessionId: string, newTitle: string) => {
    // 1. 更新本地状态
    setSessionTitle(newTitle);
    setTitleLocked(true);
    
    // 2. 持久化到后端
    await ensureTitlePersisted(sessionId, newTitle);
    
    // 3. 同步 sessionStorage
    saveState();
    
    // 4. 记录日志
    console.log(`标题已更新: ${newTitle}`);
};

// 所有标题修改都通过这个函数
```
