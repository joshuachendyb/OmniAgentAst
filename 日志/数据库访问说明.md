## 二、数据库表结构

### 2.1 conversations 表（会话表）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | TEXT (PK) | 会话唯一标识（UUID） |
| title | TEXT | 会话标题 |
| created_at | TIMESTAMP | 创建时间 |
| updated_at | TIMESTAMP | 更新时间 |
| user_id | TEXT | 用户ID |
| model | TEXT | 使用的AI模型 |
| tags | TEXT | 会话标签（JSON格式） |

**示例数据**：
```sql
INSERT INTO conversations VALUES (
  'b5ffec3c-1c18-4d78-af20-7fba42e06733',
  '记录1003',
  '2026-03-12 08:30:00',
  '2026-03-12 08:35:00',
  'user_001',
  'gpt-4',
  '["work", "file-management"]'
);
```

### 2.2 messages 表（消息表）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | INTEGER (PK) | 消息唯一标识（自增） |
| conversation_id | TEXT (FK) | 关联的会话ID |
| role | TEXT | 角色：user/assistant/system |
| content | TEXT | 消息内容 |
| timestamp | TIMESTAMP | 消息时间戳 |
| execution_steps | TEXT | 执行步骤（JSON格式） |
| model | TEXT | 使用的AI模型 |
| is_streaming | BOOLEAN | 是否流式传输 |
| is_error | BOOLEAN | 是否错误消息 |
| display_name | TEXT | 显示名称 |

**execution_steps 字段说明**：
- **类型**: JSON 数组
- **结构**: 每个步骤是一个对象，包含 `type`、`content` 等字段
- **常用 type 值**：
  - `thought`: 思考过程
  - `action_tool`: 工具执行
  - `observation`: 执行结果
  - `chunk`: 流式传输的中间结果
  - `final`: 最终回复内容
  - `error`: 错误信息
  - `paused/resumed/interrupted/retrying`: 状态变化

**示例数据**：
```sql
INSERT INTO messages VALUES (
  337,
  'b5ffec3c-1c18-4d78-af20-7fba42e06733',
  'assistant',
  '这是AI的完整回复内容...',
  '2026-03-12 08:31:00',
  '[{"type":"thought","content":"思考中..."},{"type":"chunk","content":"部分回复..."}]',
  'gpt-4',
  0,
  0,
  'GPT-4'
);
```

### 2.3 titles 表（标题历史表）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| id | INTEGER (PK) | 标题历史唯一标识（自增） |
| conversation_id | TEXT (FK) | 关联的会话ID |
| title | TEXT | 标题内容 |
| created_at | TIMESTAMP | 创建时间 |

**说明**：用于记录会话标题的修改历史

## 四、重要字段说明

### 4.1 execution_steps 字段（关键字段）

**用途**：存储 AI 执行过程的详细步骤，结构示例见 2.2 节。

**type 类型说明**：
- `thought`: AI 的思考过程
- `action_tool`: AI 调用工具执行操作
- `observation`: 工具执行结果
- `chunk`: 流式传输的中间回复内容
- `final`: AI 的最终回复内容（完整）
- `error`: 错误信息
- `paused/resumed/interrupted/retrying`: 状态变化

### 4.2 content 字段

**用途**：存储消息的完整内容

**注意**：
- 对于 AI 消息，`content` 字段可能包含完整回复
- 但前端渲染时优先使用 `execution_steps` 中的 `chunk` 或 `final` 步骤
- 如果 `execution_steps` 中没有 `chunk`，才会回退显示 `content`

## 六、常见问题排查

### 6.1 消息显示为空（空气泡）

**排查步骤**：
1. 通过 API 查询消息的 execution_steps
2. 检查 chunk 数量是否正常（正常应该 > 10）
3. 检查是否有 final 步骤
4. 对比 content 长度和 chunk/final 长度

**API 命令**：
```bash
# 查询消息的 chunk 和 final 详情
curl -s "%BASE_URL%/sessions/{session_id}/messages" | python -c "
import sys, json
data = json.load(sys.stdin)
for msg in data.get('messages', []):
    if msg.get('role') == 'assistant':
        steps = msg.get('execution_steps', [])
        chunks = [s for s in steps if s.get('type') == 'chunk']
        finals = [s for s in steps if s.get('type') == 'final']
        content_len = len(msg.get('content', '') or '')
        chunk_total = sum(len(c.get('content','')) for c in chunks)
        final_len = len(finals[0].get('content','')) if finals else 0
        
        print(f'消息ID: {msg[\"id\"]}')
        print(f'  content长度: {content_len}')
        print(f'  chunk数量: {len(chunks)}, chunk总长: {chunk_total}')
        print(f'  final数量: {len(finals)}, final长: {final_len}')
        
        if len(chunks) < 10:
            print('  ⚠️ chunk数量过少，可能显示异常')
        if final_len > 0 and chunk_total < final_len * 0.5:
            print('  ⚠️ chunk总长远小于final，可能数据不完整')
        print()
"
```

### 6.2 chunk 数据不完整

**可能原因**：
1. SSE 流式输出中断
2. 前端保存时序问题
3. 网络不稳定

**排查方法**：
使用上面 6.1 的命令检查 chunk 数量，如果发现异常，检查网络连接。

### 6.3 无法获取会话列表

**检查步骤**：
```bash
# 1. 检查后端服务是否运行

```
