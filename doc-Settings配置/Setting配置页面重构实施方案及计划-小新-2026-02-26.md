## 一、背景与目标

### 1.1 问题现状

**前端Settings页面存在以下严重问题：

| 编号 | 问题类型 | 严重程度 | 说明 |
|------|---------|----------|------|
| UI-L01 | 模型配置布局混乱 | P0 | 左右布局，一个面板尺寸不合理 |
| UX-S01 | 无Tab脏状态检测 | P0 | 用户切换Tab时丢失数据 |
| PERF-01 | 批量删除串行同步 | P0 | 10个模型删除需10秒 |
| UX-M01 | 无模型重名检测 | P0 | 允许同名模型 |
| ARCH-01 | 组件过大 | P1 | ProviderSettings组件640行 |

**后端config.yaml结构违反8条P0规则**：

| 规则编号 | 规则内容 | 当前状态 |
|----------|----------|----------|
| Rule1 | ai.provider和ai.model必须在顶层 | ❌ 违反 |
| Rule2 | 每个provider只有models列表 | ❌ 违反 |
| Rule3 | ai.model必须在ai.provider的models中 | ❌ 违反 |
| Rule4 | 只有一个ai.provider和ai.model | ❌ 违反 |
| Rule5 | 所有模型必须在models中 | ❌ 违反 |
| Rule6 | 模型ID=provider:model | ❌ 违反 |
| Rule7 | provider下无model字段 | ❌ 违反 |
| Rule8 | 用下拉选择，不能手动输入 | ❌ 违反 |

### 1.2 重构目标

**后端目标**：
- ✅ 实现config.yaml自动修复接口
- ✅ 实现config.yaml完整验证
- ✅ 实现config.yaml自动备份
- ✅ 增强PUT /config验证
- ✅ 修复POST /config/provider

**前端目标**：
- ✅ 左右分栏布局（类似VS Code）
- ✅ 全局配置区域（ai.provider+ai.model下拉框）
- ✅ provider详情区域（api_base、api_key、models列表）
- ✅ Tab脏状态检测（防止数据丢失）
- ✅ 批量删除并发优化（10x性能提升）
- ✅ 模型重名检测
- ✅ 组件架构优化

## 三、后端接口设计

### 3.1 新增后端接口

#### 3.1.1 POST /api/v1/config/fix

**功能**：自动修复config.yaml常见问题

**请求**：
```
POST /api/v1/config/fix
Content-Type: application/json
```

**响应**：
```json
{
  "success": true,
  "data": {
    "fixed": ["rule1", "rule2", "rule7"],
    "backup_file": "config/config.yaml.backup.20260226_014654",
    "message": "修复完成"
  }
}
```

#### 3.1.2 增强PUT /api/v1/config

**增强内容**：
- 写入前验证8条P0规则
- 写入前自动备份
- 验证失败返回详细错误

### 3.2 后端辅助函数

| 函数名 | 功能 |
|--------|------|
| `_backup_config_file()` | 自动备份配置文件 |
| `_validate_config_integrity()` | 验证8条P0规则 |
| `_fix_config_common_issues()` | 自动修复常见问题 |
