## 一、策略选择的完整调用链

### 1.1 策略选择的核心目标

Agent系统支持两种LLM调用策略：
- **text策略**: 直接返回文本，由下游解析器(react_output_parser)从文本中提取工具调用
- **tools策略**: 通过OpenAI Function Calling协议的`tools`参数约束LLM输出结构化函数调用

策略选择的根本目标：**自动探测LLM是否支持Function Calling，选择最优调用方式**。

### 1.2 策略数据的初始化和确定时点

策略的初始化发生在Agent子类的`__init__`中，由`_init_llm_strategies()`完成。但策略的**最终确定**延迟到**首次`_call_llm()`调用时**（懒确定+lazy detection）。

```
Agent.__init__()
  └─ _init_llm_strategies()          ← 创建策略对象，不探测
       ├─ 有llm_client + api_base →
       │   ├─ LLMAdapter(llm_client.api_base, ...)   ← 适配器创建，_strategy=None
       │   ├─ ToolsStrategy(openai_tools)              ← FC策略对象创建
       │   └─ self._strategy = None                    ← 策略未确定
       │
       └─ 无llm_client或无api_base →
           ├─ self.adapter = None
           ├─ self.tools_strategy = None
           ├─ self.use_function_calling = False
           └─ self._strategy = "text"                  ← 直接确定为text

首次_call_llm():
  └─ self._strategy is None → 调用adapter.detect_strategy()
       ├─ 探测成功(FC支持) → _strategy = "tools"
       ├─ 探测成功(FC不支持) → _strategy = "text"
       └─ 探测异常 → _strategy = "text" (降级)
       
后续_call_llm():
  └─ self._strategy is not None → 直接使用缓存策略
```

#### 1.2.1 第一次`_call_llm()`后的策略选择完整调用链

```
react_agent_mixin._call_llm()
  │
  ├─ self.llm_call_count += 1
  ├─ mb.prepare_messages_for_llm()           ← 组装完整消息列表
  │
  ├─ [首次] self._strategy is None →
  │    └─ await self.adapter.detect_strategy()
  │         ├─ for attempt in 0..2 (3次重试)
  │         │    ├─ POST {api_base}/chat/completions (tools探测请求, stream=False)
  │         │    ├─ HTTP 200 + tool_calls → return "tools"
  │         │    ├─ HTTP 200 + no tool_calls → return "text"
  │         │    ├─ HTTP 429 → 如果是第1/2次 → 指数退避重试; 第3次 → return "tools"(乐观)
  │         │    └─ HTTP 其他/异常 → 如果是第1/2次 → 退避重试; 第3次 → return "text"
  │         └─ 缓存 self._strategy
  │
  ├─ messages = self._inject_tools_hint(messages, self._strategy)
  │    ├─ strategy == "tools" → 不注入(FC协议通过tools参数传递)
  │    └─ strategy == "text" → 注入工具描述文本
  │         ├─ 首次或category变化 → 构建缓存
  │         └─ 命中缓存 → 使用缓存
  │
  ├─ [text策略] messages = self._inject_schema(messages)
  │    └─ 注入Tools Schema参考文本(仅在text策略时)
  │
  ├─ self._log_prompt(messages, self._strategy)  ← prompt_logger记录
  │
  ├─ response = await self._dispatch_strategy(self._strategy, messages)
  │    ├─ strategy == "tools" →
  │    │    ├─ 同步 self.tools_strategy.tools = self.openai_tools
  │    │    ├─ await llm_client.chat_with_tools(messages, tools=openai_tools)
  │    │    ├─ 保存 self._last_fc_raw_response (供FC协议注入使用)
  │    │    └─ return response
  │    │
  │    └─ strategy == "text" →
  │         ├─ self._last_fc_raw_response = None
  │         └─ await llm_client.chat(messages)
  │              └─ 内部: await chat_stream() → 聚合流式结果 → ChatResponse
  │
  └─ self._log_response(response)            ← prompt_logger记录响应
```

### 1.3 LLMAdapter.detect_strategy() 探测实现

```python
class LLMAdapter:
    async def detect_strategy(self) -> str:
        if self._strategy is not None:
            return self._strategy  # 缓存命中，直接返回
        
        for attempt in range(3):
            # POST {api_base}/chat/completions
            # messages: [{"role": "user", "content": "Call test_tool"}]
            # tools: [{"type": "function", "function": {"name": "test_tool", ...}}]
            # tool_choice: "auto"
            # stream: False
            result = await self._probe_tools(client)
            
            if result["works"]:
                self._strategy = "tools"
                return "tools"
            
            # 429/超时/连接类错误 - 可重试
            if "429" in reason or "timeout" in reason or "connect" in reason:
                if attempt < 2:  # 前两次重试
                    await asyncio.sleep(2.0 * (2 ** attempt))
                    continue
            
            # 不支持FC
            self._strategy = "text"
            return "text"
        
        self._strategy = "text"  # 3次都失败
        return "text"
```

### 1.4 问题分析与发现

#### 问题 STRAT-001: 首次探测失败永久降级到text模式

**错误发生原因**: `detect_strategy()` 最多重试3次，如果3次都因为瞬态网络问题失败，策略永久固定为`"text"`。后续的`_call_llm()`直接使用缓存的`_strategy`，不会再重新探测FC支持。

**全分支情况分析**:

| 分支 | 条件 | 结果 | 后续影响 |
|------|------|------|---------|
| 分支A | 3次中有1次成功+FC支持 | `_strategy="tools"` | 正常使用FC |
| 分支B | 3次中有1次成功+FC不支持 | `_strategy="text"` | 正常使用text |
| 分支C | 3次全是429限流 | 乐观返回`"tools"`(第3次429) | 乐观假设FC可用 |
| 分支D | 3次全是超时/连接错误 | `_strategy="text"` | 降级到text，永不再探测 |
| 分支E | 3次全是HTTP 5xx | `_strategy="text"` | 降级到text，永不再探测 |
| 分支F | 混合错误(429+超时+5xx) | `_strategy="text"` | 降级到text，永不再探测 |

**分支D/E/F的问题**: 瞬态网络故障导致永久降级。假设服务器短暂不可用（如重启、网络抖动），Agent会在整轮对话中以text模式运行，无法使用FC协议。

**解决思路**:
- 方案A: 在`_call_llm()`中增加周期性重新探测机制（如每N次调用后重新探测）
- 方案B: 在`_call_llm()`检测到adapter为None或strategy为text但有FC工具可用时，尝试重新创建adapter
- 方案C: 在agent初始化时尝试探测，如果失败则在首次chat请求成功后（证明网络通了）再探测一次

> **v1.2代码验证（小资）**: 问题**确认真实**。代码路径：`llm_adapter.py` L24-57 `detect_strategy()` 3次循环后永久设置 `_strategy="text"`；`react_agent_mixin.py` L210-212 `_call_llm()` 只在 `_strategy is None` 时重新探测。三种解决思路的逻辑正确性分析：
> - 方案A ✅ **可行**：周期重探最稳健，建议每10次调用或遇到连续N次text策略失败时触发重探
> - 方案B ⚠️ **部分可行**：但重新创建adapter需要api_base/api_key，这些在降级时可能不可用
> - 方案C ✅ **可行且最简单**：在`_call_llm()`成功返回后检查`_strategy=="text"`且`use_function_calling==True`，尝试重探一次

#### 问题 STRAT-002: 429时立刻乐观返回tools，无需等待3次重试

**错误发生原因**: `_probe_tools()`在收到429时直接返回`{"works": True, "reason": "429限流,乐观默认支持"}`。`detect_strategy`在L34检查到`works=True`后**立即**返回`"tools"`，**不进入下方的429重试逻辑**。这意味着**第1次探测遇到429就会乐观启用tools策略**，并非文档之前描述的"3次重试后才乐观"。

**全分支情况分析**:
```
探测器收到429:
  _probe_tools() L68-69:
    → 检查 status_code == 429
    → 直接 return {"works": True, "reason": "429限流,乐观默认支持"}  ← 不重试!

  detect_strategy() L34-37:
    → result["works"] = True
    → self._strategy = "tools"
    → return "tools"  ← 立即返回，不进入下方的for循环重试逻辑！

后续实际调用:
  假设LLM实际不支持FC:
  → _strategy = "tools"
  → _dispatch_strategy("tools", messages)
    → tools_strategy.call(llm_client, messages)
      → llm_client.chat_with_tools(messages, tools=openai_tools)
        → POST {api_base}/chat/completions (含tools参数)
        → LLM忽略tools参数，返回普通文本content
        → chat_with_tools检测：没有tool_calls，但有content
        → 返回 ChatResponse(content=普通文本)
      → tools_strategy.call() 检测response.content
        → 不是JSON数组 → 当作字符串返回
      → 返回给_call_llm
      → parse_react_response解析纯文本
        → type=chunk或thought_only或answer

流程能正常结束（LLM返回了content），只是没有用到FC。
```

**实际影响**: 429乐观行为比原来更快——第1次遇到429就乐观，不需要等3次重试。乐观决策本身不会导致崩溃。最坏情况是：探测说支持FC但实际不支持，`chat_with_tools`的响应被当作普通文本返回给下游解析器，最终正确解析。代价是浪费了一次FC探测的HTTP请求，但不会导致功能错误。

**解决思路**: 可以在`_probe_tools`中让429返回`{"works": False, "reason": "429限流"}`，让`detect_strategy`的429重试逻辑（L40-45）真正起作用。保留当前行为也无大碍（不会导致功能错误）。

> **v1.2代码验证（小资）**: 问题**确认真实**。代码路径：`llm_adapter.py` L68-69 `_probe_tools()` 收到429直接返回 `{"works": True}` → L34 `result["works"]=True` → L35-37 立即 `return "tools"`，不进入L40-45的重试逻辑。解决思路**逻辑正确**：若429返回 `{"works": False, "reason": "429限流"}`，则L34 `result["works"]=False` → 跳过L35-37 → L38 `_reason="429限流"` → L40 `"429" in _reason` 匹配 → 进入重试。但需权衡：乐观策略在429时假设FC可用，即使误判也不崩溃（最坏情况LLM忽略tools参数返回普通文本），改为重试则增加探测延迟（2+4=6秒额外等待）。

#### 问题 STRAT-003: `_init_llm_strategies`中openai_tools只包含主分类，不包含跨分类加载的工具

**错误发生原因**: 在`_init_llm_strategies()`中，`tool_registry.to_openai_tools(self.tool_category)`只加载了Agent主分类的tools定义（如FILE→文件工具）。如果后续通过`load_tools_by_intent()`动态加载了其他分类的工具，`self.openai_tools`会被扩展，同时`_dispatch_strategy`会重新赋值`tools_strategy.tools = self.openai_tools`。

**全分支情况分析**:
```
初始化:
  openai_tools = to_openai_tools(FILE)  → 10个文件工具
  tools_strategy.tools = openai_tools   → 10个文件工具

load_tools_by_intent("system"):
  new_tools = to_openai_tools(SYSTEM)   → 10个系统工具
  openai_tools.extend(new_tools)        → 20个工具

下一次_call_llm → _dispatch_strategy("tools", messages):
  tools_strategy.tools = openai_tools   → 20个工具 ✓ 正常同步
```

**结论**: 当前的`_dispatch_strategy`代码已经处理了这个问题。`self.tools_strategy.tools = self.openai_tools`确保了每次FC调用前都同步最新的工具列表。**这不是一个实际的问题**，但有一个细微的竞态条件需要考虑：

如果在`load_tools_by_intent()`和下一次`_dispatch_strategy`之间发生了LLM调用（理论上不会，因为`load_tools_by_intent`只在observation之后调用），则`tools_strategy`中的tools定义可能过期。但在当前的ReAct循环中，`load_tools_by_intent()`总是在yield observation之后、下一轮LLM调用之前执行，所以此竞态条件不会实际发生。

#### 问题 STRAT-004: DesktopReactAgent初始化时会创建LLMAdapter，但如果初始化参数中缺少api_base则静默降级

**错误发生原因**: DesktopReactAgent在`__init__`中调用`super().__init__(**kwargs)`，BaseAgent的`__init__`从`**kwargs`中取出`api_base/api_key/model`设置到self。如果上层调用时没有传递这些参数，`_init_llm_strategies()`中`self.llm_client`可能非None，但`api_base`可能从llm_client中获取为空，导致adapter创建失败，静默降级到text模式。

**全分支情况分析**:

| 分支 | llm_client | api_base | 结果 |
|------|-----------|---------|------|
| 分支A | ✅ 非None | ✅ 非空 | 正常创建adapter，探测策略 |
| 分支B | ✅ 非None | ❌ 空 | adpter=None, use_function_calling=False, _strategy="text" |
| 分支C | ❌ None | — | adpter=None, use_function_calling=False, _strategy="text" |

```python
# 关键代码 react_agent_mixin.py L58-66
_has_client = self.llm_client is not None
_self_api_base = getattr(self, 'api_base', None) or (
    getattr(self.llm_client, 'api_base', None) if _has_client else None
)
# llm_client 非 None 但 api_base 从 llm_client 也取不到时:
# → _self_api_base = None
# → 走到 else 分支，降级到 text
```

**分支B的问题**: `llm_client`的对象存在（可能是BaseAIService实例），但`llm_client.api_base`也为空字符串或None。这种情况在实际运行中不应该发生（因为`BaseAIService.__init__`总是有`api_base`参数），但如果创建llm_client时api_base被错误地设为空字符串，Agent会静默地降级到text模式，没有任何错误日志表明"降级是因为api_base为空"。

**解决思路**: 在`_init_llm_strategies`中增加分支B的检测日志，明确指出降级原因。

> **v1.2代码验证（小资）**: 问题**确认真实**。代码路径：`react_agent_mixin.py` L60 `_self_api_base` 的计算逻辑，L71 `if self.llm_client and _self_api_base:` — 当 `llm_client` 存在但 `api_base` 为空/None时走 else 分支，L89 的日志 `_init_llm_strategies 跳过adapter: 直接text模式` 没有说明**降级原因是api_base为空**。解决思路**逻辑正确**：在 L89 增加条件判断，区分 "llm_client为None" 和 "api_base为空" 两种降级原因，给出不同的日志信息。

## 二、消息构建流程分析

### 2.1 消息构建的核心职责

MessageBuilder负责管理Agent与LLM对话的完整消息列表。核心职责：
1. 管理`conversation_history`（正式历史）和`temp_history`（临时历史）
2. 组装每轮LLM调用时的消息列表（`prepare_messages_for_llm`）
3. 注入运行时动态信息（工具描述、Schema文本）
4. 对话历史裁剪（`trim_history`）
5. Observation文本的构建、截断和规范化

### 2.2 完整消息构建调用链

```
run_stream() 开始:
  message_builder.init_history(sys_prompt, task_prompt)
  → conversation_history = [
      {"role": "system", "content": sys_prompt},
      {"role": "user", "content": task_prompt}
    ]

每轮_call_llm():
  ┌────────────────────────────────────────────┐
  │ prepare_messages_for_llm()                 │
  │   messages = list(conversation_history)     │
  │   if temp_history:                         │
  │     messages += list(temp_history)          │
  │   return messages                          │
  └────────────────────────────────────────────┘
      │
      ▼
  _inject_tools_hint(messages, strategy)  ← text策略时注入工具描述
      │
      ▼
  _inject_schema(messages)  ← text策略时注入Schema参考
      │
      ▼
  LLM调用 ← messages是最终发送给LLM的完整列表
  
LLM返回后:
  parse_react_response LLM响应
  
  [action类型]:
    add_assistant(response)           ← 将LLM完整响应加入history
    build_observation_text(result)    ← 构建observation文本
    add_observation(text, ...)        ← 将observation加入history
      ├─ 预算计算 → 智能截断 → 前缀归一
      └─ trim_history()               ← 裁剪超过阈值的history

  [thought_only类型]:
    add_assistant(response)           ← 将LLM响应加入history
    trim_history()                    ← 裁剪history
  
  [chunk类型]:
    temp_history.append(chunk)        ← 加入临时历史(不进入正式history)
    (保留最近10条)
  
  [answer/implicit/finish类型]:
    flush_temp_to_history(buffer)     ← 将临时历史刷入正式history
    add_assistant(chunk_buffer)       ← LLM完整响应加入history
```

### 2.4 问题分析与发现

#### 问题 MSG-001: temp_history不参与trim_history裁剪，chunk累积可能导致消息列表无限增长

**错误发生原因**: `prepare_messages_for_llm()`将`conversation_history + temp_history`拼接后发给LLM。但`trim_history()`只裁剪`conversation_history`中的内容，`temp_history`不在裁剪范围内。在连续chunk场景下，`temp_history`最多保留10条（`if len(self.message_builder.temp_history) > 10: self.message_builder.temp_history = self.message_builder.temp_history[-10:]`），所以有上限保护。但是如果chunk内容很大（每个chunk几千字符），10条chunk总长度可能达到数万字符，且**不受MAX_CONTEXT_CHARS限制**。

**全分支情况分析**:

| 分支 | 条件 | 问题 |
|------|------|------|
| 分支A | 1-10个chunk后变成action/finish | temp_history清空，无问题 |
| 分支B | 5个chunk后达到max_consecutive_chunks提升为implicit | temp_history清空，无问题 |
| 分支C | 无工具Agent，第1个chunk就退出 | temp_history清空，无问题 |
| 分支D | 10个以上chunk(理论上不会超过max_consecutive_chunks=5) | temp_history保留最近10条 |
| **分支E** | 连续chunk但每个chunk内容巨大(5000+字符) | 10条×5000=50000字符不参与裁剪，可能超出context限制 |

**实际影响**: 连续chunk的场景有`max_consecutive_chunks=5`的上限保护，而且chunk通常较小。分支E在实际中概率较低。但如果LLM持续生成超大chunk且对chunk的解析逻辑改变（放宽阈值），这将成为问题。

**解决思路**: 在`prepare_messages_for_llm()`中或`trim_history()`中对temp_history也进行容量限制，或者计算total chars时包括temp_history。

> **v1.2代码验证（小资）**: 问题**确认真实但风险低**。代码路径：`message_builder.py` L122-124 `prepare_messages_for_llm()` 拼接 `conversation_history + temp_history`，`trim_history()` L163-222 只裁剪 `conversation_history`。temp_history 的 10 条上限保护在 `base_react.py` 中（非 MessageBuilder 内部），且 `max_consecutive_chunks=5` 限制连续 chunk 数。解决思路**逻辑正确**，推荐方案：在 `prepare_messages_for_llm()` 中计算 total chars 时包含 temp_history，超限时从 temp_history 头部裁剪。

#### 问题 MSG-003: FC协议下observation的裁剪识别不完全

**错误发生原因**: `_is_observation_role`方法用于识别哪些消息是observation。对于FC协议路径，observation以`role=tool`+`tool_call_id`的形式存在。`trim_history`通过`_is_observation_role`识别并只裁剪这些observation。但如果FC路径的`tool_call_id`为空或缺失（异常情况），`_is_observation_role`返回False，该observation不会被识别，也不会被裁剪。

**全分支情况分析**:

```python
@staticmethod
def _is_observation_role(msg: Dict) -> bool:
    # FC路径: role=tool + tool_call_id存在(且非空)
    if msg.get("role") == "tool" and msg.get("tool_call_id"):
        return True
    # Text路径: role=system + 含有[Observation]或Observation:标记
    content = msg.get("content", "")
    return msg.get("role") == "system" and (
        "[Observation]" in content or "Observation:" in content
    )
```

| 分支 | role | tool_call_id | content | 识别为observation? | 可裁剪? |
|------|------|-------------|---------|-------------------|---------|
| 分支A | tool | "call_xxx" | 任意 | ✅ 是 | ✅ 可裁剪 |
| 分支B | tool | "" | 任意 | ❌ 否(id为空字符串) | ❌ 不可裁剪，永久保留 |
| 分支C | tool | None | 任意 | ❌ 否(id为None) | ❌ 不可裁剪，永久保留 |
| 分支D | system | — | "[Observation] xxx" | ✅ 是 | ✅ 可裁剪 |
| 分支E | system | — | "Observation: xxx" | ✅ 是(兼容旧格式) | ✅ 可裁剪 |
| 分支F | system | — | "普通文本" | ❌ 否 | ❌ 不可裁剪 |

**分支B/C的问题**: 如果在FC协议中，`add_observation`创建了`role=tool`消息但`tool_call_id`为空，这条消息会永久留在history中，因为`trim_history`无法识别它为observation。FC协议路径在`add_observation`中（message_builder.py L77-81）：

```python
if fc_context and fc_context.get("tool_calls"):
    tool_calls = fc_context["tool_calls"]
    tool_call_id = fc_context.get("tool_call_id", "")
    self.conversation_history.append({
        "role": "assistant", "content": None, "tool_calls": tool_calls
    })
    self.conversation_history.append({
        "role": "tool", "content": observation_text, "tool_call_id": tool_call_id
    })
```

如果`fc_context["tool_call_id"]`是空字符串，`role=tool`消息的`tool_call_id`是`""`，`msg.get("tool_call_id")`返回`""`（Python中空字符串也是"存在"的，但bool值为False...）。Wait，`msg.get("tool_call_id")`当值为`""`时返回`""`，空字符串的bool为False，所以`if msg.get("role") == "tool" and msg.get("tool_call_id"):`中`msg.get("tool_call_id")`为`""` → False → 整个条件为False → 不识别为observation。

这就是一个实际的问题！

**实际影响**: 
```python
# base_react.py L826-827
fc_context = {"tool_calls": _chat_response.tool_calls, "tool_call_id": tc.get("id", "")}
```

`tc.get("id", "")`在id存在时返回id值（非空字符串），在id不存在时返回`""`。所以如果LLM返回的tool_calls中没有id字段，`tool_call_id`就是空字符串，这条tool消息永久驻留在history中。

**解决思路**: 
- `_is_observation_role`中加强检测：`if msg.get("role") == "tool"`即为observation，不必要求tool_call_id非空
- 或者在`add_observation`时确保tool_call_id不为空

#### 问题 MSG-004: trim_history中FC协议配对裁剪可能断裂

**错误发生原因**: `_trim_fc_pairs`要求`role=tool`消息的`tool_call_id`和`role=assistant(tool_calls)`消息中的`tool_calls[*].id`严格配对。如果observation去重或移除时，一对FC消息中只有一方被保留，整个配对被移除。

**全分支情况分析**:

`trim_history`中，observation列表被去重后只保留最近的30条。如果30条中最老的FC对中，assistant(tool_calls)在最近10条assistant中但对应的tool不在最近30条observation中（或反之），破损的配对被`_trim_fc_pairs`整体移除。

```
假设history 40条消息：
  system + user + 
  [FC对1: assistant + tool] + 
  [FC对2: assistant + tool] + 
  ... +
  [FC对20: assistant + tool]

trim_history:
  保留最近的30条observation → FC对1-10被移除
  保留最近的10条assistant → assistant_11-20保留
  
  _trim_fc_pairs:
    检查FC对11-20是否配对完整
    → 完整 → 保留
    → 不完整（assistant在但tool不在，或反之）→ 全部移除
```

**实际影响**: `_dedup_by_fingerprint`已经跳过了FC协议消息（`role=tool`+`tool_call_id`不参与去重），FC配对被整体保留或整体移除。问题不大，但需要注意的是，移除一个完整的FC对意味着损失了一轮工具调用的完整上下文。

**解决思路**: 可以接受。FC对被整体保留或整体移除是合理的——比保留半个FC对好。

### 3.5 问题分析与发现

#### 问题 PRM-001: 工具列表在System Prompt中硬编码与运行时注入重复

**错误发生原因**: System Prompt中每个分类的`get_system_prompt()`都硬编码了完整的可工具列表（工具名称、参数、示例）。同时，`_call_llm()`的`_inject_tools_hint()`又从tool_registry动态生成工具描述注入到消息中。LLM在同一次对话中看到**两次工具列表**：一次在system prompt头部，一次在runtime注入位置。

```
最终发送给LLM的消息结构(text策略):
  messages = [
    {"role": "system", "content": "[系统信息] + [工具列表-硬编码] + [OUTPUT_FORMAT] + [TOOL_CALL_RULES] + ..."},
    {"role": "system", "content": "[工具详情-动态注入] + [其他工具概要]"},
    {"role": "system", "content": "[Schema参考文本]"},
    {"role": "user", "content": "Task: ..."},
    ... 后续对话历史 ...
  ]
```

**全分支情况分析**:

| 分支 | 条件 | 问题 |
|------|------|------|
| 分支A | 工具列表完全一致 | LLM看到重复信息，浪费token |
| 分支B | 硬编码过期但registry更新 | LLM看到冲突信息，可能混淆 |
| 分支C | tools策略(text策略不注入) | tools策略不会注入动态工具描述，只有硬编码列表 |
| 分支D | 动态加载新分类后 | 新分类的工具只有动态注入中有，硬编码中无 |

**分支B的具体问题**: 例如`FileOperationPrompts`中硬编码"共11个文件工具"，但实际注册的可能更多或更少（如新增了`read_media_file`后变成12个）。如果忘记更新硬编码的"11个"，LLM会收到矛盾信息（提示说11个但实际列出12个）。

**分支D的具体问题**: 当Agent通过`load_tools_by_intent`动态加载了系统工具后，动态注入会包含新工具，但hardcoded system prompt中仍然只有文件工具。LLM可能困惑"为什么system prompt中列举的工具和动态注入的不一致"。

**解决思路**:
- 移除硬编码的工具列表，完全由运行时动态注入。这需要一个巨大的prompt改造。
- 或者确保每次工具注册变更时同步更新prompt（容易遗漏）。
- 折中方案：硬编码保持核心工具示例（不列举全部，只给示例），完整列表由动态注入提供。

---

#### 问题 PRM-002: System Prompt中的Example与动态工具列表可能不同步

**错误发生原因**: system prompt中的Tool Call Examples（如`file_prompts.py L142-152`）是静态JSON示例。如果工具的必需参数变化，examples可能过期，LLM按照过期示例生成的JSON可能被`_normalize_params`拒绝。

**具体例子**: 
```python
# file_prompts.py L143
{"thought": "用户要读取配置文件", "reasoning": "调用read_file单文件模式", 
 "tool_name": "read_file", "tool_params": {"file_paths": ["C:/config.json"]}}
```
如果`read_file`工具的参数名从`file_paths`改为`paths`，但prompt中的example没更新，LLM会按照旧参数名生成JSON，`_normalize_params`检查到`file_paths`不是合法参数（通过input_schema校验），将其删除，最终工具执行的参数为空。

**全分支情况分析**:

| 分支 | Example中的参数名 | Registry中的参数名 | 结果 |
|------|-------------------|-------------------|------|
| 分支A | `file_paths` | `file_paths` | ✅ 匹配，正常执行 |
| 分支B | `file_paths` | `paths` | ❌ `file_paths`被`_normalize_params`删除，参数为空 |
| 分支C | `file_paths` | `file_paths`(必需) + `encoding`(可选) | ✅ 必需参数存在，正常 |
| 分支D | 无example | `file_paths`(必需) | LLM可能用错的参数名 |

**解决思路**: 工具的Example应该从tool_meta或tool_schema中自动生成，而不是手写硬编码。

#### 问题 PRM-003: `_build_system_prompt`的candidates_hint和cross_tool_hint在非首次LLM调用时也重复注入

**错误发生原因**: `_build_system_prompt`在每轮Agent的`run_stream()`开始时调用一次，结果拼入`init_history`的system prompt。但**这个system prompt在整个run_stream的生命周期中不变**。候选意图和跨分类提示在system prompt中写入一次后就固定了，不会重复注入。所以这不是重复问题。

但是——这里有一个更微妙的问题：`_build_candidates_hint`的内容是基于`self._candidates`在Agent初始化时设置的。如果后续通过`load_tools_by_intent`加载了新工具，`_candidates`不会更新。在后续Observation中加载新工具后，system prompt中仍然只有最初的候选意图。

**全分支情况分析**:

```
初始化时candidates=["file", "system"]:
  system prompt = "...【候选意图】已识别出以下可能的意图类别: file, system..."

后续observation → load_tools_by_intent("network"):
  self._loaded_categories = {"file", "network"}  # 更新了
  self._candidates = ["file", "system"]          # 没更新！
  # system prompt中的candidates_hint仍然说"file, system"
  # 但实际已加载了"network"工具
```

**实际影响**: 候选意图提示仅作为"建议"性质，不影响功能正确性。LLM仍然可以从动态注入的工具描述中知道`network`工具可用。

## 四、Observation注入流程与裁剪机制

### 4.1 Observation的整体数据流

```
工具执行结果
  │
  ▼
build_observation_text(execution_result)  ← MessageBuilder@staticmethod
  └─ _format_llm_observation(execution_result)  ← tool_result_formatter.py
       ├─ code == "SUCCESS"         → "Observation: success - {message}"
       │    └─ 有display_data(data/llm_data) → 追加json.dumps数据
       │    └─ 有warning → 追加警告
       │    └─ 有next_actions → 追加推荐操作
       │
       ├─ code.startswith("WARNING_") → "Observation: warning - {message}"
       │    └─ 有data → 追加部分数据
       │    └─ 有next_actions → 追加推荐操作
       │
       └─ 其他(ERR_*等) → "Observation: error [{code}] - {message}"
  
  ▼
add_observation(observation_text, llm_call_count, fc_context)
  ┌─ 1. 预算计算: _get_observation_budget(llm_call_count)
  │     budget = 20000 + 10000 * max(0, 5 - call_count)
  │     前5轮递减: 第1轮=70000, 第2轮=60000, ..., 第5轮=30000, 第6轮=20000
  │
  ├─ 2. 智能截断: 超过预算时_smart_truncate(text, budget)
  │     head_ratio=0.6, head保留60%, tail保留剩余(减省略标记), 中间省略
  │
  ├─ 3. 前缀归一: _normalize_observation_prefix(text)
  │     确保以"[Observation] "开头
  │     兼容旧格式"Observation:"和"observation:"
  │
  └─ 4. 分别按FC/text协议路径注入
       ├─ text路径:
       │    conversation_history.append({"role":"system", "content":observation_text})
       │
       └─ FC路径:
            conversation_history.append({"role":"assistant", "content":None, "tool_calls":calls})
            conversation_history.append({"role":"tool", "content":text, "tool_call_id":id})
  
  ▼
  trim_history()  ← 容量感知裁剪
```

### 4.2 预算计算与衰减机制

```
_get_observation_budget(llm_call_count) → budget
  公式: min(20000 + 10000 * max(0, 5 - llm_call_count), 50000)
  注意: 代码中有 min(budget, 50000) 上限约束（message_builder.py L275）

轮次   | 原始公式值 | 实际预算(经min cap) | 说明
1      | 70000     | 50000              | 首轮被cap到50000，不是70000
2      | 60000     | 50000              | 同样被cap到50000
3      | 50000     | 50000              | 刚好等于上限
4      | 40000     | 40000              | 不受cap影响
5      | 30000     | 30000              | 不受cap影响
6+     | 20000     | 20000              | 稳定在最小值
```

> **v1.2代码验证修正（小资）**: 文档v1.1中预算表格第1-2轮的70000/60000是错误的，实际代码 `message_builder.py` L274-275 有 `return min(budget, 50000)` 上限，第1-3轮预算均为50000。这显著降低了OBS-002的严重性——大文件在第1轮也只能展示50000字符而非70000。

### 4.3 智能截断机制

```
_smart_truncate(content, budget, head_ratio=0.6):
  head_budget = budget * 0.6        ← 头部保留60%
  tail_budget = budget - head - 50  ← 尾部保留剩余(减省略标记开销)
  
  结果:
    head部分 + "中间省略N字符" + tail部分
```

### 4.4 容量感知裁剪机制

```
trim_history() 触发条件:
  total_chars > MAX_CONTEXT_CHARS * 0.8 (= 120000)
  (延迟裁剪优化：不到80%阈值跳过)

裁剪步骤:
  1. 分离: system_msgs(保留) + obs_list(裁剪对象) + assistant_msgs(保留)
  2. observation去重: _dedup_by_fingerprint(MD5前16位)
  3. assistant保留: 最近10条
  4. observation保留: 最近30条(然后从最旧开始移除直到满足预算)
  5. FC配对裁剪: _trim_fc_pairs(确保tool<->assistant配对完整)
  6. 兜底保护: 如果裁剪后<2条，保留前2条+后8条

目标预算: MAX_CONTEXT_CHARS * 0.7 = 105000
```

### 4.5 问题分析与发现

#### 问题 OBS-001: `llm_data`和`data`的优先级可能导致LLM看到空数据

**错误发生原因**: `_format_llm_observation`中，给LLM的observation优先使用`llm_data`，其次使用`data`。如果工具返回了`llm_data=None`且`data`也不存在，LLM看到的observation是只包含"success - {message}"的文本，没有具体数据。这可能导致LLM无法基于结果做出正确决策。

**全分支情况分析**:

```python
# tool_result_formatter.py L88
display_data = result.get("llm_data") or result.get("data")
```

| 分支 | llm_data | data | LLM看到的数据 |
|------|----------|------|-------------|
| 分支A | {"files": [...], "total": 10} | — | llm_data内容 ✅ |
| 分支B | None | {"files": [...], "total": 10} | data内容 ✅ |
| 分支C | None | None | 无数据(只有success消息) |
| 分支D | None | "string data" | "string data" ✅ |
| 分支E | ""(空字符串) | 有数据 | `"".strip()`... wait, `"" or {...}` → 返回 {...}` 因为空字符串的bool为False |

等等，让我重新检查：`result.get("llm_data") or result.get("data")` — 如果`llm_data`是`None`则取`data`。但如果`llm_data`显式设置为了`None`，`None or ...`取后者。但如果`llm_data`是`{}`(空字典)，`{} or ...`取后者。所以**只有llm_data和data都是None/空/不存在时**才会没有数据。

**分支C的问题**: 如果工具实现者忘了返回data（如`return {"code": "SUCCESS", "message": "操作完成"}`），LLM只看到"success - 操作完成"，没有具体结果。这可能影响后续决策，但不是崩溃性错误。

**解决思路**: 在`_format_llm_observation`中增加日志，当既没有llm_data也没有data时给出warning提示，帮助开发者发现遗漏。

#### 问题 OBS-002: 预算衰减到20000后，大型工具结果被严重截断

**错误发生原因**: 第6轮后observation预算固定为20000字符。如果某工具返回了大量数据（如`read_file`读取10万字符的文件），只有前12000字符（60%）和后7950字符（剩余）被保留，中间76000字符被省略。LLM看到的内容是断裂的，可能误解文件内容。

**全分支情况分析**:

```
假设第6轮，budget=20000:
  head = 20000 * 0.6 = 12000 字符
  tail = 20000 - 12000 - 50 省略标记 = 7950 字符

文件内容100000字符:
  Observation: success - 已读取文件

  数据: {文件头12000字符}... [中间省略 88000 字符] ...{文件尾7950字符}

注意: 第1-3轮budget实际为50000(受min cap限制)，非原始公式计算的70000/60000/50000。
```

| 轮次 | budget(实际) | head(60%) | tail | 对大文件的展示能力 |
|------|-------------|-----------|------|-------------------|
| 第1轮 | 50000 | 30000 | 19950 | 良好，展示大部分文件 |
| 第2轮 | 50000 | 30000 | 19950 | 良好 |
| 第3轮 | 50000 | 30000 | 19950 | 良好 |
| 第4轮 | 40000 | 24000 | 15950 | 一般 |
| 第5轮 | 30000 | 18000 | 11950 | 较差 |
| 第6+轮 | 20000 | 12000 | 7950 | 差，大文件严重截断 |

**实际影响**: 后几轮LLM无法完整看到大型工具结果，可能基于不完整信息做错误判断。这是token预算的硬约束，需要在产品层面告知用户。

**解决思路**:
- 根据文件大小和已读轮次动态调整budget
- 让工具通过llm_data字段控制给LLM的数据量（如read_file在大文件时只返回摘要）
- 保持现状并在设计文档中说明预算约束

#### 问题 OBS-003: `_smart_truncate`在budget太小时tail可能为空

**错误发生原因**: 当`head_budget`接近`budget`时（head_ratio=0.6），`tail_budget = budget - head_budget - 50`可能<=0，导致tail部分为空。

**全分支情况分析**:

#### 问题 OBS-004: （已核实非实际漏洞）前缀归一化已有`.strip()`保护，不会产生双空格

**实际代码验证结果——非实际漏洞**: `_normalize_observation_prefix`中，去掉旧前缀后调用了`.strip()`：


**解决思路**: 无需修复。`.strip()`已正确处理。

### 5.4 问题分析与发现

#### 问题 LLM-001: TextStrategy和ToolsStrategy在API错误时的处理策略不一致

**错误发生原因**: 两者的错误处理策略完全不同：

| 场景 | TextStrategy | ToolsStrategy |
|------|-------------|---------------|
| API返回error | 返回tool_name="finish" → **退出循环** | 返回type="parse_error" → **重试** |
| 空响应 | 同上，退出循环 | fallback到TextStrategy |
| HTTP错误 | 同上，退出循环 | 同上，返回parse_error |

**TextStrategy退出循环的行为分析**:
```python
# llm_strategies.py L100-113
if error_info:
    return self._make_result(
        content=f"[错误] {error_hint}",
        tool_name="finish",           # ← 这里直接 finish
        tool_params={"result": f"[错误] {error_hint}"}
    )
# 无error的空内容:
return self._make_result(
    content=f"⚠️ {user_message}",
    tool_name="finish",               # ← 也是 finish
    tool_params={"result": f"⚠️ {user_message}"}
)
```

**ToolsStrategy重试的行为分析**:
```python
# llm_strategies.py L432-444
if hasattr(response, 'error') and response.error:
    error_msg = response.error
    return json.dumps({
        "type": "parse_error",        # ← parse_error，base_react会重试
        "error": error_message,
        ...
    })
```

**全分支情况分析**:

| 分支 | 策略 | 错误场景 | 行为 | 结果 |
|------|------|---------|------|------|
| 分支A | text | API限流429 | finish退出 | Agent结束，用户看到错误 |
| 分支B | text | 连接超时 | finish退出 | Agent结束 |
| 分支C | text | 5xx服务错误 | finish退出 | Agent结束 |
| 分支D | tools | API限流429 | parse_error重试(最多3次) | Agent继续尝试 |
| 分支E | tools | 连接超时 | parse_error重试(最多3次) | Agent继续尝试 |
| 分支F | tools | 5xx服务错误 | parse_error重试(最多3次) | Agent继续尝试 |

**问题**: text策略在API短暂不可用时直接放弃（分支A-D），而tools策略会重试（分支E-H）。这意味着如果text策略遇到网络抖动，用户看到的是"AI服务错误，已退出"，失去了自动恢复的机会。而tools策略在同样的场景下会重试并可能恢复。

**解决思路**: text策略应返回parse_error而非finish，让base_react统一管理重试。或者text策略在底层传输层（baseAIService的`_post_with_retry`）已经做了3次429重试，3次都失败后表示网络确实不可用，此时退出也是合理的。但tools策略在`_post_with_retry`重试3次后如果还失败，返回的parse_error会被base_react再重试3次，导致**总共6次重试**。

#### 问题 LLM-002: ToolsStrategy在fallback到TextStrategy时，消息中没有工具信息

**错误发生原因**: 当ToolsStrategy检测到`llm_client`没有`chat_with_tools`方法，或FC请求返回空响应时，会fallback到TextStrategy：

```python
if not hasattr(llm_client, 'chat_with_tools'):
    text_strategy = TextStrategy()
    return await text_strategy.call(
        llm_client=llm_client,
        messages=messages,           # ← 这里messages是原始消息
        ...
    )
```

但在`_call_llm`中，如果策略是`tools`，`_inject_tools_hint`被跳过：

```python
def _inject_tools_hint(self, history_dicts, strategy_method):
    if strategy_method != "tools":     # ← tools策略时不注入
        detail = self._get_tools_detail()
        ...
```

所以fallback到TextStrategy时，`messages`中**没有工具描述信息**。TextStrategy收到后，LLM看到的只有OUTPUT_FORMAT规则（说"必须使用tool_name和tool_params"），但**不知道有哪些工具可用**。LLM可能生成"finish"（因为不知道工具名），也可能生成错误的工具名。

**全分支情况分析**:

```
工具描述注入: initial_messages中无工具描述

ToolsStrategy.call():
  llm_client.chat_with_tools(messages, tools=openai_tools)
  → 但llm_client没有chat_with_tools方法！
  → fallback到TextStrategy.call(llm_client, messages, ...)
  → 这里的messages中没有工具描述！
  
  TextStrategy内部:
    → llm_client.chat(messages)
    → LLM回复纯文本(不知道有任何工具)
    → parse_react_response → type=chunk或answer
    → 不会调用工具
```

| 分支 | fallback原因 | messages中有工具信息? | LLM行为 |
|------|-------------|---------------------|---------|
| 分支A | 无chat_with_tools方法 | ❌ 无 | 纯文本输出，不调用工具 |
| 分支B | FC请求返回空响应 | ❌ 无 | 同上 |
| 分支C | FC请求返回error后 | ❌ 无 | 返回parse_error(不是fallback) |

**实际影响**: 理论上这个分支不会发生——因为`adapter.detect_strategy()="tools"`的前提是LLM支持FC，此时llm_client一定有chat_with_tools方法。但如果在探测和第一次tools调用之间发生了llm_client的变化（极不可能），或者chat_with_tools方法被动态替换，就会出现此问题。

**解决思路**: fallback到TextStrategy前，手动注入工具描述。或者在`_call_llm`中，对tools策略也做工具描述注入（但标记为"仅fallback时使用"）。

#### 问题 LLM-003: chat()非流式调用内部使用流式API，存在性能浪费

**错误发生原因**: `BaseAIService.chat()`内部调用`chat_stream()`，但Agent场景（text策略）只需要最终结果，不需要流式中间过程。每次text策略的LLM调用都要遍历整个SSE流并聚合，比直接使用非流式API多消耗了协议解析和处理开销。

**全分支情况分析**:

```
策略确定后:_call_llm():
  └─ _dispatch_strategy("text", messages)
       └─ text_strategy.call(llm_client, messages, ...)
            └─ await llm_client(message="", history=messages)
                 └─ BaseAIService.__call__() → chat()
                      └─ chat(message, history)
                           └─ 内部: chat_stream()  ← 流式API
                                ├─ POST {api_base}/chat/completions?stream=true
                                ├─ 逐行读取 SSE 数据(data: {...})
                                ├─ 解析 JSON delta → 聚合 content
                                ├─ 处理 reasoning_content
                                └─ 返回聚合后的 ChatResponse
```

| 方法 | stream参数 | 适用场景 | Agent场景效率 |
|------|-----------|---------|--------------|
| chat() | 内部用stream=true | 流式/非流式都支持 | ❌ 浪费，不需要流式 |
| chat() with stream=false | 用非流式 | 仅非流式 | ✅ 直接返回完整结果 |

**实际影响**: 对大部分LLM API来说，流式和非流式的性能差异不大（都是单次HTTP请求）。SSE解析开销很小。性能影响主要在：
1. 额外的JSON解析（每行SSE都做一次`json.loads`）
2. 额外的event loop调度（每1秒超时检查取消标志）

对于典型Agent对话（5-15轮），这种性能损耗可以忽略。但如果Agent在极端场景（50+轮），累积开销可能明显。

**解决思路**: 
- 为text策略添加非流式路径（stream=false），减少解析开销
- 或者保留现状——agent场景的流式支持提供了更好的取消能力（1秒内响应取消），这是非流式API难以做到的

---

#### 问题 LLM-004: chat_with_tools的取消检查使用后台任务+心跳，但存在取消后响应无法立即返回的窗口

**错误发生原因**: `chat_with_tools`使用`asyncio.ensure_future` + `asyncio.wait_for(..., timeout=1.0)` + 每隔1秒检查`self._cancelled`。取消响应最多有1秒延迟。

```
chat_with_tools:
  request_task = ensure_future(_post_with_retry(...))
  
  while not request_task.done():
    try:
      await asyncio.wait_for(asyncio.shield(request_task), timeout=1.0)
    except asyncio.TimeoutError:
      if self._cancelled:                # 取消检查
        request_task.cancel()            # 取消HTTP请求
        return ChatResponse(error="任务已取消")
      continue                           # 未取消，继续等待
  
  response = await request_task           # 请求完成
  # 此时如果刚取消，但request_task.done()为True
  # 上面if self._cancelled不会进入
  # 而是走到这里，处理response
```

**窗口问题**: 如果取消发生在`request_task.done()`变为True之后，但在`response = await request_task`之前，代码不会检查`self._cancelled`，而是直接处理response。之后才返回给调用方。

**全分支情况分析**:

```
时间线:
t0: _post_with_retry 发出HTTP请求
t1: 服务器返回响应 → request_task.done() = True
t2: cancel()被调用 → self._cancelled = True
t3: wait_for超时检查 → request_task.done() = True → 跳出循环
t4: response = request_task.result() → 得到response
t5: 处理response.json()
t6: 返回ChatResponse(content=tool_calls)
```

在t2-t6期间，即使取消已经发生，`chat_with_tools`仍会完整处理响应并返回tool_calls，而不是"任务已取消"。

**实际影响**: 取消响应最多延迟一个心跳周期（1秒）+ 响应处理时间（~100ms）。返回的tool_calls会在base_react的"LLM返回后中断检查"中被拦截（`base_react.py L405-416`），所以最终表现是：
1. chat_with_tools返回tool_calls ✅
2. base_react检测到`task_data.get("cancelled")`为True ✅
3. yield interrupted_data并return ✅

所以**取消最终会被正确处理**，只是响应延迟了最多1秒。这不是功能性缺陷，而是用户体验问题。

**解决思路**: 在循环退出后、处理response前再加一次`if self._cancelled`检查。但即使这样，如果取消发生在`response = request_task.result()`和`return ChatResponse(...)`之间，仍然是竞争条件。asyncio无法做到立即取消外部HTTP请求。

#### 问题 LLM-005: chat_stream的取消标志检查在每次line超时(1秒)时进行，高延迟场景下取消响应慢

**错误发生原因**: `chat_stream`使用`asyncio.wait_for(line_iterator.__anext__(), timeout=1.0)`，每1秒超时一次检查`self._cancelled`。如果API返回的流很长（如1000+行），取消最快响应时间是1秒。但如果API返回流很慢（如长思考模型的SSE间隔>1秒），每个line之间可能间隔数秒，此时取消响应时间在1秒到下一个line到达时间之间。

```
chat_stream 取消响应时间 = max(1秒, SSE行间隔)

最坏情况:
  LLM正在生成长文本，SSE间隔3秒
  t0: cancel()被调用
  t1: wait_for超时(1秒后)
  t2: 检查_cancelled → True → 立即退出
  总延迟: ~1秒

极端情况:
  如果SSE流在cancel后立即进入了一个长间隔(如thinking模型思考10秒)
  但wait_for的超时确保最晚1秒后就会检查取消
  所以极端情况也是~1秒
```

**实际影响**: 取消响应延迟最坏约1秒，对用户来说可以接受。

#### 问题 LLM-006: `chat_with_tools`非流式路径的httpx client配置`read=None`，LLM慢响应时无read超时保护

**错误发生原因**: `BaseAIService.__init__`中httpx client的read timeout显式设为`None`：

```python
# llm_core.py L167-173
self.client = httpx.AsyncClient(
    timeout=httpx.Timeout(
        connect=30.0,
        read=None,        # ← read timeout为None!
        write=10.0,
        pool=10.0,
    ),
    ...
)
```

`chat_with_tools`内部调用`_post_with_retry`（L576），其核心是`self.client.post()`（L237）——**没有透传timeout参数**。当LLM响应缓慢时（如thinking模型的首次token可能需要30-60秒），`client.post()`会无限等待，因为`read=None`意味着httpx在等待读取HTTP响应体时不做任何超时判断。

虽然`chat_with_tools`的心跳循环（`wait_for(request_task, timeout=1.0)`）每隔1秒检查取消标志，但这**不限制HTTP请求本身的持续时间**——`_post_with_retry`作为一个后台任务仍在运行，只是心跳在等待它完成。

**关键区别**：

| 机制 | 作用 | 局限性 |
|------|------|--------|
| 心跳`wait_for(..., 1.0)` | 检查取消标志，1秒内响应cancel | ❌ 不限制请求时长 |
| httpx `read=None` | 不对读操作设超时 | ❌ 无限等待 |
| `_post_with_retry`的429重试 | 处理限流 | ❌ 不影响正常请求 |

**全分支情况分析**:

| 分支 | LLM响应速度 | `_post_with_retry`行为 | 心跳循环行为 | 用户感知 |
|------|------------|----------------------|-------------|---------|
| 分支A | <1秒 | 快速完成 | 第1次wait_for看到done | 正常 ✅ |
| 分支B | 1-30秒 | client.post等待 | 每1秒超时→检查取消→继续等待 | 等待中，可取消 ✅ |
| 分支C | 60+秒 | client.post无限等待 | 每1秒超时→检查取消→继续等待 | 等待长，可取消 ⚠️ |
| 分支D | 永不返回（LLM卡死） | **client.post无限等待** | 每1秒检查取消，若不取消则**永远等下去** | Agent永久挂起 🔴 |

**分支D的严重性**: 如果LLM server因故障、网络断开等原因不返回响应，Agent会永久挂起在`chat_with_tools`调用中。因为没有read timeout，httpx不会主动断开连接。唯一的退出方式是外部取消（用户点击取消按钮 → 设置`_cancelled=True` → 心跳检测到→返回取消）。如果外部取消也失败（如UI线程卡死），Agent永远无法恢复。

**解决思路**:
- 方案A: 将`read`设为具体值（如60秒），确保HTTP层有最大等待上限
- 方案B: 在`chat_with_tools`中增加一个"最大等待时间"限制，超时后返回error
- 方案C: 在`_post_with_retry`中为`client.post()`传递`timeout=httpx.Timeout(read=60)`覆盖全局配置
