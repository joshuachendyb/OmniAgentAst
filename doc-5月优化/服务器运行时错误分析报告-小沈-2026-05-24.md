## 二、Q1：404 会话不存在

### 2.1 现象

```
save_session: HTTP error: 404, detail=会话不存在: xxxxxx
```

日志中几乎所有 session_id 都返回404，贯穿整个运行过程。

### 2.2 代码追踪

**前端流程**（已完成）:

| 步骤 | 操作 | HTTP调用 | 文件:行号 |
|------|------|----------|----------|
| 1 | 创建session(首次) | POST /api/v1/sessions | useChatSend.ts:117-126 |
| 2 | saveMessage(用户消息) | POST /sessions/{id}/messages | useChatStreaming.ts:197-204 |
| 3 | SSE POST(带session_id) | POST /chat/stream/v2 | sse.ts:685 |
| 4 | saveMessage返回404 → 清空sessionId | - | useChatStreaming.ts:227-233 |
| 5 | SSE继续(无session_id) | - | sse.ts:685 (sessionId: undefined) |
| 6 | 后端生成随机UUID替代 | - | chat_router.py:207 |
| 7 | 所有后续save_execution_steps全部404 | - | 因随机UUID不存在 |

**后端session创建**（已完成）:

| 组件 | DB路径 | 初始化方式 |
|------|--------|----------|
| chat_db.py | `Path.home() / ".omniagent" / "chat_history.db"` | `CREATE TABLE IF NOT EXISTS` |
| sessions.py create_session | 同上 | `INSERT INTO chat_sessions (id, title, ...)` |
| sessions.py save_message | 同上 | `SELECT ... WHERE id = ? AND is_deleted = FALSE` |

### 2.3 根因分析

**代码层面结论：代码逻辑正确，不是代码bug。**

`create_session` 和 `save_message` 使用同一份 SQLite 文件（`C:\Users\chend\.omniagent\chat_history.db`）。
创建→保存→SSE 的时序正确（先创建，再保存，最后SSE）。

**404 的可能原因**：

| 原因 | 概率 | 说明 |
|------|------|------|
| 配置的`baseURL`与后端端口不匹配 | 中 | 前端请求了错误的端口/服务，`createSession`和`saveMessage`去了不同的后端实例 |
| 服务器重启导致session丢失 | 低 | SQLite文件持久化，重启后session仍在 |
| 多实例部署 | 中 | 如果有多个后端实例（如热重载），`createSession`和`saveMessage`可能去了不同实例 |
| DB文件被重建 | 低 | `CREATE TABLE IF NOT EXISTS` 不会覆盖已有数据 |

### 2.4 结论

**⚠️ 不是代码bug，但影响最严重**（所有聊天记录丢失）。需要运行时排查以下2点：
1. 确认 `~/.omniagent/chat_history.db` 是否存在并包含已创建的session
2. 检查前端 `config.baseURL` 是否指向正确的后端端口

## 三、Q2：400 ThinkingMode 空 tool_calls

### 3.1 现象

```
400 ... invalid message without reasoning_content/tool_calls 
`{"tool_calls": [], "content": '{"thought": "Calling tool: get_time", ...}', "role": "assistant"}`
after last user message
```

LLM API 返回400错误，拒绝请求，原因是消息中出现了 `tool_calls: []`。

### 3.2 代码追踪

**问题定位路径**（已完成全链路追踪）：

| 追踪点 | 文件:行号 | 结论 |
|--------|----------|------|
| `add_assistant(content)` | message_builder.py:63 | ✅ 添加 `{"role": "assistant", "content": content}`，无tool_calls键 |
| `add_observation(fc_context)` | message_builder.py:77-83 | ✅ 只有非空tool_calls才进入FC分支(line 80)，空 `[]`走else → 加system消息 |
| `base_react.py:861 _strategy检查` | base_react.py:861-868 | ✅ 修复后：仅 `_strategy=="tools"` 时才检查 `_last_fc_raw_response` |
| `_dispatch_strategy()` | react_agent_mixin.py:306-309 | ✅ text策略时 `_last_fc_raw_response=None`（防止跨策略污染） |
| `chat_with_tools()` | llm_core.py:648-655 | ✅ `tool_calls = msg.get("tool_calls", [])` → 空则走else，ChatResponse不设tool_calls |
| `_trim_fc_pairs()` | message_builder.py:337-368 | ✅ 只过滤消息，不修改内容 |

**唯一可能产生 `tool_calls: []` 的旧代码路径**（已修复）：

```
旧代码（2026-05-24前）：
1. Round N: strategy="tools" → LLM返回tool_calls(非空) → _last_fc_raw_response.set
2. Round N+1: strategy不经过_dispatch_strategy或_策略检查不完整
   → _last_fc_raw_response.tool_calls仍保留上一轮的tool_calls
   → add_observation 使用旧tool_calls → 添加FC协议消息到conversation_history

修复 commit 50d46f35:
- base_react.py:861 增加 _strategy检查：仅tools策略才用_last_fc_raw_response
- react_agent_mixin.py:309 text策略时清空 _last_fc_raw_response
```

### 3.3 根因分析

**当前代码中已无产生 `tool_calls: []` 的路径。** `message_builder.py:80` 仅在 `fc_context.get("tool_calls")` 为真值（非空列表）时才添加 `{"role": "assistant", "content": None, "tool_calls": [...]}`。

空列表 `[]` 是 Python 的 falsy 值，不会进入该分支。

**400错误的余下可能原因**：

| 原因 | 说明 |
|------|------|
| 历史数据污染 | conversation_history中保留着旧代码存储的消息（含tool_calls: []） |
| LLM API供应商的特殊校验 | 某些供应商（如DeepSeek）对空tool_calls有额外校验 |
| response_format路径 | 其他策略路径可能产生类似问题（已不在当前代码中） |

### 3.4 结论

**✅ 已修复（旧代码bug）。** 当前代码已通过 `_strategy` 检查和 `_last_fc_raw_response` 清空机制，消除了 `tool_calls: []` 被写入 conversation_history 的路径。

如果实际运行时仍出现400错误，建议：
1. 在 `_build_messages` 或 `chat_with_tools` 中加日志：`logger.debug(f"messages: {json.dumps(request_json['messages'], ensure_ascii=False)[:2000]}")`
2. 确认是否从旧DB加载了含 `tool_calls: []` 的历史消息

## 四、Q3：无效工具名

### 4.1 现象

```
没有工具 '具体工具名'
```

### 4.2 代码追踪

| 追踪点 | 文件:行号 | 结论 |
|--------|----------|------|
| `tool_registry.get_tool(name)` | registry.py:734 | ✅ 模块级单例，注册了所有工具 |
| `list_tools()` | registry.py:457-464 | ✅ 返回 `List[Dict]`，含 `"name"` 键 |
| `base_react.py` 工具名称验证 | base_react.py 工具调用前 | ✅ 验证逻辑正确工作 |

### 4.3 根因分析

**这不是bug，是预期行为。**

`tool_name='具体工具名'` 是 LLM 幻觉输出的**不存在**的工具名（日志原文中的占位符，实际输出不存在的工具名）。

工具验证逻辑正确地捕获了幻觉工具名，返回 `parse_error` 给 LLM，LLM 重新思考并重试。这是正常的容错流程。

### 4.4 结论

**✅ 不是bug。** LLM偶尔产生幻觉工具名是正常现象，验证逻辑正确拦截。

## 八、总结

| 问题 | 类型 | 代码bug? | 修复状态 | 严重程度 |
|------|------|----------|----------|---------|
| Q1: 404会话不存在 | 运行时/环境 | ❌ 非代码bug | - | **🔴 严重**（聊天记录丢失）|
| Q2: 400空tool_calls | 旧代码bug | ✅ 已修复 | 50d46f35 + 2831626a | 🟡 中（已消除）|
| Q3: 无效工具名 | 预期行为 | ❌ 不是bug | - | ⚪ 低（正常容错）|
| Q4: 429限流 | 预期行为 | ❌ 不是bug | - | ⚪ 低（正常限流）|
| Q5: 任务取消 | 预期行为 | ❌ 不是bug | - | ⚪ 低（用户操作）|
| Q6: PromptLogger无数据 | 预期行为 | ❌ 不是bug | - | ⚪ 低（无数据时不记录）|

### 8.1 核心结论

1. **Q3-Q6均为预期行为，不是bug。** 工具验证、限流重试、取消操作、日志记录都正常工作。
2. **Q2已修复。** 旧代码中 `_last_fc_raw_response` 跨策略污染导致 `tool_calls: []` 进入对话历史。当前代码通过 `_strategy` 检查（base_react.py:861）和 `_last_fc_raw_response=None` 清空（react_agent_mixin.py:309）解决了此问题。
3. **Q1影响最严重但非代码bug。** 所有session都无法保存到DB，导致全部聊天记录丢失。需运行时排查前端 `baseURL` 配置是否正确指向后端实例。

### 8.2 建议

1. **Q1排查**：运行时执行 SQLite 查询确认 session 是否存在：
   ```sql
   SELECT * FROM chat_sessions ORDER BY created_at DESC LIMIT 5;
   ```

2. **Q2监控**：如仍出现400错误，在 `llm_core.py:568` 前添加日志记录完整 `request_json["messages"]`，捕获实际发送的消息内容。
