## 零、为什么之前的方案是"修修补补"

之前的方案（预处理管线重构方案）本质上是在现有"意图分类→Agent分发→工具执行"框架上做改良：
- 矫正前置 → 仍是先矫正再意图分类
- 安全增强 → 仍是检查黑名单+正则
- chat fallback → 仍是意图分类后的路由选择

**根本问题没有触及**：

1. **为什么需要意图分类？** 意图分类的存在是因为"先把用户输入归类，再给对应Agent对应的工具"。但LLM完全有能力自己从全量工具中选择合适的工具——Function Calling本身就是做这件事的。

2. **为什么需要9个Agent？** 9个Agent中7个代码完全相同，差异仅为工具集+Prompt。这说明"意图→Agent→工具"这条链路中，Agent这个中间层是冗余的——意图可以直接映射到工具集。

3. **为什么安全检测放在预处理阶段？** 安全检测的真正拦截点应该是工具执行时，而不是用户输入时。用户说"帮我清理系统"本身不危险，危险的是Agent执行`rm -rf /`——所以安全应该跟着工具走，而不是跟着意图走。

## 一、根本性思考：三种架构范式

### 1.1 范式A：意图路由范式（当前）

```
用户输入 → 意图分类 → Agent选择 → 工具加载 → LLM(工具子集) → 工具执行
```

**本质**：人帮LLM做工具筛选——先把意图分类出来，再给LLM对应类别的工具。

**优点**：工具集小（每个Agent 5~11个工具），LLM选择准确，token消耗低。  
**缺点**：意图分类可能错→给错工具集→LLM无法完成任务；跨分类操作需要动态加载；9个Agent维护成本高。

### 1.2 范式B：全量工具范式（OpenAI Function Calling原生模式）

```
用户输入 → LLM(全量57工具) → 工具执行
```

**本质**：让LLM自己从全量工具中选——Function Calling本身就是工具选择器。

**优点**：无需意图分类，无需9个Agent，无需动态加载；跨分类操作天然支持。  
**缺点**：57个工具的Schema一次性传给LLM，token消耗大（约8000 tokens）；LLM在工具数量多时选择准确率下降；每次API调用都要传全量tools参数。

### 1.3 范式C：语义工具发现范式（两阶段路由+LLM自选）

```
用户输入 → 语义匹配(工具摘要) → LLM(候选工具子集) → 工具执行
```

**本质**：第一阶段用轻量方法（规则/向量/LLM分类）从57个工具中筛选出候选子集（10~20个），第二阶段让LLM从候选子集中选择并执行。

**优点**：比范式A更灵活（不绑定意图→Agent硬映射），比范式B更高效（工具子集小）；支持跨分类；安全可跟工具走。  
**缺点**：第一阶段筛选质量决定第二阶段效果；比范式A复杂度略高。

---

### 1.4 三种范式对比

| 维度 | 范式A(当前) | 范式B(全量工具) | 范式C(语义发现) |
|------|-----------|--------------|---------------|
| 意图分类 | 必须 | 不需要 | 替代为工具筛选 |
| Agent数量 | 9个(7同质) | 1个 | 1个 |
| 工具选择准确性 | 依赖意图分类准确率 | 依赖LLM在57工具中选对 | 依赖第一阶段筛选+LLM在子集中选对 |
| 跨分类支持 | 需动态加载 | 天然支持 | 天然支持 |
| token消耗 | 低(5~11工具Schema) | 高(57工具~8000tokens) | 中(15~20工具~3000tokens) |
| 维护成本 | 高(9Agent+Factory+CRSS) | 低(1Agent) | 低(1Agent+筛选器) |
| 安全模型 | 意图层+工具层 | 工具层 | 工具层(跟工具走) |
| 扩展性 | 新分类需改6+处 | 新工具注册即可 | 新工具注册+摘要即可 |
| 纯对话支持 | 需chat意图hack | 天然支持(不调工具) | 天然支持(工具集可为空) |

---

### 1.5 选择建议

**推荐范式C**，理由：

1. 范式A（当前）的核心矛盾——意图分类可能错→给错工具集→任务失败——无法通过修补解决
2. 范式B（全量工具）的token消耗在当前模型（约8000 tokens）下不可接受，且LLM在57工具中选择准确率实测会下降
3. 范式C（语义发现）是范式A和B的折中：第一阶段筛选解决token问题，第二阶段LLM自选解决准确性问题

**但范式C需要解决的核心问题是：第一阶段如何从57个工具中筛选出候选子集？**

## 二、范式C的核心设计：两阶段工具发现

### 2.1 整体架构

```
用户输入
  │
  ├─ Phase 1: 工具发现（轻量级，<50ms）
  │   ├─ 文本矫正（规则级，<1ms）
  │   ├─ 工具相关性评分（57个工具各打一个分）
  │   └─ 取Top-K（K=15~20）作为候选工具子集
  │
  ├─ Phase 2: LLM执行（标准ReAct循环）
  │   ├─ 构建system prompt（含候选工具描述+安全策略）
  │   ├─ LLM Function Calling（仅传候选工具的Schema）
  │   ├─ 工具执行（带安全守卫）
  │   └─ 动态扩展：LLM请求未在候选中的工具时，按需加载
  │
  └─ Phase 3: 安全贯穿（工具级，非意图级）
      ├─ 工具风险标签（每个工具有risk_level）
      ├─ 高风险工具需确认
      └─ Shell类工具内容检测
```

### 2.2 关键变化：意图分类 → 工具相关性评分

**当前**：`用户输入 → 意图分类(file/shell/...) → 加载该分类全部工具`

**重构后**：`用户输入 → 每个工具的相关性评分 → 取Top-K工具`

**好处**：
- 不再需要"意图"这个中间概念——直接算每个工具和用户输入的相关性
- 天然支持跨分类——Top-K中可以同时包含file和network工具
- 不存在"意图分类错误→给错工具集"的问题——每个工具独立评分

---

### 2.3 工具相关性评分方法

#### 2.3.1 方法对比

| 方法 | 延迟 | 准确率 | 维护成本 | 实现复杂度 |
|------|------|--------|---------|-----------|
| **M1: 关键词匹配** | <1ms | 中 | 高(关键词需维护) | 低 |
| **M2: TF-IDF** | <5ms | 中 | 低(自动从描述提取) | 低 |
| **M3: Embedding向量** | ~50ms | 高 | 低(自动计算) | 中 |
| **M4: LLM分类** | ~1s | 最高 | 低 | 中 |
| **M5: 混合(关键词+Embedding)** | ~50ms | 高 | 中 | 中 |

**推荐M5：关键词+Embedding混合**，理由：
- 关键词匹配处理明确意图（"截图"→screen_capture工具），零延迟
- Embedding处理模糊意图（"帮我整理一下"→可能是file或document工具），~50ms
- 两者融合：有明确关键词匹配时以关键词为主，无明确匹配时以Embedding为主

#### 2.3.2 关键词匹配层（替代CRSS意图分类）

不再对意图分类，而是直接对每个工具打分：

```python
# backend/app/services/pipeline/tool_relevance_scorer.py

from dataclasses import dataclass, field
from typing import Dict, List, Tuple
from app.services.tools.registry import tool_registry, ToolMetadata, ToolCategory
from app.utils.logger import logger


@dataclass
class ToolRelevanceScore:
    """工具相关性评分结果"""
    tool_name: str
    score: float           # 0.0 ~ 1.0
    source: str            # "keyword" / "embedding" / "default"
    category: ToolCategory


class KeywordScorer:
    """
    关键词工具相关性评分器
    
    思路：每个工具的description + name中包含的关键词，
    与用户输入做匹配，匹配度越高分数越高。
    
    与CRSS意图评分的区别：
    - CRSS: 用户输入 → 意图分类(file/shell/...) → 该分类全部工具
    - 本方法: 用户输入 → 每个工具独立打分 → 取Top-K
    
    好处：不需要维护"意图→工具"的映射关系，
    直接利用工具自身的description做匹配。
    """
    
    # 工具级关键词增强映射（补充工具description中可能缺少的中文同义词）
    # 仅维护高频/安全关键的增强词，其余靠description自动匹配
    _TOOL_KEYWORD_BOOST: Dict[str, List[str]] = {
        # FILE类
        "read_file": ["读取", "查看", "打开", "浏览", "cat", "less", "more"],
        "write_text_file": ["写入", "保存", "创建文件", "新建"],
        "edit_file": ["编辑", "修改文件", "替换", "插入"],
        "search_files": ["搜索文件", "查找文件", "find", "locate"],
        "rename_file": ["重命名", "改名", "移动文件", "mv"],
        "archive_tool": ["压缩", "解压", "打包", "zip", "tar", "归档"],
        
        # SHELL类
        "execute_shell_command": ["命令", "执行", "运行", "终端", "cmd", "bash", "shell"],
        "execute_python": ["Python", "python", "py"],
        "execute_javascript": ["JavaScript", "js", "node"],
        
        # NETWORK类
        "search_web": ["搜索", "查询", "百度", "Google", "bing"],
        "download_file": ["下载", "wget", "curl"],
        "http_request": ["请求", "API", "HTTP", "接口"],
        "fetch_webpage": ["网页", "抓取", "爬虫", "scrape"],
        
        # DESKTOP类
        "screen_capture": ["截图", "截屏", "screenshot", "屏幕截图"],
        "mouse_control": ["鼠标", "点击", "click", "双击"],
        "keyboard_control": ["键盘", "按键", "输入文字", "type", "快捷键"],
        "ocr": ["OCR", "文字识别", "识别图片"],
        "list_windows": ["窗口", "应用", "程序"],
        
        # SYSTEM类
        "get_system_info": ["系统信息", "CPU", "内存", "磁盘", "系统"],
        "list_processes": ["进程", "任务管理", "tasklist", "ps"],
        "kill_process": ["杀进程", "结束进程", "kill", "终止"],
        
        # DOCUMENT类
        "read_document": ["文档", "PDF", "Word", "Excel", "读取文档"],
        "write_document": ["写文档", "创建文档", "生成报告"],
        "query_sql": ["SQL", "查询数据", "数据库查询"],
        "analyze_data": ["数据分析", "统计", "图表"],
        
        # META类
        "get_time": ["时间", "日期", "几点", "星期", "今天"],
        "tool_help": ["帮助", "工具列表", "有什么工具"],
        "tool_search": ["搜索工具", "找工具"],
    }
    
    def score(self, user_input: str) -> List[ToolRelevanceScore]:
        """
        对所有已注册工具打相关性评分
        
        Args:
            user_input: 用户输入（已矫正）
            
        Returns:
            按score降序排列的评分列表
        """
        input_lower = user_input.lower()
        results = []
        
        for tool_name, metadata in tool_registry._tools.items():
            if not metadata.expose_to_llm:
                continue  # 内部工具不参与评分
            
            score = self._score_single_tool(tool_name, metadata, input_lower)
            results.append(ToolRelevanceScore(
                tool_name=tool_name,
                score=score,
                source="keyword",
                category=metadata.category,
            ))
        
        # 归一化到[0, 1]
        if results:
            max_score = max(r.score for r in results)
            if max_score > 0:
                for r in results:
                    r.score = r.score / max_score
        
        # 降序排列
        results.sort(key=lambda r: r.score, reverse=True)
        return results
    
    def _score_single_tool(
        self, 
        tool_name: str, 
        metadata: ToolMetadata, 
        input_lower: str
    ) -> float:
        """对单个工具计算相关性评分"""
        score = 0.0
        
        # 1. 增强关键词匹配（权重最高）
        boost_keywords = self._TOOL_KEYWORD_BOOST.get(tool_name, [])
        for kw in boost_keywords:
            if kw.lower() in input_lower:
                score += 2.0  # 中文关键词命中+2.0
        
        # 2. 工具description中的关键词匹配
        desc_lower = metadata.description.lower()
        desc_words = set(desc_lower.split())
        input_words = set(input_lower.split())
        overlap = desc_words & input_words
        score += len(overlap) * 0.5  # description词重叠+0.5/个
        
        # 3. 工具名部分匹配
        name_parts = tool_name.replace("_", " ").split()
        for part in name_parts:
            if part.lower() in input_lower:
                score += 1.0  # 工具名部分命中+1.0
        
        return score
```

```python
# backend/app/services/pipeline/tool_relevance_scorer.py 续

class HybridToolScorer:
    """
    混合工具相关性评分器 — 关键词 + Embedding 融合
    
    融合策略：
    - 关键词分数 > 0 时：以关键词为主（明确意图，不需要embedding）
    - 关键词分数 = 0 时：以embedding为主（模糊意图，需要语义理解）
    - 两者都有分数时：加权融合（关键词0.7 + embedding0.3）
    """
    
    def __init__(self, embedding_enabled: bool = False):
        self._keyword_scorer = KeywordScorer()
        self._embedding_scorer = EmbeddingScorer(enabled=embedding_enabled)
    
    async def score_and_select(
        self, 
        user_input: str, 
        top_k: int = 15,
        min_score: float = 0.1
    ) -> Tuple[List[ToolRelevanceScore], List[ToolRelevanceScore]]:
        """
        评分并选择候选工具
        
        Args:
            user_input: 用户输入（已矫正）
            top_k: 返回Top-K个工具
            min_score: 最低分数阈值
            
        Returns:
            (selected, all_scores): 选中的工具 + 全部评分
        """
        # 1. 关键词评分
        keyword_scores = self._keyword_scorer.score(user_input)
        
        # 2. Embedding评分（可选）
        embedding_scores = await self._embedding_scorer.score_async(user_input)
        
        # 3. 融合
        fused = self._fuse(keyword_scores, embedding_scores)
        
        # 4. 选择Top-K（分数>=min_score的）
        selected = [s for s in fused if s.score >= min_score][:top_k]
        
        # 5. 保底：如果没有任何匹配，返回meta类工具（help/search/pipeline）
        if not selected:
            selected = self._get_fallback_tools()
        
        return selected, fused
    
    def _fuse(
        self,
        keyword_scores: List[ToolRelevanceScore],
        embedding_scores: List[ToolRelevanceScore],
    ) -> List[ToolRelevanceScore]:
        """融合关键词和Embedding评分"""
        if not embedding_scores:
            return keyword_scores  # 无embedding时直接用关键词
        
        # 构建embedding索引
        emb_map = {s.tool_name: s.score for s in embedding_scores}
        
        result = []
        for ks in keyword_scores:
            es = emb_map.get(ks.tool_name, 0.0)
            
            if ks.score > 0:
                # 关键词有匹配 → 关键词为主
                fused_score = ks.score * 0.7 + es * 0.3
                source = "keyword"
            else:
                # 关键词无匹配 → 用embedding
                fused_score = es
                source = "embedding"
            
            result.append(ToolRelevanceScore(
                tool_name=ks.tool_name,
                score=fused_score,
                source=source,
                category=ks.category,
            ))
        
        result.sort(key=lambda r: r.score, reverse=True)
        return result
    
    def _get_fallback_tools(self) -> List[ToolRelevanceScore]:
        """保底工具集（meta类：help/search/time）"""
        return [
            ToolRelevanceScore("tool_help", 0.5, "default", ToolCategory.META),
            ToolRelevanceScore("tool_search", 0.5, "default", ToolCategory.META),
            ToolRelevanceScore("get_time", 0.3, "default", ToolCategory.META),
        ]
```

## 三、统一Agent：UnifiedReactAgent

### 3.1 核心设计

当前9个Agent（7个同质+2个特殊）合并为1个统一Agent。差异化通过**AgentProfile配置**驱动，不再需要子类。

```python
# backend/app/services/agent/unified_react.py

from app.services.agent.base_react import BaseAgent
from app.services.agent.mixins.react_agent_mixin import ReactAgentMixin
from app.services.pipeline.tool_relevance_scorer import HybridToolScorer
from app.services.agent.tool_execution_guard import ToolExecutionGuard
from typing import List, Optional


@dataclass
class AgentProfile:
    """Agent运行时配置 — 由RequestPipeline生成"""
    # 工具配置
    selected_tools: List[str]           # Phase 1筛选的候选工具名
    all_relevance_scores: List[ToolRelevanceScore]  # 全部评分（供参考）
    
    # Prompt配置
    category_hint: str = ""             # 主要工具分类名（用于system prompt）

    # 行为配置

    enable_session: bool = False         # 是否启用文件session管理

class UnifiedReactAgent(ReactAgentMixin, BaseAgent):
    """
    统一ReAct Agent — 替代9个分类Agent
    
    核心变化：
    1. 不再按意图分类创建不同Agent — 统一一个Agent
    2. 工具集由AgentProfile.selected_tools决定 — 由管线Phase 1筛选
    3. 动态扩展：LLM请求未在selected_tools中的工具时，自动加载
    4. 安全守卫：工具执行层拦截
    """
    
    def __init__(
        self, 
        profile: AgentProfile,
        llm_client,
        task_id: str,

    ):

        # 推导ToolCategory（从selected_tools的主要分类）

        super().__init__(
            llm_client=llm_client,
            task_id=task_id,
            tool_category=primary_category,
            max_steps=profile.max_steps,
            **kwargs
        )
        
        # 加载筛选的工具子集（而非全分类工具！）
        self._init_selected_tools(profile.selected_tools)
        self._init_llm_strategies()
        self._init_task_tracking(enable=profile.enable_session)
        
        # 安全守卫
        self._guard = ToolExecutionGuard(enabled=profile.guard_enabled)
        
        # Prompt
        self.prompts = self._build_prompts(profile)
    
    def _init_selected_tools(self, tool_names: List[str]):
        """
        初始化筛选的工具子集
        
        与当前_init_tools_and_executor的区别：
        - 当前：加载某个ToolCategory下的全部工具（5~11个）
        - 本方法：加载AgentProfile指定的工具（可能跨多个分类，15~20个）
        """
        ensure_tools_registered()
        
        self._tools_dict = {}
        self.openai_tools = []
        self._loaded_categories = set()
        
        for tool_name in tool_names:
            impl = tool_registry.get_implementation(tool_name)
            meta = tool_registry._tools.get(tool_name)
            if impl and meta:
                self._tools_dict[tool_name] = impl
                self._loaded_categories.add(meta.category.value)
        
        # 生成OpenAI FC Schema
        categories = [ToolCategory(c) for c in self._loaded_categories if c]
        self.openai_tools = tool_registry.to_openai_tools(categories=categories)
        # 过滤：只保留selected_tools中的工具
        self.openai_tools = [
            t for t in self.openai_tools 
            if t["function"]["name"] in tool_names
        ]
        
        self.executor = ToolExecutor(self._tools_dict)
    
    def _infer_primary_category(self, tool_names: List[str]) -> ToolCategory:
        """从selected_tools推导主要ToolCategory"""
        category_counts: Dict[ToolCategory, int] = {}
        for name in tool_names:
            meta = tool_registry._tools.get(name)
            if meta:
                category_counts[meta.category] = category_counts.get(meta.category, 0) + 1
        if category_counts:
            return max(category_counts, key=category_counts.get)
        return ToolCategory.META  # 默认
    
    def _get_system_prompt(self) -> str:
        """构建system prompt — 动态组合而非固定Prompt类"""
        return self._build_dynamic_system_prompt()
    
    def _build_dynamic_system_prompt(self) -> str:
        """
        动态构建system prompt
        
        与当前方案的差异：
        - 当前：每个Agent硬编码一个Prompt类（FileOperationPrompts/ShellPrompts/...）
        - 本方法：根据selected_tools的分类动态组合prompt
        """
        # 获取涉及分类的Prompt类
        categories = self._loaded_categories
        
        # 基础prompt（复用现有BasePromptTemplate）
        parts = []
        parts.append(self._build_role_section(categories))
        parts.append(self._build_tools_section(categories))
        parts.append(OUTPUT_FORMAT)
        parts.append(TOOL_CALL_RULES)
        parts.append(self._build_safety_section(categories))
        
        return "\n\n".join(p for p in parts if p)
    
    def _build_role_section(self, categories: set) -> str:
        """构建角色定义 — 根据工具分类动态生成"""
        category_names = {
            "file": "文件操作", "shell": "命令执行", "network": "网络通信",
            "desktop": "桌面操作", "system": "系统信息", "document": "文档读写",
            "meta": "元工具",
        }
        names = [category_names.get(c, c) for c in categories]
        return f"你是一个{'、'.join(names)}助手。根据用户需求选择合适的工具完成任务。"
    
    async def _get_llm_response(self) -> str:
        return await self._call_llm_with_summary()
    
    async def _execute_tool(self, action: str, params: dict) -> str:
        """工具执行 — 带安全守卫"""
        # 安全守卫检查
        is_safe, reason = self._guard.check(action, params)
        if not is_safe:
            return json.dumps({"code": 403, "data": None, "message": reason}, ensure_ascii=False)
        
        # 高风险工具确认
        if action in self.profile.confirm_tools:
            # TODO: yield确认步骤给前端
            pass
        
        # 标准执行
        normalized = self.executor._normalize_params(action, params)
        return await self.executor.execute(action, normalized)
```

## 四、工具级安全模型

### 4.1 核心思想：安全跟着工具走，不跟着意图走

当前安全模型的问题：安全检测在用户输入层执行，但真正危险的是工具执行。

**重构后安全模型**：

```
安全 = 工具风险标签 + 执行时守卫 + 用户确认
```

每个工具有明确的风险等级：


### 4.3 ToolExecutionGuard 改造

```python
# backend/app/services/agent/tool_execution_guard.py

class ToolExecutionGuard:
    """
    工具执行安全守卫 V2
    
    核心变化：安全跟着工具走，不跟着意图走
    
    三级检查：
    1. 工具元数据风险等级（ToolMetadata.risk_level）
    2. 工具参数内容检测（Shell命令黑名单/正则）
    3. 用户确认（高风险工具requires_confirm=True）
    """
    
    def check(self, tool_name: str, tool_params: dict) -> Tuple[bool, str, SafetyDecision]:
        """
        三级安全检查
        
        Returns:
            (is_safe, reason, decision)
        """
        # Level 1: 工具元数据风险等级
        meta = tool_registry._tools.get(tool_name)
        if meta and meta.risk_level == "critical":
            # critical工具：参数内容必须通过安全检测
            if tool_name in _GUARDED_TOOLS:
                param_name = _COMMAND_PARAM_NAMES.get(tool_name, "")
                command = tool_params.get(param_name, "")
                if command:
                    security_result = check_command_safety(command)
                    if not security_result.get('is_safe', True):
                        return False, f"危险命令: {security_result.get('risk')}", SafetyDecision.BLOCK
        
        # Level 2: Shell类工具内容检测
        if tool_name in _GUARDED_TOOLS:
            param_name = _COMMAND_PARAM_NAMES.get(tool_name, "")
            command = tool_params.get(param_name, "")
            if command:
                security_result = check_command_safety(command)
                if security_result.get('blocked', False):
                    return False, f"安全拦截: {security_result.get('risk')}", SafetyDecision.BLOCK
        
        # Level 3: 高风险工具确认
        if meta and meta.requires_confirm:
            return True, "需要用户确认", SafetyDecision.CONFIRM
        
        return True, "", SafetyDecision.ALLOW
```

## 五、RequestPipeline 完整设计

### 5.1 管线流程

```python
# backend/app/services/pipeline/request_pipeline.py

@dataclass
class PipelineContext:
    """管线上下文"""
    # 原始输入
    raw_input: str
    
    # Phase 0: 矫正
    corrected_input: str = ""
    is_corrected: bool = False
    
    # Phase 1: 工具发现
    selected_tools: List[str] = field(default_factory=list)
    all_scores: List[ToolRelevanceScore] = field(default_factory=list)
    primary_category: Optional[ToolCategory] = None
    
    # Phase 2: Agent配置
    agent_profile: Optional[AgentProfile] = None
    
    # 安全
    safety_decision: SafetyDecision = SafetyDecision.ALLOW
    blocked_reason: str = ""
    
    @property
    def effective_input(self) -> str:
        return self.corrected_input if self.is_corrected else self.raw_input


class RequestPipeline:
    """
    请求处理管线 — 两阶段工具发现架构
    
    Phase 0: 文本矫正（规则级，<1ms）
    Phase 1: 工具相关性评分 + Top-K选择（<50ms）
    Phase 2: 构建AgentProfile → 创建UnifiedReactAgent
    """
    
    def __init__(self, embedding_enabled: bool = False):
        self._corrector = TextCorrectorV2()
        self._scorer = HybridToolScorer(embedding_enabled=embedding_enabled)
    
    async def process(self, user_input: str) -> PipelineContext:
        ctx = PipelineContext(raw_input=user_input)
        
        # Phase 0: 矫正
        ctx = await self._corrector.correct(ctx)
        
        # Phase 1: 工具发现
        text = ctx.effective_input
        selected, all_scores = await self._scorer.score_and_select(
            text, top_k=15, min_score=0.1
        )
        ctx.selected_tools = [s.tool_name for s in selected]
        ctx.all_scores = all_scores
        ctx.primary_category = selected[0].category if selected else ToolCategory.META
        
        # Phase 2: 构建AgentProfile
        ctx.agent_profile = AgentProfile(
            selected_tools=ctx.selected_tools,
            all_relevance_scores=all_scores,
            category_hint=ctx.primary_category.value if ctx.primary_category else "",
            task_description=text,
        )
        
        logger.info(
            f"[Pipeline] 输入='{text}', "
            f"Top-{len(selected)}工具={[s.tool_name for s in selected]}, "
            f"主分类={ctx.primary_category}"
        )
        
        return ctx
```

## 六、动态工具扩展

### 6.1 问题：LLM请求的工具不在selected_tools中

尽管Phase 1筛选了Top-15工具，LLM仍可能请求未在候选中的工具。例如：
- 用户："下载文件并读取内容" → Phase 1选中`download_file`+`read_file`
- LLM在执行`download_file`后，决定用`search_files`查找文件 → `search_files`不在Top-15中

### 6.2 解决方案：运行时动态加载+二次评分

```python
# UnifiedReactAgent._execute_tool() 改造

async def _execute_tool(self, action: str, params: dict) -> str:
    # 工具不在当前候选中 → 动态加载
    if action not in self._tools_dict:
        # 从全局registry查找
        impl = tool_registry.get_implementation(action)
        if impl is not None:
            # 加载该工具 + 其同分类工具
            meta = tool_registry._tools.get(action)
            if meta:
                self.load_tools_by_intent(meta.category.value)
                logger.info(
                    f"[DynamicLoad] LLM请求工具'{action}'不在候选中，"
                    f"已加载{meta.category.value}分类"
                )
        else:
            return json.dumps({"code": 404, "message": f"工具不存在: {action}"})
    
    # 安全守卫
    is_safe, reason, decision = self._guard.check(action, params)
    if not is_safe:
        return json.dumps({"code": 403, "message": reason})
    
    # 执行
    return await self.executor.execute(action, self.executor._normalize_params(action, params))
```

## 十、总结

本方案与之前"修修补补"方案的根本区别：

| 维度 | 之前方案 | 本方案 |
|------|---------|--------|
| 核心范式 | 意图路由范式(A) | **语义工具发现范式(C)** |
| 意图分类 | 修补CRSS+LLM两阶段 | **取消意图分类**，用工具相关性评分替代 |
| Agent架构 | 9个Agent+PipelineContext修补 | **1个UnifiedAgent+AgentProfile配置** |
| 安全模型 | 用户输入层+工具层双层修补 | **工具级安全模型**，安全跟着工具走 |
| 纯对话 | 新增chat意图hack | **工具集为空时自然退化**，无需特殊处理 |

| 代码量 | 不变或增加 | **减少53%** |

**一句话总结**：不再"先猜意图再给工具"，而是"算每个工具和用户输入的相关性，取最相关的给LLM选"——把工具选择权还给LLM，人只做初筛。
