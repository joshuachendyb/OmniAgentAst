## 一、 架构重构背景与痛点总结

在上一阶段的《Agent与意图分类架构重构方案》中，我们解决了 Agent 代码同质化和意图定义分散的问题，引入了 `AgentRegistry` 和 `GenericReactAgent`。然而，在路由入口与安全防线侧，系统依然存在严重的"古典主义"包袱：

1. **废弃的预处理组件**：`PreprocessingPipeline` 已被抽空，仅剩下无意义的 `strip()`，增加了系统调用层级却没有产生实际价值。
2. **割裂的意图路由**：当前采用 `CRSS (正则表达式) + LLM 兜底` 的两阶段分类。正则匹配无法应对用户自然语言的多样性，导致分类准确率低且难以维护。
3. **脆弱的安全防线**：`command_security.py` 采用黑名单模式拦截危险命令（如 `rm -rf`）。面对大模型的变量拼接、编码执行等手段，黑名单形同虚设，极易被绕过。

为了构建符合现代大模型特性（LLM-Native）的智能 Agent 系统，我们亟需对这三个组件进行根本性重构。

## 二、 总体架构设计理念

本次重构抛弃原有的"补丁式"防御与正则匹配，全面拥抱 **LLM驱动与人机协同**：

1. **奥卡姆剃刀原则**：如无必要，勿增实体。彻底删除无效的 `PreprocessingPipeline`。
2. **LLM Native 路由 (Semantic Router)**：完全摒弃基于正则的 `CRSS` 意图匹配，利用大模型原生的 **Function Calling (函数调用)** 或 **Structured Output (结构化输出)** 能力进行精准调度。
3. **HITL (Human-In-The-Loop) 绝对安全机制**：放弃纯文本安全黑名单。将安全防线从"正则拦截"转变为"大模型意图识别 + 人类授权 (Approval)"，确保用户对最高危操作的绝对掌控。

## 三、 核心模块重构方案

### 3.1 预处理层极简改造

**方案**：直接删除 `app/services/preprocessing/pipeline.py` 及相关调用。
**替代逻辑**：在 `chat_router.py` 接收到 `user_input` 时，直接在一行内完成清洗：
```python
user_input = user_input.strip()
if not user_input:
    return {"error": "输入不能为空"}
```

### 3.2 基于 Function Calling 的 Semantic Router

**废除**：删除 `crss_scorer.py` 及其对应的硬编码词库。
**新方案**：构建全新的 `SemanticRouter`。
利用大模型出色的意图理解能力，将我们注册在 `IntentRegistry` 中的意图描述（descriptions）作为 Function 的参数描述提供给 LLM。

**核心流程**：
1. 从 `IntentRegistry` 获取所有激活的意图（如 file, shell, desktop 等）及其精确描述。
2. 将这些意图打包为一个 Function Tool（例如名为 `route_user_request`，包含一个枚举类型的参数 `intent_name`）。
3. 让 LLM（如七牛的 deepseek-v3.1）根据用户的自然语言输入，调用该 Function 选出最匹配的意图。

**优势**：LLM 具备极强的泛化能力，无论是用户说"帮我把C盘清理一下"还是"查看下那个文档的配置"，LLM 都能精确命中 `file` 或 `document`，彻底告别正则漏判的问题。

### 3.3 HITL (Human-in-the-Loop) 统一权限拦截器

**废除**：废除现存的 `command_security.py` 黑名单校验。
**新方案**：引入 `PermissionInterceptor` (权限拦截器)，基于**工具白名单与人工授权**机制。

**核心流程**：
1. **风险分级**：在工具注册时，为每个工具（Tool）标记风险等级。
   - `SAFE`：无破坏性（如查看时间、搜网、读文件）。
   - `HIGH_RISK`：具备系统修改能力（如写文件、删除文件、所有的 Shell 执行）。
2. **执行拦截**：在 `tool_executor.py` 执行工具前，检查该工具的风险等级。
3. **SSE 授权流**：如果工具是 `HIGH_RISK` 级别，执行器（Executor）挂起当前大模型的推理流。
   - 通过 SSE 向前端发送一个特殊的事件 `AUTHORIZATION_REQUIRED`。
   - 携带即将执行的代码/命令及解释。
   - 客户端弹出"安全风险确认框"，用户点击【允许执行】或【拒绝】。
4. **恢复执行**：前端回传用户的授权结果，执行器根据结果放行或抛出异常给大模型让其放弃。

**优势**：这是防御 AI 幻觉和危险操作的降维打击。不管命令如何混淆，最终要由人类决定是否在宿主机上按下回车键。

## 四、 核心伪代码实现

### 4.1 Semantic Router
```python
async def route_intent(user_input: str) -> str:
    """基于 Function Calling 的语义路由"""
    # 动态组装意图枚举
    intents = intent_registry.active_intents()
    intent_enum = [i.intent_type for i in intents]
    intent_descriptions = {i.intent_type: i.description for i in intents}
    
    # 构造函数调用 Schema
    tools = [{
        "type": "function",
        "function": {
            "name": "route_request",
            "description": "根据用户输入选择最合适的处理Agent",
            "parameters": {
                "type": "object",
                "properties": {
                    "selected_intent": {
                        "type": "string",
                        "enum": intent_enum,
                        "description": f"意图分类。对应关系：{intent_descriptions}"
                    }
                },
                "required": ["selected_intent"]
            }
        }
    }]
    
    # 调用 LLM，强制要求调用此 Function
    response = await llm_client.chat(
        messages=[{"role": "user", "content": user_input}],
        tools=tools,
        tool_choice={"type": "function", "function": {"name": "route_request"}}
    )
    
    # 解析并返回意图
    arguments = response.tool_calls[0].function.arguments
    return arguments.get("selected_intent", "chat") # 兜底至 chat 通用意图
```

### 4.2 HITL 拦截控制
```python
# app/services/agent/tool_executor.py

async def execute(self, action: str, params: dict) -> str:
    tool = self._tools_dict.get(action)
    
    # 风险阻断：如果是高危工具，等待用户授权
    if tool.risk_level == RiskLevel.HIGH_RISK:
        auth_result = await self.request_user_authorization(
            action=action, 
            params=params
        )
        if not auth_result.approved:
            return f"Error: 危险操作被用户拒绝执行。原因：{auth_result.reason}"
            
    # 执行原生工具
    return await tool.execute(params)
```

## 五、 实施计划与里程碑

本方案的实施应当在上一期（AgentRegistry）重构完成后进行，分为以下几个 Phase：

| 阶段 | 内容 | 风险评估 | 核心改动模块 |
|------|------|----------|-------------|
| **Phase 1** | 剃刀清理：删除 PreprocessingPipeline 及其外围调用，合并清洗逻辑到 ChatRouter。 | 极低 | `pipeline.py`, `chat_router.py` |
| **Phase 2** | Semantic Router 实现：删除 CRSS 评分模块，引入基于 Function Calling 的意图路由服务。 | 中 | `crss_scorer.py`, `chat_router.py` |
| **Phase 3** | HITL 交互通道建设：打通后端 SSE 挂起与前端授权确认接口。 | 高（需前后端联调） | `tool_executor.py`, 前端组件 |
| **Phase 4** | 安全配置迁移：废弃 `command_security.py` 黑名单，在工具注册表（Registry）为现有 135 个工具打上 RiskLevel 标签。 | 中 | 工具注册模块, `command_security.py` |

## 六、 风险分析与缓解策略

| 风险点 | 影响 | 缓解措施 |
|--------|------|----------|
| **Function Calling 延迟高** | 用户感知提问后响应变慢 | 意图路由采用体积小、首token响应极快（TTFT < 0.5s）的专属模型，而非主线大模型。 |
| **频繁授权导致打断心流** | "骚扰式"弹窗降低用户体验 | 允许前端提供"本会话对该命令信任"的选项（Session Trust），一定时间内对相同模式的高危命令免审批。 |
| **前后端交互卡死** | HITL挂起期间网络闪断导致死锁 | 增加挂起超时机制（如挂起 60 秒未授权，则自动视为拒绝并恢复大模型运行）。 |
