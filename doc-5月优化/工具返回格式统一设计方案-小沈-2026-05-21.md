# 工具返回格式统一设计方案

**版本**: v2.0 | **创建时间**: 2026-05-20 | **更新时间**: 2026-05-21 | **作者**: 小沈 | **补充**: 小健

---

## 零、混乱根源

三个根本原因：

1. **没人区分"给LLM看"和"给前端看"是两回事**，用同一个 `_format_result` 函数强行猜格式、暴力压缩，code/message/warning 全丢了
2. **code 值没有命名规范**，想到什么写什么，166个 code 各自为政
3. **file_tools 双重转换**：先 `_to_unified_format` 注入冗余字段（code+status双写、message=summary重复），再 `_format_result` 猜格式拆开——两层转换互相抵消

### 零点一、现状数据流（要砍掉的）

```
工具函数 return
    │
    ├─ file_tools: return _to_unified_format({success, error, ...}, tool_name)
    │     │
    │     ▼ 输出混合格式: {code, status, summary, data, message, retry_count}
    │     （code=status重复，message=summary重复）
    │
    └─ 其他6类: return {code, data, message}
          │
          ▼ 
    ToolExecutor._format_result() ← 7个分支猜格式
          │
          ├─ B1: status=="warning"     → {status, summary, data, retry_count}
          ├─ B2: 有 status+summary     → {status, summary, data, retry_count}   ← file走这
          ├─ B3: code=="SUCCESS"       → {status, summary, data:=整个result}    ← 其他工具成功
          ├─ B4: code以ERR_开头        → {status, summary, data:=整个result}
          ├─ B5: success==True         → ...
          ├─ B6: 兜底                  → ...
          └─ B7: 非dict               → ...
          │
          ▼ 输出: {status, summary, data, retry_count}
          │
    base_react.py ← 读 status + summary + data + error
```

**核心问题**：
- B2分支 `data = result.get("data")`（取子字段），B3分支 `data = result`（整个dict）——**data语义不一致**
- `code`、`message` 等字段在 `_format_result` 后丢失，Agent层面永远读不到
- 双重转换做无用功

## 一、统一内部格式

### 1.1 所有工具函数的返回格式

```python
{
    "code": str,                # 必须。见第二节 code 命名规范
    "data": Any,                # 必须。成功=业务数据dict，失败=None
    "message": str,             # 必须。人类可读的描述文字

    "warning": str | None,      # 可选。code=SUCCESS但有异常时
    "llm_data": dict | None,    # 可选。给LLM的精简数据（省token）
    "next_actions": list | None, # 可选。前端下一步按钮
    "retry_count": int,         # 可选。实际重试次数，默认0（Agent层填充）
    "return_direct": bool,      # 可选。true=直接返回给用户，跳过LLM继续推理
}
```

**3个必须，5个可选。没有冗余字段。**

### 1.2 data 字段语义规范

| code值 | data内容 | 说明 |
|--------|---------|------|
| `SUCCESS` | 业务数据dict | 只含业务字段，不含code/message等格式元信息 |
| `WARNING_*` | 业务数据dict | 部分成功的业务数据 |
| `ERR_*` | `None` | 失败时无业务数据 |

**禁止**：data中嵌套code/message/status等格式字段（旧B3分支的毛病）。

### 1.3 现状字段清理

| 现状字段 | 去向 | 原因 |
|---------|------|------|
| `success: bool` | ❌ 删除 | 被 `code="SUCCESS"` 取代 |
| `status: str` | ❌ 删除 | 被 `code` 取代（"success/error/warning" 太粗糙） |
| `summary: str` | ❌ 删除 | 被 `message` 取代（一模一样的东西） |
| `error: str` | ❌ 删除 | 被 `code="ERR_XXX" + message` 取代 |
| `code: str` | ✅ 保留 | 从"部分工具有"变成"所有工具有" |
| `message: str` | ✅ 保留 | 从"部分工具有"变成"所有工具有" |
| `data: Any` | ✅ 保留 | 不变，但语义统一（见1.2节） |
| `llm_data` | ✅ 保留 | 不变 |
| `next_actions` | ✅ 保留 | 不变 |
| `warning` | ✨ 新增 | code=SUCCESS但有情况要提醒时 |
| `retry_count` | ✅ 保留 | 从"部分工具有"变成统一字段 |
| `return_direct` | ✅ 保留 | 从agent层下沉到工具返回 |
| `capabilities_used/missing` | ❌ 删除 | 调试信息，走日志 |

## 二、code 命名规范 ← 重点

### 2.1 三段式结构

```
ERR_分类_具体错误
SUCCESS
```

三段之间用下划线连接，全大写。

### 2.2 三个允许值前缀

| 前缀 | 含义 | LLM行为 | 前端行为 |
|------|------|---------|---------|
| `SUCCESS` | 执行成功 | 正常处理结果 | 绿色图标 |
| `ERR_` | 执行失败 | 如果可重试则重试，否则报错 | 红色图标 + toast |
| `WARNING_` | 执行成功但有重大问题 | 注意但不阻断 | 黄色图标 |

> `WARNING_` 和 `SUCCESS + warning字段` 的区别：
> - `WARNING_` prefix = 比较严重的问题，LLM 和 Agent 应该关注
> - `SUCCESS` + `warning` 字段 = 轻微提示，只有前端展示，Agent 不关心

### 2.3 分类标识（第二段）

| 分类 | 说明 | 示例 |
|------|------|------|
| `SHELL` | Shell命令执行 | `ERR_SHELL_TIMEOUT` |
| `EXEC` | 代码执行（Python/JS） | `ERR_EXEC_TIMEOUT` |
| `NETWORK` | 网络请求 | `ERR_NETWORK_TIMEOUT` |

### 2.4 第三段取值规范

| 尾段 | 含义 | 示例 |
|------|------|------|
| `NOT_FOUND` | 找不到目标 | `ERR_FILE_NOT_FOUND` |
| `TIMEOUT` | 超时 | `ERR_SHELL_TIMEOUT` |
| `DENIED` | 权限拒绝 | `ERR_PERMISSION_DENIED` |
| `FAILED` | 执行失败 | `ERR_OCR_FAILED` |
| `INVALID_xxx` | 参数不合法 | `ERR_INVALID_PARAM` |
| `EMPTY_xxx` | 参数为空 | `ERR_EMPTY_QUERY` |
| `MISSING_xxx` | 缺少参数 | `ERR_MISSING_PARAM` |
| `CONNECTION_ERROR` | 连接错误 | `ERR_DB_CONNECTION_ERROR` |
| `NOT_SUPPORTED` | 不支持 | `ERR_PLATFORM_NOT_SUPPORTED` |
| `xxx` | 具体操作 | `ERR_TIMER_SET` |

### 2.5 禁止的行为

| ❌ 禁止 | ✅ 正确 | 原因 |
|--------|--------|------|
| `ERR_READ_CSV` 和 `ERR_READ_EXCEL_DATAFRAME` 并存 | 统一 `ERR_READ_FILE` 或 `ERR_READ_XLSX` | 同一类错误code不一致 |
| `ERR_INVALID_PARAM` 和 `ERR_INVALID_PARAMS` 并存 | 统一 `ERR_INVALID_PARAM` | 单复数混用 |
| `ERR_SYSTEM_ACCESS_DENIED` | `ERR_PERMISSION_DENIED` | 跨分类统一 |
| `ERR_TIME_NOW` | `ERR_TIME_FORMAT` 或具体说明 | 按场景选，不要工具名 |

### 2.6 当前 code 审计结果

当前后端共 166 个不同 ERR code。审计发现以下不合规项：

| 不合规code | 问题 | 建议替换 |
|-----------|------|---------|
| `ERR_SHELL_EXEC` | 太笼统 | 保留（通用shell执行失败） |
| `ERR_TIME` | 没有第三段 | `ERR_TIME_FORMAT` 或具体 |
| `ERR_TIMER` | 没有第三段 | `ERR_TIMER_SET` 或具体 |
| `ERR_EXEC_JAVASCRIPT` | 应归入EXEC分类 | `ERR_EXEC_JS` |
| `ERR_RESIZE_WINDOW` | 应归入WINDOW分类 | `ERR_WINDOW_RESIZE` |
| `ERR_SET_WINDOW_STATE` | 应归入WINDOW分类 | `ERR_WINDOW_SET_STATE` |
| `ERR_SHORTCUT` | 应归入KEYBOARD分类 | `ERR_KEYBOARD_SHORTCUT` |
| `ERR_SNAPSHOT` | 应归入SCREEN分类 | `ERR_SCREEN_SNAPSHOT` |
| `ERR_SCROLL` | 应归入DESKTOP分类 | `ERR_DESKTOP_SCROLL` |
| `ERR_TYPE_TEXT` | 应归入KEYBOARD分类 | `ERR_KEYBOARD_TYPE` |
| `ERR_SEARCH_QUERY_TOO_SHORT` | 太具体 | `ERR_PARAM_INVALID` |
| `ERR_SYSTEM_ACCESS_DENIED` | 应归入PERMISSION分类 | `ERR_PERMISSION_DENIED` |
| `ERR_TABLE_NOT_FOUND` | 应归入DB分类 | `ERR_DB_TABLE_NOT_FOUND` |
| `ERR_GENERATE_CHART` | 应归入CHART分类 | `ERR_CHART_GENERATE` |
| `ERR_CONVERT_DOCUMENT` | 应归入CONVERT分类 | `ERR_CONVERT_FAILED` |
| `ERR_FILTER_DATA` | 应归入FILTER分类 | `ERR_FILTER_INVALID` |

**策略**：现有166个code本轮不动（避免大规模改动风险），新增code必须遵守规范。下个版本逐步替换不合规code。

## 三、两条输出路径

### 3.1 为什么需要两条路径

同一个结果要给两个完全不同的消费者：

| 消费者 | 关注点 | 不需要的字段 |
|--------|--------|-------------|
| **LLM（observation文本）** | 成功/失败 + 关键数据 + 错误原因 | next_actions、retry_count、warning |
| **前端（SSE事件）** | 状态颜色 + toast + 展示数据 + 按钮 | llm_data |

一个格式硬扛两边，要么LLM收到一堆没用的前端字段，要么前端拿不到需要的字段。

### 3.2 路径A：`_format_llm_observation()` — 给LLM看

```python
def _format_llm_observation(result: dict) -> str:
    """格式化工具结果为LLM observation文本 — 小沈 2026-05-20"""
    code = result.get("code", "SUCCESS")

    if code == "SUCCESS":
        display_data = result.get("llm_data") or result.get("data")
        text = f"Observation: success - {result.get('message', '')}"
        if result.get("warning"):
            text += f"\n⚠ 警告: {result['warning']}"
        if display_data:
            text += f"\n数据: {json.dumps(display_data, ensure_ascii=False)}"
        return text

    elif code.startswith("WARNING_"):
        text = f"Observation: warning - {result.get('message', '')}"
        if result.get("data"):
            text += f"\n部分数据: {json.dumps(result['data'], ensure_ascii=False)}"
        return text

    else:  # ERR_*
        text = f"Observation: error [{code}] - {result.get('message', '')}"
        return text
```

**规则**：
- LLM 只需要：是否成功 + 关键数据 + 错误描述
- `llm_data` 优先于 `data`（省token）
- 不传 `next_actions`、`retry_count`、`return_direct`
- `warning` 字段（轻微警告）拼入observation文本；`WARNING_`前缀（严重）走独立分支

### 3.3 路径B：`_format_frontend_event()` — 给前端

```python
def _format_frontend_event(result: dict) -> dict:
    """格式化工具结果为前端SSE事件 — 小沈 2026-05-20"""
    event = {
        "code": result.get("code", "SUCCESS"),
        "message": result.get("message", ""),
        "data": result.get("data"),
        "retry_count": result.get("retry_count", 0),
        "return_direct": result.get("return_direct", False),
    }
    if result.get("warning"):
        event["warning"] = result["warning"]
    if result.get("next_actions"):
        event["next_actions"] = result["next_actions"]
    return event
```

**规则**：
- 前端需要：code（确定图标颜色）、message（toast）、data（展示内容）、next_actions（按钮）
- 不传 `llm_data`（前端不需要）
- `warning` 字段传给前端展示黄色警告标识
- `execution_time_ms` 由 StepFactory 注入（不来自工具返回值）

### 3.4 字段归属矩阵

| 统一字段 | LLM observation | 前端 event | 逻辑层消费 |
|---------|----------------|-----------|-----------|
| `code` | 拼入失败文本 | ✅ 透传 | Agent判定是否重试 |
| `data` | ✅ 展示 | ✅ 展示 | StepFactory |
| `message` | ✅ 主体文本 | ✅ toast文本 | — |
| `warning` | ⚠ 轻微时拼入 | ✅ 展示警告 | — |
| `llm_data` | ✅ **优先于data** | ❌ 不传 | — |
| `next_actions` | ❌ 不传 | ✅ 按钮列表 | — |
| `retry_count` | ❌ 不传 | ✅ 展示 | Agent填充 |
| `return_direct` | ❌ 不传 | ✅ 透传 | Agent消费 |

## 四、改造步骤

### Step 1：`file_tools.py` — 删除 `_to_unified_format`

- 130 处 `_to_unified_format({...}, tool_name)` 替换为直接返回 `{code, data, message}`
- 删除 `_to_unified_format` 函数定义
- 删除 `_generate_summary` 函数定义（仅被_to_unified_format使用）
- 删除 `file/__init__.py` 中的导出

**替换规则**：

| 旧写法 | 新写法 |
|--------|--------|
| `_to_unified_format({"success": True, "data": x, ...}, "tool_name")` | `{"code": "SUCCESS", "data": x, "message": "具体描述"}` |
| `_to_unified_format({"success": False, "error": "原因"}, "tool_name")` | `{"code": "ERR_xxx", "data": None, "message": "原因"}` |

**关键**：每处替换时，`message` 要写具体描述（不是统一模板），`code` 要用有意义的ERR_xxx（不是笼统的ERROR）。

### Step 2：`tool_executor.py` — 删除 `_format_result`

- 删除 `_format_result` 整个函数（7个分支全部删除）
- `execute()` 和 `_execute_with_retry()` 所有 `self._format_result(result, action)` → `return result`
- 重试路径 `return {}` 改为统一格式 `{code, data, message, retry_count}`
- 新增 `_format_llm_observation()` 和 `_format_frontend_event()` 两个函数（见第三节）

### Step 3：`base_react.py` — 适配新格式

- `result.get("status")` → 根据 `result.get("code")` 判定：
  - `"SUCCESS"` → success
  - `"WARNING_*"` → warning
  - `"ERR_*"` → error
- `result.get("summary")` → `result.get("message")`
- `result.get("error")` → `result.get("message")`（失败时message就是错误描述）
- 调用 `_format_llm_observation()` 生成 observation 文本
- 调用 `_format_frontend_event()` 生成前端事件

### Step 4：StepFactory — code 映射

| code条件 | execution_status |
|----------|-----------------|
| `code == "SUCCESS"` 且无 warning | `"success"` |
| `code == "SUCCESS"` 且有 warning 字段 | `"warning"` |
| `code.startswith("WARNING_")` | `"warning"` |
| `code.startswith("ERR_")` | `"error"` |

其他映射：
- `message` → `summary`
- `data` → `execution_result`
- `retry_count` → `action_retry_count`
- `execution_time_ms` — StepFactory自行注入，不来自工具返回值

### Step 5：`/tool/execute` 端点（health.py）适配

当前 health.py 的 `/tool/execute` 端点直接返回工具原始结果。统一格式后：
- 工具返回值已经统一，无需额外格式化
- 但需要确保 `_current_task_id` ContextVar 已设置（已修复）

### Step 6：其他 6 类工具 — 补齐字段

- 检查所有 `return {"code": ..., "data": ..., "message": ...}` 是否缺 `retry_count`
- 检查新增 `warning` 字段的入口
- 确认 `data` 字段不含格式元信息（旧B3分支的问题）
