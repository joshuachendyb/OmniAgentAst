## 零、混乱根源

三个根本原因：

1. **没人区分"给LLM看"和"给前端看"是两回事**，用同一个 `_format_result` 函数强行猜格式、暴力压缩，code/message/warning 全丢了
2. **code 值没有命名规范**，想到什么写什么，166个 code 各自为政
3. **file_tools 双重转换**：先 `_to_unified_format` 注入冗余字段（code+status双写、message=summary重复），再 `_format_result` 猜格式拆开——两层转换互相抵消

### 零点一、现状数据流（要砍掉的）

```
工具函数 return
    │
    ├─ file_tools: return _to_unified_format({success, error, ...}, tool_name)
    │     │
    │     ▼ 输出混合格式: {code, status, summary, data, message, retry_count}
    │     （code=status重复，message=summary重复）
    │
    └─ 其他6类: return {code, data, message}
          │
          ▼ 
    ToolExecutor._format_result() ← 7个分支猜格式
          │
          ├─ B1: status=="warning"     → {status, summary, data, retry_count}
          ├─ B2: 有 status+summary     → {status, summary, data, retry_count}   ← file走这
          ├─ B3: code=="SUCCESS"       → {status, summary, data:=整个result}    ← 其他工具成功
          ├─ B4: code以ERR_开头        → {status, summary, data:=整个result}
          ├─ B5: success==True         → ...
          ├─ B6: 兜底                  → ...
          └─ B7: 非dict               → ...
          │
          ▼ 输出: {status, summary, data, retry_count}
          │
    base_react.py ← 读 status + summary + data + error
```

**核心问题**：
- B2分支 `data = result.get("data")`（取子字段），B3分支 `data = result`（整个dict）——**data语义不一致**
- `code`、`message` 等字段在 `_format_result` 后丢失，Agent层面永远读不到
- 双重转换做无用功

## 一、统一内部格式

### 1.1 所有工具函数的返回格式

```python
{
    "code": str,                # 必须。见第二节 code 命名规范
    "data": Any,                # 必须。成功=业务数据dict，失败=None
    "message": str,             # 必须。人类可读的描述文字

    "warning": str | None,      # 可选。code=SUCCESS但有异常时
    "llm_data": dict | None,    # 可选。给LLM的精简数据（省token）
    "next_actions": list | None, # 可选。前端下一步按钮
    "retry_count": int,         # 可选。实际重试次数，默认0（Agent层填充）
    "return_direct": bool,      # 可选。true=直接返回给用户，跳过LLM继续推理
    "attachment": dict | None,  # 可选。二进制附件(base64图片/文件)，前端渲染
}
```

**3个必须，6个可选。没有冗余字段。**

### 1.2 data 字段语义规范

| code值 | data内容 | 说明 |
|--------|---------|------|
| `SUCCESS` | 业务数据dict | 只含业务字段，不含code/message等格式元信息 |
| `WARNING_*` | 业务数据dict | 部分成功的业务数据 |
| `ERR_*` | `None` | 失败时无业务数据 |

**禁止**：data中嵌套code/message/status等格式字段（旧B3分支的毛病）。

### 1.3 现状字段清理

| 现状字段 | 去向 | 原因 |
|---------|------|------|
| `success: bool` | ❌ 删除 | 被 `code="SUCCESS"` 取代 |
| `status: str` | ❌ 删除 | 被 `code` 取代（"success/error/warning" 太粗糙） |
| `summary: str` | ❌ 删除 | 被 `message` 取代（一模一样的东西） |
| `error: str` | ❌ 删除 | 被 `code="ERR_XXX" + message` 取代 |
| `code: str` | ✅ 保留 | 从"部分工具有"变成"所有工具有" |
| `message: str` | ✅ 保留 | 从"部分工具有"变成"所有工具有" |
| `data: Any` | ✅ 保留 | 不变，但语义统一（见1.2节） |
| `llm_data` | ✅ 保留 | 不变 |
| `next_actions` | ✅ 保留 | 不变 |
| `warning` | ✨ 新增 | code=SUCCESS但有情况要提醒时 |
| `retry_count` | ✅ 保留 | 从"部分工具有"变成统一字段 |
| `return_direct` | ✅ 保留 | 从agent层下沉到工具返回 |
| `attachment` | ✨ 新增(v3.0) | 二进制附件(base64图片/文件)，前端渲染 |
| `capabilities_used` | ❌ 删除(v3.0) | 僵尸字段：64处写入，全链路0消费 |
| `capabilities_missing` | ❌ 删除(v3.0) | 僵尸字段：13处写入，全链路0消费 |

## 二、code 命名规范 ← 重点

### 2.1 三段式结构

```
ERR_分类_具体错误
SUCCESS
```

三段之间用下划线连接，全大写。

### 2.2 三个允许值前缀

| 前缀 | 含义 | LLM行为 | 前端行为 |
|------|------|---------|---------|
| `SUCCESS` | 执行成功 | 正常处理结果 | 绿色图标 |
| `ERR_` | 执行失败 | 如果可重试则重试，否则报错 | 红色图标 + toast |
| `WARNING_` | 执行成功但有重大问题 | 注意但不阻断 | 黄色图标 |

> `WARNING_` 和 `SUCCESS + warning字段` 的区别：
> - `WARNING_` prefix = 比较严重的问题，LLM 和 Agent 应该关注
> - `SUCCESS` + `warning` 字段 = 轻微提示，只有前端展示，Agent 不关心

## 三、两条输出路径

### 3.1 为什么需要两条路径

同一个结果要给两个完全不同的消费者：

| 消费者 | 关注点 | 不需要的字段 |
|--------|--------|-------------|
| **LLM（observation文本）** | 成功/失败 + 关键数据 + 错误原因 | next_actions、retry_count、warning |
| **前端（SSE事件）** | 状态颜色 + toast + 展示数据 + 按钮 | llm_data |

一个格式硬扛两边，要么LLM收到一堆没用的前端字段，要么前端拿不到需要的字段。

### 3.2 路径A：`_format_llm_observation()` — 给LLM看

```python
def _format_llm_observation(result: dict) -> str:
    """
    格式化工具结果为LLM observation文本 — 小沈 2026-05-20
    更新 2026-05-21（小健审计+北京老陈决策）：
      - 去掉一刀切截断，改为工具自控+安全兜底
      工具通过 llm_data 自行控制给LLM的数据量，格式化层不做业务截断
    """
    import json

    code = result.get("code", "SUCCESS")
    LLM_SAFE_LIMIT = 100_000  # 仅防 json.dumps OOM，非业务截断

    if code == "SUCCESS":
        # 优先 llm_data（工具自控精简数据），无则用完整 data
        display_data = result.get("llm_data") or result.get("data")
        text = f"Observation: success - {result.get('message', '')}"
        if result.get("warning"):
            text += f"\n⚠ 警告: {result['warning']}"
        if display_data:
            # 不做业务截断，只防 json.dumps 爆内存
            if isinstance(display_data, (dict, list)):
                display_data = _safe_truncate(display_data, LLM_SAFE_LIMIT)
            text += f"\n数据: {json.dumps(display_data, ensure_ascii=False)}"
        return text

    elif code.startswith("WARNING_"):
        text = f"Observation: warning - {result.get('message', '')}"
        if result.get("data"):
            data = result["data"]
            if isinstance(data, (dict, list)):
                data = _safe_truncate(data, LLM_SAFE_LIMIT)
            text += f"\n部分数据: {json.dumps(data, ensure_ascii=False)}"
        return text

    else:  # ERR_*
        text = f"Observation: error [{code}] - {result.get('message', '')}"
        return text
```

**规则**：
- **工具自控原则**：工具通过 `llm_data` 字段自行控制给LLM的数据量。`llm_data` 存在时，格式化层原样传递，**不做任何截断**。
- **无 llm_data 回退**：工具未提供 `llm_data` 时，使用完整 `data`。格式化层**不做业务截断**——LLM需要看到全貌数据才能正确推理。
- **安全兜底**：仅在 `data` 极端大（>100000元素）时做 `_safe_truncate`，目的**不是限制LLM视角**，而是防止 `json.dumps` 内存溢出。阈值设得很大（100K），正常业务数据不会触发。
- 不传 `next_actions`、`retry_count`、`return_direct`
- `warning` 字段（轻微警告）拼入observation文本；`WARNING_`前缀（严重）走独立分支
- 现有 `tool_result_utils.py` 中的 `DEFAULT_MAX_*` 常量（5000/8000/10000）是**工具内部**生成 `llm_data` 时用的，不是格式化层用的。格式化层只做安全兜底。

### 3.3 路径B：`_format_frontend_event()` — 给前端

```python
def _format_frontend_event(result: dict) -> dict:
    """格式化工具结果为前端SSE事件 — 小沈 2026-05-20"""
    event = {
        "code": result.get("code", "SUCCESS"),
        "message": result.get("message", ""),
        "data": result.get("data"),
        "retry_count": result.get("retry_count", 0),
        "return_direct": result.get("return_direct", False),
    }
    if result.get("warning"):
        event["warning"] = result["warning"]
    if result.get("next_actions"):
        event["next_actions"] = result["next_actions"]
    return event
```

**规则**：
- 前端需要：code（确定图标颜色）、message（toast）、data（展示内容）、next_actions（按钮）
- 不传 `llm_data`（前端不需要）
- `warning` 字段传给前端展示黄色警告标识
- `execution_time_ms` 由 StepFactory 注入（不来自工具返回值）

### 3.4 字段归属矩阵

| 统一字段 | LLM observation | 前端 event | 逻辑层消费 |
|---------|----------------|-----------|-----------|
| `code` | 拼入失败文本 | ✅ 透传 | Agent判定是否重试 |
| `data` | ✅ 展示 | ✅ 展示 | StepFactory |
| `message` | ✅ 主体文本 | ✅ toast文本 | → summary |
| `warning` | ⚠ 轻微时拼入 | ✅ 展示警告 | — |
| `llm_data` | ✅ **优先于data** | ❌ 不传 | — |
| `next_actions` | ✅ **推荐文本** | ✅ 按钮列表 | — |
| `retry_count` | ❌ 不传 | ✅ 展示 | Agent填充 |
| `return_direct` | ❌ 不传 | ✅ 透传 | Agent消费 |
| `attachment` | ❌ 不传 | ✅ 透传 | — |

## 四、改造步骤

> **步骤依赖关系**：
> ```
> Step 1 + Step 2  →  Step 3  →  Step 4
> （工具层统一格式）  （执行器直通）  （Agent适配）
>                         Step 5（StepFactory映射，可独立进行）
>                         Step 6（health端点，已就绪）
> ```
> Step 1、Step 2 并行进行，完成后才推进 Step 3（否则 `_format_result` 删除后旧格式会透传断裂）。

### Step 0：`/tool/execute` 端点（health.py）适配

当前 health.py 的 `/tool/execute` 端点直接返回工具原始结果。统一格式后：
- 工具返回值已经统一，无需额外格式化
- 但需要确保 `_current_task_id` ContextVar 已设置

**`_current_task_id` 重构：移到公共位置**

现状问题：`_current_task_id` 定义在 `file_tools.py:39`（模块私有 `_current_task_id`），但被3个模块交叉引用：

| 引用方 | 用途 |
|--------|------|
| `file_tools.py` 自身（7处） | FileSearcher/FileReader 等类读取 task_id 做操作记录 |
| `base_react.py:792` | Agent 执行工具前设置 task_id（函数内 import，原 L52 模块级已删除） |
| `health.py:116` | `/tool/execute` 测试端点手动设置 task_id |

重构方案：
- 新增 `backend/app/services/context_vars.py`，定义 `_current_task_id` 为公共 ContextVar，名称改为 `"tool_task_id"`（不再限定 file_tools）
- `file_tools.py`：删除本地 L39 定义，改为 `from app.services.context_vars import _current_task_id`
- `base_react.py`：同上

## 八、`_response.py` 统一构建函数方案 — 小健 2026-05-21

### 8.1 问题背景

工具函数手写返回字典，731处（粗估，含字段行/变量行；实际独立return语句619处）格式不完全统一：
- 字段名拼错不会报错
- 可选字段有时写有时不写，消费端行为不一致
- 新人不知道该写哪些字段

`_response.py`提供`build_success()`/`build_error()`/`build_warning()`三个构建函数，统一出口。

### 8.2 设计原则

1. **必填字段(code/data/message)始终写入**，可选字段仅非默认值时写入
2. **函数签名简洁**，80%场景只需2-3个参数
3. **扩展性通过`**extra`**，未来新增字段无需改签名
4. **工具层和Agent层分离**：`_response.py`只负责生成标准格式，不承担Agent消费格式的转换

### 8.3 函数清单

| 函数 | 用途 | 典型调用 |
|------|------|---------|
| `build_success(data, message, ...)` | 成功返回 | `build_success({"path": fp}, f"写入成功: {fp}")` |
| `build_error(code, message, ...)` | 错误返回 | `build_error("ERR_FILE_NOT_FOUND", f"文件不存在: {fp}")` |
| `build_warning(code, message, ...)` | 警告返回(部分成功) | `build_warning("WARNING_DB_SAFETY", "SQL含CREATE", data=result)` |
| `is_success(result)` | 判断是否成功(WARNING_视为成功) | — |
| `is_error(result)` | 判断是否失败 | — |

### 8.4 字段规范

**必填(3个)**：始终写入
- `code`: "SUCCESS" | "ERR_MODULE_OPERATION_DETAIL" | "WARNING_MODULE_OPERATION_DETAIL"
- `data`: 结构化返回数据(给前端)
- `message`: 人类可读结果描述

**可选(6个)**：仅非默认值时写入
- `warning`: 默认None — 成功但有风险时的警告信息
- `llm_data`: 默认None — 给LLM的精简数据(控制上下文占用)
- `next_actions`: 默认None — 推荐后续操作列表
- `retry_count`: 默认0 — 重试次数(仅>0时写入)
- `return_direct`: 默认False — 是否直接返回前端(仅True时写入)
- `attachment`: 默认None — 二进制附件(base64图片/文件等，前端渲染，仅非None时写入)

**扩展**: 通过`**extra`支持任意新字段，无需改签名
```python
# 2027年新增字段，无需改_response.py
return build_success(data, msg, confidence=0.95, cache_hit=True)
```

### 8.5 可选字段写入规则

默认值不写入，避免消费端`result.get("retry_count")`返回0而非None的歧义：

```python
# _OPTIONAL_FIELDS注册表 — 新增可选字段只需加一行
_OPTIONAL_FIELDS = {
    "warning": None,
    "llm_data": None,
    "next_actions": None,
    "retry_count": 0,
    "return_direct": False,
}

# _add_optionals内部函数：仅写入非默认值
def _add_optionals(result, **kwargs):
    for field_name, default_value in _OPTIONAL_FIELDS.items():
        if field_name in kwargs and kwargs[field_name] != default_value:
            result[field_name] = kwargs[field_name]
```

### 8.6 WARNING_的语义

WARNING_开头的code表示**部分成功**，语义链路：

```
工具返回 WARNING_DB_SAFETY
    ↓
_format_llm_observation() → "Observation: warning - SQL含CREATE，部分数据: {...}"
    ↓
LLM看到warning文本，知道部分成功，可以继续推理
    ↓
Agent继续执行（不中断，WARNING_视为成功）
```

`is_success()`对WARNING_返回True，`is_error()`返回False。

### 8.7 `to_agent_step()` — 不应有，已删除

**理由**：`_response.py`属于工具层，`{status,summary,data,error}`属于Agent内部消费格式。两者耦合会导致：
- Agent改格式要改`_response.py`——违反分层原则
- 工具层承担了不该承担的转换责任

**正确做法**：把base_react.py和reasoning_steps.py的重复`code→status`转换逻辑抽到`tool_result_formatter.py`的`extract_status()`函数，消除重复但不跨层耦合。

### 8.8 `extract_status()` — 消除重复转换

当前`code→status`转换逻辑在两处重复且不完全一致：

| 位置 | 行号 | 逻辑 |
|------|------|------|
| base_react.py | 770-772 | SUCCESS→success, WARNING_→warning, ERR_→error |
| reasoning_steps.py | 858-864 | SUCCESS+warning→warning, WARNING_→warning, ERR_→error |

统一为：

```python
# tool_result_formatter.py 新增
def extract_status(code: str, has_warning: bool = False) -> str:
    """code→status统一映射 — 消除base_react.py和reasoning_steps.py的重复"""
    if code == "SUCCESS":
        return "warning" if has_warning else "success"
    elif code.startswith("WARNING_"):
        return "warning"
    else:
        return "error"
```

base_react.py和reasoning_steps.py都调用`extract_status()`，**消除重复**。

## 十一、前端完整视图：一次工具执行前端收到两个SSE事件 — 小健 2026-05-22

### 11.1 前端视角：连续收到两个事件

前端对一次工具执行的感知是**连续收到两个SSE事件**，两者职责分明、字段互补：

```
工具执行完成
     │
     ├─→ SSE event 1: action_tool  ← 执行摘要：status/data/耗时/重试
     │
     └─→ SSE event 2: observation  ← 详细信息：code/warning/summary/error_message/next_actions/attachment
```

**设计**：action_tool 和 observation 各自负责不同维度的信息，不重复、不遗漏。前端可从 action_tool 取业务数据和执行状态，从 observation 取摘要文本、错误码、警告等。

### 11.2 完整的三分支图

```
                    execution_result（工具原始返回）
                           │
           ┌───────────────┼───────────────┐
           │               │               │
      分支1: LLM       分支2: 前端      分支3: return_direct
           │               │               │
  _format_llm_observation() │         L883: if is_done()
  + 质量检测                │         L884-901: json.dumps(data)
  + 缓存前缀                │         create_final_step()
  + 失败汇总                │         yield final_step
           │               │         return ← 终止循环
           │               │
  add_observation()        │
  → conversation history   │
  → LLM下一轮输入          │
                           │
                  ┌────────┴────────┐
                  │                 │
            SSE event 1:       SSE event 2:
            action_tool        observation
            （执行摘要）        （详细信息）
            status/data/       code/warning/summary/
            耗时/重试          error_message/next_actions/attachment
```

### 11.3 前端收到的两个SSE事件对比

**事件1：action_tool**（执行摘要）

```json
{
    "type": "action_tool",
    "step": 3,
    "tool_name": "write_text_file",
    "tool_params": {"file_path": "/tmp/test.txt", "text": "hello"},
    "execution_status": "success",
    "execution_result": {"path": "/tmp/test.txt", "size": 5},
    "raw_data": {"path": "/tmp/test.txt", "size": 5},
    "action_retry_count": 0,
    "execution_time_ms": 12
}
```

**事件2：observation**（详细信息）

```json
{
    "type": "observation",
    "step": 3,
    "tool_name": "write_text_file",
    "tool_params": {"file_path": "/tmp/test.txt", "text": "hello"},
    "observation": "写入成功: /tmp/test.txt",
    "execution_status": "success",
    "code": "SUCCESS",
    "return_direct": false,
    "summary": "写入成功: /tmp/test.txt",
    "error_message": ""
}
```

### 11.4 两个事件的字段覆盖分析

| 字段 | action_tool SSE | observation SSE | 谁有 | 前端取哪个？ |
|------|-----------------|-----------------|------|-------------|
| `execution_status` | ✅ | ✅ | 两者 | 任一 |
| `execution_result` (业务数据) | ✅ | ❌ | action_tool | action_tool |
| `execution_time_ms` | ✅ | ❌ | action_tool | action_tool |
| `action_retry_count` | ✅ | ❌ | action_tool | action_tool |
| `tool_name` | ✅ | ✅ | 两者 | 任一 |
| `tool_params` | ✅ | ✅ | 两者 | 任一 |
| `summary` | ❌ | ✅ | observation | observation |
| `error_message` | ❌ | ✅ | observation | observation |
| `observation` (文本) | ❌ | ✅ | observation | observation |
| `return_direct` | ❌ | ✅ | observation | observation |
| `code` (原始错误码) | ❌ | ✅ | observation | observation |
| `warning` | ❌ | ✅ | observation | observation |
| `next_actions` | ❌ | ✅ | observation | observation |
| `attachment` | ❌ | ✅ | observation | observation |

**结论**：两个事件职责分明、字段互补不重复。action_tool 负责执行摘要（status/data/耗时/重试），observation 负责详细信息（code/warning/summary/error_message/next_actions/attachment）。前端无需跨事件拼凑信息。

### 11.5 分支3：return_direct 提前终止

**触发条件**：工具返回 `return_direct=True`（如 `execute_code` 要求直接返回结果给用户）

**执行路径**：`base_react.py:883-902`

```python
if observation_step.is_done():  # return_direct == True
    _result_data = execution_result.get("data")
    try:
        _response_text = json.dumps(_result_data, ensure_ascii=False) if _result_data is not None else ""
    except (TypeError, ValueError):
        _response_text = str(_result_data)
    _msg = execution_result.get("message", "")
    if _msg:
        _response_text = _msg + "\n" + _response_text
    final_step = StepFactory.create_final_step(
        step=step_count,
        response=_response_text,
        thought="工具执行要求直接返回结果",
        model=..., provider=...
    )
    self.steps.append(final_step)
    yield final_step.to_dict()  # SSE event: final
    self._on_after_loop()
    return  # 终止ReAct循环
```

**前端收到3个SSE事件**：

```
SSE event 1: action_tool   ← 正常
SSE event 2: observation   ← 正常，但 return_direct=true
SSE event 3: final         ← 分支3特有，包含最终结果
```

**final事件JSON**（B类修复后，使用 json.dumps 而非 str）：

```json
{
    "type": "final",
    "step": 3,
    "response": "写入成功: /tmp/test.txt\n{\"path\": \"/tmp/test.txt\", \"size\": 5}",
    "thought": "工具执行要求直接返回结果",
    "model": "glm-4",
    "provider": "zhipu"
}
```

注：B类修复已将 `str(data)` 改为 `json.dumps(data)`，确保输出合法JSON。

### 11.6 三分支完整对比

| 维度 | 分支1：LLM history | 分支2：前端SSE | 分支3：return_direct |
|------|-------------------|----------------|---------------------|
| **产出** | observation_text → conversation | action_tool + observation SSE | + final SSE + 终止循环 |
| **数据格式** | 纯文本（拼装） | 结构化dict（Step.to_dict） | 结构化dict（FinalStep.to_dict） |
| **信息量** | 最丰富（数据+推荐+警告+汇总） | action_tool(摘要) + observation(详细)，互补不重复 | message + json.dumps(data) |
| **消费方** | LLM下一轮推理 | 前端UI渲染 | 前端最终展示 |
| **构建入口** | `_format_llm_observation()` | `StepFactory.create_*_step()` | `StepFactory.create_final_step()` |
