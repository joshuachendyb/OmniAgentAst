# LLM核心方法架构全新设计

版本: v1.0 | 创建时间: 2026-05-16 13:30:00 | 作者: 小沈

---

## 一、设计原则

1. **从需求出发**，不迁就现有代码
2. **每个方法都有明确的使用场景**，没有死代码
3. **调用者不需要选择**，场景决定方法，方法决定一切
4. **格式逻辑集中**，不在每个方法里重复

---

## 二、需求分析：为什么有流式和非流式两种？

不是前端决定，不是用户决定，是**使用目的**决定：

### 2.1 要实时看输出 → 流式

场景：用户直接对话，LLM回答可能要30秒
- 如果等30秒才一次性出来完整回答，用户体验极差
- 逐字出来，用户感觉LLM在"思考"，体验好
- **谁调用**：SSE服务层，把流式chunk逐个推给客户端渲染

### 2.2 要完整结果做后续处理 → 非流式

场景：Agent调LLM，拿到结果后要解析tool_calls，决定下一步
- 必须**拿到完整响应**才能解析
- 流式没有意义——最终还是要等完整结果
- **谁调用**：Agent的2种策略（TextStrategy / ToolsStrategy）

### 2.3 流式和非流式都是对LLM API说的

- `stream=True`：LLM逐token返回SSE事件
- `stream=False`：LLM一次返回完整JSON

这是LLM API的**协议特征**，不是前端概念。调用者根据使用目的选择。

### 2.4 FC模式也可以流式

OpenAI API支持 `stream=True` + `tools=[...]`，LLM会流式返回思考文本+tool_calls。

场景：Agent用FC模式时，LLM在决定调tool之前先"思考"，这段思考文字流式给用户看，体验远比等完整响应好。

### 2.5 4种组合全部有场景

| | 非流式 | 流式 |
|---|---|---|
| **text** | ✅ Agent拿完整文本 | ✅ SSE实时看 |
| **tools** | ✅ Agent拿完整tool_calls | ✅ 实时看思考+流式拿tool_calls |

> **说明**：response_format模式已删除，因为当前使用的LLM不支持JSON Schema，Agent解析不可靠。

### 2.6 当前实际使用

| 组合 | 当前是否在用 | 说明 |
|------|-------------|------|
| text + 流式 | ✅ | SSE服务层 → chat_stream() |
| text + 非流式 | ✅ | chat()内部聚合chat_stream() |
| tools + 非流式 | ✅ | ToolsStrategy → chat_with_tools() |
| tools + 流式 | ❌ 未用 | 有场景，现有chat_with_tools_stream()是死代码 |

## 三、架构设计

### 3.1 正确的划分

**底层**：2个方法，覆盖流式/非流式

| 方法 | 返回 | 说明 |
|------|------|------|
| `request()` | `ChatResponse` | 非流式，一次拿到完整响应 |
| `request_stream()` | `AsyncGenerator[StreamChunk]` | 流式，逐token返回 |

**上层**：1个便捷方法

| 方法 | 返回 | 说明 |
|------|------|------|
| `chat()` | `AsyncGenerator[StreamChunk]` | 前端SSE专用，就是request_stream的别名 |

**调用模式**通过参数控制，不通过方法名：

```python
# 前端SSE
async for chunk in service.request_stream(
    messages=messages,
    mode="text"              # 纯文本
):
    yield chunk

# Agent FC模式
response = await service.request(
    messages=messages,
    mode="tools",            # Function Calling
    tools=tools
)


# Agent纯文本模式
response = await service.request(
    messages=messages,
    mode="text"
)
```

### 3.3 完整方法清单

```python
class BaseAIService:
    
    async def request(
        self,
        messages: List[Dict],
        mode: str = "text",                    # "text" / "tools"
        tools: Optional[List[Dict]] = None,
        tool_choice: str = "auto",
    ) -> ChatResponse:
        """非流式请求 - Agent用"""
    
    async def request_stream(
        self,
        messages: List[Dict],
        mode: str = "text",                    # "text" / "tools"
        tools: Optional[List[Dict]] = None,
        tool_choice: str = "auto",
    ) -> AsyncGenerator[StreamChunk, None]:
        """流式请求 - SSE服务层用"""
    
    async def chat(
        self,
        message: str,
        history: Optional[List[Message]] = None,
    ) -> AsyncGenerator[StreamChunk, None]:
        """流式对话便捷方法 - SSE服务层用"""
```

### 3.4 方法之间的关系

```
chat(message, history)                    # SSE服务层入口
  └─ _build_messages(message, history)    # 构建消息（格式由_array_format控制）
  └─ request_stream(messages, mode="text")  # 流式纯文本

request(messages, mode, tools, ...)      # Agent非流式入口
  └─ _build_request_body(messages, mode, tools, ...)  # 构建请求体
  └─ httpx.post(...)                     # 发请求
  └─ _parse_response(response, mode)     # 解析响应

request_stream(messages, mode, tools, ...)  # 流式入口
  └─ _build_request_body(messages, mode, tools, ...)  # 构建请求体
  └─ httpx.stream(...)                   # 发流式请求
  └─ _parse_stream_chunk(chunk)          # 解析每个chunk
```

### 3.5 与现有代码的对比

| 现有 | 新设计 | 说明 |
|------|--------|------|
| `chat()` | `chat()` | 保留，前端SSE入口 |
| `chat_stream()` | `request_stream(mode="text")` | 统一为request_stream |
| `chat_with_tools()` | `request(mode="tools")` | 统一为request |
| `chat_with_tools_stream()` | 删除 | 死代码 |
| 无 | `request_stream(mode="tools")` | 如未来需要FC流式，无需新增方法 |

**核心变化**：从**4个方法按名字分模式** → **2个方法按参数分模式**

## 四、mode参数详解

### 4.1 mode="text"（纯文本）

```python

response = await service.request(messages=msgs, mode="text")
# request_body: {"model": ..., "messages": msgs}

# 流式
async for chunk in service.request_stream(messages=msgs, mode="text"):
    yield chunk
# request_body: {"model": ..., "messages": msgs, "stream": True}
```

### 4.2 mode="tools"（Function Calling）

```python

response = await service.request(messages=msgs, mode="tools", tools=tools)
# request_body: {"model": ..., "messages": msgs, "tools": tools, "tool_choice": "auto"}

# 流式（预留，当前Agent不需要）
async for chunk in service.request_stream(messages=msgs, mode="tools", tools=tools):
    yield chunk
# request_body: {"model": ..., "messages": msgs, "tools": tools, "stream": True}
```

## 五、_build_request_body 统一构建请求体

```python
def _build_request_body(
    self,
    messages: List[Dict],
    mode: str = "text",
    stream: bool = False,
    tools: Optional[List[Dict]] = None,
    tool_choice: str = "auto",
) -> Dict:
    """构建LLM API请求体 - 小沈 2026-05-16"""
    body = {
        "model": self.model,
        "messages": messages,
        "stream": stream,
        "max_tokens": self.max_tokens,
        "temperature": self.temperature,
    }
    if self.seed is not None:
        body["seed"] = self.seed
    
    if mode == "tools" and tools:
        body["tools"] = tools
        body["tool_choice"] = tool_choice
    
    return body
```

## 六、消息格式兼容（_build_messages）

```python
def _build_messages(self, message: str, history: Optional[List[Message]] = None) -> List[Dict]:
    """构建消息列表 - 格式由self._array_format控制 - 小沈 2026-05-16"""
    messages = []
    if history:
        for msg in history:
            if self._array_format and msg.role in ("system", "user"):
                messages.append({"role": msg.role, "content": [{"type": "text", "text": msg.content}]})
            else:
                messages.append({"role": msg.role, "content": msg.content})
    
    if self._array_format:
        messages.append({"role": "user", "content": [{"type": "text", "text": message}]})
    else:
        messages.append({"role": "user", "content": message})
    
    return messages
```

`_array_format` 由 CapabilityDetector 探测后设置，详见消息格式兼容设计文档。

## 九、Tool动态加载：跨类别tool需求解决方案

### 9.1 问题

Agent有135个tool分12类别，首次只传相关类别tool给LLM。但LLM执行中可能发现需要其他类别的tool，当前没有该类tool可用。

### 9.2 业界方案调研

| 方案 | 代表项目 | 核心思路 | 处理跨类别 | 缺点 |
|------|---------|---------|-----------|------|
| A. 少而精 | AutoGPT, OpenHands | 只5-10个通用tool | 不存在此问题 | 能力受限 |
| B. 两阶段路由 | LangChain Tool Router | LLM先选tool子集再执行 | 只选一次，执行中不能补 | 额外LLM调用开销 |
| C. RAG for Tools | LangChain社区 | tool描述做向量索引检索 | 只请求前选一次 | 不能执行中补 |
| D. 多Agent分治 | CrewAI, MetaGPT | 不同Agent配不同tool | 靠delegation/消息委托 | 需多Agent架构 |
| E. 工作流编排 | Dify, Coze | 人工设计路由分支 | 人工预判 | 不灵活 |
| F. Plugin动态注册 | Semantic Kernel | 运行时add_plugin | 需自己写调度 | 无自动调度 |

**关键发现**：
- OpenAI官方：**超过10-20个tool，LLM选择准确率显著下降**
- 业界没有"LLM执行中主动发现缺tool并请求加载"的方案
- CrewAI的delegation最接近，但它是多Agent委托，不是单Agent内动态加载

### 9.3 我们的方案：request_tools

给LLM一个特殊tool `request_tools`，LLM调它告诉系统"我需要什么类别的tool"，系统动态加载后LLM即可使用。

```python
# 特殊tool定义
request_tools = {
    "type": "function",
    "function": {
        "name": "request_tools",
        "description": "请求加载其他类别的工具。当你发现当前工具不足以完成任务时调用。",
        "parameters": {
            "type": "object",
            "properties": {
                "categories": {
                    "type": "array",
                    "items": {"type": "string"},
                    "description": "需要加载的工具类别列表"
                }
            },
            "required": ["categories"]
        }
    }
}
```

### 9.4 执行流程

```
首次：意图分类 → 加载file类tool + request_tools → 传给LLM

第1轮：LLM调file_read → 成功
第2轮：LLM发现需要搜索 → 调request_tools(categories=["search"]) 
       → 系统加载search类tool → tool列表更新
第3轮：LLM调web_search → 成功
第4轮：LLM调file_write → 完成
```

### 9.5 system prompt配合

```
你可以随时调用 request_tools(categories=[...]) 来动态加载其他类别的工具。
当前已加载: [file, shell]。
可用但未加载: [search(网络搜索), database(数据库操作), desktop(桌面控制), 
               network(网络操作), code_execution(代码执行), document(文档处理), ...]
```

### 9.6 方案优势与注意事项

**优势**：
- 业界首创：LLM执行中主动发现并请求加载tool
- 单Agent解决跨类别，无需多Agent架构
- LLM最清楚自己需要什么，比请求前RAG检索更精准
- 渐进式暴露：避免一次传135个tool导致选择准确率下降

**注意事项**：
- 每请求新类别多一轮LLM调用（可接受）
- 需要system prompt列出所有类别名（让LLM知道可以请求哪些）
- 需设最大加载类别数防失控（如最多5个类别）
- 已加载的类别不能重复加载

### 9.7 组合策略

**首次用意图分类选初始类别 → 执行中用request_tools动态补充 → 设最大类别数防失控**

与RAG for Tools可组合：首次用RAG选初始tool，执行中用request_tools补充。

## 十、opencode参考分析

> 参考项目：F:\agenttool\opencode（Go语言，AI编程Agent）

### 10.1 Tool体系

**10个内置tool + MCP动态tool，按Agent角色分2组**：

| Agent | 可用tool | 思路 |
|-------|---------|------|
| CoderAgent | bash, edit, fetch, glob, grep, ls, sourcegraph, view, patch, write, agent, diagnostics, MCP | 全量（读写+搜索+子Agent） |
| TaskAgent | glob, grep, ls, sourcegraph, view | 只读搜索 |

**关键设计**：
- **没有分类/Registry**，Go接口+工厂函数，`BaseTool`接口只需实现`Info()`和`Run()`
- **跨分类用"agent"元工具解决**：CoderAgent调`agent`工具启动TaskAgent子代理，TaskAgent只配搜索tool
- **tool使用指南写在Description里**（bash的Description有150行指南），通过API的tools字段传给LLM，**不注入prompt文本**
- MCP动态加载tool（支持stdio+sse两种连接方式）
- 有权限系统（permission.Service），非只读命令需用户确认

**与我们的对比**：

| 维度 | opencode | 我们 |
|------|----------|------|
| Tool数量 | 10+MCP | 135 |
| 分类方式 | 按Agent角色2组 | 按功能12类别 |
| 跨分类 | agent元工具→子Agent | request_tools（设计中） |
| Tool→Prompt | 不注入，靠API tools字段 | 注入prompt文本（text模式） |
| MCP | ✅ 支持 | ❌ |

### 10.2 Prompt架构

**4层结构**：

```
第1层：基础Prompt（Go const硬编码，按Provider分Anthropic版/OpenAI版）
第2层：环境信息（<env>工作目录+git+平台+日期</env> + <project>目录树</project>）
第3层：LSP信息（条件加载）
第4层：项目上下文（自动读OpenCode.md/CLAUDE.md/.cursorrules等文件）
```

**关键设计**：
- **Tool信息不注入System Prompt**，纯靠LLM API的tools字段传递
- 按Provider不同给不同Prompt（Anthropic版比OpenAI版详细得多，约95行vs45行）
- 自动从项目根目录读配置文件注入上下文（`.cursorrules`, `CLAUDE.md`, `opencode.md`等10种路径）
- 4种Agent各有独立System Prompt：Coder/Task/Title/Summarizer

### 10.3 Conversation History处理

**核心策略：摘要截断，不做细粒度裁剪**

| 机制 | 有无 | 方式 |
|------|-----|------|
| Token级裁剪/截断 | ❌ 无 | 不做任何主动截断 |
| 滑动窗口 | ❌ 无 | 不按轮数/token数限制 |
| 摘要/压缩 | ✅ 有 | LLM生成摘要，截断到摘要点 |
| 预检查是否超窗口 | ❌ 无 | 发送前不检查 |
| Anthropic Cache | ✅ 有 | 最后3条消息标记ephemeral cache |

**摘要触发条件**：累计token >= ContextWindow的95% 且 AutoCompact=true（也支持手动触发）

**摘要流程**：
```
1. 加载session全部历史消息
2. 用独立的SummarizerProvider调LLM生成摘要（无tool，纯文本）
3. 创建摘要消息（assistant角色）
4. 更新session.SummaryMessageID = 摘要消息ID
5. 清零PromptTokens
```

**摘要生效方式**（后续请求时）：
```
1. 从DB加载全部历史消息
2. 找到SummaryMessageID对应的消息位置
3. 截断：msgs = msgs[摘要位置:]  ← 丢弃摘要之前的所有历史
4. 把摘要消息的Role从assistant改为user ← 关键！LLM当作用户提供的上下文来理解
5. 正常发送给LLM
```

**核心哲学**：信任大上下文窗口（128K-1M），不做细粒度token管理。只在接近窗口极限时做一次LLM全量摘要，然后截断到摘要点。

**代价**：如果单轮对话中tool-use循环累积的消息就已超出窗口，没有任何保护措施——LLM API返回400错误，opencode不处理。

### 10.4 对我们设计的启发

| opencode做法 | 我们可以借鉴 |
|-------------|------------|
| Tool信息不注入prompt，靠API tools字段 | FC模式下我们也应如此，当前text模式注入prompt是FC不支持时的降级方案 |
| 按Agent角色分tool组 | 我们按意图分类加载tool已有类似思路 |
| agent元工具→子Agent解决跨分类 | 我们request_tools方案更轻量，不需要子Agent |
| 摘要截断策略 | 我们也可以用，当前没有摘要机制 |
| 摘要消息Role改User | 巧妙设计，让LLM把摘要当上下文而非自己的回复 |
| MCP动态tool | 值得引入，扩展tool来源 |
| 自动读项目配置文件注入上下文 | 我们已有AGENTS.md，类似思路 |

## 十一、实施修改方案（10大原则版）

### 11.1 设计原则遵循

本方案严格遵守代码10大原则：

| 原则 | 在本方案中的体现 |
|------|----------------|
| **SRP** | SDK层只管发HTTP请求，Service层只管业务逻辑，职责分离 |
| **DRY** | 统一`_build_request_body()`和`_build_messages()`，消除重复 |
| **KISS** | 3个方法+1个参数，比4个方法更简单 |
| **SLAP** | SDK层和Service层在同一抽象层，不混搭 |
| **YAGNI** | 只实现当前需要的功能，不预留未来扩展 |
| **禁止backward compatibility** | **彻底删除旧代码，不保留任何兼容代码** |
| **OCP** | 新增mode值即可扩展新模式，不修改现有代码 |
| **LSP** | 新方法完全替代旧方法，行为一致 |
| **ISP** | 接口职责单一，每个方法只做一件事 |
| **复用优先** | 复用现有`_build_request_body()`逻辑 |

```python
async def request(self, messages, mode="text", tools=None, tool_choice="auto") -> ChatResponse:
    """非流式请求 - Agent用 - 小沈 2026-06-09"""
    self._ensure_client()
    try:
        response = await self._llm_sdk.request(
            messages=messages,
            mode=mode,
            tools=tools,
            tool_choice=tool_choice,
            max_tokens=self.max_tokens,
            temperature=self.temperature,
            seed=self.seed,
        )
        # 解析响应
        choices = response.get("choices", [])
        if not choices:
            return ChatResponse(content="", model=self.model, provider=self.provider, error="无响应")
        
        msg = choices[0].get("message", {})
        content = msg.get("content", "")
        tool_calls = msg.get("tool_calls", [])
        
        return ChatResponse(
            content=content,
            model=self.model,
            provider=self.provider,
            tool_calls=tool_calls,
        )
    except Exception as e:
        return ChatResponse(content="", model=self.model, provider=self.provider, error=str(e))

async def request_stream(self, messages, mode="text", tools=None, tool_choice="auto") -> AsyncGenerator[StreamChunk, None]:
    """流式请求 - SSE服务层用 - 小沈 2026-06-09"""
    self.reset_cancel()
    self._ensure_client()
    
    async for data_str in self._llm_sdk.request_stream(
        messages=messages,
        mode=mode,
        tools=tools,
        tool_choice=tool_choice,
        max_tokens=self.max_tokens,
        temperature=self.temperature,
        seed=self.seed,
    ):
        if self._cancelled:
            yield self._create_cancelled_chunk()
            return
        
        chunk = self._parse_sse_data(data_str)
        if chunk:
            yield chunk
            if chunk.is_done:
                return
    
    yield StreamChunk(content="", model=self.model, is_done=True)

async def chat(self, message, history=None) -> AsyncGenerator[StreamChunk, None]:
    """流式对话便捷方法 - SSE服务层用 - 小沈 2026-06-09"""
    messages = self._build_messages(message, history)
    async for chunk in self.request_stream(messages=messages, mode="text"):
        yield chunk

def _build_messages(self, message, history=None):
    """构建消息列表 - 小沈 2026-06-09"""
    messages = []
    if history:
        for msg in history:
            messages.append({"role": msg.role, "content": msg.content})
    messages.append({"role": "user", "content": message})
    return messages
```
