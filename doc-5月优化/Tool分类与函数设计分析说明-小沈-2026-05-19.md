## 一、总体架构

### 1.1 分类体系

系统采用 `ToolCategory` 枚举定义7个工具分类，共59个LLM可见工具：

```python
class ToolCategory(Enum):
    FILE = "file"          # 文件操作 — 11个工具
    SHELL = "shell"        # 命令执行 — 5个工具
    NETWORK = "network"    # 网络通信 — 5个工具
    SYSTEM = "system"      # 系统管理 — 10个工具
    DOCUMENT = "document"  # 文档处理 — 9个工具
    DESKTOP = "desktop"    # 桌面GUI — 10个工具
    META = "meta"          # 元工具+时间 — 9个工具
```

### 1.2 设计原则（依据精简方案§2）

| 编号 | 原则 | 轨道 | 在本系统中的体现 |
|------|------|------|-----------------|
| P11 | 统一入口+内部分发 | A | read_document按后缀路由、window_control按action路由、get_time按action路由 |
| P12 | 工具间互相调用 | A | read_document内部自动调用_check_pdf_readable等前置检查 |
| P13 | 分层架构Tool/Helper | A | env_check验证工具降级为document内部helper、gui_helpers降级为内部helper |
| P14 | 优雅降级 | A | read_excel缺pandas降级openpyxl、desktop缺pywin32返回ERR |
| P15 | 返回值自解释 | A | 工具返回含next_actions建议下一步操作 |
| P16 | 幂等性 | A | kill_process对已退出PID返回"无需终止" |
| P17 | 参数正交 | A | rename_file用mode区分single/batch，替代互斥参数组 |
| P1 | 重复合并 | B | read_pdf+read_docx+read_xlsx→read_document |
| P2 | 相似整合 | B | set_window_state+focus_window+resize_window→window_control |
| P3 | 功能互斥 | B | 任意两个工具功能无交集，LLM看到场景只匹配一个工具 |
| P5 | 后缀自动路由 | B | read_document("a.pdf")自动调pdfplumber |
| P9 | 直接删除 | B | 旧函数直接删除不保留委托 |

### 1.3 每个工具的4部分代码结构

| 部分 | 文件类型 | 职责 |
|------|---------|------|
| **Tool函数** | `*_tools.py` | 功能实现，接受参数→执行逻辑→返回{code,data,message} |
| **Schema** | `*_schema.py` | Pydantic参数模型，定义字段名/类型/必填/默认值/描述 |
| **Register** | `*_register.py` | 注册到tool_registry，关联函数+Schema+description+examples |
| **Prompt** | `prompts/*/*.py` | Agent的系统提示，列出可用工具名+参数+用例 |

## 二、FILE分类（11个工具）

### 2.1 分类概述

文件操作分类，覆盖文件的读取、写入、编辑、搜索、重命名、压缩、移动/复制/删除、结构化格式读写等全部文件管理场景。

### 2.2 涉及代码文件

| 文件 | 路径 | 行数 | 职责 |
|------|------|------|------|
| file_tools.py | `backend/app/services/tools/file/file_tools.py` | 3019 | 11个工具函数实现 |
| file_schema.py | `backend/app/services/tools/file/file_schema.py` | 527 | 11个Pydantic参数模型 |
| file_register.py | `backend/app/services/tools/file/file_register.py` | 353 | 注册点：description+examples+注册 |
| file_prompts.py | `backend/app/services/prompts/file/file_prompts.py` | 398 | Prompt模板(F1-F11工具说明+用例) |

### 2.3 工具清单

#### F1. read_file — 统一文件读取

- **设计原则**: P11统一入口(合并read_text_file+read_batch_file)，P17互斥校验(file_path与file_paths二选一)
- **Schema**: `ReadFileInput` — file_path, file_paths, head, tail, offset, limit, encoding
- **参数**: file_path(单文件)与file_paths(批量)互斥；head/tail/offset/limit仅单文件模式
- **实现逻辑**: 单文件→逐行读取支持分页；批量→并行读取多个文件返回列表
- **适用场景**: 读取代码文件、配置文件、日志文件；批量读取多文件；分页读取大文件

#### F2. write_text_file — 写入文本文件

- **设计原则**: P7悲观默认值(encoding默认utf-8, append默认False)
- **Schema**: `WriteTextFileInput` — file_path, text, encoding, append, create_parents, unescape
- **参数**: text必填(必须传实际内容，禁止传思考/计划)；append=True追加模式
- **实现逻辑**: append=False→覆盖写入；append=True→追加写入；自动创建父目录；自动反转义\n等
- **适用场景**: 创建新文件、覆盖写入、追加日志、保存代码

#### F3. read_media_file — 读取媒体文件

- **设计原则**: P4单一职责(只做媒体读取)
- **Schema**: `ReadMediaFileInput` — file_path
- **参数**: file_path必填，支持JPG/PNG/GIF/BMP/WebP/MP3/WAV等
- **实现逻辑**: 读取文件→Base64编码→返回{mime_type, data, size}
- **适用场景**: 读取图片用于OCR、读取音频、展示图片内容

#### F4. edit_file — 统一文件编辑

- **设计原则**: P11统一入口(合并precise_replace_in_file+edit_text_file)，P17互斥(old_string与edits二选一)
- **Schema**: `EditFileInput` — file_path, old_string, new_string, edits, replace_all, ignore_case, dry_run, encoding
- **参数**: old_string+new_string=单处替换；edits=多处结构化编辑[{oldText,newText}]；dry_run=预览不修改
- **实现逻辑**: 单处→精确查找替换；多处→按edits数组顺序依次替换；dry_run→只返回预览diff
- **适用场景**: 修改代码函数名、批量替换配置项、预览修改效果

#### F5. list_directory — 统一目录列表

- **设计原则**: P11统一入口(合并list_directory+get_directory_tree+file_statistics)，P15返回值自解释(始终含statistics)
- **Schema**: `ListDirectoryInput` — dir_path, format, recursive, max_depth, sortBy, include_hidden, exclude_patterns, page_token
- **参数**: format="list"扁平列表/"tree"JSON树；recursive=True递归；page_token分页
- **实现逻辑**: format=list→扁平文件列表+统计；format=tree→JSON树结构+统计；分页支持
- **适用场景**: 查看目录内容、查看项目树结构、统计文件数量/大小

#### F6. search_files — 搜索文件名

- **设计原则**: P8自描述性(名字即功能)
- **Schema**: `SearchFilesInput` — pattern, search_dir, recursive, max_depth, excludePatterns, ignore_case, type, sortBy, page_token
- **参数**: pattern必填(支持glob通配符**/*.py)；search_dir默认~；type=file/directory过滤
- **实现逻辑**: 用glob递归匹配文件名；支持中文文件名；排除node_modules等
- **适用场景**: 查找所有Python文件、查找配置文件、定位特定文件

#### F7. grep_file_content — 搜索文件内容

- **设计原则**: P8自描述性
- **Schema**: `GrepFileContentInput` — pattern, search_dir, output_mode, glob, type, after_lines, before_lines, context_lines, ignore_case, head_limit, show_line_no
- **参数**: pattern必填(正则)；output_mode=content/files_with_matches/count；glob过滤文件类型
- **实现逻辑**: 用ripgrep(rg)高效搜索；支持正则+中文；上下文行显示
- **适用场景**: 搜索TODO标记、查找函数定义、搜索错误关键字

#### F8. rename_file — 统一重命名

- **设计原则**: P11统一入口(合并rename_file+batch_rename)，P17互斥(path与directory二选一)
- **Schema**: `RenameFileInput` — path, new_name, directory, pattern, replacement, preview, use_regex, recursive, conflict_strategy
- **参数**: path+new_name=单文件；directory+pattern+replacement=批量正则；preview=True预览
- **实现逻辑**: 单文件→直接重命名；批量→遍历目录+正则匹配+替换；冲突策略skip/overwrite/append_number
- **适用场景**: 重命名单个文件、批量添加前缀、正则替换文件名

#### F9. archive_tool — 统一压缩/解压

- **设计原则**: P11统一入口(合并compress_files+extract_archive)，action路由
- **Schema**: `ArchiveToolInput` — action, source_path, output_path, archive_path, output_dir, format, compression_level, password, overwrite, exclude_patterns, preserve_permissions
- **参数**: action="compress"/"extract"；compress需source_path+output_path；extract需archive_path
- **实现逻辑**: compress→zipfile/tarfile打包；extract→解压到目标目录；支持密码加密ZIP
- **适用场景**: 压缩项目备份、解压下载的压缩包、加密压缩

#### F10. file_operation — 统一文件操作

- **设计原则**: P11统一入口(合并move_file+copy_file+delete_file)，action路由，P16幂等性
- **Schema**: `FileOperationInput` — action, source, destination, recursive, overwrite, force, preserve_metadata
- **参数**: action="move"/"copy"/"delete"；move/copy需destination；delete不需destination；force=永久删除跳过回收站
- **实现逻辑**: move→shutil.move；copy→shutil.copy2(保留元数据)；delete→send2trash(回收站)或os.remove(force)
- **适用场景**: 移动文件到新目录、复制文件、删除文件(安全到回收站)

#### F11. data_file_format — 结构化数据文件读写

- **设计原则**: P11统一入口(合并json/yaml/toml/ini/xml读写)，P5后缀自动路由
- **Schema**: `DataFileFormatInput` — action, file_path, format, data, encoding, indent
- **参数**: action="read"/"write"；format可选(不填则按扩展名自动检测)；data=写入数据
- **实现逻辑**: read→按格式解析(json.loads/yaml.safe_load/toml.load等)；write→按格式序列化
- **适用场景**: 读取JSON配置、修改YAML文件、解析TOML/INI/XML

## 三、SHELL分类（5个工具）

### 3.1 分类概述

命令执行分类，覆盖Shell命令执行、命令查找、后台会话管理、Python/JavaScript代码执行。

| code_execution_prompts.py | `backend/app/services/prompts/shell/code_execution_prompts.py` | 61 | 代码执行Prompt(2个工具) |

### 3.3 工具清单

#### S1. execute_shell_command — 执行Shell命令

- **设计原则**: P0最高优先级(参数易混淆command/cmd/script)，P14优雅降级(超时自动终止)
- **Schema**: `ExecuteShellCommandInput` — command, shell_type, timeout, run_in_background, cwd, encoding, env_vars, run_as_admin
- **参数**: command必填；cwd指定工作目录(替代change_directory)；run_in_background=True→返回shell_id
- **实现逻辑**: subprocess.Popen执行；超时→process.kill()；后台→存入_shell_sessions字典
- **适用场景**: 执行系统命令、运行脚本、安装依赖、查看进程

#### S2. find_command — 查找命令路径

- **设计原则**: P1重复合并(合并check_command_available+locate_command)
- **Schema**: `FindCommandInput` — command, all_paths
- **参数**: command必填；all_paths=False返回第一个路径，True返回所有路径
- **实现逻辑**: shutil.which()查找；Windows额外查PATHEXT扩展
- **适用场景**: 检查git是否安装、查找python路径、确认工具可用性

#### S3. shell_session — 后台Shell会话管理

- **设计原则**: P11统一入口(合并get_shell_output+terminate_shell)，action路由
- **Schema**: `ShellSessionInput` — shell_id, action, filter, encoding, max_lines, tail, force, cleanup
- **参数**: shell_id必填；action="output"/"terminate"；filter=正则过滤输出行
- **实现逻辑**: output→读取后台进程stdout/stderr；terminate→process.kill()；filter→正则匹配过滤
- **适用场景**: 获取后台命令输出、终止长时间运行的进程、过滤错误日志

#### S4. execute_python — 执行Python代码

- **设计原则**: P4单一职责，P14优雅降级(安全检查可关闭)
- **Schema**: `ExecutePythonInput` — code, timeout, working_dir, safety_check
- **参数**: code必填(多行Python代码)；timeout默认30最大300秒；safety_check=True检测危险模式
- **实现逻辑**: 安全检查→检测os.system/subprocess等→subprocess执行python -c；超时→终止
- **适用场景**: 执行计算脚本、运行数据处理代码、快速验证Python表达式

#### S5. execute_javascript — 执行JavaScript代码

- **设计原则**: P4单一职责
- **Schema**: `ExecuteJavascriptInput` — code, timeout, working_dir
- **参数**: code必填；timeout默认30最大300秒
- **实现逻辑**: 检查Node.js可用→subprocess执行node -e；超时→终止
- **适用场景**: 执行JS脚本、运行npm命令、验证JS表达式

## 四、NETWORK分类（5个工具）

### 4.1 分类概述

网络通信分类，覆盖HTTP请求、文件下载、网页抓取、搜索引擎、网络诊断。

### 4.2 涉及代码文件

| 文件 | 路径 | 行数 | 职责 |
|------|------|------|------|
| network_tools.py | `backend/app/services/tools/network/network_tools.py` | 1293 | 5个工具函数实现 |
| network_schema.py | `backend/app/services/tools/network/network_schema.py` | 197 | 7个模型(含2弃用PingInput/PortCheckInput) |
| network_register.py | `backend/app/services/tools/network/network_register.py` | 219 | 注册点(5个) |
| network_prompts.py | `backend/app/services/prompts/network/network_prompts.py` | 114 | NETWORK Prompt(5个工具) |

### 4.3 工具清单

#### N1. http_request — 发送HTTP请求

- **设计原则**: P0最高优先级(URL/参数易错)，P14优雅降级(指数退避重试429/5xx)
- **Schema**: `HttpRequestInput` — url, method, headers, params, body, json_body, timeout, verify_ssl, proxy, retry, follow_redirects
- **参数**: url必填；method默认GET；body/json_body为POST请求体；timeout毫秒(默认30000)；retry重试次数(默认3)
- **实现逻辑**: httpx.AsyncClient发送请求；重试→指数退避；代理→三步走(直连→环境变量→报错)
- **适用场景**: 调用REST API、获取公网IP、提交表单、测试接口

#### N2. download_file — 下载文件

- **设计原则**: P7悲观默认值(超时300秒, 断点续传默认开启)
- **Schema**: `DownloadFileInput` — url, destination_path, headers, timeout, chunk_size, resume, proxy
- **参数**: url+destination_path必填；resume=True断点续传；chunk_size分块大小
- **实现逻辑**: httpx.stream流式下载；resume→检测已下载大小+Range头续传；分块写入
- **适用场景**: 下载安装包、下载远程文件、断点续传大文件

#### N3. fetch_webpage — 抓取网页内容

- **设计原则**: P5后缀路由(extract_format选择输出格式)
- **Schema**: `FetchWebpageInput` — url, prompt, extract_format, js_render, timeout, max_tokens, user_agent, proxy

### 8.3 工具清单

#### M1. tool_help — 查询工具详细用法

- **设计原则**: 元工具，帮助LLM在不确定参数时查询用法
- **Schema**: `ToolHelpInput` — tool_name
- **参数**: tool_name必填(工具名称)
- **实现逻辑**: tool_registry.get(tool_name)→返回{name,category,description,params,examples}
- **适用场景**: LLM不确定参数时查用法、用户问"read_csv怎么用"

#### M2. tool_search — 按关键词搜索工具

- **设计原则**: 元工具，帮助LLM发现可用工具
- **Schema**: `ToolSearchInput` — query
- **参数**: query必填(搜索关键词)
- **实现逻辑**: 遍历tool_registry→模糊匹配name/description→按相关度排序
- **适用场景**: 用户描述需求但不确定用哪个工具、发现有啥工具能读取Excel

#### M3. pipeline — 工具编排管道

- **设计原则**: P12组合复用，将多个工具按顺序编排执行
- **Schema**: `PipelineInput` — steps, stop_on_error
- **参数**: steps=JSON数组[{tool,params}]；stop_on_error=True某步失败则停止
- **实现逻辑**: 顺序遍历steps→逐个调用tool_registry.get_implementation→收集结果→返回{completed_steps,results}
- **适用场景**: 自动化流程(先读取→再分析→再生成图表)、减少ReAct循环步数

#### M4. get_time — 统一时间入口

- **设计原则**: P11统一入口(四合一：get_current_time+time_format+time_to_timestamp+timestamp_to_time)，action路由
- **Schema**: `GetTimeInput` — action, time_value, format, timezone, locale, unit, target_tz
- **参数**: action="now"/"format"/"to_timestamp"/"from_timestamp"；now时+timezone指定时区；format时+time_value+format指定格式
- **实现逻辑**: now→datetime.now(tz)；format→strftime；to_timestamp→(dt-epoch).total_seconds()；from_timestamp→datetime.fromtimestamp
- **适用场景**: 获取当前时间、格式化日期、时间戳转换

#### M5. time_add — 时间加减

- **Schema**: `TimeAddInput` — delta, start, unit
- **参数**: delta必填(正数=加/负数=减)；start=基准时间(默认当前)；unit="days"/"hours"/"minutes"/"seconds"/"months"
- **实现逻辑**: timedelta计算(days/hours/minutes/seconds)；months→relativedelta精确计算
- **适用场景**: 明天是几号、3天后、100天前、2小时后

#### M6. time_diff — 时间差值计算

- **设计原则**: P1重复合并(time_compare合入，增加is_after/is_before/is_equal)
- **Schema**: `TimeDiffInput` — start, end
- **参数**: start必填；end可选(默认当前时间)
- **实现逻辑**: end-start→timedelta→返回{humanized,seconds,minutes,hours,days,is_future,is_after,is_before}
- **适用场景**: 距离某天多久、相差多少天、哪个时间更早

#### M7. check_date — 日期综合检查

- **设计原则**: P11统一入口(四合一：time_is_weekend+time_is_holiday+time_is_workday+time_next_n_workday)
- **Schema**: `CheckDateInput` — date, check_type, n
- **参数**: date可选(默认今天)；check_type="weekend"/"holiday"/"workday"/"next_workday"；n=next_workday时的第N个
- **实现逻辑**: weekend→weekday>=5；holiday→24个公历+农历节日表匹配；workday=not weekend and not holiday；next_workday→逐天递推
- **适用场景**: 明天是周末吗、国庆放不放假、下个工作日是几号

#### M8. timezone_convert — 时区转换

- **设计原则**: P11统一入口(三方向：utc_to_local+local_to_utc+any)
- **Schema**: `TimezoneConvertInput` — time_value, direction, tz, source_tz, target_tz
- **参数**: time_value必填；direction="utc_to_local"/"local_to_utc"/"any"；any时+source_tz+target_tz必填
- **实现逻辑**: pytz.timezone→astimezone转换
- **适用场景**: UTC转北京时间、北京时间转纽约时间、跨国时间转换

#### M9. timer — 定时器管理

- **设计原则**: P11统一入口(三合一：timer_set+timer_clear+timer_list)
- **Schema**: `TimerInput` — action, delay, callback, callback_data, timer_id, limit
- **参数**: action="set"/"clear"/"list"；set时+delay+callback必填；clear时+timer_id必填
- **实现逻辑**: set→threading.Timer(delay,callback)；clear→timer.cancel()；list→返回所有活跃定时器
- **适用场景**: 3分钟后提醒、设置定时任务、取消提醒

## 九、一致性保障

### 9.1 四方一致性要求

每个LLM可见工具必须保证以下4部分一致：

| 检查项 | 说明 |
|--------|------|
| 工具名一致 | register注册名 = schema类名对应 = prompt中列出的工具名 |
| 参数名一致 | schema字段名 = examples中参数名 = prompt中参数名 |
| 必填性一致 | schema的required字段 = 函数签名中无默认值的参数 |
| description存在 | 每个register工具必须有≥10字符的description |

### 9.2 自动化检查脚本

- `check_tool_consistency.py` — 检查register/schema/examples三方一致性
- `check_tool_func_consistency.py` — 以tool函数为准检查4方一致性

- 59个LLM可见工具全部通过4方一致性检查 ✅
