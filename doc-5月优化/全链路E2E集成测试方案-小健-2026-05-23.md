## 一、测试定义

### 1.1 测试名称

**全链路端到端集成测试**（Full-Link End-to-End Integration Test）

### 1.2 为什么叫这个名

| 关键词 | 含义 | 在这个测试中的体现 |
|--------|------|-------------------|
| 全链路 | 覆盖系统所有组件 | 前端API→后端→Agent→LLM(Ollama)→工具执行→数据库→SSE响应 |
| 端到端 | 从用户发消息到收到完整结果 | POST /chat → SSE事件流 → final事件 |
| 集成 | 验证各组件对接是否正确 | API路由、SSE封装、Agent循环、LLM调用、工具执行、数据库写入 |

### 1.3 与现有测试的区别

| 测试类型 | 测试什么 | 用什么驱动 | 能发现什么问题 |
|---------|---------|-----------|--------------|
| 单元测试（现有~200个） | 单个工具函数 | Mock/Patch | 函数级bug |
| MessageBuilder测试（43个） | Prompt构建逻辑 | Mock | Prompt构建bug |
| 现有Ollama E2E（7个） | LLM连通性+基础对话 | 真实Ollama | LLM通不通 |
| **全链路E2E（本方案）** | **Agent行为+系统集成** | **真实Ollama+真实后端** | **死循环、工具选错、参数传错、会话污染、日志异常、数据库异常** |

### 1.4 测试目标

**唯一目标**：用真实LLM模拟真实用户操作，把系统中隐藏的问题炸出来。

**具体来说**：
1. Agent在真实LLM下能否正确完成用户请求
2. Agent在异常情况下（LLM返回垃圾/空内容/错误工具）能否容错
3. 系统各组件的串联是否完整（API→Agent→LLM→工具→DB）
4. 日志和数据库记录是否完整准确

## 三、完整测试流程

每个测试用例都严格按照以下步骤执行。

### 3.1 前置条件

| 条件 | 要求 | 如何检查 |
|------|------|---------|
| Ollama已启动 | 必须运行，监听11434 | `curl http://localhost:11434/api/tags` 返回200 |
| qwen2.5:1.5b已就绪 | 模型已拉取，可调用 | `ollama list` 能看到模型 |
| 后端已启动 | uvicorn运行中，监听8000 | `curl http://localhost:8000/docs` 返回200 |
| 测试数据就绪 | 测试目录、测试文件已创建 | `Test-Path` 检查路径存在 |
| 日志目录就绪 | `backend/logs/prompt-logs/`可写 | `Test-Path` 检查目录 |
| 数据库路径就绪 | `~/.omniagent/chat_history.db`可写 | 确认无权限问题 |

### 3.2 测试执行（核心流程）

```
步骤 1: 记录测试起始状态
─────────────────────────────────────────────────────────────────
  · 记录当前时间 T0
  · 记录数据库当前会话数（base count）
  · 记录日志文件当前大小

步骤 2: 发送用户请求
─────────────────────────────────────────────────────────────────
  · 用 httpx 发送 POST /chat
  · 请求体：{"messages": [{"role": "user", "content": "用户输入"}]}
  · 设置 stream=True 以接收 SSE
  · 记录请求时间 T1

步骤 3: 接收并解析 SSE 事件流
─────────────────────────────────────────────────────────────────
  · 逐行读取 SSE 响应
  · 按 event: xxx 和 data: xxx 解析每个事件
  · 记录每个事件的：
    - 事件类型（start/thought/action/observation/chunk/final/error/interrupted）
    - 时间戳
    - step编号
    - data内容（工具名、参数、结果等）
  · 持续接收直到 final 事件或超时
  · 记录接收完成时间 T2

步骤 4: 验证事件流正确性
─────────────────────────────────────────────────────────────────
  · [必须] 最终收到了 final 事件 → 正常结束
  · [必须] 没有收到 error 事件 → 没有异常中断
  · [必须] 事件总数在合理范围内（如 < 50个） → 没有死循环
  · [警告] 如果事件总数 > 20，可能存在空转/无意义循环
  · [检查] final 事件中是否包含结果内容
  · [检查] 事件顺序是否正确（start→thought→action→observation→...→final）

步骤 5: 验证数据库记录
─────────────────────────────────────────────────────────────────
  · 查询 chat_history.db 的 sessions 表
    - 对应当前测试的 session 记录是否存在
    - session 的开始时间、结束时间、状态是否正常
  · 查询 messages 表
    - 对应的消息记录是否存在
    - user message 和 assistant response 是否都保存了
  · 查询 execution_steps 表
    - 步骤数量与 SSE 事件流中的 action 数量是否一致
    - 每个 step 的 type、tool、status 是否正常

步骤 6: 验证日志文件
─────────────────────────────────────────────────────────────────
  · 扫描后端日志文件
    - 是否有 ERROR 级别日志（有则失败）
    - 是否有异常 traceback（有则失败）
    - 记录 WARNING 级别数量（过多则标记）
  · 扫描 prompt-logs/
    - 检查本次请求的 LLM 调用日志是否已保存
    - 检查 LLM 调用次数（过多则可能死循环）

步骤 7: 记录测试完成状态
─────────────────────────────────────────────────────────────────
  · 记录测试总耗时 T2 - T1
  · 记录 LLM 调用次数
  · 记录 SSE 事件总数
  · 记录最终结论（通过/失败/警告）
```

### 3.3 验证标准总表

| 验证点 | 通过标准 | 失败标准 | 验证来源 |
|--------|---------|---------|---------|
| 最终事件 | 收到 final | 未收到 final（超时或中断） | SSE事件流 |
| 错误事件 | 无 error 事件 | 任意 error 事件 | SSE事件流 |
| step数量 | < 50 | >= 50（可能死循环） | SSE事件流 |
| LLM调用次数 | < 10 | >= 10（可能死循环） | prompt-logs / 事件计数 |
| 数据库记录 | session + messages + steps 完整 | 缺少任意记录 | SQLite DB |
| 日志级别 | 无 ERROR | 有 ERROR | 后端日志文件 |
| 异常traceback | 无 | 有 | 后端日志文件 |
| 响应时间 | < 300s（CPU推理） | >= 300s | 计时 |
| 执行工具 | 与预期一致 | 调用了不相关的工具 | SSE action事件 |

### 3.4 后置清理

| 操作 | 说明 |
|------|------|
| 清理测试数据库记录 | 删除本次测试产生的session记录 |
| 清理测试文件 | 删除测试过程中创建的文件 |
| 恢复被修改的配置 | 还原 config.yaml 等 |
| 记录测试报告 | 写入测试结果到报告文件 |

**P5覆盖率：附加覆盖约20个工具（与P1-P4有重叠），全部分类已覆盖 ✅**

### 4.7 P6：异常与容错场景（系统健壮性验证，10个任务）

| 编号 | 场景 | 方法 | 预期行为 | 覆盖的风险点 |
|------|------|------|---------|-------------|
| E2E-ERR-01 | 路径不存在 | 请求读`C:\no_such_file_xxx.txt` | Agent报错后finish，不崩溃 | 文件不存在容错 |
| E2E-ERR-02 | 目录不存在 | 请求列出`Z:\nonexistent\` | Agent报错后finish | 目录不存在容错 |
| E2E-ERR-03 | 无效域名 | 请求访问`http://this-domain-does-not-exist-12345.com` | Agent报错后finish | 网络不可达容错 |
| E2E-ERR-04 | 无效SQL | 请求执行`SELECT * FROM non_existent_table` | Agent报错后finish | SQL错误容错 |
| E2E-ERR-05 | 连续5次不同请求 | 按顺序发5个不同请求 | 每次独立完成，不被前一次污染 | session状态隔离 |
| E2E-ERR-06 | 并发3个请求 | 同时发3个不同请求 | 各自正常结束，互不干扰 | 并发安全 |
| E2E-ERR-07 | 长输入 | 输入2000字长文本 | Agent能处理，不OOM，正常finish | 输入长度容错 |
| E2E-ERR-08 | 快速连续相同请求 | 连续2次发同样请求 | 第2次结果合理（可缓存可不同） | 重复请求处理 |
| E2E-ERR-09 | 中途关闭Ollama | 请求执行中停掉Ollama服务 | Agent检测到连接错误，优雅finish | LLM服务中断 |
| E2E-ERR-10 | 恢复后继续 | Ollama恢复后重新发请求 | 能正常完成 | 服务恢复能力 |

async def test_e2e_p0_01_chat_hello():
    """P0-01: 闲聊输入"hello"，Agent应直接finish不调工具"""

    base_session_count = get_session_count()

    assert result["final_event"] is not None, "必须有final事件"

    db_check = check_database(result["final_event"]["session_id"])
    assert db_check["session_exists"], "session应已保存"
    
    # 7. 验证日志
    log_check = check_logs("e2e-p0-01", test_start)
    assert len(log_check["errors"]) == 0, f"日志不应有ERROR: {log_check['errors']}"
