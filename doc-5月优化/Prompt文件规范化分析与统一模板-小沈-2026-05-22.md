## 1. 背景

工具 register 文件的 description 已完成三段式规范化（使用场景→使用示例+常用名转换说明→返回数据说明），但 prompt 文件（注入 LLM system prompt 的内容）格式各异，需统一规范，确保 LLM 收到一致、清晰、有效的工具指引。

## 3. 现状不一致问题（7类）

### 3.1 角色定义语言不统一

```
# file_prompts: 全英文
"You are a professional file management assistant. You help users organize..."

# meta_prompts: 英文+中文RULES
"You are a professional Meta-tool assistant..."
"【CORE BEHAVIOR RULES】:\n1. **直接调用工具**..."

# time_prompts: 英文角色+跨分类工具声明
"You have access to the following tool categories:\n- TIME tools: ...\n- FILE tools: ..."
```

**问题**：角色定义风格不一，有的纯英文，有的穿插中文，有的声明跨分类工具。

### 3.2 工具列表标题格式不统一

```
# shell:    【Available SHELL Tools — 共4个】（2026-05-22 小沈 5→4 ...）
# network:  【Available NETWORK Tools — 共5个】（2026-05-18 小健 合并后）
# system:   【Available SYSTEM Tools】:
# document: 【Available DOCUMENT Tools】（共9个）:
# meta:     【Available META Tools】:
# time:     【Available TIME Tools】(精简后7个统一入口工具):
# desktop:  【Available DESKTOP Tools — 共9个】（2026-05-22 小沈 10→9 Ch19精简后）:
```

**问题**：6种不同写法，括号位置、是否标注数量、是否附带变更说明均不一致。

### 3.3 工具描述粒度差异极大

**极详细**（file_prompts.py）：
```
F1. read_file(file_paths, head=None, tail=None, offset=None, limit=None, encoding=None)
   Read text file(s) - unified entry. Supports single file with head/tail/offset/limit, or batch read.
   - file_paths: List of file paths. 1 path=single file(pagination supported), multiple=batch read
   - head/tail/offset/limit: Pagination for single file mode only
   - encoding: File encoding, default utf-8
   Example (single): {"file_paths": ["C:/config.json"], "offset": 1, "limit": 100}
   Example (batch): {"file_paths": ["C:/file1.txt", "C:/file2.txt"]}
```

**极简略**（document_prompts.py）：
```
1. read_document(file_path) - 统一读取文档（按后缀路由：.pdf/.doc/.docx/.xlsx/.xls/.pptx/.csv/.tsv）
```

**问题**：file写6行，document写1行。LLM对document工具的理解严重不足，容易传错参数。

### 3.4 合并工具标注格式不统一

```
# shell:    find_command - Find command path (replaces check_command_available + locate_command)
# network:  network_diagnose - Network connectivity diagnostics (replaces ping + port_check)
# desktop:  window_info - Unified window info query (replaces list_windows + get_window_info)
# time:     get_time - 统一时间入口（四合一）
           action="now": 获取当前时间（替代原get_current_time）
```

**问题**：4种标注方式 — `replaces` / `Unified` / `四合一` / `替代原`。

> **结论**：合并标注在 prompt 文件中**不需要保留**。原因：register 的 description 已通过 OpenAI Schema 暴露了 `【常用名转换说明】`，prompt 里再写一遍是冗余。LLM 不关心工具历史，只关心当前怎么调。

### 3.5 Tool Call Examples 格式不统一

```
# shell/network/desktop:  Example 1: List directory (Windows)      ← 冒号
# system:                 Example 1 - 获取系统信息:                 ← 短横线
# document/database:      Example 1 - 读取文档:                    ← 短横线
# time:                   Example 1 - 查询当前时间:                 ← 短横线
```

**问题**：冒号 vs 短横线混用。

### 3.6 get_safety_reminder / get_rollback_instructions 覆盖不一致

| 文件 | safety_reminder | rollback_instructions |
|------|:-:|:-:|
| file_prompts | ✗ (默认空) | ✗ (默认) |
| shell_prompts | ✅ | ✅ |
| network_prompts | ✗ | ✗ |
| system_prompts | ✗ | ✗ |
| desktop_prompts | ✗ | ✗ |
| document_prompts | ✅ | ✗ (默认) |
| database_prompts | ✅ | ✅ |
| meta_prompts | ✅ | ✗ (默认) |
| time_prompts | ✅ | ✗ (默认) |
| code_execution_prompts | ✗ | ✗ (默认) |

**问题**：Shell/Database有安全风险但没覆盖的（如document写操作覆盖文件无提醒），部分无安全风险的却覆盖了（如time的timer_clear提醒意义不大）。

> **结论**：safety/rollback **对LLM有实际用处**，应保留但精简。理由：`"Destructive commands require extra caution"` 确实能让LLM在调rm前多一步确认；`"Do NOT concatenate user input into SQL"` 确实能防止SQL拼接。但只写能改变LLM行为的关键提醒，不写冗长免责声明。

### 3.7 模块docstring元数据不统一

```
# shell:      P0优先级：参数易混淆（command/cmd/script），安全风险高
# network:    P0优先级：URL/参数易错，超时重试策略需引导
# system:     P2优先级
# document:   P2优先级
# desktop:    P1优先级：截图/窗口操作参数特殊
# meta:       无优先级标注
# time:       无优先级标注
```

> **结论**：模块docstring的优先级/变更历史/Author对LLM**完全无用**（不会注入prompt），属于代码维护规范，不在prompt内容模板范围内。

## 4. 统一模板规则

### 4.1 核心原则

1. **只规范 `*_prompts.py` 子类的 `get_system_prompt()` 实现**，不修改 `BasePromptTemplate.py`
2. **代码维护规范**（模块docstring、优先级标注）不在本模板范围
3. **合并标注不在prompt中保留**（register的description已覆盖常用名转换说明）
4. **工具描述中的参数说明不在prompt中保留**（register的OpenAI Schema已覆盖参数类型、必填/可选、默认值）
5. **When to use 和 Returns 只对参数易混淆或副作用明显的工具添加**，不强制每个工具
6. **safety/rollback精简有效** — 只写能改变LLM行为的关键提醒
7. **不设P0/P1工具分组** — LLM靠When to use和Examples匹配意图，不靠分组标题；平铺1-N序号即可
8. **工具列表按使用频率降序排列**（高频工具在前）

### 4.2 工具描述要素

| 要素 | 格式 | 条件 |
|------|------|------|
| When to use | `- When to use: {场景关键词}` | 仅参数易混淆或副作用明显的工具添加 |
| Returns | `- Returns: {返回字段列表}` | 仅返回结构非显而易见的工具添加 |
| Examples | `- Examples:\n  * {tool_name}({param}: {value})` | 每个工具1-2个最常用调用（JSON键值格式） |

**参数说明不在此处写** — register的OpenAI Schema已向LLM暴露参数类型、必填/可选、默认值，prompt中重复反而增加维护成本。

### 4.3 Tool Call Examples 规则

- 统一格式：`Example {N}: {中文场景描述}`（冒号分隔）
- 全部使用JSON格式，不使用函数调用式
- JSON中 thought/reasoning 用中文，tool_name/tool_params 用英文
- 必须包含一个 `finish` 示例（任务完成退出循环）
- 示例数：最少2个（1个工具调用 + 1个finish），最多5个

### 4.4 safety/rollback 规则

| 分类 | safety_reminder | rollback_instructions |
|------|:-:|:-:|
| Shell | ✅ 命令破坏性提醒 | ✅ 失败后检查命令可用性 |
| Database | ✅ SQL注入+DROP确认 | ✅ 失败后建议手动ROLLBACK |
| Desktop | ✅ 窗口操作不可逆提醒 | ✗ |
| Document | ✅ 写入覆盖提醒 | ✗ |
| Network | ✅ HTTP请求注入风险提醒 + URL参数检查 | ✗ |
| System | ✅ 注册表写删提醒 | ✗ |
| File | ✅ 写入覆盖提醒 + text禁传思考 | ✗ |
| CodeExec | ✅ 代码执行安全提醒（不执行破坏性命令/超时退出） | ✗ |
| Meta/Time | ✗ | ✗ |

**精简原则**：
- safety只写1-3条最关键的（如"Destructive commands require extra caution"）
- rollback只写出错后的第一步应对策略

### 4.5 include_commands 参数规则

| 分类 | include_commands | 原因 |
|------|:-:|------|
| Shell | True | Shell Agent需要知道命令格式 |
| File | False | File Agent不需要命令格式 |
| Network | False | 避免LLM幻觉调execute_shell_command |
| System | False | 不需要命令格式 |
| Desktop | False | 不需要命令格式 |
| Document/Database | False | 不需要命令格式 |
| Meta/Time/CodeExec | False | 不需要命令格式 |

### 4.6 get_system_prompt() 输出结构

```
① system_info = get_system_info(include_commands={True/False})

② 角色定义（全英文）
   You are a professional {category} assistant. You help users {English summary of capabilities}.

③ 工具列表
   【Available {CATEGORY} Tools — 共{N}个】:
   
   1. {tool_name} - {一句话功能}
      - When to use: {场景关键词}
      - Returns: {返回字段}
      - Examples:
        * {tool_name}({param}="{value}")
   
   2. ...

④ 【Tool Call Examples】:
   Example 1: {中文场景描述}
   {
       "thought": "{中文思考}",
       "reasoning": "{中文推理}",
       "tool_name": "{tool_name}",
       "tool_params": {参数JSON}
   }
```

## 7. 全部注入LLM的文字段对比分析

除 `get_system_prompt()` 外，还有3个方法会向LLM注入文字，当前格式严重不一致。

### 7.1 get_task_prompt() — 10个文件全对比

| 文件 | 注入内容 | 语言 | 步骤数 | 有Remember? | 有Current time? |
|------|---------|------|--------|:-:|:-:|
| **基类(默认)** | `Task: {task}\n\nPlease think step by step and use the available tools to complete this task.` | 英 | 0 | ✗ | ✗ |
| file | `Task: {task}\nCurrent time: ...\nPlease help me...\n1. analyze\n2. use tools\n3. summary\n\nRemember:\n- multiple tools\n- well-reasoned\n- explain failures\n- tracked for safety` | 英 | 3 | ✅4条 | ✅ |
| shell | `1. check command available\n2. execute with timeout\n3. clear summary` | 英 | 3 | ✗ | ✗ |
| network | `1. identify operation\n2. use tool with correct params\n3. handle errors gracefully\n\nRemember:\n- URL scheme\n- json_body\n- timeout\n- fallback\n- avoid repeat` | 英 | 3 | ✅5条 | ✗ |
| desktop | `1. identify target window\n2. use tool\n3. confirm result` | 英 | 3 | ✗ | ✗ |
| document | `1. analyze operation\n2. use tool\n3. summary` | 英 | 3 | ✗ | ✗ |
| database | `1. check schema\n2. query_sql/execute_sql\n3. summary` | 英 | 3 | ✗ | ✗ |
| system | `1. analyze info needed\n2. use tool\n3. summary in Chinese` | 英 | 3 | ✗ | ✗ |
| time | `Current time: ...\n1. analyze operation\n2. use tool\n3. friendly Chinese response` | 英 | 3 | ✗ | ✅ |
| meta | `1. understand user wants\n2. use tool\n3. friendly Chinese response` | 英 | 3 | ✗ | ✗ |
| code_exec | `1. analyze code/language\n2. execute\n3. provide result` | 英 | 3 | ✗ | ✗ |

**不一致问题**：
1. **有无Current time**：file/time注入了当前时间，其他7个没有 → LLM不知道当前时间可能误判"今天"/"现在"
2. **有无Remember**：file有4条、network有5条，其他0条 → 同一对话中不同Agent的task提示详略差异大
3. **步骤描述泛化程度**：shell/network/database步骤具体到工具名，document/desktop/system极泛化
4. **中文回复引导**：system/time/meta要求"summary in Chinese"，其他不提 → 部分Agent可能用英文回复

### 7.2 get_observation_prompt() — 仅file覆盖

| 文件 | 是否覆盖 | 注入内容 |
|------|:-:|---------|
| **基类(默认)** | 默认 | `Observation: {observation}\n\n` |
| file | ✅ | 解析JSON → 成功时列出operation_type/file_path/message → 失败时列出error → `What's your next step?` |
| 其他9个 | ✗ | 用基类默认（原始字符串透传） |

**问题**：file的observation格式化其他Agent没有。但这个差异不大，因为现在observation已改为JSON对象（第13章方案），StepContent做统一渲染。**不改**。

### 7.3 get_safety_reminder() — 7个覆盖 vs 3个不覆盖

| 文件 | 覆盖? | 注入内容 |
|------|:-:|---------|
| file | ✅ | 6条（全操作备份/移动追踪/历史记录/可回滚/写入覆盖/搜索只读） |
| shell | ✅ | 4条（破坏性命令/检查可用性/timeout/验证工作目录） |
| database | ✅ | 4条（SQL注入/确认DROP/读写分离/WHERE子句） |
| document | ✅ | 1条（写入覆盖） |
| time | ✅ | 1条（timer_clear只影响当前session） |
| meta | ✅ | 1条（pipeline JSON验证） |
| **network** | ✗ | 无 — 但HTTP请求有注入风险、download有路径风险 |
| **desktop** | ✅ | 1条（只操作可见窗口） |
| **system** | ✗ | 无 — 但注册表写删有破坏性风险 |
| **code_exec** | ✗ | 无 — 但代码执行有安全检查相关提醒 |

**问题**：
- file的safety有6条，太冗长（"All operations tracked""You can rollback"这些基类rollback已覆盖）
- network/system/code_exec应该加safety但没加
- time的timer提醒意义不大

### 7.4 get_rollback_instructions() — 仅3个覆盖

| 文件 | 覆盖? | 注入内容 |
|------|:-:|---------|
| **基类(默认)** | 默认 | `If operation fails: 1. analyze 2. try alternative 3. report error` |
| file | ✅ | 详细的回滚说明（单步回滚/session回滚/恢复删除/移动还原） |
| shell | ✅ | `1. check_command 2. check cwd 3. verify before destructive` |
| database | ✅ | `1. SELECT preview 2. WHERE clause 3. suggest manual ROLLBACK` |
| 其他7个 | ✗ | 用基类默认 |

**问题**：file的rollback太详细（7行），shell/database的3行刚好。基类默认3行也合理。**file应精简**。

### 7.5 file_prompts.py 独有的额外prompt段

file_prompts.py 在 get_system_prompt() 末尾还有2个额外段，其他文件都没有：

```
【⚠️ P17互斥参数规则 - 极其重要】:
- read_file: file_paths传1个路径=单文件, 传多个=批量
- edit_file: old_string 和 edits 不能同时使用
...

【⚠️ write_text_file text规则 - 极其重要】:
- text参数必须传入实际的文件内容
- ❌ 绝对禁止将你的思考/计划/状态确认当作text传入
...
```

**分析**：这两段对LLM有实际用处（防止互斥参数同传、防止text传思考内容），但只有file有。其他分类如果存在同类约束（互斥参数、禁止传思考内容），按相同格式在工具列表末尾添加专有段。

## 9. 不在本模板范围的事项

| 事项 | 原因 | 归属 |
|------|------|------|
| 模块docstring格式 | 不注入LLM，属代码规范 | 代码规范文档 |
| 优先级标注(P0-P3) | 不注入LLM，属维护规范 | 代码规范文档 |
| 变更历史/Author | 不注入LLM | 代码规范文档 |
| 合并工具标注 | register的description已覆盖 | register规范 |
| 参数说明(类型/必填/默认值) | register的OpenAI Schema已暴露 | register规范 |
| get_parameter_reminder() | 由tool_registry自动生成+forbidden列表 | 不变 |
| get_observation_prompt() | StepContent统一渲染，不影响LLM理解 | 不变 |
| BasePromptTemplate公共段 | OUTPUT_FORMAT/TOOL_CALL_RULES/避免重复规则已稳定 | 不变 |
