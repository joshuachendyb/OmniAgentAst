## 一、事故概述

### 1.1 事故现象

用户要求：查看E盘下载目录 + 写10章中篇小说（每章10000-20000字）

实际结果：
- 第一章：写入了约400字的小说正文（1197字节），内容正确但不满足10000字要求
- 第二至第二十五章：每章仅写入131字节的一句话，如"已成功创建并写入第X章内容。现在需要继续创建第X+1章..."
- 最终被用户中断（第83步），错误类型：empty_response

### 1.2 核心问题

LLM（LongCat-Flash-Lite）在tool call中，**从第二章开始，把thought/计划文字直接当作write_file的content参数写入文件**，而非生成实际的小说正文内容。且系统没有任何机制检测到这一异常，导致错误持续循环25+次。

### 2.4 _trim_history裁剪机制

代码位置：`base_react.py:639-694`

```python
MAX_HISTORY_TURNS = 5

def _trim_history(self) -> None:
    if len(self.conversation_history) <= 2:
        return
    if len(self.conversation_history) <= 15:
        return  # 15条以内不裁剪
    
    system_msg = self.conversation_history[0]       # 保留第0条system
    recent = self.conversation_history[-5:]          # 保留最后5条
    
    important = []
    for msg in self.conversation_history[1:-5]:      # 遍历中间消息
        content = msg.get("content", "")
        role = msg.get("role", "")
        if role == "user" or role == "assistant" or content.startswith("Observation:"):
            important.append(msg)
    
    if len(important) > 10:
        important = important[-10:]  # 重要消息最多10条
    
    self.conversation_history = [system_msg] + important + recent
```

**关键问题**：原始user消息（`conversation_history[1]`）在important的遍历范围内，但important最多10条，当中间消息超过10条时**只保留最后10条**，原始user消息被裁剪掉。

## 三、系统层面缺陷分析（逐项深入）

### 缺陷1（致命级）：Observation回传缺少写入内容验证和质量信息

**代码位置**：`base_react.py:531-569`（给LLM通道）、`reasoning_steps.py:874-921`（给前端通道）

**现状**：

- 给LLM的observation_text：只有"success + bytes_written + data字典"，不包含content摘要，不包含质量校验结果
- 给前端的observation字段：只有summary（如"成功写入文件...共131字节"），不包含任何警告

**问题**：
1. LLM看到"success"就认为写入正确，**没有机会发现自己写错了内容**
2. 前端用户看到"成功写入131字节"，**也不知道内容有问题**
3. 两条通道都没有传递"内容质量"信息——这是最根本的缺失

**对比**：read_file工具的Observation会回传文件内容（data中包含content字段），但write_file的data中只有success/operation_id/file_path/bytes_written，没有写入的content。

### 缺陷2（严重级）：write_file的思维泄漏检测只针对.py文件

**代码位置**：`file_tools.py:675-686`

```python
# 【修复 2026-05-01 序号7】.py文件写入前：检测思维泄漏
if path_preview.suffix.lower() == '.py' and content:  # ← 只对.py文件生效！
    python_keywords = ['def ', 'class ', 'import ', ...]
    has_python = any(kw in content for kw in python_keywords)
    chinese_chars = sum(1 for c in content if '\u4e00' <= c <= '\u9fff')
    if not has_python and chinese_chars > len(content) * 0.3 and len(content) < 500:
        return _to_unified_format({"success": False, "error": "内容保护：...可能是思维文本而非代码"})
```

**问题**：
1. 条件 `path_preview.suffix.lower() == '.py'`——**只对.py文件生效**
2. 本案例写入的是`.txt`文件，思维泄漏检测完全不触发
3. 写入的content如"已成功创建并写入第一章内容。现在需要继续创建第二章..."——典型的自我指涉文本（LLM在描述自己的操作，而非真正的文件内容），但因为不是.py文件而逃过了检测

### 缺陷2.5（严重级）：无法判断写入内容是否完整和正确

**核心问题**：不是"字数"问题，也不是"截断"问题，而是**无法判断LLM写入的内容是否完整和正确**。

**现象分析**：

第一种情况（第一章是小说内容）：
- LLM想写10000字小说
- 实际写入了1197字节（可能是完整的1197字小说，也可能只是LLM想写的全部内容的一部分）
- **问题**：我们不知道1197字节是不是LLM想要输出的全部内容
- 如果LLM想写10000字但只输出了1197字，说明LLM的输出不完整（被截断或提前结束）

第二种情况（第一章也是思维文本）：
- LLM把思维文本当成小说内容写入
- 1197字节的"小说"实际上是LLM的思考过程
- **问题**：内容类型错误，不是真正的文件内容

**两种情况的共同问题**：系统没有机制判断LLM输出的content是否：
1. **完整**——是LLM想要输出的全部，还是只是其中一部分？
2. **正确**——是真正的文件内容，还是思维泄漏？

**结构完整性的真正含义**：

| 维度 | 说明 | 当前状态 |
|------|------|---------|
| **JSON结构** | 返回的JSON格式是否正确 | 基本保证（有parser处理） |
| **Content完整性** | content是否是LLM想要输出的全部 | **无法判断** |
| **Content正确性** | content是否是对的文件内容 | **无法判断** |

**当前缺失**：
1. 无法判断LLM的content是否完整输出（可能只输出一部分）
2. 无法判断LLM的content是否是正确的文件内容（可能是思维泄漏）

---

### 处理方法

#### 方法A（完整性验证）：检测content是否可能不完整

**问题**：LLM可能想输出更多内容，但只输出了部分就结束了。

**检测思路**：分析content长度是否明显短于预期。

> ⚠️ **注意**：本版本已移除"句子中断检测"（如判断"林凡是一"是否不完整）。该检测在中文语境下极易误报——中文句末词如"的"、"了"、"中"等常见于完整句子末尾，无法可靠判断句子是否中断。保留长度检测作为唯一的完整性指标。

**修改位置**：`react_output_parser.py` - 解析后验证content完整性

```python
def _check_content_completeness(content: str, tool_name: str) -> str:
    """
    检查content长度是否明显短于预期
    返回警告信息，如果content看起来不完整
    
    【v1.8】移除了句子中断检测（中文语境误报率高），仅保留长度检测。
    """
    if not content or not isinstance(content, str):
        return ""
    
    warnings = []
    
    # 仅检查：内容长度是否明显短于预期
    if tool_name == "write_file":
        # 小说类：预期>10000字
        # 如果只有几百字，可能是LLM提前结束了
        if len(content) < 1000:
            warnings.append(f"⚠️ content长度({len(content)}字符)明显短于预期，可能输出不完整")
    
    return "；".join(warnings)
```

#### 方法C（Prompt约束）：规范LLM输出格式

#### 综合建议

采用方法A（设置max_tokens）作为主要防护，确保LLM完整输出；配合方法B（结构验证）作为兜底，检测并拒绝不完整的JSON；方法C（Prompt约束）作为辅助提醒。

**现状**：Observation阶段只做两件事：
1. 根据exec_status构建observation_text
2. 把observation_text存入conversation_history

**没有任何对write_file写入内容的独立质量检测**。即使工具层（file_tools.py）的思维泄漏检测没拦住，agent层也不会做二次检查。

**这是最关键的缺失**：agent层应该有独立的、与工具层无关的内容质量检测逻辑，检测结果通过Observation同时传给LLM和前端。

### 缺陷4（严重级）：assistant消息存入history包含完整tool_params——自我强化幻觉

**代码位置**：`base_react.py:525`

```python

```

response是`_make_result()`返回的JSON字符串，包含完整的tool_params（含write_file的content）。

**幻觉闭环机制**：
```

→ thought："已成功创建并写入第一章内容，继续创建第二章"
→ 弱模型把thought复制到content参数（而不是真正写小说）
→ write_file执行成功，131字节
→ Observation：success, 131字节 ← 没有告诉LLM内容有问题

→ 下轮LLM看到history：assistant说"写第二章成功了" ← 自我确认
→ LLM想：我要写第三章
→ 继续循环...永远写不出真正内容
```

## 四、双层防线架构设计

### 4.1 设计原则

**核心原则**：内容质量检测不能只靠工具层拦截，agent层必须有独立检测，检测结果通过两条Observation通道同时传递：

1. **给LLM的通道**：通过`observation_text`附加警告信息 → 存入conversation_history → LLM下一轮能看到
2. **给前端的通道**：通过ObservationStep的`observation`字段附加警告信息 → yield → 前端用户能看到

### 4.2 双层防线架构

```
┌──────────────────────────────────────────────────────────┐
│ 第1层防线：工具层（file_tools.py write_file）              │
│                                                          │
│ 检测时机：写入文件前                                      │
│ 检测范围：所有文件类型（不再限.py）                        │
│ 检测条件：自我指涉检测>60%（主要，权重70%）              │
```

### 改进1（P0）：write_file思维泄漏检测扩展到所有文件类型

**修改文件**：`backend/app/services/tools/file/file_tools.py`  
**修改位置**：write_file方法中，第675-686行之后

**检测条件设计（关键——v1.3修订）**：

思维泄漏的判断不能简单用"是否包含计划性关键词"来做AND条件，否则会误判正常的小说叙事：
- content = "接下来他走向了那座古堡..." → 含"接下来"，是小说叙事 → **不应误判**
- content = "已成功创建并写入第一章内容。现在需要继续创建第二章" → 全是计划性语言 → **应拦截**

**核心设计：自我指涉文本检测率（主要因素，权重70%）**

不是问"有没有自我指涉文本"，而是问"自我指涉/完成性描述在content中占了多大比例"。

具体方法：将content按句子拆分，计算"自我指涉句子数/总句子数"的检测率。当大部分句子都是LLM在描述自己的操作/状态时，才判定为思维泄漏。

| 判定因子 | 权重 | 说明 | 阈值 |
|---------|------|------|------|
| **自我指涉检测率** | **70%（主要）** | content中自我指涉/完成性句子占总句子的比例 | >60%时高度可疑 |
| **content长度** | **30%（辅助）** | 越短越可疑，但不是主要因素 | <50字符时加分 |

自我指涉文本的特征（与文件类型无关）：
- 自我行为描述："已成功"、"已完成"、"已创建"、"写入成功"
- 完成性确认："已成功创建并写入"、"内容已写入"
- 计划性描述（LLM的思考过程）："需要继续"、"现在需要"、"接下来将"、"按照要求"、"继续创建"

真正的文件内容（小说、文档、代码）**不会**大量包含这类语言——即使偶有"接下来"这样的叙事词，覆盖率也远低于60%。

因此检测条件按文件类型区分：
- **代码类文件**(.py/.js/.ts等)：中文占比高(>30%) + 自我指涉检测率>60%
- **文档类文件**(.txt/.md/.doc等)：自我指涉检测率>60%（不需要中文占比，不需要极短条件）
- **未知文件**：中文占比高(>70%) + 自我指涉检测率>60%（保守）

**修改方法**：在现有.py检测逻辑之后，新增全文件类型的思维泄漏检测：

```python
                         '现在应该', '接下来需要', '需要先', '然后需要']
    self_ref_sentences = 0
    for sent in sentences:
        if any(kw in sent for kw in self_ref_keywords):
            self_ref_sentences += 1
    self_ref_rate = self_ref_sentences / total_sentences  # 自我指涉检测率
    
    # 辅助指标：content极短加分
    is_very_short = len(content) < 50
    
    # 综合判定：自我指涉检测率>60% 为主要判定条件
    # 极短(<50字符)时降低阈值到40%（因为极短文本一两个自我指涉词就占很高比例）
    self_ref_threshold = 0.4 if is_very_short else 0.6
    is_high_self_ref = self_ref_rate >= self_ref_threshold
    
    # 代码类文件
    code_extensions = {'.py', '.js', '.ts', '.java', '.go', '.c', '.cpp', '.rs', '.rb'}
    if path_preview.suffix.lower() in code_extensions:
        # 保留原有.py检测逻辑（中文占比>30% + 极短 + 无代码关键字）
        # ...
        
        # 代码类通用：中文占比高 + 自我指涉检测率高
        if chinese_ratio > 0.3 and is_high_self_ref:
            return _to_unified_format({
                "success": False,
                "error": f"内容保护：代码文件写入内容中{int(self_ref_rate*100)}%为自我指涉描述，疑似思维泄漏。请在content参数中传入实际代码内容。",
                "content": None
            }, "write_file")
    
    # 文档类文件：自我指涉检测率高即可（不需要中文占比，不需要极短！）
    # 原因：小说/文档可能全中文且长度不限，只要自我指涉率低就是正常的
    doc_extensions = {'.txt', '.md', '.doc', '.docx', '.csv', '.log', '.ini', '.cfg', '.yml', '.yaml'}
    if path_preview.suffix.lower() in doc_extensions:
        if is_high_self_ref:
            return _to_unified_format({
                "success": False,
                "error": f"内容保护：写入内容中{int(self_ref_rate*100)}%为自我指涉描述（如'已成功'、'需要继续'），疑似将思考过程当作文件内容写入。请在content参数中传入实际的文件内容。",
                "content": None
            }, "write_file")
    
    # 其他文件类型：保守策略，需要中文占比高 + 自我指涉检测率高
    if path_preview.suffix.lower() not in code_extensions and path_preview.suffix.lower() not in doc_extensions:
        if chinese_ratio > 0.7 and is_high_self_ref:
            return _to_unified_format({
                "success": False,
                "error": f"内容保护：写入内容中{int(self_ref_rate*100)}%为自我指涉描述，疑似思维泄漏。请在content参数中传入实际内容。",
                "content": None
            }, "write_file")
```

```python
    分层保留对话历史
    - 保留 system message
    - 保留原始用户消息（任务需求）← 【新增 2026-05-01 小沈】
    - 保留所有 observation 消息（工具执行结果）
    - 保留最近5条消息
    
    【修复 2026-05-01 小沈】
    - 问题：原始user消息在important裁剪时可能被丢弃，导致LLM丢失任务约束（如"10章"）
    - 修复：显式保留conversation_history[1]（原始user消息），不参与important裁剪
    """
    if len(self.conversation_history) <= 2:
        return

    if len(self.conversation_history) <= 15:
        return
    
    system_msg = self.conversation_history[0]
    
    # 【新增 2026-05-01 小沈】显式保留原始用户消息，不参与裁剪
    original_user_msg = self.conversation_history[1] if len(self.conversation_history) > 1 else None
    
    recent = self.conversation_history[-5:]
    
    original_len = len(self.conversation_history)
    
    # 从index=2开始遍历（跳过已显式保留的原始user消息）
    important = []
    start_idx = 2 if original_user_msg else 1
    for msg in self.conversation_history[start_idx:-5]:
        content = msg.get("content", "")
        role = msg.get("role", "")
        if role == "user" or role == "assistant" or content.startswith("Observation:"):
            important.append(msg)
    
    if len(important) > 10:
        important = important[-10:]
    
    # 重建：system + original_user + important + recent
    rebuilt = [system_msg]
    if original_user_msg:
        rebuilt.append(original_user_msg)
    rebuilt.extend(important)
    rebuilt.extend(recent)
    
    self.conversation_history = rebuilt
    
    logger.info(f"[History] Trimmed from {original_len} to {len(self.conversation_history)} messages (important={len(important)}, recent={len(recent)})")
```

### 改进4（❌ 不做）：assistant消息存入history时精简大字段

**结论：不实施。** 经审查评估，此改进弊大于利。

**不做理由（详细分析）**：

1. **破坏LLM续写能力**：写小说时，LLM需要在下一轮看到上一轮写入的完整内容才能连贯续写。截断到500字符后，LLM看不到10000字小说的后9500字，只能猜测故事走向→更容易写出"已成功创建第一章，现在继续创建第二章"这种空洞文本。

2. **与改进2（自我修正）自相矛盾**：改进2的核心是通过Observation附加content摘要和质量警告让LLM自我修正。但改进4在另一条通道（assistant消息）把同样的content截断——LLM看到"疑似思维泄漏"的警告，想翻看自己上一轮写了什么错误内容时，却只能看到截断后的片段，自我修正效果大打折扣。

3. **不解决幻觉闭环的真正原因**：幻觉闭环的形成不是因为"content太长"，而是因为"系统只反馈'success+bytes_written'，没有反馈内容质量"。截断content解决不了这个核心问题。

4. **数据丢失不可逆**：截断后的content存入conversation_history后，原始内容永久丢失。后续如果改进2需要全量content做分析，或者rollback需要还原文件内容，已经拿不到了。

5. **token收益有限，风险不对称**：即使10000字content（约30K token），在30+轮对话中，省掉的token约900K——但代价是LLM可能完全写偏。省token有其他更安全的方法（如给LLM调用加max_tokens限制）。

**如果未来确实需要省history token**，正确做法是：**传给LLM时动态截断**（不修改conversation_history中的原始数据），而不是存储时就截断。但当前阶段，改进1+2+3已经能解决事故根因，不需要冒这个风险。

### 改进7（P1）：react_output_parser中增加reasoning验证

**修改文件**：`backend/app/services/agent/react_output_parser.py`  
**修改位置**：解析tool call结果后（ParseResult返回前）

**现状问题**：在文本返回模式下，Schema的required字段不能强制LLM提供reasoning。即使Schema写required=["thought", "reasoning", "tool_name"]，LLM仍可能不提供reasoning字段，后端Parser只是"尽力解析"，不会报错。

**修改方法**：在react_output_parser中，当解析到tool call时，检查reasoning字段是否存在且非空。如果reasoning为空或缺失，标记为parse_warning，但不阻塞流程（因为阻塞会导致任务失败）：

```python
def parse_response(self, response_text: str) -> ParseResult:
    # ... 原有解析逻辑 ...
    
    if result.get("tool_name"):
        reasoning = result.get("reasoning", "")
        # 检查reasoning是否为空或极短（<10字符视为无效）
        if not reasoning or len(reasoning.strip()) < 10:
            # 标记warning，但不阻塞（避免任务失败）
            result["parse_warning"] = (
                "⚠️ reasoning字段为空或过短(<10字符)。"
                "有效的reasoning应包含：为什么选择这个工具、参数如何确定。"
            )
            # 【v1.8】不再修改result["content"]，改用独立parse_warning字段
            # 原因：修改content会污染原始LLM输出，存入history后影响后续LLM调用。
            # 调用方(base_react.py)可根据parse_warning决定是否在Observation中附加警告。
    
    return result
```

**效果**：当LLM跳过reasoning时，后端标记`parse_warning`但不修改LLM的原始输出。base_react.py在构建Observation时可以检查`parse_warning`字段，将警告信息附加到observation_text中传给LLM，而不会污染history中的原始assistant消息。

### 改进8（P1）：LLM调用参数完备化（max_tokens + temperature + seed）

**修改文件**：
- `backend/app/services/llm_core.py` — 请求体增加参数
- `backend/app/services/__init__.py` — 工厂方法传递参数
- `config/config.yaml` — 每个 provider 新增参数配置

**现状问题**：

`llm_core.py:168-172` 的 HTTP 请求体几乎是裸奔状态：

```python
json={
    "model": self.model,
    "messages": messages,
    "stream": True
    # 什么都没有了！
}
```

而 `ChatRequest`（API层）虽然定义了 `temperature: 0.7`，但只传给简单对话，Agent tool call 路径根本没传过去。`generate_sse_stream` 接收了 `temperature` 参数但从未使用——**是个死参数**。

#### 参数分析总表

| 参数 | 范围 | 对本项目是否该加 | 理由 |
|------|------|----------------|------|
| **max_tokens** | 正整数 | **✅ 必须加** | 无此参数时弱模型输出可能被静默截断，本事故中第1章1197字节→第2章起131字节与此相关 |
| **temperature** | 0-2 | **✅ 必须加** | API接口已定义(0.7)但实际未生效，是个bug。tool use场景推荐0.3-0.5降低随机性 |
| **seed** | 整数 | **✅ 建议加** | 固定seed可使输出可复现，调试时方便对比模型行为变化。不影响质量，零成本有收益 |
| **top_p** | 0-1 | **❌ 不加** | 与 temperature 功能重叠。设了 temperature 就不需要额外设 top_p，大多数模型推荐二选一 |
| **presence_penalty** | -2~2 | **❌ 不加** | 惩罚已出现token，理论上可降低"已成功"这类重复，但也会影响正常写作用词多样性，副作用不可控 |
| **frequency_penalty** | -2~2 | **❌ 不加** | 同上。且自我指涉检测（改进1）是对此问题的精准解法，不需要通用参数来间接干预 |
| **stop** | 字符串/数组 | **❌ 不加** | 适用于对话场景的停止标记，tool use场景下不需要 |

**关于 temperature 的关键说明**：

temperature 不能阻止本事故（思维泄漏是模型能力问题，不是随机性问题），但它是 LLM 调用的**基础参数**，API 接口收了、配置文件该配、调用时必须传。当前"接口定义但实际没生效"就是 bug。

**设计思路**：遵循现有 `timeout` 的配置传递模式，3个参数走同一条链。

```
config.yaml → provider.{max_tokens, temperature, seed}
                ↓
AIServiceFactory.get_service()
  → provider_config.get("max_tokens", 4096)
  → provider_config.get("temperature", 0.7)
  → provider_config.get("seed", None)
                ↓
BaseAIService.__init__(max_tokens, temperature, seed)
  → self.max_tokens / self.temperature / self.seed
                ↓
BaseAIService.chat_stream()
  → json["max_tokens"] = self.max_tokens
  → json["temperature"] = self.temperature
  → json["seed"] = self.seed (可选)
```

**修改方法**：

1. **`config/config.yaml`**：在每个 provider 下新增参数：


各参数建议值：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| max_tokens | 4096 | 通用任务够用；长文本写作设为8192 |
| temperature | 0.5 | tool use场景低温更好，比API默认0.7略低 |
| seed | None(不设) | 调试时可固定，生产环境不设 |

2. **`backend/app/services/__init__.py`**（AIServiceFactory）：在创建 BaseAIService 时传入：

## 七、事故复盘总结

### 7.1 根因链条

```
LLM弱模型(LongCat-Flash-Lite)
  → 把thought复制到write_file的content参数（触发条件）
    → write_file思维泄漏检测只对.py生效（缺陷2，未拦截）
      → write_file执行成功，131字节（系统认为正确）
        → agent层没有独立内容质量检测（缺陷3，未自愈）
          → Observation两条通道都不传质量信息（缺陷1，LLM和前端都不知道有问题）
            → assistant消息含"已成功写入"声明存入history（缺陷4，自我确认）
              → LLM看到自己说"已成功"，继续幻觉循环
                → _trim_history裁剪丢失"10章"约束（缺陷5，不停止）
                  → 循环25+次直到用户中断
```

### 7.2 防线缺失分析

| 防线 | 应有功能 | 实际状态 | 如有此防线能否阻止事故 |
|------|---------|---------|---------------------|
| 工具层校验 | write_file检测思维泄漏 | 仅.py文件 | ✅ 能阻止（131字中文计划语言会被拦截） |
| agent层校验 | Observation阶段独立检测+警告LLM | 无 | ✅ 能阻止（通过Observation告知LLM内容有问题） |
| Observation回传 | 回传content摘要+质量警告供LLM自查 | 无 | ✅ 能阻止（LLM看到自己写的内容不对，有机会修正） |
| 前端警告 | ObservationStep传质量警告给前端 | 无 | ⚠️ 不能阻止内容错误，但用户能发现并干预 |
| Prompt约束 | Schema中禁止思维确认语言+content语义说明 | 无 | ✅ 能大幅降低概率（从源头减少LLM犯错） |
| reasoning必填 | 强制LLM给出推理过程 | 非必需 | ⚠️ 缓解但不根治 |
| History保护 | 保留原始user约束 | 可能裁剪掉 | ⚠️ 不能阻止内容错误，但能阻止写到25章 |

**结论**：7道防线全部缺失。前4道（工具层+agent层+Observation回传+Prompt约束）任意一道有效即可大幅减轻事故。其中防线1（工具层校验）效果最好且成本最低；防线5（Prompt约束）是从源头预防，对所有LLM都有效。
