## 二、问题清单

### P0-紧急：必须立即修复

#### 问题1: `_add_within_budget` 预算检查不一致
**文件**: `base_react.py:1308`
**现象**: 方法接收 `budget` 参数但实际使用 `MAX_CONTEXT_CHARS` 做判断
```python
@staticmethod
def _add_within_budget(target: list, msg: dict, budget: int) -> bool:
    chars = len(msg.get("content", ""))
    budget_used = BaseAgent._total_chars(target)
    if budget_used + chars <= BaseAgent.MAX_CONTEXT_CHARS:  # ← 应该用 budget
        target.append(msg)
        return True
    return False
```
**影响**: 容量预算制形同虚设，单条超长消息可能撑爆窗口
**修复**: 将 `MAX_CONTEXT_CHARS` 改为 `budget`
**风险评分**: 5×4×3=60

---

### P1-高：必须修复

#### 问题2: `load_tools_by_intent` 中 `openai_tools` 可能未定义
**文件**: `base_react.py:292`
**现象**: 当 `tools_strategy` 存在但 `openai_tools` 未初始化时，`self.openai_tools.extend()` 抛 AttributeError
```python
if hasattr(self, 'tools_strategy') and self.tools_strategy is not None:
    # ...
    self.openai_tools.extend(...)  # ← openai_tools可能不存在
```
**影响**: 动态加载工具时可能崩溃
**修复**: 增加 `hasattr(self, 'openai_tools')` 检查或初始化默认值
**风险评分**: 4×3×2=24

#### 问题3: `_check_and_load_missing_tools` 死代码未清理
**文件**: `base_react.py:311-373`
**现象**: 全量注册后该方法第一行就 return，但保留了60+行未使用的关键词检测代码
**影响**: 代码维护负担，误导后续开发者
**修复**: 删除 return 后的全部死代码，保留方法签名和 return
**风险评分**: 2×5×3=30

#### 问题4: 并行工具执行不更新失败计数器
**文件**: `base_react.py:1069-1121`
**现象**: 并行工具执行结果不更新 `_failed_attempts` 和 `_executed_cache`
**影响**: 并行工具连续失败不会被拦截，可能浪费token
**修复**: 在并行工具 observation 构建后同步更新失败计数和缓存
**风险评分**: 3×3×2=18

### P2-中：应该修复

#### 问题5: `ensure_tools_registered` 异常吞没
**文件**: `tools/__init__.py:97-98`
**现象**: 注册失败只记录 error 日志，不抛出异常，`_tools_registered` 仍设为 True
```python
try:
    _import_and_register(module_path, register_func)
    _registered_categories.add(cat_name)
except Exception as e:
    logger.error(f"[Tools] 注册分类{cat_name}失败: {e}")
# ← 即使失败，_tools_registered 仍为 True
```
**影响**: 部分工具未注册但系统认为已全部注册，后续查找工具时返回空
**修复**: 注册失败时不设置 `_tools_registered = True`，或抛出异常
**风险评分**: 3×3×3=27

#### 问题6: `_executed_cache` 和 `_failed_attempts` 无TTL清理
**文件**: `base_react.py:144-149`
**现象**: 缓存字典只增不减，长时间运行的Agent可能内存泄漏
**影响**: 内存持续增长
**修复**: 在 `_trim_history` 中同步清理过期缓存条目
**风险评分**: 2×4×2=16

#### 问题7: `_format_result` 对非标准格式可能丢失 llm_data
**文件**: `tool_executor.py:448-455`
**现象**: 当工具返回既无 status/summary 也无 code 字段时，走 else 分支，不检查 llm_data
```python
else:
    r = {
        "status": "error",
        "summary": result.get("error", result.get("message", ...)),
        "data": result,
        "retry_count": 0
    }
    if _llm: r["llm_data"] = _llm  # ← 这行实际有，但逻辑分支复杂易漏
```
**影响**: 实际代码有透传，但分支逻辑复杂，后续维护容易遗漏
**修复**: 统一在方法开头提取 llm_data，所有返回路径都透传
**风险评分**: 2×2×3=12

#### 问题8: `search_web` 的 Bing 跳转链接未解码
**文件**: `network_tools.py:622-624`
**现象**: `bing.com/ck/a` 跳转链接直接展示给LLM，LLM无法直接使用
**影响**: LLM收到不可点击的跳转链接，降低搜索质量
**修复**: 尝试从URL参数中提取真实目标URL（`bing.com/ck/a?...&u=真实URL`）
**风险评分**: 2×3×3=18

#### 问题9: `response_format_strategy` 刷新时依赖 `openai_tools` 可能为空
**文件**: `base_react.py:297-303`
**现象**: 动态加载后刷新 enum 时，`self.openai_tools` 可能未初始化
**影响**: 刷新失败导致 response_format 的 enum 不完整
**修复**: 增加空值检查和 fallback 逻辑
**风险评分**: 3×2×2=12

### P3-低：建议修复

#### 问题10: `_is_observation_legacy` 可能误判
**文件**: `base_react.py:1276-1277`
**现象**: `"Observation:" in content and ("exec_status" not in content)` 可能误判包含"Observation:"的正常文本
**影响**: 极低，实际使用中罕见
**修复**: 增加 `startswith("Observation:")` 前缀检查
**风险评分**: 1×2×2=4

#### 问题11: `prompt_logger.py` 的 `log_llm_call` 记录完整消息列表可能内存大
**文件**: `prompt_logger.py:231`
**现象**: `"完整消息列表": messages` 直接保存所有消息，超长对话时JSON文件可能很大
**影响**: 日志文件过大，影响磁盘和读取性能
**修复**: 限制完整消息列表只保存前N条和最后N条
**风险评分**: 1×3×2=6

#### 问题12: `react_agent_mixin.py` 中 `_last_strategy_method` 初始化时机
**文件**: `react_agent_mixin.py:118`
**现象**: `_last_strategy_method` 在 `_init_llm_strategies` 中初始化为 None，但 `_build_alternative_tools_hint` 在第一次LLM调用前就可能读取
**影响**: 首次工具失败时策略提示可能不准确
**修复**: 在 `_call_llm_with_summary` 首次执行后设置默认值
**风险评分**: 1×2×2=4

## 三、代码质量亮点

| 亮点 | 说明 |
|------|------|
| **llm_data双通道** | 工具自主决定LLM需要什么数据，避免原始数据污染上下文 |
| **容量感知裁剪** | 用字符数预算替代计数制，更精确控制上下文窗口 |
| **内容指纹去重** | 取content前200字符做指纹，语言无关、工具无关 |
| **hashlib防碰撞** | 工具缓存key用MD5替代截断，杜绝碰撞 |
| **失败拦截机制** | 连续失败3次强制跳过，避免LLM重复调用 |
| **策略前置** | FC模式跳过工具文本注入，减少冗余token |
| **全量注册** | 一次性注册所有工具，消除动态加载复杂度 |
