# llm_core 消息格式兼容设计方案

版本: v1.3 | 创建时间: 2026-05-16 11:50:25 | 作者: 小沈

---

## 一、需求背景

### 1.1 问题描述

系统当前所有LLM消息均使用text格式发送：
```json
{"role": "user", "content": "你好"}
```

LongCat-Flash-Omni-2603等多模态模型要求system/user使用数组格式：
```json
{"role": "system", "content": [{"type": "text", "text": "你是一个智能助理"}]}
{"role": "user", "content": [{"type": "text", "text": "你好"}]}
```

用text格式发请求，Omni模型返回400错误：
```
{"error":{"code":"invalid_format","message":"json format error"}}
```

需要llm_core自动适配，对上层调用方透明。

### 1.2 日志报错详解

**错误1（08:42，text格式发Omni）**：
```
HTTP 400: {"error":{"message":"json format error","code":"invalid_format"}}
```
Omni不认text格式，要求system/user的content必须是数组。

**错误2（10:02/10:09，全数组格式发Omni）**：
```
HTTP 400: {"error":{"code":"invalid_parameter","message":"Assistant message can only contain text data, multimodal content (images, videos, audio, etc.) is not supported"}}
```
Omni要求assistant角色的content只能用text字符串，不能用数组。

### 1.3 国内LLM多模态格式调研结论

| 厂商 | system | user(纯文本) | user(多模态) | assistant | 响应content |
|------|--------|-------------|-------------|-----------|------------|
| 智谱GLM-4V | string | string | array | string | string |
| 通义Qwen-VL | string | string | array | string | string |
| 月之暗面Kimi | string | string | array | string | string |
| 百度文心ERNIE | string | string | array | string | string |
| 讯飞星火Spark | string | string | array | string | string |
| DeepSeek | string | string | array | string | string |
| **LongCat-Omni** | **array** | **array** | **array** | **string** | **string** |

6大厂商全部OpenAI兼容，system/assistant始终string。**LongCat-Omni是唯一特例**：system/user纯文本也用数组。

### 1.4 需求定义

1. 默认text格式（所有角色string）—— 覆盖绝大多数模型
2. 支持混合格式（system/user数组 + assistant text）—— 给LongCat-Omni
3. 在CapabilityDetector探测阶段确定格式，**不靠请求试错**
4. 探测结果一次性设置到BaseAIService，后续零开销

## 二、现有架构链路（tools/response_format能力的传递方式）

```
CapabilityDetector.detect()
  → LLMProbeResult(feature=LLMFeature(supports_tools=True/False, ...))
    → LLMAdapter.ensure_capability()
      → StrategySelector.select(feature)
        → SelectedStrategy(method="text"/"tools"/"response_format")
          → react_agent_mixin._call_llm_with_summary()
            → strategy.call(llm_client=self.llm_client, ...)
              → llm_client.chat_stream() / chat_with_tools() / ...
```

**关键**：strategy的method（text/tools/response_format）决定了调用llm_client的哪个方法。但BaseAIService本身不感知LLMFeature。

## 三、详细设计

### 3.1 格式定义

| 格式名 | system | user | assistant | 适用场景 |
|--------|--------|------|-----------|---------|
| **text**（默认） | `"文字"` | `"文字"` | `"文字"` | 绝大多数模型 |
| **混合** | `[{"type":"text","text":"文字"}]` | `[{"type":"text","text":"文字"}]` | `"文字"` | LongCat-Omni |

### 3.2 修改文件清单

| 文件 | 修改点 |
|------|--------|
| `capability.py` | LLMFeature加 `supports_array_content: bool = False` |
| `capability_detector.py` | 加 `_probe_array_content()` 方法 |
| `llm_adapter.py` | `ensure_capability()` 中把探测结果设置到BaseAIService |
| `llm_core.py` | `_build_messages`加`array_format`参数 + `set_array_content()`方法 |

### 3.3 LLMFeature 加字段

```python
# capability.py
@dataclass
class LLMFeature:
    # ... 现有字段 ...
    supports_array_content: bool = False  # system/user用数组格式（如LongCat-Omni）
```

### 3.4 CapabilityDetector 加探测方法

```python
# capability_detector.py
async def _probe_array_content(self, client: httpx.AsyncClient) -> dict:
    """
    探测消息是否需要数组格式（system/user用数组, assistant用text）
    
    用混合格式发一个简单请求，200=支持，400格式错误=不支持
    """
    try:
        response = await client.post(
            f"{self.api_base}/chat/completions",
            headers={"Authorization": f"Bearer {self.api_key}", "Content-Type": "application/json"},
            json={
                "model": self.model,
                "messages": [
                    {"role": "system", "content": [{"type": "text", "text": "hi"}]},
                    {"role": "user", "content": [{"type": "text", "text": "hi"}]}
                ],
                "stream": False,
                "max_tokens": 5
            }
        )
        if response.status_code == 200:
            return {"works": True, "reason": "OK"}
        # 400格式错误 → 不支持数组格式，用默认text
        return {"works": False, "reason": f"HTTP {response.status_code}"}
    except Exception as e:
        return {"works": False, "reason": str(e)}
```

### 3.5 CapabilityDetector.detect() 加Step 0

在现有的 `Step 1: 探测tools` 之前加：

```python
# Step 0: 探测消息格式（最快，max_tokens=5，开销极小）
array_result = await self._probe_array_content(client)
feature.supports_array_content = array_result["works"]
```

### 3.6 BaseAIService 加属性和方法

```python
# llm_core.py - BaseAIService

def __init__(self, ...):
    # ... 现有代码 ...
    self._array_format: bool = False  # 默认text格式

def set_array_content(self, supported: bool):
    """设置消息格式能力 - 由LLMAdapter探测后调用"""
    self._array_format = supported
    fmt_label = "混合(system/user数组+assistant text)" if supported else "text"
    logger.info(f"[BaseAIService] 消息格式设为: {fmt_label}")
```

### 3.7 _build_messages 加参数

```python
def _build_messages(self, message: str, history: Optional[List[Message]] = None) -> List[Dict]:
    """构建消息列表 - 小沈 2026-05-16 支持混合格式"""
    messages = []
    if history:
        for msg in history:
            if self._array_format and msg.role in ("system", "user"):
                messages.append({"role": msg.role, "content": [{"type": "text", "text": msg.content}]})
            else:
                messages.append({"role": msg.role, "content": msg.content})
    
    # 当前用户消息
    if self._array_format:
        messages.append({"role": "user", "content": [{"type": "text", "text": message}]})
    else:
        messages.append({"role": "user", "content": message})
    
    return messages
```

**注意**：不新增参数，直接用 `self._array_format` 实例属性。4个LLM方法**零改动**——它们都调用 `_build_messages`，`_build_messages` 内部自动按格式处理。

### 3.8 LLMAdapter 传递探测结果

```python
# llm_adapter.py - LLMAdapter

async def ensure_capability(self, llm_client=None) -> SelectedStrategy:
    """
    确保能力已探测，返回选中的策略
    
    【小沈 2026-05-16】新增llm_client参数，探测完后设置消息格式
    """
    if self._strategy is None:
        result = await self._detector.detect()
        if result.success:
            self._feature = result.feature
            self._strategy = StrategySelector.select(self._feature)
            
            # 【小沈 2026-05-16】设置消息格式到BaseAIService
            if llm_client and hasattr(llm_client, 'set_array_content'):
                llm_client.set_array_content(self._feature.supports_array_content)
    
    return self._strategy
```

### 3.9 react_agent_mixin 调用处传入 llm_client

```python
# react_agent_mixin.py - _call_llm_with_summary()

# 现有代码：
strategy = await self.adapter.ensure_capability()

# 改为：
strategy = await self.adapter.ensure_capability(llm_client=self.llm_client)
```

**仅改这一处调用**，传入llm_client，让adapter能设置格式。

## 四、完整调用链路（修改后）

```
1. 用户切换模型 → AIServiceFactory创建新BaseAIService实例（_array_format=False）
2. Agent首次调用LLM → react_agent_mixin._call_llm_with_summary()
3.   → adapter.ensure_capability(llm_client=self.llm_client)
4.     → CapabilityDetector.detect()
5.       → _probe_array_content()  ← 新增：用混合格式发探测请求
6.       → _probe_tools()
7.       → _probe_response_format()
8.       → _probe_reasoning()
9.     → LLMFeature(supports_array_content=True/False, ...)
10.    → llm_client.set_array_content(True/False)  ← 新增：设置格式
11.    → StrategySelector.select(feature) → SelectedStrategy(method="...")
12.  → strategy.call(llm_client=self.llm_client, ...)
13.    → llm_client.chat_stream() / chat_with_tools() / ...
14.      → _build_messages()  ← 内部自动按self._array_format处理
```

## 八、LLM核心方法架构重设计

### 8.1 现状问题分析

llm_core.py 中现有6个方法，但**架构混乱**：

| 方法 | 流式? | 谁调用 | 问题 |
|------|-------|--------|------|
| `chat()` | 非流式 | `react_sse_wrapper` → 普通对话 | ✅ 正常 |
| `chat_stream()` | 流式 | `chat()`内聚合 + `chat_stream_query` → 前端SSE | ✅ 正常 |
| `chat_with_tools()` | 非流式 | `ToolsStrategy.call()` → Agent FC模式 | ✅ 正常 |
| `chat_with_tools_stream()` | 流式 | **无人调用** | ❌ 死代码 |
| `chat_with_response_format()` | 非流式 | `ResponseFormatStrategy.call()` → Agent JSON模式 | ✅ 正常 |
| `chat_with_response_format_stream()` | 流式 | **不存在** | ❌ 缺失 |

**3个问题**：
1. `chat_with_tools_stream()` 写了没人用——死代码
2. `chat_with_response_format_stream()` 根本不存在——架构不完整
3. 6种组合只有4种实现，不对称

### 8.2 调用场景分析

**场景1：普通对话（前端SSE流式推送）**
```
用户发消息 → chat_router → chat_stream_query → ai_service.chat_stream() → SSE推送
```
用 `chat_stream()`，纯文本流式。✅

**场景2：普通对话（非流式聚合）**
```
chat() 内部调 chat_stream() 聚合成完整响应
```
用 `chat()`，是 `chat_stream()` 的包装。✅

**场景3：Agent用Function Calling**
```
react_agent_mixin → ToolsStrategy.call() → llm_client.chat_with_tools()
```
用 `chat_with_tools()`，非流式。**为什么非流式？** 因为Agent需要拿到完整响应后解析tool_calls，再决定下一步。这个场景**不需要流式**。✅

**场景4：Agent用Structured Outputs**
```
react_agent_mixin → ResponseFormatStrategy.call() → llm_client.chat_with_response_format()
```
用 `chat_with_response_format()`，非流式。同理，Agent需要完整JSON。**不需要流式**。✅

**场景5：前端SSE + Function Calling流式（潜在）**
Agent用FC模式时，前端也想逐token看到LLM的思考过程？**当前没实现**。

**场景6：前端SSE + Structured Outputs流式（潜在）**
同理，**当前没实现**。

### 8.3 关键判断：Agent的FC/RF模式需不需要流式？

| 维度 | 分析 |
|------|------|
| Agent逻辑 | 必须拿到完整响应才能解析tool_calls/JSON → **不需要流式** |
| 用户体验 | 前端想看到Agent的实时思考过程 → **但这是Agent层面的流式（step by step），不是LLM层面的流式** |
| LLM层面 | FC模式返回tool_calls，流式返回delta拼装复杂，且最终还是要等完整tool_calls → **非流式更简单可靠** |

**结论**：FC和RF模式在LLM层面用**非流式**就够了。流式需求在Agent层面通过step事件解决。

### 8.4 合理架构设计

```
BaseAIService
├── chat_stream()                    # 纯文本流式（SSE推送到前端）
├── chat()                           # 纯文本非流式（内部调chat_stream聚合）
├── chat_with_tools()                # Function Calling非流式（Agent用）
└── chat_with_response_format()      # Structured Outputs非流式（Agent用）
```

**4个方法，职责清晰**：

| 方法 | 返回类型 | 调用者 | 用途 |
|------|---------|--------|------|
| `chat_stream()` | `AsyncGenerator[StreamChunk]` | `chat_stream_query` | 前端SSE逐token推送 |
| `chat()` | `ChatResponse` | `react_sse_wrapper` | 普通对话完整响应 |
| `chat_with_tools()` | `ChatResponse` | `ToolsStrategy` | Agent FC模式 |
| `chat_with_response_format()` | `ChatResponse` | `ResponseFormatStrategy` | Agent JSON模式 |

**删除**：
- `chat_with_tools_stream()` — 死代码，Agent不需要FC流式

**不补**：
- `chat_with_response_format_stream()` — 没有场景，Agent不需要RF流式

**关系**：
- `chat()` 是 `chat_stream()` 的非流式包装（实现复用，内部聚合）
- 其余3个方法各自独立实现

### 8.5 每个方法的内部结构（统一模板）

```python
async def xxx(self, message, history, ...特定参数) -> 返回类型:
    """
    统一结构：
    1. _build_messages(message, history)  → 构建消息（格式由_array_format控制）
    2. 构建request_body（加特定参数如tools/response_format/stream）
    3. 发HTTP请求
    4. 解析响应
    """
```

**关键**：`_build_messages()` 是唯一的消息构建入口，格式逻辑集中在此。

### 8.6 重构清单

| 操作 | 对象 | 说明 |
|------|------|------|
| **保留** | `chat_stream()` | 核心流式方法，前端SSE依赖 |
| **保留** | `chat()` | 非流式包装，react_sse_wrapper依赖 |
| **保留** | `chat_with_tools()` | Agent FC模式，ToolsStrategy依赖 |
| **保留** | `chat_with_response_format()` | Agent RF模式，ResponseFormatStrategy依赖 |
| **删除** | `chat_with_tools_stream()` | 死代码，无任何调用者 |
| **不补** | `chat_with_response_format_stream()` | 无场景，不实现 |

### 8.7 与消息格式兼容的关系

4个方法全部调 `_build_messages()`，格式逻辑集中在 `_build_messages` 内部：

```
chat_stream()               → _build_messages() → 按self._array_format处理
chat()                      → chat_stream()     → 间接走_build_messages
chat_with_tools()           → _build_messages() → 按self._array_format处理
chat_with_response_format() → _build_messages() → 按self._array_format处理
```

**4个方法零改动**，格式兼容只在 `_build_messages` 一处实现。
