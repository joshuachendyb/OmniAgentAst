## 一、精简必要性分析

### 1.1 当前状态

系统注册工具总数 **148个**（经逐件普查确认），分属15个注册分类。其中用户关注的"文档及数据分析事务处理"和"日常工具"合计 **66个**，占总量 44.6%。

### 1.2 核心问题

| 问题 | 症状 | 后果 |
|------|------|------|
| **工具过多** | 148个工具，LLM每次需从大量候选中选择 | 选择错误率高，频繁补救调用 |
| **功能重叠** | 多个工具能做同一件事（如读CSV有2个、读Excel有2个） | LLM不知道该选哪个，浪费token |
| **粒度过细** | 一个逻辑概念拆成3-4个工具（如时间检查拆4个） | LLM看到场景后需要多步推理才能选对 |
| **低频冗余** | 存在 parse_properties、file_monitor 等几乎用不到的工具 | 占用工具位，增加LLM选择负担 |
| **命名不一致** | get_file_hash vs file_checksum，precise_replace_in_file vs edit_text_file | LLM无法从名字推断功能关系 |

### 1.3 精简目标

- 工具总数从 **148 → ~112**（减少 24%）
- 核心区域（文档+日常）从 **66 → 37**（减少 44%）
- 任意两个工具功能互斥，LLM看到一个场景只对应一个工具
- 保留100%的场景覆盖率，删除和合并只为消除冗余

> 🔍 **小沈深度审查 — P9 低风险：缺少 token 节省估算**
>
> **原方案问题**：精简目标只列出了工具数量减少（148→112），但没有估算对 LLM token 消耗的实际影响：
> - 工具数量减少不代表 token 线性减少——每个工具的 description 长度不同，合并后单个工具的 description 可能更长
> - 实际 token 节省来自：① 工具列表传入 API 时的描述总量减少 ② LLM 选择错误工具后的重试减少
> - 缺少估算，无法量化精简的实际收益
>
> **估算补充**：
>
> | 项目 | 精简前 | 精简后 | 节省 |
> |------|--------|--------|------|
> | 工具数量 | 148 | 112 | -36 (-24%) |
> | 估算描述总量（每工具均约80 token） | ~11,840 | ~8,960 | ~2,880 token (-24%) |
> | 实际节省（合并后描述更长，按×1.3系数） | — | ~11,648 | ~192 token（描述本身节省少） |
> | 选择错误率降低带来的间接节省 | 频繁选错需重试 | 选错率预计降50% | 每次对话省1-2轮重试 ≈ 500-1000 token |
> | **综合估算** | — | — | **每次对话约节省 500-1200 token** |
>
> **结论**：token 直接节省较少（工具描述总量只减~192 token），但间接节省（减少LLM选择错误和重试）更显著，每次对话约省500-1200 token。精简的核心价值不是省token，而是**提高LLM选择准确率**。

## 二、精简核心原则

### 2.1 用户要求的三条原则

| 编号 | 原则 | 解释 | 判断标准 |
|------|------|------|---------|
| **P1** | 重复类似的合并或删除 | 两个工具能做同一件事 → 只保留一个 | 对比两个工具的参数和返回值，功能交集 > 80% 则合并 |
| **P2** | 相似的进行功能整合 | 同一逻辑概念拆成多个工具 → 合并为一个，用参数区分 | 工具名包含同一关键词（如 read_*, time_*）的优先合并 |
| **P3** | 功能互斥，留下主要的 | 任意两个工具之间功能不应有交集 | LLM 看到任何一个业务场景，只应匹配一个工具 |

### 2.2 补充设计原则

| 编号 | 原则 | 解释 | 示例 |
|------|------|------|------|
| **P4** | 单一职责 | 每个工具只做一件事，名字清晰表达意图 | `read_text_file` → `read_file`（不做写操作） |
| **P5** | 后缀自动路由 | 读写类工具按文件后缀自动选择底层实现 | `read_document("a.pdf")` 自动调 pdfplumber；`read_document("b.xlsx")` 自动调 openpyxl |
| **P6** | 最小完备性 | 用最少工具覆盖最多场景，不为低频场景单独设工具 | 删除 parse_properties（几乎用不到），保留 read_json（高频） |
| **P7** | 悲观默认值 | 参数有合理默认值，减少 LLM 需要传递的参数 | algorithm 默认 sha256，encoding 默认 auto_detect |
| **P8** | 自描述性 | 工具名直接表达功能，LLM 不需要读 description 就知道该用哪个 | `read_file` > `read_text_file`（删掉无意义的 text 限定词） |
| **P9** | 向下兼容 | 精简后保留别名映射，确保已有调用不中断 | tool_aliases.py 增加 `read_text_file → read_file` 映射 |
| **P10** | 先易后难分批执行 | 按影响范围和风险分批，每批独立可验证 | Time 先做（独立性强），File 最后（影响最广） |

### 2.3 架构层设计原则（北京老陈提出——工具间复用与增强）

| 编号 | 原则 | 解释 | 示例 |
|------|------|------|------|
| **P11** | **统一入口 + 内部分发** | 对外暴露一个入口工具，内部按条件分发到底层实现 | `read_document("file.xlsx")` → 内部调用 `_read_excel_base(use_pandas=False)`，与 `read_excel()` 共用同一套底层 |
| **P12** | **工具间互相调用（组合复用）** | 工具A可以调用工具B的函数来增强自身，形成能力网络 | `get_file_info` 内部调用 `file_statistics` 的单文件模式，额外返回"目录占比" |
| **P13** | **分层架构：Tool层 vs Helper层** | 区分"对外暴露的Tool"和"内部复用的Helper" | `get_file_hash` 和 `file_checksum` 不合并，但都调用同一个 `_calculate_hash` Helper |
| **P14** | **优雅降级（Graceful Degradation）** | 增强功能不可用时，自动回退到基础功能 | `read_excel(use_pandas=True)` 时若 pandas 未安装 → 自动降级到 openpyxl 基础模式 |
| **P15** | **返回值自解释（Self-Describing Result）** | 工具返回结果时告诉LLM"下一步可以做什么" | `read_csv` 返回 `next_actions: [{"tool": "analyze_data", "when": "需要统计分析时"}]` |
| **P16** | **幂等性（Idempotency）** | 同一操作多次执行，结果一致，不产生副作用累积 | `create_directory` 对已存在目录返回"已存在，无需创建"而非报错 |
| **P17** | **参数正交（Orthogonal Parameters）** | 参数之间应独立，避免"传A就不能传B"的互斥关系 | `rename_file(mode="single", source="a.txt", target="b.txt")` 替代互斥参数 |

### 2.4 原则优先级

```
P1 重复合并 > P2 相似整合 > P3 功能互斥 > P4 单一职责 > P5 后缀路由
                                                    ↓
P6 最小完备 > P7 悲观默认 > P8 自描述 > P9 兼容 > P10 分批
                                                    ↓
P11 统一入口 > P12 组合复用 > P13 分层架构 > P14 优雅降级
                                                    ↓
P15 返回值自解释 > P16 幂等性 > P17 参数正交
```

> **优先级说明**：P1-P10 是"精简层面"的原则（决定哪些工具保留/合并/删除），P11-P17 是"架构层面"的原则（决定工具内部如何设计和实现）。精简先行，架构优化跟进。

## 四、问题诊断（5类问题，逐工具标注）

### 4.1 完全重复（P1级——必须解决）

#### D1: file_checksum vs get_file_hash

| 维度 | file_checksum | get_file_hash |
|------|-------------|--------------|
| 位置 | `file_tools.py:1640` | `file_tools.py:1568` |
| 参数 | file_path, algorithm(SHA256), verify_hash, chunk_size, timeout | file_path, algorithm(SHA256), verify_against, timeout |
| 实现 | 调用 `file_checksum_impl` | 调用 `_get_file_hash` helper |
| 底层 | 同一套哈希计算逻辑 | 另一套哈希计算逻辑 |
| 结论 | **两个工具在同一类里，提供完全相同的功能，LLM 看到两个名字不知道该用哪个** |

**处理**: 保留 `get_file_hash`（参数更简洁），删除 `file_checksum`，删除 `file_checksum.py`。

#### D2: list_windows 双重注册

| 维度 | desktop/list_windows | gui/list_windows |
|------|---------------------|------------------|
| 来源 | desktop_register.py | gui_register.py |
| 功能 | 枚举 Windows 窗口 | 枚举 GUI 窗口 |
| 结论 | **同一工具名注册两次，分类不同但功能重叠** |

**⚠️ 运行时Bug分析**：由于 ToolRegistry 允许重复注册（后注册覆盖前注册），而 `_CATEGORY_MODULES` 的遍历顺序是 desktop 在先、desktop_gui 在后，所以**最终生效的是 gui_register 中的 list_windows 实现**。desktop_register 中的 list_windows 实际上被覆盖了，是一个**已有的运行时Bug**。

**处理**: 删除 gui_register.py 中的 list_windows 注册。desktop_register 中的 list_windows 保留不变，因为它是独立的窗口枚举功能，不依赖 GUI 自动化。如果 GUI 自动化场景需要专用窗口列表，可以新增 `list_gui_windows` 工具（带额外窗口过滤能力）。

### 4.2 功能重叠（P1级——LLM选择困难）

#### O1: read_csv_basic vs read_csv_dataframe

| 维度 | read_csv_basic | read_csv_dataframe |
|------|---------------|-------------------|
| 分类 | data_format | data_analysis |
| 底层 | csv stdlib（零依赖） | pandas（需安装） |
| 返回 | headers + rows（简单列表） | columns + rows + dtypes（DataFrame格式） |
| 上限 | max_rows=500 | max_rows=1000 |
| 场景 | 快速浏览 CSV | 需要后续分析 |
| 冲突 | LLM "帮我读这个 CSV" → 两个都匹配，随机选错 | |

**处理**: 合并为 `read_csv(path, use_pandas=False)`。默认不加 pandas（零依赖，快速），传 `use_pandas=True` 时走 DataFrame 路径。

#### O2: read_xlsx vs read_excel_dataframe

| 维度 | read_xlsx | read_excel_dataframe |
|------|----------|---------------------|
| 分类 | document | data_analysis |
| 底层 | openpyxl 直接读 | pandas + openpyxl |
| 返回 | headers + rows | columns + rows + dtypes + sheet_name |
| 冲突 | LLM "帮我读这个 Excel" → 两个都匹配 | |

**处理**: 合并为 `read_excel(path, use_pandas=False)`。默认用 openpyxl（不装 pandas 也能用），传 `use_pandas=True` 时走 DataFrame。

#### O3: edit_text_file vs precise_replace_in_file

| 维度 | edit_text_file | precise_replace_in_file |
|------|---------------|------------------------|
| 接口 | `edits: [{oldText, newText}]` | `old_string, new_string` |
| 多编辑 | 支持（一次传多个 edits） | 不支持单次多编辑 |
| 预览 | 支持 dryRun | 不支持 |
| 冲突 | 都是替换文件内容，接口不同 | |

**处理**: 合并为 `edit_file(path, old_str=None, new_str=None, edits=None, dry_run=False)`。优先用 old_str+new_str 单替换，传 edits 时批量替换。

#### O4: rename_file vs batch_rename

| 维度 | rename_file | batch_rename |
|------|------------|-------------|
| 粒度 | 单文件重命名 | 正则批量重命名 |
| 参数 | file_path + new_name | directory + pattern + replacement |
| 冲突 | 单文件重命名"a.txt → b.txt"，用 batch_rename("*.txt", "b.txt") 也能实现 | |

**处理**: 合并为 `rename_file(path=None, new_name=None, directory=None, pattern=None, replacement=None, preview=False)`。传 path+new_name→单文件；传 directory+pattern→批量。

#### O5: search_files vs grep_file_content

| 维度 | search_files | grep_file_content |
|------|-------------|------------------|
| 搜索对象 | 文件名（glob匹配） | 文件内容（ripgrep/正则） |
| 冲突 | LLM"找一下 main.py"→两个都可能选（main.py 既匹配文件名 also 可能匹配内容） | |

**处理**: 合并为 `search_files(path, pattern, search_content=None, glob=None)`。有 search_content → grep 内容；无 search_content → glob 文件名。

#### O6: list_directory vs get_directory_tree

| 维度 | list_directory | get_directory_tree |
|------|---------------|-------------------|
| 返回 | 平铺列表（name/path/type/size/mtime） | JSON 树（递归 children） |
| 冲突 | LLM"看看这个目录"→两个都可能选 | |

**处理**: 合并为 `list_directory(path, format="list"|"tree", recursive=False)`。format="list" 返回平铺列表，format="tree" 返回 JSON 树。

#### O7: get_file_info vs file_statistics

| 维度 | get_file_info | file_statistics |
|------|--------------|----------------|
| 粒度 | 单个文件/目录的元数据 | 整个目录的统计汇总 |
| 冲突 | LLM"这个文件多大"vs"这个目录多大"→语义边界模糊，可能选错 | |

**处理**: 保留两个独立工具，但在 description 中明确区分："单文件用 get_file_info"、"整个目录用 file_statistics"。

>
> **P3 原方案问题**：将 `search_files` + `grep_file_content` 合并为一个 `search_files(search_content=...)` 工具：
> - 两个工具的**核心语义不同**：`search_files` 是"按文件名模式搜索"（glob语义），`grep_file_content` 是"按文件内容搜索"（grep语义）
> - 合并后 LLM 需要理解 `search_content` 参数的存在改变了工具的整个行为模式——这不是"参数差异"而是"模式切换"
> - 实际场景中 LLM 经常混淆：传了 `pattern` 却也传了 `search_content`，或忘记传 `search_content` 却期望搜索内容
> - grep_file_content 有4个独有高级参数（context_lines、ignore_case、show_line_numbers、output_mode），合并后参数膨胀到8个
>
> **P10 原方案问题**：根据P3分析，search_files应保留独立工具，不合并grep_file_content
>
> **修正方案**：保留 `search_files` 和 `grep_file_content` 两个独立工具
> - `search_files`：按文件名搜索（glob模式），参数：path, pattern, recursive
> - `grep_file_content`：按文件内容搜索（grep模式），参数：path, pattern, context_lines, ignore_case, show_line_numbers, output_mode
> - 两者语义清晰互斥，LLM看到"找文件"选search_files，看到"找内容"选grep_file_content，零歧义
> - File精简效果从26→16调整为26→17（多保留1个工具）

> 注：`file_statistics` 和 `get_file_info` 按 O7 分析保留两个，因它们服务不同场景（一个查单文件，一个统计目录）。

>
> **原方案问题**：16→5（将5个时间获取/计算类工具全部合并为 `get_time(action=...)`），过于激进：
> - `get_time` 一个工具承担5种功能（now/format/add/diff/compare），参数达6个，LLM 传参复杂度高
> - `action` 枚举值5个，每种 action 的必填参数不同（now无需参数，add需time_value+delta，diff需time_value+time2），LLM 容易传错
> - 合并后 `get_time` 的 description 需要解释5种模式，token开销反而比5个独立工具更大
> - 违反 P4 单一职责原则

> - `check_date`、`timezone_convert`、`time_convert`、`timer` 维持原合并方案不变
>
> **修正理由**：合并的核心价值是减少LLM选择范围，16→8已经将选择范围减半，同时避免单个工具过于复杂。OpenAI建议单个工具参数不超过5个，修正后 `get_time` 参数从6个降至4个。

### 10.3 Timer callback 风险说明

`timer(action="set")` 的 `callback` 参数目前接收字符串描述（如"提醒我开会"），由 Agent 在定时器触发时执行对应的动作。**不建议**让 LLM 传递可执行代码字符串，存在安全风险。当前实现中 timer 触发后仅返回通知消息，不执行任意代码。

## 十一、架构设计优化（北京老陈提出——深化设计模式）

> **核心思想**：不是"把多个工具捏成一个大工具"，而是"**工具之间通过共享底层、互相调用来形成能力网络**"。

### 11.1 Helper层提取（解决 D1 根本问题）

**问题**：`file_checksum` vs `get_file_hash` 完全重复，但简单合并会丢失一个接口。

**更好方案**：提取 Helper 层，两个工具保留独立接口，但共用底层实现。

```python
# helper层（工具间共享，不暴露给LLM）
def _calculate_hash(file_path: str, algorithm: str = "sha256") -> str:
    """底层哈希计算 - 所有哈希工具共用 - 小沈 2026-05-17"""
    import hashlib
    h = hashlib.new(algorithm)
    with open(file_path, 'rb') as f:
        for chunk in iter(lambda: f.read(8192), b''):
            h.update(chunk)
    return h.hexdigest()

# tool层（LLM可调用的工具）
def get_file_hash(file_path: str, algorithm: str = "sha256") -> dict:
    """计算文件哈希值 - 小沈 2026-05-17"""
    hash_value = _calculate_hash(file_path, algorithm)
    return {"code": "OK", "data": {"hash": hash_value, "algorithm": algorithm}}

def verify_file_hash(file_path: str, expected_hash: str, algorithm: str = "sha256") -> dict:
    """验证文件哈希是否匹配 - 小沈 2026-05-17"""
    actual_hash = _calculate_hash(file_path, algorithm)
    match = actual_hash.lower() == expected_hash.lower()
    return {
        "code": "OK",
        "data": {
            "match": match,
            "expected": expected_hash,
            "actual": actual_hash,
            "algorithm": algorithm
        }
    }
```

**好处**：
- 不丢失任何接口（get_file_hash 和 verify_file_hash 都保留）
- 底层逻辑只维护一份（_calculate_hash）
- LLM 看到两个独立工具，语义清晰

**目标**：让工具多次执行同一操作不产生副作用，LLM 不需要先"检查是否存在"再操作。

def create_directory(path: str, exist_ok: bool = True) -> dict:
    """创建目录（幂等）- 小沈 2026-05-17"""
    import os
    if os.path.exists(path):
        if exist_ok:
            return {"code": "OK", "message": "目录已存在，无需创建", "data": {"path": path}}
        else:
            return {"code": "ERR_EXISTS", "message": f"目录已存在: {path}"}
    os.makedirs(path)
    return {"code": "OK", "message": "目录创建成功", "data": {"path": path}}

```python
# 真实事务实现 - 小沈 2026-05-17
import uuid
import sqlite3
import threading

_transaction_lock = threading.Lock()
_active_transactions: Dict[str, sqlite3.Connection] = {}

def begin_transaction(db_path: str) -> Dict[str, Any]:
    """开始真实数据库事务 - 小沈 2026-05-17"""
    try:
        conn = sqlite3.connect(db_path)
        conn.execute("BEGIN TRANSACTION")
        tx_id = str(uuid.uuid4())[:8]
        with _transaction_lock:
            _active_transactions[tx_id] = conn
        return {"code": "SUCCESS", "data": {"transaction_id": tx_id, "db_path": db_path},
                "message": f"事务已开始: {tx_id}"}
    except Exception as e:
        return {"code": "ERR_TX_BEGIN", "message": f"开始事务失败: {str(e)}"}

def commit_transaction(transaction_id: str) -> Dict[str, Any]:
    """提交真实数据库事务 - 小沈 2026-05-17"""
    with _transaction_lock:
        conn = _active_transactions.pop(transaction_id, None)
    if conn is None:
        return {"code": "ERR_TX_NOT_FOUND", "message": f"事务不存在或已结束: {transaction_id}"}
    try:
        conn.execute("COMMIT")
        conn.close()
        return {"code": "SUCCESS", "data": {"transaction_id": transaction_id},
                "message": f"事务已提交: {transaction_id}"}
    except Exception as e:
        try:
            conn.execute("ROLLBACK")
        except:
            pass
        conn.close()
        return {"code": "ERR_TX_COMMIT", "message": f"提交事务失败: {str(e)}"}

def rollback_transaction(transaction_id: str) -> Dict[str, Any]:
    """回滚真实数据库事务 - 小沈 2026-05-17"""
    with _transaction_lock:
        conn = _active_transactions.pop(transaction_id, None)
    if conn is None:
        return {"code": "SUCCESS", "message": f"事务不存在或已结束: {transaction_id}"}  # 幂等
    try:
        conn.execute("ROLLBACK")
        conn.close()
        return {"code": "SUCCESS", "data": {"transaction_id": transaction_id},
                "message": f"事务已回滚: {transaction_id}"}
    except Exception as e:
        conn.close()
        return {"code": "ERR_TX_ROLLBACK", "message": f"回滚事务失败: {str(e)}"}
```

def check_screen_capture_permission() -> Dict[str, Any]:

    if not WIN32_AVAILABLE:

def check_network_connectivity(method: str = "auto") -> Dict[str, Any]:
    """检查网络连通性（增强版）- 小沈 2026-05-17
    
    Args:
        method: 检测方式
            - "auto": 依次尝试 DNS → HTTPS，任意成功即返回可用
            - "dns": 仅DNS端口检测
            - "https": 仅HTTPS请求检测
    """
    # DNS 检测（原有方法）
    if method in ("auto", "dns"):
        dns_result = _check_dns_connectivity()
        if dns_result["data"]["connected"]:
            return dns_result
    
    # HTTPS 检测（新增，更可靠）
    if method in ("auto", "https"):
        try:
            import httpx
            r = httpx.get("https://www.google.com", timeout=5)
            return {"code": "SUCCESS", "data": {"connected": True, 
                    "method": "https", "latency_ms": r.elapsed.total_seconds() * 1000},
                    "message": "网络连通（HTTPS检测）"}
        except ImportError:
            try:
                import requests
                r = requests.get("https://www.google.com", timeout=5)
                return {"code": "SUCCESS", "data": {"connected": True,
                        "method": "https", "latency_ms": r.elapsed.total_seconds() * 1000},
                        "message": "网络连通（HTTPS检测）"}
            except ImportError:
                # 降级：用 socket 做 HTTP HEAD 请求
                try:
                    sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
                    sock.settimeout(5)
                    sock.connect(("www.google.com", 80))
                    sock.send(b"HEAD / HTTP/1.0\r\nHost: www.google.com\r\n\r\n")
                    sock.recv(1024)
                    sock.close()
                    return {"code": "SUCCESS", "data": {"connected": True,
                            "method": "http_socket"}, "message": "网络连通（HTTP原始检测）"}
                except Exception:
                    pass
            except Exception:
                pass
    
    return {"code": "SUCCESS", "data": {"connected": False},
            "message": "网络不可用（DNS和HTTPS均无法连接）",
            "next_actions": [
                {"tool": "validate_url", "desc": "验证URL格式",
                 "when": "需要确认URL格式是否正确"},
            ]}
