## 一、精简必要性分析

### 1.1 当前状态

系统注册工具总数 **148个**（经逐件普查确认），分属15个注册分类。其中用户关注的"文档及数据分析事务处理"和"日常工具"合计 **66个**，占总量 44.6%。

### 1.2 核心问题

| 问题 | 症状 | 后果 |
|------|------|------|
| **工具过多** | 148个工具，LLM每次需从大量候选中选择 | 选择错误率高，频繁补救调用 |
| **功能重叠** | 多个工具能做同一件事（如读CSV有2个、读Excel有2个） | LLM不知道该选哪个，浪费token |
| **粒度过细** | 一个逻辑概念拆成3-4个工具（如时间检查拆4个） | LLM看到场景后需要多步推理才能选对 |
| **低频冗余** | 存在 parse_properties、file_monitor 等几乎用不到的工具 | 占用工具位，增加LLM选择负担 |
| **命名不一致** | get_file_hash vs file_checksum，precise_replace_in_file vs edit_text_file | LLM无法从名字推断功能关系 |

### 1.3 精简目标

- 工具总数从 **148 → ~112**（减少 24%）
- 核心区域（文档+日常）从 **66 → 37**（减少 44%）
- 任意两个工具功能互斥，LLM看到一个场景只对应一个工具
- 保留100%的场景覆盖率，删除和合并只为消除冗余

> 🔍 **小沈深度审查 — P9 低风险：缺少 token 节省估算**
>
> **原方案问题**：精简目标只列出了工具数量减少（148→112），但没有估算对 LLM token 消耗的实际影响：
> - 工具数量减少不代表 token 线性减少——每个工具的 description 长度不同，合并后单个工具的 description 可能更长
> - 实际 token 节省来自：① 工具列表传入 API 时的描述总量减少 ② LLM 选择错误工具后的重试减少
> - 缺少估算，无法量化精简的实际收益
>
> **估算补充**：
>
> | 项目 | 精简前 | 精简后 | 节省 |
> |------|--------|--------|------|
> | 工具数量 | 148 | 112 | -36 (-24%) |
> | 估算描述总量（每工具均约80 token） | ~11,840 | ~8,960 | ~2,880 token (-24%) |
> | 实际节省（合并后描述更长，按×1.3系数） | — | ~11,648 | ~192 token（描述本身节省少） |
> | 选择错误率降低带来的间接节省 | 频繁选错需重试 | 选错率预计降50% | 每次对话省1-2轮重试 ≈ 500-1000 token |
> | **综合估算** | — | — | **每次对话约节省 500-1200 token** |
>
> **结论**：token 直接节省较少（工具描述总量只减~192 token），但间接节省（减少LLM选择错误和重试）更显著，每次对话约省500-1200 token。精简的核心价值不是省token，而是**提高LLM选择准确率**。

## 二、设计原则体系

> **两轨并行，架构优先**：本方案包含两条设计轨——
> - **轨道A：架构优化原则（P11-P17）**—— 先做。加固底层，提取Helper，修正假数据，真实化stub。让工具"变强壮"。
> - **轨道B：精简原则（P1-P10）**—— 后做。在稳固的底层之上，合并重复，删除冗余，统一入口。让工具"变少"。
>
> **依赖关系**：轨道A是轨道B的前提。例如 `database` 的事务stub——如果先精简，可能把"看起来没用的" `begin_transaction` 删了；但如果先真实化（P12），会发现它是必须保留的核心功能。

### 2.1 架构优化原则（轨道A —— 先行加固底层）

> 北京老陈提出——核心思想：不是"把多个工具捏成一个大工具"，而是"**工具之间通过共享底层、互相调用来形成能力网络**"。

| 编号 | 原则 | 解释 | 示例 |
|------|------|------|------|
| **P11** | **统一入口 + 内部分发** | 对外暴露一个入口工具，内部按条件分发到底层实现 | `read_document("file.xlsx")` → 内部调用 `_read_excel_base(use_pandas=False)`，与 `read_excel()` 共用同一套底层 |
| **P12** | **工具间互相调用（组合复用）** | 工具A可以调用工具B的函数来增强自身，形成跨分类能力网络 | `query_sql` 内部调 `check_db_exists` 自动验证；`screenshot` 返回 `next_actions: ["ocr"]` 引导跨分类流程 |
| **P13** | **分层架构：Tool层 vs Helper层** | 区分"对外暴露的Tool"和"内部复用的Helper"。**目录级隔离**：`tools/toolhelper/` 独立存放所有共享Helper，与各分类Tool实现物理分离，注册时不会误将Helper暴露给LLM | `get_file_hash` 和 `file_checksum` 都调用 `toolhelper/hash_helper.py` 的 `_calculate_hash`；`service_start`/`stop`/`list` 共用 `toolhelper/service_helper.py`；原有分散的 `_check_xxx` 等内部函数全部迁入 toolhelper |
| **P14** | **优雅降级（Graceful Degradation）** | 增强功能不可用时，自动回退到基础功能；检查类工具缺少依赖时应报错而非返回假数据 | `read_excel(use_pandas=True)` 缺 pandas → 降级 openpyxl；`get_mouse_position` 缺 pyautogui → 返回 ERR 而非假 `(0,0)` |
| **P15** | **返回值自解释（Self-Describing Result）** | 工具返回结果时主动告诉LLM"下一步可以做什么"，**支持跨分类建议** | `read_csv` → `next_actions: ["analyze_data"](document), ["filter_data"](document)`；跨分类链接形成能力网络 |
| **P16** | **幂等性（Idempotency）** | 同一操作多次执行，结果一致，不产生副作用累积 | `create_directory` 对已存在目录返回"已存在，无需创建"；`kill_process` 对已退出PID返回"无需终止" |
| **P17** | **参数正交（Orthogonal Parameters）** | 参数之间应独立，避免"传A就不能传B"的互斥关系 | `rename_file(mode="single"|"batch"|"regex")` 替代互斥的 path/directory 参数组 |

### 2.2 精简原则（轨道B —— 在稳固底层上做减法）

#### 2.2.1 用户要求的三条核心原则

| 编号 | 原则 | 解释 | 判断标准 |
|------|------|------|---------|
| **P1** | 重复类似的合并或删除 | 两个工具能做同一件事 → 只保留一个 | 对比两个工具的参数和返回值，功能交集 > 80% 则合并 |
| **P2** | 相似的进行功能整合 | 同一逻辑概念拆成多个工具 → 合并为一个，用参数区分 | 工具名包含同一关键词（如 read_*, time_*）的优先合并 |
| **P3** | 功能互斥，留下主要的 | 任意两个工具之间功能不应有交集 | LLM 看到任何一个业务场景，只应匹配一个工具 |

#### 2.2.2 补充精简原则

| 编号 | 原则 | 解释 | 示例 |
|------|------|------|------|
| **P4** | 单一职责 | 每个工具只做一件事，名字清晰表达意图 | `read_text_file` → `read_file`（不做写操作） |
| **P5** | 后缀自动路由 | 读写类工具按文件后缀自动选择底层实现 | `read_document("a.pdf")` 自动调 pdfplumber；`read_document("b.xlsx")` 自动调 openpyxl |
| **P6** | 最小完备性 | 用最少工具覆盖最多场景，不为低频场景单独设工具 | 删除 parse_properties（几乎用不到），保留 read_json（高频） |
| **P7** | 悲观默认值 | 参数有合理默认值，减少 LLM 需要传递的参数 | algorithm 默认 sha256，encoding 默认 auto_detect |
| **P8** | 自描述性 | 工具名直接表达功能，LLM 不需要读 description 就知道该用哪个 | `read_file` > `read_text_file`（删掉无意义的 text 限定词） |
| **P9** | 向下兼容 | 精简后保留别名映射，确保已有调用不中断 | tool_aliases.py 增加 `read_text_file → read_file` 映射 |
| **P10** | 先易后难分批执行 | 按影响范围和风险分批，每批独立可验证 | Time 先做（独立性强），File 最后（影响最广） |

### 2.3 执行顺序（关键）

```
第1步：轨道A — P11-P17 架构优化
    ├── 创建 toolhelper/ 目录（P13）→ 集中所有共享Helper，与Tool目录级隔离
    ├── 提取Helper层（P13）→ 消除代码重复，helper函数迁入 toolhelper/
    ├── 工具间调用加固（P12）→ 形成能力网络
    ├── 假数据修正（P14/P16）→ 消除误导LLM的Bug
    ├── next_actions 全覆盖（P15）→ 跨分类引导链
    └── 参数规范化（P17）→ 消除互斥参数
            ↓ 底层稳固后
第2步：轨道B — P1-P10 精简
    ├── 重复合并（P1）→ 消除功能重叠
    ├── 相似整合（P2）→ 多合一用参数路由
    ├── 功能互斥（P3）→ 保证一对一映射
    └── ... P4-P10 持续优化
```

> **为什么架构优化必须在精简之前？**
>
> | 反面案例 | 如果先精简 | 先架构优化 |
> |---------|-----------|-----------|
> | database 事务stub | 可能把"看起来没用"的 `begin_transaction` 删了 | 先真实化（P12），发现是核心功能，保留 |
> | `list_windows` 双重注册 | 可能只删一个保留另一个 | 先解决冲突（P13+P12），再决定是否合并 |
> | `get_mouse_position` 假数据 | 精简看不出来问题 | 先修正假数据（P14），再评估是否需要独立存在 |

### 2.4 原则层级关系

```
轨道A：架构优化（底层加固）            轨道B：精简（上层减法）
    P13 分层架构                              P1 重复合并

    P14 优雅降级                              P3 功能互斥

    P15 返回值自解释                           P7 悲观默认
    P17 参数正交                               P8 自描述
                                               P9 向下兼容
```

## 四、问题诊断（5类问题，逐工具标注）

### 4.1 完全重复（P1级——必须解决）

#### D1: file_checksum vs get_file_hash

| 维度 | file_checksum | get_file_hash |
|------|-------------|--------------|
| 位置 | `file_tools.py:1640` | `file_tools.py:1568` |
| 参数 | file_path, algorithm(SHA256), verify_hash, chunk_size, timeout | file_path, algorithm(SHA256), verify_against, timeout |
| 实现 | 调用 `file_checksum_impl` | 调用 `_get_file_hash` helper |
| 底层 | 同一套哈希计算逻辑 | 另一套哈希计算逻辑 |
| 结论 | **两个工具在同一类里，提供完全相同的功能，LLM 看到两个名字不知道该用哪个** |

**处理**: 保留 `get_file_hash`（参数更简洁），删除 `file_checksum`，删除 `file_checksum.py`。

#### D2: list_windows 双重注册

| 维度 | desktop/list_windows | gui/list_windows |
|------|---------------------|------------------|
| 来源 | desktop_register.py | gui_register.py |
| 功能 | 枚举 Windows 窗口 | 枚举 GUI 窗口 |
| 结论 | **同一工具名注册两次，分类不同但功能重叠** |

**⚠️ 运行时Bug分析**：由于 ToolRegistry 允许重复注册（后注册覆盖前注册），而 `_CATEGORY_MODULES` 的遍历顺序是 desktop 在先、desktop_gui 在后，所以**最终生效的是 gui_register 中的 list_windows 实现**。desktop_register 中的 list_windows 实际上被覆盖了，是一个**已有的运行时Bug**。

**处理**: 删除 gui_register.py 中的 list_windows 注册。desktop_register 中的 list_windows 保留不变，因为它是独立的窗口枚举功能，不依赖 GUI 自动化。如果 GUI 自动化场景需要专用窗口列表，可以新增 `list_gui_windows` 工具（带额外窗口过滤能力）。

### 4.2 功能重叠（P1级——LLM选择困难）

#### O1: read_csv_basic vs read_csv_dataframe

| 维度 | read_csv_basic | read_csv_dataframe |
|------|---------------|-------------------|
| 分类 | data_format | data_analysis |
| 底层 | csv stdlib（零依赖） | pandas（需安装） |

**处理**: 合并为 `rename_file(path=None, new_name=None, directory=None, pattern=None, replacement=None, preview=False)`。传 path+new_name→单文件；传 directory+pattern→批量。

| 维度 | search_files | grep_file_content |
|------|-------------|------------------|

| 维度 | list_directory | get_directory_tree |

**处理**: 合并为 `list_directory(path, format="list"|"tree", recursive=False)`。format="list" 返回平铺列表，format="tree" 返回 JSON 树。

#### O7: get_file_info vs file_statistics

| 维度 | get_file_info | file_statistics |

> - 实际场景中 LLM 经常混淆：传了 `pattern` 却也传了 `search_content`，或忘记传 `search_content` 却期望搜索内容

> - `grep_file_content`：按文件内容搜索（grep模式），参数：path, pattern, context_lines, ignore_case, show_line_numbers, output_mode
> - 两者语义清晰互斥，LLM看到"找文件"选search_files，看到"找内容"选grep_file_content，零歧义

```python
        return time_add(start=time_value, delta=delta, unit=unit)
    else:
        return {"code": "ERR_INVALID_ACTION", "message": f"不支持的操作: {action}"}

def time_diff(start=None, end=None):
    """计算时间差值 - 小沈 2026-05-17"""
    return _time_diff_impl(start=start, end=end)

def time_compare(time1=None, time2=None):
    """比较时间先后 - 小沈 2026-05-17"""
    return _time_compare_impl(time1=time1, time2=time2)

def check_date(date=None, check_type="weekend", n=1):
    """检查日期属性 - 小沈 2026-05-17"""
    if check_type == "weekend":
        return time_is_weekend(date=date)
    elif check_type == "holiday":
        return time_is_holiday(date=date)
    elif check_type == "workday":
        return time_is_workday(date=date)
    elif check_type == "next_workday":
        return time_next_n_workday(n=n, start_date=date)
    else:
        return {"code": "ERR_INVALID_CHECK_TYPE", "message": f"不支持的检查类型: {check_type}"}
```

### 9.2 功能验收

- [ ] 场景覆盖表（第七章）中的所有场景都有对应工具
- [ ] 合并后的工具通过 action/check_type/direction 参数能正确路由到原实现
- [ ] 删除的工具对应的场景被保留工具覆盖

### 9.3 兼容性验收

- [ ] `tool_aliases.py` 中包含所有旧工具名→新工具名的映射

- [ ] 现有单元测试通过（或仅因工具名变更而需要更新测试中的工具名）

### 9.4 互斥性验收

- [ ] 任意两个保留工具不存在功能交集
- [ ] 对每个常见用户场景，唯一映射到一个工具

> 🔍 **小沈深度审查 — P8 低风险：验收标准互斥性过于理想**
>
> **原方案问题**：9.4节"任意两个保留工具不存在功能交集"过于理想化：
> - 实际中 `read_document("file.xlsx")` 和 `read_excel("file.xlsx")` 存在功能交集（都能读Excel），只是返回格式不同（简洁格式 vs DataFrame格式）
> - `search_files` 和 `grep_file_content` 理论上都能"搜索文件相关内容"，只是语义不同（按名 vs 按内容）
> - 严格要求"零交集"会导致过度合并或过度拆分，不切实际
>
> **修正方案**：将互斥性验收标准从"零交集"调整为"可接受的语义区分度"：
> - 核心验收标准改为：**对每个常见用户场景，存在唯一最优工具**（不要求工具间完全没有交集）
> - 允许功能有少量交集，但交集场景必须有明确的优先选择规则
> - 示例：读Excel时，`read_document` 优先（简洁格式），需DataFrame时选 `read_excel`——虽然两者都能读，但选择规则清晰

## 十一、架构设计优化（北京老陈提出——深化设计模式）

> **核心思想**：不是"把多个工具捏成一个大工具"，而是"**工具之间通过共享底层、互相调用来形成能力网络**"。

### 11.1 Helper层提取（解决 D1 根本问题）

**问题**：`file_checksum` vs `get_file_hash` 完全重复，但简单合并会丢失一个接口。

**更好方案**：提取 Helper 层，两个工具保留独立接口，但共用底层实现。

```python
# helper层（工具间共享，不暴露给LLM）
def _calculate_hash(file_path: str, algorithm: str = "sha256") -> str:
    """底层哈希计算 - 所有哈希工具共用 - 小沈 2026-05-17"""
    import hashlib
    h = hashlib.new(algorithm)
    with open(file_path, 'rb') as f:
        for chunk in iter(lambda: f.read(8192), b''):
            h.update(chunk)
    return h.hexdigest()

# tool层（LLM可调用的工具）
def get_file_hash(file_path: str, algorithm: str = "sha256") -> dict:
    """计算文件哈希值 - 小沈 2026-05-17"""
    hash_value = _calculate_hash(file_path, algorithm)
    return {"code": "OK", "data": {"hash": hash_value, "algorithm": algorithm}}

def verify_file_hash(file_path: str, expected_hash: str, algorithm: str = "sha256") -> dict:
    """验证文件哈希是否匹配 - 小沈 2026-05-17"""
    actual_hash = _calculate_hash(file_path, algorithm)
    match = actual_hash.lower() == expected_hash.lower()
    return {
        "code": "OK",
        "data": {
            "match": match,
            "expected": expected_hash,
            "actual": actual_hash,
            "algorithm": algorithm
        }
    }
```

**好处**：
- 不丢失任何接口（get_file_hash 和 verify_file_hash 都保留）
- 底层逻辑只维护一份（_calculate_hash）
- LLM 看到两个独立工具，语义清晰

### 11.4 幂等性统一处理（P16）

**目标**：让工具多次执行同一操作不产生副作用，LLM 不需要先"检查是否存在"再操作。

**改造清单**：

| 工具 | 当前行为 | 幂等改造后 |
|------|---------|-----------|
| `create_directory` | 目录已存在时报错 | 返回 `"已存在，无需创建"` |
| `delete_file` | 文件不存在时报错 | 返回 `"不存在，无需删除"` |
| `write_text_file` | 直接覆盖 | 增加 `mode="append"\|"overwrite"\|"skip"`，默认 overwrite |
| `move_file` | 目标存在时报错 | 增加 `on_exist="overwrite"\|"skip"\|"error"`，默认 error |
| `copy_file` | 目标存在时报错 | 同上 |

**示例实现**：

```python
def create_directory(path: str, exist_ok: bool = True) -> dict:
    """创建目录（幂等）- 小沈 2026-05-17"""
    import os
    if os.path.exists(path):
        if exist_ok:
            return {"code": "OK", "message": "目录已存在，无需创建", "data": {"path": path}}
        else:
            return {"code": "ERR_EXISTS", "message": f"目录已存在: {path}"}
    os.makedirs(path)
    return {"code": "OK", "message": "目录创建成功", "data": {"path": path}}
```

## 附录 A：别名映射全表

> 🔍 **小沈深度审查 — P6 中风险：alias 运行时机制未设计**
>
> **原方案问题**：别名映射表只定义了"旧名→新名"的静态映射关系，但缺少运行时解析机制的设计：
> - LLM调用旧工具名时，谁负责查找别名并路由到新工具？
> - 路由发生在注册阶段（注册时同时注册旧名和新名）还是执行阶段（执行时查找别名表）？
> - 别名路由的性能开销是多少？是否影响Agent执行延迟？
> - 别名冲突（两个旧名映射到同一个新名但参数不同）如何处理？
>
> **修正方案**：补充别名运行时解析机制设计：
>
> | 机制 | 说明 |
> |------|------|
> | **解析时机** | 执行阶段：Agent调用工具时，先在 `tool_aliases` 中查找，找到则替换为新工具名再执行 |
> | **解析位置** | `ToolRegistry.get_tool(name)` 方法中：先查 `self._tools[name]`，未找到则查 `self._aliases[name]` |
> | **参数转换** | 别名映射只替换工具名，不转换参数（因为合并后工具的 action 参数已对应旧工具的语义） |
> | **日志记录** | 别名命中时 log.warning 记录：`"工具别名: {old_name} → {new_name}，建议更新调用方"` |
> | **冲突处理** | 同一新工具名的多个旧别名，通过不同 action 值区分，不存在真正的参数冲突 |

#### 13.2.3 P16 幂等性 + action 统一

`set_env` 合并 `delete_env`，新增 `action` 参数；同时补充幂等性 `exist_ok`：

```python
def set_env(
    name: str,
    value: Optional[str] = None,     # action="delete" 时不需要
    scope: str = "process",
    append_mode: bool = False,
    exist_ok: bool = True,           # 幂等：值已相同则返回"已存在"
    action: str = "set",             # "set" | "delete" — 合入 delete_env
) -> dict:
    """设置或删除环境变量（幂等增强版）- 小沈 2026-05-17"""
    if action == "delete":
        # 原 delete_env 逻辑
        if name in os.environ:
            del os.environ[name]
            return {"code": "SUCCESS", "data": {"name": name, "deleted": True},
                    "message": f"已删除: {name}"}
        return {"code": "SUCCESS", "data": {"name": name, "deleted": False},
                "message": f"不存在: {name}"}

    # 原有 set 逻辑 + exist_ok 幂等增强
    if exist_ok and value is not None:
        existing = os.environ.get(name)
        if existing == value:
            return {"code": "SUCCESS", "data": {"name": name, "value": value},
                    "message": f"环境变量已存在且值相同: {name}={value[:50]}..."}
    # ... 原有 set 逻辑
```

> **用户场景**：
> - `set_env(name="MY_VAR", value="abc")` → 设置（原 behavior）
> - `set_env(name="MY_VAR", action="delete")` → 删除（原 delete_env）

### 13.3 environment(check)——4个检查工具取消LLM暴露，下沉为Helper

#### 13.3.1 取消理由

北京老陈指出：这4个工具本质上都是"检查型"工具，本应由执行方自己负责验证，而非让LLM先检查再执行。

| 工具 | 现状 | 取消理由 |
|------|------|---------|
| `check_python_available` | 检查Python是否可用 | `execute_python` 已内置 `FileNotFoundError` 捕获，环境不可用时自动返回 `ERR_EXEC_PYTHON_NOT_FOUND`。LLM无需提前检查 |
| `validate_code_safety` | 检查代码安全性 | 按13.9设计，`execute_python` 的 `safety_check=True` 会在执行前自动调用。LLM无需单独检查 |
| `check_node_available` | 检查Node.js是否可用 | `execute_javascript` 已内置 `FileNotFoundError` 捕获，环境不可用时自动返回 `ERR_EXEC_NODE_NOT_FOUND` |
| `check_module_available` | 检查Python模块是否已安装 | 等价于 `execute_python("import pandas; print(pandas.__version__)")`，LLM已可以通过 `execute_python` 实现 |

#### 13.3.2 Helper化方案

4个函数从LLM工具 → `toolhelper/exec_helper.py`（内部Helper，不注册给LLM）：

```python
# toolhelper/exec_helper.py - 小沈 2026-05-17
"""代码执行辅助函数（不暴露给LLM）- 代码执行内部使用"""

import importlib
import subprocess
import sys
import re

DANGEROUS_PATTERNS = [
    (r"os\.system\s*\(", "系统调用(os.system)"),
    (r"eval\s*\(", "动态执行(eval)"),
    (r"exec\s*\(", "动态执行(exec)"),
    (r"subprocess\.(call|run|Popen)\s*\(", "子进程调用"),
]

def _check_python_available() -> bool:
    """检查Python是否可用（Helper，不暴露给LLM）"""
    try:
        # execute_python 已通过 try/except FileNotFoundError 覆盖此场景
        return sys.executable is not None
    except Exception:
        return False

def _validate_code_safety(code: str) -> list[str]:
    """验证代码安全性（Helper，不暴露给LLM）"""
    warnings = []
    for pattern, desc in DANGEROUS_PATTERNS:
        if re.search(pattern, code):
            warnings.append(desc)
    return warnings

def _check_node_available() -> bool:
    """检查Node.js是否可用（Helper，不暴露给LLM）"""
    try:
        result = subprocess.run(["node", "--version"],
                                capture_output=True, timeout=5)
        return result.returncode == 0
    except (FileNotFoundError, subprocess.TimeoutExpired):
        return False

def _check_module_available(module_name: str) -> tuple[bool, str]:
    """检查Python模块是否可用（Helper，不暴露给LLM）"""
    try:
        mod = importlib.import_module(module_name)
        return True, getattr(mod, "__version__", "unknown")
    except ImportError:
        return False, ""
```

```python
    force: bool = False,            # action="stop" 时强制
    wait_for_started: bool = False, # action="start"/"restart" 时等待
    wait_for_stopped: bool = False, # action="stop" 时等待
    timeout: int = 30,
) -> dict:
    """Windows/Linux 服务统一管理入口 - 小沈 2026-05-17"""
    if action == "list":
        return _service_list_logic(service_name or None, state)
    elif action == "start":
        return _service_start_logic(service_name, wait_for_started, timeout)
    elif action == "stop":
        return _service_stop_logic(service_name, force, wait_for_stopped, timeout)
    elif action == "restart":
        # 先stop再start
        stop_result = _service_stop_logic(service_name, force, wait_for_stopped, timeout)
        if stop_result["code"] == "SUCCESS":
            return _service_start_logic(service_name, wait_for_started, timeout)
        return stop_result
```

#### 13.12.2 编排工具：帮LLM做批量/组合操作

| 工具 | 功能 | 为什么需要 | 不增加会怎样 |
|------|------|-----------|------------|
| `batch_process` | 对匹配到的文件列表执行同一个操作 | "把所有.log转成.txt" → LLM需要先 `list_directory`→逐个 `rename_file`→多轮交互 | 用户觉得慢，LLM多次推理 |
| `pipeline` | 定义"先A→再B→如果失败则C"的工具执行链 | 多个工具串行调用时 LLM 每步都要推理，pipeline一行定义即可 | 多步操作需要N轮ReAct循环 |

```python
# batch_process 实现 - 小沈 2026-05-17
def batch_process(
    source_pattern: str,    # "*.txt" 或 "logs/*.log"
    action: str,            # "rename" | "delete" | "copy" | "compress" | "chmod"
    target_pattern: Optional[str] = None,  # rename/copy 的目标模式
    target_dir: Optional[str] = None,      # copy 的目标目录
    dry_run: bool = True,   # 默认预览，保护用户数据
    max_files: int = 500,   # 安全上限
) -> dict:
    """批量处理文件 - 小沈 2026-05-17
    
    对匹配 glob 模式的所有文件执行同一操作。
    
    示例:
    - "把所有.txt改成.md": batch_process("*.txt", "rename", target_pattern="*.md")
    - "清空所有.log": batch_process("logs/*.log", "delete", dry_run=False)
    """
    import glob as glob_module
    files = glob_module.glob(source_pattern)[:max_files]
    
    if not files:
        return {"code": "ERR_NO_MATCH", "message": f"没有匹配到文件: {source_pattern}"}
    
    plan = {
        "action": action, "file_count": len(files),
        "files": files[:20],  # 预览前20个
        "total_files": len(files)
    }
    
    if dry_run:
        return {"code": "SUCCESS", "data": plan,
                "message": f"【预览模式】将 {action} {len(files)} 个文件",
                "next_actions": [
                    {"tool": "batch_process", "desc": "确认执行",
                     "params": {"dry_run": False, "source_pattern": source_pattern, "action": action}}
                ]}
    
    # ... 实际执行 action ...
    results = {"success": 0, "failed": 0, "errors": []}
    for f in files:
        try:
            if action == "rename":
                new_name = f[:-len(f.split('.')[-1])-1] + "." + target_pattern.split('.')[-1]
                os.rename(f, new_name)
            elif action == "delete":
                os.remove(f)
            elif action == "copy":
                dest = os.path.join(target_dir, os.path.basename(f))
                shutil.copy2(f, dest)
            results["success"] += 1
        except Exception as e:
            results["failed"] += 1
            results["errors"].append({"file": f, "error": str(e)})
    
    return {"code": "SUCCESS", "data": results,
            "message": f"处理完成: {results['success']} 成功, {results['failed']} 失败"}

# pipeline 实现 - 小沈 2026-05-17
def pipeline(
    steps: list,            # [{"tool": "read_csv", "params": {...}}, {"tool": "analyze_data", ...}]
    stop_on_error: bool = True,
    timeout_per_step: int = 60,
) -> dict:
    """定义工具执行管道 - 小沈 2026-05-17
    
    将多个工具按顺序编排执行，前一步的输出自动成为后一步的输入。
    
    示例:
    pipeline([
        {"tool": "read_csv", "params": {"file_path": "data.csv"}},
        {"tool": "filter_data", "params": {"condition": "age > 30"}},
        {"tool": "generate_chart", "params": {"chart_type": "bar"}}
    ])
    """
    from app.services.tools.registry import tool_registry
    
    context = {}  # 步骤间数据传递
    results = []
    
    for i, step in enumerate(steps):
        tool_name = step["tool"]
        tool = tool_registry.get_tool(tool_name)
        if not tool:
            return {"code": "ERR_TOOL_NOT_FOUND", "message": f"步骤{i+1}: 工具 {tool_name} 不存在"}
        
        # 合并上下文（前一步输出作为当前步骤的输入上下文）
        params = {**step.get("params", {}), "_context": context}
        
        try:
            result = tool.func(**params)
            results.append({"step": i+1, "tool": tool_name, "result": result})
            if result.get("code") != "SUCCESS" and stop_on_error:
                return {"code": "ERR_PIPELINE_STOPPED",
                        "message": f"步骤{i+1} ({tool_name}) 失败，管道停止",
                        "data": {"step": i+1, "error": result, "results": results}}
            # 将结果存入上下文供下一步使用
            context[f"step_{i+1}"] = result.get("data")
        except Exception as e:
            return {"code": "ERR_PIPELINE_FAILED",
                    "message": f"步骤{i+1} ({tool_name}) 执行异常: {str(e)}",
                    "data": {"results": results}}
    
    return {"code": "SUCCESS", "data": {"steps": len(steps), "results": results},
            "message": f"管道执行完成: {len(steps)} 个步骤"}
```

#### 13.14.4 实施要点

1. **toolhelper 不注册给LLM**：`toolhelper/` 目录下的函数不被 `ensure_tools_registered()` 扫描，只在Tool层通过 `import` 调用
2. **单向依赖**：Tool层 → toolhelper层。toolhelper 不反向依赖任何 Tool 实现
3. **迁移顺序**：先创建 `toolhelper/` 目录结构 → 逐个提取Helper → 更新Tool层import → 删除旧实现 → 验证测试
4. **不新增外部依赖**：toolhelper 只使用 Python 标准库 + 项目已有的第三方库（如 `psutil`、`win32gui`），不引入新依赖
