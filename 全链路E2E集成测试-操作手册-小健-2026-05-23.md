## 一、测试定义

### 1.1 测试名称

**全链路端到端集成测试**（Full-Link End-to-End Integration Test）

### 1.2 为什么叫这个名

| 关键词 | 含义 | 在这个测试中的体现 |
|--------|------|-------------------|
| 全链路 | 覆盖系统所有组件 | 前端API→后端→Agent→LLM→工具执行→数据库→SSE响应 |
| 端到端 | 从用户发消息到收到完整结果 | POST /chat → SSE事件流 → final事件 |
| 集成 | 验证各组件对接是否正确 | API路由、SSE封装、Agent循环、LLM调用、工具执行、数据库写入 |

### 1.5 与现有测试的区别

| 测试类型 | 测试什么 | 用什么驱动 | 能发现什么问题 |
|---------|---------|-----------|--------------|
| 单元测试（现有~200个） | 单个工具函数 | Mock/Patch | 函数级bug |
| MessageBuilder测试（43个） | Prompt构建逻辑 | Mock | Prompt构建bug |
| **全链路E2E（本方案）** | **Agent行为+系统集成** | **真实后端（读取config.yaml的LLM配置）+ 真实SQLite** | **死循环、工具选错、参数传错、会话污染、日志异常、数据库异常** |

### 1.6 测试目标

**唯一目标**：用真实LLM（后端从config.yaml读取配置）模拟真实用户操作，把系统中隐藏的问题炸出来。

**具体来说**：
1. Agent在真实LLM下能否正确完成用户请求
2. Agent在异常情况下（LLM返回垃圾/空内容/错误工具）能否容错
3. 系统各组件的串联是否完整（API→Agent→LLM→工具→DB）
4. 日志和数据库记录是否完整准确

### 1.7 核心测试原则（铁律）

**铁律1：测试的目的是发现问题，不是跑脚本。**

**铁律2：严禁批量跑测试用例。一次只能跑一个测试用例。**

| 允许 | 禁止 |
|------|------|
| ✅ 写一个脚本 → 跑一个 → 观察 → 检查 | ❌ 一次写完所有脚本 |
| ✅ 一个用例跑通后再写下一个 | ❌ 一次批量跑多个用例 |
| ✅ 每次跑完仔细观察系统运行过程 | ❌ 只看pass/fail就过 |
| ✅ 失败了停下来分析修复 | ❌ 失败了跳过继续下一个 |

**铁律3：所有测试统一基于真实后端运行，禁止Mock：**

| 环节 | 要求 | 禁止 |
|------|------|------|
| **后端** | 真实uvicorn启动，从config.yaml读取LLM配置 | ❌ 不启动后端、Mock后端 |
| **LLM** | 真实调用config.yaml中配置的provider/model | ❌ Mock LLM返回、假响应 |
| **工具** | 真实执行文件操作、命令、网络请求等 | ❌ Mock工具返回值 |
| **数据库** | 真实读写SQLite（~/.omniagent/chat_history.db） | ❌ Mock数据库、内存数据库 |
| **SSE** | 真实解析后端返回的SSE事件流 | ❌ 假造SSE事件 |

**一句话：测试输入就是用户会说的话，测试过程就是真实系统的完整运行。**

## 三、完整测试流程

每个测试用例严格按照 **测试前→测试中→测试后** 三阶段执行。
**测试前准备不到位，后续所有验证都不可能做好。**

### 3.1 测试前：准备工作（必须全部完成才能开始测试）

**准备工作分两大块：清除旧脚本 + 逐个编写新脚本。**

#### A. 清除旧测试脚本

```
操作:
  · 删除 backend/tests/ 下所有旧的 E2E 测试文件
  · 删除 backend/tests/e2e_artifacts/ 下所有旧的测试产物
  · 确认 conftest.py 中的 fixtures 是最新的

原因:
  · 旧脚本可能基于旧的系统行为,与当前代码不匹配
  · 旧脚本的断言可能与新的验证标准冲突
  · 清除后从零开始,确保每个脚本都是正确且最新的
```

#### B. 逐个编写测试脚本（铁律：一次只写一个）

```
⚠️ 禁止一次性编写所有脚本！

测试的目的 = 发现问题,不是跑10000遍
  · 一次性写完 → 无法验证每个脚本是否正确
  · 一次性写完 → 错误会累积,难以定位
  · 一次性写完 → 执行人员无法逐步推进
  · 批量测试 = 自欺欺人,发现不了真正的问题

正确做法: 一个用例一个脚本,写完跑通再写下一个

  第1步: 看测试用例 E2E-P0-01 "详细介绍自己"
         → 这个用例要测什么? (API→SSE通路)
         → 要发什么指令? ("详细介绍一下你自己，你能做什么")
         → 要验证什么? (收到final、无error、DB有记录)
         → 编写 test_e2e_p0_01_introduce_self()
         → 运行这个脚本
         → 观察系统运行过程,检查结果
         → 通过 → 进入第2步
         → 失败 → 分析错误、修复、重跑、通过后进入第2步

  第2步: 看测试用例 E2E-P0-02 "创建文件"
         → 编写 test_e2e_p0_02_create_file()
         → 运行、观察、检查
         → 通过 → 进入第3步

  ...逐个推进...

编写顺序: P0 → P1 → P2 → ... → P10
  · 先写最简单的（P0 hello）
  · 确认基础流程通了再写复杂的
  · 每写完一个,立即运行观察系统行为
```

#### C. 系统就绪检查（6项全部通过才能开始测试）

```
检查项 0: 重启后端服务（每次测试前必须执行）
  · 先杀死旧的 uvicorn 进程
    Windows: taskkill /F /IM python.exe /FI "WINDOWTITLE eq uvicorn*"
    或找到占用 8000 端口的进程杀掉
  · 重新启动后端
    cd backend && python -m uvicorn app.main:app --reload
  · 等待启动完成（约3-5秒）
  · 检查后端是否正常:
    curl http://localhost:8000/docs 返回 200
  · 失败处理: 检查端口占用/代码错误/配置问题,修复后重新启动
  · ⚠️ 即使使用 --reload,也必须确认后端正常响应才能继续

检查项 1: 后端服务已确认正常
  · 接检查项0的结果,确认返回200

检查项 2: LLM 配置
  · config.yaml 中 ai.provider 是否已配置有效 provider
  · 检查方法: 后端启动日志无 LLM 连接错误
  · 失败处理: 检查 config.yaml 的 ai 配置段

检查项 3: 安全开关
  · config.yaml 中 security.enabled 是否为 false（纯功能验证模式）
  · 检查方法: grep "enabled" config/config.yaml
  · 失败处理: 修改 config.yaml 设 security.enabled: false

检查项 4: 数据库
  · ~/.omniagent/chat_history.db 是否存在且可写
  · 检查方法: sqlite3 查询 SELECT count(*) FROM sessions
  · 失败处理: 检查目录权限

检查项 5: 日志目录
  · backend/logs/ 和 backend/logs/prompt-logs/ 是否可写
  · 检查方法: 尝试写入测试文件
  · 失败处理: 创建目录或修复权限

检查项 6: 测试数据完备性（铁律：每个用例跑前必须自检）
  · 测试用的目录和文件是否就绪
  · 检查方法: Test-Path 检查关键路径
  · 失败处理: 创建测试目录和文件

  ⚠️ 测试数据完备性铁律（经验总结 2026-06-15）:

  6.1 基础测试数据必须与手册严格一致:
    · E:\test_dir\test.txt 内容必须是 "hello world"（F03搜索hello、F04编辑hello改你好依赖此内容）
    · E:\test_dir\test_old.txt 内容必须是 "old content"（F07压缩测试依赖此内容）
    · E:\test_dir\backup\ 必须是空目录（F06复制测试需验证backup下只有新副本，旧文件干扰判断）
    · E:\test_dir\data.csv 内容必须包含2026年数据（DM06筛选2026年记录依赖此数据）
    · E:\test_dir\test.docx 必须是可读Word文档（DM01读取文档依赖此文件）
    · E:\test_dir\empty_dir\ 必须是空目录（ERR-10空目录搜索依赖此目录）

  6.2 每个测试用例脚本必须在开头自检前置数据:
    · 脚本中用 assert TEST_FILE.exists() 检查依赖文件
    · 脚本中用 assert "hello" in content 检查文件内容符合预期
    · 如果用例会修改公共文件（如F04编辑test.txt），必须在finally中恢复原始内容
    · 如果用例创建新文件，必须在finally中清理

  6.3 公共文件被修改后的恢复规则:
    · test.txt 被编辑后 → 恢复为 "hello world"
    · backup/ 目录 → 清空为空目录
    · E盘根目录的临时文件 → 删除（如 e2e_test_p0.txt）

  6.4 数据完备性检查清单（每个用例跑前核对）:
    | 检查项 | 验证方法 | 不通过的后果 |
    |--------|---------|-------------|
    | test.txt存在且内容正确 | read + assert "hello world" | F02/F03/F04/P0-03全部失败 |
    | test_old.txt存在且内容正确 | read + assert "old content" | F07失败 |
    | backup/为空目录 | listdir + assert len==0 | F06误判 |
    | data.csv存在且含2026数据 | read + assert "2026" | DM05/DM06失败 |
    | test.docx存在 | exists | DM01失败 |
    | empty_dir/为空目录 | listdir + assert len==0 | ERR-10误判 |

以上 A + B + C 全部完成 → 进入测试中阶段
任一项未完成 → 不开始测试
```

### 3.2 测试中：执行与验证

```
步骤 1: 记录测试起始状态
  · 记录当前时间 T0
  · 记录数据库当前会话数（base count）
  · 记录日志文件当前大小

步骤 2: 发送用户请求
  · 用 httpx 发送 POST /chat
  · 请求体: {"messages": [{"role": "user", "content": "用户输入"}]}
  · 设置 stream=True 以接收 SSE
  · 记录请求时间 T1

步骤 3: 接收并解析 SSE 事件流
  · 逐行读取 SSE 响应
  · 按 event: xxx 和 data: xxx 解析每个事件
  · 记录每个事件的:
    - 事件类型 (start/thought/action/observation/chunk/final/error)
    - 时间戳
    - step 编号
    - data 内容 (工具名、参数、结果等)
  · 持续接收直到 final 事件或超时
  · 记录接收完成时间 T2

步骤 4: 验证事件流正确性
   · [SHOULD] 收到 final/error/中断任一 → 流已结束(三方式均有效，不阻断)
   · [SHOULD WARN] 无 error 事件为佳，有则记录不阻塞(has_error降级)
  · [必须] 事件总数 < 50 → 无死循环
  · [检查] final 事件包含结果内容
  · [检查] 事件顺序正确 (start→thought→action→observation→...→final)

  ⚠️ 发现 FAIL → 执行步骤 4.5 错误处理

步骤 4.5: 错误处理（发现 FAIL 时的必须流程）
  错误分两类:
    A类(运行时): LLM超时/工具崩溃/死循环/SSE断流 → 查日志traceback
    B类(数据不一致): session缺失/messages丢失/steps不匹配 → 对比SSE vs DB

  处理流程:
    ① 分类 → A类还是B类
    ② 定位 → 查后端日志 + SSE事件 + prompt-logs,定位到文件:行号
    ③ 分析 → 是代码bug还是测试用例问题
    ④ 修复 → 遵守10大规范 (SRP/DRY/KISS/YAGNI,禁止backward compat)
    ⑤ 复测 → 重跑该用例 + 跑关联测试确认无回归
    ⑥ 记录 → 写入测试报告

  严重等级:
    P0(致命): 系统崩溃/死循环 → 立即停止,修复后从P0重跑
    P1(严重): 功能异常但不崩溃 → 修复后重跑该用例
    P2(一般): 非核心异常 → 记录,批量修复
    P3(警告): 性能偏差 → 记录,不阻塞

  禁止事项:
    ❌ 看到FAIL就跳过   ❌ 只改测试不改代码
    ❌ 修复时不跑单元测试  ❌ 不记录直接修复
```

### 3.4 验证标准总表

| 验证点 | 权重 | 通过标准 | 失败标准 | 验证来源 |
|--------|------|---------|---------|---------|
| 流结束 | SHOULD | 收到 final/error/中断任一 | 超时无任何事件(null消息) | SSE事件流 |
| 错误事件 | SHOULD | 无 error 事件 | 有 error 事件(记录不阻塞，v1.8从MUST降级) | SSE事件流 |

| 事件总数 | MUST | 1~50（无死循环） | >= 50 或 0 | SSE事件流 |
| LLM调用次数 | SHOULD | 1~10次 | >= 10（可能死循环） | prompt-logs |
| DB session记录 | MUST | 存在且状态completed | 缺失或状态异常 | SQLite DB |
| DB messages记录 | MUST | user+assistant都保存 | 缺少任一 | SQLite DB |

| 回复语义 | SHOULD | 长度>10字，语义与用户指令相关 | 回复为空或答非所问 | SSE final事件 |

### 3.7 柔性验证原则（最优美的验证方法）

真实LLM的输出具有**非确定性**（同一输入每次回复不同），因此验证方法必须从"刚性精确匹配"升级为**柔性语义验证**。

| 原则 | 刚性验证（禁止） | 柔性验证（最优美） |
|------|----------------|-------------------|
| **工具名验证** | `assert tool_name == "write_text_file"` | `assert any("write" in t for t in tool_names)` |
| **事件顺序** | `assert events[1]["type"] == "thought"` | `assert "thought" in event_types` |
| **步骤数量** | `assert steps == 5` | `assert 1 <= steps <= 20` |
| **回复内容** | `assert "hello" in response` | `assert len(response) > 10`（语义完整即可）|
| **文件验证** | `assert content == "hello world"` | 文件存在 + 内容非空 + 内容与指令相关 |
| **LLM调用** | `assert llm_calls == 3` | `assert 1 <= llm_calls <= 8`（无死循环即可）|

**核心思想**：
- **语义为王**：验证"结果是否符合用户意图"，不验证"是否走了预期路径"
- **自适应断言**：使用范围、包含、模式匹配代替精确值
- **权重分层**：核心属性（文件存在/DB有记录）严格验证，行为属性（选了哪个工具）宽松验证
- **异常分级处理**：死循环/崩溃必须零容忍(MUST)，error事件记录不阻塞(SHOULD WARN)，工具选择差异保持弹性(MAY)

**事件流验证的优美模式**：
```python
# 最优美的验证方式：分层验证
# L1 - 强制层（不通过就是系统缺陷）
end_type = assert_stream_ended(result)   # 返回final/error/中断，不阻断
assert result["total_steps"] < 50        # 不能死循环(MUST)

# L2 - 语义层（验证结果质量）
assert len(result["response_text"]) > 10 # 回复不能为空或敷衍(MUST)
if result["has_error"]:
    print(f"  [WARN] 有error事件(SHOULD)，流结束: {end_type}")

# L3 - 检查层（记录但不阻塞）
if file_path_exists:
    assert file_exists, f"文件应已创建: {file_path}"
```

**验证权重定义**：

| 权重 | 含义 | 失败处理 | 示例 |
|------|------|---------|------|
| **MUST** | 必须满足 | 测试失败，停止 | 无死循环、DB有记录、文件存在 |
| **SHOULD** | 应该满足 | 记录警告，不失败 | 工具名合理、步骤数合理 |
| **MAY** | 建议检查 | 仅记录日志 | LLM调用了A工具而非B工具 |

## 五、实现方式--测试和测试脚本编写指南

### 5.1 测试框架选择

使用 **pytest**，与现有测试体系一致。

### 5.2 测试文件结构

```
backend/
  ├── e2emodel/                   ← 模板目录（公共函数、模板脚本的唯一来源）
  │   ├── e2e_helpers.py          ← 公共函数库（所有测试脚本从这里导入）
  │   ├── conftest.py             ← fixtures
  │   ├── test_e2e_p0_01_introduce_self.py   ← 模板A: 无工具调用
  │   ├── test_e2e_p0_02_tool_call.py         ← 模板B: 单工具调用
  │   ├── test_e2e_p0_03_multi_step.py        ← 模板C: 多步推理
  │   └── test_e2e_p0_04_data_persistence.py  ← 模板D: 数据持久化
  └── tests/                      ← 实际测试脚本（从 e2emodel/ 导入公共函数）
      ├── test_e2e_p0_01_introduce_self.py
      ├── test_e2e_p6_err01_file_not_found.py
      └── ...
```

**铁律1：公共函数的唯一来源是 `backend/e2emodel/e2e_helpers.py`。**
测试脚本中禁止在本地维护一份独立的 `e2e_helpers.py`，必须通过 `from e2emodel.e2e_helpers import ...` 导入。

**铁律2：1个测试用例 = 1个测试函数，禁止合并。**

| 规则 | 说明 |
|------|------|
| 每个 E2E-XX-XX 编号对应一个 `async def test_e2e_xx_xx_xxx()` | 一一对应，不合并 |
| 每个函数只发一条用户指令 | 一个函数测一个场景 |
| 每个函数独立验证、独立清理 | 函数间无依赖 |
| 函数名 = `test_` + 编号小写 | 如 `test_e2e_p0_01_introduce_self` |

**为什么不能合并：**
- 合并后无法定位哪个场景失败
- 合并后无法单独重跑失败的用例
- 合并后一个场景的残留数据会影响下一个场景

### 5.3 测试辅助函数

**⚠️ 公共函数必须放在 `backend/e2emodel/e2e_helpers.py` 中，测试脚本从 `e2emodel` 导入：**

```python
# 正确：从模板目录导入公共函数
from e2emodel.e2e_helpers import (send_chat, check_db, verify_steps,
                                   write_test_record, ...)

# 错误：在 tests/ 目录下维护自己的 e2e_helpers.py
```

**注意：`e2e_helpers.py` 只存放在 `e2emodel/` 中，`tests/` 目录下不能有独立的副本。**

每个测试脚本从 `e2emodel.e2e_helpers` 导入并使用以下公共函数。

**注意：以下函数名、参数签名以 `backend/e2emodel/e2e_helpers.py` 中的实际实现为准，手册只列出各函数的核心用途。**

| 函数 | 用途 | 必用/可选 |
|------|------|----------|
| `ensure_backend_ready()` | 检查后端服务是否就绪 | **必用** |
| `send_chat(user_input, session_id=None, timeout_seconds=180, partial_result=None)` | 发POST /chat，接收SSE事件流；`partial_result`参数用于超时/中断时保留已收集events/tool_calls | **必用** |
| `assert_stream_ended(result)` | 返回流结束方式(`final`/`error`/`中断`)，不引发断言；替代旧`assert result["final_event"]` | **必用**(v1.8新增) |
| `check_db(session_id)` | 验证DB session/messages/steps完整性 | **必用** |
| `verify_consistency(result, session_id)` | 对比SSE事件与DB记录的一致性 | **必用** |
| `verify_steps(result, session_id)` | 验证执行步骤的合理性 | **必用** |
| `check_logs(ts, session_id)` | 检查日志文件中的ERROR/traceback | **必用** |
| `print_report(test_id, desc, r, db, lc, ci, si, passed, elapsed)` | 控制台输出结构化测试报告 | **必用** |

| `get_security_enabled()` / `set_security_enabled(flag)` | 读取/设置安全开关 | **必用** |

**`send_chat` 返回值结构（实际字段）：**
```python
{
    "events": [...],            # 所有SSE事件列表
    "final_event": {...},       # 最后一个final事件(可能为None：超时/中断时)
    "has_error": bool,          # 是否有error事件
    "total_steps": int,         # SSE事件总数（含chunk）
    "logical_step_count": int,  # 非chunk的逻辑事件数
    "unique_step_numbers": int, # 不重复的step编号数
    "tool_calls": [...],        # [{tool_name, tool_params}, ...]
    "llm_call_count": int,      # LLM调用次数
    "total_time_ms": int,       # 总耗时(ms)
    "response_text": str,       # final事件的回复文本(可能为空)
    "session_id": str,          # session_id
    "event_types": [...]        # 事件类型列表
}
```

**注意：`partial_result` 参数**——传入空dict后，`send_chat`在`finally`块中会将返回值写入该dict。
即使超时/中断导致协程取消，已收集的events/tool_calls/session_id也不会丢失：

```python
pr = {}  # 保证即使超时也能拿到部分数据
result = await send_chat(user_input, timeout_seconds=60, partial_result=pr)
result = pr  # 完整或部分数据都在pr中
```

### 5.4 测试用例模板

**模板提供骨架结构，每个脚本必须根据具体测试用例修改以下内容：**

| 模板中需修改的部分 | 说明 | 示例 |
|-------------------|------|------|
| 函数名 | = `test_` + 用例编号小写 | `test_e2e_p0_01_introduce_self` |
| docstring | 写清楚这个用例测什么 | `"P0-01: 核心链路验证 - 自我介绍"` |
| 用户指令 | = 测试用例中的用户输入 | `"详细介绍一下你自己"` / `"读取test.txt"` |
| 预期行为 | = 测试用例中预期的系统行为 | LLM回复 / 调用read_file / 创建文件 |
| 额外断言 | 根据用例场景增加特定检查 | 文件存在/内容非空/命令执行等 |

**验证方式分层（遵循3.6柔性验证原则）：**

| 层级 | 断言方式 | 示例 |
|------|---------|------|
| **MUST层** | `assert` 硬断言 | 无死循环、DB有记录、文件存在 |
| **SHOULD层** | `if...assert` 条件断言 | DB steps数偏差在2以内 |
| **MAY层** | `if...print` 记录不阻塞 | 工具名差异仅记录日志 |

**以下是与实际测试脚本一致的模板（E2E-ERR-01，含工具调用 + 测试记录写入）：**

```python
"""P6-ERR-01: 文件不存在 - Agent正常告知"找不到"
手册: 读取E:\test_dir\no_such_file.txt -> 回复"不存在"或"找不到"
-- 小健 2026-06-15
"""
from datetime import datetime
import pytest
from e2emodel.e2e_helpers import (ensure_backend_ready, send_chat, check_db,
    verify_consistency, verify_steps, check_logs, print_report,
    write_test_record, get_security_enabled, set_security_enabled,
    assert_stream_ended)

@pytest.mark.e2e_full_link
@pytest.mark.asyncio
async def test_e2e_p6_err01_file_not_found():
    ts = datetime.now()
    orig = get_security_enabled(); set_security_enabled(False)
    passed = False; r = None; sid = None; db = {}; ci = []; si = []
    lc = {"errors":[],"tracebacks":[]}; tn = []
    try:
        assert ensure_backend_ready()
        u = "读取E:\\test_dir\\no_such_file.txt"
        r = await send_chat(u)
        sid = r["session_id"]; el = r["total_time_ms"]/1000.0
        tn = [t["tool_name"] for t in r["tool_calls"]]

        # L1 - 流结束验证(所有方式有效，不阻断)
        end_type = assert_stream_ended(r)
        print(f"  流结束: {end_type}")

        # L1 - 无死循环(MUST)
        assert r["unique_step_numbers"] < 50, f"疑似死循环(MUST ERR-01)"

        # L2 - has_error降级为SHOULD WARN
        if r["has_error"]:
            print(f"  [WARN] 有error事件(SHOULD)，流结束: {end_type}")

        resp = r["response_text"]
        assert any(k in resp for k in ["不存在","找不到","没有"]), "回复应告知找不到"
        db = check_db(sid); assert db["session_exists"]
        ci = verify_consistency(r,sid); assert len(ci)==0
        si = verify_steps(r,sid); assert len(si)==0
        lc = check_logs(ts,sid); assert len(lc["errors"])==0
        passed = True
        print_report("E2E-ERR-01","文件不存在",r,db,lc,ci,si,True,el)
    except Exception as e:
        passed = False
        if sid: lc = check_logs(ts,sid)
        raise
    finally:
        if orig is not None: set_security_enabled(orig)
        write_test_record("ERR-01","文件不存在",u,r or {},db,ci,si,lc,passed)
    print(f"  [DONE] E2E-ERR-01 PASSED")
```

### 5.5 测试记录写入规范（铁律）

**每个测试用例跑完后，无论通过还是不通过，都必须自动生成测试记录文件。**

| 规则 | 说明 |
|------|------|
| **触发位置** | 测试脚本的 `finally` 块中调用 `write_test_record()` |
| **即使失败也要写** | assert失败/异常退出时，记录文件必须写入（含失败原因） |
| **记录内容** | 测试ID、任务描述、用户命令、执行时间、结果、SSE事件序列、工具链、DB验证、各层验证结果、失败原因 |
| **文件位置** | `G:\OmniAgentAs-desk\notes\测试记录-{用例编号}-{日期}.md` |
| **实现方式** | `e2e_helpers.py` 中的 `write_test_record()` 函数自动生成 |

**v1.8新增增强功能：**

| 增强项 | 说明 |
|--------|------|
| **`resp_has_error` 检查** | 回复文本中含`"错误:"` `"错误："` `"超时,"` `"超时，"` `"超时)"` `"超时）"` `"出错"` `"failed:"` `"exception:"` `"traceback:"` 等关键词时，记录自动标FAIL |
| **fallback DB check** | `db`参数为空且`result`有`session_id`时，自动调用`check_db(sid)`补查，测试无需各自加DB验证代码 |
| **流结束方式记录** | `assert_stream_ended(result)` 返回的 final/error/中断 写入报告"流结束"行 |

**测试脚本结构要求**：

```python
passed = False; r = None; sid = None; db = {}; ci = []; si = []; lc = {"errors":[],"tracebacks":[]}
try:
    # ... 测试逻辑 ...
    passed = True
except Exception as e:
    passed = False; raise
finally:
    write_test_record("用例编号","任务描述",u,r or {},db,ci,si,lc,passed,...)
```

**注意事项**：
- 所有变量必须在`try`前初始化默认值（防止assert失败后未定义）
- `write_test_record`必须在`finally`中调用（确保无论通过/失败都写入）
- 记录文件用`utf-8`编码，手动打开查看不会乱码
