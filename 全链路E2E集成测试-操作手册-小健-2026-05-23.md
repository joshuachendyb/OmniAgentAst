## 一、测试定义

### 1.1 测试名称

**全链路端到端集成测试**（Full-Link End-to-End Integration Test）

### 1.2 为什么叫这个名

| 关键词 | 含义 | 在这个测试中的体现 |
|--------|------|-------------------|
| 全链路 | 覆盖系统所有组件 | 前端API→后端→Agent→LLM→工具执行→数据库→SSE响应 |
| 端到端 | 从用户发消息到收到完整结果 | POST /chat → SSE事件流 → final事件 |
| 集成 | 验证各组件对接是否正确 | API路由、SSE封装、Agent循环、LLM调用、工具执行、数据库写入 |

### 1.3 与现有测试的区别

| 测试类型 | 测试什么 | 用什么驱动 | 能发现什么问题 |
|---------|---------|-----------|--------------|
| 单元测试（现有~200个） | 单个工具函数 | Mock/Patch | 函数级bug |
| MessageBuilder测试（43个） | Prompt构建逻辑 | Mock | Prompt构建bug |
| **全链路E2E（本方案）** | **Agent行为+系统集成** | **真实后端（读取config.yaml的LLM配置）+ 真实SQLite** | **死循环、工具选错、参数传错、会话污染、日志异常、数据库异常** |

### 1.4 测试目标

**唯一目标**：用真实LLM（后端从config.yaml读取配置）模拟真实用户操作，把系统中隐藏的问题炸出来。

**具体来说**：
1. Agent在真实LLM下能否正确完成用户请求
2. Agent在异常情况下（LLM返回垃圾/空内容/错误工具）能否容错
3. 系统各组件的串联是否完整（API→Agent→LLM→工具→DB）
4. 日志和数据库记录是否完整准确

### 1.5 核心测试原则（铁律）

**铁律1：测试的目的是发现问题，不是跑脚本。**

**铁律2：严禁批量跑测试用例。一次只能跑一个测试用例。**

| 允许 | 禁止 |
|------|------|
| ✅ 写一个脚本 → 跑一个 → 观察 → 检查 | ❌ 一次写完所有脚本 |
| ✅ 一个用例跑通后再写下一个 | ❌ 一次批量跑多个用例 |
| ✅ 每次跑完仔细观察系统运行过程 | ❌ 只看pass/fail就过 |
| ✅ 失败了停下来分析修复 | ❌ 失败了跳过继续下一个 |

**铁律3：所有测试统一基于真实后端运行，禁止Mock：**

| 环节 | 要求 | 禁止 |
|------|------|------|
| **后端** | 真实uvicorn启动，从config.yaml读取LLM配置 | ❌ 不启动后端、Mock后端 |
| **LLM** | 真实调用config.yaml中配置的provider/model | ❌ Mock LLM返回、假响应 |
| **工具** | 真实执行文件操作、命令、网络请求等 | ❌ Mock工具返回值 |
| **数据库** | 真实读写SQLite（~/.omniagent/chat_history.db） | ❌ Mock数据库、内存数据库 |
| **SSE** | 真实解析后端返回的SSE事件流 | ❌ 假造SSE事件 |

**一句话：测试输入就是用户会说的话，测试过程就是真实系统的完整运行。**

## 三、完整测试流程

每个测试用例严格按照 **测试前→测试中→测试后** 三阶段执行。
**测试前准备不到位，后续所有验证都不可能做好。**

### 3.1 测试前：准备工作（必须全部完成才能开始测试）

**准备工作分两大块：清除旧脚本 + 逐个编写新脚本。**

#### A. 清除旧测试脚本

```
操作:
  · 删除 backend/tests/ 下所有旧的 E2E 测试文件
  · 删除 backend/tests/e2e_artifacts/ 下所有旧的测试产物
  · 确认 conftest.py 中的 fixtures 是最新的

原因:
  · 旧脚本可能基于旧的系统行为,与当前代码不匹配
  · 旧脚本的断言可能与新的验证标准冲突
  · 清除后从零开始,确保每个脚本都是正确且最新的
```

#### B. 逐个编写测试脚本（铁律：一次只写一个）

```
⚠️ 禁止一次性编写所有脚本！

测试的目的 = 发现问题,不是跑10000遍
  · 一次性写完 → 无法验证每个脚本是否正确
  · 一次性写完 → 错误会累积,难以定位
  · 一次性写完 → 执行人员无法逐步推进
  · 批量测试 = 自欺欺人,发现不了真正的问题

正确做法: 一个用例一个脚本,写完跑通再写下一个

  第1步: 看测试用例 E2E-P0-01 "详细介绍自己"
         → 这个用例要测什么? (API→SSE通路)
         → 要发什么指令? ("详细介绍一下你自己，你能做什么")
         → 要验证什么? (收到final、无error、DB有记录)
         → 编写 test_e2e_p0_01_introduce_self()
         → 运行这个脚本
         → 观察系统运行过程,检查结果
         → 通过 → 进入第2步
         → 失败 → 分析错误、修复、重跑、通过后进入第2步

  第2步: 看测试用例 E2E-P0-02 "创建文件"
         → 编写 test_e2e_p0_02_create_file()
         → 运行、观察、检查
         → 通过 → 进入第3步

  ...逐个推进...

编写顺序: P0 → P1 → P2 → ... → P10
  · 先写最简单的（P0 hello）
  · 确认基础流程通了再写复杂的
  · 每写完一个,立即运行观察系统行为
```

#### C. 系统就绪检查（6项全部通过才能开始测试）

```
检查项 0: 重启后端服务（每次测试前必须执行）
  · 先杀死旧的 uvicorn 进程
    Windows: taskkill /F /IM python.exe /FI "WINDOWTITLE eq uvicorn*"
    或找到占用 8000 端口的进程杀掉
  · 重新启动后端
    cd backend && python -m uvicorn app.main:app --reload
  · 等待启动完成（约3-5秒）
  · 检查后端是否正常:
    curl http://localhost:8000/docs 返回 200
  · 失败处理: 检查端口占用/代码错误/配置问题,修复后重新启动
  · ⚠️ 即使使用 --reload,也必须确认后端正常响应才能继续

检查项 1: 后端服务已确认正常
  · 接检查项0的结果,确认返回200

检查项 2: LLM 配置
  · config.yaml 中 ai.provider 是否已配置有效 provider
  · 检查方法: 后端启动日志无 LLM 连接错误
  · 失败处理: 检查 config.yaml 的 ai 配置段

检查项 3: 安全开关
  · config.yaml 中 security.enabled 是否为 false（纯功能验证模式）
  · 检查方法: grep "enabled" config/config.yaml
  · 失败处理: 修改 config.yaml 设 security.enabled: false

检查项 4: 数据库
  · ~/.omniagent/chat_history.db 是否存在且可写
  · 检查方法: sqlite3 查询 SELECT count(*) FROM sessions
  · 失败处理: 检查目录权限

检查项 5: 日志目录
  · backend/logs/ 和 backend/logs/prompt-logs/ 是否可写
  · 检查方法: 尝试写入测试文件
  · 失败处理: 创建目录或修复权限

检查项 6: 测试数据
  · 测试用的目录和文件是否就绪
  · 检查方法: Test-Path 检查关键路径
  · 失败处理: 创建测试目录和文件

以上 A + B + C 全部完成 → 进入测试中阶段
任一项未完成 → 不开始测试
```

### 3.2 测试中：执行与验证

```
步骤 1: 记录测试起始状态
  · 记录当前时间 T0
  · 记录数据库当前会话数（base count）
  · 记录日志文件当前大小

步骤 2: 发送用户请求
  · 用 httpx 发送 POST /chat
  · 请求体: {"messages": [{"role": "user", "content": "用户输入"}]}
  · 设置 stream=True 以接收 SSE
  · 记录请求时间 T1

步骤 3: 接收并解析 SSE 事件流
  · 逐行读取 SSE 响应
  · 按 event: xxx 和 data: xxx 解析每个事件
  · 记录每个事件的:
    - 事件类型 (start/thought/action/observation/chunk/final/error)
    - 时间戳
    - step 编号
    - data 内容 (工具名、参数、结果等)
  · 持续接收直到 final 事件或超时
  · 记录接收完成时间 T2

步骤 4: 验证事件流正确性
  · [必须] 收到 final 事件 → 正常结束
  · [必须] 无 error 事件 → 无异常中断
  · [必须] 事件总数 < 50 → 无死循环
  · [检查] final 事件包含结果内容
  · [检查] 事件顺序正确 (start→thought→action→observation→...→final)

  ⚠️ 发现 FAIL → 执行步骤 4.5 错误处理

步骤 4.5: 错误处理（发现 FAIL 时的必须流程）
  错误分两类:
    A类(运行时): LLM超时/工具崩溃/死循环/SSE断流 → 查日志traceback
    B类(数据不一致): session缺失/messages丢失/steps不匹配 → 对比SSE vs DB

  处理流程:
    ① 分类 → A类还是B类
    ② 定位 → 查后端日志 + SSE事件 + prompt-logs,定位到文件:行号
    ③ 分析 → 是代码bug还是测试用例问题
    ④ 修复 → 遵守10大规范 (SRP/DRY/KISS/YAGNI,禁止backward compat)
    ⑤ 复测 → 重跑该用例 + 跑关联测试确认无回归
    ⑥ 记录 → 写入测试报告

  严重等级:
    P0(致命): 系统崩溃/死循环 → 立即停止,修复后从P0重跑
    P1(严重): 功能异常但不崩溃 → 修复后重跑该用例
    P2(一般): 非核心异常 → 记录,批量修复
    P3(警告): 性能偏差 → 记录,不阻塞

  禁止事项:
    ❌ 看到FAIL就跳过   ❌ 只改测试不改代码
    ❌ 修复时不跑单元测试  ❌ 不记录直接修复
```

### 3.3 测试后：数据验证与清理

```
步骤 5: 验证数据库记录完整性
  · chat_sessions 表
    - session 记录是否存在
    - is_valid 是否为 true（会话已生效）
    - created_at、updated_at 时间是否合理
  · chat_messages 表
    - user message 是否保存 (role=user, content=用户输入)
    - assistant response 是否保存 (role=assistant, content=AI回复)
    - 消息顺序是否正确 (user在前, assistant在后)
  · execution_steps (存储在 chat_messages.execution_steps 字段中)
    - 步骤数量是否 > 0 (有工具调用的任务必须有步骤记录)
    - 每个 step 的字段完整性:
      · step 编号连续
      · tool_name 非空
      · tool_params 已保存
      · observation (工具返回结果) 已保存
      · status 为 success 或 error

步骤 6: 三方数据一致性验证 (SSE事件 vs DB记录)
  数据来源: SSE事件从HTTP响应解析; DB记录从chat_messages表查询
  · SSE 事件数 vs DB execution_steps 数 → 数量必须一致
  · SSE 中 tool_name+tool_params vs DB step → 内容必须一致
  · SSE 中 observation vs DB step → 工具返回结果必须一致
  · SSE final response vs DB messages assistant content → 最终回复必须一致

步骤 7: step 步骤合理性验证
  数据来源: DB execution_steps 中的 ThoughtStep.thought/reasoning 字段
  · 工具选择合理性 → thought 中的推理与用户意图匹配
  · 参数正确性 → tool_params 中的路径/关键词/类型正确
  · 步骤顺序合理性 → step 编号顺序符合操作逻辑（先读后写等）
  · 结果完整性 → 每个工具调用都有 observation, 最终回复总结所有结果

步骤 8: 验证日志文件
  · app 日志 (backend/logs/app_*.log):
    - 无 ERROR 级别日志
    - 无异常 traceback
    - 有 session 操作记录（创建/更新/删除）
    - 有 SSE 事件发送记录
  · prompt-logs (backend/logs/prompt-logs/):
    - LLM 调用记录已保存（含 model/provider/消息摘要）
    - LLM 调用次数合理（无死循环）

步骤 9: 后置清理与记录
  · 记录测试总耗时 T2 - T1
  · 记录 LLM 调用次数、SSE 事件总数
  · 记录最终结论 (通过/失败/警告)
  · 清理测试产生的 session 记录 (session_id LIKE 'e2e_test_%')
  · 清理测试产生的文件
  · 恢复 config.yaml 中 security.enabled 为 true
```

### 3.4 验证标准总表

| 验证点 | 权重 | 通过标准 | 失败标准 | 验证来源 |
|--------|------|---------|---------|---------|
| 最终事件 | MUST | 收到 final | 未收到 final（超时或中断） | SSE事件流 |
| 错误事件 | MUST | 无 error 事件 | 任意 error 事件 | SSE事件流 |
| 事件总数 | MUST | 1~50（无死循环） | >= 50 或 0 | SSE事件流 |
| LLM调用次数 | SHOULD | 1~10次 | >= 10（可能死循环） | prompt-logs |
| DB session记录 | MUST | 存在且状态completed | 缺失或状态异常 | SQLite DB |
| DB messages记录 | MUST | user+assistant都保存 | 缺少任一 | SQLite DB |
| DB steps记录 | SHOULD | tool_name非空，observation存在 | 字段全空或缺失 | SQLite DB |
| SSE vs DB步骤数 | SHOULD | 数量偏差 ≤ 2 | 偏差 > 2（步骤丢失） | SSE事件 vs DB |
| SSE vs DB核心字段 | SHOULD | tool_name可匹配，observation语义一致 | tool_name完全不一致 | SSE事件 vs DB |
| 回复语义 | SHOULD | 长度>10字，语义与用户指令相关 | 回复为空或答非所问 | SSE final事件 |
| 工具选择 | MAY | 工具大类与用户意图匹配 | 工具大类完全不一致 | SSE action事件 |
| 参数正确性 | SHOULD | 关键参数（路径/关键词）正确 | 关键参数明显错误 | SSE action事件 |
| 步骤顺序 | MAY | 符合操作逻辑 | 顺序明显颠倒 | SSE事件序列 |
| 日志级别 | MUST | 无 ERROR | 有 ERROR | 后端日志文件 |
| 响应时间 | SHOULD | 单步<120s，多步<600s | 超时 | 计时 |

### 3.5 验证执行规范（铁律）

**核心要求**：每次测试必须手动（显式）执行步骤5+步骤6验证，不得依赖 fixture 或框架自动执行。

```
验证铁律:
  1. 每个测试用例必须显式调用 verify_db_events_consistency()
  2. 验证结果必须在测试代码中可见，不得藏在 fixture teardown 中
  3. 步骤5（DB记录完整性）必须验证:
     · session_exists
     · user message + assistant message 都存在
     · execution_steps 中每个 step 的 tool_name/tool_params/observation/status 字段完整性
  4. 步骤6（SSE vs DB一致性）必须验证:
     · tool_calls 数量一致
     · tool_name 一致
     · observation 内容一致（相似度 >= 80%）
     · final response 内容一致（相似度 >= 50%）
  5. 日志检查:
     · 无 ERROR 级别日志
     · 无异常 traceback
  6. 不满足任一验证 → 该测试标记为 FAILED
```

```
测试代码模板（必须遵守）:

  result = await send_chat_and_collect_events("...", session_id=session_id)

  # L1 - MUST层：SSE事件完整性
  assert result["final_event"] is not None, "必须收到final事件"
  assert result["has_error"] is False, "不应有error事件"

  # L2 - 步骤5：DB记录完整性验证（显式调用）
  db = check_database(session_id)
  assert db["session_exists"], "session必须保存到DB"
  assert db["has_user_message"], "应有用户消息"
  assert db["has_assistant_message"], "应有助手回复"

  # L3 - 步骤6：SSE vs DB一致性验证（显式调用）
  issues = verify_db_events_consistency(result, session_id)
  assert len(issues) == 0, (
      f"DB-SSE一致性验证失败:\n" + "\n".join(f"  - {i}" for i in issues)
  )
```

**禁止事项**:
  ❌ 禁止把步骤5+步骤6的验证逻辑隐藏在 fixture 或 conftest 中
  ❌ 禁止 fixture 自动执行 DB-SSE 一致性验证
  ❌ 禁止不显式调用而依赖 teardown 隐式验证

### 3.6 柔性验证原则（最优美的验证方法）

真实LLM的输出具有**非确定性**（同一输入每次回复不同），因此验证方法必须从"刚性精确匹配"升级为**柔性语义验证**。

| 原则 | 刚性验证（禁止） | 柔性验证（最优美） |
|------|----------------|-------------------|
| **工具名验证** | `assert tool_name == "write_text_file"` | `assert any("write" in t for t in tool_names)` |
| **事件顺序** | `assert events[1]["type"] == "thought"` | `assert "thought" in event_types` |
| **步骤数量** | `assert steps == 5` | `assert 1 <= steps <= 20` |
| **回复内容** | `assert "hello" in response` | `assert len(response) > 10`（语义完整即可）|
| **文件验证** | `assert content == "hello world"` | 文件存在 + 内容非空 + 内容与指令相关 |
| **LLM调用** | `assert llm_calls == 3` | `assert 1 <= llm_calls <= 8`（无死循环即可）|

**核心思想**：
- **语义为王**：验证"结果是否符合用户意图"，不验证"是否走了预期路径"
- **自适应断言**：使用范围、包含、模式匹配代替精确值
- **权重分层**：核心属性（文件存在/DB有记录）严格验证，行为属性（选了哪个工具）宽松验证
- **异常不被忽视**：error事件、崩溃、死循环必须零容忍，但对工具选择差异保持弹性

**事件流验证的优美模式**：
```python
# 最优美的验证方式：分层验证
# L1 - 强制层（不通过就是系统缺陷）
assert result["has_error"] == False      # 不能有error事件
assert result["final_event"] is not None # 必须有final（不能死循环）
assert result["total_steps"] < 50        # 不能死循环

# L2 - 语义层（验证结果质量）
assert len(result["response_text"]) > 10 # 回复不能为空或敷衍

# L3 - 检查层（记录但不阻塞）
if file_path_exists:
    assert file_exists, f"文件应已创建: {file_path}"
```

**验证权重定义**：

| 权重 | 含义 | 失败处理 | 示例 |
|------|------|---------|------|
| **MUST** | 必须满足 | 测试失败，停止 | 无error、有final、文件存在 |
| **SHOULD** | 应该满足 | 记录警告，不失败 | 工具名合理、步骤数合理 |
| **MAY** | 建议检查 | 仅记录日志 | LLM调用了A工具而非B工具 |

## 五、实现方式

### 5.1 测试框架选择

使用 **pytest**，与现有测试体系一致。

### 5.2 测试文件结构

```
backend/tests/
  ├── test_e2e_full_link.py       ← 主测试文件（本方案的核心）
  ├── conftest.py                  ← fixtures（backend_client、db检查、日志检查等）
```

**铁律：1个测试用例 = 1个测试函数，禁止合并。**

| 规则 | 说明 |
|------|------|
| 每个 E2E-XX-XX 编号对应一个 `async def test_e2e_xx_xx_xxx()` | 一一对应，不合并 |
| 每个函数只发一条用户指令 | 一个函数测一个场景 |
| 每个函数独立验证、独立清理 | 函数间无依赖 |
| 函数名 = `test_` + 编号小写 | 如 `test_e2e_p0_01_introduce_self` |

**为什么不能合并：**
- 合并后无法定位哪个场景失败
- 合并后无法单独重跑失败的用例
- 合并后一个场景的残留数据会影响下一个场景

### 5.4 测试用例模板

**模板提供骨架结构，每个脚本必须根据具体测试用例修改以下内容：**

| 模板中需修改的部分 | 说明 | 示例 |
|-------------------|------|------|
| 函数名 | = `test_` + 用例编号小写 | `test_e2e_p0_01_introduce_self` |
| docstring | 写清楚这个用例测什么 | `"P0-01: 核心链路验证 - 自我介绍"` |
| 用户指令 | = 测试用例中的用户输入 | `"详细介绍一下你自己"` / `"读取test.txt"` |
| 预期行为 | = 测试用例中预期的系统行为 | LLM回复 / 调用read_file / 创建文件 |
| 额外断言 | 根据用例场景增加特定检查 | 文件存在/内容非空/命令执行等 |

**验证方式分层（遵循3.6柔性验证原则）：**

| 层级 | 断言方式 | 示例 |
|------|---------|------|
| **MUST层** | `assert` 硬断言 | 无error、有final、文件存在 |
| **SHOULD层** | `if...assert` 条件断言 | DB steps数偏差在2以内 |
| **MAY层** | `if...print` 记录不阻塞 | 工具名差异仅记录日志 |

**以下是一个具体用例的脚本示例（E2E-P0-01，柔性验证版）：**

```python
@pytest.mark.e2e_full_link
@pytest.mark.asyncio
async def test_e2e_p0_01_introduce_self():
    """P0-01: 核心链路验证 - 详细介绍自己，验证API→SSE通路"""
    
    # === 测试前 ===
    # 3.1 系统就绪验证（由conftest fixture自动检查）
    test_start = datetime.now()
    session_id = None
    
    # === 测试中 ===
    # 2. 发送请求并接收SSE事件流
    result = await send_chat_and_collect_events(
        "详细介绍一下你自己，你能做什么"
    )
    
    # === 验证（柔性分层） ===
    
    # L1 - MUST层（不通过就是系统缺陷）
    assert result["has_error"] == False, "系统不应产生error事件"
    assert result["final_event"] is not None, "系统必须正常结束(final事件)"
    assert len(result["events"]) >= 2, "至少需要start+final两个事件"
    assert result["total_steps"] < 50, f"疑似死循环: {result['total_steps']}步"
    
    # L2 - SHOULD层（验证结果质量）
    if result.get("response_text"):
        resp = result["response_text"]
        assert len(resp) > 10, f"回复太短({len(resp)}字)，疑似敷衍"
        # 语义验证：回复应包含自我介绍相关内容
        key_terms = ["助手", "AI", "可以", "能够", "帮助"]
        assert any(t in resp for t in key_terms), "回复内容与自我介绍无关"
    
    # L3 - MAY层（记录不阻塞）
    session_id = result["final_event"].get("session_id")
    event_types = [e["type"] for e in result["events"]]
    if len(event_types) > 2:
        print(f"  [INFO] 事件序列: {'→'.join(event_types)}")
    
    # === 测试后 ===
    if session_id:
        # 5. 验证DB记录
        db_check = check_database(session_id)
        assert db_check["session_exists"], "session必须保存到数据库"
        assert db_check["messages_count"] >= 2, "必须保存user+assistant消息"
        
        # 8. 验证日志
        log_check = check_logs("e2e-p0-01", test_start)
        assert len(log_check["errors"]) == 0, \
            f"日志不应有ERROR: {log_check['errors']}"
        
        # 9. 清理
        cleanup_test_data(session_id)
```

- **工具层**（P1-P4）：57个工具在真实LLM下能否被正确调用
- **流程层**（P5-P7）：多步推理、任务控制、系统串联是否完整
- **容错层**（P6/P9-P10）：异常输入、工具失败、服务中断时能否优雅处理
