## 一、背景与需求

### 1.1 问题背景

在原有架构中，AI服务采用**专用文件模式**：
```
services/
├── zhipuai.py    # 智谱专用服务
├── opencode.py   # OpenCode专用服务
└── ...           # 每个provider一个文件
```

**存在的问题**：
| 问题 | 影响 |
|------|------|
| 扩展困难 | 新增provider需要编写新的专用代码 |
| 维护成本高 | 每个provider都有重复代码 |
| 配置不透明 | 用户不知道配置是否正确 |
| 错误无反馈 | 前端无法获取配置验证结果 |

### 1.2 解决方案

采用**通用架构 + 配置驱动**模式：
```
services/
├── base.py       # 通用AI服务类（支持所有OpenAI兼容API）
└── __init__.py   # 工厂模式 + 配置验证
```

**架构优势**：
| 优势 | 说明 |
|------|------|
| 一个类支持所有provider | `BaseAIService` 兼容所有OpenAI API格式 |
| 配置驱动 | 只需修改 `config.yaml`，无需改代码 |
| 零代码扩展 | 新增provider只需添加配置项 |
| 配置可验证 | 提供完整的验证结果供前端显示 |

### 1.3 接口设计目的

本接口的设计目的是：

1. **配置透明化**：让前端能够获取配置验证结果，不再"盲目使用"
2. **错误可追溯**：当配置出错时，提供明确的错误列表和警告列表
3. **用户体验提升**：前端可以根据验证结果引导用户修复配置
4. **调试便利**：开发和测试时快速定位配置问题

## 三、接口详情

### 2.1 完整配置验证（新增）

**用途**：验证当前配置文件是否正确，返回详细的错误和警告列表，供前端显示。

**请求**：
```http
GET /api/v1/config/validate-full
```

**响应**：
```json
{
  "success": true,
  "provider": "opencode",
  "model": "minimax-m2.5-free",
  "message": "配置验证通过: provider=opencode, model=minimax-m2.5-free",
  "errors": [],
  "warnings": []
}
```

**字段说明**：

| 字段 | 类型 | 说明 |
|------|------|------|
| `success` | boolean | 验证是否成功 |
| `provider` | string | 当前使用的Provider名称 |
| `model` | string | 当前使用的模型名称 |
| `message` | string | 验证结果消息 |
| `errors` | string[] | 错误列表，需要用户处理 |
| `warnings` | string[] | 警告列表，提示但不阻塞 |

**验证项**：
1. 配置文件是否存在
2. ai 配置块是否存在
3. provider 字段是否存在
4. provider 配置是否存在
5. api_key 是否存在且非空
6. model 是否存在且非空
7. api_base 是否存在

**失败响应示例**：
```json
{
  "success": false,
  "provider": "zhipuai",
  "model": "",
  "message": "配置验证失败: 2 个错误",
  "errors": [
    "provider 'zhipuai' 缺少 api_key 配置",
    "provider 'zhipuai' 缺少 model 配置"
  ],
  "warnings": [
    "provider 'zhipuai' 未配置 api_base，将使用默认值"
  ]
}
```

### 2.2 单个API Key验证（已修复）

**用途**：验证用户输入的API Key是否有效，测试与AI服务的连接。

**请求**：
```http
POST /api/v1/config/validate
Content-Type: application/json

{
  "provider": "opencode",
  "api_key": "your-api-key-here"
}
```

**请求参数**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `provider` | string | 是 | AI提供商名称（如 opencode、zhipuai） |
| `api_key` | string | 是 | 要验证的API Key |

**成功响应**：
```json
{
  "valid": true,
  "message": "API Key验证成功，当前使用 opencode",
  "model": "minimax-m2.5-free"
}
```

**失败响应**：
```json
{
  "valid": false,
  "message": "API Key无效，请检查是否正确",
  "model": null
}
```

## 四、前端调用示例

### 3.1 完整配置验证

```typescript
// 调用完整配置验证
async function validateFullConfig() {
  try {
    const response = await fetch('/api/v1/config/validate-full');
    const result = await response.json();
    
    if (result.success) {
      console.log('配置验证通过');
      console.log(`当前Provider: ${result.provider}`);
      console.log(`当前Model: ${result.model}`);
    } else {
      // 显示错误列表
      console.error('配置验证失败:');
      result.errors.forEach(err => console.error(`❌ ${err}`));
      result.warnings.forEach(warn => console.warn(`⚠️ ${warn}`));
      
      // 可以在UI上显示这些错误
      showConfigErrors(result.errors, result.warnings);
    }
    
    return result;
  } catch (error) {
    console.error('验证请求失败:', error);
    throw error;
  }
}
```

### 3.2 单个API Key验证

```typescript
// 验证用户输入的API Key
async function validateApiKey(provider: string, apiKey: string) {
  try {
    const response = await fetch('/api/v1/config/validate', {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({
        provider: provider,
        api_key: apiKey,
      }),
    });
    
    const result = await response.json();
    
    if (result.valid) {
      console.log('API Key验证成功');
      // 可以保存配置
    } else {
      console.error('API Key验证失败:', result.message);
      // 提示用户检查API Key
    }
    
    return result;
  } catch (error) {
    console.error('验证请求失败:', error);
    throw error;
  }
}
```

### 6.5 发现的真正Bug

**问题描述**：`backend/app/api/v1/config.py` 的 `validate_config` 函数中，临时创建的 `BaseAIService` 在异常情况下未关闭 `httpx.AsyncClient`。

**问题代码**（修改前）：
```python
temp_service = BaseAIService(...)

# ❌ 如果 validate() 抛出异常，close() 不会被调用

```

**修复代码**（修改后）：
```python
temp_service = BaseAIService(...)
try:
    is_valid = await temp_service.validate()
    # 返回结果...
finally:
    await temp_service.close()  # ✅ 异常时也会执行
```

## 八、前端配合分析（小新补充）

### 7.1 当前获取下拉列表的流程（有问题）

```
进入Web页面
    ↓
Layout.useEffect() 
    ↓
直接调用 configApi.getModelList() ← 没有验证配置！
    ↓
后端直接读取config.yaml返回模型列表
    ↓
问题：如果config.yaml不存在/字段缺失 → 返回空列表，前端不知道原因
```

### 7.2 正确的流程应该是

```
进入Web页面
    ↓
先调用 GET /api/v1/config/validate-full 验证配置
    ↓
验证成功(success=true) → 调用 getModelList() 获取下拉列表
验证失败(success=false) → 显示 errors[] + warnings[]，不获取列表或显示空列表
```

### 7.3 前端需要修改的地方

**文件**: `frontend/src/components/Layout/index.tsx`

**修改点**: `useEffect` 里的 `initApp` 函数

```tsx
// 现在是这样的（有bug）
const initApp = async () => {
  const [modelData, config] = await Promise.all([
    configApi.getModelList(),      // ← 直接获取，没验证
    configApi.getConfig()
  ]);
  // ...
};

// 应该改成这样
const initApp = async () => {
  // 1. 先验证配置
  const validation = await configApi.validateFullConfig();
  
  if (!validation.success) {
    // 验证失败，显示错误
    setValidationErrors(validation.errors);
    setValidationWarnings(validation.warnings);
    return;  // 不获取模型列表，或显示空列表
  }
  
  // 2. 验证成功才获取模型列表
  const [modelData, config] = await Promise.all([
    configApi.getModelList(),
    configApi.getConfig()
  ]);
  // ...
};
```

### 10.8 注意事项

1. **验证失败不阻塞**：验证失败只是提示用户，不影响应用基本运行
2. **错误和警告区分**：
   - **错误(errors)**：必须修复，否则功能无法正常使用
   - **警告(warnings)**：建议修复，但不影响基本功能
3. **弹框复用**：Layout和Settings页面的验证弹框代码相同，可以抽取为公共组件
4. **避免重复调用**：验证结果可以缓存，避免频繁请求后端
